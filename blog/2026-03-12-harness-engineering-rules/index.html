给 AI 立规矩：Harness Engineering 的落地实践

Posted 2026-03-12 00:00 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 ##

上一篇文章聊了可验证循环——它是从 AI Assist 进化到 AI First 的核心机制。但要让这个循环真正转起来，团队面临一个更实际的问题：怎么把工程经验告诉 AI，让它持续稳定地产出符合团队标准的代码？

这就是 Harness Engineering 要解决的问题。

从 Prompt Engineering 到 Harness Engineering ##

回顾一下 AI coding 领域的概念演进：

Prompt Engineering (2023)
  "如何写好提示词让 AI 给出好回答"
  问题：每次都要重写，无法复用，效果不稳定
        │
        ▼
Context Engineering (2024)
  "如何组织好上下文让 AI 理解完整信息"
  问题：上下文组织有了章法，但仍然是被动的
        │
        ▼
Harness Engineering (2025-2026)
  "如何系统性地约束、引导、驾驭 AI 的完整行为"
  核心：从"每次手动告诉 AI"变成"建立一套可复用的约束体系"

Harness Engineering 的本质：把团队积累的工程经验、编码规范、领域知识，编码为 AI IDE 可以自动加载和执行的规则。这些规则不依赖单次提示词的质量，而是在每次 AI 生成代码时自动生效。

可以这样理解三者的区别：

Prompt EngineeringContext EngineeringHarness Engineering
核心动作写好单条提示词组织好对话上下文建立规则体系
生效范围单次对话单次会话持续、跨会话
复用性低（每次手写）中（模板化）高（自动加载）
对人的要求写好提示词组织好信息设计好规则
对 AI 的约束软约束（靠措辞）中约束（靠信息完整度）硬约束（靠规则强制）

Harness Engineering 的三层组件 ##

在落地 Harness Engineering 时，我们可以把约束分为三个层次：

Rules（规矩层） ##

Rules 回答：AI 应该怎么做。

Rules 是最基础的约束，定义了编码规范、技术选型、安全要求等"不可协商"的底线。它们应该是 AI 每次生成代码时自动加载的上下文。

Rules 的设计原则：

  1. 具体可执行：AI 能直接照做，不需要"理解精神"
    • ❌ “写好代码”
    • ✅ “函数体不超过 50 行，参数不超过 5 个”
  2. 正反例对照：给出正确和错误示例，AI 通过对比学习更快
  3. 区分强制和建议：用 MUST / SHOULD / MUST NOT 明确优先级
  4. 持续演进：基于实际使用效果迭代，不是写完就不管了

Skills（手册层） ##

Skills 回答：AI 应该怎么做一件复杂的事。

Skill 是比 Rule 更高级的抽象，它是一个完整的、多步骤的工作流模板。如果说 Rule 是"编码规范"，Skill 就是"操作手册"。

Rule 和 Skill 的区别：

Rule  ≈ 交规（红灯停、绿灯行、限速 60）—— 被动约束
Skill ≈ 导航路线（从 A 到 B 经过哪些路口）—— 主动引导

Commands（快捷键层） ##

Commands 回答：人能一键触发什么操作。

Commands 是最贴近开发者日常体验的一层。它把常用的 AI 操作封装为 /command 形式的快捷键，降低使用门槛。

Rule    ≈ 规矩（被动约束，AI 自动遵守）
Skill   ≈ 手册（主动引导，AI 按流程执行）
Command ≈ 快捷键（一键触发，人主动调用）

实战：设计一套 Rules ##

以 Go 项目为例，一套基础的 Rules 可以这样组织：

文件结构 ##

project-root/
├── .cursor/rules/          # AI IDE 的 Rules 目录
│   ├── go-code-style.mdc   # always: 核心编码规范
│   ├── go-test.mdc         # always: 测试规范
│   ├── go-error.mdc        # always: 错误处理规范
│   ├── proto-style.mdc     # auto: Proto 规范（按需加载）
│   └── arch-review.mdc     # manual: 架构规范（手动触发）
│
├── .cursor/skills/
│   ├── requirement-analysis.md   # 需求分析
│   ├── tdd-workflow.md          # TDD 工作流
│   └── code-review.md           # AI 辅助 Code Review
│
└── docs/ai-rules/               # 工具无关的通用规范
    ├── go-code-style.md
    └── ...

Rule 文件模板 ##

---
description: Go 语言编码规范，适用于所有 Go 源代码
globs: "**/*.go"
alwaysApply: true
---

# Go 编码规范

## 目标
确保团队 Go 代码风格统一、可读性强、易于维护。

## 规则

### 命名规范（MUST）
- 包名：小写单词，不使用下划线。✅ `userservice` ❌ `user_service`
- 导出标识符：PascalCase。✅ `UserService`
- 接口名：单方法接口以 `-er` 后缀。✅ `Reader`、`Closer`

### 函数设计（MUST）
- 函数体不超过 50 行（不含注释和空行）
- 函数参数不超过 5 个，超过时使用 Option 结构体
- 返回值：正常值在前，error 在最后

### 错误处理（MUST）
- 绝不忽略 error
- 使用 `fmt.Errorf("context: %w", err)` 包装错误
- 错误信息小写开头

## 示例

### ✅ 正确
// 略，正例

### ❌ 错误
// 略，反例

## 原因
统一的编码风格降低代码审查的认知负担。遵循 Go 社区标准降低新人上手成本。

有几个关键点：

  1. alwaysApply: true：核心编码规范应该每次对话都加载，这是底线性约束
  2. globs 匹配：**/*.go 确保只在编辑 Go 文件时触发，编辑 Proto 时不干扰
  3. 正反例对照：AI 通过对比学习效果更好，比单纯文字描述更稳定
  4. 必须说明原因：让使用 Rule 的人（和未来的维护者）理解决策背景

常见踩坑 ##

在实际使用中，容易犯的几个错误：

  1. 规则太模糊：“写好代码”——AI 不知道怎么执行
  2. 规则太多：100 条规则 = 没有规则，AI 的上下文窗口是有限的
  3. 只给规则不给原因：维护者不知道为什么这么定，后续迭代容易跑偏
  4. 写完就不管了：规则需要基于实际使用效果持续迭代

实战：设计一套 Skills ##

Skill 比 Rule 更复杂，它是一个有步骤、有输入输出、有检查点的完整工作流。

Skill 文件结构 ##

---
description: 将产品需求拆解为可执行的开发任务
---

# 需求分析与任务拆解

## 触发条件
当开发者收到新需求时使用。

## 步骤

### Step 1: 需求理解与澄清
**输入**：原始需求描述
**操作**：提炼核心目标、识别关键实体、列出模糊点
**输出**：需求理解文档（含待确认事项）
**检查点**：开发者确认理解无误 → 进入下一步

### Step 2: 技术方案设计
**输入**：确认后的需求理解
**操作**：分析影响范围、设计数据模型、设计 API
**输出**：技术方案文档
**检查点**：开发者 Review 方案 → 确认或修改

### Step 3: 任务拆解
**输入**：确认后的技术方案
**操作**：拆解为独立可测试的子任务（每个不超过 4h）
**输出**：任务清单（含依赖关系和验收标准）

Skill 设计的核心原则：每步都有明确的输入、输出和检查点。这样人可以在任一步骤暂停、确认、调整，而不是等 AI 一次性输出所有内容后再修改。

关于 AI Code Review 的一个关键洞察 ##

很多人问：如果代码大部分是 AI 生成的，那 AI Code Review 还有意义吗？

我的看法是：不仅有意义，而且必须升级 Review 的模式。

分层 Review 模型 ##

L0 格式检查 ──── CI 自动（100% 自动化）
L1 规范检查 ──── AI Review（90% 自动化）
L2 安全检查 ──── AI + 人（70% 自动化）
L3 性能检查 ──── AI + 人（50% 自动化）
L4 逻辑正确性 ── 人为主（10% 自动化）
L5 架构合理性 ── 人（0% 自动化）

核心理念：AI Review 不是替代人工 Review，而是过滤噪音，让人类聚焦高价值审查。

  • L0-L2 的问题（格式、命名、常见 bug 模式）AI 可以处理得很好——不应该让人类逐行检查这些
  • L3-L4 的问题（性能、业务逻辑）需要人机配合
  • L5 的问题（架构合理性、设计意图）目前只能靠人

分层之后，人类 Reviewer 的注意力从"找低级错误"转移到"判断设计决策"——这才是真正需要资深工程师经验的地方。

如何避免"规则腐烂" ##

Rules 和 Skills 写完之后最大的风险是规则腐烂：时间一长，规则和实际行为脱节，AI 被过时的规则误导，团队失去对规则的信任。

避免规则腐烂的机制：

1. 建立反馈闭环 ##

团队成员在 AI 协作中发现问题
        │
        ▼
  判断：是 Rules 不够好，还是 AI 没遵守？
        │
        ▼
  提出 Rule/Skill 改进 → PR → Tech Lead Review → 合并生效
        │
        ▼
  观察效果 → 进入下一个反馈周期

2. 设定迭代节奏 ##

频率动作
每周收集 AI 辅助编码中遇到的问题案例
每两周Review 和更新 Rules，修复不好用的规则
每月评审 Skills 使用频率和效果
每季度全面回顾，淘汰无效规则，新增必要规则

3. 保持工具无关性 ##

不要 All-in 单一工具的 Rules 格式。建议的做法是：

docs/ai-rules/          ← 工具无关的通用规范（Markdown）
  ├── go-code-style.md
  └── ...

.cursor/rules/          ← Cursor 适配（引用核心规范 + IDE 特有元数据）
.codebuddy/rules/       ← CodeBuddy 适配
CLAUDE.md               ← Claude Code 适配

核心规范用通用 Markdown 维护，各 IDE 的 Rules 文件从核心规范派生。这样团队成员使用不同工具时，遵循的是同一套标准。

总结 ##

Harness Engineering 不是什么高深的概念，它的本质就是把资深工程师的经验从"人脑中"搬到"文件里"，让 AI 成为这些经验的忠实执行者。

几点关键认知：

  1. Rules 不是越多越好。10 条好的 Rules 胜过 100 条模糊的 Rules。每条 Rule 都必须是"可执行的"——AI 能直接照着做，不需要揣测精神。
  2. 正反例比纯文字描述更有效。AI 通过对比学习规则的效果，远好于只看文字描述。
  3. Rules 需要像代码一样维护——有版本控制、有 Review、有迭代、有淘汰。
  4. 工具无关性很重要。不要把规则锁死在一个 AI IDE 的格式里。用通用 Markdown 维护核心规范，各 IDE 做适配。
  5. Harness Engineering 是"可验证循环"的落地手段。Rules 定义了"什么是好代码"，自动化测试验证了"代码是不是好代码"，两者配合才能让 AI First 模式真正运转起来。
