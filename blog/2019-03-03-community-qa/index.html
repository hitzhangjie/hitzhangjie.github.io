Go 社区问答与设计考量

Posted 2019-03-03 12:00 +0800 by ZhangJie ‐ 8 min read

分享:  

本文整理了 Go 语言官方 FAQ 及社区讨论中反复出现的问题，涵盖语言设计、类型系统、内存模型、并发模型等方面的常见疑问及其背后的设计考量。

Go 泛型支持 #

Go 只有有限的泛型支持，只有内置的某些函数支持泛型（如 fmt.Println），用户自定义的函数不支持泛型。

泛型是 Go 开发者被询问最多的问题之一。Go 团队表示，Go 的核心是 simplicity and powerful，在实际开发中，代码写得越多越觉得没有必要支持泛型。

Q: Do you have plans to implement generics?

A: Many proposals for generics-like features have been mooted both publicly and internally, but as yet we haven’t found a proposal that is consistent with the rest of the language. We think that one of Go’s key strengths is its simplicity, so we are wary of introducing new features that might make the language more difficult to understand. Additionally, the more Go code we write (and thus the better we learn how to write Go code ourselves), the less we feel the need for such a language feature.

Go 适用场景 #

Go 设计之初就将系统级编程考虑在内。开发网络 server、client、database、cache、balancer、distributor 都适合用 Go，这也是 Google 内部最初使用 Go 的地方。然而 Go 发布之后，社区开发者在不同场景下使用 Go，使得 Go 在更多领域发光发热。由于 Go 最初将系统级编程的思想考虑在内，设计足够通用，可以说限制 Go 应用的仅仅是 library 的多少而已。

Q: What can I use Go for?

A: Go was designed with systems programming in mind. Servers, clients, databases, caches, balancers, distributors - these are applications Go is obviously useful for, and this is how we have begun to use it within Google. However, since Go’s open-source release, the community has found a diverse range of applications for the language. From web apps to games to graphics tools, Go promises to shine as a general-purpose programming language. The potential is only limited by library support, which is improving at a tremendous rate. Additionally, educators have expressed interest in using Go to teach programming, citing its succinct syntax and consistency as well-suited to the task.

Go 是面向对象语言吗？ #

Go 是一门"object-oriented but not type-oriented“的语言。面向对象编程只是一种思想，C 虽然是过程式语言，但在 Linux Kernel 开发中仍然可以看到很多 OO 的设计思想；C++ 支持面向对象，Java 强制面向对象，但这些特性使得语言变得过重了。

Go 提供了 interface，允许其他 type 实现 interface，以一种 loose coupled 的形式近似实现了 runtime polymorphism（运行时多态）；另外，Go 也提供了 interface 组合、struct 组合能力，通过组合来近似实现了 class inheritance（类继承）。

可以说 Go 是一门在语言层面支持面向对象编程的语言，但它不像 C++、Java 那样强制 implements、extends，是一种 loose coupled 的 object-oriented、not type-oriented 的语言。从这个层面上讲，Go 兑现了 simple & powerful 的目标。

Go 移动平台支持 #

Go 编译器支持生成 ARM 代码，因此支持移动平台开发。尽管 Go 团队早期在这方面的投入较少，但到 2019 年，gomobile 已经发布，开发者可以用 Go 开发移动平台应用。目前多还是以 library 的形式，然后通过 Java 调用（Android），iOS 平台则支持 native 开发。

Q: Are there plans to support Go under Android?

A: Both Go compilers support ARM code generation, so it is possible. While we think Go would be a great language for writing mobile applications, Android support is not something that’s being actively worked on.

Go 代码风格 #

为什么要统一风格？每个人都有不同的代码风格，尽管存在那么多 best practices。代码风格是重要的——代码写出来是给机器运行的，但更可能是要给其他开发人员看的。代码风格一致可以减少开发者浪费在代码风格上的精力，专注于代码逻辑本身。gofmt 会自动完成代码风格的处理，提高代码的可读性、可维护性。

关于 Tab vs Space：Go 官方建议使用 Tab，但某些 code review 平台对于 tab、space 混排的展示会有问题，鉴于此，我个人建议使用 Space。

Indentation We use tabs for indentation and gofmt emits them by default. Use spaces only if you must.

Go Getters & Setters #

Go 不会为结构体字段自动生成 Getters & Setters，开发者需要自己按需定义。Setter 方法名加 Set 前缀，Getter 方法名不要加 Get——没必要，加了反而显得多余、累赘。

Getters: Go doesn’t provide automatic support for getters and setters. There’s nothing wrong with providing getters and setters yourself, and it’s often appropriate to do so, but it’s neither idiomatic nor necessary to put Get into the getter’s name. If you have a field called owner (lower case, unexported), the getter method should be called Owner (upper case, exported), not GetOwner. The use of upper-case names for export provides the hook to discriminate the field from the method. A setter function, if needed, will likely be called SetOwner.

Both names read well in practice:

owner := obj.Owner()
if owner != user {
    obj.SetOwner(user)
}

Go interface 及 method 命名 #

接口名以 “er” 结尾，方法名表示动作，尽量不要和接口名重复。

Interface names: By convention, one-method interfaces are named by the method name plus an -er suffix or similar modification to construct an agent noun: Reader, Writer, Formatter, CloseNotifier etc. There are a number of such names and it’s productive to honor them and the function names they capture. Read, Write, Close, Flush, String and so on have canonical signatures and meanings. To avoid confusion, don’t give your method one of those names unless it has the same signature and meaning. Conversely, if your type implements a method with the same meaning as a method on a well-known type, give it the same name and signature; call your string-converter method String not ToString.

Go identifiers #

变量和结构体字段命名，不要用下划线分割的形式，使用 MixedCaps 风格（导出）或者 mixedCaps（不导出）。

MixedCaps: Finally, the convention in Go is to use MixedCaps or mixedCaps rather than underscores to write multiword names.

Go 特殊操作符 #

首先，Go 没有逗号运算符。在 C/C++ 中可以写 for (i=0, j=1, k=2; i<100; i++)，但 Go 中必须使用多重赋值：

for i, j, k := 0, 1, 2; i<100; i++ {
    ...
}

另外，++、-- 在 Go 中是 statement 而非 operator，并且只有后置这一种写法（如 i++，不能写 ++i）。在 C/C++ 中可能写 printf("%d %d\n", i++, j++)，但在 Go 中 ++、-- 是语句，没有返回值，fmt.Println(i++, j++) 会编译报错 syntax error: unexpected ++, expecting )。正确的写法：

i++
j++
fmt.Println(i, j)

Finally, Go has no comma operator and ++ and – are statements not expressions. Thus if you want to run multiple variables in a for you should use parallel assignment (although that precludes ++ and –).

// Reverse a
for i, j := 0, len(a)-1; i < j; i, j = i+1, j-1 {
    a[i], a[j] = a[j], a[i]
}

Go defer 函数 #

defer 入栈的函数 f，在函数退出阶段 f 执行时，它的参数才会开始计算。这里涉及传值、传引用，以及闭包中变量的按值捕获、按引用捕获问题。

比如下面这个示例，defer func f(v int) 中 v 是按值传递的，output 时输出 0；变量 i 则是按引用捕获的，因为后续 i++ 对其有修改，最终输出 1。所以最终输出 0 1：

package main
import "fmt"

func main() {
    i := 0
    defer func(v int) {
        fmt.Println(v, i)
    }(i)
    i++
}

Output: 0 1

We can do better by exploiting the fact that arguments to deferred functions are evaluated when the defer executes.

Go 内存分配：new vs make #

Go 里面有两个内存分配的方法：new 和 make。前者分配一块内存并将内存清零，返回指向内存的指针；后者分配内存并完成初始化过程。

slice 可以通过 SliceHeader 来描述：

SliceHeader {
    data *unsafe.Pointer
    cap  int
    len  int
}

make([]int, 10, 100) allocates an array of 100 ints and then creates a slice structure with length 10 and a capacity of 100 pointing at the first 10 elements of the array. (When making a slice, the capacity can be omitted; see the section on slices for more information.)

In contrast, new([]int) returns a pointer to a newly allocated, zeroed slice structure, that is, a pointer to a nil slice value.

Go 反射的限制 #

Go 反射能力尽管很强大，但还是有一些限制。比如出于安全性方面的考虑，不允许在已经定义的类型上添加方法。

总的来说，有两方面的原因：一是因为 Go 语言本身不允许在一个已经定义的类型上再添加方法，Go 反射应该与此保持一致；二是因为如果在运行时给某个类型动态添加方法，会导致接口断言、转换时出现不一致。综上，不允许在已定义的类型上通过反射来添加新方法。详见 Go issue：golang/go#20189

As a general rule, the reflect package should provide exactly the capabilities of the language. In the language you can not define a method on an existing struct, so reflect shouldn’t let you do it either.

Also, if you could define a struct in one package and add methods in another the general handling of interface conversion would be inconsistent, depending on whether the methods had been added or not.

为什么时间格式化是 “2006-01-02 15:04:05”？ #

使用参考时间（reference time）实际上比使用格式化代码更简单——一旦你理解了参考时间的含义。这些数字不是随机的：2006-01-02 15:04:05 是 time 包中所有格式化函数使用的唯一参考时间点。你只需要按照你希望的时间格式来格式化这个参考时间，time 包就会模仿该格式进行输出。

参考：golang-nuts 讨论

为什么没有构造函数和析构函数？ #

C++ 有多种构造函数，也有析构函数，这是为了兼顾内存占用和性能。但在可读性、可维护性与内存占用、性能优化之间的权衡中，开发者更关心可读性和可维护性。况且我们有更好的方式（GC）来优化内存分配和性能，所以 Go 不需要构造函数和析构函数。

为什么用 conversion 而非 casting？ #

“type is life”——类型就是生命。如果没有数据类型支撑，就无法解读内存中的数据。比如一个字节的数据，该解释成字符、整数还是浮点数？

术语 conversion 和 casting 字面上都有"类型转换"的意思，但有细微区别：

  • conversion：将 a 变量的内存数据按需拷贝到 b，对 a 无影响，b 是新变量；
  • casting：除了上述操作，还可能涉及将 a 的地址之后的连续内存区强制解释成 b 类型的数据。

Go 将这两种操作进行了更细致的划分：

  • conversion：a 转成 b，将 a 变量的内存数据按需拷贝到 b，对 a 无影响，b 是新变量；
  • casting：只包含将 a 的地址之后的连续内存区强制解释成 b 类型的数据这一种最底层的操作。

Go 中 conversion 相比 casting 会创建新变量，多消耗内存，但仍然建议用 conversion 而非 casting，因为 conversion 更能保证数据类型的完整性。类似 casting 的操作，Go 将其移到了 unsafe 包下面。

The idea of conversion over casting, there would be a cost of new memory, but we always be safe than sorry. Conversion over casting, it’s an integrity play to keep our software, our data and memory safe.

conversion 示例：

var b int32 = 6

a := int8(b)
c := int16(b)
d := int(b)

casting 示例：

var b int32 = 0
var a int16 = -1

p := unsafe.Pointer(&a)
b = *(*int32)(p)          // 注意这里可能导致内存访问段错误 (SEGMENTATION FAULT)
fmt.Println(`value:`, b)

命名类型转换 vs 字面量类型转换 #

  • named type：有名字的类型
  • literal type（unnamed type）：没有名字的类型

Go 中 named type 间的转换必须是显式的，而 literal type 的转换看起来像隐式转换。但实际上 literal type 的转换并非真正的隐式转换——完整性由源码保证。

type A struct {
    a int
}

type AA struct {
    a int
}

a := A{}
aa := AA{}
a = A(aa)            // named type AA 必须显式转换为 A

a = struct {         // literal type struct{int} 转换为 A，完整性由源码保证
        a int
    }{
        100,
    }

简单来说：Conversion is not the same as Casting. Conversion is not pretending. Casting is pretending.

为什么需要结构体字段对齐和填充？ #

内存对齐是为了让读写内存尽可能高效。要理解这一点，必须先理解硬件字边界（word boundary）。

|----------|----------|
0          8          16

假设硬件字长为 8 字节。如果有如下结构体：

struct V {
    f1 int8
    f2 int8
}

如果没有字段对齐，V.f1 和 V.f2 可能在内存中这样存储：

|------v.f1|v.f2-------|
0          8          16

由于硬件指令一次只能读 8 字节，读取 v.f1 和 v.f2 需要执行 2 条指令——效率低下！

而有对齐时，v.f1 和 v.f2 会这样存储：

|-v.f1-v.f2|----------|
0          8          16

只需一条读指令，效率大幅提升。

为了满足结构体字段对齐，某些字节会被填充（padding），例如：

struct V {
    b  bool
    n1 int8
    n2 int4
}

没有对齐和填充时，sizeof(struct V) 为 6 字节。有对齐时：

struct v {
    b  bool  // 放在 [0,1)
    n1 int8  // 放在 [1,2)
    ...      // 填充 2 字节
    n2 int4  // 放在 [4,8)
}

现在 sizeof(struct V) 为 8 字节。

为什么 goroutine 只能直接访问其栈帧内存？ #

我们写的代码不是在读内存就是在写内存。Goroutine 只能直接访问它当前操作的栈帧（frame）内的内存。

func main() {
    v := 10
}

在 main 这个 frame 中，main goroutine 要给变量 v 赋值 10，必须在 frame 内分配内存来存储这个值：

|---main-frame---|
|
|--data-section--|
|      10        |
|--text-section--|
|
|--    ...     --|
|                |

如果当前活跃 frame 是 main-frame，这个 goroutine 只能直接读写 main-frame 区域。这意味着什么？

意味着如果数据转换要由该 goroutine 完成，且它只能操作当前 frame 内的内存，那么 goroutine 所需的所有数据都必须在当前 frame 中。

理解 frame 的重要作用：它创建了一个沙箱（sandbox），一层隔离。它提供了不可变性（immutability）的保证——goroutine 只能在这个区域内产生变更（mutate），不会影响代码中的其他地方。这是非常强大的构造，也引出了值语义（value semantics）的概念。

为什么函数参数要传值？ #

“What we see is what we got!"——是的，但还有更深层的原因，其中之一就是值机制（value mechanics）。

函数参数传值实际上是对原始值的拷贝，在活跃 frame 层面提供了隔离——这非常重要！

区分两个术语：mechanics（机制）和 semantics（语义）：

  • mechanics：事物如何工作
  • semantics：事物如何表现

一切都有代价，没有什么是免费的。值语义的代价是什么？程序中存在数据的多个副本。值语义没有效率优势，有时要让一个被修改的数据在所有需要它的地方都得到更新会非常复杂。

但值语义也非常强大，因为它能减少副作用（side effects）——只能在活跃 frame 内产生变更。它提供了隔离，提供了对完整性至关重要的不可变性层级。

但有时，值语义的低效可能导致代码更复杂，甚至引发性能问题——而性能确实很重要。我们需要学会平衡值语义和指针语义：

func main() {
    v := 10
    increment(v)
}

// 值语义：只在 increment 的 frame 内变更，不会产生副作用
func increment(v int) {
    v++
}

// 指针语义：参数 v 是数据的拷贝（依然传值），
// 但通过指针语义，可以在当前 frame 之外产生副作用
func increment(v *int) {
    *v = (*v)++
}

为什么需要指针语义？ #

函数调用中传值始终成立，但值语义使得 goroutine 只能直接变更当前活跃 frame 内的内存。有时我们需要变更活跃 frame 之外的内存——这就是指针语义的用途。

指针语义使 goroutine 有能力间接变更其活跃 frame 之外的内存。

值语义应用于变量构造，指针语义应用于共享场景。只有在 return 语句中，才应在变量构造时使用指针语义。

为什么需要逃逸分析（Escape Analysis）？ #

如果函数返回一个指向其活跃 frame 的内存地址，该 frame 在函数返回后会被销毁。在 C++ 中，如果后续访问这个返回的地址，会触发段错误（SEGMENTATION FAULT）。C++ 开发者必须时刻注意这种情况——如果编译器能识别这种场景，为什么还要开发者一直操心呢？这是一个巨大的负担！

在 Go 中，编译器可以通过分析源码来判断一个变量是否会逃逸到当前活跃 frame 之外被访问。如果是，该变量会分配在堆内存中；否则分配在栈内存中。

编译器的这种分析就叫逃逸分析（Escape Analysis）。

可通过以下命令查看逃逸分析结果：

go build -gcflags="-m"
go build -gcflags="-m -m"

Go GC 的设计目标 #

  • 减少堆内存大小
  • 以良好的节奏执行 GC 任务
  • 将 StopTheWorld 时间控制在 100 微秒以内

Go GC 使用**三色并发标记-清除算法（Tri-color Concurrent Mark-Sweep Algorithm）来清理堆内存中的垃圾，并使用调步算法（Pacing Algorithm）**来决定何时执行 GC 任务。
