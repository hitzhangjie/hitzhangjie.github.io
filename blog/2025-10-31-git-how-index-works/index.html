Git 暂存区（.git/index）是如何工作的

Posted 2025-10-31 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

很多 Git 用户每天都用 git add、git commit，但对中间那个"暂存区"的理解往往停留在"一个中间状态"的程度。更具体的问题来了：如果我只改了一个文件的 10～20 行然后 git add，暂存区里存的是什么？是这几行的 diff，还是别的什么？

答案是：暂存区不存 diff，它存的是完整文件快照的指针。这个设计直接影响了 Git 的很多行为，理解它之后，git status 为什么快、合并冲突的 stage 是怎么回事，就都清楚了。

.git/index 是什么 ##

.git/index 文件，也叫 staging area 或者 cache，是工作区和版本库之间的一个中间层。它做的事情很简单：记录你"准备放进下一次 commit"的文件状态。

每次 git commit 的时候，Git 不会去看你的工作目录里有什么文件，而是直接根据 index 里的内容生成一个新的 tree 对象，然后创建 commit 指向它。

这个设计有几个关键行为：

初始状态：刚 clone 完仓库或者 git init 之后，index 里的内容和 HEAD commit 指向的 tree 完全一致。也就是说，刚开局的时候，暂存区反映的是最新一次 commit 的状态。

修改文件但不 add：改了工作区的文件之后，Git 能检测到文件变了，但 index 里还保留着上一次 add 或 commit 时的记录。此时 git status 会显示"modified but not staged"。

git add 之后：Git 把当前工作区中这个文件的完整内容压缩成一个 blob 对象存到 .git/objects/ 里，然后把 index 中该文件的条目更新为指向这个新 blob。同时更新的还有文件的元数据——路径、权限、时间戳等。

git commit 之后：Git 从 index 的内容生成一个新的 tree 对象，再创建一个 commit 指向这个 tree。commit 完成之后，index 的内容通常和新 commit 一致（除非你 commit 之前没有把所有改动都 add 进去，那就出现了"staged"和"committed"不一致的情况）。

为什么 git status 这么快 ##

git status 很快，不是因为 Git 会去对比文件内容，而是因为 index 里缓存了每个文件的文件系统元数据：mtime（最后修改时间）、ctime、文件大小、inode 等。

判断一个文件是否被修改的流程是：

  1. 拿工作区文件当前的 mtime、size 等元数据，和 index 里缓存的元数据对比
  2. 如果元数据一致，认为文件没变，跳过
  3. 如果元数据不一致，才真正读取文件内容，计算 hash，和 index 中记录的 blob hash 对比

也就是说，对于没改动的文件，Git 根本不用读文件内容，只做一次 stat 调用就够了。这也是为什么在大型仓库里 git status 依然能秒出的原因。

ps: 这也是为什么 git update-index --assume-unchanged 能加速 status —— 它直接告诉 Git “这个文件不用检查了”，连 stat 都省了。当然代价是你得自己记得什么时候又改了这个文件。

合并冲突时 index 的特殊用法 ##

正常情况下，index 中一个文件只有一个条目（stage 0）。但在合并发生冲突时，index 可以同时存储同一个文件的多个版本：

  • stage 1：共同祖先（base）版本
  • stage 2：当前分支（ours）版本
  • stage 3：合并来源分支（theirs）版本

这就是为什么解决冲突时可以用 git diff --ours、git diff --theirs 这样的命令 —— 它们的依据就是 index 中不同 stage 的 blob。

当你解决了冲突并 git add 之后，Git 会把 stage 0 更新为你解决后的版本，删除 stage 1/2/3 的条目。再 git commit 就完成了合并。

回到那个问题：改了几行，index 里存了什么？ ##

现在可以完整回答开头的问题了。

改文件但还没 git add 时：index 里没有任何变化。它还是指向旧 blob，元数据也是旧的。Git 通过对比工作区文件的 mtime 和 index 缓存的 mtime 发现不一致，于是知道这个文件"modified but not staged"。

执行 git add 之后，发生了两件事：

  1. 创建新 blob：Git 读取这个文件的全部内容（不是只读改动的几行），压缩后存入 .git/objects/，文件内容的 SHA-1 hash 就是这个 blob 的文件名。

  2. 更新 index 条目：index 中这个文件的记录被更新 —— blob 指针换成新的 SHA-1 hash，元数据（mtime、ctime、size 等）也更新为当前文件的最新值。

关键就在这里：index 里存的是整个文件快照的指针，而不是"10~20 行改了什么"的 diff。

你可能觉得这很浪费——就改几行而已，为什么要存整个文件的快照？这其实是 Git 的设计哲学：存快照而非增量。Git 在概念层面就是"一系列文件系统快照"，而不是"一系列文件变更集"。

当然，Git 并不是真的傻到每次存一份完整副本。在存储层面，它会定期运行 git gc 把多个松散对象打包成 packfile，packfile 内部会做增量压缩（delta compression），把相似的对象之间只存差异。但这是存储优化，不影响 index 和 Git 对象模型的语义——从概念上讲，每个 commit 就是一份完整的快照。

总结 ##

  • .git/index 是工作区和 commit 之间的中间层，存的是文件元数据和指向完整内容快照（blob）的指针，不存 diff。
  • git status 快，是因为它优先通过 mtime 等元数据判断文件是否变更，只有元数据不一致时才真正读内容并计算 hash。
  • 合并冲突时，index 用多个 stage 同时存储 base/ours/theirs 三个版本，解决冲突的本质是清掉多 stage 只留 stage 0。
  • Git 的概念模型是"快照流"，不是"变更集"。packfile 的 delta 压缩只是存储层优化，不影响这个语义。

理解了 index 的工作方式之后，再回头看 git add -p（部分暂存）就会好奇：不是说存完整快照吗，怎么又能只 add 一部分？这个留到下一篇再展开。
