Go 汇编语言详解

Posted 2026-01-23 00:00 +0800 by ZhangJie ‐ 9 min read

分享:  

引出问题 ##

写 Go 的人，迟早会碰到需要读汇编的场景。比如用 pprof 做 CPU 分析，火焰图上函数名后面的指令你看不懂；比如你想搞清楚一个 channel 操作到底编译成了什么；比如 Go 编译器的 nil 检查到底长什么样。

Go 的汇编不是 x86 AT&T，也不是 Intel 语法——它基于 Plan 9 汇编，Go 编译器在此基础上做了大量定制。这篇文章的目标是让你读完能看懂 go tool compile -S 的输出，也能自己写 .s 文件。

Go Assembly 是什么 ##

Go 编译器（gc，不是 gccgo）的汇编器源自 Plan 9 操作系统的汇编器。Plan 9 是 Bell Labs 在 Unix 之后开发的操作系统，Go 的三位创始人（Rob Pike、Ken Thompson、Robert Griesemer）都曾在 Bell Labs 工作，他们在设计 Go 时沿用了 Plan 9 汇编的风格。

为什么要自己搞一套而不是直接用 AT&T 或 Intel 语法？核心原因是 可移植性。Go 汇编本质上是一层"伪汇编"——它是架构无关的中间表示。同一段 Go 汇编代码，经过不同架构的汇编器处理后，会生成对应的机器码。这一点和 LLVM IR 的思路类似，只不过 Go 汇编更贴近机器，保留了寄存器和具体指令，但用统一的助记符和操作数顺序屏蔽了平台差异。

ps: Go 编译器实际上有两个后端。一个是直接生成机器码的"旧后端"（cmd/compile/internal/gc，输出的就是 Plan 9 风格汇编），另一个是基于 SSA 的"新后端"。从 Go 1.7 开始引入 SSA 后端，到 Go 1.8 成为默认。本文讨论的汇编是 SSA 后端生成之后、由汇编器处理的文本汇编，它是整个编译管线中最后一步。

第一个例子：从 Go 到汇编 ##

先看一个简单的 add 函数，然后逐行解释编译后的汇编。

package main

func add(a, b int) int {
    return a + b
}

用 go tool compile -S 查看汇编输出：

go tool compile -S main.go

add 函数对应的汇编大致如下（这里用的是基于栈传参的旧约定，方便理解基本概念）：

TEXT ·add(SB), $0-24
    MOVQ a+0(FP), AX      // 将第一个参数 a 放入 AX
    MOVQ b+8(FP), BX      // 将第二个参数 b 放入 BX
    ADDQ BX, AX           // AX = AX + BX
    MOVQ AX, ret+16(FP)   // 将结果写入返回值位置
    RET

关键点：操作数方向

Plan 9 汇编的数据流向是从左到右。MOVQ $1, AX 是把立即数 1 搬到 AX。这和 Intel 语法一致，和 AT&T 语法（movq $1, %rax）相反。

如果你习惯了 AT&T 的 mov src, dst，切到 Go 汇编时需要适应这一点。

TEXT 指令：汇编函数的身份证 ##

TEXT ·add(SB), $0-24

这一行包含了函数的全部元数据，拆开来看：

  • TEXT：指令助记符，表示这是一段代码段（Text Segment），不是数据。
  • ·add：函数名。中间的点 ·（U+00B7，middle dot）是 Plan 9 的惯例，用于分隔包名和函数名。如果函数属于当前包，包名可以省略，只写 ·add；如果是其他包的函数，写成 math·Add(SB)。
  • (SB)：Static Base 虚拟寄存器，表示这是一个全局符号，地址相对于静态基址计算。
  • $0-24：
    • $0：栈帧大小（Frame Size）。这个 add 函数没有本地变量，不需要额外分配栈空间，所以是 0。
    • 24：参数和返回值总大小（Args Size）。两个 int 参数（16 字节）+ 一个 int 返回值（8 字节）= 24 字节。

注意：$0-24 中的数字是编译时确定的。如果你改了参数类型，这个数字会跟着变。

TEXT 的完整格式：

TEXT symbol(SB), [flags,] $framesize-argsize

其中 flags 是可选的，后面会讲到 NOSPLIT、WRAPPER 等标记。

四大虚拟寄存器 ##

Go 汇编引入了 4 个虚拟寄存器（伪寄存器），它们不是真实的硬件寄存器，而是汇编器层面的抽象。理解它们是读懂 Go 汇编的关键。

寄存器全称作用用法示例
FPFrame Pointer指向调用者传入的参数和返回值区域a+0(FP)
PCProgram Counter程序计数器，用于跳转和分支JMP (PC)
SBStatic Base全局静态基址，所有全局符号都相对于 SB·add(SB), data+0(SB)
SPStack Pointer指向当前函数栈帧的局部变量区域x-8(SP)

FP：参数和返回值 ##

FP 伪寄存器指向调用者压入的参数区域的起始位置。访问参数时必须使用符号名 + 偏移的形式，如 a+0(FP)、b+8(FP)。

这里的偏移是相对于 FP 的字节偏移。第一个参数 a 在偏移 0 处（占 8 字节），第二个参数 b 在偏移 8 处，返回值在偏移 16 处。

MOVQ a+0(FP), AX     // 正确：符号名+偏移+FP
MOVQ 0(FP), AX       // 错误：不允许直接写偏移

这种强制命名的方式让代码更可读——你知道每个偏移对应的是哪个参数。

SP：局部变量和栈顶 ##

SP 是最容易搞混的。Go 汇编中 SP 有两种含义，取决于写法：

  • symbol+offset(SP)：伪寄存器，指向当前函数局部变量区域的起始。用法和 FP 类似，偏移是相对于栈帧基址。
  • offset(SP)：硬件 SP，指向物理栈顶。用于分配栈空间等操作。
SUBQ $16, SP          // 硬件 SP：栈顶下移 16 字节（分配空间）
MOVQ AX, x-8(SP)      // 伪 SP：将 AX 写入局部变量 x 的位置

这个区分很坑。一句话记：带符号名的 SP 是伪寄存器，给局部变量用；裸的 SP 是硬件寄存器，做栈操作。

SB：全局符号基址 ##

SB 是 Static Base，所有全局符号（函数、全局变量）都相对于它寻址。

TEXT ·add(SB), $0-24        // 函数 add 相对于 SB
DATA myvar+0(SB)/8, $42     // 全局变量 myvar 相对于 SB
GLOBL myvar(SB), $8         // 声明 myvar 为全局符号，大小 8 字节
CALL runtime·newobject(SB)  // 调用 runtime 包的函数

SB 的值对用户代码没有意义——它是链接器在最终定位时确定的。你只需要知道：全局的东西都挂到 SB 上。

PC：程序计数器 ##

PC 在 Go 汇编中主要用于跳转。不过大多数情况下你写 label 就够了：

    CMPQ AX, $0
    JEQ  done        // 如果 AX == 0，跳转到 done
    // ...
done:
    RET

直接操作 PC 的场景很少，除非在写很底层的 runtime 代码。

操作数与寻址模式 ##

Go 汇编支持多种操作数形式。以 x86-64 为例：

立即数 ##

MOVQ $42, AX     // $ 前缀表示立即数
MOVQ $0x10, BX   // 十六进制
MOVQ $1<<3, CX   // 表达式（汇编器支持常量运算）

寄存器 ##

MOVQ AX, BX      // 物理寄存器
MOVQ R8, R9      // x86-64 扩展寄存器

可用的物理寄存器（x86-64）：AX, BX, CX, DX, SI, DI, BP, SP, R8-R15, X0-X15（浮点）。

内存寻址 ##

Go 汇编的内存寻址和 x86 的寻址模式对应，用括号表示：

MOVQ (AX), BX            // BX = *AX（寄存器间接寻址）
MOVQ 8(AX), BX           // BX = *(AX + 8)（基址+偏移）
MOVQ (AX)(BX*2), CX      // CX = *(AX + BX*2)（基址+索引*倍数）
MOVQ 8(AX)(BX*4), CX     // CX = *(AX + 8 + BX*4)（基址+偏移+索引*倍数）

倍数（scale）只能是 1、2、4、8，对应于 1、2、4、8 字节宽的数据。

虚拟寄存器寻址 ##

前面已经提到过，FP 和 SP 伪寄存器的寻址有特殊要求：

// FP：必须带符号名
MOVQ a+0(FP), AX     // ✅
MOVQ 0(FP), AX       // ❌ 不允许

// SP（伪）：必须带符号名，用于局部变量
MOVQ AX, x-8(SP)     // ✅
// SP（硬件）：裸偏移，用于栈操作
SUBQ $32, SP         // ✅

ps: 为什么要区分伪 SP 和硬件 SP？因为 Go 的栈是动态增长的（分段栈 → 连续栈 → 栈复制），栈帧布局和传统 C 调用约定不同。伪 SP 提供的是一个稳定的、相对于栈帧起始点的视角，而硬件 SP 随时可能因为栈扩容而改变。

调用约定的演变 ##

这是 Go 汇编中变化最大的部分。

Go 1.16 及之前：全栈传参 ##

Go 最早的设计选择是全栈传参——所有参数和返回值都通过栈传递，寄存器只用于临时计算。这样做的好处是实现简单，可以精确支持栈复制（用于 goroutine 栈扩容）。

TEXT ·add(SB), $0-24        // 2 参数(16字节) + 1 返回值(8字节) = 24
    MOVQ a+0(FP), AX        // 从栈上取参数 a
    MOVQ b+8(FP), BX        // 从栈上取参数 b
    ADDQ BX, AX
    MOVQ AX, ret+16(FP)     // 返回值写到栈上
    RET

go tool compile -S 加上 -N -l（禁用优化和内联）就能看到这种栈传参的汇编：

go tool compile -N -l -S main.go

Go 1.17+：寄存器传参 ##

Go 1.17 引入了基于寄存器的调用约定（register-based calling convention），x86-64 上使用以下寄存器传参：

位置寄存器
第 1 个整数参数AX
第 2 个整数参数BX
第 3 个整数参数CX
第 4 个整数参数DI
第 5 个整数参数SI
第 6~9 个整数参数R8, R9, R10, R11
浮点参数X0-X14
整数返回值AX, BX, CX, DI, SI, R8, R9, R10, R11
浮点返回值X0-X14

寄存器传参后，同一个 add 函数的汇编变得非常简洁：

TEXT ·add(SB), $0-16
    ADDQ BX, AX     // AX 中已经是 a，BX 中已经是 b
    RET             // 返回值已经在 AX 中了

$0-16 也变了：帧大小还是 0，但参数+返回值只有 16 字节了（因为参数走了寄存器，这里统计的只是栈上的参数空间）。

注意：Go 1.17+ 的寄存器传参是 Go ABI 内部约定，不影响用户调用 Go 函数（编译器会自动处理）。只有当参数太多（超过 9 个整数或 15 个浮点数）时，超出部分会回退到栈传递。

如何判断当前用的是哪种约定 ##

看一眼 $framesize-argsize 就能猜个大概。如果 argsize 很大，大概率是栈传参；如果 argsize 很小甚至为 0，大概率是寄存器传参。

另外可以通过 go build -gcflags="-S" 的输出交叉验证：

# 看完整汇编
go build -gcflags="-S" main.go 2>&1 | less

数据传送 ##

指令含义示例解释
MOVQ传送 8 字节MOVQ $10, AXAX = 10
MOVL传送 4 字节MOVL $1, AXAX 低 32 位 = 1
MOVB传送 1 字节MOVB $0, (AX)*AX = 0
LEAQ加载有效地址LEAQ 8(AX)(BX*4), CXCX = AX + 8 + BX*4

LEAQ 和 MOVQ 的区别在于：MOVQ 8(AX), BX 是读取 *(AX+8) 内存处的值；LEAQ 8(AX), BX 只计算地址 AX+8 本身，不访问内存。前者访存但慢，后者只做计算但快。

条件跳转 ##

条件跳转通常跟在 CMPQ 或 TESTQ 之后：

指令含义条件
JEQ / JZ等于 / 零ZF = 1
JNE / JNZ不等于 / 非零ZF = 0
JLT / JL小于（有符号）SF != OF
JGT / JG大于（有符号）ZF = 0 且 SF = OF
JLE小于等于（有符号）ZF = 1 或 SF != OF
JGE大于等于（有符号）SF = OF
JLO / JB小于（无符号）CF = 1
JHI / JA大于（无符号）CF = 0 且 ZF = 0
JLS / JBE小于等于（无符号）CF = 1 或 ZF = 1
JHS / JAE大于等于（无符号）CF = 0
JCS / JC进位CF = 1

实际用法：

    CMPQ AX, $10     // 比较 AX 和 10
    JGT  bigger      // 如果 AX > 10，跳转到 bigger
    // AX <= 10 的路径
    RET
bigger:
    // AX > 10 的路径
    RET

    TESTQ AX, AX     // AX & AX，等价于检查 AX 是否为 0
    JEQ  isZero      // 如果 AX == 0

CMPQ vs TESTQ ##

  • CMPQ a, b 计算 b - a，只设置标志位不写回。用于比较两个值的大小。
  • TESTQ a, b 计算 a & b，只设置标志位不写回。常用于测试某个位是否为 1，或者判断寄存器是否为 0（TESTQ AX, AX，比 CMPQ AX, $0 短一字节）。

还记得开头 nil 检查文章里提到的 TESTB AL, (AX) 吗？那就是利用了 TEST 只设标志位不写回的语义——实际是在试探 (AX) 地址是否可访问。如果 AX 是 0，访问地址 0 触发 page fault，比后续代码访问同一地址崩溃要好，因为崩溃位置更精确。

DATA 与 GLOBL：定义全局变量 ##

除了函数，Go 汇编还可以定义数据。

DATA：初始化数据 ##

DATA symbol+offset(SB)/width, value
  • symbol+offset(SB)：符号相对于 SB 的偏移
  • /width：数据宽度（1, 2, 4, 8）
  • value：初始值（可以是立即数或地址）
DATA ·mydata+0(SB)/8, $42        // 在偏移 0 处存储 8 字节的 42
DATA ·mydata+8(SB)/8, $100       // 在偏移 8 处存储 8 字节的 100
DATA ·mystr+0(SB)/8, $"hello\n"  // 字符串常量

GLOBL：声明全局符号 ##

GLOBL symbol(SB), [flags,] size
  • RODATA：只读数据
  • NOPTR：不含指针（帮助 GC 跳过扫描）
  • DUPOK：允许重复定义（多个编译单元中可以出现同名符号）
  • TLSBSS：线程局部存储
GLOBL ·mydata(SB), $16           // 16 字节的可读写全局变量
GLOBL ·myrodata(SB), RODATA, $8  // 8 字节的只读数据
GLOBL ·nocan(SB), NOPTR, $8      // 不含指针的全局变量

一个完整的例子，定义一个全局计数器：

#include "textflag.h"

DATA ·counter+0(SB)/8, $0
GLOBL ·counter(SB), NOPTR, $8

TEXT ·Increment(SB), NOSPLIT, $0
    INCQ ·counter(SB)
    RET

特殊标记与元数据 ##

NOSPLIT ##

TEXT ·myfunc(SB), NOSPLIT, $0-0

NOSPLIT 告诉编译器：这个函数的栈帧不会超过 StackGuard（也就是不需要栈分裂检查）。通常用在：

  • 非常短小的函数（帧大小已知且很小）
  • 被 runtime 在特殊上下文中调用的函数（如信号处理、栈扫描）
  • 自己管理栈的函数

不加 NOSPLIT 时，编译器会在函数入口自动插入栈检查代码（比较 SP 和 g->stackguard0）。如果加了 NOSPLIT 但栈帧太大，编译会直接报错。

WRAPPER ##

TEXT ·mywrapper(SB), WRAPPER, $0-16

WRAPPER 标记表示这是一个包装函数（wrapper），通常是编译器为方法值（method value）或接口转换生成的。标记为 WRAPPER 的函数在栈回溯和 GC 扫描时有特殊处理。

NOFRAME ##

TEXT ·leaf(SB), NOFRAME|NOSPLIT, $0-0

NOFRAME 告诉汇编器不要生成栈帧（不保存 BP）。只适用于没有本地变量、不调用其他函数的叶子函数。省一条 PUSHQ BP 和 POPQ BP。

FUNCDATA 和 PCDATA ##

你在编译器输出中经常会看到这类指令：

FUNCDATA $0, gclocals·xxx(SB)
FUNCDATA $1, gclocals·yyy(SB)
PCDATA  $0, $0
PCDATA  $1, $0

这些是 Go 运行时需要的元数据：

  • FUNCDATA：函数级别的元数据。$0 是栈映射表（stack map），用于 GC 扫描栈上的指针；$1 是参数/返回值的指针信息；$2 是栈对象信息（Go 1.18+）。
  • PCDATA：程序计数器级别的元数据。$0 是栈映射索引（当前 PC 处哪些栈槽存了指针），$1 是 unsafe point 标记。

日常读汇编时一般不需要深入理解这些，知道它们是 GC 和栈展开用的就够了。

其他 TEXT flags ##

TEXT ·myfunc(SB), NOSPLIT|NOFRAME|WRAPPER|NEEDCTXT, $16-8
  • NEEDCTXT：函数需要一个闭包上下文指针（传在 DX 寄存器中）
  • TOPFRAME：标记为栈顶帧（用于栈回溯的终止条件）
  • ABIInternal：使用 Go 内部 ABI（寄存器传参）
  • ABI0：使用 Go 旧 ABI（栈传参）

textflag.h ##

写 .s 文件时，建议 include textflag.h：

#include "textflag.h"

这个头文件定义了上述 flag 常量（NOSPLIT、NOFRAME 等），直接写常量名比写数字可读性好得多。

栈帧布局详解 ##

把一个有局部变量和参数传递的函数的栈帧画出来：

调用者的栈帧
+-------------------+  ← 高地址
| 调用者的局部变量    |
+-------------------+
| 返回值（栈传参时）  |  ← FP + argsize - retsize
+-------------------+
| 参数 N             |
| ...               |
| 参数 1             |  ← FP（伪）指向这里
+-------------------+
| 返回地址            |  ← 硬件 SP 在 CALL 之后指向这里
+-------------------+
| 调用者的 BP         |  ← 保存的 BP（如有帧指针）
+-------------------+
| 局部变量 1          |  ← SP（伪）指向这里
| 局部变量 2          |
| ...               |
+-------------------+  ← 硬件 SP（分配完栈帧后）
| （未使用的栈空间）   |  ← 低地址
+-------------------+

几点注释：

  1. 参数和返回值在调用者的栈帧里。被调用者通过 FP 向上（高地址方向）访问它们。这就是为什么参数偏移是正的（a+0(FP)）。
  2. 局部变量在被调用者的栈帧里。通过 SP 伪寄存器向下（低地址方向）访问，偏移通常是负的（x-8(SP)）。
  3. 硬件 SP 指向物理栈顶的最低地址。分配栈空间是 SUBQ $N, SP（栈向下增长）。

ps: Go 的函数调用中，参数空间的分配和清理都由调用者负责。这和 C 的 __cdecl 风格类似，也是支持可变参数的自然选择。

实战：手写 Go 汇编文件 ##

下面通过一个完整例子展示如何写 .s 文件。

场景 ##

我们要在 amd64 上用汇编实现一个 Add 函数和一个 Swap 函数。同时写一个纯 Go 的 fallback 实现（在其他架构上使用）。

Go 文件（add.go） ##

package mymath

// Add returns a + b.
func Add(a, b int) int

// Swap returns b, a.
func Swap(a, b int) (int, int)

只声明函数签名，不写函数体——这样编译器就会去 .s 文件中找实现。

汇编文件（add_amd64.s） ##

#include "textflag.h"

// func Add(a, b int) int
TEXT ·Add(SB), NOSPLIT, $0-16
    ADDQ BX, AX    // Go 1.17+: a 在 AX，b 在 BX，结果在 AX
    RET

// func Swap(a, b int) (int, int)
TEXT ·Swap(SB), NOSPLIT, $0-32
    MOVQ AX, CX    // CX = a
    MOVQ BX, AX    // AX = b (第一个返回值)
    MOVQ CX, BX    // BX = a (第二个返回值)
    RET

Fallback（add_generic.go） ##

//go:build !amd64

package mymath

func Add(a, b int) int {
    return a + b
}

func Swap(a, b int) (int, int) {
    return b, a
}

编译时，add_amd64.s 文件名中的 _amd64 是 Go 的构建约束，只有目标平台是 amd64 时才会被编译。add_generic.go 里的 //go:build !amd64 确保它只在非 amd64 平台编译。

关于 NOSPLIT ##

上面用了 NOSPLIT 因为这两个函数栈帧为 0，不会触发栈分裂。对于有本地变量的汇编函数，栈空间 > 0 时，需要判断是否加 NOSPLIT：

  • 如果函数很小（几字节栈空间），加 NOSPLIT 没问题
  • 如果函数可能分配较大栈空间或有循环，不要加 NOSPLIT，让编译器自动插入栈检查

总结 ##

Go 汇编是基于 Plan 9 的体系结构无关伪汇编。本文覆盖了以下核心点：

  1. TEXT 指令：$framesize-argsize，分别表示栈帧大小和参数+返回值总大小。
  2. 四大虚拟寄存器：FP（参数/返回值）、SP（局部变量/栈顶）、SB（全局符号）、PC（程序计数器）。SP 伪寄存器和硬件 SP 的区分是最常见的陷阱。
  3. 调用约定：Go 1.17 引入了寄存器传参，绝大多数场景下参数通过 AX、BX、CX 等整数寄存器传递，提升了函数调用性能。
  4. 常用指令：Q/L/W/B 后缀表示操作宽度，指令集覆盖数据传送、算术、控制流等。
  5. DATA 和 GLOBL：定义全局变量。
  6. 特殊标记：NOSPLIT（禁止栈分裂）、NOFRAME（无栈帧）、WRAPPER（包装函数）、FUNCDATA 和 PCDATA（GC 元数据）。

实际工作中，绝大多数场景你不需要手写汇编。但能够阅读汇编，意味着你可以：

  • 理解性能敏感代码到底编译成了什么
  • 分析 pprof 火焰图中被内联、被优化的代码
  • 搞清楚运行时内部机制的细节（goroutine 切换、channel 操作、map 访问）

都是有用的时候才想起来的技能。
