AI Agent 技术全景：规划、记忆、工具、多 Agent 与人机协同

Posted 2026-01-06 00:00 +0800 by ZhangJie ‐ 8 min read

分享:  

背景 ##

AI Agent 是 2025-2026 年最热门的 AI 话题之一。它的核心公式很简单：

Agent = LLM + Planning + Memory + Tools

但这四个组件各自的内涵远比表面看起来复杂。Planning 不是让 LLM 多输出几行字；现在的模型「看起来会想」，很多人却说不清这能力是 prompt 调出来的，还是训练里刻意灌的。Memory 也不等于把聊天记录全塞进 context。Tool Calling 不是模型内置了某个工具。Web Search 有一整套 API 生态。Multi-agent 经常被塞进 planning 那一格——其实更像编排。Human-in-the-Loop 则是 2026 年 AI coding 从「追求全自动」往回拉的关键趋势。

这篇文章按这条线拆完：Planning → Memory → Tools（含联网搜索）→ Multi-agent 编排 → HITL。Planning 和 Memory 会写细一点——这也是写过 Agent 之后最容易跟别人对不齐口径的地方。

一、Planning：不只是 LLM 的"碎碎念" ##

在 Agent 循环里，Planning 的本质是将高阶目标分解为可执行的原子步骤，并在执行中根据观察结果动态调整。

核心就两件事：

  • 任务分解（Task Decomposition）：把复杂问题拆成 step-by-step 的子任务
  • 自我反思与修正（Self-Reflection）：观察中间结果，偏了就改计划

工程上常见两条路径：

  • 内在规划（模型驱动）：靠 prompt / 模型自身推理，输出计划、再执行。灵活，长链路容易幻觉和逻辑漂移。
  • 外部约束规划（架构驱动）：状态机、DAG、workflow 把跳转写死，LLM 只在节点里做「选择题」。稳，但表达力受图约束。

真实系统往往是二者混合：图管骨架，节点内仍可能跑 CoT / ReAct。

1.1 模型里的 CoT 是怎么来的？ ##

一个常见问题：现在模型好像「天生」会逐步推理，这是训练时故意喂了思维链数据吗？

不完全是「故意喂一套 CoT 数据集」这么简单，也不是纯涌现。 大致分了几代做法：

阶段做法代表
Prompt 时代不改权重，靠 few-shot 示例或 zero-shot 触发词引出中间步骤Wei et al. CoT (2022)、“Let’s think step by step”
SFT 时代用带中间步骤的解答数据做监督微调，教模型「先写过程再给答案」数学/代码题解、人工或合成的 long CoT 轨迹
过程监督不只看最终对错，对中间步骤打分/奖励Process Reward Model (PRM)
推理 RL 时代用可验证奖励做 RL，鼓励更长、可回溯的推理；不必每步都有人工标注o1 系、DeepSeek-R1 / R1-Zero

再拆开说：

1）先有 prompt，后有「内置」

2022 年的 Chain-of-Thought 论文核心是：同样的模型，你让它「写出中间步骤」再答，复杂推理准确率会明显上去。当时并不是换了一个「CoT 专用模型」，而是推理形态被 prompt 撬出来了。规模够大时，这种能力更容易被激活——所以早期会说「涌现」，但涌现不等于「训练里完全没见过分步解答」。

2）SFT：是的，会刻意用思维链数据

指令微调阶段，高质量样本里经常包含「分步推导 → 最终答案」。数学题、代码题、逻辑题尤其如此。模型学到的不只是答对，还有输出形态：先铺过程，再收敛到结论。所以你的直觉对了一半——训练里确实有人在喂带过程的轨迹，但那通常是 SFT/对齐数据的一部分，不是单独再开一门「CoT 课」。

3）从 outcome 到 process，再到 RL

  • Outcome supervision：只奖励最终答案对不对——中间瞎编只要蒙对也给分。
  • Process supervision：中间每一步也评——更贵，但对推理轨迹约束更紧。
  • RL with verifiable rewards（RLVR）：对数学、代码等「结果可自动校验」的任务，用对错信号做 RL。DeepSeek-R1-Zero 甚至展示过：纯 RL、几乎不做 reasoning SFT，也能长出长 CoT、自我纠正一类行为；R1 则是 cold-start SFT（少量优质 long CoT）再加 RL，更稳、更可读。

4）今天你看到的「内置 CoT」通常是混合产物

预训练（文本里本就有大量解题/推导）
    → SFT（指令数据里混入分步解答 / cold-start long CoT）
    → RL / 偏好优化（拉长、纠正、对齐可读性与格式）
    → 推理时再花 token「想一会儿」（inference-time compute）

所以：

  • 会逐步想：很大程度上是训练目标 + 数据形态塑造的，不是魔法；
  • prompt 仍然有用：没做强推理对齐的模型，仍常靠 “think step by step” 或 few-shot 激发；
  • o1 / R1 一类：把「想多久」变成一等公民，中间过程可能折叠成 reasoning tokens，对应用层是「黑盒想完再答」，和你在 Agent 外层写的 ReAct 循环不是同一层。

ps: 训练侧细节和 SFT/RLHF 管线，另一篇 LLM 训练全景 写得更完整。这里只卡在「Agent 规划依赖的模型能力从哪来」。

1.2 单 Agent 规划范式：从直线想到搜索树 ##

自己写 Agent 时，planning 模式往往不是「选一个永远用」，而是按任务难度和容错要求升级。下面按搜索空间形状排一版，基本覆盖工程里最常被点名的那些：

CoT（Chain of Thought）——直线推理 ##

问题 → 步骤1 → 步骤2 → … → 答案

单路径、低成本。适合确定性强、步骤不太长的任务。弱点很明确：一步错，后面全跟着错，没有天然回溯。

和「模型内置 CoT」的关系：模型可能已经会在一条 completion 里写中间步骤；你也可以在外层强制「先 plan 再 act」。Agent 语境里的 CoT，更多是把中间态显式化，方便日志、截断和调试。

Self-Consistency——同一条直线采多样 ##

仍是 CoT，但同一题采样多条推理链，投票或聚合最终答案。不引入工具，也不改搜索拓扑，只是用更多 inference compute 换稳一点。数学题上很常见。

ReAct（Reason + Act）——推理与行动交错 ##

Thought → Action(工具) → Observation → Thought → …

Yao et al. 2022。Planning 不再只在脑子里推，而是边想边调工具，用环境反馈修正下一步。绝大多数「能查文档 / 跑命令 / 调 API」的 Agent，骨架就是 ReAct 或其变体。

注意：ReAct 解决的是「想和做怎么接」，默认仍是单路径；某一步 Action 选错了，后面照样可能走歪，除非你再叠 Reflection 或搜索。

Plan-and-Execute / ReWOO——先计划后执行 ##

  • Plan-and-Execute：先产出完整或阶段性计划，再按步执行；执行器可换模型、可并行子任务。
  • ReWOO：把推理与工具调用的依赖显式拆开，减少「每一步都把完整 thought 塞回上下文」的 token 浪费。

适合步骤边界清楚、工具结果可缓存的任务。计划错了要有**重规划（replan）**出口，否则只是把失败推迟。

Reflection / Reflexion——做完再批自己 ##

尝试 → 结果/评分 → 反思（错在哪）→ 带着教训再试

Reflexion（Shinn et al.）把失败经验写进记忆，下一轮 prompt 带上。和 CoT 的差别：CoT 是「这一次顺着想」；Reflection 是跨尝试的纠错回路。工程上很便宜——多一轮 critic prompt 就能上——也因此容易变成「反思套话」，需要可检验的失败信号（单测挂了、工具报错、规则校验不过）。

ToT（Tree of Thoughts）——树搜索 ##

        根问题
       /   |   \
     方案A 方案B 方案C
     / \         \
   …   …         …

每步展开多个候选「想法」，用评价器打分，BFS/DFS/束搜索选路，可回溯。适合开放式决策、多方案对比，成本明显高于单链 CoT。

GoT（Graph of Thoughts）——图，而不只是树 ##

想法节点之间可以合并、聚合、回流，不限于严格树。表达力更强，工程复杂度也更高；多数业务 Agent 用不到这一层。

LATS（Language Agent Tree Search）——ToT × MCTS × ReAct ##

LATS 把几件事拧在一起：

  • 像 ToT 一样维护搜索树；
  • 用 MCTS（选择 → 扩展 → 模拟 → 回传）分配探索预算；
  • 节点上的动作可以是 ReAct 式工具调用；
  • 用语言模型当价值/启发式，用环境反馈当真实信号。

可以把它理解成：会用工具的、带反思的、在树里做蒙特卡洛搜索的 Agent。适合高价值、可模拟/可评估的任务；延迟和费用都不便宜，一般不是 CRUD 聊天的默认档。

一张对照表 ##

| 范式 | 形状 | 是否用工具 | 容错 | 成本 | 典型场景 | |:—|:—|:—|:—|:—| | CoT | 链 | 否 | 低 | 低 | 短推理、解释型问答 | | Self-Consistency | 多链投票 | 否 | 中 | 中 | 数学/逻辑求稳 | | ReAct | 链 + 环境环 | 是 | 中 | 中 | 检索、API、编码 Agent | | Plan-and-Execute | 计划链 + 执行 | 可选 | 中（依赖 replan） | 中 | 多步骤工作流 | | Reflection | 多轮链 | 可选 | 中高 | 中 | 可自动验错的任务 | | ToT | 树 | 通常否 | 高 | 高 | 多方案规划 | | LATS | 树 + MCTS + 工具 | 是 | 高 | 很高 | 难搜索、可评估环境 | | 状态机 / DAG | 固定图 | 节点内可选 | 高（结构上） | 可控 | 生产工作流 |

1.3 Multi-task 算 Planning 吗？（Multi-agent 见第五节） ##

概念回答的问题更贴近的层
Planning下一步想什么、做什么、是否回溯单（逻辑）Agent 内的决策策略
Multi-task多个子任务怎么拆、排序、并行多数仍是 planning / 调度
Multi-agent谁来做、怎么分工、怎么通信与仲裁编排（orchestration），第五节单独讲

CoT / ReAct / Reflection / ToT / LATS 都是单个决策主体内部怎么探索解空间——planning。 Multi-task 若仍是一个 Agent、一份记忆、一套工具，只是任务队列里有多件事——任务分解 + 调度，还在 planning 周边；Plan-and-Execute 天然干这个。 Multi-agent 是系统架构问题，不是又一种 CoT 变体。

1.4 Planning 怎么落到工程 ##

  1. 简单、短链路 → 模型内置 CoT 或轻量 ReAct 足够
  2. 要接工具、要可观测 → ReAct（Thought/Action/Observation 落日志）
  3. 结果可自动验 → 加 Reflection / 重试；有预算再上 Self-Consistency
  4. 解空间大、一步错代价高 → ToT / LATS，或直接 代码状态机/DAG 收束
  5. 角色必须隔离或并行专才 → multi-agent 编排（规划仍可嵌在某个角色里）

最稳健的生产默认项，往往不是「上最炫的搜索算法」，而是：

用状态机/图约束骨架，节点内用 ReAct，关键边用人审（HITL），只在少数高价值分支打开树搜索。

二、Memory：Agent 凭什么「记得」 ##

公式里有 Memory，很多人实现时却只写了 messages.append(...)。上下文窗口是工作记忆的上限，不是记忆系统本身。没有分层，长任务会一边烧 token 一边丢关键约束。

2.1 为什么必须单独做 Memory ##

LLM 调用是无状态的：每次请求只看见你塞进 prompt 的东西。Agent 要跨多步、多会话、甚至多用户保持一致，就得在应用层维护「什么该进上下文、什么该落库、什么该检索回来」。

三件事叠在一起，才构成可用的记忆：

  1. 写什么（write）：原始对话？摘要？结构化事实？工具结果？
  2. 存哪（store）：进程内 / Redis / 向量库 / 关系库 / 文件
  3. 何时读回（read）：每步全量？top-k 检索？按任务阶段注入？

2.2 常见分层 ##

层存什么生命周期工程形态
工作记忆（Working）当前 turn 的 messages、本步计划、最近工具结果一次任务 / 一次会话直接进 context
短期记忆（Short-term）本会话轨迹、scratchpad、未完成 TODO会话级滚动窗口 + 摘要
长期记忆（Long-term）用户偏好、项目约定、历史结论、可复用经验跨会话向量检索 / KV / 图谱
情景 / 情节记忆（Episodic）「某次任务怎么做成的」完整轨迹按需轨迹日志 + 检索
语义 / 知识记忆（Semantic）稳定事实、文档、领域知识慢变RAG 语料、知识库
程序 / 技能记忆（Procedural）可复用的工具组合、playbook、成功 prompt慢变模板库、skill 文件

不必六层全上。最小可用通常是：

Working（当前 messages）
  + Short-term（会话摘要 / 最近 N 轮）
  + Long-term 里的一块（用户偏好 or 项目文档 RAG）

编码 Agent 往往还要加：代码库检索（semantic） + 本次改动 diff / 测试结果（working/episodic）。

2.3 读写策略：比「存进去」更难 ##

写入

  • 全量追加：实现简单，context 很快爆。
  • 滑动窗口：只留最近 K 轮，丢早期约束（用户第一条里的硬性要求最容易没）。
  • 摘要压缩：周期性把旧对话收成 summary，再替换原文——信息有损，但可续跑长任务。
  • 结构化抽取：从对话里抽出 {user_name, repo, coding_style, decisions[]} 写入 KV，比纯向量更可控。
  • Reflection 写入：失败教训、成功 recipe 显式落库（Reflexion 那一路）。

读出

  • 总是注入：系统提示 + 会话摘要 + 当前任务状态。
  • 检索注入（RAG）：按当前问题 embedding 召回相关记忆/文档 chunk。
  • 按阶段注入：规划阶段给目标与约束；执行阶段给工具 schema 与局部上下文；收尾阶段给验收标准。

注意：向量检索召回的是「语义像」的片段，不是「逻辑上必须遵守」的约束。硬约束（权限、预算、禁止操作）应放在结构化存储 + 系统提示强制位，别全指望相似度。

2.4 Memory 和 RAG、和 Planning 的边界 ##

MemoryRAG（作为工具/子系统）
典型内容对话状态、用户/任务私有事实、轨迹相对静态的文档与知识
更新频率高低～中
谁消费几乎每一步需要外部知识时
  • RAG 可以看成「语义记忆的一种实现」或「一种 tool」——查文档、查代码、查工单。
  • Planning 产生的 plan / scratchpad 属于工作记忆；Reflection 的教训 适合进长期/情节记忆。
  • 多 Agent 时还要决定：共享记忆 vs 私有记忆——全共享易污染，全私有又协调不动，常见是共享任务状态 + 私有推理上下文。

2.5 工程上几条硬约束 ##

  • Token 预算要显式：系统提示、工具 schema、记忆、用户输入、模型输出各占多少，写进配置，不要靠感觉截断。
  • 可观测：每次注入了哪些记忆 ID / chunk，日志里要能回放——不然「它怎么又忘了」无法 debug。
  • 遗忘与失效：密码、临时 token、过期结论要能删、能 TTL；记忆不是越大越好。
  • 权限：长期记忆按用户/租户隔离，工具结果里的敏感字段写库前先脱敏。

一句话：Planning 决定「下一步干什么」，Memory 决定「下一步还能看见什么」。 看不见的约束，等于不存在。

三、Tool Calling：LLM 是"翻译官"，不是"执行者" ##

很多人对 Tool Calling 有一个根本性的误解：以为大模型本身就能调用工具。实际上：

LLM 只是一个翻译官——它告诉你「我需要查字典」，但真正去拿字典查的是你（Agent/应用）。

3.1 完整流程 ##

用户提问
    ↓
LLM 分析 → 判断需要调用工具
    ↓
LLM 输出结构化调用意图：
  {"name": "search_web", "arguments": {"query": "..."}}
    ↓
Agent/应用 解析
    ↓
Agent/应用 真正执行（HTTP / 本地函数 / shell / MCP server）
    ↓
工具返回结果（Observation）
    ↓
Agent 将结果作为消息传回 LLM
    ↓
LLM 继续想：再调工具 or 给出最终回答

这就是 ReAct 环在工程上的落点：模型出 Action，运行时出 Observation。

3.2 谁定义可用的工具 ##

不是模型，是应用。 调用 API 时通过 tools（或各家等价字段）把 JSON Schema 交给模型：

response = client.chat.completions.create(
    model="gpt-4.1",
    messages=[...],
    tools=[
        {
            "type": "function",
            "function": {
                "name": "get_weather",
                "description": "获取指定城市的实时天气",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "location": {
                            "type": "string",
                            "description": "城市名，如 Beijing"
                        }
                    },
                    "required": ["location"]
                }
            }
        }
    ]
)

模型侧通常返回 tool_calls；你的代码 dispatch[name](**arguments)，再把结果以 role: tool（或厂商约定角色）塞回 messages。

3.3 为什么需要模型「支持」工具调用 ##

需要在训练/对齐阶段学会：

  • 读懂工具定义的 JSON Schema
  • 在该调的时候输出 tool call，而不是瞎编天气
  • 参数类型与必填字段尽量合法
  • 读懂 tool 返回，再决定下一步

未对齐的模型常见翻车：该调不调、JSON 半截、参数幻觉、把 tool 结果当自己知道的事实继续编。

支持 vs 不支持：

  • 支持（GPT-4 系、Claude、Llama 3.1+ 等）：问「北京天气？」→ 结构化 get_weather
  • 不支持（早期 GPT-3 等）：「抱歉我无法查询实时天气…」或散文式伪调用

3.4 工程里还要补的几块 ##

光有一次 function call 还不能够做成可靠的 Agent：

点说明
并行 tool calls一次响应里多个调用；运行时要会并发执行、按 id 回填
多轮 tool loop上限（max steps）、超时、重复调用熔断，防止死循环烧钱
校验schema 校验参数；危险工具（删库、打钱、rm -rf）二次确认或 HITL
错误即 Observation工具失败不要吞掉——把 error 文本返回模型，让它改参数或换策略
描述即接口文档description 写清楚「何时用 / 何时不用 / 参数语义」；模型主要靠这个选工具
权限与沙箱工具以谁的身份跑？文件系统、网络、密钥作用域必须最小
可观测记录 name、args、latency、成功/失败；和 Planning 日志串在同一 trace

3.5 工具从哪来：本地函数 → OpenAPI → MCP ##

  • 进程内函数：最快，适合单一后端。
  • OpenAPI / HTTP：把已有微服务挂成 tools。
  • MCP（Model Context Protocol）等协议：把工具/资源标准化，IDE 与 Agent 运行时可以插拔同一套 server（读文件、查库、浏览器等）。

协议解决的是发现与接线；是否调用、何时调用仍是模型 + 你的循环策略。

3.6 工具设计的实用原则 ##

  • 少而精：工具越多，选错率越高；按任务阶段动态裁剪 tool 列表往往比一次塞 50 个强。
  • 副作用分级：只读默认开；写入/不可逆默认关或要审批。
  • 返回给模型的内容要「可推理」：巨型 HTML dump 不如提取后的 Markdown/JSON 要点；太长就截断并注明「已截断」。
  • 能确定性做的别交给模型：鉴权、配额、路径规范化用代码；模型只填业务参数。

四、Web Search：Agent 的「联网能力」 ##

Web Search 是 Agent 最常用的工具之一。模型权重是训练截止前的快照；要实时信息、长尾网页、竞品文档，就得外挂搜索 + 抓取。

4.1 AI-Native 搜索 API（专为 Agent 管线） ##

API特点
Firecrawl搜索 + 全页面抓取一体化，返回 Markdown/JSON，适合多步研究
Exa神经网络语义搜索，按含义匹配，偏学术/高质量网页
Tavily面向 Agent 优化，带引用，和 LangChain 等集成深
Perplexity Sonar对话式搜索 API，一次调用返回带引文的答案

4.2 传统 SERP API ##

API特点
SerpApiGoogle/Bing 等多引擎结构化结果，企业向
Serper高性价比 Google SERP JSON
Brave Search API自有索引，偏隐私
Google Custom Search API官方，已限制新客；存量需规划迁移

4.3 选型时真正要看的 ##

为 Agent 选搜索，优先：

  1. 内容完整度：只有 title+snippet 往往不够，模型会瞎补全文。
  2. 结构化与引用：方便二次引用、减少幻觉、做 HITL 展示。
  3. 延迟与价格：ReAct 可能连搜 5～10 次，单价 × 步数很伤。
  4. 可抓取性：搜到链接后能否稳定拿到正文（反爬、JS 渲染）。

当前常见组合：Tavily / Exa 做发现 + Firecrawl 类做深挖，或一个 AI-native 端到端搞定。

ps: 搜索只是 Tool 的一种；内部 Wiki、代码库、工单系统往往比公网搜索更重要——别把「联网」当成唯一知识入口。

五、Multi-agent：编排，不是又一种 Planning ##

第一节说过边界：Planning 回答「下一步怎么想/怎么做」；Multi-agent 回答「谁来做、怎么协作」。 这一节把编排侧补全。

5.1 什么时候真的需要多 Agent ##

先别上。下列信号出现再考虑：

  • 上下文互相污染：调研材料和最终写代码挤在同一窗口，模型开始「用错资料」
  • 职责必须隔离：评审者不能兼作者（否则自卖自夸）
  • 要并行专才：检索、编码、测试可同时跑，对任务耗时敏感
  • 工具权限不同：只读研究 Agent vs 可写仓库的实现 Agent

没有这些，一个 Agent + 多 tool + 清晰 system prompt 通常更简单、更好调。

5.2 常见拓扑 ##

1) Pipeline（流水线）
   A → B → C → 结果
   例：研究 → 起草 → 润色

2) Supervisor / Hierarchical（主管 + 工人）
        Supervisor
       /    |    \
   Worker  Worker  Worker
   例：Planner 拆任务，Coder / Tester 执行

3) Router（路由）
   请求 → 分类 → 某个专家 Agent
   例：客服里账单问题 vs 技术问题

4) Debate / Ensemble（辩论 / 投票）
   Agent1 ↘
   Agent2 → Judge → 最终答案
   Agent3 ↗

5) 黑板 / 共享状态（Blackboard）
   多角色读写同一 Task State / 文档，由调度器决定下一手
拓扑优点代价
Pipeline清晰、易测前级错后级放大；缺动态回退时僵
Supervisor贴近真实团队；易插 HITLSupervisor 成为单点；提示词与状态设计重
Router省 token、专家提示可特化路由错则全错
Debate提高难度题正确率延迟与费用线性涨；需好 Judge
Blackboard灵活、适合复杂任务状态模型难设计；竞态与一致性要自己管理好

5.3 通信与状态：多 Agent 的脏活 ##

  • 消息传递：A 的输出当 B 的输入——简单，但丢结构。
  • 共享 Task State：{goal, plan, artifacts, open_questions, decisions} 用 typed 对象，各 Agent 读写字段——生产里更稳。
  • 私有 scratchpad + 公开结论：推理过程私有，只提交「结论/补丁/引用」到共享层，减少污染。
  • 终止条件：最大轮数、达成定义的 done、连续无进展、人工接管——和单 Agent 的 max steps 一样，没有终止条件就会跟开会一样没有结束。

5.4 和 Planning 模式怎么叠 ##

  • 每个 Worker 内部仍可用 CoT / ReAct / Reflection。
  • Supervisor 的「派工」本身是一种 Plan-and-Execute。
  • 树搜索（ToT/LATS）一般放在单个角色内部；用五个 Agent 硬模拟 ToT 往往更慢更乱。
  • Multi-task：一个 Supervisor 队列里多个 subtask ≠ 多 Agent；一个 Agent 顺序执行也叫 multi-task。

5.5 框架只是脚手架 ##

LangGraph、AutoGen、CrewAI、Semantic Kernel、自研状态机……解决的是图、状态、人机断点、重试这些运行时问题。它们不自动让你的分工合理。 选框架前先画清：角色、输入输出契约、共享状态、失败与重试、哪里要人批。图画不清，换框架也救不了。

选型结论再强调一次：先把单 Agent 的 ReAct + Memory + Tools + HITL 做稳；多 Agent 解决的是组织问题，不自动提高单步推理质量。

六、Human-in-the-Loop：2026 年的趋势转变 ##

截至 2026 年，越来越多团队从「纯自动化」转向 Human-in-the-Loop（HITL）。原因很具体：完全自主的 Agent 会引入缺陷、安全漏洞、难维护的代码，以及「它自作主张改了不该改的东西」。

6.1 核心原则 ##

  • 主动参与：人是流程的一环，不是事后看日志的观众
  • 迭代交互：AI 产出 → 人审 → 反馈进入下一轮上下文/记忆
  • 验证检查点：关键边（合并代码、部署、打钱、删数据）暂停等批准
  • 补齐大局观：业务约束、政治与合规、模型看不到的隐性目标

6.2 常见挂载点 ##

挂载点行为例子
计划审批执行前展示 plan「将改这 5 个文件，是否继续」
工具门禁高危 tool 前 interruptgit push --force、生产 DB 写
结果抽检完成后 diff / 报告给人PR 描述 + 测试结果
不确定时提问模型/规则触发 clarify需求歧义、多方案互斥
异常升级连续失败转人工重试 N 次仍红

LangGraph 一类框架把 interrupt / resume 做成一等能力，本质就是：把 HITL 嵌进状态机边，而不是做完再人工救火。

6.3 产品形态（2026） ##

  • Cursor：IDE 内 diff 级接受/拒绝
  • Claude Code / 终端 Agent：对话式改代码，关键命令可确认
  • GitHub Copilot Workspace 等：规划与实现建议 + 人最终确认
  • 自建 Agent：审批队列、Slack/飞书按钮、工单状态回写

6.4 HITL 不是「加个人工就变慢」 ##

做得好的 HITL：

  • 只挡不可逆或高成本步骤，只读探索可以自动
  • 给人看的是可裁决的摘要（diff、影响面、引用），不是 20 页 chain-of-thought
  • 人的决定写回 Memory（偏好、规范），减少重复打断

核心价值仍然是：质量与安全、责任边界清晰、模糊需求可推进——信任和责任无法完全托付给模型。

七、串起来：一次 Agent 运行时长什么样 ##

把前面模块放进同一条时间线：

用户目标
    │
    ▼
┌─────────────┐
│  Memory 读入 │  摘要 / 偏好 / RAG 片段 / 任务状态
└──────┬──────┘
       ▼
┌─────────────┐
│  Planning   │  CoT 计划 或 图上的当前节点决策
└──────┬──────┘
       ▼
┌─────────────┐
│ Tool Calling│  模型提出调用 → 运行时执行 → Observation
└──────┬──────┘     （搜索、代码、DB、MCP…）
       │
       ├─ 未完成 → 写回 Memory / 更新状态 → 再 Planning
       │
       ├─ 触碰高危边 → HITL 审批 → 通过则继续 / 驳回则重规划
       │
       └─ 完成 → 最终回答 +（可选）长期记忆写入

单 Agent 时，一条进程跑完上面的环。 Multi-agent 时，环出现在每个角色内部，外层再套 Supervisor/Router/共享状态。

还值得单独建的工程能力（本文不展开成章） ##

能力干什么
Eval固定题集 + 轨迹打分；改 prompt/工具前后可回归
TracingOpenTelemetry 式 span：每步模型 I/O、工具、耗时、费用
Guardrails输入输出过滤、PII、越权工具拦截
成本与延迟预算max steps、max dollars；模型路由（常见：大模型规划 / 小模型执行；或小模型先试、卡住再升级大模型）

没有 eval 和 trace，Planning 范式换来换去只是体感。

总结 ##

AI Agent 不是某一个技术突破，而是多个子系统的工程整合：

  1. Planning：模型侧 CoT 来自 prompt → SFT 轨迹 → RL/过程监督；应用侧从 CoT / ReAct / Reflection 到 ToT / LATS，是在换搜索形状和容错预算。Multi-task 多半是规划与调度。生产默认：图约束骨架 + 节点内 ReAct + 关键点 HITL。
  2. Memory：工作记忆 ≠ 把历史全塞进 context。分层写/存/读；硬约束走结构化，别全靠向量相似。Planning 定「干什么」，Memory 定「还能看见什么」。
  3. Tool Calling：LLM 是翻译官，应用是执行者；循环、校验、沙箱、熔断和工具描述质量决定能不能上生产。
  4. Web Search：AI-Native 搜索/抓取优先于纯 SERP snippet；内部知识源往往更关键。
  5. Multi-agent：是编排层——拓扑、共享状态、终止条件；不替代单步推理，也不是默认必选项。
  6. HITL：把人嵌进关键边，而不是事后背锅；信任与责任托付不出去。

把这几块边界划清，写 Agent 时就不容易把「模型会不会想」「上下文里有什么」「工具谁执行」「多角色怎么编」「哪里必须人批」揉成一锅粥。
