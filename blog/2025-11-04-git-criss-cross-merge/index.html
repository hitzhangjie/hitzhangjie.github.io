Git 交叉合并：两个分支可以有多个共同祖先？

Posted 2025-11-04 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

Git 里两个分支的公共祖先（merge base）通常只有一个。但有一个问题：有没有可能不止一个？

答案是有。典型场景就是交叉合并（criss-cross merge）。

正常的合并基础 ##

先回顾一下正常情况。从提交 C1 拉出两个分支，各自往前走了若干提交：

     C2 (branch-A)
    /
  C1
    \
     C3 (branch-B)

这时候 branch-A 和 branch-B 的共同祖先是 C1，唯一。git merge-base branch-A branch-B 也只会返回 C1。

交叉合并怎么产生多个共同祖先 ##

问题出在下面这一步。假设两个分支分别被合入对方：

  • 有人把 branch-A 合到了 branch-B，产生合并提交 M1（父提交是 C3 和 C2）
  • 同时另一个人把 branch-B 合到了 branch-A，产生合并提交 M2（父提交是 C2 和 C3）
     C2---M2 (branch-A)
    / \ /
  C1   X
    \ / \
     C3---M1 (branch-B)

这个 X 形历史就是 criss-cross merge。现在来看 branch-A（在 M2）和 branch-B（在 M1）的共同祖先：

  • C1 是共同祖先
  • C2 也是共同祖先——它同时是 M1 和 M2 的祖先
  • C3 也是共同祖先——同样同时是两者的祖先

C2 和 C3 彼此之间没有父子关系，都是"最近的"共同祖先。这时候两个分支就有了两个 merge base。

ps: 准确地说，C2 和 C3 都是"最佳"共同祖先（best common ancestor），因为它们之间不存在祖先关系。C1 虽然也是共同祖先，但它同时是 C2 和 C3 的祖先，可以被约掉。

Git 怎么处理多 merge base ##

当你在这种历史上再次尝试合并两个分支时，Git 的默认合并策略（recursive）会检测到多个 merge base。处理方式是：

  1. 找出所有 merge base（C2 和 C3）
  2. 先对这两个 merge base 做一次虚拟合并，产生一个临时提交——“虚拟祖先”（virtual ancestor），比如图中的X。
  3. 以这个虚拟祖先为参考点，对两个分支的 HEAD（M1 和 M2）做三方合并

简单说：Git 先自己内部合并一次 merge base，用合并结果当新的基准点，再拿它来合并真正的两个分支。所以叫"递归"（recursive）——它在合并之前先递归地合并了祖先。

这个虚拟合并本身也可能产生冲突。如果 merge base 之间的虚拟合并出现冲突，Git 会直接在合并过程中报冲突，让你来解决。不过这种情况在实际中很少见。

验证方法 ##

用 git merge-base 就能看到：

# 默认只返回一个（best）merge base
$ git merge-base branch-A branch-B
<C2 或 C3 的 commit hash>

# --all 列出所有 merge base
$ git merge-base --all branch-A branch-B
<C2 的 commit hash>
<C3 的 commit hash>

第二条命令如果输出多个 commit ID，就说明这两个分支有多个共同祖先。

你也可以自己构造一个 criss-cross 场景来验证：

# 初始化仓库
git init criss-cross-demo && cd criss-cross-demo
echo "hello" > README.md && git add . && git commit -m "C1"

# 创建两个分支各自提交
git checkout -b branch-A
echo "from A" >> README.md && git commit -am "C2"

git checkout main
git checkout -b branch-B
echo "from B" >> README.md && git commit -am "C3"

# 交叉合并
git checkout branch-A
git merge branch-B -m "M2"   # branch-A 合并 branch-B

git checkout branch-B
git merge branch-A -m "M1"   # branch-B 合并 branch-A

# 查看 merge base
git merge-base --all branch-A branch-B
# 输出两个 hash → 确认多个共同祖先

什么时候会碰到 ##

Criss-cross merge 在实际项目中不常见，但也不是纯理论场景。几种可能出现的情况：

  • 多分支并行开发 + 双向合并：两个团队各自维护分支，互相合入对方的改动
  • 长期分支相互同步：比如 main 和 release 分支在发版周期内互相合并
  • Git 自动化脚本：如果你写脚本自动处理分支间合并，可能不经意间构造出这种历史

大多数时候你不会感知到这个问题，因为 Git 的 recursive 策略已经默默处理了。但知道这个机制，至少在看 git merge-base --all 输出不止一行时不会慌。

小结 ##

两个分支可以有多个共同祖先。它发生在交叉合并（两个分支互相合入对方）的历史中。Git 的 recursive 合并策略通过"先合并 merge base 得到虚拟祖先，再用虚拟祖先合并真正分支"的方式处理这种情况，对用户基本透明。

git merge-base --all 是最直接的检测方式——如果你看到多行输出，就碰到了 criss-cross。
