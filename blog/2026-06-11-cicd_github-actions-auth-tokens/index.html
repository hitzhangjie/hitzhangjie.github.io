GitHub Actions 中的认证方式：GITHUB_TOKEN、PAT 与 Deploy Key

Posted 2026-06-11 12:00 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 #

我的个人博客采用 Hugo 构建，源代码放在私有仓库 hitzhangjie/myspace，构建产物需要部署到公开仓库 hitzhangjie/hitzhangjie.github.io。原先的 workflow 部署在目标仓库上，每次 push 源仓库后还需要手动推一次目标仓库，很痛苦。

迁移 workflow 到源仓库（myspace）时，发现 GITHUB_TOKEN 无法完成跨仓库推送。这引出了一个问题：GitHub Actions 中到底有哪几种认证方式？它们分别怎么配置、作用域是什么？

三种认证方式速览 #

认证方式谁来生成作用域适用场景
GITHUB_TOKENGitHub 自动注入当前仓库同仓库内操作（checkout、发 release、写 PR 评论等）
Personal Access Token (PAT)用户手动创建用户有权访问的所有仓库跨仓库操作、需要比 GITHUB_TOKEN 更广的权限
SSH Deploy Key用户手动生成单个仓库（写权限可选）跨仓库推送（如部署静态站点到另一个仓库）

GITHUB_TOKEN #

它是什么 ##

GITHUB_TOKEN 是 GitHub Actions 在每次 workflow 运行时自动生成的一个临时令牌。你不需要、也没办法手动配置它——在任何仓库的 Settings → Secrets and variables 或 Settings → Deploy keys 里都找不到它。

它由 GitHub 在 workflow 启动时注入，workflow 结束后立即失效。

如何使用 ##

在 workflow 文件中通过 ${{ secrets.GITHUB_TOKEN }} 引用即可，无需任何额外配置：

steps:
  - uses: actions/checkout@v3
    with:
      token: ${{ secrets.GITHUB_TOKEN }}

权限范围 ##

严格限制在当前仓库内。 默认拥有读/写权限（可在仓库 Settings → Actions → General → Workflow permissions 中调整），但无法访问任何其他仓库。

典型用途 ##

  • actions/checkout 检出代码
  • 在同一仓库内创建 release、tag
  • 在同一仓库内对 PR 添加评论、label
  • peaceiris/actions-gh-pages 部署到同一仓库的 gh-pages 分支

什么情况下用不了 ##

当你的 workflow 需要对另一个仓库做任何操作时——比如我的场景，workflow 跑在 myspace，需要把构建产物推到 hitzhangjie.github.io——GITHUB_TOKEN 就无能为力了。

Personal Access Token (PAT) #

它是什么 ##

PAT 是用户手动在 GitHub 上生成的静态令牌。你需要去 Settings → Developer settings → Personal access tokens 创建。分为两种：

类型说明
Fine-grained tokens（细粒度）可以精确到单个仓库、特定权限，但只在 GitHub 组织/企业版中功能完整
Classic tokens（经典）作用域粗（repo、workflow 等大类），但简单够用，个人项目推荐

创建方法（Classic） ##

  1. 打开 https://github.com/settings/tokens
  2. 点击 Generate new token (classic)
  3. Note 填一个有意义的名称，如 myspace-deploy
  4. 勾选 repo（对私有仓库需要）或 public_repo（只推送公开仓库）
  5. 生成后立即复制保存，离开页面后无法再次查看

如何存入 Secrets ##

打开源仓库的 Settings → Secrets and variables → Actions → New repository secret：

  • Name: PERSONAL_TOKEN
  • Value: 粘贴上一步生成的 PAT

同一个 PAT 可以用于多个仓库的 workflow。

在 workflow 中使用 ##

对于 peaceiris/actions-gh-pages 跨仓库部署：

- name: Deploy to external repo
  uses: peaceiris/actions-gh-pages@v3
  with:
    personal_token: ${{ secrets.PERSONAL_TOKEN }}
    external_repository: hitzhangjie/hitzhangjie.github.io
    publish_branch: master
    publish_dir: ./public

权限范围 ##

以创建这个 Token 的用户的身份，访问该用户有权访问的所有仓库。注意：

  • 如果创建 PAT 的用户离开了组织，令牌立即失效
  • Classic token 的 repo 权限覆盖了用户所有私有+公开仓库，权限很大

注意 ##

GITHUB_TOKEN 和 PAT 虽然用法相似（都是 secrets.XXX），但本质不同：

GITHUB_TOKENPAT
生成方式自动手动
存储位置无（运行时注入）Secrets
作用域当前仓库用户所有仓库
生命周期单次 workflow 运行永久（可设过期）
跨仓库❌✅

SSH Deploy Key #

它是什么 ##

Deploy Key 是一对 SSH 密钥，公钥添加到目标仓库，私钥存入源仓库的 Secrets。与 PAT 不同，它是仓库对仓库的授权，不绑定具体用户。

创建和配置 ##

第一步：生成 SSH 密钥对 ##

ssh-keygen -t ed25519 -C "actions-deploy" -f ~/.ssh/myspace-deploy-key

生成两个文件：

  • myspace-deploy-key → 私钥（存到源仓库 Secrets）
  • myspace-deploy-key.pub → 公钥（加到目标仓库 Deploy keys）

第二步：添加公钥到目标仓库 ##

打开目标仓库（部署产物要推到的那一个）→ Settings → Deploy keys → Add deploy key：

  • Title: 随意命名，如 myspace-actions
  • Key: 粘贴 .pub 文件全部内容
  • 勾选 “Allow write access”（必须！workflow 需要推送代码）

第三步：添加私钥到源仓库 Secrets ##

打开源仓库（跑 workflow 的那一个）→ Settings → Secrets and variables → Actions → New repository secret：

  • Name: ACTIONS_DEPLOY_KEY
  • Value: 粘贴私钥文件的全部内容

关键：必须包括 -----BEGIN OPENSSH PRIVATE KEY----- 头尾行，不能只复制中间部分。

第四步：在 workflow 中引用 ##

- name: Deploy to external repo
  uses: peaceiris/actions-gh-pages@v3
  with:
    deploy_key: ${{ secrets.ACTIONS_DEPLOY_KEY }}
    external_repository: hitzhangjie/hitzhangjie.github.io
    publish_branch: master
    publish_dir: ./public

权限范围 ##

精确到单个仓库。公钥在哪個仓库，私钥就只能操作那个仓库。这是与 PAT 最大的区别——PAT 是"用户能碰的所有仓库"，deploy key 是"就是这一个仓库"。

三种方式对比 #

┌─────────────────┬──────────────┬───────────────┬──────────────────┐
│                 │ GITHUB_TOKEN │     PAT       │   Deploy Key     │
├─────────────────┼──────────────┼───────────────┼──────────────────┤
│ 谁来生成         │ GitHub 自动   │ 用户手动       │ 用户手动          │
│ 存在哪里         │ 运行时注入    │ Secrets       │ Secrets+Deploy k │
│ 作用域           │ 当前仓库      │ 用户所有仓库    │ 单个目标仓库      │
│ 跨仓库           │ ❌           │ ✅            │ ✅               │
│ 依赖用户身份      │ ❌ 服务账号   │ ✅ 绑定用户    │ ❌ 仅绑定仓库     │
│ 过期             │ 运行结束      │ 可设/永不      │ 永不             │
│ 配置复杂度        │ 零配置       │ 低             │ 中               │
└─────────────────┴──────────────┴───────────────┴──────────────────┘

回到我的场景：为什么 GITHUB_TOKEN 不行 #

我的博客部署链路：

hitzhangjie/myspace (私有，Hugo 源文件)
        │
        │  push 触发 workflow
        │  hugo build → public/
        │
        ▼
hitzhangjie/hitzhangjie.github.io (公开，静态 HTML)

workflow 在 myspace 仓库运行，需要把 public/ 目录推送到 hitzhangjie.github.io：

  • GITHUB_TOKEN 作用域只在 myspace → 不可行
  • PAT 或 Deploy Key 可以跨仓库 → 可行

最终我选择了 PAT，因为它只需生成一个 token，不需要 SSH 密钥管理。对于个人项目来说最省事。

安全建议 #

  1. PAT 设置合理的过期时间。Classic token 默认 7 天过期，建议设为 30~90 天，到期前续期或轮换。
  2. 不要用个人主账号的 PAT 跑 CI。如果有组织，用机器账号（service account）生成 PAT，避免离职导致所有 CI 挂掉。
  3. Deploy Key 的私钥不要共享。一个 deploy key 只用于一个源-目标仓库对，多个仓库用不同的 key。
  4. GITHUB_TOKEN 的权限调成最小化。在仓库 Settings → Actions → General 中把默认权限从"Read and write"改成"Read"（大部分场景不需要写权限）。

总结 #

  • GITHUB_TOKEN 是天上掉下来的——自动、临时、只认当前仓库
  • PAT 是你自己去申请的——手动、持久、能跨仓库，但绑定用户身份
  • Deploy Key 是仓库之间的 SSH 通道——手动、持久、精确到单个仓库，不绑定用户

搞清楚了这三种方式的生成机制和作用域，就不会在配置 CI/CD 时被权限问题困住了。
