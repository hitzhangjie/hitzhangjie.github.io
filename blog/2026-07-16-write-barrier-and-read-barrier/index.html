写屏障与读屏障：Store Buffer 与 Invalidate Queue 的双重挑战

Posted 2026-07-16 20:10 +0800 by ZhangJie ‐ 5 min read

分享:  

多核 CPU 中，一个核心的写操作什么时候能被另一个核心看到？上一篇文章 梳理了 Store Buffer 在这条链路中的作用。但光有 Store Buffer 还不够——另一个同样关键的硬件机制 Invalidate Queue 在文章中没有展开。本文把这两个缓冲放在一起，重点讲清楚写屏障和读屏障分别解决什么问题，以及为什么经典的"生产者-消费者"场景必须两者配合。

Store Buffer 回顾：为什么写操作"延迟可见"？ ##

先快速回顾 Store Buffer。当一个 CPU 核心执行 x = 1 时，如果 x 对应的缓存行还不在本核心的 Exclusive/Modified 状态，CPU 不会阻塞等待——它把写操作丢进 Store Buffer，然后继续执行后续指令。Store Buffer 在后台异步申请缓存行权限，拿到后再把数据刷入 Cache。

CPU Core 0                        CPU Core 1
┌─────────────┐                  ┌─────────────┐
│  store x=1  │                  │             │
│     ↓       │                  │             │
│ ┌─────────┐ │  ──invalidate──→ │ ┌─────────┐ │
│ │Store Buf│ │                  │ │  Cache  │ │
│ │ x = 1   │ │                  │ │  x = 0  │ │
│ └─────────┘ │                  │ └─────────┘ │
└─────────────┘                  └─────────────┘

核心问题：本核心能通过 store forwarding 看到自己 Store Buffer 中的新值，但其他核心看不到，直到数据被刷入 Cache 并进入缓存一致性协议的管辖范围。

Invalidate Queue：为什么"失效消息"也延迟生效？ ##

Store Buffer 解释不了所有问题。看另一个经典场景：

// 初始: x = 0, flag = 0

// Core 0                  // Core 1
x = 1;                     while (flag == 0) {}
flag = 1;                  r1 = x;   // 可能读到 0！为什么？

即使 Core 0 的 x = 1 和 flag = 1 都已经刷入 Cache 并发送了 Invalidate 消息给 Core 1，Core 1 拿到 flag == 1 之后再去读 x，仍然可能读到旧值 0。

这用 Store Buffer 解释不通——Core 0 的写入都已经离开 Store Buffer 了。问题的根源是 Core 1 这边的 Invalidate Queue。

Invalidate Queue 是什么？ ##

CPU 接收到其他核心发来的 Invalidate 消息后，需要一个完整的处理周期：查 cache tag、更新状态、可能还要写回脏数据。如果缓存系统正忙，这个处理就要排队等。为了不让发送方干等 ACK，CPU 会：

  1. 把 Invalidate 消息直接丢进 Invalidate Queue
  2. 立刻回复 ACK 给发送方
  3. 后台异步处理 Invalidate Queue 中的条目
CPU Core 0                       CPU Core 1
┌──────────────┐                ┌──────────────────┐
│   Cache      │                │ 收到 Invalidate   │
│   x = 1      │                │      ↓            │
│              │                │ ┌──────────────┐  │
│              │ ←Invalidate── │ │Invalidate Q  │  │
│              │   Ack ──────→ │ │  x: invalid  │  │
└──────────────┘                │ └──────────────┘  │
                                │   Cache: x = 0   │
                                └──────────────────┘

关键后果：Invalidate Queue 中的失效操作尚未真正生效。Core 1 的缓存里 x 仍然是旧值 0，所有后续的 load 指令都可能命中这个本应失效的缓存行。

这就解释了上面的例子：Core 1 的 Invalidate Queue 中可能有 flag 和 x 两条失效消息，处理顺序不确定。当 Core 1 读到 flag == 1 退出循环时，x 的 Invalidate 可能还没处理——于是 r1 = x 读到了旧值 0。

写屏障：强制清空 Store Buffer ##

定义 ##

写屏障（Write Barrier，在 Linux 内核中叫 smp_wmb()）做一件事：把 Store Buffer 中已有的条目全部强制刷入 Cache，后续的 store 指令不能越过它。

CPU Core 执行:

store x = 1       // 进入 Store Buffer
store y = 2       // 进入 Store Buffer
─── smp_wmb() ─── // 写屏障：等待上面所有 store 刷入 Cache
store flag = 1    // 只有上面都完成后才能执行

保证什么 ##

写屏障保证屏障前的所有写操作对系统中所有核心全局可见之后，屏障后的写操作才能被其他核心看到。换句话说，它阻止了 Store-Store 重排——多个写操作的全局可见顺序与程序顺序一致。

只靠写屏障够吗？ ##

回到生产者-消费者：

// Core 0
x = 1;
smp_wmb();   // 保证 x=1 先于 flag=1 对其他核可见
flag = 1;

// Core 1
while (flag == 0) {}
r1 = x;      // 仍然可能读到 0！

Core 0 的写屏障保证了 x = 1 在 flag = 1 之前全局可见。但 Core 1 这边没有做任何保证——它的 Invalidate Queue 中可能还有 x 的失效消息没处理，导致读到 x 的旧值。

写屏障只管"发出"，不管"接收"。

读屏障：强制清空 Invalidate Queue ##

定义 ##

读屏障（Read Barrier，在 Linux 内核中叫 smp_rmb()）做一件事：把 Invalidate Queue 中已有的条目全部处理完毕，后续的 load 指令不能越过它。

CPU Core 执行:

load flag         // 读 flag
─── smp_rmb() ─── // 读屏障：清空 Invalidate Queue
load x            // 现在读到的 x 是最新的

保证什么 ##

读屏障保证屏障之后的 load 指令，一定能看到屏障之前其他核心已经全局可见的写入。它阻止了 Load-Load 重排——如果程序先读 flag 再读 x，硬件层面 x 的 load 不会在 flag 的 load 之前完成（从可见性角度）。

两者协同：经典的生产者-消费者 ##

写屏障和读屏障各管一端，只有配合使用才能正确同步：

// 初始: x = 0, flag = 0

// Core 0 (生产者)                  // Core 1 (消费者)
x = 1;                              while (flag == 0) {}
smp_wmb();   // 刷 Store Buffer    smp_rmb();   // 刷 Invalidate Queue
flag = 1;                           r1 = x;      // 保证读到 1 ✅

逐步追踪：

步骤Core 0Core 1
1x = 1 进入 Store Buffer
2smp_wmb() 强制 drain Store Buffer → x = 1 进入 Cache → 发送 Invalidate 到 Core 1
3收到 x 的 Invalidate → 丢入 Invalidate Queue（立刻回复 ACK）
4flag = 1 进入 Store Buffer → drain → 发送 Invalidate 到 Core 1
5收到 flag 的 Invalidate → 处理 → 读到 flag == 1，退出循环
6smp_rmb() 强制处理 Invalidate Queue 中所有条目（包括 x 的失效）
7r1 = x 现在从 Cache 读到最新的 x = 1 ✅

缺一不可：

  • 少了写屏障：flag = 1 可能在 x = 1 之前被 Core 1 看到。
  • 少了读屏障：即使 Core 1 看到 flag == 1 后再读 x，Invalidate Queue 中还有 x 的旧值。

全屏障：一次解决所有问题 ##

当不确定该用哪种屏障时，可以用全屏障（Full Barrier，Linux 的 smp_mb()，x86 的 mfence）：

// Core 0
x = 1;
smp_mb();    // 刷 Store Buffer + 刷 Invalidate Queue
flag = 1;

// Core 1
while (flag == 0) {}
smp_mb();
r1 = x;

全屏障同时做了写屏障和读屏障的事情，还额外阻止了 Store-Load 重排（这是写屏障和读屏障各自无法单独阻止的）。代价是更重——x86 上 mfence 比单纯阻止编译器重排多消耗几十个 cycle。

架构差异：x86 vs ARM ##

不同 CPU 架构对屏障的需求不同，根源就在于它们的微架构如何处理 Store Buffer 和 Invalidate Queue：

x86 (TSO)ARM (弱一致性)
Store Buffer 对嗅探可见？是否
Invalidate Queue也存在，但 load 在特定条件下会触发 drain是，大量依赖 Invalidate Queue
写屏障 smp_wmb()仅编译器屏障（store 顺序硬件保证）dmb st
读屏障 smp_rmb()仅编译器屏障（load 顺序硬件保证）dmb ld
全屏障 smp_mb()mfence / LOCK 前缀dmb sy

x86 之所以"强"，是因为它的 Store Buffer 对嗅探可见（上一篇文章提到），并且硬件保证 Store-Store、Load-Load、Load-Store 不重排。所以 x86 上 smp_wmb() 和 smp_rmb() 实际上只需要编译器屏障——但代码里还是要写，因为在内核这样的跨平台代码中不能假设自己跑在 x86 上。

ARM 则不然。它的 Store Buffer 对嗅探不可见，Invalidate Queue 的异步程度也更高。所以 ARM 上 smp_wmb() / smp_rmb() 会编译成真正的硬件屏障指令（dmb st / dmb ld）。

编程语言层面的对应 ##

理解硬件屏障之后，再看编程语言的原语就很清晰了：

写屏障（Release 语义）读屏障（Acquire 语义）全屏障
Linux 内核smp_wmb()smp_rmb()smp_mb()
C/C++memory_order_releasememory_order_acquirememory_order_seq_cst
JavaVarHandle.storeRelease()VarHandle.loadAcquire()volatile 读写
Go无直接 API无直接 APIsync/atomic 全屏障
RustOrdering::ReleaseOrdering::AcquireOrdering::SeqCst

Release-Acquire 对（写屏障 + 读屏障）就是从硬件机制中抽象出来的：release store 刷 Store Buffer，acquire load 刷 Invalidate Queue，一个负责"发出"，一个负责"接收"，刚好覆盖了整个通信链路。

总结 ##

┌─────────────────────────────────────────────────────────┐
│                     CPU Core                            │
│  ┌──────────┐              ┌──────────┐                │
│  │  Store   │──── drain ──→│  Cache   │←── invalidate ─│
│  │  Buffer  │              │  子系统   │                │
│  └────┬─────┘              └────┬─────┘                │
│       │ 写屏障                  │ 读屏障                 │
│       │ smp_wmb()               │ smp_rmb()              │
│       │ 清空 Store Buffer       │ 清空 Invalidate Queue  │
│       │                         ↓                       │
│       │                  ┌──────────────┐               │
│       │                  │ Invalidate   │               │
│       └──────── 不影响 ─→│   Queue      │               │
│                          └──────────────┘               │
└─────────────────────────────────────────────────────────┘
  • Store Buffer 让写操作异步化——写屏障 (smp_wmb()) 强制 drain，保证"我写的别人能看到"。
  • Invalidate Queue 让失效处理异步化——读屏障 (smp_rmb()) 强制 drain，保证"我读到的是别人最新写的"。
  • 两个缓冲各管一端，少一个同步都不完整。
  • 全屏障 (smp_mb()) 两个都刷，还额外阻止 Store-Load 重排，代价最重。

简单记忆：Store Buffer 是"发了但对方没收到的消息"，Invalidate Queue 是"收到了但没来得及看"。写屏障解决前者，读屏障解决后者。
