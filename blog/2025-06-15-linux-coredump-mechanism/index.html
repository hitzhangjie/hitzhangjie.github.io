Linux 核心转储机制深度解析

Posted 2025-06-15 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

最近排查一个线上服务的崩溃问题，需要在测试环境开启 core dump 来捕获崩溃现场。本以为就是 ulimit -c unlimited 改个 core_pattern 的事情，结果踩了不少坑：目录权限明明设好了却写不进去、WSL2 里 systemd-coredump 罢工……这些问题追下去，发现核心转储的机制比想象中要复杂。

这篇文章把核心转储的几个关键问题系统梳理一下，希望能帮到遇到类似问题的同学。

Core dump 到底是谁写入的？ #

直觉上可能会认为：核心转储是内核在进程崩溃时写入的，内核拥有最高权限，所以应该能写到任何目录才对。

但实际情况是：

内核确实是写入动作的执行者，但它会以崩溃进程的 EUID/EGID 来检查文件系统权限。

具体来说：

  1. 当进程收到致命信号（如 SIGSEGV、SIGBUS）时，内核检测到这个信号，并决定生成核心转储。
  2. 内核在崩溃进程的上下文中执行写入操作——这意味着文件创建、写入时的权限检查，基于的是崩溃进程的有效用户 ID 和有效组 ID，而非 root。
  3. 因此，core_pattern 指定的目标目录（如 /var/cores/）必须对崩溃进程的用户可写——而不是对 root 可写。

打个比方：秘书（内核）拿着你的文件（core dump 数据），按你的要求放进某个抽屉（目标目录）。秘书虽然职务很高，但那个抽屉只允许 boss（root）打开的话，秘书没有钥匙也放不进去。

例外情况 ##

有几个情况会改变这个行为：

  • fs.suid_dumpable = 1：对于 SUID/SGID 程序，如果设置了此参数，内核会以 root 身份写入 core 文件，并限制文件权限为只有 root 可读写。
  • Pipe handler（| 模式）：当 core_pattern 以管道符开头时，内核将 core dump 数据通过管道传给处理程序，被调用的程序通常以 root 权限运行，由它来执行实际的写入。这也是管道模式更灵活的原因之一。

systemd-coredump：现代化的管道处理模式 #

现代 Linux 发行版（RHEL 8+、Ubuntu 16.04+ 等）通常不再直接写 core 文件，而是使用 systemd-coredump 通过管道模式来统一处理。

典型的配置如下：

kernel.core_pattern = |/usr/lib/systemd/systemd-coredump %P %u %g %s %t %c %h %e

管道模式的工作原理 ##

以 | 开头的 core_pattern 表示管道模式。内核不再直接将 core dump 写到磁盘，而是：

  1. 内核将 core dump 数据通过标准输入管道传输给指定的程序（此处为 systemd-coredump）。
  2. systemd-coredump 接收原始数据，进行压缩，并存储到 /var/lib/systemd/coredump/（或其他配置路径）。
  3. 同时将元数据（PID、信号、时间戳、可执行文件名等）记录到 systemd journal。

公式中各占位符的含义：

占位符含义
%P崩溃进程的 PID
%u崩溃进程的 UID
%g崩溃进程的 GID
%s导致崩溃的信号编号
%tcore dump 发生时的 Unix 时间戳
%c崩溃进程的 core 文件大小软限制
%h主机名
%e可执行文件名

管道模式的优势 ##

相比直接写 core 文件的方式，管道模式有几个明显的优点：

  • 统一管理：所有 core dump 由 systemd-coredump 集中处理，不会散落各处。
  • 自动压缩：节省磁盘空间，大内存进程的 core dump 可能非常大。
  • 丰富元数据：崩溃信息与 journald 集成，可以用 coredumpctl 方便地查询、过滤。
  • 安全性更好：避免了在任意目录创建 core 文件可能带来的安全隐患。

可以用以下命令查看捕获到的 core dump：

# 查看最近的 core dump 列表
coredumpctl list

# 查看特定 PID 的 core dump 信息
coredumpctl info <PID>

# 查看最近一次 core dump 的详细信息
coredumpctl info -1

WSL2 下的 systemd-coredump 问题与解决 #

在 WSL2 + RHEL 8.5 环境中，systemd-coredump 管道模式经常无法正常工作。

症状 ##

当向目标进程发送 SIGBUS 触发 core dump 时，dmesg 会报告：

systemd-coredump[31738]: Failed to connect to coredump service: No such file or directory

检查 socket 和 service 状态：

  • systemd-coredump.socket 状态正常（active listening），但 Accepted: 0; Connected: 0。
  • systemd-coredump.service 文件根本不存在（Unit systemd-coredump.service could not be found.）。

原因 ##

WSL2 的内核和 systemd 集成度有限，systemd-coredump 服务组件可能未完整安装，或者 socket 激活机制在 WSL2 环境下无法正常工作。

解决方案 ##

有两种 workaround：

方案一：回退到传统 core 文件模式 ##

直接将 core_pattern 设为传统文件路径：

sudo sysctl -w kernel.core_pattern=/var/cores/core.%e.%p.%t

注意：确保目标目录 /var/cores/ 对崩溃进程的 EUID/EGID 可写。

sudo mkdir -p /var/cores
sudo chmod 1777 /var/cores    # 或按需设置合适的权限

方案二：使用自定义 pipe handler ##

如果仍想用管道模式的灵活性，可以写一个简单的脚本来替代 systemd-coredump：

# /usr/local/bin/simple-coredump-handler
#!/bin/bash
DEST_DIR="/var/cores"
mkdir -p "$DEST_DIR"
cat > "$DEST_DIR/core.$1.$8.$(date +%s)"

然后设置：

sudo sysctl -w kernel.core_pattern='|/usr/local/bin/simple-coredump-handler %P %u %g %s %t %c %h %e'

总结 #

  1. Core dump 由内核写入，但权限检查基于崩溃进程的 EUID/EGID——这是最容易踩的坑。
  2. 现代 Linux 普遍使用 systemd-coredump 管道模式，core dump 被压缩后存入 /var/lib/systemd/coredump/，配合 coredumpctl 使用体验很好。
  3. WSL2 中 systemd-coredump 可能不可用，需要回退到传统文件模式或使用自定义 handler。
  4. 排查 core dump 问题时，dmesg 和 journalctl -b -g coredump 是最有用的两个信息来源。

了解这些机制后，在容器化环境、WSL 开发环境、线上排查崩溃问题时，就能快速定位 core dump 失败的原因，而不是在权限问题上反复试错了。
