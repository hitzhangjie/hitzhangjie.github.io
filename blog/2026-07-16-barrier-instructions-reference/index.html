内存屏障指令参考：从 x86 到 ARM 的完整对照

Posted 2026-07-16 22:00 +0800 by ZhangJie ‐ 7 min read

分享:  

前面的文章分别从 Store Buffer 与 Invalidate Queue、指令重排序、LOCK 前缀 等角度讨论了内存屏障"为什么存在"和"解决什么问题"。但当你打开 Linux 内核源码、Go runtime 汇编，或者用 objdump 反汇编一段并发代码时，真正映入眼帘的是一条条具体的汇编指令。

这些指令就是本文的主角。下面按架构逐一梳理——每条指令长什么样、做什么事、在什么场景下用。

x86 / x86-64 ##

x86 提供三条专用的屏障指令，外加一个"全能选手" LOCK 前缀。

LFENCE — Load Fence ##

指令：LFENCE
操作码：0F AE E8

语义：序列化所有 load 操作。LFENCE 之后的 load 指令不会在 LFENCE 之前执行（从执行完成的角度）。注意它不约束 store。

在 Intel SDM 中的定义：

LFENCE 之后的读指令不会在 LFENCE 完成前执行。LFENCE 不序列化写指令。

什么时候用：

  1. 配合 non-temporal load：MOVNTDQA（non-temporal load）配合 LFENCE 确保数据完全进入 cache 后再被使用。
  2. Spectre 缓解：LFENCE 可以阻止后续 load 的推测执行，被用作 Spectre v1 的软件缓解（__fence_reg / lfence 作为 speculation barrier）。
  3. 理论上充当读屏障，但实际上 x86 TSO 已经保证 Load→Load 和 Load→Store 不重排，所以普通场景下不需要显式 LFENCE 做读屏障。

SFENCE — Store Fence ##

指令：SFENCE
操作码：0F AE F8

语义：序列化所有 store 操作。SFENCE 之前的所有 store 必须在 SFENCE 之后的 store 之前全局可见。注意它不约束 load。

SFENCE 之后的写指令，在 SFENCE 之前的写指令全局可见之前，不会被提交。

什么时候用：

  1. 配合 non-temporal store：MOVNTQ / MOVNTDQ 等 write-combining（WC）写操作不经过 cache，而是直接写到 write-combining buffer，因此不受 MESI 协议的自然保序。用 SFENCE 强制 WC buffer 排空。
  2. 普通 WB（write-back）store 场景不需要：因为 x86 TSO 保证 Store→Store 不重排。
; 例子：WC 写之后用 SFENCE
MOVNTDQ [buf], xmm0    ; non-temporal store → WC buffer
SFENCE                  ; 强制排空 WC buffer
; 现在 [buf] 的数据对其他核可见

MFENCE — Memory Fence ##

指令：MFENCE
操作码：0F AE F0

语义：全屏障。MFENCE 之前的所有 load 和 store 必须在 MFENCE 之后的任何 load 和 store 之前完成。

MFENCE 之后的读写指令，在 MFENCE 之前的读写全部完成之前，不会执行。

实际上做的事：

  1. 排空 Store Buffer（之前的 store 全部全局可见）
  2. 阻止 Store→Load 重排（后续 load 不越过 MFENCE 去读旧值）
  3. 阻止 Load→Load、Load→Store、Store→Store 重排（虽然 x86 TSO 已隐含保证）

什么时候用：

  • 需要一个显式的全屏障但不想引入 LOCK 语义的额外副作用时。
  • 但实际中 LOCK 前缀指令常比 MFENCE 更快（在某些微架构上），所以 Linux 内核和 JVM 更倾向用 LOCK 前缀。

LOCK 前缀 — 事实上的全屏障 ##

指令：LOCK + {ADD, CMPXCHG, XADD, INC, DEC, ...}
前缀字节：F0

语义：对被修饰的指令提供：

  1. 原子性：读-改-写成为一个不可分割的事务
  2. 全屏障：等效于 MFENCE，排空 Store Buffer + 阻止所有重排

这是 x86 上最常用的全屏障实现方式。很多看似无意义的指令，唯一的使命就是带上 LOCK 前缀充当屏障：

; JVM 的 StoreLoad barrier - 往栈上加 0，只为了借 LOCK 的屏障语义
LOCK ADDL $0, 0(%rsp)

; Linux 内核的 smp_mb() 常见实现
LOCK; ADDL $0, -4(%rsp)

; Go atomic 在 amd64 上的实现
XCHGQ AX, (BX)        ; XCHG 自动隐含 LOCK 语义 = 全屏障

关于 LOCK 前缀的完整分析见前文。

x86 的特殊性：TSO 下的隐式保证 ##

回顾一下 x86 TSO 模型对四类重排的态度：

重排类型x86 是否发生需要的屏障
Store→Store不重排（硬件保证）无需 SFENCE（WC 写除外）
Load→Load不重排（硬件保证）无需 LFENCE（NT load 除外）
Load→Store不重排（硬件保证）无对应单屏障指令
Store→Load可重排MFENCE 或 LOCK 前缀

因此，x86 上大于 90% 的屏障需求只需要处理 Store→Load 重排。 其他三种重排硬件已经帮你拦住了。这就是为什么 x86 的 smp_wmb()（写屏障）和 smp_rmb()（读屏障）在 Linux 内核中本质上只是编译器屏障——不需要硬件指令。

ARM / AArch64 ##

ARM 是弱内存模型，四种重排都可能发生，所以屏障指令体系比 x86 丰富得多。

DMB — Data Memory Barrier ##

指令：DMB <option>
A64 编码：DMB {SY|ST|LD|ISH|ISHST|ISHLD|NSH|NSHST|NSHLD|OSH|OSHST|OSHLD}

语义：数据内存屏障。DMB 前的指定类型的内存访问，必须在 DMB 后的同类型访问之前被所有观察者看到。

option 字段（组合了屏障类型和作用域）：

Option屏障类型作用域含义
DMB SY全屏障全系统所有 load/store 保序
DMB ST写屏障全系统只保序 store
DMB LD读屏障全系统只保序 load
DMB ISH全屏障Inner Shareable同一个 Inner Shareable domain 内保序
DMB ISHST写屏障Inner Shareable同上，仅 store
DMB ISHLD读屏障Inner Shareable同上，仅 load
DMB OSH全屏障Outer Shareable更大共享域
DMB NSH全屏障Non-shareable仅本核

什么时候用：

  • DMB SY：全屏障，对应 x86 的 MFENCE 和 Linux 的 smp_mb()
  • DMB ST：写屏障，对应 Linux 的 smp_wmb()
  • DMB LD：读屏障，对应 Linux 的 smp_rmb()

与 x86 的核心区别：x86 上 smp_wmb() 和 smp_rmb() 只是编译器屏障；ARM 上它们会生成真正的 DMB ST 和 DMB LD 硬指令。

DSB — Data Synchronization Barrier ##

指令：DSB <option>
A64 编码：DSB {SY|ST|LD|ISH|...}（option 同 DMB）

语义：比 DMB 更强。DSB 不仅保序，还会阻塞流水线——DSB 之后的所有指令（不只是内存操作，是所有指令）必须等 DSB 之前的所有内存操作完成后才能开始执行。

DMB 只保序，不阻塞；DSB 既保序又阻塞。

什么时候用：

  1. 修改系统控制寄存器（如 MMU 配置、Cache 维护）后需要立即生效时
  2. WFI（Wait For Interrupt）之前，确保之前的 store 已完成
  3. 上下文切换、电源管理
; 修改 TTBR（页表基址寄存器）后
DSB ISH       ; 确保 TLB 刷新完成
ISB           ; 确保后续指令使用新页表

ISB — Instruction Synchronization Barrier ##

指令：ISB

语义：刷新 CPU 流水线。ISB 之后的所有指令从头重新取指。ISB 本身不保序数据访问（不约束 load/store），它只约束指令流。

什么时候用：

  • 修改了指令相关的系统寄存器（如 SCTLR_EL1）后，确保后续指令按新配置执行
  • 开启/关闭 MMU 后
  • 配合 DSB 使用：DSB 确保数据操作完成，ISB 确保流水线重来

LDAR / STLR — 单向屏障 ##

AArch64 特有的单向屏障机制，性能远优于全 DMB。

LDAR  <Rt>, [<Rn>]    ; Load-Acquire
STLR  <Rt>, [<Rn>]    ; Store-Release
指令语义对应高级语义
STLR这条 store 之前的所有 load/store 不会被重排到它之后memory_order_release
LDAR这条 load 之后的所有 load/store 不会被重排到它之前memory_order_acquire
; Release-Acquire 对 —— ARM 上最高效的同步模式

; 生产者
STR  X1, [data]       ; 普通 store
STLR X2, [flag]       ; release store —— 前序操作不得越过

; 消费者
LDAR X3, [flag]       ; acquire load —— 后序操作不得穿越
; 如果 X3 == 1，则
LDR  X4, [data]       ; 保证看到生产者写入的值

单向屏障的优势：STLR 和 LDAR 不像 DMB 那样双向阻断，它们只在一个方向上施加约束，硬件实现开销远低于全 DMB。现代 C++ memory_order_release / memory_order_acquire 在 ARMv8 上就编译为 STLR / LDAR。

ARM 屏障速查 ##

需要什么x86 等价ARM 指令说明

写屏障编译器屏障DMB ST保序 store

RISC-V ##

RISC-V 的屏障设计非常简洁，只有两条 FENCE 指令。

FENCE — 内存屏障 ##

指令：FENCE pred, succ
      FENCE  [iorw, iorw]

pred（predecessor）和 succ（successor）各是一个 4 位掩码，分别表示 FENCE 前后需要保序的内存操作类型：

位操作类型
iInput（load）
oOutput（store）
rRead（load，同 i）
wWrite（store，同 o）

常见组合：

FENCE rw, rw      ; 全屏障（等价于 x86 MFENCE / ARM DMB SY）
FENCE w, w        ; 写屏障（等价于 ARM DMB ST）
FENCE r, r        ; 读屏障（等价于 ARM DMB LD）
FENCE rw, w       ; release 屏障：前序读写不重排到后续写之后
FENCE r, rw       ; acquire 屏障：后序读写不重排到前序读之前
FENCE             ; 等价于 FENCE iorw, iorw（全屏障）

RISC-V 的独特之处：FENCE 的参数化设计允许更细粒度的组合——可以指定"前面的 load 不能重排到后面的 store 之后"（FENCE r, w）这种不对称屏障，这在 x86 和 ARM 中都没有直接对应。

FENCE.I — 指令同步 ##

指令：FENCE.I

类似 ARM 的 ISB——同步指令和数据流。当软件修改了指令内存后（如 JIT 编译器写完代码），用 FENCE.I 确保后续取指能拿到最新的指令。

PowerPC ##

PowerPC 也是弱内存模型，屏障指令体系介于 ARM 和 x86 之间。

SYNC — 全屏障 ##

指令：SYNC    （老写法）
      MSYNC   （Book E）

语义：全屏障。前面所有操作完成并对其他处理器可见后，后面的操作才能开始。等价于 x86 MFENCE、ARM DMB SY。

  • Linux 内核 smp_mb() 在 PowerPC 上就是 SYNC。

LWSYNC — 轻量同步 ##

指令：LWSYNC    （Lightweight SYNC）

语义：比 SYNC 更轻的全屏障。LWSYNC 前后的 load/load、load/store、store/store 保序，但不保序 store→load 的跨核顺序——这一点恰恰和 x86 TSO 的"洞"完全一致。

因此 LWSYNC 的语义非常接近 x86 TSO 的硬件保证，代价比 SYNC 低得多。Linux 内核中的 smp_wmb() 在 PowerPC 上用 LWSYNC 实现。

ISYNC — 指令同步 ##

指令：ISYNC

类似 ARM 的 ISB，刷新指令流水线。通常配合上下文切换或 SYNC 使用。

PowerPC 屏障速查 ##

Linux 内核PowerPC 实现说明
smp_mb()SYNC全屏障
smp_wmb()LWSYNC轻量，不保序 store→load
smp_rmb()LWSYNC读写屏障都用轻量同步
smp_rmb() 替代ISYNC + load某些实现用指令同步串行化 load

编译器屏障 ##

编译器屏障不生成硬件指令，只约束编译器。

GCC / Clang ##

asm volatile("" ::: "memory");

这告诉编译器三件事：

  1. 不要重排：这条 asm 语句前后的 load/store 不能跨越它
  2. “memory” 被破坏了：编译器必须假设所有内存都可能被修改，把寄存器中的变量值视为过期——强制从内存重新加载
  3. volatile：告诉编译器不要把这行 asm 优化掉

它不生成任何 CPU 指令。在 x86 上这往往就够了（因为 TSO 已经保证绝大部分序），但在 ARM 上远远不够。

Linux 内核中的编译器屏障 ##

#define barrier() __asm__ __volatile__("" ::: "memory")

Linux 把编译器屏障提取为 barrier() 宏。smp_wmb() 在 x86（CONFIG_X86_TFO）上的实现就是：

// x86: 硬件已保证 Store→Store 不重排，只需编译器屏障
#define smp_wmb()   barrier()

编译器屏障的局限性 ##

防止编译器优化重排     ✅
生成 CPU 屏障指令      ❌
防止 CPU 乱序执行重排   ❌
防止 Store Buffer 延迟  ❌

在 x86 TSO 上编译器屏障直接等于 smp_wmb() / smp_rmb() 是安全的，因为硬件已经补足了剩下的保证。但在 ARM 等弱内存模型上，编译器屏障必须和真正的硬件屏障配合使用。

全架构对照速查表 ##

操作x86-64ARM64RISC-VPowerPC编译器屏障

写屏障编译器屏障（TSO）DMB STFENCE w,wLWSYNC✅ TSO 下够

核心规律：

  • x86 最省事：只有 Store→Load 需要硬件屏障，其他三种依赖 TSO 的硬件保证 + 编译器屏障就够了
  • ARM 最费事：每种重排都需要显式硬件指令，所以 DMB SY/ST/LD + STLR/LDAR 得根据场景精细选择
  • RISC-V 最灵活：FENCE 的可编程掩码允许任意组合的不对称屏障
  • PowerPC 走中间路线：全屏障用重的 SYNC，其他用轻量的 LWSYNC

一道检验题 ##

下面这道题是理解屏障指令差异的好试金石——判断以下说法是否正确：

“x86 和 ARM 的写屏障（smp_wmb）都是硬件指令级别的内存屏障。”

答案：错误。 x86 的 smp_wmb() 在 TSO 下就是一个编译器屏障（barrier()），不生成任何 CPU 指令；ARM 的 smp_wmb() 会生成 DMB ST。二者在汇编层面完全不同。

查看实际生成的屏障指令 ##

写代码时想知道编译器到底生成了什么屏障指令，可以这样：

C/C++：

# x86
gcc -O2 -S atomic_example.c -o - | grep -E 'mfence|sfence|lfence|lock'

# ARM (交叉编译)
aarch64-linux-gnu-gcc -O2 -S atomic_example.c -o - | grep -E 'dmb|dsb|stlr|ldar'

Go：

# 编译为汇编输出
go build -gcflags="-S" atomic_example.go 2>&1 | grep -E 'XCHGQ|MFENCE|LOCK'

Linux 内核：

# 查看特定架构的 smp_mb 实现
grep -r 'smp_mb' arch/x86/include/asm/barrier.h
grep -r 'smp_mb' arch/arm64/include/asm/barrier.h

总结 ##

┌──────────────────────────────────────────────────────────────────┐
│ 屏障指令选择原则                                                 │
│                                                                  │
│ 1. 能用高级语言原语就用原语（atomic、volatile、Mutex）           │
│    → 编译器自动选择对应架构的最佳屏障指令                        │
│                                                                  │
│ 2. x86 写业务汇编时记住：                                        │
│    · Store→Store / Load→Load / Load→Store = 编译器屏障就够       │
│    · Store→Load = MFENCE 或 LOCK 前缀                            │
│    · 优先用 LOCK 前缀（常比 MFENCE 快）                          │
│                                                                  │
│ 3. ARM 写时记住：                                                │
│    · 能单向不双向 —— 优先 STLR/LDAR，而非 DMB SY                │
│    · DMB 保序不阻塞，DSB 保序且阻塞                              │
│    · 改系统寄存器后 DSB+ISB 一起上                               │
│                                                                  │
│ 4. 编译器屏障是"声明"，硬件屏障是"执行"                          │
│    · 前者告诉编译器：别动这里                                    │
│    · 后者告诉 CPU：别动这里                                      │
│    · x86 上前者常常等于后者，ARM 上二者必须搭配                  │
└──────────────────────────────────────────────────────────────────┘

从写代码的角度，记住两条原则就够了：

  • 遇到"其他线程什么时候能看到我写的数据"这个问题→你至少需要一个释放/写屏障。在 x86 上编译器会在适当位置插入 LOCK 前缀，ARM 上会生成 STLR 或 DMB。
  • 遇到"我怎么保证读到绝对是最新的数据"这个问题→你至少需要一个获取/读屏障。在 x86 上普通 MOV 就够（TSO 保证），ARM 上会生成 LDAR 或 DMB。

至于具体是 MFENCE 还是 LOCK ADD、是 DMB ST 还是 STLR，那是编译器的工作。你只需要知道每当写下 atomic_store(memory_order_release) 或 volatile write 时，对应的那些指令就已经在最终的二进制里等着被执行了。
