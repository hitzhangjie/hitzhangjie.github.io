版本控制系统选型：Git、Git-LFS 与 Perforce 的综合对比

Posted 2026-07-22 08:19 +0800 by ZhangJie ‐ 4 min read

分享:  

背景 #

团队项目最初用 SVN 管理游戏代码和资产，后来 SVN 频繁出问题，评估后切到了 Perforce。我没有直接参与这个过程，但对背后的选型逻辑很好奇：为什么不用 Git？Git-LFS 能不能解决二进制资产管理的问题？如果 Git 不适合，Perforce 又做对了什么？

这些问题其实指向一个更大的话题：不同版本控制系统各自的设计哲学和权衡。Git 诞生于 Linux 内核社区——海量贡献者、文本为主、分布式协作；Perforce 脱胎于游戏和大型企业——巨型仓库、二进制资产、集中式管控。它们在各自的土壤里长成了完全不同的东西。

本文尝试从多个维度梳理 Git、Git-LFS 和 Perforce 的设计差异，也顺带提一下 SVN 和一些大型公司的自研方案，希望能提供一个相对完整的视角。

架构：集中式 vs 分布式 #

集中式（SVN / Perforce） ##

SVN 和 Perforce 都是集中式架构：一台中心服务器持有完整的仓库历史和元数据，客户端只保留工作区的一份快照。

  • SVN：服务器存储每个文件的所有历史版本。客户端通过 checkout 拿某一版本的快照，commit 把本地修改推回服务器。没有本地仓库的概念。
  • Perforce：类似，但设计上更面向企业级场景。服务器（P4D）持有所有文件的完整历史、changelist、客户端工作区映射等元数据。客户端通过 p4 sync 同步文件到本地工作区，p4 edit/add/delete 向服务器"预定"修改意图，p4 submit 提交。

集中式的一个关键特征是：几乎所有操作都需要与服务器通信。p4 diff 要连服务器，p4 opened 要连服务器，甚至连查看文件历史都要连服务器。断开服务器后，Perforce 基本只剩下手动编辑文件的能力。

优势也很明显：

  • 权限控制精确：可以在文件/目录级别设置读写权限，适合需要细粒度访问控制的企业场景。
  • 强制锁机制：二进制文件（如 PSD、模型）可以要求先 p4 lock 再编辑，防止合并冲突——这对于无法合并的资产文件至关重要。
  • 单一事实来源：所有历史、元数据都在服务器上，不存在"谁有最新版本"的问题。

分布式（Git / Mercurial） ##

Git 把完整的仓库历史克隆到每个客户端本地。.git 目录包含了所有 commit、tree、blob，是一份完整的、自包含的数据库。

这带来几个质的改变：

  • 离线工作：commit、diff、log、branch、merge 全部本地完成，只在 push/pull/fetch 时才需要网络。
  • 轻量分支：Git 的分支本质上是一个指向 commit 的引用（41 字节的文件），创建和切换几乎零开销。这让 feature branch、hotfix branch 等分支工作流成为可能。
  • 冗余备份：每个 clone 都是一份完整备份。服务器挂了？任意开发者的本地仓库都能恢复整个历史。

代价也同样存在：

  • clone 成本高：仓库大的时候，全量 clone 非常慢。Git 后来引入了 --depth（shallow clone）、--filter=blob:none（partial clone）等机制来缓解。
  • 不适合巨型二进制仓库：Git 的 delta 压缩对文本文件很高效，但对经常变化的二进制文件基本无效。每次改动都存完整副本，.git 目录迅速膨胀。

大仓库：谁更擅长处理"胖"仓库？ #

这是游戏和大型企业项目最关心的问题。动辄几百 GB 甚至 TB 级别的仓库，各系统怎么应对？

Git 的应对 ##

Git 的设计前提是"仓库应该能完整 clone 到本地"。当这个前提不成立时，就需要各种补丁式方案：

Shallow Clone（浅克隆）

git clone --depth 1 <url>

只拉取最新一次 commit，历史全部省略。适合 CI 构建场景——我只需要最新代码编译，不需要历史。缺点是本地能做的事情受限，git log 看不到老的，git bisect 也没法用。

Partial Clone（部分克隆）

git clone --filter=blob:none <url>

这是更精细的控制：历史（commit、tree）全拉下来，但 blob（文件内容）按需下载。当你 git checkout 某个文件时，Git 才从服务器动态获取它的内容。结合 --filter=tree:0 甚至可以连 tree 对象也延迟获取。

本质上是一种"用网络延迟换本地存储"的权衡——仓库大到一定程度，这是唯一可行的办法。

Scalar / VFS for Git

微软针对 Windows 仓库（~300GB，300 万文件）自研的方案。核心思路：

  • 用虚拟文件系统（VFS）按需填充文件，而不是 checkout 时全部写入磁盘
  • 用后台进程预取可能需要的对象
  • 用 sparse-checkout 只 checkout 需要的目录

Git 后来内置了 git sparse-checkout 和 git maintenance 等命令，吸收了这些思路。Scalar 作为微软维护的 .NET 工具继续存在，核心优化也在逐步合入 Git 上游。

Perforce 的应对 ##

Perforce 不需要 clone。客户端只同步需要的文件：

p4 sync //depot/project/main/...#head
p4 sync //depot/project/main/src/...#head  # 只同步源码目录

仓库哪怕几 TB，客户端也可以只拉几百 MB。文件历史和版本信息都在服务器上，查历史（p4 filelog）直接查询服务器，不消耗本地存储。

这种架构天然适合大仓库——仓库多大是服务器的事，客户端只关心自己要用的文件。代价是离线查不了历史，也离不了线提交。

为什么没有完美的方案？ ##

这里有根本性的矛盾：你不可能同时拥有"完整的本地历史"“快速的首次同步"和"小的本地存储”。

本地有完整历史首次同步快本地存储小
Git 全量 clone✅❌（大仓库）❌
Git shallow clone❌✅✅
Git partial clone✅（commit/tree）✅✅（按需）
Perforce❌（在服务器）✅✅
SVN❌（在服务器）✅✅

Git 的 partial clone 在三个维度上都"还不错"，但这本质上是在用网络延迟换存储。如果你需要频繁查看历史、切换分支，网络抖动会明显影响体验。

Perforce 把这个三角直接拆开了：历史永远在服务器，客户端只关心当前。这在网络稳定的内网环境（大多数企业场景）下完全合理，但不适合经常离线的场景。

权限控制：路径级 vs 仓库级 #

这是企业选型中经常被忽视但其实非常重要的维度。

Git 的权限模型 ##

Git 的权限控制是仓库级别的：你有某个仓库的读权限，就能看到整个仓库的所有文件和所有历史。想做到路径级别的读权限控制（比如外包只能看 assets/textures/ 不能看 src/）？Git 本身做不到。

Git 托管平台（GitHub、GitLab、Gerrit）在此基础上做了增强：

  • 分支保护规则
  • CODEOWNERS 要求特定文件必须特定人员 review
  • Protected branch 禁止 force push

但这些仍然是"操作限制"而非"读权限控制"——一旦你 clone 了仓库，所有文件都在本地。

Perforce 的权限模型 ##

Perforce 的保护表（Protections Table）支持路径级别的细粒度权限：

write user alice * //depot/project/...
read  user bob   * //depot/project/assets/...
list  user bob   * //depot/project/...
  • read：可以同步文件到本地
  • write：可以修改和提交
  • list：只能看到文件存在，不能读取内容
  • 甚至可以对特定文件、特定分支、特定用户组设置不同权限

这种设计在游戏开发中非常实用：外包团队只能访问资产目录，引擎组能看到引擎代码，策划只能看配置表。所有权限都在服务器端控制，客户端无法越权。

实际影响 ##

  • Git 方案：大项目需要拆成多个仓库（如 Google 的 repo 工具、Android 的数百个 git 子仓库），然后通过 repo/gerrit 做统一管理和权限控制。拆分本身就是一个不小的工程决策。
  • Perforce 方案：一个 depot 搞定，通过保护表控制权限粒度。

二进制资产与锁机制 #

文本代码可以合并，但二进制文件（PSD、3D 模型、纹理贴图、视频）不能。两个人同时改了同一个 PSD，后提交的人一定覆盖前一个——因为没法 merge。

Perforce 的方案：文件锁 ##

Perforce 对二进制文件默认要求先 checkout（p4 edit → p4 lock）才能修改。锁是服务器端强制执行的，同一时刻只有一个人能拿到写锁。这从根本上避免了二进制文件的合并冲突。

p4 lock  //depot/project/assets/hero.fbx   # 锁定，别人只能读
p4 edit  //depot/project/assets/hero.fbx   # 打开编辑
# ... 修改 ...
p4 submit -d "更新角色模型"
p4 unlock //depot/project/assets/hero.fbx  # 释放锁

Git LFS 的方案：指针 + 锁 ##

Git LFS 用指针文件代替大文件存储在 Git 仓库中，实际二进制内容存在 LFS 服务器上。

version https://git-lfs.github.com/spec/v1
oid sha256:abc123...
size 52428800

Git LFS 2.0 之后也加入了文件锁功能：

git lfs lock hero.fbx      # 锁定文件
git lfs unlock hero.fbx    # 释放锁

锁信息存储在 LFS 服务器上，其他开发者可以看到谁锁了哪些文件。

但 LFS 的锁和 P4 的锁有一个关键区别：

  • P4：锁是工作流的内置部分，提交后自动释放；所有 p4 opened 都能看到当前锁状态
  • Git LFS：锁是一个"附加功能"，需要开发者主动 git lfs lock。如果有人没锁直接改了文件然后 push，Git 端不会阻止——除非在 LFS 服务端配置了强制锁策略

这意味着 Git LFS 的锁更多依赖团队规范，P4 的锁是技术强制。

分支模型 #

Git：分支就是一切 ##

Git 的分支极其轻量——一个 41 字节的引用文件。这催生了各种分支工作流：

  • Git Flow：master、develop、feature、release、hotfix，多分支严格管理
  • GitHub Flow：main + feature branch + PR
  • Trunk-Based Development：短生命周期分支，频繁合入主干

cherry-pick、rebase、squash merge 等操作让 Git 的历史编辑能力非常灵活。你可以把一个 feature 的 20 个 WIP commit squash 成 1 个有意义的 commit，也可以通过 rebase -i 重排、拆分、合并提交。

这也是 Git 在开源社区和互联网公司如此流行的核心原因——低成本的实验性分支让"试一下，不行就扔"变得没有心理负担。

Perforce：分支是重量级操作 ##

Perforce 也有分支，但它的分支本质上是在服务器上创建文件副本：

p4 integrate //depot/main/... //depot/dev/feature-x/...

这对小项目还好，但对大仓库来说，分支创建本身就是一个小型工程——几百 GB 的仓库，每个分支都要拷贝文件元数据。所以 Perforce 的分支通常比较少，而且生命周期较长。

Perforce 的 p4 integrate 会记录源和目标的对应关系，后续 p4 resolve 时可以追溯。但整体分支操作比 Git 重得多，不太可能"随手开个分支试试"。

流（Streams） ##

Perforce 后来引入了 Streams，把分支关系抽象成流图：

//depot/main
    ├── //depot/dev
    │   ├── //depot/dev-alice
    │   └── //depot/dev-bob
    └── //depot/release-1.0

类似 Git 的分支模型，但仍然是服务器端操作。Streams 让 Perforce 的分支管理更接近 Git 的体验，但底层实现决定了它不可能做到 Git 那种"瞬间创建、零成本丢弃"。

离线能力 #

这个话题在源聊天笔记中有详细讨论，这里简要总结：

  • Git：完整的本地仓库，99% 的操作不需要网络。飞机上写代码、commit、branch、rebase 都没问题，落地后 push。
  • Perforce：断开服务器等于"残废"。p4 diff 不行、p4 opened 不行、p4 submit 不行。只有在服务器可达时才能正常工作。
  • SVN：和 Perforce 类似，但 svn diff 不需要网络（本地保留了原始副本）。

如果你的团队永远在内网开发、VPN 稳定，这个差异可能不重要。但如果有远程办公、出差、不稳定网络的需求，分布式 VCS 的优势巨大。

实际场景与选择 #

没有银弹。选型取决于团队规模、项目类型和工作方式。

游戏开发 ##

典型的游戏项目：几百 GB 的仓库（大量二进制资产）、多部门协作（引擎、策划、美术、外包）、内网环境。

Perforce 通常是默认选择：

  • 路径级权限控制（外包管理）
  • 文件锁防止二进制资产冲突
  • 不需要 clone 整个仓库
  • 集中式管理与游戏 studio 的层级结构匹配

但 Git + Git-LFS 也在渗透：

  • 独立游戏或小团队
  • 资产规模可控（几十 GB 以内）
  • 团队偏好 Git 工作流
  • 有远程协作需求

实际案例：一些游戏项目使用 Git + Git-LFS，但配合了定制工具链。也有用被微软收购前的 GitHub 开发的自定义 Git 服务端来应对大仓库的。

互联网 / 软件公司 ##

典型的软件项目：代码为主、仓库规模适中（几十 MB 到几 GB）、大量开源依赖。

Git 是事实标准：

  • 文本代码的 diff/merge 能力
  • 轻量分支适合 code review 和 CI/CD
  • 开源生态完全围绕 Git
  • monorepo 可以用 partial clone 或 Scalar 缓解

大型企业的 monorepo ##

Google、Meta 这类公司的 monorepo 规模超出了任何现有 VCS 的能力范围：

  • Google 自研了 Piper，基于 CitC（客户端云端）架构，文件在云端存储，客户端按需访问
  • Meta 在 Mercurial 基础上深度定制
  • 微软开发了 VFS for Git（后演化为 Scalar）

这些方案的核心思路趋同：不要让开发者 clone 整个仓库。要么文件在云端（Piper），要么虚拟文件系统按需拉取（VFS for Git），要么深度魔改 Mercurial。

Linus Torvalds 对此的态度很明确：Git 不是为这种规模设计的。如果一个项目大到 Git 处理不了，项目团队应该自己想办法。这不是技术傲慢——这是对自己项目定位的诚实。

总结 #

维度GitGit-LFSPerforceSVN
架构分布式分布式 + 外部存储集中式集中式
大仓库shallow/partial clone大文件外存，仓库仍小原生支持，按需同步按需 checkout
二进制锁无可选，需服务端配置原生强制锁需 svn:needs-lock
权限控制仓库级仓库级路径级路径级
分支轻量，瞬间创建同 Git重量级，服务器操作重量级（拷贝）
离线工作完整支持完整支持（LFS 文件需缓存）基本不支持部分支持
学习曲线中等中等+平缓（GUI 成熟）低
生态系统极其丰富依赖 Git 生态商业支持逐渐萎缩

最后补充几点：

  1. Git partial clone 正在缩小差距。--filter=blob:none 配合 sparse-checkout 让 Git 也能处理较大的仓库，但体验仍不如 Perforce 原生支持那么流畅。

  2. Perforce 在它擅长的领域做到了极致。细粒度权限、文件锁、大仓库、GUI 工具链（P4V），这些是 20 多年在游戏和企业市场的积累，Git 短期内很难追上。

  3. 选型看核心矛盾。如果你的项目核心矛盾是"大量二进制资产 + 细粒度权限"，Perforce 可能更合适；如果核心矛盾是"频繁分支实验 + 分布式协作"，Git 更合适。

  4. 混合方案也是一种选择。有些团队代码用 Git，资产用 Perforce 或对象存储，通过脚本桥接。虽然运维成本高，但在特定场景下可能是最优解。
