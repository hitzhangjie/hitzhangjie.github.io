CPU软中断与硬中断：从INT指令到硬件中断的完整路径

Posted 2025-07-25 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

INT n 是一条指令，定时器中断是一个外部信号——来源完全不同，但 CPU 处理它们的流程为什么是一样的？

这个问题触及了 CPU 硬件设计里一个很精妙的地方：不同的前端接入方式，汇聚到同一个后端处理流程。

一个类比：CEO 和紧急电话 ##

把 CPU 想象成一个正在专注处理工作清单的 CEO：

  • 软件中断（INT n）：CEO 在清单上读到一行"立即给安全部门打电话报告内部问题"。这是计划内的、同步的，写在工作流里。
  • 硬件中断（网卡收到包）：桌上的红色紧急电话响了。这是计划外的、异步的，随时可能发生。

关键在于：不管是 CEO 主动打电话，还是接起紧急电话，后续动作完全一样——记下当前工作进度（保存现场），处理电话（执行 ISR），翻开笔记本继续之前的工作（恢复现场）。

CPU 硬件设计遵循的就是这个逻辑。

三个硬件模块，各司其职 ##

1. 外部信号接收：中断控制器（APIC） ##

CPU 不会给每个设备（键盘、网卡、硬盘、定时器）都留一个引脚——太多了。实际架构是：

  • 早期：独立芯片 8259A PIC，像电话总机一样汇集所有 IRQ，通过一根 INTR 引脚通知 CPU。
  • 现代：APIC（Advanced Programmable Interrupt Controller），集成在 CPU 内部（Local APIC），通过系统总线与 I/O APIC 通信，支持多核路由和 MSI（Message Signaled Interrupts）。

流程：

  1. 网卡收包 → 通过 I/O APIC 发送中断信号
  2. I/O APIC → Local APIC（优先级排序、决定是否打扰 CPU）
  3. Local APIC → 通过内部总线向 CPU 核心发送中断消息

2. 指令解码：控制单元（CU） ##

控制单元在同一时刻只能感知一种中断来源：

  • 软件中断：解码器识别到 0xCD nn（INT n）或 0xCC（INT 3）操作码，立即知道这是一个中断请求，不需要等外部信号。中断号就是指令操作数里的 n。
  • 硬件中断：每条指令执行完后，控制单元检查 APIC 是否有中断请求。有的话，向 APIC 发送确认信号，APIC 把配置好的中断向量号（网卡可能是 32，定时器是 33）发给 CPU。

ps: 为什么是"每条指令执行完后"检查？因为 CPU 不能在一条指令执行到一半的时候响应中断——那会破坏指令的原子性。所以硬件中断的响应延迟至少是当前指令的剩余执行时间。

3. 统一后端：IDT 驱动的中断处理流程 ##

到这里，软件中断和硬件中断的差异就消失了。CPU 硬件自动执行以下不可打断的序列：

  1. 完成当前指令：保证指令原子性。
  2. 获取中断向量号：软中断来自指令操作数，硬中断来自 APIC。
  3. 查 IDT：用向量号作为索引，查中断描述符表（地址在 IDTR 寄存器里）。
  4. 权限检查：IDT 条目里有 DPL，确保用户态代码不能伪装成内核级中断处理。
  5. 保存上下文：CPU 自动压栈关键寄存器。
    • 如果涉及特权级切换（用户态→内核态），先切到内核栈，再压入 SS/RSP/RFLAGS/CS/RIP。
    • 自动清除 TF（陷阱标志），防止 ISR 被单步执行。
  6. 跳转 ISR：从 IDT 条目加载新的 CS:RIP，控制权交给内核代码。
  7. 内核接管：接下来就是操作系统的中断处理逻辑了——保存剩余寄存器、执行处理函数、iret 返回。

整个步骤 1~6 全是 CPU 硬件自动完成的，操作系统只负责把 IDT 准备好、把 IDTR 寄存器设置好。

软中断 vs 硬中断对比 ##

软件中断 (INT n)硬件中断 (定时器、网卡)
触发源CPU 执行 INT 指令外部设备通过 APIC 发信号
同步性同步：发生在代码流的确定位置异步：随时可能发生
中断号来源指令操作数APIC 返回的配置值
后端流程完全相同——IDT 查询和上下文保存完全相同

为什么要统一 ##

如果软中断和硬中断走两套不同的硬件处理流程，CPU 设计会复杂得多——两套上下文保存逻辑、两套权限检查、两套恢复机制。统一到 IDT 的好处是：不管中断从哪来，硬件只负责"保存现场 + 查表跳转"，剩下的交给软件。

这也是为什么 Linux 内核可以复用同一套中断处理框架来处理所有类型的中断——不管是 NMI、硬件 IRQ、还是系统调用的 INT 0x80 / syscall，内核看到的世界是一样的。

ps: 系统调用在 x86-64 上已经用 syscall 指令替代了 INT 0x80，但原理类似——都是触发一个预定义的入口点，走内核的处理流程。
