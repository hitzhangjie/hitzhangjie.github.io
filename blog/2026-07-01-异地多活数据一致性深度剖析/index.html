异地多活的阿喀琉斯之踵：写一致性的技术真相与工业实践

Posted 2026-07-01 20:16 +0800 by ZhangJie ‐ 6 min read

分享:  

背景 #

在上一篇文章中，我们讨论了容错、高可用、容灾三者的本质区别。在容灾部分提到，异地多活是容灾体系中的"最高形态"——RTO ≈ 0，RPO ≈ 0，一个数据中心挂了，另一个无缝接管。

听起来很美好。但如果你深入问一句：“多活"到底是读多活，还是写多活？ 很多号称实现了异地多活的系统就要开始支支吾吾了。

本文试图诚实回答一个问题：异地多活中，数据的写一致性到底怎么解决？工业界真正落地的方式是什么？以及，我们到底需要"多活"到什么程度？

一、异地多活的挑战全景 #

在讨论具体方案之前，我们需要先把面临的挑战看清楚。异地多活的"异地"二字意味着机房之间隔着几十到上千公里——这带来了一系列环环相扣的难题。

1.1 根源：CAP 定理 ##

Eric Brewer 在 2000 年提出的 CAP 定理告诉我们：一个分布式系统最多只能同时满足以下三个性质中的两个：

C (Consistency)  — 所有节点在同一时刻看到相同的数据
A (Availability) — 每个请求都能收到（非错误的）响应
P (Partition Tolerance) — 系统在任意网络分区下仍能正常运转

异地多活场景下，机房之间的网络延迟不可消除（北京到上海光速极限约 12ms 往返），网络分区不可杜绝（光缆被挖断、运营商 BGP 抖动）。P 是必须接受的现实。

所以我们真正面对的取舍是：在 P（网络分区必然发生）的前提下，你选 C 还是选 A？ 这就是异地多活写一致性所有技术方案的分叉路口。

1.2 读很容易，写才是难题 ##

读多活的核心思路非常简单：数据复制 + 就近读取。无论用 MySQL 异步复制、Redis 主从同步，还是消息队列的广播模式，本质都是把一份数据拷贝到多个机房，然后让每个机房的用户读自己本地的那份。即使复制有延迟（通常秒级），绝大多数业务场景——看文章、刷 feed、查商品详情——对"读到 3 秒前的数据"完全没意见。

写多活则完全不同。它的核心矛盾可以抽象为一个简洁的问题：

当两个数据中心同时修改数据，甚至修改同一条数据 或者 事务性的对多条记录的修改，而且彼此不知道对方的修改——谁说了算？

举个例子：

时间线：
T1: 数据中心 A 收到请求，用户余额从 100 扣到 80（买了一张 20 块的电影票）
T2: 数据中心 B 收到请求，用户余额从 100 扣到 50（买了一张 50 块的电话卡）
T3: 两条写入各自同步到对方机房

问题来了：最终余额应该是 80？50？

1.3 写多活的挑战维度 ##

很多人讨论异地多活时，习惯把问题简单二分为"单条记录的一致性"和"多条记录的事务性”。但现实比这复杂得多。写多活面临的挑战实际上是多个维度交织在一起的：

维度一：副本一致性。 一条写操作在机房 A 执行后，机房 B、C 的副本应该看到什么值？这是对"值"的共识——Paxos/Raft 等共识算法要解决的核心问题。

维度二：事务排序。 当 T1（在北京扣 20）和 T2（在上海扣 50）同时发起，谁先谁后？这个顺序直接决定了最终余额是 80 还是 50。更致命的是：如果 T3 是一个基于余额条件的操作（“余额大于 60 时允许某操作”），顺序不同就可能导致 T3 在一个副本上成功、另一个副本上失败。

维度三：物理延迟。 这是所有讨论的底层约束：

机房对往返延迟 (RTT)一次 Paxos 提交
上海 ↔ 杭州~6ms~12ms
上海 ↔ 深圳~30ms~60ms
北京 ↔ 深圳~45ms~90ms
上海 ↔ 新加坡~180ms~360ms

电信号以光速传播，光速是不可协商的。任何跨地域的强一致性方案，延迟下限都由地理距离决定。

维度四：冲突检测与解决。 当两个写入确实冲突了——改的是同一条记录、且彼此不知道对方的存在——系统要么在冲突发生前阻止它（强一致性），要么在冲突发生后修复它（最终一致性）。前者代价高，后者语义复杂。

维度五：成本与复杂度。 不是所有公司都有 Google 的全球原子钟基础设施，也不是所有团队都能驾驭单元化改造。方案的选择从来不只是技术问题，而是技术、组织、成本的三角平衡。

这些维度不是非此即彼的，而是在不同场景下以不同方式组合。接下来我们看看工业界是如何在这个多维挑战空间中寻找出路的。

二、从写单活到写多活：工业实践的演进之路 #

既然挑战是多维的，工业界的应对也是逐步演进的。让我们沿着一条自然的思考路径——从最简单的方案开始，一步步追问"这够吗？不够的话下一步怎么走？"——来看看各种方案是如何在挑战维度之间做取舍的。

2.1 起点：读多活 + 写单活 ##

这是绝大多数公司的现实选择。架构简单得近乎朴素：

          ┌──────────────┐
          │   主库 (北京)  │  ← 唯一的写入者
          └──────┬───────┘
     ┌───────────┼───────────┐
     │ 异步复制   │           │ 异步复制
     ▼           │           ▼
┌─────────┐     │     ┌─────────┐
│ 只读副本 │     │     │ 只读副本 │
│ (上海)   │     │     │ (深圳)   │
└─────────┘     │     └─────────┘

所有写操作路由到主库，读操作可以在本地只读副本完成。当主库所在机房故障时，通过 VIP 漂移或 DNS 切换将写流量切到另一个机房的备库（提升为主库）。MySQL 主从复制、Redis 主从同步都是这个模式。

解决了什么：维度三（延迟）——读操作就近访问，写的延迟用户也能接受（互联网场景下几十ms的写延迟不是瓶颈）。

没解决什么：维度一（副本一致性）依赖异步复制，RPO 不是零；维度二（事务排序）不是问题——因为只有一个写入者。

代价：很低。加几台只读副本而已。这也是为什么云厂商的"多可用区部署"在读取侧几乎"开箱即用"。

很多团队在架构评审时执着于"做异地多活"，但当被追问到具体场景时，能想到的只是"机房断电了怎么办"。而"机房断电"实际上只需要热备 + 分钟级切流就能覆盖——这比做真正的多活便宜一个数量级。如果你的业务核心链路允许分钟级的写不可用，停在这里就很好。

2.2 单元化：换个维度实现"全局多写" ##

但有些业务停不下来。金融核心交易不能丢数据，IM 消息对延迟零容忍，电商下单链路停机直接等于营收损失。这些场景下，即使分钟级的写不可用也是不可接受的。

于是有了单元化（Unit Architecture）。它的核心哲学简洁而深刻：

既然两个人不能同时改同一条数据，那就设计系统，让每条数据在任何时刻只有一个写入者。

这不是"解决"冲突，而是从业务建模层面消除冲突的可能性。

蚂蚁 LDC 架构 ##

蚂蚁金服（支付宝）在 2013-2015 年间完成了可能是业界最大规模的异地多活改造，其核心架构称为 LDC（Logical Data Center，逻辑数据中心）。它把数据分为三种类型，分别处理：

┌─────────────────────────────────────────────────┐
│                   GZone                          │
│         全局数据（系统配置等）                       │
│         同一时刻只在一地提供写服务                     │
└─────────────────────────────────────────────────┘

┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│  RZone 0 │  │  RZone 1 │  │  RZone 2 │  │  RZone 3 │
│ 用户ID   │  │ 用户ID   │  │ 用户ID   │  │ 用户ID   │
│ [00-24]  │  │ [25-49]  │  │ [50-74]  │  │ [75-99]  │
│          │  │          │  │          │  │          │
│ 上海机房 │  │ 杭州机房 │  │ 上海机房 │  │ 杭州机房 │
└──────────┘  └──────────┘  └──────────┘  └──────────┘

┌─────────────────────────────────────────────────┐
│                   CZone                          │
│  有"写读时间差"的共享数据（账号、会员信息）          │
│  异步同步到各城市，本地读                          │
└─────────────────────────────────────────────────┘
单元数据类型写策略一致性策略
RZone（Region Zone）用户流水型数据（订单、交易、行为记录）只在本单元内写天然隔离，无冲突
GZone（Global Zone）不可拆分的全局数据（系统配置等）同一时刻只有一个地方写写单活
CZone（City Zone）写后有"读延迟容忍"的共享数据（账号信息）在原始单元写，异步同步到各城市最终一致

流量路由与切流 ##

单元化的前提是每条数据都有一个"归属单元"。以支付宝为例，按用户 ID Hash 分配：

user_id % 100 → 路由到对应 RZone

[00-24] → RZone 0（上海机房 A）
[25-49] → RZone 1（杭州机房 A）
[50-74] → RZone 2（上海机房 B）
[75-99] → RZone 3（杭州机房 B）

在网关层，每个请求根据用户 ID 被路由到正确的单元。关键的设计是可以做到单元内闭环——一个用户的所有操作，从接入层到计算层到数据层，全部在自己的单元内完成，不跨单元调用。

当一个单元挂了，切流遵循先切数据库映射，再切流量路由的原则。字节跳动的"两阶段切换"方案值得借鉴：

阶段 1: 禁写 — 先对切流用户在源单元禁写
阶段 2: 等待同步 — 等待增量数据同步追平
阶段 3: 开通写入 — 在目标单元开放写入，完成切换

库存问题：单元化的"终极 BOSS" ##

单元化最棘手的场景是库存扣减。库存天然是一个"全局量"——一件商品的总库存是所有用户共享的，不能简单地按用户 ID 拆分。

蚂蚁的库存单元化方案的思路是：

          ┌──────────────┐
          │  中心库存单元  │  ← 持有"全局缓冲"库存
          │  (逻辑集中)  │
          └──────┬───────┘
      ┌──────────┼──────────┐
      │          │          │
  ┌───▼───┐  ┌───▼───┐  ┌───▼───┐
  │单元 A  │  │单元 B  │  │单元 C  │  ← 各自预分配一部分库存
  │库存:10 │  │库存:10 │  │库存:10 │
  └───────┘  └───────┘  └───────┘
  1. 将商品总库存按单元拆分为多个"单元库存"
  2. 每个单元在本地扣减自己分配的库存
  3. 某单元库存不足时，从中心库存单元申请调拨
  4. 网络断开时，各单元仍可独立扣减已分配库存，网络恢复后重新汇总

这本质上是一个库存分段锁的方案，用一个"可能有少量偏差"的代价换来了"单元可以独立运行"的能力。

单元化的"真相" ##

如果我们诚实地审视单元化架构：

单元化 = 按用户维度把数据分区，每个分区只有一个活跃的写入者。

对于任意一条具体的用户数据，在任何时刻，只有一个单元在写它。你可以说这是"多活"——因为不同单元同时为不同用户提供读写服务。你也可以说这不是真正的"多活"——因为对于任一条记录，它永远只有一个写入者。

这取决于你的定义。但无论如何，单元化是目前唯一能在超大规模（数亿用户、数十万 TPS）下稳定运行、成本可控的"广义多活"方案。

一个思想的转变：很多架构师想做"异地多活"时，想的是"任何人在任何地方都可以写，系统自动解决冲突"——这是一个美好的愿景，但物理定律不支持。单元化的智慧在于，它把"分布式一致性"这个极难的技术问题，转化成了"用户分区 + 路由规则 + 切换流程"这个相对可控的工程问题。

2.3 数据同步：同城强一致，跨城异步 ##

单元化解决了"谁写"的问题，但没有解决"写完之后怎么同步"的问题。当用户在广州 RZone 写入了一条交易记录，这条记录怎么同步到其他机房的副本？

答案取决于距离。

同城多机房（延迟 <2ms）：可以用 Paxos/Raft 做同步复制，RPO=0。OceanBase（蚂蚁）、TiDB（PingCAP）都将 Raft/Paxos 实现在数据库层。同城延迟低，多数派确认的开销在可接受范围内——每次写入增加几毫秒的延迟，换来的是零数据丢失的保证。

跨城多机房（延迟 >30ms）：退化为异步复制。这时候 Raft/Paxos 的多数派确认开销太大——如果 leader 在北京，上海用户发起的写请求光网络往返就要几十毫秒。“就近写入"的优势荡然无存。

注意：大部分存储系统的数据同步是主从架构，支持可配置的同步、半同步或异步复制，通过快照或增量日志来实现。Paxos/Raft 是其中最"强"的一档——同步、自动故障转移、RPO=0——但代价也最高。

到这里，一个自然的追问产生了：真的没有办法在跨城场景下做同步复制吗？

2.4 跨城强一致：Google Spanner 的方案 ##

有，而且目前大规模验证过的只有一个：Google Spanner。它的做法值得深入理解，不是因为它适合所有人，而是因为它展示了这个问题的理论上界是什么。

为什么需要跨城强一致？ ##

回到 2.2 的结论：单元化要求业务数据可以完美按地域切分。但有些数据天然不适合切分——全局配置、跨域库存、以及某些金融场景下的全局一致性要求。如果某条数据的 Paxos Group 的副本分布在多个地域，那对这条数据的写入就需要跨城 Paxos 共识。

Spanner 的答案是：用大量的 Paxos Group 管理数据，每个 Group 负责一部分数据的同步复制。

Paxos Group：化整为零 ##

如果全球所有数据都由同一个 Paxos Group 来投票，那么这个组的 Leader 会成为巨大的性能瓶颈。Spanner 的思路是：

  • 数据分片：将数据切分成很多小块（称为 Tablets 或 Splits），每个分片对应一个独立的 Paxos Group
  • 独立运行：不同的 Paxos Group 独立运行。对 A 数据的写入和对 B 数据的写入可以在不同的服务器、不同的地域上并行处理，互不干扰
  • 自动分裂：随着数据量增长，Spanner 会自动将一个 Tablet 分裂成两个，并创建新的 Paxos Group。这些 Group 被自动调度到不同的物理服务器上，分散 CPU 和存储压力
┌──────────────────────────────────────┐
│           Spanner 集群                │
│                                      │
│  Paxos Group 1     Paxos Group 2    │
│  ┌──────────┐      ┌──────────┐     │
│  │Leader:北京│      │Leader:上海│     │
│  │副本:沪,深 │      │副本:京,深 │     │
│  │负责:用户A-M│     │负责:用户N-Z│     │
│  └──────────┘      └──────────┘     │
│                                      │
│  Paxos Group 3     Paxos Group 4    │
│  ┌──────────┐      ┌──────────┐     │
│  │Leader:深圳│      │Leader:北京│     │
│  │ ...      │      │ ...      │     │
│  └──────────┘      └──────────┘     │
│         ... 数以万计的 Group ...      │
└──────────────────────────────────────┘

当一个事务只需要修改某个特定主键的数据时，只需要这一个 Paxos Group 内部投票即可。这里跨城的网络延迟是必须支付的代价——但通过合理调度 Leader 的位置（让 Leader 靠近写入流量的来源），可以把延迟控制在可接受的范围内。

跨 Group 事务：2PC over Paxos ##

真正棘手的是：当事务需要修改跨多个 Paxos Group 的多个 Key 时怎么办？

比如从账户 A 转账到账户 B，A 和 B 在不同的 Paxos Group 里。这时 Spanner 启动两阶段提交（2PC），让涉及到的 Paxos Group 协同工作：

  • 协调者 Group 的 Leader 发起 2PC Prepare
  • 所有参与者 Group 的 Leader 在各自的 Paxos Group 内部投票，把"我准备好提交了"写入 Paxos 日志并同步给副本
  • 协调者 Group 的 Leader 决定最终提交，也把"事务已提交"写入 Paxos 日志并同步

到这里，跨 Group 的数据修改已经能保证原子性了（要么全提交，要么全回滚）。但还有一个更微妙的问题没解决。

事务排序：TrueTime 登场 ##

每个 Paxos Group 只能保证自己内部的日志是有序的。但 Group 1 并不知道 Group 2 发生了什么。如果没有一个全球统一的"时钟”，我们就无法判断"Group 1 的 A 扣款"和"Group 2 的 B 存款"到底哪一个先发生——分布式事务的隔离性（Isolation）就会崩溃。

普通时钟为什么不行？

实际时间线（上帝视角）：
  10:00:00.000  事务 T1 提交完成（在北京）
  10:00:00.020  事务 T2 开始执行（在上海）

各节点的本地时钟：
  北京节点时钟: 10:00:00.050 ← 快了 50ms
  上海节点时钟: 09:59:59.980 ← 慢了 20ms

分配的时间戳：
  TS(T1) = 10:00:00.050（来自北京时钟）
  TS(T2) = 09:59:59.985（来自上海时钟）

结果：TS(T2) < TS(T1)，但实际 T2 发生在 T1 之后！

普通时钟的致命缺陷：误差是未知的、无界的。 你不知道自己的时钟偏了多少。

TrueTime 的洞察：不确定没关系，框住就行。

TrueTime 不给你一个点，而是给你一个区间：

TT.now() → [earliest, latest]

  earliest = 真实时间绝对不会早于这个值（下界）
  latest   = 真实时间绝对不会晚于这个值（上界）
  ε = latest - earliest（通常 1~7ms）

这是怎么做到的？Google 在每个数据中心同时部署 GPS 接收器和原子钟——两者互为校验。GPS 提供绝对时间基准（但有信号抖动），原子钟提供短时间内的极高稳定性（但会长期漂移）。TrueTime 守护进程持续从两个来源采样，计算出一个数学上保证正确的区间。

Commit-Wait：用等待穿越不确定性窗口。

对于一个读写事务 T：

1. 持有锁、准备提交时，取 latest 作为提交时间戳：
   s = TT.now().latest
   此时我们知道：真实时间 ≤ s（latest 是上界）

2. Commit-Wait：等待，直到 TT.now().earliest > s
   此时我们确定：真实时间 > s（earliest 是下界）
   在所有节点上，s 都已经是"过去"了

3. 释放锁，返回客户端

排序正确性的直观证明：假设 T1 在 T2 开始前完成。T1 的 commit-wait 完成意味着真实时间 > s1。T2 在 T1 之后才开始，所以 T2 开始时真实时间 > s1。T2 取时间戳 s2 = TT.now().latest ≥ 真实时间 > s1。因此 s2 > s1 严格成立。这就是外部一致性（External Consistency / Linearizability）。

只读事务的零延迟捷径 ##

Spanner 的真正效率体现在只读事务上（99%+ 的工作负载）。读事务选择一个"稍旧"的时间戳 t（只要 t < TT.now().earliest - ε），就可以直接从本地副本读取，零跨机房网络往返、零锁。这也是为什么 commit-wait 的成本（1~7ms）是值得的——它让占绝大多数的读事务得以极低成本完成。

为什么只有 Google 能做？ ##

  • 硬件依赖：TrueTime 依赖 Google 自建的全球原子钟+GPS 基础设施。CockroachDB 用混合逻辑时钟（HLC）+ 不确定性重启做了类似的事，但一致性等级弱一级（Serializable 而非 External Consistency）
  • 网络基础设施：Google 有全球专线，跨地域延迟比公网低得多
  • ε 并不总是 1ms：在 GPS 信号差或原子钟故障时，ε 会增大到几十 ms，commit-wait 也会相应变长

Spanner 的设计哲学：Paxos 是在空间上达成共识（多数派同意这个值），TrueTime 是在时间上达成共识（等真实时间超过这个时刻）。Spanner 的厉害之处在于——它意识到这两个共识可以分开做，并且各自选择最优方案。这个方案很美，也很贵。

2.5 另一条路：CRDT 与冲突合并 ##

前面讨论的所有方案都在试图避免或预防冲突。还有一条完全不同的路：接受冲突必然发生，然后在冲突发生后去修复它。

LWW（Last Write Wins）：最简单也最危险 ##

每条记录带一个时间戳，冲突时以时间戳大的为准。问题在于——分布式系统中时间戳不可靠，不同机器上的时钟不可能绝对同步。LWW 的适用场景极其有限：低价值、独立写入、对丢失不敏感的数据（如用户偏好设置、最近登录时间）。

版本向量：能检测冲突，但不解决冲突 ##

版本向量是 LWW 的升级版：每条记录携带它在每个副本上的修改计数，形如 {A:3, B:2, C:5}。当两个版本在合并时：

  • 如果一个是另一个的"祖先"（如 {A:3, B:2} vs {A:3, B:3}），后者覆盖前者——没有冲突
  • 如果两者互为"兄弟"（如 {A:3} vs {B:2}），则标记为冲突，交由应用层仲裁

版本向量解决了"检测冲突"的问题，但没有解决"自动修复冲突"的问题。应用层需要写一堆 if-else 来处理各种冲突场景——这恰恰是业务开发人员最不想做的事情。

听说过向量时钟的概念没？在并发冲突检测领域，向量时钟用得非常广泛，比如 go test -race 就使用向量时钟来进行 data race 检测。和这里的版本向量其实是一个意思。

CRDT：自动合并的数学保证 ##

CRDT（Conflict-free Replicated Data Type，无冲突复制数据类型）是近十年来分布式系统领域从学术走向工业的重要成果。它的数学基础是：合并操作必须满足交换律、结合律、幂等律，使得无论副本以什么顺序接收更新，最终都能收敛到相同的结果。

类型工作原理典型应用
G-Counter每个副本维护独立计数向量，合并时取各分量的最大值，查询时求和点赞数、浏览量
PN-Counter增量和减量分别用两个 G-Counter库存计数（可增可减）
OR-Set每个元素带唯一 ID，删除只移除"已观察"的元素，不会误删并发添加的元素购物车、标签
LWW-Register时间戳 + 值，最后写入胜出用户状态

CRDT 在以下场景已经证明非常成功：

  • 协作文档（Google Docs 级别的并发编辑）：Yjs、Automerge 已被广泛使用
  • Redis Enterprise 多活：内置 CRDT，自动处理跨区域写入冲突
  • Microsoft CosmosDB：Azure 的全球分布式数据库，使用 CRDT 实现多活

CRDT 的致命边界 ##

但 CRDT 不是万能药。它的致命局限是：只能处理"值运算"型的数据变更，处理不了"条件写入"。

if balance >= amount:     # 条件判断
    balance -= amount      # 值变更

CRDT 可以保证 balance 字段的合并是数学上正确的，但它不能保证 “balance >= amount” 这个条件在所有副本上同时为真——因为两个并发扣减可能在各自的数据中心都通过了条件检查，导致了透支。

CRDT 解决的是"怎么合并"的问题，解决不了"该不该写"的问题。 后者需要分布式事务或共识算法来保证。

四、如何决策：你真正的需求是什么？ #

经过以上分析，一个务实的决策框架呼之欲出：

第一问：你的业务真的需要「写」多活吗？ ##

很多团队在架构评审时执着于"做异地多活"，但当被追问到具体场景时，能想到的只是"机房断电了怎么办"。而"机房断电"实际上只需要热备 + 分钟级切流就能覆盖。

真正需要写多活的场景其实很窄：

  • 金融核心交易：不能丢数据，停机损失巨大
  • 社交基础设施（IM 消息、动态发布）：用户对延迟和不可用零容忍
  • 电商下单链路：停机直接等于营收损失

如果你的业务不在上面这个列表里，同城双活 + 异地灾备大概率够了。

第二问：你的数据模型适合做分区吗？ ##

观察蚂蚁和字节的实践，你会发现它们成功的前提是：核心业务数据可以按用户维度做分区。如果一个用户的大部分操作都在自己的数据范围内，那单元化的冲突规避就能生效。

但如果你的业务核心是一张全局热点表（如秒杀库存、排行榜），那单元化也帮不了你——你仍然需要面对真正的分布式一致性挑战。

第三问：你能接受多大的成本？ ##

方案年化增量成本（相对）需要的团队能力
读写分离 + 异地灾备1-2x运维 + DBA
同城 Paxos/Raft 多活3-5x运维 + DBA + 分布式数据库经验
单元化5-10x运维 + DBA + 架构师 + 中间件团队 + 业务改造配合
Spanner 级别10x+以上全部 + 特殊硬件 + GCP 深度绑定

成本不是线性的——每往右走一步，组织能力和基础设施的要求都会指数级上升。

五、总结：共识之外的思考 #

回到文章开头的问题。我对异地多活写一致性的理解，可以用几句话概括：

1. 读多活容易，写多活极难。 “读多活"通过数据复制 + 就近读取已经基本解决，绝大多数公司也只需要到这个程度。写多活的挑战是多维的——副本一致性、事务排序、物理延迟、冲突解决、成本——这些维度在不同场景下以不同方式交织，没有一个方案能同时完美解决所有维度。

2. 工业实践是逐步演进的。 这不是一个"哪个方案更好"的选择题，而是一个"你的业务约束下能接受哪个方案"的匹配题。大多数公司停在读写分离 + 异地灾备就够了；需要更高可用性的公司走单元化路线；极少数有特殊硬件和基础设施的公司（如 Google）能做到真正的全球写多活。

3. 冲突规避（单元化）比冲突解决（CRDT/共识算法）在工程上更可行。 这不是说 CRDT 和 Paxos 没有价值——它们在各自的适用场景下极其成功——但在通用的 OLTP 场景下，“让冲突不发生"比"让冲突自动修复"在工程上更可控，在业务语义上更安全。

4. 分布式一致性没有银弹，只有适合你业务模型的方案。 架构的终极评判标准不是"用的技术是不是最酷的”，而是"在给定的预算和业务约束下，这套方案能不能活下来”。

一个思想的转变：很多架构师一开始想做"异地多活"时，想的是"任何人在任何地方都可以写，系统自动解决冲突"——这是一个美好的愿景，但物理定律不支持。真正落地时，他们发现更可行的问题是：“怎样划分数据让每个数据只有一个写入者，以及这个写入者挂了之后，怎样平滑地换一个写入者？” 前者是分布式系统的理想，后者是分布式系统的现实。

好的架构师，不是那个实现了理想的人，而是那个在现实的约束下找到最优解的人。
