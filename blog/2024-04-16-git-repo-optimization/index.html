Git 仓库膨胀到 30GB 之后

Posted 2024-04-16 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

团队有一个服务端仓库，差不多 4 年历史，最近发现 git clone 巨慢，CI 拉代码也卡得要命。一查，.git 目录 30GB。

为什么这么大 ##

几个原因叠在一起：

二进制文件入库。 git-lfs 一直没启用，大家往仓库里塞了各种东西——编译工具链的 binary、服务构建产物（二进制、.so、.a），甚至有人把测试用的数据文件也传上去了。这些东西本来不该进仓库，应该用 Bazel 之类的构建工具统一管理工具链和依赖，但团队在这方面基础建设比较少，就一直没做。

.gitignore 没配好。 项目初期没认真写 .gitignore，构建中间产物、IDE 临时文件、依赖缓存目录都没忽略。后面想补，但文件已经进了历史，忽略规则只管增量。

git commit hook 缺位。 没有 pre-commit hook 检查大文件或禁止的文件类型，漏进来完全没感知。等发现的时候已经攒了几百个不该在仓库里的文件。

主干一顿狂怼。 团队没有养成分支开发的习惯。即使主干开发，也应该用较短生命周期的分支，整理好再 squash 合入。但实际情况是：一个本该一次性提交的完整功能，被拆成好几个快照式 commit——“先传上去看看"“再加点"“补一下”。每个 commit 都是当时代码树的完整快照，二进制文件的每个版本都被记录，.git 膨胀得飞快。

怎么清理 ##

AI 给的思路其实是对的：备份 → 清理大文件 → 可选压缩历史 → 垃圾回收 → 强制推送。但操作历史重写类命令之前，务必先备份源仓库或源分支，不然不可逆。

1. 备份 ##

cp -R your-repo your-repo-backup

没什么好说的，安全第一。最好连备份也推到远程另一个位置，防止本地误删。

2. 清理大文件 ##

两个常用工具：git filter-branch 和 BFG Repo-Cleaner。推荐 BFG，速度快且用法简单：

# 清理大于 100M 的 blob
java -jar bfg.jar --strip-blobs-bigger-than 100M your-repo.git

# 或者按文件名/目录清理
java -jar bfg.jar --delete-files '*.{jar,war,zip,tar.gz,tgz}' your-repo.git
java -jar bfg.jar --delete-folders build your-repo.git

BFG 默认保护当前 HEAD 上的文件不会被删——这比 git filter-branch 安全。

ps: 如果仓库里有敏感信息（密钥、token）也需要清理，BFG 的 --replace-text 可以处理。

3. 可选：压缩历史 ##

如果历史 commit 太多且没有保留价值（比如上面说的那种快照式提交），可以考虑 squash 或 shallow clone 替代方案：

# 方案 A：将老提交合并为一次
git rebase --root --onto <new-base>

# 方案 B：对 CI 而言，shallow clone 是最简单的
git clone --depth 1 <repo-url>

不过历史重写会影响所有人的本地仓库，需要提前和团队沟通，约定好时间窗口，所有人重新 clone。

4. 垃圾回收 ##

清理完引用后，Git 还需要物理删除这些对象：

# 先让 reflog 过期
git reflog expire --expire=now --all

# 强制 gc
git gc --prune=now --aggressive

--prune=now 会删除所有未被引用的对象（包括 reflog 里引用的），所以前面 reflog expire 必须跑。

5. 强制推送 ##

git push origin --force --all
git push origin --force --tags

之后团队成员需要重新 clone，或者 git fetch --all && git reset --hard origin/master。

怎么预防 ##

事后清理是应急手段，关键还是不让问题重现：

补 .gitignore。 认真梳理项目里哪些文件不应入库。至少覆盖常见构建产物、IDE 配置、依赖缓存。GitHub 的 gitignore 模板 是个好起点。

启用 git-lfs。 对于确实需要版本管理的二进制文件（比如设计稿、测试固件），用 git-lfs 管理。它在仓库里只存指针文件，大文件存到 LFS 存储后端。

git lfs track "*.psd" "*.png" "*.bin"
git add .gitattributes

配置 pre-commit hook。 结合 lint-staged 或自写脚本，在提交前检查文件大小、类型，超过阈值直接拦截。

规范分支和提交策略。 即使是主干开发，也建议用短生命周期分支 + squash merge。一个功能一次 squash commit，保持历史干净。

从源头治理。 工具和依赖的管理应该通过 XaC（Everything as Code）手段解决——用 Bazel 或类似构建系统声明式地描述工具链版本和依赖，CI 和本地开发按需拉取，而不是塞进仓库里。

总结 ##

30GB 的 .git 目录是多个问题叠加的结果：无 lfs、无 ignore、无 hook 检查、无分支规范。事后清理可以用 BFG + gc 的组合，但要注意备份和团队协调。更重要的事是把预防措施做到位——补齐 .gitignore、启用 git-lfs、配好 pre-commit hook、规范提交流程。仓库的臃肿不是一朝一夕造成的，别指望一次清理就一劳永逸。
