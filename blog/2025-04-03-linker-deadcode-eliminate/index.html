Dead Code Elimination: A Linker's Perspective

Dead Code Elimination: A Linker’s Perspective ##

Linkers are crucial components of the build process, responsible for combining object files and libraries into a final executable. A key, often unseen, function they perform is dead code elimination. This process removes code that is never called, reducing the executable size and potentially improving performance. Let’s demonstrate this with a practical example.

Link Dependent *.o files in *.a ##

1. Setting the Stage: add.c and minus.c

First, we create two simple C files:

add.c:

#include <stdio.h>

int add(int a, int b) {
  printf("Adding %d and %d\n", a, b);
  return a + b;
}

minus.c:

#include <stdio.h>

int minus(int a, int b) {
  printf("Subtracting %d and %d\n", a, b);
  return a - b;
}

2. Compilation and Assembly

We compile and assemble these files into object files:

gcc -c add.c -o add.o
gcc -c minus.c -o minus.o

3. Creating a Static Library (libmath.a)

Now, we create a static library libmath.a containing both add.o and minus.o:

ar rcs libmath.a add.o minus.o

4. main.c: Calling Only add()

This is the critical part. We create main.c and only call add():

#include <stdio.h>

int main() {
  int result = add(5, 3);
  printf("Result: %d\n", result);
  return 0;
}

5. Building the Executable (Static Linking)

We build the executable, linking against our static library:

gcc main.c -L. -lmath -o myprogram

-L. tells the linker to look for libraries in the current directory. -lmath links against libmath.a.

6. Inspecting with objdump (Initial Verification)

Now, let’s use objdump to see what’s included in the executable. We’ll focus on the symbol table:

$ objdump -t myprogram | grep minus
$ |

You should not see any symbols related to minus() in the output. Why? The linker only need to link the object file (add.o) which defines function add. So no need to link minus.o. That’s why there’s no any grepped minus ouput.

Link Dependent *.o without Deadcode Elimination ##

1. Combining into a Single File: math.c

Now, let’s rewrite the code to combine add() and minus() into a single file, math.c:

#include <stdio.h>

int add(int a, int b) {
  printf("Adding %d and %d\n", a, b);
  return a + b;
}

int minus(int a, int b) {
  printf("Subtracting %d and %d\n", a, b);
  return a - b;
}

2. Revisiting main.c

We keep main.c the same:

#include <stdio.h>

int main() {
  int result = add(5, 3);
  printf("Result: %d\n", result);
  return 0;
}

3. Compilation of math.c

Compile math.c into math.o:

gcc -c math.c -o math.o

4. Building and Inspecting (Second Verification)

Now, build the executable using math.o:

gcc main.c math.o -o myprogram2

Let’s check with objdump again:

$ objdump -t myprogram2 | grep minus
00000000004005ff g     F .text  000000000000002d              minus

This time, you will see symbols related to minus(). Because math.o contains both add() and minus(), and main.c calls add(), the linker includes minus() because it’s part of the math.o object file. This demonstrates that if the code is present in an object file, and any symbol from that object file is referenced, the entire object file is included.

Link Dependent *.o with Deadcode Elimination ##

1. Explicit Dead Code Elimination with Compiler Flags

Now, let’s use compiler flags to force the linker to remove minus() even though it’s present in math.o. This is where things get more advanced.

  • -ffunction-sections: This tells the compiler to place each function in its own section.
  • -fdata-sections: This tells the compiler to create data sections for each function.
  • -Wl,-gc-sections: This is a linker flag that tells the linker to remove unused sections. -Wl passes options directly to the linker.

Let’s recompile math.c with these flags:

gcc -ffunction-sections -fdata-sections -c math.c -o math.o

And build the executable:

gcc main.c math.o -Wl,-gc-sections -o myprogram3

Now, run objdump again:

$ objdump -t myprogram3 | grep minus
$ |

You should not see any symbols related to minus(). The -gc-sections flag, combined with -ffunction-sections and -fdata-sections, has successfully removed the minus() function from the final executable.

2. How does the linker do that?

The flags -ffunction-sections and -fdata-sections will tells the compiler and linker generate new section for any exported function and data definition. In our example, add(a,b) and minus(a,b) will not be put into section .text, they will be put into .text.add and .text.minus.

Then when build the final executable, only the referenced add function definition is included, that means, the .text.add section will be merged into the .text section in the final executable, while the .text.minus section is ignored.

These flags can be used to reduce binary size, which in turn reduce allocated memory size. It’s more friendly to disk usage and memory usage.

Conclusion

This demonstration highlights the power of the linker’s dead code elimination capabilities. By default, the linker removes unused code. However, when code is present in an object file, it’s included unless explicitly removed using linker flags. Understanding these principles is crucial for optimizing executable size and improving performance, especially in resource-constrained environments. The -gc-sections flag is a powerful tool, but it should be used with caution, as it can potentially remove code that is dynamically linked or used in other ways. Always test thoroughly after using aggressive optimization flags.
