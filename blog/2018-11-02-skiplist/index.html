跳表（Skip List） #

背景 #

在程序开发中，我们经常需要维护一个有序集合，支持插入、删除、查找、范围查询等操作。对于这类需求，有哪些数据结构可选？它们各有什么优缺点？

有序数组 ##

有序数组通过二分查找可以在 O(log n) 时间内定位元素，查找效率很高。但插入和删除需要移动大量元素，时间复杂度为 O(n)。如果集合元素频繁变动，性能会急剧下降。

有序链表 ##

有序链表在已知插入位置时，插入和删除操作只需 O(1) 时间。但问题在于，链表不支持随机访问，查找一个元素必须从头遍历，时间复杂度为 O(n)。即使链表本身已经有序，也无法像数组那样二分查找。

二叉查找树（BST） ##

二叉查找树试图兼顾两者：理想情况下，每个节点的左子树都小于它、右子树都大于它，查找、插入、删除都是 O(log n)。

但 BST 有一个致命缺陷——它可能退化。如果按升序或降序插入元素，BST 会退化成一条单链表，查找性能退化到 O(n)。

平衡树（AVL、红黑树） ##

为了解决退化问题，人们发明了平衡二叉查找树——通过严格的平衡条件和旋转操作保证树的高度始终为 O(log n)，从而保证 O(log n) 的查找性能。

平衡树的代价也很明显：

  1. 实现复杂：插入和删除时需要大量的条件判断和旋转操作来维持平衡，代码写起来容易出错；
  2. 旋转开销大：当节点数量很多时，旋转调整的代价不可忽视；
  3. 键比较成本高：如果存储的键是长字符串或其他比较耗时的类型，每次旋转带来的额外比较开销会进一步放大。
  4. 并发不友好：旋转操作会影响大片子树，实现高效并发平衡树需要极其复杂的锁或无锁机制。

相对于平衡树，RBTree红黑树是放宽了一些条件，但是上面的开销依然存在。

正如 William Pugh 在 1989 年发表的论文 Skip Lists: A Probabilistic Alternative to Balanced Trees 中所指出的：平衡树的实现过于复杂，我们需要一种更简单的替代方案。

跳表的诞生 ##

William Pugh 提出了一个大胆的想法：用概率来替代严格的平衡条件。

与其在每次插入、删除后费尽心思地旋转调整，不如让每个节点随机决定自己的"高度"，用概率保证整体结构的平衡性。这就是跳表（Skip List）。

核心思想：概率平衡似乎比严格强制平衡更简单且更高效。——William Pugh

跳表的设计思想 #

跳表本质上是对有序链表的改造。

想象一个有序的单链表，查找只能一个一个节点地遍历，效率 O(n)。要提高查找效率，直观的做法是增加索引——在链表之上再构建一层稀疏的索引链表，使其可以"跳过"部分节点。

查找时从顶层索引开始，向右移动直到下一个节点大于目标值，然后下降到下一层继续查找，直到找到目标元素或确认不存在：

节点层数的确定 ##

节点层数的确定方式是跳表的核心——随机。每个新插入的节点，通过抛硬币（以概率 p）来决定是否增加一层：

  • 第 1 层（数据层）：所有节点都在这一层（概率 1）
  • 第 2 层：概率 p（通常 p = 1/2 或 1/4）
  • 第 3 层：概率 p²
  • 第 k 层：概率 p^(k-1)

这种随机化策略保证了期望上每一层节点数是下一层的 1/p，索引结构自然保持平衡，无需任何旋转操作。

复杂度分析 ##

时间复杂度：设数据层有 n 个节点，第 2 层约 n·p 个，第 3 层约 n·p² 个……最顶层约 1 个。总层数约为 log_{1/p}(n)，即 O(log n)。每次查找从顶层开始，每层最多比较 1/p 次（常数），因此查找、插入、删除的时间复杂度均为 O(log n)。

空间复杂度：所有层的节点数之和为：

n + n·p + n·p² + … = n/(1-p)

当 p = 1/2 时，总和约 2n，即 O(n)，额外空间约为原数据的 1 倍。当 p = 1/4 时（如 Redis），总和约 1.33n，额外空间仅 33%。

与平衡树的对比 ##

维度平衡树（AVL/红黑树）跳表
平衡方式严格平衡条件 + 旋转概率随机化
实现复杂度高（大量条件分支、旋转）低（核心逻辑几十行）
查找/插入/删除O(log n)O(log n)（期望）
空间开销2 个指针/节点（左右子树）平均 1.33~2 个指针/节点
范围查询需要中序遍历底部链表天然有序，直接遍历
并发实现复杂（锁子树）简单（可逐层加锁，或实现无锁）

实际应用 #

跳表在工程中有广泛应用，最经典的当属 Redis。

Redis ZSet（有序集合） ##

Redis 的 ZSet（Sorted Set）是跳表最具代表性的应用。它底层使用跳表 + 哈希表的双重结构：

  • 哈希表：member → score 的映射，O(1) 查找成员分值（如 ZSCORE）
  • 跳表：按 score 排序，O(log n) 插入删除和范围查询（如 ZADD、ZRANGE、ZRANK）

两者共享同一份 member 字符串（通过指针），不会产生额外的内存浪费。

Redis 跳表在经典跳表基础上做了几个关键的工程优化：

  1. span 字段：每个索引层都记录了跨度（跨越的节点数），使得按排名查询（ZRANK、ZREVRANK）也能做到 O(log n)；
  2. backward 指针：底层链表的每个节点都有后退指针，支持反向遍历（ZREVRANGE）；
  3. score 相等时的排序：当 score 相同时，按 member 的字典序排序，保证有序性稳定；
  4. 节点层数概率 p = 1/4：平均每个节点指针数约 1.33，空间效率更高。

Redis 作者 antirez 曾解释过为什么选择跳表而非平衡树：

  • 跳表的实现更简单，调试和扩展更容易（ZRANK 支持只需少量代码改动）；
  • 对于内存数据库来说，插入/删除时不需要旋转，缓存局部性更好；
  • B+ 树的优势在于减少磁盘 I/O，对纯内存数据库并无帮助。

LevelDB / RocksDB MemTable ##

LevelDB 和 RocksDB 都是 Google 开源的 LSM-Tree 存储引擎。它们的 MemTable（内存表）就是基于跳表实现的，用于在内存中暂存写入数据，数据达到一定大小后刷写到磁盘。

RocksDB 在较新版本中支持多种 MemTable 实现，跳表（SkipList MemTable）仍是默认选项，因为其简单的并发控制非常适合高频写入场景。

其他应用 ##

项目用途
Apache HBaseMemStore 使用跳表存储待刷写数据
Apache CassandraMemtable 默认基于跳表实现
Apache Lucene倒排索引的 Posting List 中使用跳表加速多条件合并
InfluxDBTSI（Time Series Index）索引中使用跳表

Go 语言跳表 / ZSet 库推荐 #

如果你在 Go 项目中需要使用跳表或有序集合，以下是几个值得关注的库。

高性能并发跳表集合：skipset / skipmap ##

github.com/zhangyunhao116/skipset 是目前 Go 生态中性能最好的跳表实现之一。它基于 MIT 的 Lazy Skip List 算法，支持泛型，并发安全。

特点：

  • 支持泛型 New[T]()，Go 1.18+
  • Contains 和 Range 是 wait-free 的（不阻塞，比 lock-free 更强）
  • 相比 sync.Map，混合读写场景下快 3~15 倍，内存节省 50% 以上
  • 元素始终有序，而 sync.Map 是无序的

适合用于需要高性能并发有序集合的场景，如缓存、索引等。

同系列还有 skipmap，提供有序的 key-value 映射。

Redis 风格 ZSet：bytedance/gopkg ##

github.com/bytedance/gopkg/collection/zset 是字节跳动开源的 Redis 风格有序集合，可作为 Redis ZSet 的本地内存替代。

特点：

  • 完整的 Redis ZSet 命令支持：ZADD、ZRANGE、ZRANK、ZREVRANK、ZREM、ZCOUNT、ZUNION、ZINTER 等
  • 并发安全（细粒度锁）
  • Add/Remove 复杂度 O(log n)，Contains 复杂度 O(1)
  • 支持 Float64 分值类型

适合场景：不想引入 Redis 依赖，但需要 ZSet 数据结构的单机场景，比如排行榜系统。使用方式与 Redis ZSet 几乎一致：

import "github.com/bytedance/gopkg/collection/zset"

z := zset.NewFloat64()
z.Add(90, "Alice")
z.Add(89, "Bob")
z.Add(95, "Charlie")

// 按分数排名
rank, score := z.Rank("Alice", true)   // rank=1 (0-based)
// 范围查询
members := z.Range(0, -1)               // 从高到低全部

存储引擎级实现：CockroachDB/pebble arenaskl ##

github.com/cockroachdb/pebble/internal/arenaskl 是 CockroachDB 存储引擎 Pebble 内部使用的跳表，面向极致性能优化。

特点：

  • Arena 内存分配（减少 GC 压力）
  • 无锁并发（lock-free）
  • 性能随 CPU 核心数线性扩展
  • 支持正向和反向迭代

适合场景：对性能和内存分配有极致要求的系统级开发。但它是内部包，API 稳定性不保证。

库选型建议 ##

场景推荐库
高性能并发有序集合zhangyunhao116/skipset
排行榜 / 本地 ZSet 替代bytedance/gopkg/collection/zset
极致性能、无 GC 压力cockroachdb/pebble/internal/arenaskl

总结 #

跳表是一个精妙的数据结构设计。它用一个简单的随机化技巧，替代了平衡树复杂的旋转逻辑，在保持 O(log n) 性能的同时，大幅降低了实现复杂度和并发控制难度。

Redis ZSet 的成功也证明了跳表的工程实用性——它不是理论上好看但工业界不敢用的小众结构，而是在高性能、高并发的存储系统中经过了充分验证的选择。

如果你需要维护一个有序集合，且插入、删除、查找、范围查询都需要高性能，跳表值得认真考虑。
