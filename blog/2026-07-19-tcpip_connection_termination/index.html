tcp状态机：TCP 连接终止场景分析

Posted 2026-07-19 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 ##

TCP 连接终止看似简单（四次挥手），但在实际系统中，连接的关闭方式受到多种因素影响：进程是正常退出还是被 kill -9 杀掉？主动、被动建连接一侧行为是否一致？socket buffer 中有无数据行为是否一致？写半连接关闭/连接关闭期间收到数据？原来管理连接的进程是否存在行为是否一致？…… 这几个场景都是网络编程期间会观察到的一些令人困惑的现象，实际上它们结论很简单，大一统。但是对于网络编程新手，经常会偶尔踩到其中1条或几条，但是有没有深入探究，最后可能会模糊认为这些场景下有各种特殊case。

本文针对这几个典型场景，分别给出一些介绍、分析，最后收到一个一致的Linux内核处理行为上去。后续，希望碰到这类问题时，大家能够非常直截了当地做出判断。

场景一：kill -9 杀进程 ##

kill -9（SIGKILL）无法被进程捕获，进程直接退出。内核在回收资源时会关闭其打开的全部 fd，对 TCP 走的是与正常 close() 同一类路径（sock_close → tcp_close）。

关键点：发 FIN 还是 RST，不取决于「主动打开 / 被动打开」（Client / Server），而取决于该 socket 关闭时的状态。

常见判定：

条件结果
接收缓冲区没有未读数据，且未设置 SO_LINGER(on, 0)发FIN，正常挥手
接收缓冲区有未读数据发RST（未读数据将被丢弃）
SO_LINGER：l_onoff=1 且 l_linger=0发RST，立即掐断

SO_LINGER 的语义、三种配置与取舍见专题：SO_LINGER：close() 时 FIN 还是 RST？。

因此：

  • Client 被 kill -9：缓冲已读空 → FIN；有未读数据 → RST
  • Server 被 kill -9（含 accept() 出来的已连接 socket）：同一套规则，不是「一杀 server 就 RST」

可用同目录下的 kill9_fin_rst_demo.go 自测：buffer 空时 kill client / server，存活侧 Read 均为 EOF（FIN）；任一侧有未读数据再 kill，则为 connection reset by peer（RST）。

场景二：socket buffer 中是否有数据 ##

当连接关闭时，发送 FIN 还是 RST 取决于接收缓冲区是否有未读数据（client / server 对称）：

  • buffer 为空：发送 FIN，正常四次挥手
  • buffer 中有未读数据：发送 RST，因为数据将被丢弃，不需要正常关闭

这在实际应用中很常见：server 还没读完 client 发来的数据就被关闭了，client 会收到 RST。场景一里的 kill -9 也遵循同一规则。

场景三：Write() 不立即感知对端关闭 ##

Go 中 conn.Write() 返回 nil 并不代表数据已成功送达对端。因为：

  1. Server 端 conn.Close() 只发了 FIN 给 client，相当于 server 写端关闭
  2. Client 端认为还可以继续向 server 发数据
  3. conn.Write() 返回成功——数据被写入了内核发送缓冲区
  4. 当数据真正发送到 server 时，server 因连接已关闭返回 RST
  5. Client 直到第二次 Write() 或 Read() 时才发现错误

应用层的 write() 成功 ≠ 对端收到数据。

场景四：进程不在了但连接未完全关闭 ##

操作系统层面：如果连接的四元组还未完全销毁（如处于 TIME_WAIT），但 socket 归属的进程已经不存在了。此时对端发来的包到达：

  • 操作系统认为该连接没有上层应用来处理
  • buffer 中有数据也没用——没有进程来读
  • 直接 RST 快速断开

统一视角，处理行为是一致的！ ##

这几个场景可以统一理解：内核处理 TCP 关闭时，不看 Client/Server 角色，而看关闭瞬间的 socket 状态：

  • 接收缓冲区已读空（默认 linger）→ 正常关闭（FIN）；kill -9 在缓冲为空时也是这条路径
  • 接收缓冲区有未读数据，或 SO_LINGER(on, 0) → 异常/快速断开（RST）
  • 进程已退出、TCB 已销毁后再收到对端数据 → 无连接可归，回 RST（场景四）

总结 ##

TCP 连接终止场景看似纷繁，内核判定却始终一致：不看 Client/Server 角色，只看关闭瞬间的 socket 状态。进程被 kill -9 时内核代为 close，接收缓冲区已读空则发 FIN 正常挥手，有未读数据或设置了 SO_LINGER(on, 0) 则发 RST；应用主动关连接时规则相同。Write() 返回成功只说明数据进了本机发送缓冲，对端若已关连接，往往要等到后续读写才感知 RST。进程已退出且 TCB 销毁后，对端再发数据同样会收到 RST——没有上层可读，内核只能快速断开。理解这一点，碰到「kill 后是 EOF 还是 reset」「关连接为何 reset」「写成功却报错」等现象，便可直接从缓冲是否读空、连接是否仍在、半关闭状态等角度判断，而无需记忆一堆特殊 case。
