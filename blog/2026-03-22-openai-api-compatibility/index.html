OpenAI 协议兼容性：契约、厂商与半兼容边界

Posted 2026-03-22 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

问题 ##

接 LLM 服务，最烦的不是模型本身，是每次换一家 provider，客户端代码要不要重写————新API？请求参数要改吗？响应解析要改吗？API能力上能对齐吗？等等。

理想情况是这样的：

# 只改这两处
client = OpenAI(
    api_key=os.environ["API_KEY"],
    base_url="https://api.deepseek.com/v1",  # 或 ollama、openrouter、自建 vLLM
)
resp = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": "hello"}],
)

路径还是 /v1/chat/completions，字段还是 messages / choices / stream，SDK 不用换。这就是大家说的「兼容 OpenAI 协议」。

反过来，如果对方坚持自家 SDK（Gemini 的 generateContent、Claude 的 Messages API、Bedrock 的签名调用），那业务层要么双端适配，要么中间再塞一层 gateway——成本一下就不一样了。

所以这篇文章只回答三个问题：

  1. 「兼容」到底兼容到哪一层
  2. 谁兼容、谁故意不兼容
  3. 联调时「路径兼容、行为半兼容」会在哪遇到阻碍

兼容的是契约，不是模型 ##

大模型权重本身是个黑盒函数：吃 token 序列（tensor），吐 next-token 分布。你 HTTP 里发的 JSON，它直接吃不下。

API Provider 干的事，是在模型前面加一层适配：

HTTP JSON (OpenAI 形态)
    ↓  解析 messages / tools / stream 等
chat template 拼装（按模型家族不同）
    ↓
tokenize → 推理引擎
    ↓
decode → 再包装成 choices / delta 返回

兼容 OpenAI，指的是这层适配的对外契约长得像 OpenAI：

  • 路径：/v1/chat/completions（以及常见的 /v1/models、/v1/embeddings）
  • 请求字段：model、messages、temperature、tools、stream …
  • 响应字段：choices[].message、usage、SSE 的 data: {...} chunk

内部用 vLLM 还是 TensorRT-LLM、chat template 是 Llama 3 还是 Qwen，可以完全不同。开发者只改 base_url + api_key + model，业务逻辑不动。

注意：契约对齐 ≠ 语义能力对齐。同一字段在不同后端的默认值、截断策略、tool 参数校验，经常对不齐。

谁兼容 ##

按接入方式粗分四类：

类别代表实际观感
商业 APIDeepSeek、Groq、Together、Mistral基本按 OpenAI 形态交付，流式和 tool call 可用性较好
国内大厂Kimi、智谱 GLM、通义 DashScope通常另开 OpenAI 兼容 endpoint，原生接口可能是另一套
聚合 / 云OpenRouter、Azure OpenAIOpenRouter 把大量模型统一成 OpenAI 形态；Azure 是官方合作路径，鉴权与部署名有自己的规矩
本地推理Ollama、vLLM、LocalAI、LM Studio起服务后本地就有兼容 HTTP 口，本地开发、离线 demo 很方便

工程上的收益很直接：一套 client 可以在「云上 DeepSeek → OpenRouter 兜底 → 本机 Ollama 调试」之间切换。CI 里 mock 也简单，对着同一套 schema 录回放就行。

国内厂商那一栏要小心：文档里常同时存在「原生 API」和「OpenAI 兼容 API」，base_url 不同。接错地址会表现为 404 或字段对不上，不是 key 的问题。

谁不兼容，以及为什么 ##

厂商形态换 base_url 行不行
Google GeminigenerateContent 等自有结构不行，要 SDK 或自写适配
Anthropic ClaudeMessages API，字段名/结构不同不行（虽有社区适配器）
AWS BedrockAWS 签名 + 平台 I/O不行，绑的是云账号体系

不兼容通常不是「技术做不到」。常见三类取舍：

  1. 原生能力装不下Gemini 的超长上下文、原生多模态输入（直接塞视频等），硬塞进 OpenAI 的 messages 形态，要么字段变形，要么信息丢失。厂商更愿意用自己的请求模型表达能力边界。
  2. 生态绑定让你深度用自家 SDK、自家控制台、自家观测体系，迁移成本就上去了。这是产品策略，不是接口设计失误。
  3. 历史存量 OpenAI 协议成为事实标准之前，自家 API 已经有一批生产用户。推倒重来的兼容层要双写维护，短期不一定划算。

ps: 社区里「Claude/Gemini → OpenAI 形态」的 proxy 很多。那是调用侧或网关做的兼容，不等于厂商承诺稳定契约。版本一升，字段语义漂移很常见。

适配层中间到底转了什么 ##

以一次 chat/completions 为例，服务端大致是：

  1. 收到 JSON：messages 数组 + 可选 tools / temperature / stream
  2. 按 model 选 chat template（同一套 OpenAI 字段，落到 Llama 3 / Qwen / DeepSeek 上模板不同）
  3. 拼成模型真正认的字符串，例如 Llama 3 风格：
<|begin_of_text|><|start_header_id|>system<|end_header_id|>

You are a helpful assistant.<|eot_id|><|start_header_id|>user<|end_header_id|>

hello<|eot_id|><|start_header_id|>assistant<|end_header_id|>
  1. Tokenize → 推理 → detokenize
  2. 再包装回 OpenAI 风格：choices[0].message.content，流式则是一串 delta

所以「一个 client 打遍 DeepSeek / Groq / Ollama」的关键，不在权重长得一样，而在适配层把差异吞掉了。

这也解释了半兼容为什么难查：HTTP 200、路径对、字段名对，但：

  • tool call 的 arguments 是字符串还是对象，各家不一致
  • stream_options.include_usage 有的支持有的忽略
  • stop、logit_bias、response_format=json_object 支持度参差
  • system prompt 在模板里的位置不同，同样 messages 效果不同
  • 多模态 image_url 有的只认 url，有的要 base64，有的直接 400

联调时别只测 hello。至少验收：非流式、流式、tool call、长 context 截断、错误码形态。名字叫兼容，不等于每个字段、每种能力 100% 对齐。

ps: 如果是模型私有化部署，并且在这基础上封装兼容API的话，应该就涉及到上面这些内容。如果是作为LLM Provider的代理，那么可能就是在模型API基础上继续套一层API，可能牵扯到的就是API请求参数转换、响应参数转换以及流式这方面的支持。

写客户端时怎么选 ##

我的默认策略比较简单：

  • 能走 OpenAI 兼容 endpoint 就走。业务代码只依赖这一套契约，provider 当可替换后端。
  • 边缘能力单独验收。tool call、vision、json mode、超长上下文，不要假设「兼容了 chat 就等于兼容了全家桶」。
  • 真要接 Claude/Gemini/Bedrock：要么官方 SDK 隔离在 adapter 包里，要么用成熟 gateway（自建或 OpenRouter 一类），不要把三套协议散落在业务 handler 里。

OpenAI 协议已经是事实标准。兼容降低的是接入与迁移成本；不兼容的大厂多半在保差异化能力或既有生态。选型时把「协议形态」和「模型能力」分开看，少踩「文档写兼容、联调才发现半套」的坑。
