CI 与 CD 的本质区别：为什么 CI 不包含 CD

Posted 2026-07-01 20:02 +0800 by ZhangJie ‐ 1 min read

分享:  

工作在现代软件开发团队中，你一定对 “CI/CD” 这个词不陌生。然而，在日常交流中，我们经常会听到诸如"我们的 CI 流水线挂了"、“这个项目的 CD 做得不好"等说法。久而久之，很多人会产生一种模糊的印象：CI 和 CD 是一回事，或者 CI 包含了 CD。

严格来说，CI（持续集成）并不包含 CD（持续交付/持续部署）。 它们是 DevOps 流水线中紧密衔接但职责不同的两个独立阶段。本文将为你深度拆解 CI 与 CD 的本质区别、各自的核心任务，以及它们是如何协同工作，共同打造高效的现代软件交付流水线的。

1. 什么是 CI（Continuous Integration，持续集成）？ ##

CI 的核心关注点是代码的集成与质量验证。它的终点通常是生成一个可部署的产物（如 Jar 包、Docker 镜像等）。

当开发者提交代码到代码仓库（如 GitLab、GitHub）后，CI 流程会自动触发：

  • 代码拉取与编译： 确保新代码没有语法错误，能跑得通。
  • 运行单元测试（Unit Tests）： 验证业务逻辑是否被破坏。
  • 代码质量扫描： 检查代码规范和潜在的安全漏洞。
  • 构建产物： 把代码打包成制品（Artifact）。

2. 什么是 CD（持续交付 / 持续部署）？ ##

CD 接收 CI 生成的产物，并负责把它们安全地带到各个运行环境中。CD 实际上包含两个层次：

  • Continuous Delivery（持续交付）： 代码通过了 CI 的自动化测试后，会自动部署到测试环境或预发环境。但是否发布到生产环境，需要人工点击按钮来触发（手动决策）。
  • Continuous Deployment（持续部署）： 比持续交付更进一步。代码只要通过了 CI 和所有自动化测试，就不需要任何人工干预，直接由系统全自动部署到生产环境。

3. 一图看懂 CI 与 CD 的界限与联系 ##

我们可以用一个经典的流水线来表示它们的分工：

[ 编码 ] -> [ 提交 ] 
  |
  +-- 1. CI (持续集成) 开始 -----------+
  |   - 编译构建                        |
  |   - 运行单元测试                    |
  |   - 代码质量检查                    |
  +-- 终点：生成可部署制品 -------------+
        |
        +-- 2. CD (持续交付) 开始 -----------+
        |   - 自动部署到测试/预发环境         |
        |   - 运行集成测试/端到端测试         |
        |   - 🛑 人工审核审批                 |
        |   - 部署到生产环境                  |
        +-----------------------------------+
              |
              +-- 3. CD (持续部署) -------------+
                  - 🟢 连审批都省了，全自动上线  |
                  +-----------------------------+

总结 ##

  • CI 是 CD 的基础： 没有高质量的 CI 自动化测试和构建，CD 就如同空中楼阁（你不敢把未经测试的代码直接推上线）。
  • CD 是 CI 的延伸： CI 负责把代码变成"好用的包”，CD 负责把"好用的包"变成"在线的服务"。

所以，它们是相辅相成的合作关系，而不是包含关系。
