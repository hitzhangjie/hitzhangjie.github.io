Git LFS 工作机制：从3d模型文件不见了说起

Posted 2026-04-10 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

仓库里有大量 *.glb 模型资源，某台机器没装 Git LFS，git clone 之后发现模型目录几乎是空的。装上 Git LFS，文件又"自己回来了"。

这背后就一个问题：Git 到底存了什么？LFS 又存了什么？

一句话结论 ##

Git 管"版本关系 + 指针文本"，Git LFS 管"大文件本体"。两者各干各的，靠一个 pointer 文件对接。

Git LFS 的存储设计 ##

Git 仓库（.git/objects）里有什么？ ##

Git 本身仍然是三类对象：commit、tree、blob。这一点没变。

变化在 blob 的内容。当某个文件（比如 ship6.glb）被 LFS 跟踪后，对应的 blob 不再是二进制模型数据，而是一个很小的 pointer 文件（纯文本）：

version https://git-lfs.github.com/spec/v1
oid sha256:xxxxxxxx...
size 12345678

Git 历史里存的是"这个大文件的身份信息"——oid、size——不是文件本体。所以不管模型文件多大，Git 仓库本身的增长微乎其微。

LFS 本地缓存（.git/lfs/objects）里有什么？ ##

这里放的才是真正的大文件内容，按 oid sha256 组织。

Git LFS 根据 pointer 里的 oid 去远端 LFS 存储拉取本体，落到 .git/lfs/objects 目录下。这个目录跟 .git/objects 是两套体系，互不干扰。

工作区里看到的文件是什么？ ##

取决于 LFS 过滤器是否生效：

  • 生效且对象已拉取：工作区里是真实二进制文件，正常使用
  • 未生效 / 未安装 Git LFS：工作区里大概率是 pointer 文本文件——也就是把那段 version ... oid ... size ... 当成了文件内容
  • 特殊场景（sparse checkout 排除了路径、提交本身不含该路径等）：工作区可能直接没有这个文件

开篇说的"模型目录几乎是空的"，就是第三种情况——没装 LFS，路径下文件直接缺失。

为什么"装了 LFS 就自动恢复"？ ##

装上 Git LFS 之后，checkout / smudge 流程开始生效：

  1. Git 检出时发现某个文件是 LFS pointer
  2. smudge filter 读取 pointer 里的 oid
  3. 按 oid 去远端 LFS 存储拉对象，落到 .git/lfs/objects
  4. 把工作区文件替换为真实内容

整个过程对用户透明——你看到的就只是"文件回来了"。实际上它走了一整套拉取逻辑。

ps: smudge 是 Git 的 filter 机制之一。简单说，git checkout 把文件从 .git 写到工作区时，会经过 smudge filter；反过来 git add 时走 clean filter，把大文件转成 pointer 再存进 Git。

常见误区 ##

误区 1：Git 自动管理大文件本体

不是。在 LFS 跟踪的前提下，Git 只存 pointer。没装 LFS 的机器上 clone 下来，大文件本体根本不会下载。

误区 2：没装 LFS 就会看到 0 字节占位文件

不一定。更常见的情况是工作区里放着 pointer 文本文件，内容就是那三行 version / oid / size。某些场景下甚至直接缺文件——取决于路径是否在提交中、是否有 sparse checkout 等。

误区 3：models 目录空了就是 LFS 坏了

先确认路径对不对。资产的实际路径可能是 demo/public/models，不是仓库根的 models。路径搞错了，自然看不到文件，跟 LFS 没关系。

一个实用心智模型 ##

把 Git LFS 想成"Git 的大文件外置仓库"：

  • Git 提供目录索引和版本时间线——谁在什么时候改了什么
  • LFS 提供对象存储和按哈希取回——文件本体存在哪、怎么拿
  • pointer 是两者之间的凭证——Git 里存这个凭证，LFS 凭它取文件

排查清单 ##

怀疑 LFS 异常时，按这个顺序查：

  1. .gitattributes 里有没有对应路径的 filter=lfs 规则
  2. git ls-files "path/*.glb" 看看文件是否被 Git 跟踪
  3. 是否启用了 sparse checkout，可能把目标路径排除了
  4. 本机是否装了 git-lfs 并且执行过 git lfs install
  5. 工作区文件内容是 pointer 文本还是真实二进制——head -n 3 看一眼就知道

最后 ##

Git LFS 要解决的不是"让 Git 学会存大文件"，而是"让 Git 只存可版本化的指针，把大文件交给专用对象存储"。这个设计选择直接解释了你会遇到的所有现象——clone 后文件缺失、装 LFS 后恢复、diff 看到的是 pointer 变化而不是二进制内容变化。

理解了这个，剩下的就是按清单排查。
