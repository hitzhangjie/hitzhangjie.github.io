Embedding 与 Prompt：LLM 如何"理解"语言

Posted 2026-05-23 00:00 +0800 by ZhangJie ‐ 5 min read

分享:  

背景 ##

我们每天都在跟 LLM 对话——输入一句话，它返回一段回复。但这中间发生了什么？“你好"这两个汉字是怎么变成 GPU 能算的矩阵的？Embedding 为什么是"分布式"的而不是"一个维度对应一个特征”？Prompt 处理有哪两条路径？RAG（检索增强生成）里的向量检索又是什么？

这篇文章用同一套记号贯穿全文，一次性讲清楚。

记号含义
V词表大小（有多少个 token id）
d隐藏维度（每个 token 向量的长度）
T序列长度（这条 prompt 有多少个 token）

一、从 One-Hot 到 Embedding ##

One-Hot：最"笨"的表示法 ##

One-hot 就是"只有一个位置是 1，其余全是 0"。词汇表有 50000 个词，“猫"排在第 42 位，那它的 one-hot 向量就是：

[0, 0, 0, ..., 0, 1, 0, ..., 0]
         第42位是1，其余49999位全是0

核心问题：

  • 向量维度 = 词汇表大小，几万到几十万维
  • 任意两个词的 one-hot 向量点积都是 0——完全正交
  • “猫"和"狗"在 one-hot 空间里的距离，跟"猫"和"量子力学"一样远
  • 不携带任何语义信息，纯粹是一个"排他性编号”

Embedding：可训练的语义查找表 ##

Embedding 的本质是一个可训练的查找表。以 Word2Vec 为例：

输入层                    隐藏层（Embedding）
"猫" → [0,0,...,1,...,0] × W → [0.23, -0.15, 0.78, ..., 0.41]
      (50000维 one-hot)    (50000×300)     (300维 embedding)

关键操作：one-hot 乘以权重矩阵，等价于直接从 W 中"抽出一行”。所以 W 的每一行就是一个词的 embedding 向量。训练时，模型通过预测上下文（CBOW）或用中心词预测周围词（Skip-gram），不断反向传播梯度，更新 W 中对应行的数值。训练目标迫使语义相近的词在向量空间中靠拢。

“分布式表示"到底什么意思 ##

很多人以为 embedding 的维度是这样分工的：

局部表示（假设的理想情况，实际不存在）：
维度1: 是否动物    维度2: 是否宠物    维度3: 是否哺乳动物
猫 = [1, 1, 1]
狗 = [1, 1, 1]
石头 = [0, 0, 0]

真实情况是：

分布式表示（真实的 embedding）：
猫  = [0.23, -0.15, 0.78, 0.41, -0.62, ...]
狗  = [0.31, -0.09, 0.72, 0.35, -0.58, ...]
石头 = [-0.88, 0.02, -0.34, 0.91, 0.17, ...]

分布式表示的意思是：一个语义特征不是存在某一个维度里，而是分散编码在所有维度上，同时每个维度又参与编码多个不同的语义特征。 就像一个 300 人的合唱团唱"水果之歌”——你不能指着某个歌手说"就是他唱的水果"，是所有人的声音叠加在一起形成了"水果"这个和声。

“连续"的两层含义 ##

第一层：取值是连续实数（不像 one-hot 只有 0 和 1）。这让向量可以精确量化语义距离。

第二层（更本质）：语义空间是连续的。因为取值连续，你可以做语义运算：

国王 - 男人 + 女人 ≈ 女王

这之所以成立，是因为"国王"和"男人"之间的语义差异是一个连续的方向向量，可以被加减。在 one-hot 空间里这是完全不可能的——词汇之间是孤立的离散点。

One-HotEmbedding
维度高维（几万）低维（几十到几百）
稀疏性稀疏（只有一个 1）稠密（几乎所有维非零）
取值离散（0 或 1）连续（实数）
语义编码无语义，纯粹编号分布式编码语义
词间关系全部正交，距离一样近义词靠近，可做向量运算

二、Prompt 处理全流程：路径 A（Chat LLM） ##

A.1 分词：字符串 → Token ID ##

输入 UTF-8 字符串 "你好"。输出 input_ids，类型 int64，形状 [B, T]（B 是 batch size，推理时常为 1）：

input_ids = tensor([[872, 1962]], dtype=int64)  # 形状 [1, 2]

这些整数来自词表文件（HuggingFace 里常见 tokenizer.json），词表在训练分词器时已固定：

id 0   → "<|endoftext|>"
id 1   → "!"
...
id 872 → "你"
id 1962→ "好"

tokenizer 的工作流程：按 BPE/SentencePiece 规则把字符串切成子词序列 → 对每个子词查表得到下标 → 返回 input_ids。同一条字符串、同一个 tokenizer → 永远得到同一个 input_ids。

A.2 Token Embedding：ID → 向量 ##

在 GPT-2 / Llama 里，权重名叫 model.embed_tokens.weight（或 wte），形状：

W ∈ R^{V×d}    # float32，可训练参数

前向就是按行索引取子矩阵：

# input_ids: [B, T]  int64
# W:         [V, d]   float32
h = W[input_ids]      # 结果 h: [B, T, d]

不是对 id 做任何数学运算——id 只用来当行号。再加上位置编码（常见 h + pos_emb[pos]），得到 Transformer 第 0 层输入，形状 [B, T, d]。

A.3 用极小数字走一遍 ##

设玩具词表 V=5，d=4：

id | 子词   | W 的第 id 行
---|--------|---------------------------
0  | <bos>  | [0.1, 0.0, 0.0, 0.0]
1  | 你     | [1.0, 0.0, 0.0, 0.0]
2  | 好     | [0.0, 1.0, 0.0, 0.0]
3  | 世界   | [0.0, 0.0, 1.0, 0.0]
4  | <eos>  | [0.0, 0.0, 0.0, 1.0]

encode("你好") → input_ids = [1, 2]：

h = W[[1, 2]]   # 形状 [2, 4]
# h[0] = [1,0,0,0]
# h[1] = [0,1,0,0]

模型"看到"的是矩阵 h，不是汉字。

A.4 生成：向量 → 下一个 Token ID ##

最后一层某个位置输出向量 x，形状 [d]。对每个候选 id 打分：

logits = x @ W.T          # [V]，每项 = 与第 j 个 token 的匹配分数
next_id = argmax(logits)  # 或按 temperature 采样

这叫 weight tying：输入用 W[id] 取向量，输出用 x @ W.T 选 id。本质是对 V 个类做 softmax 分类。

A.5 三个容易混淆的操作 ##

操作是否合法结果
正常encode 得 input_ids，再 W[input_ids]合法正确
把"你"的 id 从 872 改成 10，但不交换 W 对应行非法读到的是 id=10 那个子词的向量，不是"你”
用 GPT 的input_ids 去索引 Llama 的 W非法行号与词表项对不上，输出无意义

核心原则：tokenizer 和 model 必须配对。W 在预训练时与词表一一绑定——换 Chat 模型必须换 tokenizer。

三、路径 B：RAG 的 Embedding 模型 ##

在 RAG（检索增强生成）场景中，还有一套独立的 Embedding 体系：

路径 AW[input_ids]路径 B retrieval 模型
权重所属LLM checkpoint 内独立模型（如BAAI/bge-small-en-v1.5）
输入int64 的 id 序列 [B,T]原始字符串（内部自己分词）
输出[B, T, d]，每个 token 一个向量[B, d_ret]，整段文本一个向量
用途自回归生成向量库 cosine 相似度检索

路径 B 的典型流程：

# 伪代码，以 sentence-transformers 为例
tokens = tokenizer_bge(text)           # 另一套词表，与 GPT 无关
ids_bge = tokens["input_ids"]          # [1, T_bge]
h = encoder_bge(ids_bge)              # Transformer → [1, T_bge, d_bge]
vec = mean_pool(h, attention_mask)    # [1, d_ret]，如 d_ret=384
vec = normalize(vec)                  # L2 归一化

写入向量库的是 vec（float32 [d_ret]），不是 token id。

Agent 完整数据流 ##

【离线建库 — 路径 B】
file.md
  → chunk_i（纯文本）
  → vec_i = RetrievalModel(chunk_i)     # float32[d_ret]
  → INSERT INTO vector_db (id, text, vec_i)

【在线问答】
user_query
  → q = RetrievalModel(user_query)      # 必须与建库同一模型
  → top_k texts = vector_db.search(q)
  → prompt_str = system + join(top_k texts) + user_query

【生成 — 路径 A】
prompt_str
  → ids = ChatTokenizer(prompt_str)
  → h = ChatModel.embed_tokens(ids)     # [1, T, d]
  → ... Transformer ... → 生成文本

关键：vec_i 与 Chat LLM 的 h 维度、词表、权重均无交集。两条路径之间传的是纯文本字符串，不是向量拼接。

四、ANN：让向量检索"快"起来 ##

RAG 建库后，向量库里可能有成千上万甚至上百万条 chunk 向量。用户提问时，要从中找出最相关的几条——这件事在算法上叫最近邻搜索（Nearest Neighbor Search）。

先分清两个词：NN 与 ANN ##

缩写全称含义
NNNearest Neighbor精确最近邻：和库里每一条向量算相似度，再取 Top-K。结果正确，n 很大时很慢
ANNApproximate Nearest Neighbor近似最近邻：不保证全局最优，但通常「够近」，换的是搜索速度的数量级提升

ANN 不是另一种相似度公式，而是一类加速检索算法 / 索引结构的统称。FAISS、HNSW、IVF、ScaNN 等，走的都是这条路线。

常见误解：余弦不是每条都算吗？索引预构建又是什么？ ##

初学者容易卡在这里：

  1. 向量检索不就是算余弦相似度吗？
  2. 那不就是查询时对库里每条向量重新算一遍？
  3. 「索引需要预构建」——难道是提前把文档两两相似度都算好？

答案：度量（余弦 / 内积 / L2）确实是查询时算的；预构建的不是 query–doc 分数表，而是对已入库向量搭好的加速数据结构。

查询事先不知道，没法预计算「用户问题 vs 每一条 chunk」的余弦。预构建干的事，更像 B 树 / 倒排索引：不是提前算好答案，而是提前整理数据布局，查的时候走捷径，不必扫全库。

阶段做什么
离线（建库 / 建索引）把所有 chunk 的 embedding 组织成索引（聚类、近邻图、量化表等）
在线（用户提问）只对少量候选算距离 / 相似度，再返回 Top-K

暴力精确检索（brute-force / Flat）：

对库里每一个向量：cosine(query, vec_i)
取分数最高的 k 个

每次查询都和 n 条全算一遍。n = 1e6、d = 768 时，延迟和算力都吃不消。

ANN 检索：

1. 用预建索引，快速缩小到几百 / 几千个候选（不扫全库）
2. 只对候选算 cosine / L2 / 内积
3. 在候选里取 Top-K（可能略差于全局最优）

所以：仍然是「查询时算相似度」；不是「预先算完所有两两相似度」；是「预先把向量排好队形，查询时只和一小撮人比」。

预构建索引到底长什么样？ ##

不是「预先算两两相似度再挂到某个索引项下」。常见做法是按向量空间的几何结构做划分或连边。

IVF（倒排文件 / 聚类） ##

离线：
  1. 对全部文档向量做 k-means，得到例如 1000 个簇中心
  2. 每条向量只属于离它最近的那个簇
  3. 索引 = 「簇中心 → 该簇内向量列表」

在线查询 q：
  1. 先算 q 和 1000 个簇中心的距离，挑最近的 nprobe 个簇（比如 10 个）
  2. 只在这 10 个簇里的向量上算 cosine
  3. 取 Top-K

这里预建的是聚类 + 倒排列表，不是 query–doc 分数表。

HNSW（图索引） ##

离线：向量之间按「近邻」连成多层图
在线：从图上贪心 / 束搜索走几跳，很快落到附近区域

预建的是近邻图，查询时在图上走，而不是全表扫描。

ANN 的输入输出 ##

输入：
- 数据集向量：n × d（n 个 d 维向量，如 100 万个 512 维的文本 chunk）
- 查询向量：1 × d（用户问题经过同一个 embedding 模型）
- 搜索参数：k（返回几个结果）、搜索精度（如 nprobe、efSearch）

输出：
- 近似最近邻的向量索引和距离分数

FAISS 示例 ##

import faiss
import numpy as np

d = 128                                    # 向量维度
xb = np.random.random((10000, d)).astype('float32')  # 数据集
xq = np.random.random((1, d)).astype('float32')      # 查询向量
k = 5                                      # 返回 5 个最近邻

# IndexFlatL2 是暴力 baseline：几乎没有复杂结构，查询时全量扫 L2
# 适合小数据或做对照；真正靠预构建换速度的是 IndexIVFFlat、IndexHNSW 等
index = faiss.IndexFlatL2(d)
index.add(xb)

D, I = index.search(xq, k)                # D: 距离, I: 索引
# I[0] = [123, 456, 789, ...]
# D[0] = [0.1, 0.2, 0.3, ...]

若要 IVF 这类「先建索引再搜」：

nlist = 100                                # 簇数量
quantizer = faiss.IndexFlatL2(d)
index = faiss.IndexIVFFlat(quantizer, d, nlist)
index.train(xb)                            # 离线：用数据训练簇中心（预构建）
index.add(xb)
index.nprobe = 10                          # 在线：只探 10 个簇
D, I = index.search(xq, k)

train / add 是离线预构建；search 时仍在算距离，但只落在 nprobe 个簇里。

整条链路 ##

【离线】
chunk 文本 → Embedding 模型 → 向量库
                              ↓
                    构建 ANN 索引（聚类 / 图 / 量化…）
                    （对已有向量做空间组织，不是预算用户问题）

【在线】
user_query → 同一 Embedding 模型 → 查询向量 q
                              ↓
                    用索引跳过大部分向量
                              ↓
                    只对候选算 cosine / L2 / 内积
                              ↓
                    返回近似 Top-K

关键点 ##

  1. 输入必须同维度：查询向量与数据集向量维度必须一致（来自同一个 embedding 模型）
  2. 输出是近似值：可能不是绝对最近邻，但在可接受误差内
  3. 距离度量可配置：L2、内积、余弦等不同距离影响结果；度量是查询时算的，不是预算死的
  4. 索引需要预构建：离线搭好空间加速结构（聚类、近邻图等），不是预先算好 query–doc 相似度
  5. Flat ≠ 真正的 ANN 加速：IndexFlatL2 是全量暴力 baseline；IVF / HNSW 等才靠预构建换速度

ANN 的核心价值在于：用可接受的精度损失，换取搜索速度的数量级提升。

总结 ##

整条链路串起来就是：

  1. Tokenization：自然语言 → 整数 token ID（词表查表）
  2. Embedding Lookup：token ID → 稠密向量 W[id]（nn.Embedding 查表）
  3. Transformer：向量序列 → 逐层变换 → 输出向量
  4. 输出投影：输出向量 → x @ W.T → 下一个 token 的概率分布
  5. RAG 分支：独立 embedding 模型 → 向量入库并预建 ANN 索引 → 查询时只对候选算相似度 → 拼接上下文 → 回注到步骤 1

理解这条链路，你就不会再混淆 “token ID 是怎么分配的”、“embedding 向量能不能跨模型混用”、“RAG 的向量为什么不能直接塞进 Chat LLM”、“ANN 预构建到底预建了什么” 这些问题了。
