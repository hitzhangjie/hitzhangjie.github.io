50行代码构建Linux容器：理解容器技术的本质

Posted 2018-11-11 00:00 +0800 by ZhangJie ‐ 4 min read

分享:  

背景 ##

容器 vs 虚拟机 ##

在容器技术流行之前，虚拟机（VM）是实现资源隔离和应用的打包部署的主流方案。虚拟机通过 Hypervisor 在物理硬件之上模拟出一套完整的虚拟硬件，每台虚拟机拥有独立的操作系统内核，隔离性极强，但代价也很明显：

  • 资源开销大：每台 VM 需要运行完整的 OS，占用大量内存和磁盘；
  • 启动慢：需要经历完整的 OS 启动流程，通常以分钟计；
  • 密度低：单台物理机能承载的 VM 数量有限。

容器则走了一条完全不同的路：它不虚拟化硬件，而是虚拟化操作系统。同一宿主机上的所有容器共享同一个 Linux 内核，通过 Linux namespace 和 cgroups 等机制实现进程级别的隔离与资源限制。这使得容器：

  • 启动极快：本质上是启动一个（组）进程，毫秒级；
  • 资源开销极小：没有额外的 OS 开销，内存和磁盘占用远小于 VM；
  • 密度高：单机可以运行成百上千个容器。

下面的表格总结了容器与 VM 的核心区别：

维度容器虚拟机
隔离粒度进程级（共享宿主机内核）硬件级（独立内核）
启动速度毫秒级分钟级
资源开销MB 级内存，几乎没有 CPU 开销GB 级内存，需模拟硬件
镜像大小通常几十 MB通常几 GB
密度单机成百上千单机几十上百

容器技术的本质：Linux namespace ##

容器的底层技术支撑正是 Linux namespace。Linux namespace 是内核提供的一种资源隔离机制，它让不同进程组看到的系统资源视图互不相同——每个进程组仿佛独占一个独立的系统环境。

Linux 内核当前支持以下 8 种 namespace：

Namespace引入版本隔离的资源
Mount (mnt)2.4.19文件系统挂载点
UTS2.6.19主机名和域名（nodename, domainname）
IPC2.6.19System V IPC、POSIX 消息队列
PID2.6.24进程 ID 编号空间
Network (net)2.6.29网络设备、协议栈、端口等
User3.8用户 ID 和组 ID
Cgroup4.6cgroup 根目录视图
Time5.6系统时间（CLOCK_MONOTONIC, CLOCK_BOOTTIME）

其中 Mount、UTS、IPC、PID、Network、User 是最常用的六个维度，构成了容器隔离的基础。一个"完整"的容器通常需要组合使用这些 namespace，再加上 rootfs 切换（pivot_root / chroot）和资源限制（cgroups）以及安全策略（capabilities、seccomp 等）。

一个最小化的容器演示 ##

学习 Go 语言时，我无意中看到了 Liz Rice 的演讲 Building a container from scratch in Go，她用约 50 行 Go 代码演示了如何从零构建一个 Linux 容器。这个 Demo 虽然简陋，却清晰地展示了容器技术最核心的两个要素：namespace 隔离 和 rootfs 切换。

下面我们一起来看这段代码，理解每一部分在做什么。

代码解析 ##

整体结构 ##

package main

import (
    "fmt"
    "os"
    "os/exec"
    "syscall"
)

// go run main.go run cmd args
func main() {
    switch os.Args[1] {
    case "run":
        run()
    case "child":
        child()
    default:
        panic("what??")
    }
}

程序入口根据第一个参数分发到两个分支：

  • run：用户直接调用的入口，负责重新执行自身并传入 child 参数；
  • child：在 namespace 隔离环境中真正执行的逻辑。

为什么要重新执行自身？因为 Linux namespace 的创建必须通过 clone() 系统调用（或其封装），指定 CLONE_NEW* 标志，让新的子进程进入新的 namespace。这意味着我们无法在当前进程中直接"进入"新 namespace——必须 fork 出一个新进程。在 Go 中，exec.Command 调用 fork + exec，而 SysProcAttr.Cloneflags 正好可以指定 clone 的标志位。

run()：创建隔离进程 ##

func run() {
    cmd := exec.Command("/proc/self/exe", append([]string{"child"}, os.Args[2:]...)...)
    cmd.Stdin = os.Stdin
    cmd.Stdout = os.Stdout
    cmd.Stderr = os.Stderr
    cmd.SysProcAttr = &syscall.SysProcAttr{
        Cloneflags: syscall.CLONE_NEWUTS | syscall.CLONE_NEWPID,
    }
    must(cmd.Run())
}

run() 做了以下事情：

  1. 重新执行自身（/proc/self/exe），就像"自己启动自己"一次，但参数改为 child 加上原本的命令行参数。
  2. 将标准输入输出重定向到当前终端，这样容器内运行的命令才能正常交互。
  3. 设置 Cloneflags：CLONE_NEWUTS 让子进程拥有独立的 UTS namespace，CLONE_NEWPID 让子进程拥有独立的 PID namespace。

这里只用了两个 namespace flag，一个更完整的容器会在 Cloneflags 中加上更多标志，例如：

syscall.CLONE_NEWUTS | syscall.CLONE_NEWPID | syscall.CLONE_NEWNS |
syscall.CLONE_NEWNET | syscall.CLONE_NEWIPC

注意：CLONE_NEWUSER 涉及用户命名空间的映射（uid_map/gid_map），处理起来更复杂，通常不直接在 Cloneflags 中设置，而是由容器运行时（如 runc）单独管理。

child()：初始化容器环境 ##

func child() {
    fmt.Printf("running %v as pid:%v\n", os.Args[2:], os.Getpid())
    cmd := exec.Command(os.Args[2], os.Args[3:]...)
    cmd.Stdin = os.Stdin
    cmd.Stdout = os.Stdout
    cmd.Stderr = os.Stderr
    cmd.SysProcAttr = &syscall.SysProcAttr{
        Cloneflags: syscall.CLONE_NEWUTS | syscall.CLONE_NEWPID,
    }

    // 切换根文件系统
    must(syscall.Chroot("/home/rootfs"))
    must(syscall.Chdir("/"))
    // 挂载 proc 文件系统
    must(syscall.Mount("proc", "proc", "proc", 0, ""))
    must(cmd.Run())
}

进入新的 namespace 后，child() 做了一个容器运行时最关键的初始化：

1. 切换 rootfs（chroot） ##

must(syscall.Chroot("/home/rootfs"))
must(syscall.Chdir("/"))

chroot 将进程的根目录切换到 /home/rootfs，这样容器内的进程就只能看到这个目录及其子目录——这便是容器镜像的最原始形态。chdir("/") 确保当前工作目录也在新的根文件系统内。

现代容器运行时（如 runc）通常使用 pivot_root 而非 chroot，因为 pivot_root 更安全，能彻底脱离旧的 rootfs。但 chroot 对于理解容器原理已经足够了。

2. 挂载 proc 文件系统 ##

must(syscall.Mount("proc", "proc", "proc", 0, ""))

进入新的 PID namespace 后，如果不重新挂载 /proc，ps、top 等命令看到的仍然是宿主机的进程列表。重新挂载 proc 后，容器内就只会看到自己的进程。

3. 准备 rootfs ##

代码中的注释提示了如何准备一个最小的 rootfs：

mkdir /home/rootfs
cd /home/rootfs
cp -r /bin /dev /etc/ /lib /lib64 /sbin /usr /var ./
mkdir -p proc/self

这实际上是从宿主机复制了一份基础文件系统。在生产环境中，这部分工作由容器镜像（一组分层打包的文件系统）来完成，容器运行时会负责解压镜像并设置 rootfs。

容器隔离性的完整维度 ##

上述 Demo 只演示了 PID 和 UTS 两个 namespace，要构建一个真正可用的容器，还需要考虑以下维度：

维度对应机制Demo 是否实现
进程隔离PID namespace✅
主机名隔离UTS namespace✅
文件系统隔离chroot / pivot_root + Mount namespace⚠️ 用了 chroot，但未启用 Mount namespace
网络隔离Network namespace❌
IPC 隔离IPC namespace❌
用户隔离User namespace❌
资源限制cgroups（CPU、内存、I/O 等）❌
安全限制capabilities、seccomp、AppArmor/SELinux❌

Mount namespace（CLONE_NEWNS）至关重要——如果不启用，容器内的挂载操作（如上面的 proc 挂载）会传播到宿主机。Docker 在创建容器时总是启用 Mount namespace，以保证容器内的挂载不会污染宿主机。

Network namespace（CLONE_NEWNET）让每个容器拥有独立的网络栈：自己的网卡、IP 地址、路由表、端口空间。Docker 默认会为每个容器创建一个虚拟以太网卡对（veth pair），一端在容器内，一端在宿主机的 docker0 网桥上。

cgroups（Control Groups）负责限制而非隔离。如果说 namespace 解决的是"能看到什么"，cgroups 解决的则是"能用多少"——CPU 配额、内存上限、块设备 I/O 带宽等。没有 cgroups 的容器，一个进程的 fork bomb 就能拖垮整台宿主机。

User namespace 允许容器内的 root 用户映射到宿主机上的一个普通用户，大大降低了容器逃逸的危害。但也是实现最复杂的一个 namespace，涉及 uid_map/gid_map 的精细配置。

总结 ##

通过这段约 50 行的 Go 代码，我们看到了容器技术的最简实现：

  1. 利用 clone() 系统调用创建新进程，并通过 CLONE_NEWUTS | CLONE_NEWPID 等标志让其进入独立的 namespace；
  2. 使用 chroot 切换根文件系统，实现文件系统隔离；
  3. 重新挂载 /proc，让容器内的工具看到正确的进程列表。

当然，这个 Demo 距离生产级容器还有很长的路——它没有网络隔离、没有资源限制、没有安全加固。但它的价值在于去掉了容器"魔法"的外衣，让我们看到：容器不过是对 Linux 内核已有特性的巧妙组合。

现代容器运行时（runc、containerd、gVisor、Kata Containers 等）在这些基础之上做了大量工程化工作，但万变不离其宗——Namespace 隔离、Cgroups 限制、rootfs 切换始终是容器技术的三大基石。

感兴趣的读者可以观看原演讲视频 Building a container from scratch in Go，Liz Rice 的讲解非常清晰，强烈推荐。

package main

import (
    "fmt"
    "os"
    "os/exec"
    "syscall"
)

// go run main.go run cmd args
func main() {
    switch os.Args[1] {
    case "run":
        run()
    case "child":
        child()
    default:
        panic("what??")
    }
}

func run() {
    cmd := exec.Command("/proc/self/exe", append([]string{"child"}, os.Args[2:]...)...)
    cmd.Stdin = os.Stdin
    cmd.Stdout = os.Stdout
    cmd.Stderr = os.Stderr
    cmd.SysProcAttr = &syscall.SysProcAttr{
        Cloneflags: syscall.CLONE_NEWUTS | syscall.CLONE_NEWPID,
    }
    must(cmd.Run())
}

func child() {
    fmt.Printf("running %v as pid:%v\n", os.Args[2:], os.Getpid())
    cmd := exec.Command(os.Args[2], os.Args[3:]...)
    cmd.Stdin = os.Stdin
    cmd.Stdout = os.Stdout
    cmd.Stderr = os.Stderr
    cmd.SysProcAttr = &syscall.SysProcAttr{
        Cloneflags: syscall.CLONE_NEWUTS | syscall.CLONE_NEWPID,
    }

    // 切换根文件系统
    must(syscall.Chroot("/home/rootfs"))
    must(syscall.Chdir("/"))
    // 挂载 proc 文件系统
    must(syscall.Mount("proc", "proc", "proc", 0, ""))
    must(cmd.Run())
}

func must(err error) {
    if err != nil {
        panic(err)
    }
}
