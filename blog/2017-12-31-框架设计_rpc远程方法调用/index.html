RPC，不是近几年才有的新兴技术

Posted 2017-12-31 12:24 +0800 by ZhangJie ‐ 2 min read

分享:  

Remote Procedure Call (RPC) #

1 RPC简介 ##

RPC，是Remote Procedure Call的简称，译为“远程过程调用”。

RPC设计初衷是想提供一种面向“application-to-application”（而非“host-to-host”）的通信协议，RPC使得本地程序可以像访问本地资源一样来访问其他主机上的资源。

1.1 并不“新”：比近十年流行更早的一条线 ##

近十来年，随着微服务、多语言中间件的普及，各类 RPC 框架（gRPC、Thrift、Dubbo 等）重新变得很热。但 RPC 本身绝不是近十年才有的概念——它的历史可以追溯得很远：

  • RFC 层面：文中对照的 RFC 684（1975）、RFC 707（1976）已经在讨论“把过程调用做成网络协议、面向 application 共享资源”这类问题，距今半个世纪量级。后来更广为人知的 ONC RPC / Sun RPC 相关规范（如 RFC 1057 等）以及系统里仍常见的 rpcbind / portmapper，也都是上世纪八九十年代那一套 Unix/NFS 生态里的产物，算得上“古老但成体系”的实现。
  • 其他语言与中间件生态：并不只有 C/Unix 这一支。例如 CORBA（Common Object Request Broker Architecture，OMG 在 1990 年代推动）在当时跨语言、跨平台的分布式对象调用里极有名；同一时期前后还有 DCOM、Java RMI 等。它们和今天的“RPC 框架”形态不同，但核心动机类似：让远程调用尽量像本地调用，并配套命名、定位、桩代码生成等机制。

所以，若只从“最近十年 RPC 很火”来理解，很容易把问题看成全新课题；翻一翻那些年代久远的 RFC，再看看 rpcbind，以及 CORBA 一类中间件，会更清楚：注册、发现、地址透明、stub/IDL 等，老问题换了新皮，反复被各代基础设施重新实现了一遍。

Steve Vinoski 在一次 Youtube 分享中介绍了 rpc 演进的过程，点击观看分享。

1.2 对照规范时真正在意什么 ##

现在所谓的各种 rpc 技术，wiki 上列举了很多（点击查看 wiki）。若对照 RFC 684、RFC 707 所关心的问题来看，很多号称使用RPC的团队，也并没有系统性地解决规范里真正强调的那部分——尤其是面向 application 的服务注册与发现，而不只是“发一个请求、收回一个响应”。普通的网络请求/响应收发，本身并不等于已经具备了规范中追求的那种“位置透明”的 RPC 能力。

即使在今天，一些基础设施在这块做得并不透明。典型例子：服务固定监听某个端口，再和别的服务混部在同一台机器上；注册时默认“这个服务名下的 IP 列表都用同一个端口”。这样一来，服务注册至少没有做到完全透明——调用方或注册中心仍要依赖部署约定，而不是由机制把「程序身份 → 实际监听地址/端口」映射清楚。服务发现相对还好一些：一旦登记了实例，按名查找、拿到地址列表通常是通的；短板更常出在注册侧是否把端口、实例与部署细节真正解耦。

2 rpcbind ##

rpcbind（或 portmapper），点击查看 wiki，是更贴近上述目标的一类实现：由 rpcbind 维护 program number / version / 协议类型与 port number 的映射，客户端先查询再调用，从而把“应用身份”和“具体监听端口”解耦开来，更接近面向“application-to-application”的设想。

基于 rpcbind 的 rpc 开发，利用软件包提供的 rpcgen 以及服务描述文件，可以快速生成 rpc client stub、server stub 代码：rpcgen -a -C calc.x。印度小哥 Kashif Kaularikar 在 Youtube 上也做了个分享，点击观看分享。

3 RPC程序开发 ##

这里描述的 rpc 程序开发，是基于 rpcbind 服务进行的。

“像访问本地资源一样来访问其他主机上的资源”，决定了在实现形式上，RPC 的 client、server 端都需要实现服务定义的方法，只不过 client、server 端对该方法的实现逻辑是不同的。

  • client 端的方法实现，方法体内部会发起 RPC 相关网络请求

    • 先发起 portmap 请求，即向远程主机上的 rpcbind（或 portmapper）服务发起查询请求。查询服务 program number、服务版本 version number（版本中定义了不同的过程）、协议类型对应的服务监听端口 port number；

    • 收到响应的端口号后，再发起 rpc 请求，向服务主机、端口发起网络请求，并接收响应；

      rpc 请求中要调用的远程过程以及参数信息都封装在 PDU 的 continuation data 中，rpc 响应中返回的计算结果也封装在 PDU 的 continuation data 中。

  • server 端的方法实现，方法体内部会实现相关计算并 return 响应

    • 启动时先发起 portmap 请求，请求 rpcbind 取消其 program number、version number 上的端口映射设置；

    • 收到响应后再分别针对不同方法、协议类型发起 portmap 请求，请求 rpcbind 重新注册 program number、version number、协议类型对应的 port number；

      因为 server 端启动的时候，监听套接字的时候操作系统会自动分配端口号，server 端就将对应的端口号向 rpcbind 发起注册，由 rpcbind 来维护 program number、version number、协议类型与 port number 之间的映射关系。这也正是“注册侧透明”的关键一步：端口不必写死进调用约定，而由运行时登记。

总结 ##

基于 rpcbind 实现 rpc 服务注册、查询，一次有效的 rpc 需要两次网络 IO 才能完成：一次是 portmap 协议 getport 请求/响应以获取 rpc 服务监听端口，一次是 rpc 协议请求/响应。对于 QPS 要求比较高的服务场景，rpcbind 很可能会成为网络瓶颈。

现在很多流行的 RPC 框架都支持了服务注册、服务发现能力。真正的区别在于：部署时（尤其是同一台机器混部多个服务时）是否解耦了服务名与各服务实例监听地址之间的映射关系。这些服务实例可能使用配置好的固定监听端口，也可能是完全随机、对调用方透明的端口。至于实例监听地址如何写入注册中心，常见有两种、且都能做到完全透明：一是由平台下发监听地址配置，等服务实例起来后再代为注册；二是服务进程自己 listen 成功后主动注册。
