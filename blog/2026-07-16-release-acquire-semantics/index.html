Release-Acquire 语义：从锁的直觉到内存屏障的泛化

Posted 2026-07-16 20:20 +0800 by ZhangJie ‐ 3 min read

分享:  

理解写屏障（store-write barrier / smp_wmb）和读屏障（store-read barrier / smp_rmb）之后，再看 Java、Go、C++ 的并发编程资料，经常会碰到另一套术语：release 和 acquire。

  • Java volatile 变量的写被描述为"具有 release 语义"
  • Java volatile 变量的读被描述为"具有 acquire 语义"
  • C++ std::atomic 的 memory_order_release、memory_order_acquire
  • Rust 的 Ordering::Release、Ordering::Acquire
  • 甚至直接说"volatile 的语义是 release-acquire"

如果你对内存屏障有基础认知，初次看到这些词会感到困惑——为什么叫 release？为什么叫 acquire？它们跟读写屏障是什么关系？

本文就从互斥锁的直觉出发，把这些命名彻底讲清楚。

锁视角：release 和 acquire 的原型 ##

先忘掉内存屏障，回到最熟悉的并发原语——互斥锁（mutex）：

// Thread A                        // Thread B
lock.acquire()                     ...
data.x = 42                        ...
data.y = 99                        ...
lock.release()                     lock.acquire()
                                   read(data.x)  // 保证看到 42
                                   read(data.y)  // 保证看到 99
                                   lock.release()

两个关键操作的含义：

操作含义
lock.release()A 说：“我做完了，把我的修改发布 / 释放出去给其他人看”
lock.acquire()B 说：“我要获取 / 拿到最新的状态”

release 是"交出"，acquire 是"索取"——这是一对天然的匹配关系。锁保证了一件事：

A 在 release 之前的所有写，对 B 在 acquire 之后的所有读一定可见。

这就是 happen-before 关系。

从锁到无锁：把 release/acquire “剥离"出来 ##

核心观察：release 和 acquire 本质上是两个屏障点，锁只不过正好把它们分别绑在了 unlock 和 lock 上。

release 在做什么？它告诉硬件：“此刻之前的所有写，都必须在此刻真正完成，不能滞留。“换句话说，release 包含了一个写屏障。释放锁时，必须确保临界区内的所有修改都刷到了别人能看到的地方——而不是还躲在当前核心的 Store Buffer 里。

acquire 在做什么？它告诉硬件：“此刻之后的所有读，都必须从此刻起重新读取，不能使用过期的缓存。“换句话说，acquire 包含了一个读屏障。获取锁时，必须确保后续的读操作能看到最新数据——而不是用 Invalidate Queue 还没处理完的旧缓存行。

Thread A (release 视角):           Thread B (acquire 视角):

  data.x = 42                      lock.acquire()
  data.y = 99                         |
    |                           ══════╪══════════ acquire 屏障
    |                                 |           (此后的所有读必须看到最新值)
    |                                 ↓
    ↓                            read(data.x)
  lock.release()                 read(data.y)
    |
══════╪══════════ release 屏障
    |  (此前的所有写必须在此时被"发布")
    ↓

现在把锁拿掉，只保留这两个屏障点：

Thread A:                          Thread B:

  data.x = 42                      ...
  data.y = 99                      ...
    |                                 |
    |  ═══ release 屏障 ═══           |  ═══ acquire 屏障 ═══
    ↓                                 ↓
  atomic_flag.store(true,          if (atomic_flag.load(acquire)) {
    release)                           read(data.x)  // 保证看到 42
                                       read(data.y)  // 保证看到 99
                                    }

只要 B 的 acquire load 读到了 A 通过 release store 写入的值，那么 A 在 release 之前的所有写，对 B 在 acquire 之后的所有读一定可见。

这跟锁的保障完全一致——只是不再有锁的互斥（mutual exclusion），只剩下了"可见性"保证。

映射到硬件内存屏障 ##

有了上面的直觉，release/acquire 与读写屏障的映射就一目了然：

高级语义硬件屏障含义
release storewrite barrier（如 smp_wmb、sfence）此前的所有存储在此刻"发布”——不能重排到此之后，不能滞留
acquire loadread barrier（如 smp_rmb、lfence）此后的所有加载在此刻"获取”——不能重排到此之前，不能读过期的

也就是说：

  • Release = 带写屏障的 store。它保证把之前累积的修改一次性"发布"出去。
  • Acquire = 带读屏障的 load。它保证后续的读取能"获取"到最新发布的数据。

从实现角度看：

release store:                     acquire load:
  ┌──────────────────┐               ┌──────────────────┐
  │ 之前的 store      │               │                  │
  │   (不能重排至此下) │               │  (不能重排至此上)  │
  │ ←── 写屏障 ──     │               │  ── 读屏障 ──→    │
  │ atomic store      │               │ atomic load      │
  │ 之后的 store      │               │ 之后的 load      │
  │   (可以重排至此下) │               │  (可以重排至此下)  │
  └──────────────────┘               └──────────────────┘

为什么 release 只限制"之前的 store”？ ##

一个细致的点：release 是单向屏障，它只阻止此前的写重排到后面，不阻止后面的读写重排到前面。

// release store 不阻止后续的普通 store 被重排到它前面
data = 42;                          // ① 可能被重排到 release store 之后
flag.store(1, memory_order_release); // ② release 只保证 ② 之前的操作在 ② 之前完成
                                    //    不保证 ② 之后的操作不在 ② 之前执行

这意味着如果 B 侧通过 flag.load(acquire) 看到了 1，它不能保证看到 data = 42——因为 ① 可能被重排到 ② 后面去了。正确的用法是：release store 之前的写才是受保护的。

这是 release-acquire 和全屏障（memory_order_seq_cst / smp_mb）的关键区别。release-acquire 提供单向保证，足够覆盖绝大多数生产者-消费者场景。

为什么 volatile 是 release-acquire 语义？ ##

现在可以回答这个经典问题了。Java 语言规范（JLS）对 volatile 的定义是：

  1. volatile 写：写之前的所有操作不能重排到写之后 → release 语义
  2. volatile 读：读之后的所有操作不能重排到读之前 → acquire 语义

所以 volatile 的语义恰好就是"pairwise release-acquire synchronization”：

class Example {
    int data = 0;
    volatile boolean flag = false;

    // Thread A
    void writer() {
        data = 42;          // 普通写
        flag = true;        // volatile 写 = release store
    }

    // Thread B
    void reader() {
        if (flag) {                  // volatile 读 = acquire load
            int x = data;            // 保证看到 42
        }
    }
}

不是"映射到 store-load 屏障”。volatile 的读写分别承担 release/acquire 角色，构成一个单向的 happen-before 链：

Thread A:                          Thread B:
  data = 42                         ...
    |                                  |
    |  ←── release 屏障                |
    ↓                                  |
  flag = true  (volatile 写)           |
                          ╲           |
                           ╲          |
                            ╲         |
                             ╲        |
                          flag == true (volatile 读)
                                       |
                                       |  ←── acquire 屏障
                                       ↓
                                   int x = data  // 看到 42

条件：B 的 volatile 读必须读到 A 的 volatile 写所写入的值。如果 B 在 A 写之前就执行了读，那没有同步关系。

C++ 的 std::atomic 同理，memory_order_release 和 memory_order_acquire 是 volatile 语义的显式表达。Go 的 sync/atomic 包虽然没有显式的 ordering 参数，但其保证也等价于 release-acquire。

release-acquire vs 全屏障 ##

Release-Acquire 对全屏障（seq_cst / smp_mb）
屏障方向单向（release 拦 store, acquire 拦 load）双向（store + load 全拦）
保证配对的两个线程间单向同步所有线程看到一致的全局顺序
代价低（通常只需 store fence + load fence）高（x86 上 mfence 或 lock 前缀）
适用场景生产者-消费者、消息传递、锁释放-获取需要全局一致性的复杂并发算法

绝大多数场景下，release-acquire 就足够，不需要全屏障带来的额外开销。这也是为什么现代原子操作库（C++ atomic、Rust atomic）把 release/acquire 作为独立选项提供。

总结 ##

release 和 acquire 这两个名字不是凭空发明的，它们是对互斥锁语义的直接继承和泛化：

层次releaseacquire
锁的直觉释放锁，把修改"发布"出去获取锁，“接收"最新的状态
内存屏障写屏障：此前的 store 不能重排到此之后读屏障：此后的 load 不能重排到此之前
高级语义release store：我的写入在此刻发布acquire load：我从此获取最新写入
编程语言volatile 写、atomic store(release)volatile 读、atomic load(acquire)

下次再看到 release-acquire，心里默念：“释放-获取，就是发布写入和接收写入的一对同步点”——跟锁一模一样的直觉。
