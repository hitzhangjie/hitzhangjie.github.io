Git rerere：告别重复解决相同冲突的噩梦

Posted 2024-05-02 00:00 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 #

你有没有遇到过这种情况：在做一个长周期 feature 分支的开发，期间上游 main 分支也在不断演进。某天你决定把 feature 分支 rebase 到最新的 main 上，跑 git rebase main，结果哗啦啦出来好几个冲突。你一个一个解决，打完收工。

过了一周，main 又往前走了不少，你再次 rebase。同样的冲突又来了——你明明上周已经解决过一次了。

又或者更极端的情况：同一次 rebase 过程中，你在 commit #3 遇到了一个冲突，解决了。rebase 继续，到了 commit #7，同一个文件的同一段代码又冲突了。你翻出刚才的解决方案，手动再改一遍。

这感觉就像西西弗斯推石头——每次都觉得搞定了，下次又回到原点。

为什么会反复出现相同的冲突？ ##

理解这个问题，要先搞清楚 rebase 的本质。

git rebase 的工作方式是：把当前分支上相对于 base 的所有 commit 暂存起来，然后一个一个重新"播放"到新的 base 上。这个播放过程，其实就是对每个 commit 做一次 git cherry-pick。

而 cherry-pick 本质上是把某个 commit 的 diff patch 应用到一个不同的代码基上。当这个 patch 应用不上去时——比如 patch 要修改的那几行代码，在新的代码基上已经被别人改过了——就会产生冲突。

关键问题在于：Git 不会记住你之前是怎么解决这个冲突的。每次 cherry-pick 都是独立操作，每次都是 fresh start。

举个具体的例子：假设源分支 a 上有三个 commit——p1、p2、p3，都在改 config/constants.go 里的同一个 const 块：

p1: 在 const 块末尾加了 MaxRetryCount = 5
p2: 在 const 块末尾加了 DefaultTimeout = 30
p3: 在 const 块末尾加了 EnableCache = true

而目标分支 b 恰好在同一个位置加了个 MinBatchSize = 10。当你在 b 分支执行 git cherry-pick p1 p2 p3 时：

  • p1：patch 要往末尾加 MaxRetryCount，但那里已经被 MinBatchSize 占了，冲突。你手动解决。
  • p2：patch 要往末尾加 DefaultTimeout，同样撞上 MinBatchSize——同一个冲突模式。你刚才已经解决过一遍了，但 Git 不记得，再弹一次。
  • p3：同理，第三次。

三次冲突，你手动解决了三次，但问题本质是同一个。

跨 rebase 也是一样。每次 git rebase 都是全新的操作，Git 没有任何记忆——你上次花了半天解决的冲突细节，下次照样原封不动地弹出来。

这就是 git rerere 要解决的问题。

Git rerere 是什么 #

rerere 全称是 Reuse Recorded Resolution，翻译过来就是"复用已记录的解决方案"。

它的思路很朴素：既然 Git 自己不会记，那就加一个机制来帮它记。每次你解决一个冲突，rerere 就把这个冲突的"原版"和你的"解决方案"记录下来。下次遇到完全相同的冲突时，Git 直接帮你自动应用上次的解决方案，连冲突提示都不给了。

这跟 rerere 的本意有关：既然是"复用已记录的解决方案"，那说明对同一个冲突的解决方案已经"记录"过了，下次再遇到同一个冲突问题，就能直接复用解决方案，连"冲突提示"都可以"不提示"了。

rerere 如何工作 #

rerere 的工作流程可以分为两个阶段：记录和复用。

记录阶段 ##

当你解决完一个冲突、执行 git add（或 git commit）时，rerere 在后台做两件事：

  1. 记录 preimage（冲突前镜像）：冲突发生时的原始状态，就是带有 <<<<<<<、=======、>>>>>>> 标记的那个版本。
  2. 记录 postimage（冲突后镜像）：你解决冲突之后的干净版本。

这两个镜像被存储在 .git/rr-cache/ 目录下，用冲突内容的 SHA-1 哈希值来命名。同一个冲突 hunk，只要内容一样，就会命中同一条缓存。

复用阶段 ##

当下一次 Git 操作（rebase、merge、cherry-pick 等）遇到冲突时，rerere 会：

  1. 把当前冲突的 preimage 拿来算哈希
  2. 去 .git/rr-cache/ 里查有没有匹配的记录
  3. 如果有，直接把对应的 postimage 应用到文件上，标记为已解决
  4. 如果没有，就走正常的冲突提示流程，等你手动解决

这样一来，同一次 rebase 中的重复冲突、跨多次 rebase 的重复冲突，都会被自动处理掉。rebase 的中断次数大幅减少，你的心智负担也大幅降低。

一个小细节 ##

rerere 自动解决冲突后，文件已经被修改了（应用了你的解决方案），但 Git 不会自动 git add 这个文件。你需要手动执行 git add 来将它标记为已解决，然后继续 rebase。

不过实际体验下来，你会发现冲突已经"消失"了——打开文件看到的是干净代码，没有冲突标记。你只需要 git add + git rebase --continue 继续往下走就行。

如何启用 rerere #

启用方式很简单，全局开启：

git config --global rerere.enabled true

也可以用 --local 只对当前仓库开启：

git config rerere.enabled true

建议直接全局开启，这是一个"无副作用"的功能——它不会改变 Git 的行为逻辑，只是在你手动解决冲突后多做了一步记录，遇到相同冲突时少做一步手动操作而已。

启用之后，可以随时查看 rerere 的统计：

# 查看 rerere 记录了多少条解决方案
git rerere status

# 查看具体的 diff
git rerere diff

如果想清理很久没用的缓存记录：

# 清除超过 15 天未使用的记录
git rerere gc

实战演示 #

还是上面那个场景：源分支 a 上三个 commit 依次往 config/constants.go 的 const 块末尾加 MaxRetryCount、DefaultTimeout、EnableCache；目标分支 b 的同位置已经有 MinBatchSize。现在要把这三个 commit cherry-pick 过来：

git cherry-pick p1 p2 p3
# Auto-merging config/constants.go
# CONFLICT (content): Merge conflict in config/constants.go
# error: could not apply p1... add MaxRetryCount

冲突内容是这样的（const 块末尾）：

<<<<<<< HEAD
    MinBatchSize = 10
=======
    MaxRetryCount = 5
>>>>>>> p1 (add MaxRetryCount)

你手动解决，把两边都保留：

const (
    MinBatchSize  = 10
    MaxRetryCount = 5
)

git add config/constants.go
git cherry-pick --continue
# rerere 在 add 时悄悄记录了本次解决方案

这时处理 p2。p2 的 patch 是 diff(p1, p2)，要往 MaxRetryCount 后面加 DefaultTimeout，同样撞上末尾的 MinBatchSize。但这次：

# Auto-merging config/constants.go
# Resolved 'config/constants.go' using previous resolution.  ← rerere 自动解决了！

你打开文件一看，冲突标记都没了，代码已经干净了：

const (
    MinBatchSize    = 10
    MaxRetryCount   = 5
    DefaultTimeout  = 30
)

git add config/constants.go       # 仍需手动 add
git cherry-pick --continue        # 继续

p3 同理，rerere 再次自动解决。三次冲突，你只手动解决了一次。

这才是 rerere 真正闪耀的场景——单次操作处理多个 commit、反复碰到同一块"热区"的时候，rerere 能把你实际需要手动解决的冲突数量降低一个数量级。像常量块、import 块、配置文件字段这类多人频繁改动的区域，尤其受益。

rerere 的适用场景 #

rerere 在以下工作流中特别有用：

  • 长周期 feature 分支多次 rebase：最常见的使用场景。feature 分支存活越久，rebase 次数越多，rerere 的收益越大。
  • 反复 merge 同一个分支：比如你有一个 testing 分支，持续 merge 多个 topic 分支，同一个 topic 分支多次 merge 可能遇到相同冲突。
  • cherry-pick 多条 commit 到不同分支：同一组 patch 往多个 release 分支 cherry-pick 时，冲突往往是相似的。
  • git rebase --onto 迁移分支：把一组 commit 从一个 base 挪到另一个 base，中间的冲突 rerere 直接消化。

注意事项 #

rerere 虽好，也有几点要注意：

  1. 不要盲目信任自动解决。rerere 解决的依据是冲突内容完全一致。如果冲突的上下文变了，但 git 仍然认为是一样的 hunk，自动应用的方案可能不是你想要的。遇到自动解决的情况，最好还是 git diff --staged 看一眼。

  2. .git/rr-cache/ 是本地缓存，不会推送到远程。这意味着团队其他成员没法共享你的解决方案。每个人需要在自己的机器上独立积累 rerere 缓存。

  3. 不同仓库的问题域不同。rerere 是按仓库独立记录的，A 仓库的解决方案不会应用到 B 仓库。

总结 #

git rerere 解决了一个很实际的问题：Git 操作中重复出现相同冲突时，不需要每次都手动解决一遍。它通过记录 preimage（冲突状态）和 postimage（解决后状态），在下次遇到相同冲突时自动应用历史方案。

启用它只需一行命令 git config --global rerere.enabled true，没有任何副作用，却能显著减少 rebase 过程中的摩擦。尤其是维护长周期 feature 分支、经常需要 rebase 的开发者，开了就不会想关掉。

如果你还没有开启 rerere，强烈建议现在就把它加到你的全局 Git 配置里。
