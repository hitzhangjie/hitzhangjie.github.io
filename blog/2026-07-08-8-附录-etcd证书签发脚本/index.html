Etcd证书有效期延长带来的思考(8)：附录 etcd 测试证书签发脚本

Posted 2026-07-08 12:00 +0800 by ZhangJie ‐ 4 min read

分享:  

背景 #

排查 etcd 证书过期、信任链以及 CA 续签问题时，直接使用生产证书既不安全，也不便于稳定复现。下面的脚本会生成一套固定有效期的测试证书，用来模拟以下场景：

  • 原 CA 证书 ca.pem 快要过期，但其私钥 ca-key.pem 可用；
  • etcd 叶子证书 etcd.pem 由原 CA 签发，且同时具备服务端和客户端认证用途；
  • 使用同一份 CA 私钥重新签发有效期更长的 new-ca.pem；
  • 对比原 CA 与新 CA 的公钥、Subject、有效期，并验证叶子证书能否使用新 CA 建立信任链。

本脚本只用于学习和故障复现。证书有效期、Subject、密钥强度及扩展字段均为固定测试配置，不能直接用于生产环境。

脚本说明 #

脚本源码同时保存在 content/blog/infosec/scripts/generate-test-certs.sh。执行后，证书和私钥会写入脚本所在目录。

生成文件 ##

文件作用固定有效期
ca-key.pem原 CA 与续签 CA 共用的 RSA 私钥不适用
ca.pem原自签名 CA 证书2026-01-01 ～ 2026-07-01
etcd-key.pemetcd 叶子证书对应的 RSA 私钥不适用
etcd.pem由 ca.pem 签发的 etcd 叶子证书2026-01-01 ～ 2027-01-01
new-ca.pem使用 ca-key.pem 重新签发的 CA 证书2026-01-01 ～ 2027-01-01

其中 ca.pem 被刻意设置为提前过期，用于观察“叶子证书尚未过期，但签发它的 CA 已过期”时的校验结果。new-ca.pem 与 ca.pem 复用同一密钥对，因此二者公钥和 Subject Key Identifier 一致，但它们仍是序列号、有效期和签名不同的两张证书。

完整脚本 ##

#!/usr/bin/env bash

set -euo pipefail

# Generate a fixed set of test certs under ./ssl:
# - CA:      ca-key.pem + ca.pem (2026-01-01 ~ 2026-07-01)
# - ETCD:    etcd-key.pem + etcd.pem (signed by CA, 2026-01-01 ~ 2027-01-01)
# - Renewed: new-ca.pem (same CA key, 2026-01-01 ~ 2027-01-01)
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
SSL_DIR="${SCRIPT_DIR}"

CA_KEY="${SSL_DIR}/ca-key.pem"
CA_CERT="${SSL_DIR}/ca.pem"
ETCD_KEY="${SSL_DIR}/etcd-key.pem"
ETCD_CERT="${SSL_DIR}/etcd.pem"
NEW_CA_CERT="${SSL_DIR}/new-ca.pem"

CA_EXT="${SSL_DIR}/.ca-ext.cnf"
ETCD_EXT="${SSL_DIR}/.etcd-ext.cnf"
CA_CSR="${SSL_DIR}/.ca.csr"
ETCD_CSR="${SSL_DIR}/.etcd.csr"
NEW_CA_CSR="${SSL_DIR}/.new-ca.csr"
CA_SRL="${SSL_DIR}/.ca.srl"

# Remove temporary config/CSR/serial files at exit.
cleanup() {
  rm -f "${CA_EXT}" "${ETCD_EXT}" "${CA_CSR}" "${ETCD_CSR}" "${NEW_CA_CSR}" "${CA_SRL}"
}
trap cleanup EXIT

# Minimal x509v3 extensions for a usable CA cert.
cat > "${CA_EXT}" <<'EOF'
[v3_ca]
basicConstraints=critical,CA:TRUE
keyUsage=critical,keyCertSign,cRLSign
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid:always,issuer
EOF

# Extensions for etcd leaf cert (not a CA).
cat > "${ETCD_EXT}" <<'EOF'
[v3_etcd]
basicConstraints=critical,CA:FALSE
keyUsage=critical,digitalSignature,keyEncipherment
extendedKeyUsage=serverAuth,clientAuth
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid,issuer
EOF

# 1) Generate root CA key and CSR, then self-sign CA cert.
openssl genrsa -out "${CA_KEY}" 2048
openssl req -new \
  -key "${CA_KEY}" \
  -out "${CA_CSR}" \
  -subj "/C=CN/ST=shenzhen/L=shenzhen/O=etcd/OU=System/CN=etcd-ca"

openssl x509 -req \
  -in "${CA_CSR}" \
  -signkey "${CA_KEY}" \
  -out "${CA_CERT}" \
  -not_before 20260101000000Z \
  -not_after 20260701000000Z \
  -extfile "${CA_EXT}" \
  -extensions v3_ca

# 2) Generate etcd key and CSR, then sign etcd cert with CA.
openssl genrsa -out "${ETCD_KEY}" 2048
openssl req -new \
  -key "${ETCD_KEY}" \
  -out "${ETCD_CSR}" \
  -subj "/C=CN/ST=shenzhen/L=shenzhen/O=etcd/OU=System/CN=etcd"

openssl x509 -req \
  -in "${ETCD_CSR}" \
  -CA "${CA_CERT}" \
  -CAkey "${CA_KEY}" \
  -CAserial "${CA_SRL}" \
  -CAcreateserial \
  -out "${ETCD_CERT}" \
  -not_before 20260101000000Z \
  -not_after 20270101000000Z \
  -extfile "${ETCD_EXT}" \
  -extensions v3_etcd

# 3) "Renew" CA cert by re-signing CA CSR material with same CA key.
openssl x509 -x509toreq \
  -in "${CA_CERT}" \
  -signkey "${CA_KEY}" \
  -out "${NEW_CA_CSR}"

openssl x509 -req \
  -in "${NEW_CA_CSR}" \
  -signkey "${CA_KEY}" \
  -out "${NEW_CA_CERT}" \
  -not_before 20260101000000Z \
  -not_after 20270101000000Z \
  -extfile "${CA_EXT}" \
  -extensions v3_ca

# Note:
# Verifying etcd.pem against ca.pem may show "expired" at current date
# because ca.pem intentionally ends at 2026-07-01 for testing.
echo "Generated files:"
echo "  ${CA_KEY}"
echo "  ${CA_CERT}"
echo "  ${ETCD_KEY}"
echo "  ${ETCD_CERT}"
echo "  ${NEW_CA_CERT}"

实现要点 ##

  1. 生成 CA 密钥和原 CA 证书

    • 使用 RSA 2048 位私钥；
    • 通过 basicConstraints=critical,CA:TRUE 标记为 CA；
    • 通过 keyUsage=critical,keyCertSign,cRLSign 限定其证书签发和 CRL 签名用途。

  2. 生成 etcd 密钥和叶子证书

    • etcd-key.pem 只用于生成 CSR 和证明节点持有对应私钥；
    • etcd.pem 实际由 ca-key.pem 签名；
    • extendedKeyUsage=serverAuth,clientAuth 使其可用于 etcd 的双向 TLS 场景。

  3. 模拟 CA 续签

    • 先使用 openssl x509 -x509toreq 从原 CA 证书和原 CA 私钥重新生成 CSR；
    • 再使用同一份 ca-key.pem 自签得到 new-ca.pem；
    • 续签不会修改已经签发的 etcd.pem，只是提供一张有效期更长、持有相同 CA 公钥的新信任锚证书。

  4. 清理临时文件

    • CSR、序列号和扩展配置均为中间文件；
    • trap cleanup EXIT 确保脚本正常结束或异常退出时都会清理这些文件。

执行与校验 #

运行脚本 ##

脚本会覆盖同目录下同名的测试密钥和证书，执行前应确认目录中没有需要保留的文件：

cd content/blog/infosec/scripts
chmod +x generate-test-certs.sh
umask 077
./generate-test-certs.sh

建议先确认当前 OpenSSL 的 x509 子命令支持 -not_before 和 -not_after 参数；不同 OpenSSL 版本的可用参数可能不同：

openssl version
openssl x509 -help

查看证书内容 ##

openssl x509 -in ca.pem -noout -subject -issuer -serial -dates
openssl x509 -in etcd.pem -noout -subject -issuer -serial -dates
openssl x509 -in new-ca.pem -noout -subject -issuer -serial -dates

确认新旧 CA 使用同一公钥 ##

分别提取公钥并计算摘要，两个摘要应完全一致：

openssl x509 -in ca.pem -pubkey -noout \
  | openssl pkey -pubin -outform DER \
  | openssl sha256

openssl x509 -in new-ca.pem -pubkey -noout \
  | openssl pkey -pubin -outform DER \
  | openssl sha256

校验证书链 ##

由于脚本中的时间是固定的，应使用 -attime 分别复现原 CA 有效和过期后的结果：

# 2026-06-01：原 CA 与 etcd 证书都在有效期内，应校验通过
openssl verify -attime "$(date -d '2026-06-01 00:00:00 UTC' +%s)" \
  -CAfile ca.pem etcd.pem

# 2026-08-01：原 CA 已过期，应校验失败
openssl verify -attime "$(date -d '2026-08-01 00:00:00 UTC' +%s)" \
  -CAfile ca.pem etcd.pem

# 2026-08-01：续签 CA 仍有效，并且公钥未改变，应校验通过
openssl verify -attime "$(date -d '2026-08-01 00:00:00 UTC' +%s)" \
  -CAfile new-ca.pem etcd.pem

date -d 是 GNU date 的语法；在 macOS 等环境中，需要改用对应平台的时间戳生成方式。

使用限制 #

  • etcd.pem 没有配置 SAN，只适合验证签发关系和证书链；用于真实 TLS 主机名或 IP 校验时会失败。
  • 脚本为便于复现实验使用了固定时间，按当前时间直接执行 openssl verify -CAfile ca.pem etcd.pem，可能因 CA 已过期而失败，这是预期现象。
  • RSA 2048、固定 Subject 和单一叶子证书仅用于演示。生产环境应按组织安全规范选择算法、密钥长度、有效期和 SAN。
  • ca-key.pem 和 etcd-key.pem 都是敏感私钥。即使是测试文件，也不应提交到公开仓库；生产 CA 私钥更应离线保存并严格控制访问权限。
  • CA 续签能够延长 CA 证书有效期，但不能改变既有叶子证书自身的到期时间、SAN 或扩展用途；这些内容变化时必须重新签发叶子证书。

总结 #

该脚本用最小化的 X.509 扩展复现了“签发原 CA → 签发 etcd 证书 → 复用 CA 私钥续签 CA”的完整过程。通过固定时间和 openssl verify -attime，可以明确区分 CA 过期、叶子证书过期和信任锚替换三类问题，并验证复用同一 CA 密钥续签后对既有叶子证书链的影响。
