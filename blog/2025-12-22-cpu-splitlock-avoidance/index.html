Split Lock 压垮母机：一次CPU飙升的线上排查

Posted 2025-12-22 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

那天业务告警响了：服务延迟飙高，吞吐量断崖式下跌。

打开监控一看，CPU 使用率瞬间拉满。再看 atop，CPU 飙高的同时，IRQ（硬中断）处理量也同步暴增——两者的曲线几乎完全吻合。第一反应是网卡中断风暴或者定时器异常，但仔细看了中断分布，并没有哪个设备的中断计数特别高。问题出在别的地方。

后来才了解到，我们业务所在母机上的另一台子机存在严重的 Split Lock 问题——那台问题子机持续触发总线锁，把母机 CPU 资源拖垮了。母机调度器在资源争抢之下，很长一段时间没调度到我们的子机，等到终于调度上时，积压的中断一次性涌入，触发了所谓的"IRQ catchup"。表现出来的就是我们的子机上 CPU 和 IRQ 同时飙升。

所以这次事件的根因不是 IRQ catchup 本身——它只是"果"——真正的"因"是同一母机上那台问题子机的 Split Lock，持续消耗物理 CPU 资源，间接把我们的子机也拖下了水。

Split Lock 是什么 ##

x86 架构下，当一条原子指令（比如 lock cmpxchg）操作的内存地址跨越了两个 Cache Line（缓存行）边界时，CPU 无法在缓存内部保证原子性，只能锁住整个内存总线来完成操作。

正常的一次加法指令可能只消耗不到 10 个时钟周期。而一次 Split Lock 锁住总线的时间可以达到 1000 个时钟周期——是正常操作的 100 倍以上。

在这 1000 个周期里，其他 CPU 核心无法访问内存。不是"慢了一点"，是整个总线的使用权被独占。

常见的触发场景：

  • 数据结构设计不当，原子操作的字段刚好跨了 64 字节边界
  • 内存未对齐，比如在一个非 4 字节对齐的地址上做 lock cmpxchg
  • 编译器或运行时没有做 cache line padding

ps: Cache Line 在 x86 上通常是 64 字节。一个 8 字节的原子变量如果起始地址是 60（mod 64），就会跨越两个 Cache Line，触发 Split Lock。

一个问题子机如何拖垮整台母机 ##

在 KVM 虚拟化环境中，vCPU 本质上是 QEMU 的线程。当子机内的某个进程触发 Split Lock 时：

  1. CPU 硬件检测到 Split Lock → 触发 #AC（Alignment Check）异常，或直接导致严重的总线争抢
  2. KVM 介入 → 内核检测到 Split Lock 发生在 vCPU 上下文中，通知 QEMU
  3. QEMU 的 vCPU 线程主动休眠 → 这是 KVM 的保护机制，防止 Split Lock 持续消耗物理 CPU

但问题是：vCPU 休眠意味着这个子机在这段时间内失去 CPU 时间片。母机调度器会尝试给它重新分配时间片，子机被调度回来后又可能再次触发 Split Lock，再次被休眠——这是一个死循环。每次调度-休眠的切换都伴随着大量的上下文切换开销和中断处理。

更关键的是：Split Lock 发生时总线已经被锁了一段时间。在这段时间内，母机上的其他子机——包括我们业务所在的那台——也无法访问内存。一台问题子机，通过总线锁，直接影响到了整台物理机的所有租户。

这就是虚拟化环境的连锁效应：一个子机的不合理内存访问 → 总线争抢 → 其他子机性能下降 → 母机整体不可靠。

为什么 IRQ 也会飙升 ##

这是排查中最困惑的一点——没有哪个设备的中断量异常，但 IRQ 总量就是飙上去了。

一种可能的解释是：被 Split Lock 阻塞期间，设备（网卡、磁盘等）继续产生中断，但 CPU 忙于处理总线锁无法响应。等总线锁释放后，积压的中断集中涌入——这就是前面提到的"IRQ catchup"。atop 看到的 IRQ 飙升是结果，不是原因。

ps: 这里的因果链条值得注意：Split Lock → CPU 卡在总线锁 → 中断积压 → 锁释放后 IRQ 集中处理 → atop 看到 CPU + IRQ 双高。如果只看监控就冲向中断子系统，排查方向就偏了。

能怎么办 ##

Split Lock 问题的根因在应用程序：内存对齐没做好，或者数据结构布局不合理。最直接的修复是：

  • 让关键原子变量对齐到 64 字节边界（__attribute__((aligned(64))) 或 Go 里的 [64]byte padding）
  • 调整数据结构，避免一个原子字段跨 cache line
  • 使用现代 CPU 提供的 Split Lock Detection 功能（内核参数 split_lock_detect=），在检测到时直接 SIGBUS 或 warn

但在云环境里，作为子机的使用者，我们改不了母机上其他租户的代码。能做的事情很有限：

  • 迁移：把业务迁到没有问题子机的母机上。但这依赖于云平台的支持和调度策略
  • 降级：临时降级非关键服务，减少对 CPU 的竞争
  • 施压：推动云平台侧排查并隔离问题子机

复盘 ##

这件事暴露了几个层面的问题：

技术层面：Split Lock 在 x86 上是一个被很多人忽视的性能杀手。它不像死锁那样会导致服务完全不可用，而是一种"慢性病"——性能会周期性恶化，很难快速定位。理解 cache line 对齐、原子操作的硬件实现、KVM 的 Split Lock 处理机制，对排查这类问题至关重要。

云平台层面：腾讯云团队对 Split Lock 问题是知情的，但响应速度跟不上业务的需要。这可以理解——定位问题子机、联系对应业务方配合修复、或者迁移，每一步都涉及跨团队沟通。但作为受影响的租户，等不起。

业务层面：问题子机所属的业务方没有足够强的意愿去修复——可能是没意识到严重性，可能是修复成本高，可能是排期排不上。一个子机的问题，通过母机的共享资源放大后，影响到了完全无关的其他业务。

最后我们只能推动迁移，才算绕过了这个问题。但本质上的问题没有解决——只要母机上还存在不合理的 Split Lock 访问，类似的事就还会发生。

Split Lock 的排查思路值得记住：CPU 飙高 + IRQ 突增，先看有没有跨 cache line 的原子操作，别一头扎进中断子系统。
