云计算领域速查手册 #

面向架构师的结构化参考，覆盖从虚拟化到云原生的核心技术概念、对比与选型决策。

1. 概念层次与演进关系 ##

物理机（Bare Metal）
  └─ 虚拟化（Virtualization）── 资源抽象，提高利用率
       └─ 云计算（Cloud Computing）── 服务化、按需付费的商业模式
            ├─ 传统云上应用（Lift & Shift）
            └─ 云原生（Cloud Native）── 为云设计的架构方法论
层次核心问题代表技术
虚拟化如何把一台物理机当多台用？KVM, VMware, Hyper-V
云计算如何以服务形式交付 IT 资源？AWS, Azure, OpenStack
云原生如何最大化利用云的优势？Docker, Kubernetes, Serverless

类比：虚拟化 = 一户多本房产证；云计算 = 连锁酒店租赁平台；云原生 = 为酒店生活方式设计的全新建筑理念。

2. 服务模型 ##

2.1 模型总览 ##

模型全称你管什么云管什么典型产品
IaaSInfrastructure as a Service应用、数据、运行时、OS虚拟化、服务器、存储、网络AWS EC2, 阿里云 ECS, OpenStack
CaaSContainer as a Service应用、数据容器运行时、编排、基础设施GKE, AKS, EKS, 阿里云 ACK
PaaSPlatform as a Service应用代码、数据OS、运行时、中间件、基础设施Heroku, Google App Engine, OpenShift
FaaSFunction as a Service函数代码全部运行时和基础设施AWS Lambda, 阿里云函数计算
SaaSSoftware as a Service配置和数据一切Gmail, Salesforce, 钉钉

2.2 选型决策指南 ##

是否需要完全控制基础设施？
├─ 是 → 需要自定义 OS/网络/存储？ → IaaS（ECS/EC2）
└─ 否 → 是否需要长期运行的服务？
         ├─ 是 → 需要容器编排？ → CaaS（K8s 托管集群）
         │         └─ 不需要 → PaaS（Heroku/OpenShift）
         └─ 否 → 事件驱动/短期任务？ → FaaS（Lambda）

关键权衡：控制力 vs 运维负担

高控制力 ◄──────────────────────────────► 低运维负担
  IaaS          CaaS          PaaS         FaaS        SaaS
  自建 K8s      托管 K8s      Heroku       Lambda      Gmail

3. 虚拟化技术 ##

3.1 Hypervisor 类型 ##

类型架构特点代表
Type 1（裸机型）直接运行在硬件上高性能、低开销KVM, VMware ESXi, Hyper-V
Type 2（宿主型）运行在宿主 OS 上灵活、性能较低VirtualBox, VMware Workstation

3.2 虚拟机 vs 容器 ##

维度虚拟机（VM）容器（Container）
隔离层级Hypervisor（硬件级）OS 内核（进程级）
启动速度分钟级秒级
资源开销每个 VM 一个完整 OS共享宿主机内核（MB 级）
镜像大小GB 级MB 级
迁移性受限于 Hypervisor高度可移植
安全性强隔离共享内核，隔离较弱
适用场景多租户强隔离、传统应用微服务、CI/CD、弹性伸缩

3.3 容器技术栈 ##

组件代表技术职责
容器运行时containerd, CRI-O, runc实际运行容器
镜像构建Docker, Buildah, Kaniko构建和分发容器镜像
容器编排Kubernetes, Docker Swarm调度、扩缩容、服务发现
服务网格Istio, Linkerd, Consul服务间通信治理

4. 云原生技术栈（CNCF Landscape 简化版） ##

4.1 核心能力矩阵 ##

能力域技术成熟度
编排与调度Kubernetes事实标准
服务发现CoreDNS, Consul成熟
配置管理ConfigMap/Secret, Vault成熟
可观测性Prometheus + Grafana + Loki + Tempo成熟
API 网关Envoy, Kong, Nginx Ingress成熟
服务网格Istio, Linkerd快速增长
CI/CDTekton, ArgoCD, Flux成熟
镜像仓库Harbor, Docker Registry成熟
ServerlessKnative, OpenFaaS快速增长
GitOpsArgoCD, Flux主流采纳
IaCTerraform, Pulumi, Crossplane成熟
Policy as CodeOPA/Gatekeeper, Kyverno快速增长

4.2 Serverless 深入 ##

Serverless 不是"没有服务器"，而是开发者不需要感知服务器。

分类说明代表
FaaS函数级 Serverless，事件驱动AWS Lambda, Azure Functions
Serverless 容器无需管理节点的容器AWS Fargate, 阿里云 ECI
Serverless K8s无节点的 K8sGKE Autopilot, 阿里云 ASK
BaaS后端即服务（DB、Auth、Storage）Firebase, Supabase

适用场景：

  • ✅ 事件驱动、流量波峰波谷明显、短任务
  • ✅ API 后端、定时任务、数据处理管道
  • ❌ 长时间运行、延迟敏感、需要 GPU 的任务
  • ❌ 对底层网络/存储有特殊要求的场景

5. OpenStack 定位 ##

5.1 在技术栈中的位置 ##

┌─────────────────────────────────────────────┐
│               应用层                          │
├─────────────────────────────────────────────┤
│  Kubernetes / Docker Swarm    ← 容器编排     │
├─────────────────────────────────────────────┤
│  虚拟机（Nova） + 容器（Magnum/Zun）          │
├─────────────────────────────────────────────┤
│  OpenStack  ← IaaS 层                        │
│  ┌─────────────────────────────────────┐     │
│  │ Nova(计算) │ Neutron(网络) │ Cinder(块存储) │
│  │ Glance(镜像)│ Swift(对象存储)│ Keystone(认证)│
│  └─────────────────────────────────────┘     │
├─────────────────────────────────────────────┤
│  物理机 / Hypervisor（KVM 等）                 │
└─────────────────────────────────────────────┘

5.2 与其他技术的关系 ##

对比OpenStack公有云（AWS/Azure）Kubernetes
层级IaaSIaaS + PaaS + SaaSCaaS/编排
管理对象虚拟机、网络、存储全栈云服务容器工作负载
部署方式私有云公有云任意的 K8s 集群
典型组合提供底层基础设施给 K8s提供托管 K8s 服务运行在 OpenStack VM 或公有云上

关键理解：OpenStack 是 IaaS 层的开源私有云方案，K8s 是容器编排平台。K8s 的 Pod 可以运行在 OpenStack 创建的虚拟机上，也可以直接运行在物理机上。两者可以独立使用，也可以协同：OpenStack 管底层基础设施（VM、网络、存储），K8s 管上层容器工作负载。

6. 云部署模型 ##

模型说明优势挑战
公有云第三方提供商，共享基础设施弹性、按需、免运维成本不可控、供应商锁定
私有云自建数据中心，独占资源合规、安全、可控高 CAPEX、需运维
混合云私有云 + 公有云联动兼顾安全与弹性网络打通、数据一致性
多云使用多个公有云避免锁定、取长补短管理复杂、数据迁移成本

6.1 混合云关键考量 ##

  • 网络连通：专线/VPN 打通 VPC，统一服务发现
  • 数据一致性：多云数据同步延迟、脑裂处理
  • 统一管理层：Terraform/IaC 统一编排，可能是好的切入点
  • 成本优化：基准负载跑私有云，弹性负载溢出到公有云

7. 架构选型决策框架 ##

7.1 从需求到技术选择 ##

你的应用需要什么？
│
├─ 需要强隔离/合规要求？
│   └─ 是 → 虚拟机（IaaS） + 私有云/混合云
│
├─ 需要快速迭代、弹性伸缩？
│   ├─ 微服务架构 → 容器 + K8s（CaaS）
│   └─ 单体 → VM 或 PaaS
│
├─ 流量不确定/波峰波谷明显？
│   └─ 事件驱动 → Serverless/FaaS
│
└─ 已有大量 VMware/KVM 资产？
    └─ 考虑 OpenStack 构建私有云 + K8s 管理容器

7.2 技术组合推荐 ##

场景推荐组合
创业公司/小团队公有云 PaaS（Vercel/Railway）+ FaaS（Lambda）
中型互联网公司公有云 CaaS（托管 K8s）+ 部分 FaaS
大型企业混合云：自建 K8s + OpenStack 私有云 + 公有云弹性
金融/政务/合规强需求私有云（OpenStack/KVM）+ 容器化改造
传统企业上云IaaS（Lift & Shift）→ 逐步容器化 → 云原生改造

8. 关键概念速查 ##

概念一句话解释
弹性伸缩根据负载自动增减资源
不可变基础设施上线后不修改，直接替换（Immutable Infrastructure）
声明式 API描述"想要什么"而非"怎么做"，K8s 的核心理念
最终一致性分布式系统中，副本最终达到一致，但不保证实时
服务发现自动找到服务实例的地址和端口
蓝绿部署两套环境同时存在，切换流量来实现零停机部署
金丝雀发布逐步放量到小部分用户，验证后再全量
熔断器下游故障时自动切断请求，防止级联失败
IaCInfrastructure as Code，用代码定义和管理基础设施
GitOps以 Git 为唯一信源，声明式管理基础设施和应用部署
