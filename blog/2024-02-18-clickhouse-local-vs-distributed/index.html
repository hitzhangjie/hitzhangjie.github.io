ClickHouse 本地表与分布式表：读写路径和选型建议

Posted 2024-02-18 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

背景 #

刚接触 ClickHouse 时，本地表和分布式表的概念容易让人困惑。在单机数据库（如 MySQL）中，表就是表，没有本地/分布式之分。但在 ClickHouse 集群里，这两种表的读写行为差异很大，选择不当会直接影响性能和运维复杂度。

本地表 vs 分布式表 #

概念区分 ##

本地表：ClickHouse 中最基本的表类型。在某个节点上创建的表，数据只存储在该节点上，只能由该节点直接访问。通常使用 MergeTree 引擎。

分布式表：不直接存储数据，而是作为多个本地表的代理。一张分布式表背后对应所有节点的本地表，让你可以像操作单表一样跨节点读写。

-- 每个节点上创建本地表
CREATE TABLE local_events ON CLUSTER 'my_cluster' (
    event_time DateTime,
    user_id UInt64,
    event_type String
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_time)
ORDER BY (user_id, event_time);

-- 在任意节点上创建分布式表（代理）
CREATE TABLE distributed_events ON CLUSTER 'my_cluster' (
    event_time DateTime,
    user_id UInt64,
    event_type String
) ENGINE = Distributed(
    'my_cluster',          -- 集群名
    'default',             -- 数据库名
    'local_events',        -- 本地表名
    rand()                 -- 分片键
);

写入路径 ##

写入分布式表：

INSERT INTO distributed_events VALUES (...);

数据会被写入分布式表背后所有节点的本地表中。分片键（如 rand()）决定了数据落在哪个分片。简单来说，写入分布式表 = 自动分发写入。

写入本地表：

INSERT INTO local_events VALUES (...);

数据只会写入当前节点的本地表。你需要自行管理数据应该写入哪个节点——通常是在应用层按分片键路由。

查询路径 ##

查询分布式表：

SELECT count() FROM distributed_events
WHERE event_time >= today() - 7;

查询会被发送到所有相关节点，各节点在本地表上执行后，结果返回初始节点汇总。对用户透明，一行 SQL 搞定全集群查询。

查询本地表：

SELECT count() FROM local_events
WHERE event_time >= today() - 7;

只查询当前节点的数据。要查全集群数据，需要在应用层手动向每个节点发起查询并汇总结果。

为什么有人建议用本地表代替分布式表？ #

分布式表确实会增加集群内的网络流量：每次查询都需要在所有节点上执行，并将结果发送回初始节点。如果查询量大、结果集大，这会成为瓶颈。

使用本地表的优势：

  • 避免分布式查询的额外网络开销。
  • 查询性能更可预期（不会因为某个节点慢而拖累整体）。
  • 数据本地性更好，可以利用 ClickHouse 的本地优化。

使用本地表的代价：

  • 需要在应用层手动管理"查哪个节点"和"如何汇总"，增加代码复杂度。
  • 分片变更时需要同步更新应用路由逻辑。
  • 失去分布式表带来的透明查询体验。

怎么选？ #

场景建议
数据量不大，查询结果集可控用分布式表，简单方便
数据量大，但查询总是指向某个分片（如按租户分）本地表 + 应用层路由
大结果集的聚合查询本地表 + 手动汇总，避免分布式表的内网流量爆炸
写入量极高（每秒几十万行以上）本地表 + 应用层写入路由，避免 Distributed 引擎的写入放大
初期快速搭建分布式表，先跑起来再优化

实践中，很多团队的做法是：写入走本地表（应用层路由），查询走分布式表（开发方便）。这样两边的优势都能利用。当查询性能成为瓶颈时，再把高频查询逐步切换为直接查本地表。

总结 #

分布式表是 ClickHouse 提供的一个便利层——让集群操作像单机操作一样简单。但它不是免费的：网络开销、写入放大、查询汇聚都是隐含成本。理解本地表和分布式表的读写路径差异，有助于在便利性和性能之间做出合理取舍。经验法则是：先用分布式表快速迭代，遇到性能瓶颈时再优化为本地表直连。
