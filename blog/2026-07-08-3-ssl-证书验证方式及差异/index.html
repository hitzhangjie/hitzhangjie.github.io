证书校验：OpenSSL 与 Go 的行为差异 #

本文档记录对 etcd 集群 TLS 证书链的深入分析，重点说明 为何 openssl verify 报错但 etcd 集群正常运行，以及对 OpenSSL / Go 未来版本兼容性 的顾虑与应对建议。

1. 问题背景 ##

使用以下命令校验时失败：

openssl verify -CAfile new-ca.pem etcd.pem

输出：

C=CN, ST=shenzhen, L=shenzhen, O=etcd, OU=System, CN=etcd
error 18 at 0 depth lookup: self-signed certificate
error etcd.pem: verification failed

但 etcd 集群在如下配置下 长期稳定运行：

trusted-ca-file: /etc/etcd/ssl/new-ca.pem
cert-file: /etc/etcd/ssl/etcd.pem
key-file: /etc/etcd/ssl/etcd-key.pem
client-cert-auth: true

初看像是「配置与 PKI 未对齐」，实测与线上行为表明：集群是正确的，是校验工具的判断逻辑不同。

2. 真实 PKI 关系（非自签名） ##

2.1 易被误判的原因：DN 完全相同 ##

三份证书的 Distinguished Name 完全一致：

Subject / Issuer = C=CN, ST=shenzhen, L=shenzhen, O=etcd, OU=System, CN=etcd
文件Issuer == Subject公钥模数（与私钥配对）
etcd.pem是（字段相同）与 etcd-key.pem 配对
new-ca.pem是（CA 自签）与 ca-key.pem 配对
ca.pem是（CA 自签）与 ca-key.pem 配对（同 new-ca）

etcd.pem 的 Issuer 字段 与 Subject 字段 字符串相同，是因为 CA 证书也使用了同一个 DN，并不等同于「用 etcd-key.pem 给自己签了一张证」。

2.2 实际签发关系 ##

ca-key.pem ──签──> new-ca.pem / ca.pem     （CA 根证书）
ca-key.pem ──签──> etcd.pem                （节点叶子证书）
etcd-key.pem ←──配对──> etcd.pem 中的公钥  （TLS 握手加密用）
  • etcd.pem 内的公钥属于 节点密钥对（etcd-key.pem）
  • etcd.pem 这张证书本身的签名来自 CA 私钥（ca-key.pem）
  • trusted-ca-file: new-ca.pem 用于验证对端出示的 etcd.pem，职责正确，并非空转

2.3 Go 验证结果（与 etcd 行为一致） ##

使用 Go crypto/x509（etcd 底层 TLS 库）验证：

检查项结果
etcd.CheckSignatureFrom(new-ca.pem)✅ 通过
etcd.CheckSignatureFrom(etcd.pem)❌ 失败
etcd.Verify(Roots: new-ca.pem)✅ 链：etcd.pem → new-ca.pem
模拟 mTLS 双向握手（同配置）✅ 成功

结论：etcd.pem 在密码学上由 CA 签发，PKI 与 etcd 配置是对齐的。

3. 为何 OpenSSL 报错？ ##

3.1 error 18 的含义 ##

OpenSSL error 18: self-signed certificate 表示：在证书链 depth 0 处，发现 Issuer 与 Subject 相同，直接归类为自签名证书，停止继续向 -CAfile 中的 CA 寻找签发者并验签。

这是 OpenSSL 的启发式规则，而非密码学验签失败。

3.2 OpenSSL 与 Go 的流程对比 ##

OpenSSL verify:
  看到 etcd.pem 的 Issuer == Subject
    → 判定为自签名
    → error 18，不再尝试用 new-ca.pem 验签

Go crypto/x509:
  看到 Issuer DN 与信任池中 new-ca.pem 的 Subject 匹配
    → 执行 RSA 签名校验（CheckSignatureFrom）
    → 验签通过，建立证书链

3.3 这是 OpenSSL 版本 bug 吗？ ##

不是。 这是 OpenSSL 长期设计行为，跨版本基本一致：

OpenSSL 版本行为
1.0.2issuer==subject 视为自签；输出格式与 1.1 略有不同
1.1.x明确报 verification failed（被认为更正确）
3.x（本环境 3.6.2 实测）同样 error 18

因此，这主要是 OpenSSL 与 Go 的实现差异，而非 OpenSSL 各版本之间的不兼容。用不同版本的 OpenSSL 测试，大概率都会报 error 18。

尝试附加 -untrusted、证书链文件等方式，在本样本上仍无法让 openssl verify 通过——OpenSSL 在 depth 0 的短路判断优先于后续链构建。

4. 为何 etcd 集群能正常运行？ ##

etcd 使用 Go 标准库 crypto/tls + crypto/x509，而非调用 openssl verify。

在各节点共用同一份 etcd.pem + etcd-key.pem、且 trusted-ca-file 指向 new-ca.pem 时：

  1. TLS 加密：etcd-key.pem 参与握手，与证书公钥配对 ✅
  2. Peer mTLS：节点出示 etcd.pem，对端用 new-ca.pem 验证——Go 验签通过 ✅
  3. client-cert-auth: true：要求对端出示 CA 签发的证书——etcd.pem 满足 ✅

集群能跑不是侥幸，而是 Go 校验路径下 PKI 成立。

5. 对未来兼容性的顾虑 ##

5.1 OpenSSL 侧 ##

  • 各版本对「issuer==subject 即自签」的处理稳定
  • 风险不在于 OpenSSL 升级导致 etcd 挂掉（etcd 不依赖 OpenSSL verify）
  • 风险在于：用 openssl verify 做验收或监控时持续误报，造成不必要的告警或误判

5.2 Go 侧（更值得关注） ##

Go 有 Go 1 兼容性承诺，但明文保留 安全相关打破兼容性 的权利。crypto/x509 是历史上改动较多的包：

版本变更可能影响
Go 1.15不再用 CN 做主机名校验（需 SAN）缺 SAN 的老证书
Go 1.18默认禁用 SHA-1 证书企业内网老 CA（K8s 曾反馈，延长过渡期）
Go 1.24移除 x509sha1 GODEBUG 回退SHA-1 证书彻底不可用

改动方向几乎都是 收紧安全策略，而非随意放宽。

对当前 PKI 布局的潜在风险：

场景可能性影响
Go 收紧校验，对 issuer==subject 的叶子证不再尝试匹配信任池低～中可能影响 etcd 集群
Go 放松校验极低几乎不会
算法淘汰（如 SHA-1）本环境使用 SHA256目前无影响

etcd 升级时内含的 Go 版本会变化，因此 关注 etcd 版本升级 > 关注系统 OpenSSL 版本升级。

Go 对破坏性变更通常会：

  • 在 release note 中提前说明
  • 提供 GODEBUG 过渡期（参考 SHA-1 案例）
  • 给企业内网留出迁移窗口

5.3 根本风险点 ##

当前布局的特殊性在于：CA 证书与节点证书使用了相同的 DN。

这导致：

  • OpenSSL 误判为自签名
  • 依赖 Go 更完整的链构建逻辑才能正确验证
  • 若未来 Go 改为与 OpenSSL 类似的短路策略，存在理论风险

6. 建议的验收方式 ##

不要单独依赖 openssl verify 作为证书链健康的判断标准。

6.1 推荐做法 ##

# 1. 确认节点证书与私钥配对
openssl x509 -in etcd.pem -noout -modulus | openssl md5
openssl rsa  -in etcd-key.pem -noout -modulus | openssl md5
# 两者 MD5 应一致

# 2. 用 etcd 实际使用的 TLS 路径验证（最直接）
etcdctl endpoint health \
  --endpoints=https://<node-ip>:2379 \
  --cacert=/etc/etcd/ssl/new-ca.pem \
  --cert=/etc/etcd/ssl/etcd.pem \
  --key=/etc/etcd/ssl/etcd-key.pem

# 3. 用 Go 验证（接近 etcd 行为）
# 见下方「附录：Go 验证脚本」

6.2 不推荐作为唯一依据 ##

# 在本 PKI 结构下会持续误报，不代表集群有问题
openssl verify -CAfile new-ca.pem etcd.pem

7. 长期改进建议 ##

7.1 区分 CA 与节点证书的 DN（推荐） ##

签发时使用不同的 Subject，消除工具歧义：

CA 证书：  CN=etcd-ca,  O=etcd, OU=System, ...
节点证书： CN=etcd,     O=etcd, OU=System, ...

这样 OpenSSL 与 Go 的结论一致，不依赖实现细节，后续换工具、升级 etcd/Go 都更稳妥。

7.2 保持现有布局时 ##

  • 文档中明确记录：openssl verify 在此 DN 结构下会误报
  • 证书续签/轮换验收以 etcdctl 连通性 或 Go x509 验证 为准
  • 升级 etcd 版本时关注 release note 中 crypto/x509 / crypto/tls 变更

7.3 私钥与证书管理 ##

  • ca-key.pem 不下发至节点（仅证书管理机使用）—— 当前做法正确
  • etcd-key.pem 权限建议 600
  • 轮换 new-ca.pem 时同步更新所有节点的 trusted-ca-file

8. 校验结果汇总（修正版） ##

检查项openssl verifyGo crypto/x509（etcd 实际）
etcd.pem ↔ etcd-key.pem 密钥匹配N/A（不涉及）✅ 通过
etcd.pem 由 new-ca.pem / CA 签发❌ error 18（误报）✅ 通过
new-ca.pem ↔ ca.pem 公钥相同N/A✅ 相同
mTLS 双向握手N/A✅ 通过
etcd 集群实际运行N/A✅ 正常

附录 A：Go 验证脚本 ##

将以下脚本保存为 verify_chain.go，在 ssl/ 目录下执行 go run verify_chain.go：

package main

import (
	"crypto/tls"
	"crypto/x509"
	"encoding/pem"
	"fmt"
	"os"
)

func loadCert(path string) *x509.Certificate {
	b, err := os.ReadFile(path)
	if err != nil {
		panic(err)
	}
	block, _ := pem.Decode(b)
	c, err := x509.ParseCertificate(block.Bytes)
	if err != nil {
		panic(err)
	}
	return c
}

func main() {
	etcd := loadCert("etcd.pem")
	caPEM, _ := os.ReadFile("new-ca.pem")
	pool := x509.NewCertPool()
	pool.AppendCertsFromPEM(caPEM)

	// 签名校验
	if err := etcd.CheckSignatureFrom(loadCert("new-ca.pem")); err != nil {
		fmt.Printf("❌ 签名校验失败: %v\n", err)
	} else {
		fmt.Println("✅ etcd.pem 由 new-ca.pem 对应 CA 私钥签发")
	}

	// 链验证
	if _, err := etcd.Verify(x509.VerifyOptions{Roots: pool}); err != nil {
		fmt.Printf("❌ 链验证失败: %v\n", err)
	} else {
		fmt.Println("✅ 证书链验证通过（Go / etcd 行为）")
	}

	// mTLS 握手模拟
	serverCert, _ := tls.LoadX509KeyPair("etcd.pem", "etcd-key.pem")
	sCfg := &tls.Config{
		Certificates: []tls.Certificate{serverCert},
		ClientAuth:   tls.RequireAndVerifyClientCert,
		ClientCAs:    pool,
		MinVersion:   tls.VersionTLS12,
	}
	cCfg := &tls.Config{
		Certificates: []tls.Certificate{serverCert},
		RootCAs:      pool,
		MinVersion:   tls.VersionTLS12,
	}
	ln, err := tls.Listen("tcp", "127.0.0.1:0", sCfg)
	if err != nil {
		panic(err)
	}
	addr := ln.Addr().String()
	errCh := make(chan error, 1)
	go func() {
		c, err := tls.Dial("tcp", addr, cCfg)
		if err != nil {
			errCh <- err
			return
		}
		c.Close()
		errCh <- nil
	}()
	sconn, _ := ln.Accept()
	if err := sconn.(*tls.Conn).Handshake(); err != nil {
		fmt.Printf("❌ mTLS 服务端握手失败: %v\n", err)
	} else if err := <-errCh; err != nil {
		fmt.Printf("❌ mTLS 客户端握手失败: %v\n", err)
	} else {
		fmt.Println("✅ mTLS 双向握手模拟通过")
	}
	sconn.Close()
	ln.Close()
}

问题结论
openssl verify 报错是 PKI 坏了吗？不是，是 OpenSSL 对 issuer==subject 的短路判断
etcd 为什么能跑？etcd 用 Go 验签，能正确建立 etcd.pem → new-ca.pem 链
OpenSSL 版本升级会影响 etcd 吗？不会（etcd 不依赖 OpenSSL verify）
