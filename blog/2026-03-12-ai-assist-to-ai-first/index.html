从 AI Assist 到 AI First：可验证循环驱动的范式升级

Posted 2026-03-12 00:00 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 ##

过去一年多，AI coding 工具的能力提升是肉眼可见的。从最早的代码补全，到现在的 Agent 自主写代码、修 bug、做重构——工具越来越强，写代码越来越快。但如果只停留在"用 AI 写得快"这个层面，团队很快就会撞上一堵墙：代码量膨胀了，质量却没有跟上。

我一直在思考一个问题：同样是使用 AI coding 工具，为什么有些团队的体验是"AI 帮我们写了很多代码，但也制造了更多 bug"，而另一些团队却能做到"AI 帮我们既提效又提质"？

答案在于一个被很多人忽视的东西：可验证循环（verifiable loop）。

AI Assist 与 AI First 的本质区别 ##

在讨论可验证循环之前，先定义两个模式：

AI Assist 模式 ##

这是目前大多数团队的状态：

人类写需求 → AI 生成代码 → 人类手动验证 → 人类修改 → 合入

AI 充当的是"高级代码补全器"，写得快，但验证的责任全在人身上。人类必须仔细阅读每一行 AI 生成的代码，确保正确性、安全性、性能都过关。AI 提效了编码环节，但验证环节的成本并没有降低——它只是从"自己写"变成了"审查 AI 写的"。

AI First 模式 ##

这是我们想要达到的状态：

人类确认需求 → AI 生成方案 + 代码 → 自动化验证 → 人类聚焦审查关键决策 → 合入

在 AI First 模式下，验证环节被大幅自动化了。AI 生成的代码会先经过自动化测试、静态分析、安全扫描、性能基准等层层检查，只有通过了这些检查的代码，才进入人类审查环节。人类的注意力不再消耗在"这段代码有没有低级错误"上，而是聚焦在"这个方案的设计是否合理"“业务逻辑是否正确"这些更高层次的问题上。

一张表看清两种模式的区别 ##

维度AI AssistAI First
编码AI 辅助，人主导AI 主导，人审查
验证人手动验证自动化验证为主
人类注意力逐行检查代码质量聚焦架构和业务决策
AI 的定位高级补全器可靠的协作者
信任基础每次都要验证通过规则和自动化建立信任
规模化瓶颈人类审查能力自动化验证能力

可验证循环：从 Assist 到 First 的桥梁 ##

两种模式之间最关键的差异是什么？不是 AI 模型的能力，而是验证环节的自动化程度。

核心逻辑很简单：

AI 写代码很快，但如果产生的每行代码都需要人眼确认正确性，那么：
  人的审查速度 = 系统的吞吐上限

AI 写代码很快，如果大部分验证是自动完成的，那么：
  自动化验证的速度 = 系统的吞吐上限

前者受限于人的精力，后者可以横向扩展。这就是可验证循环的力量。

什么是可验证循环？ ##

一个典型的可验证循环包含以下层次：

                    ┌─ AI 生成代码 ─┐
                    │               │
                    ▼               │
           ┌──────────────────┐    │
           │  L0: 格式检查      │    │
           │  gofmt, goimports │    │
           └───────┬──────────┘    │
                   ▼               │
           ┌──────────────────┐    │
           │  L1: 静态分析      │    │
           │  go vet, lint     │    │
           └───────┬──────────┘    │
                   ▼               │
           ┌──────────────────┐    │
           │  L2: 单元测试      │    │
           │  go test -race    │    │
           └───────┬──────────┘    │
                   ▼               │
           ┌──────────────────┐    │
           │  L3: 安全扫描      │    │
           │  go vulncheck     │    │
           └───────┬──────────┘    │
                   ▼               │
           ┌──────────────────┐    │
           │  L4: 人类审查      │    │
           │  架构 + 业务逻辑   │    │
           └───────┬──────────┘    │
                   ▼               │
              合入主干 ────────────┘
              (反馈驱动 AI 改进)

每一层都是一道防线。L0-L3 是确定性规则——它们不依赖人的主观判断，可以自动化执行。L4 是人工判断——聚焦在自动化覆盖不了的高层次问题上。

这个循环之所以是"可验证"的，核心在于：每一次 AI 生成代码的变更，都有一条清晰的、可自动验证的真相路径。代码能不能格式化通过？lint 过不过？测试过不过？这些都是确切的、二值的、可自动判断的事实。

为什么 CI 自动化测试是反馈价值最大的环节？ ##

在这个可验证循环中，如果只能选一个环节来重点投入，我会毫不犹豫地选择 自动化测试。

原因很简单：

  1. 格式检查和 lint 只能发现问题，不能验证正确性。代码格式完美、lint 全过的代码，逻辑可能完全错误。
  2. 安全扫描只能找已知漏洞模式，覆盖范围有限。
  3. 只有自动化测试真正验证了"代码按预期工作”——它直接回答了"这个变更对不对"的问题。
  4. 测试是最快的质量反馈。一次 go test -race 跑下来，几分钟就能告诉你：你的代码有没有数据竞争，核心逻辑有没有被破坏。

在 AI coding 的背景下，自动化测试的价值被进一步放大：AI 生成的代码可能在风格上无可挑剔，但在边界条件、错误处理、并发安全上可能存在问题。没有自动化测试的 AI coding，就像没有刹车的赛车——速度越快，风险越大。

一个具体的循环：TDD + AI ##

把这个循环具象化到日常开发中，TDD 工作流是最自然的形式：

  1. Red：人定义接口和测试用例（这是人类在设定"什么是对的"的标准）
  2. Green：AI 生成最小实现，使测试通过
  3. Refactor：AI 在测试保护下优化代码结构和性能
  4. Verify：go test -race -cover + golangci-lint 一键验证

在这个循环中，测试用例就是人类给 AI 立的"可验证的规矩"。AI 可以自由发挥如何实现，但必须通过这些测试。人不用审查每一行代码，只需要审查测试用例是否覆盖了正确的场景，以及架构决策是否合理。

存量项目中的现实困境 ##

理论很美好，但现实很骨感。大多数存量项目的状态是：

  • 测试覆盖率低（甚至没有）
  • 代码可测试性差（全局变量、硬编码依赖、缺少接口抽象）
  • CI 流水线只有编译检查
  • 没有任何自动化质量门禁

在这样的项目中，可验证循环根本跑不起来——因为"验证"这一步本身就是缺失的。

先止血：从 CI 自动化测试开始 ##

面对这种情况，我的建议是：不要试图一步到位，先从 CI 中的自动化测试开始。

具体路径：

  1. 先铺测试基础设施：把 go test -race -cover 接入 CI，哪怕覆盖率是 0% 也要先跑起来
  2. 设定覆盖率底线：“新增代码覆盖率必须 ≥ 现有水平”——至少不让情况恶化
  3. 新功能强制测试：任何新功能、新接口，必须带测试才能合入
  4. 重构改善可测试性：每次修改旧代码时，顺手把硬编码依赖改为接口注入
  5. 逐步提高覆盖率阈值：从 30% → 50% → 70%，循序渐进

这叫止血策略：先防止新的坏代码进入，再逐步清理存量债。

持续重构是唯一出路 ##

可测试性不会从天而降。在没有测试保护的存量代码上加测试，往往需要先重构。而重构又有风险。这就形成了一个死锁。

打破这个死锁的方法是：在每次功能变更时，顺手改一点。

修改旧模块的流程：
  1. 先写一个"特征测试"（characterization test）——把当前行为固化下来
  2. 在测试保护下做必要的重构（提取接口、注入依赖）
  3. 实现新功能
  4. 补充新功能的测试

这样每次变更都让代码的可测试性好一点，而不是在"一次性重构"中赌博。

AI First 不等于无人介入 ##

一个常见的误解是：AI First 就是 AI 全权负责，人只需要最后验收。这不是真的。

AI First 中人类仍然不可或缺，只是介入的位置变了。人从"审查每一行代码"的质检员，变成了"在关键决策点把关"的架构师。

人类必须介入的关键节点 ##

阶段人类角色AI 角色
需求评审判断优先级、澄清业务意图辅助拆解、识别歧义点
方案确认评估架构合理性、技术选型生成候选方案、对比优劣
测试用例评审确认覆盖了正确的场景生成测试用例、补充边界条件
代码审查审查架构和业务逻辑规范检查、性能分析
上线决策评估风险、决定时机生成检查清单、验证兼容性

最有趣的一点是：测试用例评审可能是 AI 时代最重要的 Review 环节。如果你认可"测试用例是给 AI 的可验证规矩"，那么测试用例的质量直接决定了 AI 输出代码的质量。糟糕的测试用例意味着糟糕的验证标准，AI 通过了测试也不代表代码写对了。

这需要什么样的团队能力？ ##

要搭建和维护这样一个可验证循环，团队需要具备一些关键能力：

资深工程师的新角色 ##

在 AI Assist 模式下，资深工程师的价值是"写出好的代码"。在 AI First 模式下，他们的价值变成了：

  1. 写规矩的能力：把工程经验编码为 AI 可理解、可执行的规则（Rules）
  2. 设计验证标准的能力：知道什么样的测试验证了什么，什么没验证到
  3. 判断 AI 输出质量的能力：在自动化检查通过后，仍然能判断代码是否"真的好"
  4. 搭建验证基础设施的能力：CI 流水线、质量门禁、可观测性体系

初级工程师的新成长路径 ##

初级工程师不需要一夜之间掌握所有技能。但在可验证循环的保护下，他们可以在实践中学习：

  • AI 写的代码自动经过了格式、lint、测试的检查，初级工程师 review 时看到的已经是"相对规范"的代码
  • 测试失败时，失败信息本身就是很好的学习材料——它告诉开发者"为什么这样写不对"
  • 资深工程师的 Rules 会自动约束 AI 的行为，初级工程师甚至在不知道规则存在的情况下，就已经在遵循规则了

这就是"修渠引水"而非"拔苗助长"。

总结 ##

                    ┌──────────────────┐
                    │   可验证循环       │
                    │                   │
  AI Assist ──────▶ │ L0 格式检查       │ ──────▶ AI First
                    │ L1 静态分析       │
  人验证一切         │ L2 自动化测试     │         自动化验证为主
  AI 辅助编码        │ L3 安全扫描      │         AI 主导编码
                    │ L4 人工决策      │         人把关关键节点
                    │                   │
                    │ CI 快速反馈      │
                    └──────────────────┘

几点核心认知：

  1. 可验证循环是从 AI Assist 进化到 AI First 的唯一桥梁。没有它，AI 越快，风险越大。
  2. CI 中的自动化测试是验证体系中最有价值的一环——它直接回答"代码对不对"的问题。
  3. 存量项目的验证基础设施不会从天而降，需要先止血（不让情况恶化）+ 持续重构（每次改一点）。
  4. AI First 不是取代人，而是让人从重复性验证中解放出来，聚焦在需求理解、方案决策、架构设计这些 AI 不擅长的高层次问题上。
  5. 测试用例评审可能成为 AI 时代最重要的 Review 环节——因为测试用例就是给 AI 立的"可验证的规矩"。
