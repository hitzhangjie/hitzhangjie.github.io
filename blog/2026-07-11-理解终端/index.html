终端、TTY、PTY、Console：把这些概念一次理清

Posted 2026-07-11 17:05 +0800 by ZhangJie ‐ 3 min read

分享:  

问题背景 ##

写 asciinema 工作原理 时，我再次遇到了 Terminal、Console、TTY、PTY、master、slave 和 isatty()。这些词经常一起出现，却分属硬件历史、内核抽象和用户态程序三个层次。单独看每个定义并不难，把它们放进同一条数据链路时却很容易混淆。

日常开发中，我们接触最多的是以下几类对象：

  1. 终端模拟器：GNOME Terminal、Windows Terminal、iTerm2，以及编辑器中的终端面板；
  2. Shell：Bash、Zsh、Fish；
  3. Shell 内置命令：cd、export、alias；
  4. 外部程序：ls、cat、git、vim。

它们都出现在同一个窗口里，但并不属于同一层。本文从一条主线出发：键盘输入如何到达 Shell，程序输出又如何回到屏幕。理解这条链路后，TTY 与 PTY 的区别、Console 的含义，以及 SSH、容器和重定向中的终端行为都会变得清晰。

全局视角：打开终端时发生了什么 ##

在桌面环境中打开一个终端窗口，典型链路如下：

 Keyboard                                            Screen
    │                                                   ▲
    ▼                                                   │
┌────────────────────────────────────────────────────────────┐
│ Terminal Emulator                                          │
│ - 将按键转换为字节                                          │
│ - 解释程序输出中的控制序列，绘制字符、颜色和光标               │
└─────────────────────────┬──────────────────────────────────┘
                          │ read/write
                    PTY master
                          ⇅
                    PTY slave  ← 内核 TTY 子系统
                          │
                 stdin / stdout / stderr
                          │
                       Shell
                          │ fork/exec
                          ▼
                    CLI programs

先记住四个结论：

  • 终端模拟器负责将按键转换为字节，并将程序输出解释、绘制到屏幕；它不解析 Shell 语法，也不知道 ls 的含义；
  • Shell负责读取和解析命令、建立管道与重定向，并通过作业控制管理前后台任务；
  • cd、export 等内置命令直接改变 Shell 自身状态，ls、git 等外部命令则通常由 Shell 创建子进程执行；
  • 终端模拟器和 Shell 通过内核提供的 PTY 通信：模拟器持有 master 端，Shell 及其启动的命令默认连接 slave 端。

“默认连接 slave”并不意味着标准流始终指向终端。Shell 建立管道或重定向后，命令看到的文件描述符也会随之改变：

ls           # stdout -> PTY slave，isatty(1) 为真
ls | cat     # ls 的 stdout -> pipe，isatty(1) 为假
ls > out.txt # ls 的 stdout -> file，isatty(1) 为假

因此，同一个命令在终端、管道和文件重定向中可能采用不同的输出格式。判断问题时，应分别确认 stdin、stdout 和 stderr 连接到了哪里。

这里最容易误解的是“终端”一词。它可能指历史上的硬件终端、内核中的 TTY 设备，也可能只是口语中的终端模拟器窗口。后文会逐层区分这些含义。

概念理解：Terminal、TTY、Console ##

Term 与 TTY：从早期硬件名称到内核抽象 ##

早期的 Terminal（终端） 是一种硬件：键盘输入通过串行线路发给主机，主机返回的字符则被打印在纸上或显示在屏幕上。Teletype（电传打字机）是其中的典型设备，TTY 这个缩写也由此而来。

Unix 沿用了这套名称，但 TTY 的含义已经从某一种硬件扩展为：

内核为字符终端提供的统一设备接口和处理机制。

Linux 中的串口 /dev/ttyS0、虚拟控制台 /dev/tty1 和伪终端 slave /dev/pts/3，底层实现不同，却都接入了 TTY 子系统。该子系统不仅传递字节，还提供了终端特有的能力，例如：

  • 行规程（line discipline）：规范模式下缓存一行输入，并处理退格等字符；
  • 特殊字符与信号：例如将 Ctrl+C 转换为发给前台进程组的 SIGINT；
  • 终端属性：回显、规范模式、波特率等，可通过 termios 配置；
  • 窗口大小：全屏 TUI 可以通过 ioctl 获取行数和列数；
  • 作业控制：配合会话、进程组和控制终端管理前后台任务。

程序可以使用 isatty() 判断一个文件描述符是否连接到终端设备：

if (isatty(STDOUT_FILENO)) {
    // stdout 连接到 TTY，可以采用适合交互显示的输出方式
}

许多程序据此决定是否输出颜色、进度条或分栏布局。isatty() 不区分物理终端、虚拟控制台和伪终端；只要文件描述符指向内核认可的 TTY 设备，就会返回真。

需要注意，TERM 环境变量不由 TTY 子系统自动产生。它通常由终端模拟器、登录程序或远程登录工具设置，用来告诉应用当前终端支持哪套控制序列。

Console：强调用途，而非新的设备体系 ##

Console（控制台） 最初指主机旁用于启动、管理和排障的主控设备。今天它仍强调“系统管理入口”这一用途，而不是某种独立于 TTY 的设备类型。

Linux 中常见的相关概念如下：

名称常见设备或配置主要用途
System console/dev/console、内核参数 console=接收内核消息和启动阶段输出
Virtual console/dev/tty1、/dev/tty2 等本机全屏文本会话
Serial console/dev/ttyS0 等串口 TTY服务器、虚拟机和嵌入式设备的远程管理

一台机器可以通过 console= 指定一个或多个系统控制台。例如，串口 TTY 被指定为 console 后，它既是 TTY 设备，也承担 console 的职责。

因此，两者关注的问题不同：

  • TTY 回答“程序通过什么终端设备进行字符 I/O”；
  • Console 回答“哪个入口承担系统控制和诊断职责”。

概念理解：软件方案PTY，连接“终端前端+会话程序” ##

桌面终端没有连接真实串口，却仍要让 Bash、Vim 等程序看到完整的 TTY 语义。为此，内核提供了 PTY（Pseudo Terminal，伪终端）。

PTY 总是成对出现：

端点常见持有者作用
master终端模拟器、sshd、tmux、asciinema注入输入，读取会话输出
slaveShell 及其启动的命令表现为普通 TTY 设备

以输入 ls 为例，数据往返过程如下：

按键 "ls<Enter>"
      │
      ▼
终端模拟器 ──write──> PTY master ⇄ PTY slave ──read──> Shell
终端模拟器 <──read─── PTY master ⇄ PTY slave <──write── 程序输出
      │
      ▼
解释控制序列并绘制到窗口

slave 端接入标准 TTY 子系统，因此连接它的进程可以使用行规程、信号、窗口大小和作业控制。对 Shell 和应用程序而言，/dev/pts/3 就是一台终端；它们通常不需要知道另一端是 GUI 程序、sshd 还是会话录制器。

master 端则像这台“虚拟终端”的控制接口。持有 master 的程序既能看到 slave 发出的字节，也能向 slave 注入输入。这正是 tmux、script(1) 和 asciinema 能够转发或记录终端会话的基础。

Linux 的 Unix98 PTY 通常将 slave 设备暴露为 /dev/pts/N。本文沿用 Linux 文档中常见的 master/slave 叫法；部分资料也会使用 primary/replica 等名称。

应用实践：如何创建PTY 与其他 TTY ##

打开桌面终端：按需分配一对 PTY ##

打开一个新的终端窗口或标签页时，终端模拟器通常会完成以下操作。实际程序可能通过 openpty()、forkpty() 或更高层库封装这些步骤。

1. open("/dev/ptmx")
   └─ 获得 PTY master，内核为它分配对应的 slave

2. grantpt() / unlockpt()
   └─ 设置并解锁 slave

3. ptsname()
   └─ 获得 slave 路径，例如 /dev/pts/3

4. fork()
   ├─ 子进程：
   │    setsid()                 创建新会话
   │    打开 slave
   │    ioctl(TIOCSCTTY)         将 slave 设为控制终端
   │    dup2(..., 0/1/2)         将三个标准流连接到 slave
   │    exec("bash")             启动 Shell
   │
   └─ 父进程：
        持有 master，在窗口与 PTY 之间转发数据

其中，“控制终端”不是第四种终端设备，而是会话与某个 TTY 建立的关系。交互式 Shell 依靠这层关系进行作业控制：终端生成的信号会发送给前台进程组，后台进程直接读终端也可能被暂停。

可以在当前终端中查看自己连接的设备：

$ tty
/dev/pts/3

$ test -t 1 && echo "stdout is a tty"
stdout is a tty

每创建一个终端标签页、tmux pane，或一次请求 PTY 的 SSH 会话，通常都会分配一对新的 PTY。相关文件描述符全部关闭后，这对 PTY 即可被内核回收。

SSH 登录时，本地终端模拟器管理本地 PTY，服务端的 sshd 则另外分配一对远端 PTY；两者通过 SSH 协议传递数据和窗口尺寸，并不是同一个设备。使用 ssh -T 禁止分配远端 PTY 后，远端程序的 isatty() 通常返回假。

docker exec -it 和 kubectl exec -it 中的 -t 也表示为进程分配伪终端；-i 则表示保持标准输入打开。两者含义相关，但并不相同。

切换虚拟控制台：使用已有的 TTY ##

在支持 Linux 虚拟控制台的本机环境中，按 Ctrl+Alt+F3 通常会切换到 /dev/tty3。这不是临时创建一对 PTY，而是把键盘和显示切换到内核管理的另一个虚拟控制台。

keyboard / display
        │
        ▼
Linux virtual console (/dev/tty3)
        │
      getty
        │
      login
        │
      shell

getty 打开相应 TTY 并显示登录提示；认证完成后，login 启动 Shell，Shell 的标准流直接连接 /dev/tty3。这条链路中没有 PTY master，也没有终端模拟器。

串口 TTY 类似：驱动提供 /dev/ttyS0、/dev/ttyUSB0 等设备，进程直接打开它们，不需要通过 /dev/ptmx 动态申请。

把两种路径放在一起比较：

虚拟控制台 /dev/tty3伪终端 /dev/pts/3
实现方式内核虚拟控制台内核提供的 master/slave 对
输入和显示由谁连接内核控制台与显示子系统持有 master 的用户态程序
是否按会话动态分配通常不是是
典型场景本机文本登录桌面终端、SSH、tmux、容器

“真实 TTY”并不是一个严格的内核分类，通常只是为了与 PTY 对比，泛指串口或虚拟控制台等非 PTY 终端。判断具体行为时，直接说明“串口 TTY”“虚拟控制台”或“PTY”会更准确。

本文总结 ##

把全文压缩为一条数据链路：

键盘
  → 终端模拟器
  → PTY master
  → PTY slave（TTY 语义）
  → Shell / 命令
  → 原路返回
  → 终端模拟器绘制到屏幕

再记住五个边界：

  1. Terminal 原本是硬件终端，现在也常被用来指终端模拟器窗口；
  2. TTY 是 Linux 对字符终端的设备抽象和处理机制；
  3. PTY 是成对出现的软件终端，master 连接终端前端，slave 面向 Shell 和应用；
  4. Console 描述系统控制和诊断用途，底层可以由某个 TTY 承担；
  5. Shell 解析并执行命令，终端模拟器只负责转发输入和绘制输出。

这样的话，我感觉大家已经算是系统性地理解了终端、TTY、PTY、Console 这几个概念，以及它们之间的关系。
