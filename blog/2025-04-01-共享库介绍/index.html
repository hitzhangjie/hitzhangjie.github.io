Introduction of Shared Libraries #

Introduction ##

Recently, while improving the content related to linkers and loaders in my ebook “Design and Implementation of a Go Language Debugger,” I also read the book “Linkers and Loaders” and gained a deeper understanding of how linkers and loaders work. I’d like to take this opportunity to share and review some interesting points within it.

The Big Picture ##

To improve code reuse, modular design, reduce redundant disk space usage, and optimize memory usage, the concept of shared libraries emerged. They can be strictly divided into two types: Static Shared Library and Dynamic Shared Library. The former uses static linking and static loading, while the latter uses dynamic linking, and can use both static and dynamic loading methods.

OK, This is a macro perspective of compiling, linking, loading, and running: When performing a compilation build operation, we can consider each source file as a compilation unit. After compiling and assembling, a series of object files (*.o files) are generated. Then, the linker links these object files and dependent library files (static or dynamic link libraries) to generate a complete executable program. Finally, when the program is executed, the loader loads it into memory, and then jumps to the program entry point for execution.

Shared Lirary Structure ##

Roughly, the structure of static shared library and dynamic shared library are as following. The static library includes the list of object files, while the dynamic shared library merges rather than including the list of object files.

Here’s an exmaple: add.c and minus.c.

add.c

#include <stdio.h>

int add(int a, int b) {
  printf("Adding %d and %d\n", a, b);
  return a + b;
}

minus.c

#include <stdio.h>

int minus(int a, int b) {
  printf("Subtracting %d from %d\n", a, b);
  return a - b;
}

Static Shared Library ##

we can create static shared library like this:

  1. Compile the source files into object files:

    gcc -c add.c -o add.o
gcc -c minus.c -o minus.o
    • -c: Tells gcc to compile the source file into an object file (.o). It doesn’t link.
    • -o: Specifies the output file name.

  2. Create the static library:

    ar rcs libmath.a add.o minus.o
    • ar: The archiver utility.
    • rcs: Options for ar:
      • r: Insert or replace existing object files.
      • c: Create the archive if it doesn’t exist.
      • s: Create an index (symbol table) in the archive. This is important for linking.
    • libmath.a: The name of the static library. The lib prefix is a common convention.

The structure of libmath.a will roughly be like this:

+-----------------------+
|   Static Library      |
|                       |
| +-------------------+ |
| |  add.o            | |
| +-------------------+ |
| |  minus.o          | |
| +-------------------+ |
|                       |
|   Metadata & Index    |
+-----------------------+

Dynamic Shared Library ##

We can create dynamic shared library like this:

  1. Compile the source files into object files with position-independent code (PIC):

    gcc -fPIC -c add.c -o add.o
gcc -fPIC -c minus.c -o minus.o
    • -fPIC: Generates position-independent code. This is essential for shared libraries. PIC allows the library to be loaded at any memory address.

  2. Create the dynamic library:

    gcc -shared -o libmath.so add.o minus.o
    • -shared: Tells gcc to create a shared library.
    • -o: Specifies the output file name. The .so extension is common on Linux. On macOS, it would be .dylib.

The structure of libmath.a will roughly be like this:

+------------------------+
|   Dynamic Library      |
|                        |
| +--------------------+ |
| |  .text section     | |  <-- Code segment (executable instructions)
| +--------------------+ |
| |  .data section     | |  <-- Initialized data segment
| +--------------------+ |
| |  .bss section      | |  <-- Uninitialized data segment
| +--------------------+ |
| |  .rodata section   | |  <-- Read-only data segment
| +--------------------+ |
| |  .got section      | |  <-- Global Offset Table
| +--------------------+ |
| |  .plt section      | |  <-- Procedure Linkage Table
| +--------------------+ |
|                        |
|   Metadata & Symbols   |  <-- Symbol table, relocation info, etc.
+------------------------+

Using the Libraries ##

Let’s create a simple program main.c that uses the functions from our libraries.

main.c

#include <stdio.h>

// Function prototypes (important!)
int add(int a, int b);
int minus(int a, int b);

int main() {
  int x = 10;
  int y = 5;

  int sum = add(x, y);
  printf("Sum: %d\n", sum);

  int difference = minus(x, y);
  printf("Difference: %d\n", difference);

  return 0;
}

  1. For Static Linking:

    gcc main.c -L. -lmath -o main
    • -L.: Tells the linker to search for libraries in the current directory (.).
    • -lmath: Tells the linker to link with the library named libmath.a. The lib prefix and the .a extension are omitted.
    • -o main: Specifies the output executable file name.

  2. For Dynamic Linking:

    gcc main.c -L. -lmath -o main

    The command is the same as for static linking. The difference is how the linker resolves the symbols at runtime.

Summary ##

This article briefly introduces the purpose of shared libraries and how to complete compilation and assembly using the GCC toolchain, how to create static and dynamic shared libraries, and how to link static and dynamic shared libraries. We also demonstrated the structural differences between static and dynamic shared libraries. This difference is not the result itself, but a necessary intermediate result for linkers and loaders to complete their work normally. We will continue to introduce in more detail in the following articles.
