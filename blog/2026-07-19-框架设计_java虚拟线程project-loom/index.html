Java 虚拟线程：Project Loom 与平台级「协程式」并发

Posted 2026-07-19 17:38 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

「Java 会不会原生支持协程？」这个问题，在 Kilim / Quasar 年代只能答：靠字节码曲线救国。大约十年后，答案变成了：

会，而且形态是虚拟线程（Virtual Threads），由 Project Loom 带入 JDK，并在 Java 21 转正。

它解决的问题与 Fiber 框架一致：别让一次阻塞 I/O 白白占死一条昂贵的 OS 线程；但实现路径不同——不是让业务给每个方法贴 @Suspendable 再织入，而是 JVM 把 Thread 的一种实现做成可挂起的轻量单元。

本文说明虚拟线程是什么、怎么跑、和旧方案差在哪，以及使用时要注意什么。

一、从平台线程到虚拟线程 #

1.1 两种 Thread ##

自 Loom 起，Java 里可以粗分为：

类型本质数量级直觉
平台线程（Platform Thread）对 OS 线程的薄封装，1:1受限于栈内存与调度，通常成百上千
虚拟线程（Virtual Thread）JVM 调度的轻量线程，M:N 映射到载体可达百万级任务（视内存与业务）

创建示例（Java 21+）：

// 方式一
Thread.startVirtualThread(() -> {
    // 业务逻辑：可直接写阻塞式 API
    var body = httpClient.send(request, BodyHandlers.ofString()).body();
    process(body);
});

// 方式二：Executor
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<String> f = executor.submit(() -> fetch());
    // ...
}

关键体验：代码仍可按同步风格写，不必拆成回调地狱，也不必全项目字节码增强。

1.2 载体线程（Carrier）与挂载（Mount） ##

调度关系可以记成：

大量虚拟线程  --mount/unmount-->  少量平台线程（carrier）  -->  OS 调度

生命周期示意：

  1. Mount：调度器把就绪的虚拟线程挂到某条 carrier 上执行；
  2. Running：在 carrier 上跑应用代码，与普通线程类似；
  3. 阻塞点：如网络 I/O、LockSupport.park、部分阻塞式 JDK API；
  4. Unmount：保存 continuation（执行状态），从 carrier 卸下；carrier 可去跑别的虚拟线程；
  5. 再次就绪：I/O 完成等事件后，再 mount（可能换一条 carrier）继续。

因此：阻塞虚拟线程 ≠ 阻塞 OS 线程（在可卸载的路径上）。这正是高并发服务器想要的。

二、底层在靠什么：Continuation #

虚拟线程的魔法词是 continuation（延续）：表示「从某程序点带着状态继续执行」的能力。

当虚拟线程在阻塞点 yield 时：

  • 与该虚拟线程相关的栈状态被保存在 堆上管理的结构 中（实现细节随 JDK 版本演进，常见描述是可增长的栈段 / stack chunk）；
  • carrier 的 OS 栈不被该任务长期占用；
  • unpark 后 continuation 被重新提交给调度器。

这与 Kilim/Quasar 的「在方法里插 pc + State」同属 可恢复计算 家族，但：

Kilim / Quasar虚拟线程
改造层级应用字节码 instrumentationJDK / JVM 运行时
APIFiber、@Suspendable标准 Thread / Executor
挂起点织入识别的 suspend 调用运行时识别的阻塞点
业务传染强（注解/签名向上传）弱（普通方法即可）
栈模型典型 无栈状态机轻量线程 + 可迁移栈状态（更接近有栈/半栈语义）

所以：虚拟线程 解决同类问题，但不宜简单说成「官方版 Quasar」。它是 一等公民的轻量线程，而不是用户态库模拟的无栈协程 DSL。

三、和「协程」一词怎么对齐 #

社区常把虚拟线程叫做 Java 的协程 或 goroutine-like。更精确一点：

  • 相似点：用户态 M:N 调度、阻塞不绑死 OS 线程、适合海量并发 I/O；
  • 差异点：
    • API 是 线程模型（Thread、中断、ThreadLocal 语义等），不是 async/await 或 go 关键字；
    • 调度与挂起由 运行时 完成，不是语言级 co_await 状态机；
    • 与结构化并发（Structured Concurrency）、作用域值（Scoped Values）等 Loom 周边特性配套演进。

若你的思维模型来自 Go：虚拟线程 ≈「JVM 版 goroutine 的一种实现思路」（都是轻量线程 + 运行时），而不是 C++20 / C# 那种 函数级无栈协程语法。

四、使用注意：不是银弹 #

4.1 Pinning（钉住载体） ##

某些情况下虚拟线程 无法 unmount，会把 carrier 钉住，退化成「轻量线程占着重线程」：

  • 长期占用的 synchronized 监视器（早期版本更敏感；新版本持续改进）；
  • 本地方法（JNI） 执行期间；
  • 其他运行时无法安全卸载的路径。

表现是：并发度上不去、carrier 池耗尽、延迟毛刺。排查时常看 JFR / 线程转储里的虚拟线程与 pinning 相关事件。

4.2 线程池习惯要改 ##

旧世界：

业务线程池大小 ≈ CPU 或 略大于 CPU（CPU 密集）
或 按公式估算阻塞系数（I/O 密集）

虚拟线程世界：

任务数可以很多；
不要为虚拟线程再套一层巨大的本地线程池「缓冲」；
CPU 密集仍应限制并行度（信号量 / 固定平台线程池）。

经验法则：I/O 密集 + 同步风格 最受益；纯 CPU 密集 虚拟线程不会变快。

4.3 ThreadLocal 与池化 ##

百万虚拟线程 × 大对象 ThreadLocal = 内存事故。Loom 推荐方向包括 Scoped Values（更适合不可变、有界生命周期的上下文传递）。连接池、请求上下文等设计也要重新审视「是否假设线程很少」。

4.4 库兼容性 ##

理想情况：JDK 自己的阻塞 I/O 在虚拟线程上可卸载。现实中：

  • 老旧第三方库若在本地代码里阻塞，仍会钉线程；
  • 自己封装的「线程亲和」逻辑可能失效（虚拟线程可能换 carrier）。

五、结构化并发（顺带一提） #

Loom 叙事不只虚拟线程，还包括 Structured Concurrency：把一组相关任务当作一个工作单元（一起成功、失败或取消），避免「扔进全局线程池就不管」的生命周期泄漏。

概念形态（示意）：

在一个 scope 内 fork 多个子任务
  → 等待全部完成或第一个失败
  → scope 结束时保证子任务被收敛

它与虚拟线程互补：前者管 寿命与取消，后者管 便宜地多跑任务。预览特性随 JDK 版本推进，写生产代码前核对目标 JDK 的 JEP 状态。

六、迁移视角：从 Kilim/Quasar 到 Loom #

旧方案痛点虚拟线程如何缓解
Weaver / Agent 构建复杂标准 JDK，无织入
@Suspendable 传染普通 Java 方法即可
与生态库难集成面向标准 Thread 生态
调试栈怪异仍有特殊点，但官方工具链持续跟进

遗留 Fiber 代码迁移时，重点不是「语法替换」，而是：

  1. 阻塞点是否真能卸载；
  2. 同步原语是否从 fiber-channel 换成虚拟线程友好的结构；
  3. 池化、超时、取消是否按结构化并发重做。

总结 #

Java 已经 在新版本中提供了平台级的轻量并发机制：虚拟线程（Project Loom，Java 21 正式可用）。它让「同步风格 + 高并发 I/O」成为一等公民，终结了多数场景下对 Kilim/Quasar 类织入框架的依赖。

理解它时抓住三条链：

  1. 虚拟线程 ↔ 载体平台线程 的 mount/unmount；
  2. continuation 保存与恢复执行状态；
  3. pinning 与资源模型 决定你能不能吃到理论红利。

它与 C++20 的 co_await、与 Kilim 的无栈状态机都不同；更接近 运行时实现的 M:N 轻量线程。要和 Go、libmill、C++20 放在同一张地图上比较，请看同目录「有栈与无栈协程对比」。
