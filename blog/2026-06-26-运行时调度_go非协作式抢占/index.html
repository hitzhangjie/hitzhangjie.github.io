深入浅出 Go 非协作式抢占

Posted 2026-06-26 18:01 +0800 by ZhangJie ‐ 2 min read

分享:  

在 Go 1.14 之前，Go 语言的调度器采用的是协作式抢占。这种机制存在一个致命的缺陷：如果一个 Goroutine 陷入了没有函数调用的死循环（例如纯 CPU 计算），它就会永久霸占线程，导致其他 Goroutine 饿死，甚至触发程序卡死或垃圾回收（GC）无法完成。

为了解决这个问题，Go 1.14 引入了基于 SIGURG 信号的非协作式抢占（Non-cooperative Preemption）机制。本文将系统性地拆解这一机制的底层原理。

1. 为什么被称为"非协作式"？ ##

为了理解非协作式抢占，我们需要先对比它与传统协作式抢占的区别：

特性协作式抢占 (Go 1.14 之前)非协作式抢占 (Go 1.14 及之后)
触发前提依赖 Goroutine 主动配合。强制执行，无需 Goroutine 配合。
检查时机发生函数调用或内存分配时，检查栈溢出标志。随时随地，由操作系统信号直接中断。
死循环应对无法解决纯计算死循环（如for {}），会导致饿死。完美解决，可在 10ms 内强行剥夺 CPU 执行权。

核心痛点：协作式抢占就像坐公交车，乘客（Goroutine）必须走到站台（函数调用点）才能下车。而非协作式抢占则像直升机索降，无论你在哪里，调度器都能直接把你"抓"走。

2. 核心大招：基于 SIGURG 信号的底层看板 ##

非协作式抢占的整个流程由系统监控线程、操作系统内核以及 Go 运行时（Runtime）共同协作完成：

+------------------+     超过 10ms      +-------------------+
|                  | -----------------> |                   |
|  sysmon 监控线程  |                    | 向目标线程发送信号  |
|                  |                    |                   |
+------------------+                    +-------------------+
                                                  |
                                           (发送 SIGURG)
                                                  v
+------------------+   修改 PC 寄存器    +-------------------+
|                  |                    |                   |
| 恢复执行用户代码  | <----------------- |  sighandler 阶段  |
|                  |                    |                   |
+------------------+                    +-------------------+
         |
    (流向改变)
         v
+------------------+    保存现场 & 挂起   +-------------------+
|                  |                    |                   |
|  asyncPreempt()  | -----------------> |  调度器切换新协程  |
|                  |                    |                   |
+------------------+                    +-------------------+

详细步骤 ##

  1. 触发检测：Go 语言的后台监控线程 sysmon 周期性运行。如果发现某个 Goroutine 占用单线程时间超过 10ms，就会将其标记为可抢占，并获取其绑定的操作系统线程（M）。
  2. 发送信号：sysmon 调用系统调用，向该线程发送一个 SIGURG（紧急带外数据信号）。
  3. 信号捕获：操作系统中断当前线程的执行，将控制权交给 Go 运行时提前注册好的信号处理函数 sighandler。

为什么选择 SIGURG 信号？ ##

在 Unix/Linux 系统中，SIGALRM 常用于定时器，但容易与 CGO 或用户自定义计时器冲突。而 SIGURG 在标准应用中极少被显式使用，且各大操作系统对其支持非常成熟，不会触发默认的进程终止，因此是绝佳的"内鬼信号"。

3. 核心机制：asyncPreempt 的汇编劫持 ##

收到信号后，Go 是如何让正在运行的代码停下来并切换的？这就是最精彩的寄存器劫持技术。

3.1 篡改程序计数器（PC / RIP） ##

当进入 sighandler 时，操作系统会在内核栈中保存被中断处的 CPU 上下文，其中包括 RIP 寄存器（指向下一条要执行的业务代码指令）。

Go 运行时在 sighandler 中做了一个"偷天换日"的操作：

  • 读取该线程被中断时的上下文。
  • 直接将上下文中保存的 RIP 值的地址修改为 runtime.asyncPreempt 函数的入口地址。
  • 信号处理结束，操作系统恢复线程。此时 CPU 读取修改后的 RIP，不会回到原本的业务代码，而是直接跳转到了 asyncPreempt。

3.2 asyncPreempt 内部做了什么？ ##

asyncPreempt 是一个纯汇编编写的底层函数。因为被异步中断时，编译器无法预知当前寄存器里存了什么关键数据，所以它必须做极致的现场保护。

它的核心源码逻辑如下：

// src/runtime/preempt_amd64.s (简化概念模型)
TEXT ·asyncPreempt(SB),NOSPLIT,$0
    // 1. 极其壮观的压栈操作：保存所有通用寄存器
    PUSHQ R15; PUSHQ R14; PUSHQ R13; PUSHQ R12
    PUSHQ BP;  PUSHQ BX;  PUSHQ AX;  PUSHQ CX
    PUSHQ DX;  PUSHQ SI;  PUSHQ DI;  PUSHQ R8
    PUSHQ R9;  PUSHQ R10; PUSHQ R11

    // 2. 保存浮点数和向量寄存器（防止破坏复杂的计算现场）
    SUBQ $512, SP
    FXSAVE64 (SP)

    // 3. 调用 Go 语言实现的抢占主逻辑
    CALL ·asyncPreempt2(SB)

    // 4. 如果没被成功抢占，恢复现场
    FXRSTOR64 (SP)
    ADDQ $512, SP
    POPQ R11; POPQ R10; // ... 依次出栈
    RET

在调用 asyncPreempt2 后：

  1. 运行时会检查当前是否处于安全点（Safe Point）。如果在某些不允许抢占的底层运行时代码中（如正在持有锁或执行 mallocgc 分配内存），则会放弃本次抢占，恢复现场继续运行。
  2. 如果满足抢占条件，调用 preemptPark。
  3. 将当前 Goroutine 的状态从 _Grunning 改为 _Grunnable。
  4. 将该 Goroutine 解绑并放回全局或本地调度队列，触发新一轮的 schedule() 调度，彻底完成抢占。

4. 总结 ##

Go 语言通过结合操作系统的信号机制（SIGURG）与底层的寄存器篡改（asyncPreempt），在没有引入明显运行时开销的前提下，彻底解决了长循环饿死线程的历史顽疾。

正是这种"非协作式"的强硬手段，让 Go 语言的调度器在高并发、高计算密度的复杂业务场景下，能够展现出更加稳定和可预测的低延迟性能。

本文框架完整地展现了非协作式抢占的全局观。后续请关注，深入探讨 GC 安全点（Safe Points） 以及为什么有些位置绝对不能被抢占。
