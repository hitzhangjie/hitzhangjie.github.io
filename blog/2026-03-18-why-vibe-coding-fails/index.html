为什么 vibe coding 搞不定复杂系统

Posted 2026-03-18 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 ##

我用 AI coding 工具一年多了。从最早的代码补全，到现在的 agent 自主写代码、修 bug、做重构，能力提升是肉眼可见的。Claude Code、Cursor、Copilot 这些工具我都在用，确实大幅提高了日常效率。

但有一个感受始终没变：当软件规模到了一定复杂度之后，没有思想上的沉淀，很难短时间内靠 vibe coding 来 100% 解决问题。

Vibe coding 这个说法，大意是开发者通过自然语言 prompt 让 AI 生成代码，自己更多扮演"产品经理"或"审查员"的角色，而非亲自动手写每一行。听起来很美好——你说需求，AI 出代码，你只管 review。对快速原型、内部工具、简单 MVP，这确实管用。但一旦项目跨过某个复杂度阈值，问题就来了。

AI 擅长"看起来对"，但搞不定隐式约束 ##

AI 生成代码有一个很典型的模式：它能快速搭出"看起来像那么回事"的架子，但底层 99% 的工作——那些真正决定软件能不能在生产环境跑下去的东西——它搞不定。

边缘情况是第一个坑。AI 对 happy path 的处理相当熟练，但真实用户产生的边缘情况是无穷的。你让它写一个表单提交逻辑，它能写好；但你没告诉它的——网络断了怎么办、用户快速双击提交怎么办、服务端返回了预期外的状态码怎么办——这些它不会主动考虑。而这些恰恰是生产环境里每天都在发生的事。

架构设计是第二个坑。没有人在顶层做架构设计，AI 生成的代码往往是功能堆砌，缺乏逻辑分层。几个迭代下来，你就会得到一团"意大利面条"——不是 AI 写得不好，而是没有一个大脑在做全局的结构性决策。每一段代码单独看都没问题，拼在一起就是一盘散沙。

隐式业务逻辑是第三个、也是最隐蔽的坑。AI 经常把复杂的业务规则悄悄塞进一个 helper 函数里，没有注释，没有文档，没有类型约束。三个月后你自己都看不懂这段代码在干什么，更别说改了。

ps: 这不是 AI 的"错"。这些问题本质上不是代码生成的问题，而是理解问题域的问题。AI 不知道你的业务上下文、你的用户行为模式、你的运维约束——除非你用 prompt 把这些全部喂给它。但如果你能把这些全部精确地描述清楚，你实际上已经完成了软件工程中最难的部分。

修一个 bug，引入两个新 bug ##

Vibe coding 的另一个典型陷阱是"让它先跑起来"的心态。这在小项目上没问题，但在长期维护的项目上会积累惊人的技术债务。

AI 修 bug 的方式很有意思——它倾向于"打补丁"而非"治根"。你告诉它这里有个 null pointer，它就在那里加个 nil check；你告诉它这里并发有问题，它就加个锁。至于这个 nil 是从哪里来的、为什么会有并发冲突、整个数据流是不是设计上就有问题——它不关心。它只解决你描述的那个症状。

然后就是经典的调试死循环：修复一个 AI 生成的 bug → 引入两个新 bug → 修复那两个新 bug → 引入四个新 bug。你花在调试上的时间，可能比你亲手写的时间还多。

我自己的经验是：AI 最擅长的是你已经完全理解的代码。你知道这段代码应该长什么样、边界在哪、怎么测试——AI 帮你写，快。但如果你自己都没想清楚这段代码应该怎么设计，让 AI 替你"想"，出来的东西大概率后面要还债。

工程素养的"暗知识" ##

这引出一个更根本的问题：软件工程的核心困难从来不是"写代码"本身，而是"理解要写什么代码以及为什么"。

这就是我说的"思想沉淀"。它不是某一个具体的技术栈或框架知识——那些 AI 学得很快。它是你对一个系统的长期理解：为什么当初选了这个架构而不是那个、这个模块的边界为什么划在这里、哪些地方是历史遗留的坑碰不得、哪些看似奇怪的设计其实是在特定约束下的最优解。

这些东西很少被写进文档，更多是以"暗知识"的形式存在于团队的大脑里。AI agent 拿不到这些上下文，因为连你自己都未必意识到你知道这些——直到 AI 动了不该动的地方，你才恍然大悟"哦这个不能改"。

ps: 工程素养、软件工程原则、最佳实践这些东西，正在被 AI agent 吸收，甚至直接被内化为模型本身的能力。这一点肯定会越来越好。最近的模型在代码审查、架构建议方面的表现已经比一年前强太多了。

离 100% 还有点远 ##

但可以预见的是，离"100% 靠 vibe coding 解决问题"，还有点远。

不是因为 AI 不够聪明——它已经比大多数初级工程师聪明了。而是因为软件工程中有太多决策依赖于"上下文"而非"知识"。知识可以被训练进模型参数里，上下文不行。上下文是活的——它在每一次产品讨论中产生，在每一次 on-call 的凌晨排查中积累，在每一次重构的纠结中沉淀。

你能想象一个 AI agent 在你没告诉它的情况下，知道你们团队的 MySQL 连接池配置有一个奇怪的参数是因为三年前的一次线上事故吗？知道某个第三方 SDK 的 v1.3 版本有内存泄漏所以必须用 v1.2？知道某个看似多余的字段是因为财务部门需要一张特定的报表？

这些东西，只有人在。

Vibe fixer：一个预测 ##

如果 vibe coding 真的继续爆火下去，我预测会兴起一个新角色——vibe fixer。

就是专门收拾 vibe coding 烂摊子的人。那些"先跑起来再说"的项目，在用户量上来之后开始各种崩、各种慢、各种数据不一致，创始人发现自己的"AI 全栈工程师"产出的代码谁也看不懂、谁也不敢动。这时候就需要有人来救火——读懂那一团 AI 生成的代码，理清业务逻辑，重构成能维护的样子。

ps: 这其实不是一个新故事。低代码平台的"最后一公里"问题、外包项目的"接盘重构"需求，本质上都是同一件事：当"写代码"的门槛被大幅降低后，理解和维护已有代码的能力反而变得更加稀缺和值钱。

某种程度上，vibe fixer 就是软件工程领域的"技术债务清理师"。如果 vibe coding 降低了软件开发的进入门槛（这是好事），那么随之而来的海量低质量代码就会创造巨量的修复需求。这不是讽刺，这是市场的自我平衡。

总结 ##

Vibe coding 不是骗局，也不是银弹。它是一个工具——对某些场景极其高效，对另一些场景力不从心。

我用了一年多的 AI coding 工具，结论是：用它来加速你"已经知道怎么做"的事情，而不是替代你"还没想清楚"的思考。 前者它能做到令人惊艳的程度，后者它只会给你一种"搞定了"的错觉，然后让你在三个月后付出代价。

至于 vibe fixer 会不会真的成为一个职业——谁知道呢。但如果哪天你在招聘网站上看到了这个 title，记得这篇文章写于 2026 年 3 月。
