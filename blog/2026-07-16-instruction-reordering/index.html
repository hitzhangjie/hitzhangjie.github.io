指令重排序：编译器与 CPU 的"合法作弊"艺术

Posted 2026-07-16 19:00 +0800 by ZhangJie ‐ 6 min read

分享:  

现代 CPU 和编译器为了让程序跑得更快，会在背后做一件事：调整指令的执行顺序。只要最终结果看起来和"按代码顺序执行"一样，中间怎么排是它们说了算。

这听起来像作弊，但实际上是经过严格数学验证的合法优化。这篇文章就来拆解：谁在做重排序、为什么要做、依据什么规则来做、以及不同的 CPU 架构到底在哪些地方"放了水"。

两层重排序：编译器 + CPU ##

指令重排序发生在两个层面：

源代码
  ↓
编译器优化重排（静态，编译期）
  ↓
CPU 乱序执行重排（动态，运行期）
  ↓
最终执行顺序

编译器重排序：编译器在做优化（如寄存器分配、公共子表达式消除、指令调度）时，可能会调整生成的机器指令顺序。只要不改变单线程下的程序语义，怎么排都行。

CPU 重排序：现代处理器几乎都是乱序执行（Out-of-Order Execution）的。取指、译码后的指令进入保留站（Reservation Station），操作数就绪的就发射执行，不管它在代码里的先后位置。此外，store buffer 的存在也让写操作对其他核的可见时机变得不确定。

关于 store buffer 机制以及它如何影响跨核可见性，已在《Store Buffer、内存屏障与缓存一致性》中详细讨论，本文更聚焦于"重排序的规则"本身。

铁律：As-If Serial 语义 ##

无论编译器还是 CPU，重排序都遵循一个基本原则：

在单线程（或单个执行上下文）内，重排序后的执行结果必须与原程序顺序执行的结果完全一致。

这就是 as-if serial 语义，也叫程序顺序语义（Program Order Semantics）。它是所有重排序优化的底线——编译器或 CPU 可以随意调整顺序，但只要你站在这个线程内部观察，一切必须"看起来"是按代码顺序发生的。

这个铁律意味着：重排序不能破坏数据依赖关系。

数据依赖：哪些不能重排 ##

指令之间的依赖关系分为三类：

真依赖：RAW（Read After Write） ##

① x = 42        // 写
② y = x + 1     // 读 x 的值

指令 ② 必须等指令 ① 完成才能得到正确结果。RAW 依赖不可重排——一旦写操作的值还没产生就去读，读到的就是垃圾。这是硬件级别的硬约束，CPU 的乱序引擎会自动识别并等待。

反依赖：WAR（Write After Read） ##

① r1 = y        // 读 y
② y = 43        // 写 y

如果重排成先写后读，① 读到的就是 43 而不是原来的值。在逻辑上这不能重排，但 CPU 通过**寄存器重命名（Register Renaming）**打破了物理层面的依赖——② 被分配一个不同的物理寄存器，和 ① 用的寄存器虽然逻辑上是同一个 y，但物理上是两个不同的存储单元。于是这两条指令可以并行执行，谁也不等谁。

输出依赖：WAW（Write After Write） ##

① x = 42        // 写
② x = 43        // 写

如果重排成先 ② 后 ①，最终 x = 42，错了。同样，寄存器重命名可以消除这个假依赖——两个写操作写入不同的物理寄存器，最终提交时按程序顺序让后写的那个"胜出"。

总结：

依赖类型名称能否重排CPU 如何应对
RAW真依赖不能乱序引擎必须等待操作数就绪
WAR反依赖逻辑上不能寄存器重命名消除假依赖
WAW输出依赖逻辑上不能寄存器重命名消除假依赖

说白了，CPU 在微架构层面做了大量的"暗箱操作"，让只存在假依赖（WAR、WAW）的指令可以真正并行执行，而只对真依赖（RAW）保持等待。这就是乱序执行能提升吞吐的核心原理。

CPU 重排序的四种可能 ##

从内存操作（load/store）的角度看，CPU 重排序可以分为四种组合。在展开之前，先厘清两个关键概念：

  • 程序顺序（Program Order）：指单个线程内，源代码（及编译后的机器指令）中内存操作的先后次序。例如 x = 1; y = 2;，程序顺序是"先写 x，再写 y"。
  • 观察顺序（Observed Order）：指其他核心看到的该线程内存操作效果的先后次序。由于流水线乱序、store buffer、缓存一致性等机制的存在，观察顺序可以和程序顺序不同。

所谓"重排序"，本质上就是这两个顺序之间的差异——站在本核视角一切正常（as-if serial），但站在其他核的视角，操作的发生顺序可能已经变了。

这种差异的来源有三类：

  • 核内流水线乱序：无依赖的 load/load、load/store 可以提前执行。
  • Store Buffer 异步提交：store 先进 store buffer，后续 load 不等它排空就执行，导致 Store→Load 在跨核视角下"先读后写"。
  • 缓存一致性协议：不同地址的 store 到达其他核缓存的顺序可能与程序顺序不同（弱一致性模型下尤其常见）。

那么"其他核心观察到"具体是怎么发生的？不是另一个核在主动监视本核的流水线，而是通过缓存一致性协议间接体现的：

  1. Core A 的 store 先从 store buffer drain 到 L1 cache，再通过 MESI 等协议向其他核心广播 Invalidate 消息。
  2. Core B 执行 load 指令读同一变量时，如果自己的 cache line 已被 Invalidate，就会从 Core A 的 cache 或共享缓存中拿到新值。此时 B"观察"到了 A 的这次写。

假设 Core A 的程序顺序是 x = 1; y = 2;，Core B 通过先后读 x 和 y 来"探测"这两次写的传播时序：

  • 读到 x=1, y=0 → A 对 x 的写已到达 B，对 y 的写还没到——和程序顺序一致
  • 读到 x=0, y=2 → A 对 y 的写先到达 B，对 x 的写后到——观察顺序和程序顺序反了，这就是重排序
  • 读到 x=1, y=2 → 两次写都到达了，且和程序顺序一致

说到底，观察顺序就是其他核心通过 load 指令读到的值发生变化的先后顺序。缓存一致性协议只管"最终一致"，不管不同地址的写入按什么顺序传播——想要约束这个传播时序，就得用屏障指令。

下表中，A 和 B 表示程序顺序中先后相邻的两次内存操作（可以是 load 或 store），“重排"指观察顺序变成了先 B 后 A：

重排序方向含义单线程安全？
Load → LoadA 读之后 B 读，重排成先 B 后 A如果 A/B 读同一变量则有 RAW 依赖，不可重排；读不同变量则安全
Store → StoreA 写之后 B 写，重排成先 B 后 A如果 A/B 写同一变量则有 WAW 依赖，不可重排；写不同变量则安全
Load → StoreA 读之后 B 写，重排成先 B 后 A如果 B 写的就是 A 读的变量，则有 WAR 反依赖，逻辑上不可重排
Store → LoadA 写之后 B 读，重排成先 B 后 A最容易出问题的组合——只要读写的不是同一个变量，单线程看不出区别

对于单线程而言，只要 A 和 B 操作的是不同变量（没有数据依赖），四种重排在结果上都是无害的。这也是为什么编译器在单线程代码面前可以毫无顾忌地重排无关变量的访问。

但一到多线程共享变量的场景，问题就来了。

不同 CPU 的内存模型 ##

不同的 CPU 架构允许多大程度的"合法作弊"各不相同。这就是所谓的内存一致性模型（Memory Consistency Model）。

理想模型：顺序一致性（SC） ##

Lamport 在 1979 年定义了 Sequential Consistency：

多处理器的执行结果，必须看起来像是所有处理器的操作按某种全局交错的顺序依次执行，且每个处理器内部的操作保持程序顺序。

SC 是最直观、最容易编程的模型，但它对硬件实现太不友好——CPU 必须严格按程序顺序执行所有内存操作，几乎无法发挥现代 CPU 的流水线和缓存优势。所以没有实际的通用处理器真正实现 SC。

x86：TSO（Total Store Order） ##

x86 是相对"程序员友好"的模型，只留了唯一一个重排机会：

重排序方向x86 TSO
Store → Store❌ 不会重排
Load → Load❌ 不会重排
Load → Store❌ 不会重排
Store → Load✅可以重排！

TSO 的"Total"指的是所有核心对 store 操作的全局顺序有一致的看法——不是你有你的顺序、我有我的顺序。

唯一的"洞"就是 Store→Load 重排。一个经典的例子：

// 初始：x = 0, y = 0

// Core A
x = 1;          // store
r1 = y;         // load  ← 可能被重排到 store 之前

// Core B
y = 1;          // store
r2 = x;         // load  ← 同理

// 可能的结局：r1 = 0 且 r2 = 0
// 这在 SC 下不可能，但在 x86 TSO 下完全可能！

为什么会发生？因为 Core A 对 x 的写会被放入 store buffer，store buffer 采用 FIFO 顺序，但在写被 drain 到 cache 之前，Core A 对 y 的读就已经执行了，此时 Core B 还没写 y。同时在 Core B 那边也发生了同样的事。结果两个核都读到了对方的旧值。这就是由 store buffer 异步提交机制直接导致的 Store→Load 重排效果。

要使 Core A 先写后读的结果在多核之间看起来严格有序，需要用 mfence 或 LOCK 前缀指令来做 StoreLoad 屏障——强制 drain store buffer 再执行后续的 load。

ARM / POWER：弱一致性模型 ##

ARM 和 POWER 是更激进的弱一致性模型，几乎允许所有类型的重排：

重排序方向ARM / POWER
Store → Store✅ 可能重排
Load → Load✅ 可能重排
Load → Store✅ 可能重排
Store → Load✅ 可能重排

ARM 之所以如此宽松，是因为它的 store buffer 甚至对嗅探（snoop）不可见——其他核来读时不会去翻 store buffer。这让硬件实现更简单、功耗更低，但几乎所有的跨核同步都需要显式内存屏障。

ARMv8 通过 STLR（store-release）和 LDAR（load-acquire）提供了细粒度的单向屏障，避免了 full dmb 的开销，也让 release-acquire 语义成为弱一致性模型下的主流编程范式。

三平台对比 ##

重排序方向x86 TSOARMv8 (无屏障)ARMv8 (release-acquire)
Store → Store不重排可能重排STLR → STLR 之间不重排
Load → Load不重排可能重排LDAR → LDAR 之间不重排
Load → Store不重排可能重排需要dmb 保证
Store → Load可重排可能重排需要dmb 保证

编译器重排序：静态层面的优化 ##

除了 CPU 的动态重排，编译器也会在编译期对指令顺序动手。编译器优化有几个经典手段：

公共子表达式消除（CSE） ##

// 原始代码
a = b * c + g;
d = b * c + e;

// 编译器优化后等价于
tmp = b * c;
a = tmp + g;
d = tmp + e;

b * c 的计算被"提前"到 g 的读取之前，这在单线程下完全没问题。

指令调度 ##

// 原始代码
x = *p;         // load，可能 cache miss，耗时较长
y = a + b;      // 纯计算，不依赖 x

// 编译器可能重排为
y = a + b;      // 先执行无依赖的计算，掩盖 load 延迟
x = *p;

编译器利用 load 的延迟来做有用功，这就是指令调度的价值。

循环不变代码外提（Loop-Invariant Code Motion） ##

// 原始代码
for (int i = 0; i < n; i++) {
    a[i] = b[i] * (x * y);
}

// 编译器优化后
tmp = x * y;
for (int i = 0; i < n; i++) {
    a[i] = b[i] * tmp;
}

x * y 被提到循环外，从执行 n 次变成执行 1 次。

编译器与并发 ##

编译器的重排在单线程下完全符合 as-if serial 语义，但它不一定知晓多线程的存在。对于下面这段代码：

// 标志位模式（多线程共享 flag, data）
// 线程 A：准备数据，然后"举旗"
data = 42;
flag = 1;       // 编译器不知道 flag 被线程 B 读取

// 线程 B：看到旗子后读数据
while (flag == 0) /* spin */;
r = data;

编译器可能因为 data 和 flag 是不同变量、没有单线程依赖，而把 store 重排成先 flag = 1 后 data = 42。在线程 B 看来，flag == 1 时 data 可能还是旧值。

解决方案不是"关掉优化”，而是用正确的方式告诉编译器和 CPU"这里不能重排"——即使用 atomic / volatile / 内存屏障。

屏障：对重排序说"不" ##

内存屏障（Memory Barrier/Fence）就是程序员的"重排序刹车"。不同层级有不同的屏障：

层级手段作用
C/C++std::atomic + memory order阻止编译器和 CPU 重排
Javavolatile / synchronizedJVM 插入平台对应的屏障指令
Gosync/atomic / sync.Mutex通过LOCK 前缀或等价指令实现屏障
汇编 (x86)mfence / lfence / sfenceCPU 指令级屏障
汇编 (ARM)dmb / dsb / isbCPU 指令级屏障

各语言的高层并发原语最终都编译（或 JIT）到平台的特定屏障指令。

各语言 volatile、atomic 在 x86 和 ARM 上的具体屏障映射，已在《Store Buffer、内存屏障与缓存一致性》中有详细说明。

并发编程的启示 ##

理解了重排序机制之后，有几个关键认知：

1. 重排序不是 bug，是 feature。 它让单线程代码跑得飞快，代价是编写多线程代码时需要更小心。

2. 并发编程中，“看起来会先发生"不靠谱。 你写了 A、B 两个操作在上下的两行，如果不加同步，它们的实际执行顺序和可见顺序在另一个线程眼里可能是反的。

3. ABA 问题深层原因也跟重排序有关。 CAS 操作的 ABA 问题看似是值的循环，但根源之一是在没有正确屏障的情况下，看似成功替换的值背后，其实已经发生过一整套读写序列的乱序。

4. 语言提供的并发原语，是对硬件差异的抽象。 好的代码不依赖平台特定的屏障指令，而是使用标准库提供的 atomic、Mutex、Channel 等原语。底层的重排差异被这些原语屏蔽了。

5. Happens-Before 是理解可见性的正确姿势。 与其纠结"CPU 到底会不会重排这条指令”，不如用 happens-before 关系来判断：A happens-before B，则 A 的结果对 B 可见。JMM（Java Memory Model）、Go Memory Model 都是基于 happens-before 来定义正确同步程序的可见性保证的。

总结：重排序的全景图 ##

│ 编译器重排序                                                │

│ · 不了解多线程共享变量的语义 → 可能做"单线程安全但          │
│   多线程不安全"的重排                                       │
│ · 通过编译器屏障（compiler barrier）阻止                    │

│ CPU 重排序（微架构层）                                      │
│ · 乱序执行引擎：真依赖（RAW）不可重排，假依赖（WAR, WAW）   │
│   通过寄存器重命名消除                                       │
│ · Store Buffer 异步提交 → Store→Load 重排（x86 唯一的洞）  │
│ · Load→Load / Store→Store / Load→Store 是否重排因架构而异  │
│ · 通过硬件内存屏障指令（mfence / dmb / STLR+LDAR）阻止     │
├─────────────────────────────────────────────────────────────┤
│ 软件层面的应对                                              │
│ · 高级语言并发原语（atomic, volatile, Mutex）               │
│ · 编译为平台对应的屏障指令                                  │
│ · 建立 happens-before 关系，保证跨线程可见性                │
└─────────────────────────────────────────────────────────────┘

回到文章开头的问题：指令重排序的规则到底是什么？核心就是两条：

  1. 对单线程，必须符合 as-if serial 语义——有数据依赖（RAW）的不能重排，没有依赖的随便排。WAR 和 WAW 这些"假依赖"被 CPU 通过寄存器重命名消化掉了。注意：寄存器重命名是 CPU 硬件层面做的事，不是编译器。编译器只能看到有限的架构寄存器（x86-64 就十几个），而 CPU 内部有上百个物理寄存器；译码时每遇到一个目标寄存器写操作，就从空闲池里分配一个新的物理寄存器给它——同一个 RAX 在不同时刻可能对应完全不同的物理寄存器，WAR 和 WAW 的假依赖就这样被硬件自动消除了。
  2. 对多线程，不同架构有不同程度的"放水"——x86 只放 Store→Load 一个洞，ARM 四个洞全开。程序员用并发原语（atomic 等）来"补洞"，这些原语在编译后变成平台的屏障指令，告诉编译器和 CPU：这里不能重排。

所以指令重排序不是"可以胡来"，而是一套精细设计的规则体系——在保证单线程正确性的前提下最大化并行度，同时把多线程同步的控制权交给程序员。
