网络编程：TCP send 返回成功，真的意味着对端收到了吗？

Posted 2026-06-25 10:00 +0800 by ZhangJie ‐ 3 min read

分享:  

TCP 连接 send 返回成功，真的意味着对端收到/处理了吗？先说结论，不光可能没处理，甚至收到都不一定收到。很多人说TCP不是可靠的吗？为什么像golang tcpconn.Write 返回了nil，竟然还有可能发送不成功呢？

这是很多开发者的TCP认知误区。现如今很多网络服务器开发都是基于开发框架、RPC桩代码生成，网络通信被简化成了一个桩代码函数调用，开发者的技能也从系统调用、libev库之类的较深的理解慢慢会话为了RPC桩代码call boy :)

背景 #

前几天同事问了我一个问题：我们框架里通过 send 通知对方服务，如果 send 没有返回失败，能不能 100% 确定对端收到了这个包？

先对齐下概念：这里的send，是RPC时只发req但是不返回rsp给上层调用，在有些框架里也会用选项notify或者sendonly来标识；与send一应的还有一个call，这里的call指的是发送req并且返回rsp给上层调用，有些框架里会用选项sendrcv来标识。

答案是不能。这个问题其实触及了一个非常经典的认知误区——很多开发者把"系统调用没报错"等同于"对端进程收到了数据"。这个误区之所以普遍，是因为它跨越了多个抽象层次：从应用层 buffer 到内核协议栈，从网卡队列到网络传输，再到对端操作系统和业务进程，每一层都有"你以为成功了但其实还没有"的环节。

下面我们从几个层次来拆解这个问题。

问题一：系统调用成功 ≠ 数据送达 #

层层缓存：everything is cache ##

当你调用 tcpconn.Write(buf) 并且没有返回 error 时，这个"成功"到底意味着什么？答案是：它只意味着数据从应用层的 buffer 拷贝到了内核的 socket buffer。后面的路还很长：

应用层 buffer
    ↓ tcpconn.Write (系统调用)
内核 socket buffer      ← 系统调用到这里就返回了
    ↓ TCP 协议栈（分段、拥塞控制、重传……）
网卡 TX queue (NIC buffer)
    ↓ 物理层
网络
    ↓
对端网卡 → 内核 → 应用层

系统调用的语义是"数据已提交给内核"，它不会等待数据通过网卡发送出去，更不会等待对端的 TCP ACK。这是有意为之的设计——如果每个 write 都要等到对端确认，那么一次 RTT 就会阻塞调用线程，性能将完全不可接受（后面还会细说）。

连接池里的"幽灵连接" ##

这个问题的典型重灾区就是 TCP 连接池。考虑这个场景：

  1. 你从连接池中取出一个 tcpconn，它在上次使用时是健康的
  2. 但实际上，对端进程已经重启了，或者网络中间设备已经清理了这个连接的状态
  3. 由于你还没有收到 FIN 包（或者收到了但还没执行 Read 去感知），本地操作系统认为这个连接仍然 ESTABLISHED
  4. 你在这个连接上 Write 数据，系统调用成功返回——因为数据被拷进了内核 socket buffer
  5. 数据发到对端后，对端操作系统发现这个连接已经不存在了，回复一个 RST
  6. 这个 RST 被本地内核收到，连接被标记为错误状态
  7. 只有到下一次 Write 时，你才会收到 connection reset by peer 的错误

这就是所谓的 delayed failure：你第一次 Write 成功了，但数据实际上没有被对端进程收到；错误要等到第二次 Write 才能被感知。

能不能提前检测？ ##

一个自然的想法是：在 Write 之前先检测连接是否健康。在 Go 里，标准库的 net.Conn 提供了 Read 方法，基于 epoll/kqueue 的非阻塞 I/O 多路复用。但问题是：

  • Read 在没有数据时会阻塞，等待数据到达或 deadline 到期
  • 你无法在不阻塞的情况下"看一眼"连接是否收到了 FIN 或 RST
  • 即使你用一个极短的 deadline 去 Read，这也只是一个 workaround，不是根本解法

关于这个问题，Golang 社区有过深入的讨论：gonuts - detect broken tcp conn。里面提到了很多类似的边界情况。

我曾在自己开发的 GoNeat 框架 中实现过一个检测机制：利用 syscall 直接检查 socket 的错误状态，而不需要执行阻塞的 Read 操作。思路是在 Unix 上通过 getsockopt 获取 SO_ERROR，配合 poll 做非阻塞探活。但即使这样，也只能缩短检测延迟（取代过大的deadline），不能从根本上消除 delayed failure 的窗口。

问题二：网络送达 ≠ 业务处理 #

假设运气好，前两个环节都没出问题——数据成功发送了，对端操作系统也收到了，并且本地也收到了 ACK——这能不能保证对端业务进程处理了这条消息呢？还是不能。

考虑对端进程的重启场景：

  1. 对端进程收到了你的消息，tcpconn.Read 返回了数据（或者还没read就crash或者暴力重启）
  2. 数据被 dispatch 给一个 goroutine 处理（或者还没dispach之前就crash或者暴力重启)
  3. 但在 goroutine 处理完成之前，进程crash、暴力重启（有些号称优雅重启的方案，不一定真的优雅）
  4. 消息丢了——数据从 TCP 连接上被成功读取，但业务逻辑没有执行完

还有一种更糟糕的情况：进程在重启时没有调用 conn.Close() 或 Shutdown()，导致连接在操作系统层面仍然存活。老进程虽然不再处理数据了，但 TCP 连接还能收包。发送方看到连接是好的，数据发出去了，但对端已经没人处理了。

这些问题不是 TCP 协议层面的问题，而是应用层生命周期管理的问题。TCP 不知道你的进程什么时候退出，它只负责在两端操作系统之间可靠传输字节流。

TCP 的可靠性到底指什么？ #

说到这里，很多人会反问：TCP 不是可靠传输协议吗？为什么连"对方收到没收到"都不能保证？

这恰恰是对 TCP 可靠性最大的误解。TCP 的"可靠"指的是：

  • 字节流的按序、无差错交付：数据不会乱序、不会损坏、不会重复。如果丢包，自动重传
  • 端到端的确认机制：发送方的 TCP 协议栈保证数据被对端的 TCP 协议栈确认接收
  • 拥塞控制和流量控制：自适应调整发送速率

但 TCP 不保证：

  • 系统调用返回后，数据在多少时间内被对端收到
  • 发送方能在微秒级时间内感知到丢包
  • 对端应用程序是否真的 Read 并处理了数据
  • “发出去就收得到"这个语义——那是业务层的语义，不是传输层的

划重点：TCP 是一条操作系统到操作系统的可靠管道，不是应用到应用的可靠管道。

为什么不让 write 等到 ACK 再返回？ ##

有人可能会想：既然 TCP 本身有 ACK 机制，那为什么不让系统调用等到收到 ACK 后再返回呢？这样不就"可靠"了吗？

这个想法有两个致命问题：

1. 累计确认与 Head-of-Line 阻塞 ##

TCP 使用**延迟累计确认（delayed ACK）**机制：对端不会每收到一个包就回一个 ACK，而是等一小段时间合并确认多个包，用下一个期望的字节 offset 作为 ACK number。这能显著减少 ACK 包的数量，提高网络吞吐。

如果多个 goroutine 共用同一个 TCP 连接发送数据，那么当这个累计 ACK 到达时，它的 offset 可能已经覆盖了多个发送者的数据。这意味着：

  • 一个 ACK 的延迟会连锁影响多个等待 Write 返回的 goroutine
  • 如果网络抖动导致 ACK 延迟，所有共用这个连接的发送者都被阻塞
  • 这是典型的 head-of-line blocking

2. 线程饥饿与多米诺骨牌效应 ##

假定我们让 I/O 多路复用（epoll/kqueue）支持收到ACK后才让write系统调用返回，一个 TCP 连接的发送操作就可能导致线程被挂起“更久”。现在可能是加锁+数据copy的时间，现在变成了还要附加网络延迟的时间，更可怕的是对端crash要等到deadline超时这么久。

想象一下：

  • 一个线程因为等待 ACK 被挂起
  • 大量连接都需要等待 ACK，大量线程被挂起
  • 线程池被耗尽，新请求无法处理
  • 整个服务进入不可用状态

系统调用不等待 ACK，不是 TCP 的缺陷，而是刻意为之的设计取舍——用应用层的"部分复杂性"换取传输层的吞吐和系统的可扩展性。

你不能既要系统调用不阻塞（高性能），又要系统调用返回时保证对端收到了（强一致性）。这两个目标是矛盾的，必须在不同层次分别处理。

正确的解法：应用层区分 SendRcv 与 Notify #

归根结底，这个问题要在应用层协议交互模式上解决。你需要正确区分两种通信语义：

模式语义做法适用场景
SendRcv（Call）我需要确认对方收到了发送请求 → 接收响应 →校验响应关键操作：扣款、状态变更、数据写入
Notify（Send）我只是通知一下，丢了也无所谓只发送，不收响应（或收了不校验）非关键通知：缓存失效、配置刷新、事件推送，有其他手段兜底

关注可靠性就用 SendRcv，别用 Notify

很多框架提供了 call（SendRcv）和 send（Notify）两种原语，但业务开发者有时图省事，在关心对端是否成功的场景下也用了 send，理由是"send 也没报错，应该到了吧”。

这正是本文要纠正的认知。你需要：

  • 关心对端是否成功处理 → 用 SendRcv，发请求收响应，而且要校验响应，比如有没有逻辑错误码
  • 丢了也无所谓 → 用 Notify，只发不收

这不是框架的 bug，而是通信模型的本质限制。你就算换成 UDP，只要按照 SendRcv 的模式设计应用协议（发请求 → 等确认 → 超时重传），也能实现可靠的 UDP 通信。当然了，你的包最好小于 64KB。现在有了更好的 QUIC 协议，但核心思想是一样的——可靠性不是传输层一个层的事，而是端到端协议设计的事。

说白了就一点：如果你关心结果，就用 send+recv 并且校验结果。别信 notify 的 nil error。

ps: 有些框架包装的 notify 实现，发了请求也收了回包，但是没有检查回包中的逻辑错误码并包装成error返回，可能是为了与”通知“”不care结果“语义一致，但既然都收都收了还不处理下就有点粗糙：

  • 既然caller不关心结果，那么notify的时候协议头应该标识这是notify，callee就不用回包了 …… 这种更与”通知"“不care结果"语义一致；
  • 如果多种框架互操作，callee回包了，实际上caller不需要这个回包，直接丢了就行。但是既然都收到了回包，是不是应该检查下有没有错误会好点？

btw, 这种设计容易让使用者产生一个认知惯性：“调用没报 error、也收到回包了——那对端肯定收到并处理成功了吧？” 。但实际上，如果框架在有条件做错误检查的时候选择了吞掉错误——caller拿到的就是 nil error，而真实情况是对端已经明确告诉你处理失败了。

问题的根因在协议设计：像 tRPC、gRPC 可以把业务错误码放在框架协议头里统一传递，而不是丢在 payload 里各业务自己管（容易不一致）。如果协议先天没考虑这一点，后面靠"要求所有 RPC response 都包含 result 字段，然后接口断言去取" 这种打补丁的方式补救，本身就暴露了框架设计问题。

总结 #

回到最初的问题：send 没返回失败，能 100% 保证对端收到/处理了吗？不能保证，不光处理没保证，是否收到也没保证。

原因逐层递进：

  1. 系统调用层面：Write 成功只代表数据进入了内核 socket buffer，后面的 NIC、网络、对端操作系统都还没影
  2. 连接状态层面：连接池中的"幽灵连接"可能导致 delayed failure，第一次 Write 成功、第二次才报错
  3. 进程生命周期层面：即使网络送达了，对端进程可能在处理前退出，数据丢失
  4. 协议设计层面：TCP 保证的是 OS-to-OS 的可靠字节流传输，不是应用-to-应用的可靠消息投递。系统调用不等待 ACK 是有意为之的性能取舍

正确的做法是：在应用层协议设计时，明确区分 SendRcv（需要确认）和 Notify（丢了无所谓），并在关心可靠性的场景下，发送请求并校验响应。

理解 TCP 真正的可靠性边界，理解网络 I/O 的分层抽象——应用层 buffer → 内核 socket buffer → NIC → 网络 → 对端——每一步的"成功"都有不同的含义。理解这些，你的分布式系统才会更可靠。

如果你对进程重启阶段的"平滑重启、热重启方案"产生了更多兴趣，可以了解下不同层次的做法，

  • 比如借助发布平台的能力：服务注册发现摘除 -> 请求流量掉0 -> 服务退出 -> 服务启动 -> 加流量权重，这是大部分发布平台的基本操作。
  • 再比如借助服务框架自身的能力：tcplistener还建不建新链接、已经建立的连接还处理不处理IO、已经处理的请求是不是要等处理完、连接上收取了一半的包还处不处理？如果你要用新进程无缝接管新进程，你会如何处理上述各个阶段？可以参考tRPC，也可以参考下mosn/mosn，后者处理的比较极致。
