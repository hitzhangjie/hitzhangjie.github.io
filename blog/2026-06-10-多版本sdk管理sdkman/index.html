多版本 SDK 管理：SDKMAN! #

问题的起源 ##

最近准备开始研究一些 Java 开发相关的项目，第一件事自然是配置 Java 开发环境。装个 JDK 看起来简单，但真正动手的时候，问题就来了：

  • 不同项目依赖不同的 Java 版本——这个项目跑在 JDK 8 上，那个项目需要 JDK 17，还有一个在尝鲜 JDK 21 的虚拟线程。
  • 除了 Java runtime，还有 Maven、Gradle 这些构建工具的版本也要管理。
  • 最近这些年主要写 Go，对 JVM 生态的版本管理工具已经有些生疏了。

于是决定系统地梳理一下目前有哪些方案，以及哪个方案最适合当下的场景。

多版本管理的痛点 ##

这其实是一个老问题。如果你在 Linux 下做过开发，大概率接触过 update-alternatives：

# 每次安装新版本，都要手动注册
sudo update-alternatives --install /usr/bin/java java /opt/jdk-17/bin/java 1 --slave javac ... -- slave javap ...
sudo update-alternatives --install /usr/bin/java java /opt/jdk-21/bin/java 2 --slave javac ... -- slave javap ...

执行版本切换时要异同切换master和slave对应的二进制：

sudo update-alternatives --config java

这套机制在 Debian/Ubuntu 生态里工作了多年，但痛点也很明显：

  1. 每次安装新版本都要手动注册 master/slave 符号链接，Java 还要连带 javac、javadoc 等一堆工具。
  2. 跨平台基本不可用——macOS 用 Homebrew 管版本，Windows 只能靠安装目录和 PATH 手动切。
  3. 一个工具链一套配置——Java 配一套，Go 配一套，Node 配一套，每种的语法和机制都不一样。

于是各个语言社区陆续长出了自己的版本管理器：

工具语言特点
nvmNode.js按项目写 .nvmrc，nvm use 自动切换
pyenvPythonshim 机制，拦截 python 命令后路由到对应版本
rbenvRubyshim 机制，与 pyenv 思路类似
goenvGopyenv 的 Go 移植版
jenvJava轻量，只管理 JDK，不涉及构建工具

每种工具一套语法，一套配置，脑子里的上下文切换成本很高。更麻烦的是，在 JVM 生态里，光管好 JDK 版本还不够——Maven、Gradle、Kotlin、Scala、Groovy 这些工具的版本同样需要在不同项目间切换。

认识 SDKMAN! ##

SDKMAN! 是目前 JVM 生态里维护得最好、使用最广泛的版本管理工具。它最初就是为了解决上面说的这些问题而设计的——一个工具，管理 JVM 生态中几乎所有 SDK 的版本。

支持哪些 Candidate？ ##

SDKMAN! 里每个可管理的软件叫一个 “candidate”，目前主流的 JVM 生态工具基本都覆盖了：

  • JDK 发行版：Oracle JDK、OpenJDK、GraalVM、Liberica、Zulu、Adoptium、Amazon Corretto、Microsoft、IBM Semeru 等 20+ 种
  • 构建工具：Maven、Gradle、Ant、sbt
  • 语言：Kotlin、Scala、Groovy
  • 框架：Spring Boot CLI、Micronaut、Quarkus CLI
  • 其他：JBang、VisualVM、Spark、Leiningen 等

设计理念 ##

SDKMAN! 融合了多个工具的优秀思路：

  • 借鉴 nvm 的 .nvmrc → 用 .sdkmanrc 文件让项目自动切换到指定版本
  • 借鉴 pyenv/rbenv 的理念 → 候选版本隔离在 ~/.sdkman/candidates/ 下
  • 借鉴 update-alternatives → sdk use / sdk default 灵活切换默认版本和临时版本

SDKMAN! 安装与使用 ##

安装 ##

一行命令搞定，macOS / Linux / WSL 通用：

curl -s "https://get.sdkman.io" | bash

脚本会自动把初始化逻辑写入 ~/.bash_profile（或 ~/.zshrc）：

export SDKMAN_DIR="$HOME/.sdkman"
[[ -s "$HOME/.sdkman/bin/sdkman-init.sh" ]] && source "$HOME/.sdkman/bin/sdkman-init.sh"

安装完成后，重新打开终端或执行 source 即可使用。

注意：sdk 是一个 shell 函数，不是二进制可执行文件，所以 which sdk 查不到它。用 type sdk 可以验证。

常用命令速览 ##

# 查看所有可安装的 candidate
sdk list

# 查看 Java 可用版本
sdk list java

# 安装指定版本
sdk install java 17.0.9-tem
sdk install maven 3.9.6
sdk install gradle 8.5

# 切换当前 shell 的版本（临时生效）
sdk use java 21.0.1-graal

# 设置全局默认版本
sdk default java 17.0.9-tem

# 查看当前使用的版本
sdk current java

# 查看已安装的版本
sdk list java | grep installed

# 卸载
sdk uninstall java 8.0.392-tem

# 升级已安装的 candidate
sdk upgrade java

# 查看所有已安装 candidate 的当前版本
sdk current

项目级版本锁定：.sdkmanrc ##

SDKMAN! 支持用 .sdkmanrc 文件锁定项目的 SDK 版本，团队协作时特别有用：

# 在项目根目录打开 sdk env 配置
sdk env init

这会生成一个 .sdkmanrc 文件：

# Enable auto-env through the sdkman_auto_env config
# Add key=value pairs of SDKs to use below
java=17.0.9-tem
maven=3.9.6

之后 cd 到项目目录，SDKMAN! 会自动切换到文件中声明的版本（需要在配置中启用 sdkman_auto_env=true）。

配置 ##

配置文件在 ~/.sdkman/etc/config，几个实用的选项：

# 自动响应（跳过交互提示）
sdkman_auto_answer=true

# 进入目录时自动切换版本
sdkman_auto_env=true

# 开启原生二进制模式（更快）
sdkman_native_enable=true

SDKMAN! vs jenv ##

这两个工具经常被放在一起比较，但它们的定位其实差异很大：

SDKMAN!jenv
管理范围JDK + Maven + Gradle + Kotlin + …仅 JDK
JDK 安装内置下载安装，无需手动下载需要手动下载 JDK，jenv 只管注册和切换
体积更重，功能更多轻量，只做一件事
安装方式curl 一键安装Homebrew 或 git clone

简单来说：

  • SDKMAN! 适合 JVM 生态的全面管理，一揽子解决 JDK + 构建工具 + 其他语言版本的问题。功能更强但更"重"。
  • jenv 像一个轻量级的 JDK 路由器，你手动装好多个 JDK 然后告诉它路径，它帮你切换 JAVA_HOME 和 PATH。适合只需要管 JDK、习惯手动安装、偏好极简方案的人。

跨语言生态的展望 ##

聊完了 JVM 生态的方案，自然会想到一个问题：有没有一个工具能管所有语言的 SDK？

asdf 是这个方向的代表性尝试。它通过插件机制支持 Ruby、Node.js、Python、Go、Java 等数百种工具。但它也存在一些问题：

  • Shim 机制的性能开销：每次执行命令都需要经过 shim 路由，对频繁执行的命令（如 go、node）影响不可忽略。
  • 插件质量参差不齐：核心插件维护较好，但冷门插件可能已经停更。
  • 配置不如原生工具自然：比如 .tool-versions 的语法比 .nvmrc 或 .sdkmanrc 更通用但也更抽象。

mise（前身是 rtx）是 asdf 的 Rust 重写版，在性能上有所改善，并且兼容 asdf 插件生态。但目前社区规模和各语言原生工具的差距仍然很大。

从实用主义的角度出发，我的建议是：

  • JVM 生态 → 用 SDKMAN!，它就是这个领域的 nvm + rbenv + homebrew，一体化的体验足够好。
  • Node.js → 继续用 nvm 或 fnm（Rust 写的更快版本）。
  • Go → 官方 go install 的多版本管理已经很方便了，不一定需要第三方工具。
  • Python → pyenv 或直接用 venv/conda。

一个统一的跨语言版本管理器目前看来还不太现实——每种语言的打包、构建、版本语义都不同，统一抽象必然会损失细节和体验。目前"每个生态用自己最好的工具"反而是最务实的方案。

小结 ##

SDKMAN! 是 JVM 生态下最成熟的版本管理方案。相比 Linux 时代手动操作 update-alternatives，或者在各语言之间用不同的 *env 工具来回切换，一套 sdk 命令统一管理 JDK、Maven、Gradle、Kotlin 等所有 JVM 相关工具，体验好了很多。

如果你也久违地要搭建 Java 开发环境，SDKMAN! 是目前最省心的起点。
