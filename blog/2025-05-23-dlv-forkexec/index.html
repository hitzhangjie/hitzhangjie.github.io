Delve 如何处理 fork/exec 时的断点继承

Posted 2025-05-23 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 ##

调试多进程程序时，子进程在 fork/exec 之后还能否继承父进程的断点？直觉上，由于 fork 是写时复制（Copy-on-Write），代码段是共享的，断点指令似乎应该自动保留。但实际情况比这复杂，本文从物理内存和调试器架构两个层面来分析。

断点的物理实现 ##

先回顾一下断点是如何工作的。软件断点的实现方式是在目标代码段中插入特殊指令——在 x86 架构上通常是 int3（0xCC）。当 CPU 执行到这条指令时，会触发调试异常，内核将控制权交给调试器（tracer）。

因此，断点在内存中的存在形式就是：代码段中被替换为 int3 的若干字节。

关键前提：page cache 共享 vs 进程地址空间 ##

在讨论 fork 和 exec 的行为之前，必须区分两个容易混淆的概念：

层面是否共享说明
内核 page cache✅ 是同一个 inode 的文件内容在内核中只缓存一份
进程页表映射❌ 否每个进程有自己的页表，将虚拟地址映射到物理页

断点写入 int3 时发生的是：调试器修改进程 text 段的某个字节 → CPU 尝试写入 → 触发 CoW (Copy-on-Write) 缺页异常 → 内核为该进程分配一个新的私有物理页，复制原始内容，然后修改。

这个含 int3 的页是该进程的私有副本，不影响内核 page cache 中的原始页，也不影响其他映射同一文件页的进程。

fork 时的断点继承 ##

物理层面：继承，因为 CoW 复制的是父进程的私有页 ##

当进程调用 fork() 时，内核路径如下：

do_fork()
  └── copy_process()
       └── copy_mm()
            └── dup_mmap()              // 复制父进程的全部 VMA
                 └── copy_page_range()  // 建立 CoW 映射

子进程继承的是父进程的页表——即父进程已经 CoW 过的私有物理页，不是重新从内核 page cache 映射。因此：

  • 父进程 text 段中的 int3（在私有页上）会被子进程直接继承
  • 子进程在相同断点位置也包含了 int3 指令，命中时同样触发 SIGTRAP

一图对比 fork 和 exec 的映射来源：

                    磁盘 binary
                        │
                        ▼
                内核 page cache
               (只读，原始指令)
               /            \
        exec()启动，text映射到page cache，无0xCC
              /               \
    写断点CoW /                 \ 写断点CoW
            ▼                   ▼
      父进程A私有页            父进程B私有页
       (含 0xCC)               (含 0xCC)
          │

          |                    
          ▼                    
       子进程A2  →  映射到父进程A私有页，包括修改后的断点指令
      (含 0xCC)    ✅ 继承断点

fork 和 exec 的关键差异：

forkexec
映射来源父进程的私有页（含 int3）内核 page cache（原始干净页）
断点物理继承✅ 是❌ 否
原因dup_mmap 复制页表exec_mmap 新建 mm_struct，elf_map 重新映射

调试器层面：仍不可见 ##

即使物理上继承了 int3，fork 之后只有父进程是原始 tracer 的 tracee。子进程需要被 tracer 显式附加（或通过 PTRACE_O_TRACEFORK 等选项自动跟踪）才能让调试器感知到它的存在。

所以 fork 场景的结论是：物理断点被继承了，但调试器不知道子进程的存在。

exec 时的断点继承 ##

execve() 的行为与 fork() 完全不同——它完全替换进程的地址空间。内核路径：

do_execve()
  └── exec_binprm()
       └── load_elf_binary()
            ├── flush_old_exec()
            │    └── exec_mmap()    // 分配全新 mm_struct，释放全部旧 VMA
            └── elf_map()           // 从 page cache 重新映射各段

关键点：elf_map() 从内核 page cache 建立映射，完全不走父进程的私有页。因此即使 exec 的是同一个 binary（比如 exec("/proc/self/exe", ...)），新进程拿到的是 page cache 中的原始干净指令，不含任何 int3。

这也符合 exec 的语义——“干净的开始”：旧地址空间、信号处理、内存都是全新的。如果断点能跨 exec 继承，会带来严重的安全问题。想象一下：调试器在程序 A 的 text 段写入了 int3，然后一个完全不相关的进程 B 恰好 exec 了同一个 binary——进程 B 凭什么要在断点处停止？谁来处理这个 SIGTRAP？没有 tracer 的话，进程直接被信号杀死。同一个 binary 的多个实例之间断点隔离，是正确且必要的。

对于 exec，没有任何捷径——所有断点必须重新设置。

Delve 的处理机制 ##

Delve 通过 TargetGroup 机制 统一管理 这个流程。当 fork/exec 产生新的目标进程时，核心逻辑在 addTarget 中：

func (grp *TargetGroup) addTarget(p ProcessInternal, pid int, currentThread Thread,
    path string, stopReason StopReason, cmdline string) (*Target, error) {
    // ...
    t.Breakpoints().Logical = grp.LogicalBreakpoints
    for _, lbp := range grp.LogicalBreakpoints {
        if lbp.LogicalID < 0 {
            continue
        }
        err := enableBreakpointOnTarget(t, lbp)
        // ...
    }
    // ...
}

关键步骤：

  1. 继承逻辑断点：将 TargetGroup 中维护的 LogicalBreakpoints 赋值给新 target
  2. 逐个应用：遍历所有逻辑断点，调用 enableBreakpointOnTarget 将其写入新进程的代码段
  3. 处理条件断点：如果断点带有条件（condition）、命中计数（hit count）等属性，也在这一步中正确初始化

为什么需要重新设置 ##

即使 fork 场景下 int3 指令会被物理继承，Delve 仍然选择为每个新 target 显式重新设置断点。原因有几点：

  • 断点数据结构的完整性：Delve 内部维护的逻辑断点 ↔ 物理断点映射需要更新，断点命中计数、条件表达式等状态需要在新 target 上正确初始化
  • 地址空间可能变化：fork 后如果发生了 exec，地址空间完全不同，必须重新计算断点地址
  • 统一流程：无论 fork 还是 exec，用同一套 addTarget → enableBreakpointOnTarget 流程，简化代码逻辑

总结 ##

场景物理断点指令调试器感知Delve 的处理
fork写时复制，int3 被继承子进程需要被 tracer 跟踪通过TargetGroup.addTarget 重新应用逻辑断点
exec地址空间替换，全部丢失新进程无断点完全重新设置所有断点

核心结论：即使 fork 在物理上继承了断点指令，调试器层面仍然需要重新应用断点，以维护内部状态的一致性。Delve 通过 TargetGroup 和 LogicalBreakpoints 机制，在 fork/exec 发生时自动完成断点的继承与重设，对外暴露为一致的调试体验。
