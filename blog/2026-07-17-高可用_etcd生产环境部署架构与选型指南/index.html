etcd 生产环境部署架构与选型指南 ##

概述 ##

本规范旨在指导生产环境 etcd 集群的拓扑规划与客户端接入架构选型，以规避节点重启、扩缩容等日常维护对上游客户端造成的直接断连影响。

1. 物理层与拓扑规范 ##

  • 奇数节点原则： 生产环境必须部署奇数个节点（推荐 3 或 5 节点）。偶数节点不仅无法提升容灾能力，反而提高了投票共识（Quorum）门槛。
  • 磁盘 IO 隔离： 必须使用高 IOPS 的 SSD 或 NVMe 磁盘。etcd 对 fsync 延迟极其敏感，严禁与其他高 I/O 应用共用磁盘。
  • 网络网络平面隔离： 将 etcd Peer（集群内部 Raft 通信，默认 2380 端口）与 Client（客户端流量，默认 2379 端口）通过不同网卡或独立网段隔离，防止客户端大流量查询（如全量 List）导致集群内部失联。

2. 接入架构对比与选型 ##

目前业界主流的客户端接入方案有以下三种，推荐根据业务场景进行选型：

方案 A：gRPC 客户端直连多 Endpoint（官方推荐） ##

  • 实现方式： 将集群所有真实 IP 注入客户端参数（如 kube-apiserver 的 --etcd-servers=ip1,ip2,ip3）。
  • 优缺点： 架构最轻量，利用 gRPC 智能负载均衡，断连秒级感知，无单点故障；但运维时变更 IP 需要重启上游客户端。

方案 B：gRPC + DNS A 记录轮询（云原生推荐 ✨） ##

  • 实现方式： 注册内部域名（如 etcd.internal），将所有节点 IP 配置为该域名的 A 记录。客户端配置 dns:///etcd.internal:2379。
  • 运行机制： gRPC 内置的 DNS Resolver 会拉取所有 IP，并在底层与每一个真实节点同时建立物理长连接（SubChannel）。
  • 优缺点： 完美兼顾了“高可用”与“解耦”。某个节点重启时，客户端毫秒级自动切到其他已建连的通道，业务零感知，且增删节点只需改 DNS。

方案 C：四层负载均衡器（CLB/SLB）模式 ##

  • 实现方式： 客户端统一连接 CLB VIP，由 CLB 转发给后端 2379 端口。
  • 死穴： 长连接保持与健康检查延迟的天然冲突。gRPC 是强长连接，客户端只会与 CLB 建立极少连接并落在一台物理机上。若该节点重启，CLB 被动健康检查有秒级滞后，此期间请求会持续失败；且新连接重建时极易再次路由到正在重启的节点。
  • 避坑指南： 必须使用 L4（TCP 转发），绝对禁止使用 L7（HTTP 转发）；且运维时必须引入主动摘除机制。
