文件锁实现：从建议性锁到强制性锁

Posted 2026-07-18 12:00 +0800 by ZhangJie ‐ 5 min read

分享:  

背景 #

在多进程/多线程编程中，当多个执行流需要访问共享资源时，同步互斥是绕不开的话题。常见的同步机制有互斥锁（mutex）、信号量（semaphore）、条件变量等，这些适用于同一台机器上的线程或进程间同步。而当我们需要跨进程、甚至跨主机的同步时，文件锁（file lock）便成了一种简单有效的选择。

文件锁的实现方式有多种，从最简单的基于 open() 的排他性创建，到基于 flock() / fcntl() 的内核级锁机制。本文将系统性地梳理 Linux 下的文件锁实现方案，并深入讨论建议性锁（advisory lock）与强制性锁（mandatory lock）的区别。

实现方式一：open O_CREAT|O_EXCL #

原理 ##

这是最"原始"的文件锁实现方式，利用的是文件系统本身的原子性语义：

  • 调用 open(path, O_CREAT | O_EXCL, ...) 时，内核会原子性地检查文件是否存在：
    • 若文件不存在，则创建文件并返回 fd —— 视为获取锁成功；
    • 若文件已存在，则返回 -1，errno 设为 EEXIST —— 视为锁已被他人持有。
  • 解锁时，直接 unlink() 删除该文件，释放锁。

O_CREAT | O_EXCL 的组合保证"检查-创建"是一个原子操作，不会出现 TOCTOU（time-of-check-time-of-use）竞态问题。

代码示例 ##

#include <sys/types.h>
#include <sys/stat.h>
#include <errno.h>
#include <fcntl.h>
#include <unistd.h>
#include <iostream>

#define LOCKFILE "/tmp/tmp.lock"

namespace FileLockExcl {

class FileLock {
public:
    static int lock()
    {
        int tempfd;
        // 循环重试，直到成功创建锁文件
        while ((tempfd = open(LOCKFILE, O_CREAT | O_RDWR | O_EXCL,
                              S_IRUSR | S_IWUSR)) < 0) {
            if (errno != EEXIST) {
                std::cerr << "Failed to create lockfile: " << LOCKFILE << std::endl;
                return -1;
            }
            // 文件已存在，说明锁被他人持有，忙等待
        }
        close(tempfd); // 创建成功后立即关闭，锁文件的存在性即代表锁的状态
        return 0;
    }

    static int unlock()
    {
        return unlink(LOCKFILE); // 删除文件即释放锁
    }
};

} // namespace FileLockExcl

优缺点分析 ##

优点：

  • 实现简单，不依赖任何特殊的系统调用；
  • 跨平台性好（POSIX 标准行为）；
  • 锁文件的存在性直观可见（ls /tmp/tmp.lock 即可判断）。

缺点：

  • 忙等待（busy-wait）浪费 CPU；
  • 锁文件操作涉及实际的文件系统 I/O 和磁盘驱动，性能较低（benchmark 约 40 万次 lock/unlock 每秒）；
  • 进程异常退出时锁文件可能残留（未正常 unlink），需要额外的清理机制；
  • 不支持超时、不支持读锁/写锁分离、不支持范围锁；
  • 锁粒度只能是整个文件，无法锁定部分区间。

实现方式二：flock() — BSD 风格文件锁 #

原理 ##

flock() 是 BSD 引入的文件锁系统调用，它对整个文件加锁，锁与文件描述符（fd）关联：

#include <sys/file.h>
int flock(int fd, int operation);

operation 可选值：

操作说明
LOCK_SH共享锁（读锁），多个进程可同时持有
LOCK_EX排他锁（写锁），同时只能有一个进程持有
LOCK_UN解锁
LOCK_NB非阻塞模式（与上述任一值按位或），若不指定则默认阻塞等待

关键语义：

  • flock() 的锁与 file description（内核中的打开文件表项）关联，而非与进程或 fd 整数关联；
  • 这意味着 dup()、fork() 产生的 fd 副本共享同一把锁；
  • 进程退出时自动释放所有 flock 锁；
  • 同一个进程对同一个文件多次调用 flock()，后续调用会变更锁类型（如从 LOCK_SH 升级为 LOCK_EX，或反之）。

代码示例 ##

#include <sys/types.h>
#include <sys/stat.h>
#include <sys/file.h>
#include <errno.h>
#include <fcntl.h>
#include <unistd.h>
#include <cstdio>

#define LOCKFILE "filelockv2.lock"

namespace FileLockFlock {

class FileLockV2 {
public:
    static int lock_fd;

    static int init()
    {
        // init 阶段创建锁文件（非锁操作本身）
        int fd = open(LOCKFILE, O_CREAT | O_RDWR, S_IRUSR | S_IWUSR);
        if (fd < 0) {
            printf("failed to create lockfile.\n");
            return -1;
        }
        close(fd);
        return 0;
    }

    static int lock()
    {
        lock_fd = open(LOCKFILE, O_RDONLY);
        if (lock_fd < 0) return -1;
        flock(lock_fd, LOCK_EX); // 排他锁，默认阻塞等待
        return 0;
    }

    static int unlock()
    {
        flock(lock_fd, LOCK_UN);
        close(lock_fd);
        return 0;
    }

    static int fini()
    {
        return unlink(LOCKFILE);
    }
};

int FileLockV2::lock_fd = 0;

} // namespace FileLockFlock

性能与内核实现 ##

flock() 的操作主要发生在内核内存中（对 file table entry 中的锁字段做标记/清除），不涉及实际的磁盘 I/O。benchmark 显示约 100 万次 lock/unlock 每秒，是 O_EXCL 方案的 2.5 倍。

优缺点分析 ##

优点：

  • 性能远优于文件系统级别的实现；
  • 支持共享锁/排他锁，语义更丰富；
  • 支持阻塞等待（无需忙轮询）；
  • 进程退出时自动释放，无残留问题。

缺点：

  • 锁粒度只能是整个文件，无法锁定特定字节范围；
  • 锁与 file description 关联，fork() 后父子进程共享同一把锁（这在某些场景下可能不是期望的行为）；
  • 可移植性受限：flock 是 BSD 特性，虽然 Linux 也支持，但在某些 Unix 系统上行为可能有差异。

实现方式三：fcntl() — POSIX 风格记录锁 #

原理 ##

fcntl() 提供了 POSIX 标准的文件锁接口，支持按字节范围加锁（record locking），是最灵活的文件锁实现：

#include <fcntl.h>
int fcntl(int fd, int cmd, ... /* struct flock *arg */);

struct flock 定义：

struct flock {
    short l_type;   // 锁类型：F_RDLCK（读锁）、F_WRLCK（写锁）、F_UNLCK（解锁）
    short l_whence; // 偏移基准：SEEK_SET、SEEK_CUR、SEEK_END
    off_t l_start;  // 起始偏移
    off_t l_len;    // 锁定长度，0 表示锁定到文件末尾
    pid_t l_pid;    // 持有锁的进程 PID（仅 F_GETLK 时填充）
};

cmd 可选值：

命令说明
F_SETLK非阻塞设置锁；若冲突则立即返回-1，errno=EAGAIN/EACCES
F_SETLKW阻塞设置锁（W = wait）；若冲突则阻塞等待
F_GETLK查询当前锁状态：若已有冲突锁，则通过flock 结构返回冲突信息

关键语义 ##

fcntl 锁与 进程 + inode 关联，而非 file description。这与 flock 的核心区别在于：

  • 同一个进程对同一个文件的不同 fd 加锁，后一次会覆盖前一次（因为是同一个进程）；
  • 进程退出时自动释放所有 fcntl 锁；
  • fork() 产生的子进程不继承父进程的 fcntl 锁；
  • 如果一个 fd 被 close()，该进程通过任意 fd 在该文件上持有的所有 fcntl 锁都会被释放。

代码示例 ##

#include <fcntl.h>
#include <unistd.h>
#include <cstdio>
#include <cerrno>

int fcntl_write_lock(int fd, off_t start, off_t len)
{
    struct flock fl;
    fl.l_type   = F_WRLCK;   // 写锁（排他锁）
    fl.l_whence = SEEK_SET;
    fl.l_start  = start;
    fl.l_len    = len;       // 0 表示锁定到文件末尾

    return fcntl(fd, F_SETLKW, &fl); // 阻塞等待
}

int fcntl_read_lock(int fd, off_t start, off_t len)
{
    struct flock fl;
    fl.l_type   = F_RDLCK;   // 读锁（共享锁）
    fl.l_whence = SEEK_SET;
    fl.l_start  = start;
    fl.l_len    = len;

    return fcntl(fd, F_SETLKW, &fl);
}

int fcntl_unlock(int fd, off_t start, off_t len)
{
    struct flock fl;
    fl.l_type   = F_UNLCK;
    fl.l_whence = SEEK_SET;
    fl.l_start  = start;
    fl.l_len    = len;

    return fcntl(fd, F_SETLK, &fl);
}

// 示例：对文件的前 1024 字节加写锁
int main()
{
    int fd = open("data.bin", O_RDWR | O_CREAT, 0644);
    if (fd < 0) { perror("open"); return 1; }

    if (fcntl_write_lock(fd, 0, 1024) < 0) {
        perror("fcntl write lock");
        return 1;
    }

    // ... 临界区操作 ...

    fcntl_unlock(fd, 0, 1024);
    close(fd);
    return 0;
}

建议性锁 vs 强制性锁 #

两种方式的共同点：都是建议性锁 ##

上述三种实现方式——O_EXCL、flock()、fcntl() 默认情况下实现的都是建议性锁（advisory lock）。

建议性锁的核心特征是：内核只跟踪锁的状态，不会在 I/O 操作时主动检查锁。

也就是说，如果进程 A 通过 flock(fd, LOCK_EX) 对文件加了一把排他锁，进程 B 仍然可以完全不调用 flock，直接 read() / write() 操作该文件而不会被内核阻止。锁的语义完全依赖于所有参与进程都自觉、正确地在读写前进行加锁。

打一个比方：建议性锁就像厕所门上的"有人/无人"标识牌。如果你不检查标识直接推门进去，标识牌本身不会阻止你。只有当每个人都遵守"先看标识、再进门、出来翻牌"的约定时，系统才能正常工作。

这也是为什么文件锁被称为"建议性"的原因：内核建议你在操作前先获取锁，但不强制。

强制性锁（Mandatory Lock） ##

如果需要更严格的保护——即使某个进程不配合加锁，内核也能阻止其非法读写——那就需要强制性锁。

启用条件 ##

在 Linux 上启用强制性锁需要同时满足两个条件：

  1. 文件系统层面：挂载时加上 mand 选项

    # 挂载时启用强制性锁支持
mount -o mand /dev/sdb1 /mnt/data

# 或在 /etc/fstab 中配置
# /dev/sdb1  /mnt/data  ext4  defaults,mand  0  0

  2. 文件权限层面：要对目标文件设置 set-group-ID 位，并清除 group-execute 位

    chmod g+s,g-x /path/to/file
# 即权限位显示为 --s—— （group 位显示为 s 而非 x 或 S）

为什么要同时置 setgid、清 group-execute？

set-group-ID 位与 group-execute 位的这种组合（------s--- 即 0o2000 + ~0o0010）是一个特殊的标记。历史上 Linux 用这个组合来标识文件应受强制性锁保护，因为正常情况下一个有 setgid 的文件通常也应该有 group-execute 权限（--x--s--x），这个非正常组合不会与常规用法冲突。

强制锁的行为 ##

当强制性锁定启用后，内核会在每次 read() / write() 时主动检查文件上的锁：

操作对方持有读锁（F_RDLCK）对方持有写锁（F_WRLCK）
本方尝试read()✅ 允许（读锁兼容读）⛔ 阻塞（读写互斥）
本方尝试write()⛔ 阻塞（读写互斥）⛔ 阻塞（写写互斥）

如果调用时使用了 O_NONBLOCK，则冲突时会立即返回 EAGAIN 而非阻塞。

强制性锁的局限性 ##

  1. 只适用于 fcntl POSIX 锁：flock() 的 BSD 锁无法升级为强制性锁。
  2. 需要文件系统支持：仅适用于支持 mand 挂载选项的文件系统（ext4、XFS 等），NFS 等网络文件系统不支持。
  3. Truncate 不受控制：truncate() / ftruncate() 不会被强制性锁阻止。
  4. unlink / rename 不受控制：删除或重命名文件不会检查强制性锁。
  5. root 进程豁免：以 root 身份运行的进程不受强制性锁限制（但在某些配置下可能受影响）。
  6. 管理成本高：需要协调挂载选项和文件权限，不适用于任意目录下的临时文件。

实用建议：在实践中，强制性锁并不常用。大多数场景下，只要确保所有访问方遵循统一的加锁协议，建议性锁已足够。强制性锁主要适用于以下场景：

  • 你无法控制所有访问进程的代码（如第三方程序）；
  • 需要防范疏漏导致的写冲突；
  • 安全敏感场景，需要内核级别的访问控制兜底。

总结 #

本文梳理了 Linux 下三种文件锁的实现方式：

  1. open() + O_CREAT|O_EXCL：利用文件系统原子语义，最简单但也最受限。适用于只需"单实例运行"这类简单互斥场景。
  2. flock()：BSD 风格的整文件锁，与 file description 关联，性能较好。适用于需要读写锁但不关心锁范围的场景。注意 fork() 的继承语义。
  3. fcntl()：POSIX 风格的记录锁，与进程关联，支持灵活的范围锁。适用于需要精细控制锁粒度的场景，也是唯一可升级为强制性锁的方式。

关于建议性锁与强制性锁：

  • 上述三种方式默认都是建议性锁：内核只记录锁状态，不强制 I/O 检查。一致性完全依赖于所有进程自觉遵守加锁协议。
  • 强制性锁在满足特定条件后（挂载 mand + 文件 setgid 置位且 group-execute 清位），内核会在每次 read()/write() 时主动检查 fcntl 锁，未获锁的进程将被阻塞。但由于其配置复杂、限制较多，实践中不如建议性锁普遍。

选择哪一种方式，取决于你的具体需求：锁的粒度、是否需要进程间安全的释放语义、是否需要在不可控的进程中强制执行。理解每种方式的原理和局限，才能做出正确的选择。
