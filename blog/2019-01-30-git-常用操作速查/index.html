Git 常用操作速查 #

本文整合了日常开发中高频使用的 Git 命令，按使用场景分类，供快速查阅。

一、基础概念 ##

三种文件状态 ##

状态说明
committed已提交到本地.git 仓库，安全存储
modified已修改但未 stage（暂存）
staged已暂存，等待 commit
  • Staged 状态存储在 .git/index 文件中
  • 日常使用中可以直接跳过 staged 状态（git commit -a）

两种追踪状态 ##

  • tracked：已被 commit 或 stage 的文件
  • untracked：新创建、未被 Git 追踪的文件

Git 与其他 VCS 的核心差异 ##

  • 快照而非差异：Git 将仓库视为小型文件系统，以快照形式记录变化。未改变的文件直接引用之前的版本。
  • 离线工作：clone 后包含完整历史，查看日志、提交、创建分支等均可离线操作。
  • 数据完整性：底层通过校验和（SHA-1）保证数据不被篡改或传输错误。
  • 操作可恢复：Git 的操作几乎都是添加操作，提交过的内容很难丢失。

二、配置（git config） ##

配置文件层级（优先级从高到低） ##

  1. 仓库级：.git/config
  2. 用户级：~/.gitconfig 或 ~/.config/git/config
  3. 系统级：/etc/gitconfig

常用配置 ##

# 用户信息（必须设置）
git config user.name "Your Name"
git config user.email "your@email.com"

# 编辑器
git config core.editor /usr/bin/vim

# 查看所有配置
git config --list

别名（alias） ##

git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status

也可以直接编辑 ~/.gitconfig：

[alias]
    co = checkout
    br = branch
    ci = commit
    st = status

实用配置 ##

# 跨平台换行符处理（Linux/Mac 推荐 input）
git config --global core.autocrlf input

# 忽略文件权限变化（chmod）
git config --global core.filemode false

# 解决中文编码问题
git config core.quotepath false
git config --global gui.encoding utf-8
git config --global i18n.commitEncoding utf-8
git config --global i18n.logOutputEncoding utf-8

三、基本操作 ##

clone vs checkout ##

  • git clone：克隆完整仓库（含所有分支、标签、历史），等同于 svn 的 checkout
  • git checkout：在已有仓库中切换分支或恢复文件
# HTTPS 协议
git clone https://github.com/user/repo.git [dirname]

# SSH 协议
git clone git@github.com:user/repo.git

文件状态查看 ##

git status          # 详细状态
git status -s       # 简要状态（M=修改, ??=未追踪）

git add / git stage ##

# git add 的三重用途
git add <file>          # 1. 将 untracked 文件加入追踪
git add <file>          # 2. 将修改后的文件放入 staged
git add <file>          # 3. 标记冲突已解决

# git stage 仅用于将修改后的文件放入 staged
git stage <file>

git diff ##

git diff              # 未 staged 与 staged 的比较
git diff --staged     # 已 staged 与上次 commit 的比较
git diff --cached     # 同 --staged
git diff HEAD         # 当前文件与上次 commit 的比较

git commit ##

git commit -m 'message'              # 快速提交
git commit                           # 调用编辑器填写提交信息
git commit --amend -m 'new message'  # 修订上次提交信息（未 push 时有效）

git reset ##

git reset <file>     # 撤销 stage（将文件从 staged 移回 modified）
git reset --hard <commit>  # 回滚到指定 commit，丢弃所有后续修改

git mv ##

git mv <old> <new>   # 重命名文件，被 Git 识别为重命名操作

四、分支操作 ##

# 创建并切换到新分支
git checkout -b <branch>

# 切换分支
git checkout <branch>

# 删除本地分支
git branch -D <branch>

# 删除远程分支
git push <remote> --delete <branch>

# 推送新分支到远程
git push --set-upstream origin <branch>

# 同步远程已删除的分支信息
git fetch --prune

五、日志与历史 ##

查看日志 ##

git log --stat                          # 显示文件变更统计
git log --pretty=oneline                # 单行模式
git log --graph                         # 图形化分支历史
git log --pretty=format:"%h - %an, %ar : %s"  # 自定义格式

过滤日志 ##

git log --author=<name>                 # 按作者过滤
git log --since=<date> --before=<date>  # 按时间范围过滤
git log --no-merges                     # 排除 merge 提交

推荐的自定义 log 格式 ##

git log --abbrev-commit --graph \
    --pretty=format:'%Cgreen%h %C(bold blue)%cr %Creset>>> %C(bold yellow)%<(78,trunc)%s %Creset<<< %Cred<%an>%Creset'

按内容搜索提交 ##

# 查找添加或删除了指定字符串的 commit（非常实用）
git log -S <string>

按message搜索提交 ##

git log --grep <string>

查看指定 commit 详情 ##

git show <commit>

分支间对比 ##

# 查看 release 有但 master 没有的 commit
git log master..release

六、合并与变基 ##

git merge ##

# 将 dev 合并到 master
git checkout master
git merge dev

git rebase ##

场景 1：压缩多次提交（squash）

git rebase -i <commit-hash>
# 在弹出的编辑器中，将后续 commit 的 pick 改为 squash
# pick hash1
# squash hash2
# squash hash3

场景 2：同步上游变更

# 在 dev 分支上，将 master 的更新同步过来
git rebase master

# 对比 merge 方式（会在 dev 日志中穿插 master 的提交），
# rebase 可以保持 dev 的修改始终在日志顶部，提交历史更清晰

git cherry-pick ##

# 挑选单个 commit
git cherry-pick <commit>
git cherry-pick -x <commit> # 保留原提交commit，(cherrypicked from ...)

# 挑选连续范围的 commit（不包含 A）

git cherry-pick A..B

# 挑选连续范围的 commit（包含 A）

git cherry-pick A^..B

# 挑选多个不连续的 commit

git cherry-pick A1 A2 A3 B1 B2 B3

cherry-pick 合并提交（merge commit）

  • 合并提交有两个 parent：parent 1 是被合入的分支，parent 2 是合入来源的分支
  • 使用 -m 1 表示以 parent 1 为主线，应用来自 parent 2 的变更
git cherry-pick <merge-commit> -m 1

git patch（补丁） ##

# 为最近一次 commit 生成补丁
git format-patch -1 HEAD

# 为指定 commit 生成补丁
git format-patch -1 <sha>

# 应用补丁
git am < file.patch

适用场景：在错误分支上开发后，希望通过补丁方式将修改同步到正确分支，保持历史干净。

七、远程仓库 ##

# 查看远程仓库
git remote -v                    # 显示详细信息（fetch/push URL）
git remote show <remote>          # 查看远程仓库信息（会查询服务器）
git remote rename <old> <new>     # 重命名
git remote rm <remote>            # 删除

# 获取远程更新
git fetch <remote>               # 只获取不合并
git pull                         # 获取并合并（= fetch + merge）

# 推送
git push <remote> <branch>       # 正常推送
git push --force origin          # 强制推送（回滚后使用，慎用！）
git push <remote> <tag-name>     # 推送指定 tag
git push <remote> --tags         # 推送所有未推送的 tag

八、标签（tag） ##

两种类型 ##

类型特点
轻量级 tag相当于不可变的分支，不含描述信息
注解式 tag存储完整对象，包含描述信息（推荐）
# 创建注解式 tag（推荐）
git tag -a <version> -m 'description'

# 创建轻量级 tag
git tag <version>

# 查看 tag 详情
git show <version>

# 给历史提交补打 tag
git tag -a <version> -m 'description' <checksum>

# 基于 tag 创建分支
git checkout -b <branch> <tag>

九、.gitignore ##

# 基本模式
*           # 匹配任意文件
dir/        # 忽略整个目录
dir/*.txt   # 忽略目录中的 txt 文件
!file       # 不忽略指定文件（反转规则）

取消对已追踪文件的追踪 ##

# 保留文件，仅删除追踪信息
git rm <file> --cached

# 删除文件和追踪信息
git rm <file> -f

十、实用技巧 ##

修改历史提交的作者信息 ##

适用场景：提交时用错了用户名/邮箱（如公司项目误用了个人 GitHub 账号）。

警告：这会重写历史，协作场景下需谨慎使用。

# 克隆裸仓库
git clone --bare https://github.com/user/repo.git
cd repo.git

# 批量修改
git filter-branch --env-filter '
OLD_EMAIL="old@email.com"
CORRECT_NAME="Correct Name"
CORRECT_EMAIL="correct@email.com"

if [ "$GIT_COMMITTER_EMAIL" = "$OLD_EMAIL" ]
then
    export GIT_COMMITTER_NAME="$CORRECT_NAME"
    export GIT_COMMITTER_EMAIL="$CORRECT_EMAIL"
fi
if [ "$GIT_AUTHOR_EMAIL" = "$OLD_EMAIL" ]
then
    export GIT_AUTHOR_NAME="$CORRECT_NAME"
    export GIT_AUTHOR_EMAIL="$CORRECT_EMAIL"
fi
' --tag-name-filter cat -- --branches --tags

# 推送修正后的历史
git push --force --tags origin 'refs/heads/*'

从本地仓库重建远程仓库 ##

# 1. 复制 .git 目录内容到远程仓库目录
# 2. 删除 git init --bare 不会生成的文件（最终保留 3 文件 4 文件夹）
# 3. 设置允许向当前检出分支推送
git config receive.denyCurrentBranch ignore

恢复被误删的文件 ##

git checkout <file>     # 从版本库恢复文件（覆盖本地修改）

强制推送（回滚场景） ##

git reset --hard <commit>
git push --force origin

查看分支关系 ##

# 查看当前分支与远程分支的关系
git status

# 查看所有分支（含远程）
git branch -a
