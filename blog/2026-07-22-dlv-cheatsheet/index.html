dlv 调试速查表

Posted 2026-07-22 00:00 +0800 by ZhangJie ‐ 5 min read

分享:  

背景 #

我对 dlv 做过比较深入的源码分析，也写过一本关于调试器实现的书1。但日常使用层面，大部分需求 VS Code + dlv dap 已经覆盖了。真正需要裸调的时候往往是一些特殊场景——比如远程服务器上没装 IDE、需要脚本化批量设断点、需要追踪子进程、或者需要监控某个变量被谁写过。这些时候对命令结构的熟悉程度就决定了排查效率。

本文把 dlv 的命令分为两大部分：

  1. 启动命令——决定"怎么进入"调试会话
  2. 会话内命令——决定进入之后"怎么查、怎么走"

启动命令 #

启动命令是 dlv 的子命令，决定以什么方式启动一个调试目标。我把它们按使用频率重新排列了一下：

常用 ##

命令典型用法说明
dlv debugdlv debug ./cmd/server编译当前（或指定）package 并启动调试，日常最常用
dlv execdlv exec ./mybin调试已编译好的二进制，建议编译时加 -gcflags="all=-N -l" 关闭优化
dlv testdlv test ./pkg/repo -- -test.run TestXxx调试单元测试，-- 后面透传给 go test 的参数
dlv attachdlv attach 12345附加到已在运行的进程；--waitfor <前缀> 可等待进程出现

这四条覆盖了日常 90% 以上的调试场景。优先级是：debug > test > exec > attach。

特殊场景 ##

命令说明
dlv trace按函数名正则匹配设 tracepoint，输出函数调用信息后立即继续执行。不需要进入交互会话，适合快速了解哪些函数被执行、传了什么参数
dlv core分析 core dump，支持 Linux/amd64、Linux/arm64、Windows/amd64 minidump
dlv connect以终端客户端连接 headless 模式的 dlv 服务器
dlv dap启动 DAP 协议服务器——这是 VS Code 等 IDE 集成 dlv 的底层协议

关键的全局 flag ##

几乎所有子命令都共享以下 flag：

Flag作用
--headless以 headless server 模式运行，不进入交互终端，等待客户端连接
--accept-multiclient允许多个客户端同时连接
--listen :2345指定监听地址，默认127.0.0.1:0（随机端口）；unix: 前缀走 domain socket
--init <file>启动时自动执行文件中的 dlv 命令（批处理脚本）
--continue启动后不等待、直接 continue 直到遇到断点
--backend选择后端：default（macOS 上自动选 lldb，Linux 上 native）、native、lldb、rr
--build-flags透传给go build 的参数，如 -tags=integration -mod=vendor
--log --log-output debugger,rpc开启日志，可选的 log 组件包括debugger、gdbwire、lldbout、rpc、dap、fncall、stack 等
-r stdin:/tmp/in重定向目标进程的标准 IO

headless 模式的典型组合 ##

# 启动 headless 服务器，允许多客户端
dlv debug ./cmd/server --headless --accept-multiclient --listen :2345

# 另一个终端连接进来
dlv connect :2345

# IDE attach 到已有 headless 服务器（VS Code launch.json）
{
    "mode": "remote",
    "host": "127.0.0.1",
    "port": 2345
}

这是远程调试和容器内调试的基础。

会话内命令 #

进入 (dlv) 提示符后，命令按功能可以分为六组。下面逐一展开。

1. 执行控制 ##

命令别名作用
continuec继续执行直到下一个断点或程序结束
steps单步执行，进入函数
nextn单步执行，不进入函数
stepoutso跳出当前函数，回到调用方
step-instructionsi / stepi按 CPU 指令单步（进入 call）
next-instructionni / nexti按 CPU 指令单步（跳过 call）
restartr重启进程，断点保留
rebuild-重新编译并重启（仅限 dlv 自己编译的）
call-向进程注入一次函数调用（实验性）

指令级步进适合排查编译器代码生成问题，或者在反汇编视图下逐指令分析。

2. 断点 & 跟踪点 ##

这是 dlv 最强大也最是最常用的一组命令，但是大家往往只了解普通断点、条件断点，其实还有更加高级的“点”。

设置断点 ##

# 基本形式
break main.go:42           # 文件:行号
break main.go:42 if i > 100  # 条件断点
break myFunc               # 函数名
break main.(*Server).Start  # 带 receiver 的方法
break /regex/              # 正则匹配函数名，在所有匹配函数上设断点
break +3                   # 当前行偏移 +3
break namedBP main.go:42   # 给断点取名，方便后续引用

管理断点 ##

命令作用
breakpoints (bp)列出所有断点及其 ID、命中次数
condition <id> <expr> (cond)给已有断点设定条件
condition -hitcount <id> % 100每 100 次命中才停一次
condition -per-g-hitcount <id> == 5按 goroutine 各自计算命中次数
condition -clear <id>清除条件
toggle <id>开关断点（不影响其他属性）
clear <id>删除单个断点
clearall批量删除断点

tracepoint 和 watchpoint ##

trace main.go:55           # 设 tracepoint：命中时打印并继续执行，比断点更轻量
watch -w someVar           # 监控变量写入（-r 读，-w 写，-rw 读写）
watch -w *(*int)(0x1400007c018)  # 监控任意内存地址

watchpoint 是"谁改了这个值"的神器。某个字段莫名其妙被改了、找不出一条写入路径时，设一个 write watchpoint 命中的那一刻栈回溯就是答案。注意：值未变化的纯覆写可能不会被捕获。

on 命令——断点命中时自动执行 ##

on 2 print v           # 断点 2 命中时自动打印变量 v
on 2 stack             # 断点 2 命中时自动打印栈
on 2 goroutine         # 断点 2 命中时自动打印当前 goroutine
on namedBP cond i > 10 # 等价于 cond namedBP i > 10
on namedBP -edit       # 编辑断点命中时的命令列表

对于需要反复"停下 → 看一眼 → 继续"的循环内调试，on + condition 配合能省下大量手动操作。

3. 查看变量和内存 ##

命令别名关键说明
print <expr>p求值任意表达式，支持结构体字段、指针解引用、类型转换等
locals-打印当前栈帧所有局部变量（没有 -a flag）
args-打印当前函数参数
vars <package>-打印包级变量
display -a <expr>-每次 stop 自动打印表达式，可叠加多次
display -d <n>-移除第 n 个自动打印
set <var> = <val>-修改变量值，调试时很方便
whatis <expr>-打印表达式或类型的完整类型信息
examinemem <addr>x查看原始内存内容（hex dump）
regs-查看 CPU 寄存器值

几个实用技巧 ##

# 打印时指定格式
print %x v              # 十六进制
print %s v              # 字符串

# 打印 slice 前20个元素
print s[:20]

# 类型转换后取值
print (*reflect.rtype)(t).string

# display 管理
display -a v            # 添加 auto-print
display -a %x regs.rax  # 每次 stop 自动打印 rax 寄存器的十六进制值
display                 # 列出所有 display 项
display -d 1            # 删除第 1 项

4. goroutine 和线程 ##

命令别名作用
goroutinesgrs列出所有 goroutine 及运行状态（runnable/running/syscall/waiting 等）
goroutinegr查看当前 goroutine 信息
goroutine <id>gr切换到指定 goroutine
goroutine <id> <cmd>gr在指定 goroutine 的上下文中执行一条命令
threads-列出所有线程
thread <id>tr切换到指定线程

切换到目标 goroutine 后再 stack、locals、args，看到的就都是该 goroutine 的视角。

5. 调用栈 ##

命令别名作用
stackbt打印完整调用栈
stack 10bt打印前 10 帧
frame <n>-切换到第 n 帧
up-上移一个栈帧（往调用者方向）
down-下移一个栈帧（往被调用者方向）
deferred-在 defer 闭包上下文中执行命令

切换栈帧后再 locals 可以查看调用链上任意一层的局部变量，对于"值在什么环节变错"的排查非常有用。

6. 杂项（但常用的） ##

命令别名作用
list [locspec]ls / l显示当前位置或指定位置的源码
funcs [regex]-列出匹配正则的函数名
types [regex]-列出匹配正则的类型
sources [regex]-列出匹配的源文件
disassembledisass反汇编当前函数
source <file>-从文件加载并执行 dlv 命令。.star 后缀按 starlark（Python 方言）执行，支持编写自定义调试脚本2
config-查看或修改配置，如config max-string-len 200
config -list-列出全部配置项
config -save-保存当前配置到~/.dlv/config.yml
transcript <file>-将会话输出追加写入文件（调试记录留档用）
dump <file>-生成当前进程的 core dump，可后续dlv core 分析
target follow-exec-自动跟踪目标 fork/exec 出的子进程
target list/switch-多进程调试场景下列出和切换当前调试目标
libraries-列出已加载的动态库
edited用$DELVE_EDITOR 或 $EDITOR 打开当前所在位置

starlark 脚本能力速览 ##

source 命令加载 .star 文件时使用的是 starlark——一个受限的 Python 方言。在 starlark 脚本里可以调用所有 dlv 命令并获取返回值：

# mydebug.star — 示例：批量打印所有 goroutine 的栈
grs = dlv("goroutines")
for gr in grs:
    dlv("goroutine", gr.id, "stack 5")

除了一次性脚本，还可以通过 config alias 把自定义命令注册为快捷键2。

高级技巧速览 #

下面几个功能在日常中属于"不常用但关键时刻能救命"的类型，这里简要提及，后续会单独展开成文。

条件断点 + hit count

break main.go:100 if v != nil        # 条件断点
cond 3 -hitcount % 100               # 每100次才触发一次
cond 3 -per-g-hitcount == 5          # 按goroutine分别计数，第5次触发

watchpoint 定位"谁改了我"

watch -w sharedMap                   # 任何对 sharedMap 的写入都会被拦截

headless 模式远程调试

# 服务器端（容器内）
dlv exec --headless --accept-multiclient --listen :2345 ./app

# 本地连接
dlv connect <remote_ip>:2345

子进程跟踪

target follow-exec -on .*            # 跟踪所有 fork/exec 的子进程
target follow-exec -on "myworker"    # 仅跟踪命令匹配 myworker 的子进程
target list                          # 查看当前所有被跟踪的进程

on 命令减少手动操作

on 1 print req.URL                   # 每次命中断点1时自动打印请求URL
on 1 stack                           # 同时打印调用栈
on 1 goroutine                       # 同时打印goroutine信息

display 自动观察

display -a v                         # 每次stop自动打印v
display -a %x regs.rsi               # 每次stop自动打印rsi寄存器
display -d 1                         # 移除第1项

init 文件批量执行

# init.dlv
break main.go:42 if i > 100
break main.go:55
continue

# 启动时加载
dlv debug --init init.dlv

总结 #

把 dlv 命令拆成启动命令和会话命令两类之后结构就很清晰了：启动命令选了"怎么进来"，会话命令管"进来后怎么查"。大多数人的日常场景——设个断点、单步几下、看一眼变量——只在很小的子集里操作，但把整张表过一遍会发现在 tracepoint、watchpoint、on、starlark 脚本等几个方向上 dlv 的空间比想象中大得多。

后续我打算单独展开 watchpoint 和条件断点这两个最实用但文档又最薄弱的主题，用实际例子把排查流程串起来。
