Quasar：Java 上另一套 Fiber / 无栈协程方案

Posted 2026-07-19 17:37 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 #

Kilim 证明了：在不改 JVM 的前提下，靠 字节码编织 也能做出可暂停的轻量任务。同一时期（以及稍后）更广为人知的是 Parallel Universe Quasar——它把类似能力产品化成 Fiber API，并配套 Pulsar（Clojure 侧）、与 actor 库等生态。

若你已经读过 Kilim 一文，会感到「似曾相识」：都是 无栈（stackless）协程思路，都依赖 编译期/加载期改字节码，都要解决 可挂起方法传染。差异主要在 API 形态、instrumentation 工具链，以及与 Java 生态的集成方式。

本文从原理与用法两侧做技术介绍，并说明它在 Loom 时代的历史位置。

一、Quasar 是什么 #

Quasar 的目标可以概括成：

让开发者用 近似线程的编程模型（顺序代码、阻塞风格 API），却以 Fiber 作为调度单位，在阻塞点 释放底层 OS 线程，从而支撑极高并发。

核心组件概念：

概念含义
Fiber轻量「纤程」，用户态调度的执行单元
@Suspendable标记方法可挂起（类比 Kilim 的 Pausable）
Instrumentation构建期或 Java Agent 改写字节码，插入挂起/恢复逻辑
Scheduler把 Fiber 多路复用到少量 ForkJoin / 线程池上
Channel / Actor可选的通信与 actor 模型支持

官方叙事里常强调：Fiber 是 lightweight threads——对业务代码尽量像 Thread，对运行时则是 continuation + 调度。

二、和 Kilim 同一条技术路线 #

2.1 无栈，而不是每 Fiber 一块原生栈 ##

Quasar Fiber 不是 给每个任务 mmap 一块独立栈再 swapcontext。挂起时，运行时把 恢复所需状态 保存在堆上结构中；载体线程的 JVM 栈可被其他 Fiber 复用。这与 Kilim 的 Fiber/State 同属 stackless 一族。

对照：

方案标记可挂起改造手段通信
Kilimthrows PausableWeaver 织入Mailbox
Quasar@Suspendable 等instrumentation（含 agent）Channel / 集成库

2.2 为什么必须改字节码 ##

JVM 方法一旦调用进入，局部变量与操作数栈活在当前帧里；普通 return 会丢掉帧。要在「方法中间」让出执行权，必须：

  1. 在 挂起点 把活跃局部变量与恢复 pc 存起来；
  2. 从调用栈 展开 到调度器（语义上类似抛异常展开，但是可控的 continuation）；
  3. 再次调度时 重建 调用链或跳到保存的 continuation。

这无法单靠「写个库、不碰 class」优雅完成（除非像 Loom 那样改 JVM/运行时）。因此 Quasar 与 Kilim 一样选择 字节码增强。

2.3 可挂起性传染（The Contagion） ##

@Suspendable void handle() {
    byte[] data = socket.read();  // 若 read 可挂起
    process(data);
}

handle 要挂起，则它调用的可挂起路径上的方法通常也必须是 @Suspendable，并被正确 instrument。否则：

  • 未增强方法里发生的「阻塞」会 真的堵住载体线程；
  • 或工具链在检测时直接报错（取决于配置与检查严格程度）。

这是所有 无栈 + 显式 suspend 点 方案的共同税负；Go 的 goroutine（有栈 + 运行时协作/抢占）和 Java 虚拟线程（运行时 unmount）从语言/VM 层消解了这层税。

三、使用形态（概念级） #

下面用示意代码说明 API 直觉（细节以当时 Quasar 文档为准）：

// 启动一个 Fiber（示意）
new Fiber<Void>(() -> {
    while (true) {
        Message m = channel.receive(); // @Suspendable：可挂起，不绑死 OS 线程
        handle(m);
    }
}).start();

与 new Thread(...).start() 对比：

  • 写法：同样是「一段顺序逻辑」；
  • 代价：Fiber 创建与切换轻量得多，可开数量级更高的并发任务；
  • 约束：阻塞式 JDK API 并非自动安全——需要 已 instrument 的封装，或避免在 Fiber 里调用会钉死载体线程的代码。

集成方式通常包括：

  1. 构建期 instrumentation（Maven/Gradle 插件处理 class）；
  2. Java Agent 在加载时改写；
  3. 与 reactive / async 库桥接时的适配层。

调试、堆栈、与监控系统的集成，是这类方案的长期工程成本——栈轨迹往往是「逻辑栈 + 载体线程栈」的混合视图。

四、调度与阻塞语义 #

理想路径：

Fiber 运行在载体线程上
  → 遇到 channel 接收 / 异步 I/O 完成前
  → 保存 continuation，Fiber 挂起
  → 载体线程去跑别的 Fiber
  → 事件就绪后 Fiber 再被调度

危险路径：

Fiber 内调用未适配的阻塞系统调用 / synchronized 重锁等
  → 载体 OS 线程被占住
  → 吞吐塌方（类似虚拟线程的 pinning 问题，但机制不同）

因此生产实践里会强调：Fiber 里只使用协程友好的 I/O 与同步原语。这和写 Go 时「少用阻塞 cgo、注意 runtime 钩子」是同一类运维知识，只是 Java 侧更依赖库是否被织入。

五、生态与历史命运 #

5.1 曾经的位置 ##

在 Loom 之前，若要在 Java 里写「高并发同步风格」服务，选择大致是：

  • 线程池 + 有限并发；
  • 回调 / CompletableFuture / Reactive 流；
  • Kilim / Quasar 等织入式 Fiber。

Quasar 文档与社区一度更活跃，API 也更「现代化」，常被用作对比 Kotlin 协程、Go goroutine 的 Java 侧参照物。

5.2 与 Project Loom 的关系 ##

Project Loom 将 虚拟线程（Virtual Threads） 做进 JDK（预览后于 Java 21 转正）。虚拟线程由 JVM 调度，阻塞点可 unmount 载体线程，业务代码 largely 无需 @Suspendable 织入。

结果是：

  • 第三方字节码协程框架的 核心卖点被平台吃掉；
  • Quasar 等项目维护动力下降，逐步淡出主流选型；
  • 但 原理 仍值得学：理解 Loom 之前工业界如何「曲线救国」。

注意：虚拟线程在实现上更接近 可挂起的轻量线程（continuation + 栈段），与 Quasar 的「方法级无栈状态机」并不完全同构——对比文会细说。

六、何时还值得读 Quasar 源码 #

目的是否值得
新项目选型一般不：优先虚拟线程 / 成熟响应式栈
理解 instrumentation 协程是
对比有栈/无栈是（作为 Java 无栈代表）
维护遗留 Quasar 系统是，并评估迁 Loom

阅读路径建议：

  1. Fiber 生命周期与调度队列；
  2. @Suspendable 在字节码中的落地（挂起点、栈展开）；
  3. Channel 实现与 park/unpark；
  4. 与 ForkJoinPool 的关系。

总结 #

Quasar 与 Kilim 同属 Java 无栈协程 / Fiber 路线：

  • 用注解或签名标记可挂起方法；
  • 用 字节码 instrumentation 插入保存/恢复逻辑；
  • 用用户态调度把海量 Fiber 映射到少量 OS 线程；
  • 用 Channel 等做任务间通信。

它比回调模型更利于表达复杂业务控制流，但付出 工具链、传染性、与未增强库互操作 的成本。随着 Java 21+ 虚拟线程 成为官方方案，Quasar 更多作为 原理教材与历史坐标 存在。

同系列建议阅读顺序：Kilim（织入细节）→ 本文（Fiber API）→ Java 虚拟线程（平台方案）→ 有栈/无栈对比（全局图景）。
