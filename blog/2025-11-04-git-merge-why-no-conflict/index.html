为什么 git merge 不报冲突？

Posted 2025-11-04 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

两个分支在同一个文件的同一区域各自加了新行，merge 之后两边的改动都在，没有冲突。直觉上这应该冲突——都在改同一个地方。但 Git 没报。为什么？

Git 的三路合并 ##

Git merge 不是直接比较两个分支的差异，而是引入第三个参照点——merge base（两个分支的最近公共祖先）。

       O (base)
      / \
     A   B

Git 分别计算 diff(base, A) 和 diff(base, B)，然后尝试合并这两组差异。对每一行，判断逻辑很直接：

Base → OursBase → Theirs结果
没变没变保留原样
变了没变取 ours
没变变了取 theirs
变了，结果相同变了，结果相同自动合并
变了，结果不同变了，结果不同冲突

这个表本身没什么意外的。关键是：Git 怎么定义"变了"？

插入 vs. 修改，Git 分得很清楚 ##

Git 的合并引擎（merge-recursive 或 ort）以行为单位做 diff，比较时依赖上下文锚点——那些在两个分支中都没变的行。锚点用来定位改动发生的区域，Git 在锚点之间合并差异。

这里有一个关键区别：插入新行和修改已有行，在 Git 看来是两种完全不同的操作。

  • 修改已有行：base 中某行是 X，ours 改成 Y，theirs 改成 Z——Git 检测到两边都动了同一行，冲突。
  • 插入新行：base 中某行是 X，ours 在它后面加了一行，theirs 也在它后面加了一行——Git 看到的是"双方在锚点 X 之后各自添加了新内容"，两段插入之间没有直接冲突，直接拼接在一起。

ps: 这里的"锚点"就是 diff hunk 头部的 @@ -a,b +c,d @@ 中所依据的那些上下文行。

一个具体例子 ##

假设 base 有一个文件：

line A
line B
line C

branch1 在 line B 之后插入一行：

line A
line B
new line from branch1
line C

branch2 也在 line B 之后插入一行：

line A
line B
new line from branch2
line C

merge 结果：

line A
line B
new line from branch2
new line from branch1
line C

没有冲突。line B 和 line C 是未变的锚点，两个分支都在这个锚点区间内做了插入，Git 按 ours → theirs 的顺序拼在一起。

什么情况下会冲突？ 两边动了同一行已有内容。

// base
print("Hello")

// branch1
print("Hi")

// branch2
print("Hello, World!")

两边都把 print("Hello") 改成了不同的东西，Git 无法决定：

<<<<<<< HEAD
print("Hi")
=======
print("Hello, World!")
>>>>>>> branch2

再比如一边删除了某行，另一边修改了同一行——同样是冲突，因为两个操作对同一行做了互斥的动作。

总结起来 ##

操作类型Base → OursBase → Theirs冲突？
两边都没动不变不变否
只有一边插入新行插入不变否
两边在不同位置插入插入（位置 1）插入（位置 2）否
两边在同一锚点区间插入插入插入否
两边修改同一行（结果不同）修改修改是
一边修改，一边删除修改删除是
两边修改同一行（结果相同）修改（X→Y）修改（X→Y）否，自动合并

最后一行也值得注意——两边做了完全一样的修改，Git 不会报冲突，直接合并。

这合理吗 ##

从文本合并的角度，合理。插入是增量操作，两段独立的插入不破坏对方的改动，可以安全地交织。

但从语义角度，不一定。两段代码插在同一个位置，逻辑上很可能有关联——比如两个人都在同一个函数里加了错误处理，合并后两段处理并列，执行顺序和覆盖范围可能有问题，但 Git 不会提醒你。

所以 merge 之后即使没冲突，该跑的测试、该做的 code review 一样不能少。

ps: 如果确实需要在合并时做更严格的检测，可以配置 Git 自定义合并驱动（custom merge driver），在合并阶段引入额外的检查逻辑。相关实践见《Git 自定义合并驱动，增强冲突检测》。
