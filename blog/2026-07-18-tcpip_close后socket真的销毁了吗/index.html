tcp状态机：close(fd)后socket真的销毁了吗

Posted 2026-07-18 20:00 +0800 by ZhangJie ‐ 2 min read

分享:  

问题背景 ##

在网络编程中，许多人存在一个直觉误区：认为应用层执行 close(fd) 且引用计数归零后，内核的 socket 就立即销毁了，此时收到任何数据必然会立刻回复 RST。

这个理解是错误的。

应用层销毁 fd（钥匙）与内核销毁 socket 状态机（实体）是异步解耦的。调用 close() 后，该套接字会变成孤儿套接字（Orphan Socket）由内核接管。对于后续到来的数据，内核有一套极其严密的防御和处理机制。

前置知识：shutdown() 与 close() 的本质区别 ##

在深入内核行为之前，必须先澄清一个容易混淆的问题：为什么 shutdown() 不能替代 close()？两者到底操作的是什么？

应用层
  │
  ├── fd (文件描述符) ─── close() 操作这一层
  │
  ├── struct file ─── 引用计数为0时销毁的是这个，而非最底层的struct sock
  │   ...
  └── struct sock, socket (内核对象) ─── shutdown() 操作这一层 ─── TCP 状态机在这里
  • shutdown() 无论 how 取 SHUT_RD、SHUT_WR 还是 SHUT_RDWR，都直接操作 struct sock 层的半关闭标志，完全不动 struct file 的引用计数 (f_count)。
  • close(fd)时，引用计数-1指的是对struct file引用计数-1。引用计数为0时，销毁的是 struct file，而不是 struct sock（socket传输控制块）。释放 struct file 时通过 file->f_op->release() → sock_close() → tcp_close() 驱动 TCP 状态机流转。tcp_close() 只是驱动 TCP 状态机迁移到 FIN_WAIT_1 / FIN_WAIT_2，struct sock 本身并未直接销毁——它作为"孤儿套接字"继续由内核协议栈接管，直到对端关闭或超时。

理解了这层关系，“孤儿套接字"就很好懂了：close(fd) 进程失去了访问 socket 的"钥匙”。但 struct sock（TCP 状态机）另有其生命周期，在 tcp_close() 将其推入 FIN_WAIT_1 / FIN_WAIT_2 等状态后，它仍然存活并由内核协议栈继续处理网络数据。

接下来我们要讨论，close(fd) / shutdown(fd, write) 后，如果对端继续发送数据，该如何处理？

一、 close()后对端继续发送数据 ##

当应用层调用 close(fd) 且引用计数归零后，后续收到数据（或缓冲区残留数据）的处理直接分为两种截然不同的内核路径：

路径 A：接收缓冲区“有”未读数据 —— 立刻回 RST ##

  • 触发条件：调用 close 的那一刻，如果用户应用层还有数据滞留在接收缓冲区没读完；或者 close 触发的 FIN 发出后，对端在收到 FIN 前又抢跑发了一批新数据过来。
  • 内核行为：内核认为这是一种异常关闭（数据丢失风险），会直接跳过四次挥手，单方面将状态切为 TCP_CLOSE，并立刻向对端发送 RST 包。

路径 B：接收缓冲区“没有”未读数据 —— 正常回复 ACK ##

  • 触发条件：应用层已经把数据彻底读干净了，然后调用 close。
  • 内核行为：内核正常发送 FIN 包，并将孤儿套接字推入 FIN_WAIT_1 / FIN_WAIT_2 状态。此时如果对端（处于 CLOSE_WAIT 状态）继续发送合法的单向数据，内核不仅不会回 RST，反而会为了维护 TCP 的可靠性契约，正常回复 ACK 确认收妥。

二、 “路径B”核心机制深度拆解 ##

针对路径 B，进程已经没有 fd 了，操作系统为什么还要“多此一举”地回 ACK？这背后是一套精妙的内核自我防卫闭环：

1. 为什么还要回 ACK？ ##

TCP 是全双工的。本端关闭了写端（发送了 FIN），但在协议层面，对端依然有权利把请求或者手头没发完的存量数据发完。只要对端发送的数据包序列号（Sequence Number）落在合法的接收窗口内，本端内核协议栈就必须回复 ACK 确认。这些数据会被默默积压在孤儿套接字的接收缓冲区中。

2. 没人读，缓冲区满了怎么办？（滑动窗口防御） ##

随着对端持续发送应用层数据，由于应用层的 fd 已经不存在了，没有人能调用 read() 去消耗这些数据，导致孤儿套接字的接收缓冲区迅速被填满。 一旦缓冲区爆满，本端内核通告给对端的 接收窗口（Window Size）会瞬间跌成 0（Zero Window），从物理上卡死对端的发送能力。

3. 如果对端篡改协议栈，强行冲撞零窗口怎么办？ ##

如果对端恶意或者因 Bug 篡改了自身的协议栈，无视零窗口通告继续强行把数据包塞过来，本端内核在检测到数据超出当前接收窗口后，绝对不会再为这些非法数据回复正常的 ACK。内核会直接将数据包丢弃（Drop），仅回复一个携带当前正确序列号和窗口为 0 的 Dup ACK（重复确认） 进行警告。

4. 内核最终如何收场？（tcp_fin_timeout 的真正归宿） ##

卡死在 FIN_WAIT_2 的孤儿套接字不可能无限期常驻内存。Linux 内核通过 net.ipv4.tcp_fin_timeout 参数进行终极清算：

  • 注意误区：该参数完全管不到 TIME_WAIT 的 2MSL 时间（Linux 的 2MSL 是硬编码死在源码里的 60 秒，无法通过 sysctl 调节）。
  • 真实作用：它特指孤儿套接字卡在 FIN_WAIT_2 状态下，迟迟收不到对端 FIN 包的超时时间。
  • 微观细节：当对端一直在发有效数据时，内核认为它还在工作；只有当对端发完安静下来，或者触发零窗口死寂后，该定时器才真正激活并倒计时。一旦超时（默认 60 秒），内核彻底失去耐心，直接跳过 TIME_WAIT，强行把这个 Socket 结构体及其 Buffer 从内存中彻底抹去，释放资源。

       │

       │
       ▼
内核彻底强行销毁 Socket 实体，内存与缓冲区资源全额回收！

四、shutdown()后对端发送数据 ##

shutdown(fd, write) 就不一样了，没有释放fd、没有减引用计数、没有释放struct file，只是设置了struct sock的写半连接关闭状态，驱动TCP状态机发送了FIN包给对端，对端会送ACK。然后对端被动关闭停在CLOSE_WAIT状态。

TCP连接，支持全双工通信，我方主动关闭写端，只是告知对端我不会再发送新数据了，但是我依然可以读取数据。所以对端如果还有数据要发送，仍然可优化大大方方发送给我。

和close不同的是，应用程序此时依然持有访问struct sock的金钥匙，依然可以正常收取数据，进行处理，这完全可行。等真正不需要这个socket接受数据时再调用close(fd)即可。注意，不能通过调用shutdown，前面已经提过了，即使shutdown(fd, RDWR)也不能彻底释放socket。

本文小结 ##

最后总结下，close(fd) 并不等于立即销毁 socket：应用层的 fd 只是访问内核 socket 的"钥匙"，struct sock 才是真正的"实体"，钥匙丢了实体还在。close(fd) 释放的是 struct file，struct sock 则作为孤儿套接字由内核接管。若接收缓冲区有未读数据，内核判定异常关闭，直接发 RST 暴力终止；若已读干净，则正常走 FIN 挥手流程，即使进程已无法读取数据，内核仍会代为 ACK 对端后续发来的合法数据，并通过零窗口、Dup ACK 警告、tcp_fin_timeout 超时三层防御机制防止资源泄漏。

与之相比，shutdown() 只操作半关闭标志、不释放 fd，适用于"我不发了但还想收"的场景。
