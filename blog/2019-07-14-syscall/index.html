系统调用在不同 CPU 架构上的实现差异

Posted 2019-07-14 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

背景 #

系统调用是用户态程序请求内核服务的唯一入口。但不同 CPU 架构下，进入系统调用的指令、传参方式各不相同。理解这些差异对于阅读内核源码、进行跨平台开发和性能优化都有帮助。

正文 #

核心问题 ##

对于系统调用，每个 CPU 架构都需要解决两个问题：

  1. 用什么指令进入内核态？
  2. 如何传递系统调用号和参数？如何获取返回值？

x86（32 位） ##

机制指令说明
传统方式int 0x80通过软中断进入内核，系统调用号存入 eax，参数通过 ebx、ecx、edx、esi、edi、ebp 传递，返回值存入 eax
快速方式sysenter / sysexitIntel 引入的快速系统调用指令对，比 int 0x80 更快，无需经过完整的中断处理流程

x86_64（64 位） ##

指令说明
syscall / sysretAMD 引入的快速系统调用指令，被 x86_64 Linux 采用

参数传递：

  • 系统调用号：rax
  • 参数 1~6：rdi、rsi、rdx、r10、r8、r9
  • 返回值：rax

为何第 4 个参数用 r10 而非 rcx？因为 syscall 指令会自动将 rip 保存到 rcx，会破坏 rcx 的值。

ARM64 ##

指令说明
svc (Supervisor Call)触发异常进入内核态

参数传递：

  • 系统调用号：x8
  • 参数 1~6：x0 ~ x5
  • 返回值：x0

为什么需要多种方式？ ##

从 int 0x80 到 sysenter/syscall 的演进，主要是出于性能考虑：

  • int 0x80 走完整的中断门流程，需要进行权限检查、栈切换等大量操作
  • sysenter/syscall 是专门为系统调用优化的指令，跳过了中断处理的大部分开销
  • 现代内核（如 Linux）在启动时会根据 CPU 能力选择最优的系统调用入口，并通过 vDSO 暴露给用户态

总结 #

架构指令系统调用号寄存器
x86 传统int 0x80eax
x86 快速sysentereax
x86_64syscallrax
ARM64svcx8

虽然指令和寄存器约定不同，但概念是一致的：通过特定指令陷入内核，在寄存器中传递系统调用号和参数，内核处理后返回结果。
