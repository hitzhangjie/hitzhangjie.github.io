文件锁阻塞与信号响应

Posted 2000-01-01 00:00 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 #

文件锁（flock/fcntl）是进程间同步的常用手段。当一个进程尝试获取已被占用的文件锁时，默认行为是阻塞等待。但问题来了：此时进程还能响应信号（如 Ctrl+C 发送的 SIGINT）吗？

正文 #

结论：可以！ ##

如果文件锁被另一个进程持有，尝试获取锁的进程会被阻塞。但内核只会将此进程的状态改为**可中断等待（interruptible sleep）状态，而非不可中断等待（uninterruptible sleep）**状态。

这意味着：

  • 没有信号到达时，进程处于睡眠状态，不参与调度
  • 一旦有信号到达且未被进程屏蔽，内核会立即将进程状态修改为**可运行（running）**状态，使进程重新参与调度
  • 信号处理函数得以执行

实测：信号打断锁等待 ##

下面的 C 程序演示：一个进程持有锁，另一个进程阻塞在 fcntl(F_SETLKW) 上，此时 Ctrl+C 发送 SIGINT 能否唤醒它。

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <fcntl.h>
#include <signal.h>

void handle_sigint(int sig) {
    printf("Received signal %d, exiting...\n", sig);
    exit(0);
}

int main() {
    signal(SIGINT, handle_sigint);

    int fd = open("/tmp/test.lock", O_RDWR | O_CREAT, 0644);
    if (fd == -1) {
        perror("open");
        return 1;
    }

    struct flock fl = {
        .l_type   = F_WRLCK,
        .l_whence = SEEK_SET,
        .l_start  = 0,
        .l_len    = 0,          // 锁定整个文件
    };

    printf("Trying to acquire lock...\n");
    fcntl(fd, F_SETLKW, &fl);   // 阻塞等待锁
    printf("Lock acquired!\n");

    pause(); // 持有锁，等待测试
    close(fd);
    return 0;
}

# 终端 1：先启动一个实例持有锁
$ gcc -o main main.c && ./main &
Trying to acquire lock...
Lock acquired!

# 终端 2：启动第二个实例，它会阻塞在 fcntl(F_SETLKW) 上
$ ./main &
Trying to acquire lock...
# <阻塞中，状态为 interruptible sleep>

# 按 Ctrl+C 发送 SIGINT
^CReceived signal 2, exiting...

进程在获取不到文件锁时，内核只是将进程状态修改为可中断等待状态，因此 Ctrl+C 产生的 SIGINT 能被正常接收和处理。

进程状态观察 ##

在等待锁期间，可以通过 ps 观察进程状态：

$ ps aux | grep main
# STAT 列为 'S'（interruptible sleep），而非 'D'（uninterruptible sleep）

信号处理函数返回之后呢？ ##

前面的实验验证了"阻塞等锁时可以响应信号"。但这引出一个更关键的问题：信号处理函数执行完毕后，fcntl(F_SETLKW) 会怎样？

  • 是继续阻塞等锁？
  • 还是自动重新进入 fcntl 加锁？
  • 还是返回错误让调用方自己处理？

信号处理函数由谁执行？ ##

首先需要澄清一个常见的误解：信号处理函数到底是内核代劳，还是进程自己执行的？

答案是进程自己执行。 整个流程如下：

1. fcntl(F_SETLKW) 阻塞，进程处于 TASK_INTERRUPTIBLE
                    ↓
2. 信号到达 → signal_wake_up() 将进程状态修改为 TASK_RUNNING
   并将进程放回运行队列（runqueue），进程重新变为可调度
                    ↓
3. 调度器选中该进程，准备返回用户态
                    ↓
4. 内核在返回用户态前检查 pending signals（do_signal / exit_to_user_mode_loop）
   发现有未处理信号 → 不直接返回 fcntl 的调用点
   而是修改用户态栈帧，让进程"看起来像是"在正常执行中被信号打断
                    ↓
5. 内核将 sigreturn 的返回地址压栈，然后设置 RIP 指向信号处理函数入口
   返回用户态 → 进程开始执行信号处理函数
                    ↓
6. 信号处理函数执行完毕，执行 sigreturn 系统调用
   内核恢复原始上下文（寄存器、栈帧等）

关键点：信号不是"内核替进程跑了个函数"，而是内核在恰当的时机（进程被调度、即将返回用户态时）“劫持"了用户态执行流，让进程自己去执行信号处理函数。

fcntl(F_SETLKW) 不会被自动重启 ##

fcntl(F_SETLKW) 在等锁时通过 wait_event_interruptible() 睡眠。这个函数在收到信号后返回 -ERESTARTSYS 错误码。

但 ERESTARTSYS 的语义是”如果 SA_RESTART 标志置位则可以重启"。哪些系统调用能被 SA_RESTART 重启、哪些不能，是由内核决定的。根据 Linux signal(7) 手册的明确说明：

The following interfaces are never restarted after being interrupted by a signal handler, regardless of the use of SA_RESTART; they always fail with the error EINTR:

  • fcntl(F_SETLKW)

也就是说，无论你是否设置了 SA_RESTART，fcntl(F_SETLKW) 在信号中断后都不会自动重试，而是直接返回 -1 并设置 errno = EINTR。

验证：信号中断后返回 EINTR ##

修改测试代码，让信号处理函数不退出，而是正常返回，观察 fcntl 的行为：

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <fcntl.h>
#include <signal.h>
#include <errno.h>

static volatile sig_atomic_t sig_received = 0;

void handle_sigint(int sig) {
    sig_received = 1;
    write(STDOUT_FILENO, "\n[SIGNAL] SIGINT received\n", 26);
}

int main() {
    struct sigaction sa = { .sa_handler = handle_sigint };
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;  // 不设置 SA_RESTART
    sigaction(SIGINT, &sa, NULL);

    int fd = open("/tmp/test.lock", O_RDWR | O_CREAT, 0644);
    
    struct flock fl = {
        .l_type = F_WRLCK,
        .l_whence = SEEK_SET,
        .l_start = 0,
        .l_len = 0,
    };

    printf("Trying to acquire lock...\n");
    if (fcntl(fd, F_SETLKW, &fl) == -1) {
        // 信号中断后会执行到这里
        if (errno == EINTR) {
            printf("fcntl interrupted by signal (EINTR). sig_received=%d\n",
                   sig_received);
            // 调用方需要决定：是重试，还是放弃？
        } else {
            perror("fcntl");
        }
        close(fd);
        return 1;
    }

    printf("Lock acquired!\n");
    pause(); // 持有锁，等待另一个实例测试
    close(fd);
    return 0;
}

输出：

$ ./main
Trying to acquire lock...
^C
[SIGNAL] SIGINT received
fcntl interrupted by signal (EINTR). sig_received=1

信号处理函数正常返回了，但 fcntl 并没有继续等锁，而是直接返回 -EINTR。 进程需要自行决定是重试还是退出。

如果调用方想继续等锁怎么办？ ##

最简单的做法是在外层包一个循环：

int lock_with_retry(int fd, struct flock *fl) {
    while (1) {
        if (fcntl(fd, F_SETLKW, fl) == 0) {
            return 0;  // 加锁成功
        }
        if (errno == EINTR) {
            // 被信号打断，继续重试等锁
            continue;
        }
        // 其他错误
        return -1;
    }
}

这样就实现了"信号可以被处理，处理完后继续等锁"的效果，但重试逻辑由调用方显式控制，而不是内核自动完成。

总结 #

  • 文件锁阻塞（fcntl F_SETLKW）将进程置于可中断睡眠（S 状态），信号到来时进程被真正唤醒并参与调度
  • 信号处理函数由进程自身执行，不是内核代劳：内核在返回用户态前修改栈帧，让进程执行信号处理函数，处理完后通过 sigreturn 恢复上下文
  • fcntl(F_SETLKW) 被信号中断后不会自动重启（即使在 SA_RESTART 下也是），立即返回 -1，errno = EINTR
  • 如果需要在信号处理后继续等待锁，调用方必须显式地在 while 循环中处理 EINTR 并重试
