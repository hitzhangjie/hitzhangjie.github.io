Starlark 在 Delve 中的实战：从脚本编写到踩坑记录

Posted 2025-05-25 01:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

前一篇 介绍了 Delve Starlark 脚本系统的架构和内置函数。本文通过一个完整的实战示例，展示如何编写 Starlark 脚本来自动化调试流程，并记录编写过程中容易踩到的坑。

示例程序 #

假设我们要调试下面这个简单的 Go 程序：

package main

import (
    "fmt"
    "time"
)

type Person struct {
    Name string
    Age  int
}

func main() {
    people := []Person{
        {Name: "Alice", Age: 25},
        {Name: "Bob", Age: 30},
        {Name: "Charlie", Age: 35},
    }

    for i, p := range people {
        fmt.Printf("Processing person %d: %s\n", i, p.Name)
        time.Sleep(time.Second)
        processPerson(p)
    }
}

func processPerson(p Person) {
    fmt.Printf("Name: %s, Age: %d\n", p.Name, p.Age)
}

需求：在 processPerson 中自动打印变量 p 的值，并将调试信息保存到文件。这个流程如果手动操作需要反复 continue → print → next，用 Starlark 脚本可以一键完成。

编写 Starlark 调试脚本 #

我们逐步构建一个完整的调试脚本。

基础函数：获取作用域 ##

def print_scope():
    scope = cur_scope()
    print("Current scope:", scope)

cur_scope() 返回当前栈帧的作用域信息，是最常用的自省函数之一。

核心调试逻辑 ##

def debug_person():
    dlv_command("break main.processPerson")  # 在目标函数设断点
    dlv_command("continue")                   # 运行到断点
    print_scope()                             # 查看当前作用域
    dlv_command("print p")                    # 打印变量 p
    dlv_command("next")                       # 单步到下一行
    print_scope()                             # 再次查看作用域

保存调试信息到文件 ##

def save_debug_info():
    scope = cur_scope()
    debug_info = "Debug session at " + str(time.time()) + "\n"
    debug_info += "Current scope: " + str(scope) + "\n"
    debug_info += "Variables:\n"
    dlv_command("print people")
    write_file("debug_info.txt", debug_info)

主函数 ##

def main():
    print("Starting debug session...")
    dlv_command("break main.main")
    dlv_command("continue")
    debug_person()
    save_debug_info()
    print("Debug session completed.")

main()

保存为 debug.star，在 Delve 中通过 source debug.star 加载执行。

踩坑记录：Starlark 与 Python 的差异 #

Starlark 的语法看起来很像 Python，但它实际上是 Python 的一个受限子集，很多 Python 习以为常的用法在 Starlark 中并不支持。以下是编写调试脚本时最常遇到的两个坑。

坑 1：不支持 f-string ##

❌ Python 写法：

debug_info = f"Debug session at {time.time()}\n"

✅ Starlark 写法：

debug_info = "Debug session at " + str(time.time()) + "\n"

Starlark 只支持传统的字符串拼接方式，需要显式使用 str() 函数转换非字符串值。

坑 2：不支持 __name__ ##

❌ Python 写法：

if __name__ == "__main__":
    main()

✅ Starlark 写法：

main()

Starlark 没有 __name__ 这个 dunder 变量。如果需要控制执行入口，直接在脚本末尾调用即可——当脚本通过 source 加载时，Starlark 解释器会顺序执行顶层代码。

其他需要注意的差异 ##

Python 特性Starlark 支持？
f-string❌ 不支持
__name__❌ 不支持
list comprehension✅ 支持
dict comprehension✅ 支持
len(), str(), int()✅ 支持
import✅ 支持（有限）
class 定义❌ 不支持
try/except❌ 不支持
递归❌ 不支持
generator /yield❌ 不支持

执行方式 #

# 编译目标程序（禁用优化和内联以保留调试信息）
go build -gcflags="all=-N -l" main.go

# 启动 Delve
dlv exec ./main

# 加载并执行 Starlark 脚本
(dlv) source debug.star

脚本执行后，会在当前目录生成 debug_info.txt，包含调试会话中的变量快照。

适用场景 #

Starlark 脚本最适合以下场景：

  • 批量断点管理：一次性设置多个条件断点，或者在不同版本之间迁移断点配置
  • 自动化数据采集：在某个断点每次命中时自动打印指定变量并记录到文件
  • 调试流程复用：将复杂的调试流程写成脚本，团队成员可以直接 source 执行

总结 #

Starlark 虽然语法上接近 Python，但在实际编写时会遇到不少受限子集带来的差异。掌握了字符串拼接和顶层执行这两点之后，就可以写出实用的自动化调试脚本。对于需要反复执行的调试流程，花几分钟写一个 Starlark 脚本，远比每次手动敲命令高效。
