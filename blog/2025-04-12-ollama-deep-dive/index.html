Ollama 深度解析：本地大模型服务的工程实践

Posted 2025-04-12 00:00 +0800 by ZhangJie ‐ 5 min read

分享:  

背景 ##

Ollama 是目前最流行的本地 LLM 部署工具之一——一行命令就能下载并运行各种开源模型。它屏蔽了模型格式、量化方案、GPU 加速等复杂性，让开发者可以像使用 Docker 一样使用大模型。

但 Ollama 内部是怎么做到这些的？这篇文章从源码和设计角度，系统性地拆解 Ollama 的核心技术：架构设计、认证机制、模型格式、量化技术、GPU 加速、模板语法和推理循环。

一、架构设计：双 Server 的分离之美 ##

Ollama 的源码中有两个同名的 Server 结构体，但它们定义在不同的包中，职责完全不同。这是理解 Ollama 架构的第一把钥匙。

API Server（server/routes.go） ##

type Server struct {
    addr  net.Addr
    sched *Scheduler
}

这是 Ollama 的 HTTP API 服务器，负责：

  • 处理 HTTP 路由和 REST API 请求（/api/generate、/api/chat 等）
  • 管理模型调度器（Scheduler）
  • 模型生命周期管理（加载、卸载、调度）

LLM Runner（runner/llamarunner/runner.go） ##

type Server struct {
    ready     sync.WaitGroup
    model     *llama.Model
    status    llm.ServerStatus
    parallel  int
    batchSize int
    lc        *llama.Context
    seqs      []*Sequence
    cache     *InputCache
    // ...更多字段
}

这是 LLM 运行时核心，负责：

  • 直接管理和运行 LLM 模型
  • 处理实际的推理请求
  • 管理模型上下文（Context）和 KV Cache
  • 管理多线程和并发请求

分层协作 ##

用户请求 → API Server (HTTP路由) → Scheduler (调度) → LLM Runner (实际推理) → 返回结果

API Server 是高层抽象，处理 HTTP 和 API 层面的逻辑；LLM Runner 是底层实现，直接与 llama.cpp 的模型对象交互。API Server 通过 sched 字段间接调用 Runner 的能力。

这种设计体现了关注点分离原则——将 API 服务和模型运行时解耦，各自独立演进和维护。

二、认证机制：Bearer Token 的细粒度控制 ##

Ollama 在 pull/push 模型到 registry 时使用 Bearer Token 认证机制，其核心优势是细粒度的访问控制：

资源级别的访问控制（Scope）：精确控制用户能访问哪些仓库/模型，可区分不同资源的读写权限。

操作级别的权限控制：细分 pull/push 等不同操作的权限，不同操作可使用不同 token。

时间维度的控制：token 可设置过期时间，可随时撤销，不需要像持久性凭证那样长期有效。

审计和跟踪：每个 token 可关联到特定的请求和操作，便于安全审计和访问日志分析。

这比简单的用户名密码认证要灵活和安全得多——不需要在每个请求中传输敏感凭证；可动态调整权限而不需要更改用户凭证；即使 token 泄露，影响也受限于时间和范围。

三、模型文件格式：为什么有那么多"标准" ##

LLM 领域目前没有一个统一的文件格式标准。不同框架和项目选择了最适合自己的格式：

格式特点适用场景
PyTorch (.pt/.pth)框架原生格式，文件大训练
Safetensors (.safetensors)HuggingFace 开发，安全（防代码执行），加载快模型分发
GGML (.ggml/.bin)CPU 优化，支持量化，llama.cpp 生态CPU 推理
GGUF (新一代 GGML)更好的元数据支持，更灵活的格式设计本地推理（新标准）
TensorFlow (.pb)Google 框架格式TF 生态

Ollama 选择 GGML/GGUF 格式的原因：

  1. 专注本地 CPU 推理场景——不需要 GPU 也能运行
  2. 很好的平衡了模型大小和推理性能——通过量化技术
  3. 成熟的生态系统支持——llama.cpp 社区活跃

关键认知：GGML/GGUF 是一种存储和表示格式，而不是"只能用 CPU"的限制。Ollama 运行时会根据硬件条件自动选择最优的执行后端（CUDA/ROCm/Metal）。它既能在 CPU 上高效运行，又能在有 GPU 时充分利用 GPU 加速。

四、量化技术：从 FP32 到 INT8 ##

什么是量化 ##

模型参数默认用 FP32 存储——每个参数占 4 字节。能不能用更少的字节来存？量化就是干这个的：把 32 位浮点数压缩成 8 位整数（INT8），每个参数从 4 字节变成 1 字节。

三个好处：体积小、跑得快、省电。

量化的三种玩法 ##

  • 训完再压（PTQ）：模型训好了，直接压缩。最常用，也是 Ollama 的默认方式。
  • 边训边压（QAT）：训练的时候就模拟压缩效果，训出来的模型更耐压，但训练成本高。
  • 看人下菜（混合精度）：重要的层用高精度，不重要的层往死里压。

一个具体例子 ##

假设模型里有个参数张量，里面的值都在 0 到 100 之间，目前用 FP32 存储。怎么把它变成 INT8？

INT8 能表示的整数范围是 -128 到 127（有符号 8 位整数，共 256 个数）。思路很简单：把 0100 这根尺子，等比映射到 -128127 这根尺子上。

需要两个数来描述这个映射：

  • scale：尺子的缩放比例。scale = 127 / 100 = 1.27，意思是整数这边每走 1 格，相当于浮点那边走 1.27。
  • zero_point：浮点 0 对应整数尺子上的哪个刻度。这里取 -128（也就是 INT8 的最小值），是因为这个 tensor 的值全是非负数，最小值就是 0，所以让 0 对齐到 -128，用满整数范围。如果换一个包含负数的 tensor，比如范围 -50~50，那就该让 -50 对齐到 -128，zero_point 就不是 -128 了。

写成代码就三行：

def quantize(value, scale, zero_point):
    return round(value / scale) + zero_point   # 浮点 → 整数

def dequantize(q_value, scale, zero_point):
    return (q_value - zero_point) * scale      # 整数 → 浮点

拿 val = 50.0 试一下：

q_val = quantize(50.0, 1.27, -128)      # -89
back  = dequantize(-89, 1.27, -128)     # 49.53

50.0 压进去，解回来变成 49.53，差了 0.47——这就是量化损失。实际工程中，scale 和 zero_point 不是手算的，而是用一批校准数据跑一遍模型，自动算出来的，尽可能把损失压到最小。

内存省了多少 ##

一个 120 亿参数的模型：

  • FP32：120 亿 × 4 字节 = 48 GB
  • INT8：120 亿 × 1 字节 = 12 GB
  • 省了 36 GB，也就是 75%

五、imatrix：更智能的量化 ##

普通的量化是"一刀切"——所有参数用同样的压缩率。但模型中不同权重的重要性是不同的。

Importance Matrix（重要性矩阵） 技术通过分析模型各层权重对最终输出的贡献，来指导量化过程：

  1. 数据收集：用代表性数据集过一遍原始模型，记录每个权重参数的"贡献度"，保存为 .imatrix 文件
  2. 指导量化：量化器根据这个"重要性地图"，对重要权重分配更精细的量化等级（更少误差），对不重要的权重进行更激进的压缩

类比：普通量化像对整个合影照片用同样的 JPEG 压缩率——所有人脸都有些模糊。使用 imatrix 的量化像智能压缩——识别出人脸是关键区域，对脸部用高保真压缩，对背景树木天空用高压缩率。文件大小差不多，但人脸更清晰。

这就是为什么同样是 Q4_K_M 量化，带 imatrix 优化的版本通常有更高的准确性。

六、GPU 加速：GGML 格式不等于"只能用 CPU" ##

虽然 Ollama 使用 GGML/GGUF 格式，但它确实支持 GPU 加速。

支持的 GPU 后端 ##

  • NVIDIA：通过 CUDA
  • AMD：通过 ROCm
  • Apple Silicon：通过 Metal

关键优化特性 ##

  • Flash Attention：通过 OLLAMA_FLASH_ATTENTION=1 启用，显著减少显存使用
  • KV Cache 量化：通过 OLLAMA_KV_CACHE_TYPE 设置，支持 f16/q8_0/q4_0 等精度
  • 多 GPU 负载均衡：自动评估 VRAM 并决定模型如何分布在多个 GPU 上

自动优化策略 ##

Ollama 运行时会自动检测可用 GPU，根据模型大小和可用显存自动决策：

  • 模型完全适合单个 GPU → 优先单 GPU（减少 PCI 总线数据传输）
  • 模型太大 → 自动在多个 GPU 间分配
  • 没有 GPU → 退回到 CPU 推理（这是 GGML 格式的核心优势）

七、Modelfile：模板语法深度解析 ##

Ollama 的 Modelfile 中 TEMPLATE 定义了对话格式。以 Llama 3 为例：

TEMPLATE """{{ if .System }}<|start_header_id|>system<|end_header_id|>

{{ .System }}<|eot_id|>{{ end }}{{ if .Prompt }}<|start_header_id|>user<|end_header_id|>

{{ .Prompt }}<|eot_id|>{{ end }}<|start_header_id|>assistant<|end_header_id|>

{{ .Response }}<|eot_id|>"""

语法分解 ##

使用 Go 模板语法：

  • {{ if .System }} — 条件判断，有系统消息则渲染
  • {{ .System }} — 插入系统消息内容
  • {{ .Prompt }} — 插入用户提示
  • {{ .Response }} — 插入助手回复位置

特殊标记 ##

  • <|start_header_id|> 和 <|end_header_id|> — 标记角色的开始和结束（system/user/assistant）
  • <|eot_id|> — 表示一段对话的结束（End of Turn）

为什么模板格式如此重要 ##

大模型在训练时就是按照这个格式来训练的。 训练数据中反复出现这些特殊标记，模型通过大量训练学会了识别和理解它们的含义。

当我们使用时，输入格式与训练格式一致 → 模型表现更好。就像跟一个外国人交流时，用他最熟悉的语言和表达方式，沟通更顺畅。

这就是 Modelfile 模板的本质：用模型训练时的"语言"与模型对话。

八、推理循环：代码级视角 ##

Ollama 的文本生成核心在 runner/llamarunner/runner.go 中，只有三行代码：

// sample a token
token := seq.samplingCtx.Sample(s.lc, seq.iBatch)
seq.samplingCtx.Accept(token, true)
piece := s.model.TokenToPiece(token)

要理解这三行代码，需要先厘清几个基础概念。我们从 token 开始，逐步深入到概率分布和采样策略，最后回到这三行代码，一切就水到渠成了。

Token 是什么 ##

Token 是大模型处理文本的最小单元，它是一个整数编号，对应词汇表（vocabulary）中的某个"词片段"。它不是 embedding 向量——embedding 是模型内部把 token ID 映射成向量以便计算，而 token 本身就是一个整数。

几个关键事实：

  • Token ≠ 单词。一个 token 可能是完整单词，也可能是单词的一部分。比如 “unbelievable” 可能被拆成 “un”、“believe”、“able” 三个 token。中文里，“你好” 可能拆成 “你” + “好” 两个 token。
  • 词汇表是固定的。每个模型有一个预定义的词汇表，通常包含 32,000～100,000 个 token，每个 token 有且只有一个整数 ID。模型的输出维度就等于词汇表大小。
  • Tokenization 是将文本转成 token ID 序列的过程，由分词器完成；反向过程叫 detokenization（token ID 序列还原为文本），正是 TokenToPiece 做的事情。

一个直观的类比：token 就像一本大词典里的"词条编号"。假设词典收录了 5 万个词条，从 1 到 50000 逐一编号，每个编号只对应一个词条。模型读的不是文字本身，而是这些编号——输入文字 → 查词典找到每个词的编号 → 模型处理编号序列 → 输出"下一个最可能的编号" → 查词典还原为文字。

概率分布：模型在"猜"下一个词 ##

理解了 token 是词汇表中的编号后，下一个问题是：模型推理一次，输出究竟是什么？

模型推理每一步输出的是一个概率分布——一个长度等于词汇表大小的向量，每个位置的值表示对应 token 成为"下一个词"的概率。具体过程：

  1. 模型最后一层输出原始分数（logits），维度 = 词汇表大小（如 50000）
  2. 经过 softmax 函数，将 logits 转换为 0～1 之间的概率值，且所有值之和为 1
  3. 每个 token ID 都得到一个概率——没有哪个是绝对 0，只是很多 token 的概率微乎其微

举个例子：输入 “The sky is”，模型输出的概率分布可能是：

Token ID对应文本概率
15339blue0.72
8241clear0.15
3782dark0.05
20123the0.003
……余下 49996 个 token 各占极小概率

类比：模型像一个"完形填空高手"。给它上半句，它脑子里浮现出所有可能的接续词，每个词都有不同的"合适度"。接下来要做的，就是从这个"候选清单"里选一个。

采样策略：怎么从概率分布中选一个 token ##

有了概率分布之后，怎么选？不同的选法就是不同的采样策略。选法的差异直接决定了生成结果的风格——是保守稳妥还是天马行空。

1. 贪婪采样（Greedy） ##

直接选概率最高的 token。确定性强，但输出单调重复——模型永远选最有把握的词，导致内容缺乏变化，甚至陷入循环。几乎只有代码生成等对确定性要求极高的场景才用。

2. 温度采样（Temperature） ##

在 softmax 之前，用温度参数 T 调节分布的"陡峭程度"。做法是把所有 logits 除以 T 再做 softmax：

  • T < 1（如 0.3）：分布更"尖锐"，高概率 token 更突出 → 输出保守确定
  • T = 1：原始分布，不做调整
  • T > 1（如 1.5）：分布被"抹平"，低概率 token 获得更多机会 → 输出更随机、更有创意

温度就像一个"创造力旋钮"。拧低 → 模型只敢说最有把握的答案；拧高 → 模型敢于冒险尝试冷门选项。

3. Top-k 采样 ##

top-k是整数。只保留概率最高的 k 个候选 token，其余强制归零，然后在这 k 个中按概率随机采样。k=1 退化为贪婪采样，k 越大越随机。

4. Top-p 采样（核采样 / Nucleus Sampling） ##

top-p是浮点数。只保留概率从高到低累加、直到总和刚好超过 p 的最小候选集，其余归零。候选集大小是动态的，比如概率集中的场景（如 p=0.9 只包涵 3 个 token），概率分散时则可能更大。然后继续在这个候选集中随机采样。

实际使用中，Top-k 和 Top-p 常组合：先 Top-k 限定最大范围，再 Top-p 动态裁剪末尾的长尾。

回到三行代码 ##

有了上面这些铺垫，再来看这三行，每一行的职责就非常清晰了：

// sample a token
token := seq.samplingCtx.Sample(s.lc, seq.iBatch)
seq.samplingCtx.Accept(token, true)
piece := s.model.TokenToPiece(token)
  1. Sample（采样）：拿到模型输出的概率分布（一个 50000 维的向量，每维代表一个 token 的概率），按当前配置的采样策略（比如温度 0.7 + Top-p 0.9 + Top-k 40），从中选出一个 token ID。选中哪个取决于概率——概率高的 token 更容易被选中，但不是必然。
  2. Accept（接受）：将选中的 token ID 追加到生成序列中，并更新 KV Cache（见下文详解）。有了更新后的 KV Cache，下一次推理只需要处理这一个新的 token，之前的所有 token 已经缓存，不需要重复计算。
  3. TokenToPiece（转换为文本）：这是一个字典查表操作，用 token ID 直接查找词汇表，得到对应的文本片段（piece）。比如 ID 15339 → “blue”，ID 1234 → “ing”。片段逐一拼接起来就是最终的生成文本。

KV Cache：让生成越来越快的关键 ##

上面多次提到 KV Cache，这里展开说明它为什么是推理性能的命门。

Transformer 的自注意力机制要求每个 token 与序列中所有 token 计算注意力。在推理阶段：

  • 无缓存：生成第 n 个 token 时，需要重新计算前 n-1 个 token 的所有 Key 和 Value 矩阵 → 复杂度 O(n²)，生成越长越慢
  • 有缓存：前 n-1 个 token 的 K、V 矩阵已经缓存，只需计算新 token 的 K、V → 每步复杂度 O(n)，生成速度恒定

这就是为什么 LLM 生成第一个 token 比较慢（需要处理整个输入 prompt 的全部 token），但之后每个 token 的生成速度基本一致——KV Cache 保证每次只算增量，不重复算存量。

类比：写长文章时，你不会每写一个新字就从开头重读一遍。你记住了前面写了什么，基于记忆续写。KV Cache 就是模型的这块"记忆"。Accept 这一步，本质上就是把刚才新生成的 token 写入记忆，供后续推理使用。

完整生成流程 ##

用户输入 → Tokenization（文本 → token ID 序列）
     → Transformer 推理（基于当前序列，预测每个 token 成为"下一个"的概率）
     → Softmax → 得到一个词汇表大小的概率分布
     → Sample（按采样策略从分布中选一个 token ID）
     → Accept（追加到序列，写入 KV Cache）
     → TokenToPiece（查字典：token ID → 文本片段）
     → 拼接到输出 → 循环，直到生成 EOS 或达到停止条件

总结 ##

Ollama 的成功不在于某种单一技术创新，而在于工程上的系统性整合：

  1. 架构分离：API Server 和 LLM Runner 各司其职，清晰解耦
  2. 格式兼容：支持 GGML/GGUF 等多种格式，屏蔽底层差异
  3. 智能量化：PTQ + imatrix，在精度和效率间取得平衡
  4. 硬件自适应：自动检测 GPU/CPU，选择最优执行后端
  5. 模板系统：Modelfile 让用户用模型训练时的"语言"对话

这种设计哲学让 Ollama 能够适应从个人 MacBook 到 GPU 服务器的各种硬件环境，真正做到了"一次编写，处处运行"的 LLM 部署体验。
