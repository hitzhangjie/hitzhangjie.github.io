Store Buffer、内存屏障与缓存一致性

Posted 2026-07-16 18:41 +0800 by ZhangJie ‐ 4 min read

分享:  

写一个变量，其他线程（或 CPU 核心）什么时候能看到？这个看似简单的问题，答案跨越了编译器优化、CPU 微架构、缓存一致性协议等多个层次。这篇文章尝试把这条链路串起来——从你写下一行 *p = 1 开始，到另一个核读到新值为止，中间到底发生了什么。

C volatile：只管编译器，不管硬件 ##

先从一个最常见的误区开始——C/C++ 中的 volatile 能保证线程可见性吗？

不能。 volatile 只做一件事：告诉编译器，不要把这个变量优化到寄存器里，每次读写都要生成真正的 load/store 指令。但它对 CPU 的 cache hierarchy 和 store buffer 完全透明。

一条 *p = 1 对应的 MOV 指令执行后，数据去了哪里？不是直接到内存，而是：

CPU 流水线 → Store Buffer → L1 Cache → (通过缓存一致性协议扩散到其他核)

C volatile 只保证了第一步——编译器确实生成了 MOV 指令。但这条指令执行后，值只是被写入了 store buffer。Store buffer 什么时候 drain 到 cache？不确定。在 drain 之前，这份数据没有被缓存一致性协议跟踪，其他核根本不知道这个写操作的存在。

这也是为什么 Linux 内核文档明确说 “volatile considered harmful”——内核中用 volatile 做同步几乎永远是错的，必须搭配显式的 smp_mb() / smp_wmb() 才有意义。volatile 在内核中的真正用途是 MMIO（设备寄存器访问），而不是线程同步。

Store Buffer：为什么存在？ ##

如果每次写都要等 cache line 准备好（拿到 Exclusive/Modified 状态）再继续，CPU 流水线会被频繁停顿。Store Buffer 正是为了解决这个问题——它是一种微架构级别的 FIFO 队列，让 CPU 可以"写下即走"，store 指令执行时把值和地址扔进 store buffer，CPU 继续执行后续指令，store buffer 在后台异步地把数据提交到 cache。

但"异步提交"带来了一个副作用：写操作对其他核的可见时机变得不确定。

一次完整的写：从 store buffer 到对其他核可见 ##

让我们追踪一个正常的写操作流程：

Core A 执行 store 指令
    ↓
值写入 Core A 的 Store Buffer（微架构 FIFO 队列）
    ↓
Store Buffer Entry 等待对应 cache line 被获取（需要 Exclusive/Modified 状态）
    ↓
发送 RFO（Read For Ownership）到总线，使其他核该 cache line 变为 Invalid
    ↓
拿到所有权后，Store Buffer 该 entry 被 drain 到 L1 Cache（cache line 变成 Modified）
    ↓
此时缓存一致性协议（MESI/MESIF）开始跟踪这个 cache line
    ↓
其他核可以感知到这个修改

关键点：只有当 store buffer 被 drain 之后，cache line 的状态才变成 Modified，缓存一致性协议才开始跟踪它。在此之前，这个写操作对总线上的其他核是不可见的。

其他核来读：两种不同路径 ##

当 Core B 尝试读取同一个 cache line 所对应的内存地址时，情况取决于 Core A 的 store buffer 是否已经 drain。

路径 A：Store Buffer 已经 drain ##

Core B 发 Read 请求到总线
    ↓
Core A 的 cache controller 嗅探到请求 → 自己有 Modified 副本
    ↓
Core A 通过"缓存到缓存转发"（cache-to-cache intervention）直接把数据给 Core B
    ↓
（无需写回内存，这是现代 CPU 的标准操作）
    ↓
双方 cache line 都变成 Shared 状态

路径 B：Store Buffer 还没 drain ##

这是理解可见性问题的核心。

Core B 发 Read 请求到总线
    ↓
Core A 的 cache controller 嗅探到请求
    ↓
检查 cache → 没有！（因为数据还在 store buffer 里，还没 commit 到 cache）
    ↓
这里出现分叉：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
x86（将 store buffer 纳入一致性域）：
    cache controller 继续检查 store buffer → 找到了！
    直接把 store buffer 中的新值转发给 Core B
    （这里的转发路径：store buffer → 某种内部旁路 → 总线响应 → Core B）
    Core B 拿到最新值 ✓

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ARM / POWER（弱一致性模型，store buffer 对嗅探不可见）：
    cache controller 只查到 cache 是 Shared → 回应"没有新数据"
    Core B 从自己 cache 中拿到旧值 ✗
  
    虽然 Core B 拿到的数据是旧的，但通常这仍是一个合法的读操作
    （取决于内存一致性模型的允许范围）
  
    要想让 Core B 读到新值，Core A 必须在写操作之后执行一个
    内存屏障（如 ARM 的 dmb sy），强制 drain store buffer：
  
    Core A: store → dmb sy（drain store buffer 到 cache）
    Core B: dmb sy → load（等等，更好的方式是 Core A drain 后 Core B 直接读）
  
    实际上 ARMv8 引入了 STLR（store-release）和 LDAR（load-acquire）
    作为更轻量的单向屏障方案，比 full dmb 更高效
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

这里纠正一个常见的误解：有人会说"其他核嗅探到读请求后，可以发现 Core A 有数据在 store buffer 中，然后触发 store buffer drain 再响应"。但在弱一致性 CPU 上，store buffer 对嗅探是不可见的——嗅探只查 cache tag，不查 store buffer。这就是为什么弱一致性 CPU 必须用显式屏障来保证可见性。

x86 TSO：到底有多 “Total”？ ##

x86 的内存一致性模型叫 TSO——Total Store Order。这个名字容易让人误解为"所有操作的全局顺序完全确定"，实际上它有非常精确的含义。

TSO 保证以下四条规则：

重排序方向x86 TSO说明
Store → Store不会重排所有核心看到相同的 store 顺序
Load → Load不会重排所有核心看到相同的 load 顺序
Load → Store不会重排先读后写，这个顺序也被确保
Store → Load可以重排这是 TSO 唯一的"洞"

“Total"指的是所有核心对 store 操作的全局顺序有一致的看法——大家看到的 store 发生顺序是一致的，而不是你有你的顺序、我有我的顺序。

第四条 Store→Load 重排是 TSO 唯一的例外，也是最容易被忽视的陷阱：

// Core A
x = 1;          // store
r1 = y;         // load  ← 这个 load 可能被重排到 store 之前

// Core B  
y = 1;          // store
r2 = x;         // load  ← 同理

// 可能结果：r1 = 0 且 r2 = 0（在 TSO 下是可能发生的！）

这个"洞"就是为什么即使 x86 上 volatile 读不需要硬件屏障，volatile 写仍然需要 StoreLoad 屏障（避免 volatile 写之后的普通读被重排到写之前）。

内存屏障：把一切都串起来 ##

内存屏障是连接软件语义和硬件机制的桥梁。各平台的屏障粒度不同：

操作x86-64ARM64作用
全屏障mfence / LOCK 前缀dmb sydrain store buffer + 阻止所有重排序
Store-Release（不需要，x86 store 自带 release 语义）STLRdrain store buffer，保证之前的 store 对后续 acquire 可见
Load-Acquire（不需要，x86 load 自带 acquire 语义）LDAR保证之后的 load 不会重排到这条指令之前

不同语言在做什么 ##

了解硬件之后，再看各语言的并发原语，就会发现它们殊途同归：

Java volatile：

volatile 写 → JVM 生成 StoreStore + StoreLoad 屏障
            → x86 上即 LOCK 前缀指令
            → drain store buffer，禁止 Store→Load 重排

volatile 读 → JVM 生成 LoadLoad + LoadStore 屏障  
            → x86 上几乎零成本（只需编译器屏障）

Go sync/atomic：

sync.Mutex.Unlock/Lock → runtime/internal/atomic CAS/XCHG
                       → x86 上 LOCK 前缀 → 全屏障

atomic.Value.Store     → x86 上 XCHGQ（隐式 LOCK）→ 全屏障
atomic.Value.Load      → x86 上普通 MOV（无需硬件屏障）
                       → 形成 release-acquire 对，语义与 Java volatile 一致

C11 std::atomic：

std::atomic<T> 默认 memory_order_seq_cst
    → store: x86 上 XCHG（全屏障），ARM 上 STLR
    → load:  x86 上普通 MOV，ARM 上 LDAR
  
memory_order_release / acquire
    → 对应单向屏障，与 Go atomic.Value 语义一致

完整链路 ##

把这些串起来，就是一条从高级语言到底层硬件的完整路径：

层级做什么保证什么
语言原语（volatile / atomic / Mutex）表达并发语义，触发编译器生成正确的屏障源代码级别的 happens-before 关系
编译器插入内存屏障指令，阻止编译器级别重排指令序列符合内存模型的顺序要求
Store Buffer微架构缓冲写操作，对软件透明取决于 CPU 是否将其纳入一致性域
内存屏障指令（mfence / dmb / LOCK）drain store buffer + 阻止 CPU 重排写真正进入 cache，被一致性协议接管
缓存一致性协议（MESI + Snoop）跨核数据同步保证所有核最终看到一致的最新值

理解这套链路，就能回答最初的问题：一段 *p = 1 什么时候对另一个核可见？答案取决于：

  1. 编译器有没有把这次写优化掉？→ volatile / atomic 可以阻止
  2. 写操作是否已经从 store buffer drain 到 cache？→ 内存屏障可以强制
  3. 缓存一致性协议是否已将新值传播给其他核？→ MESI/MESIF 自动保证

而 C volatile 只解决了第 1 层，所以它不够。Java volatile、Go atomic、C atomic 解决了全部 3 层，所以它们才能真正保证线程间可见性。
