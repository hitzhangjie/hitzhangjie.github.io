如何编写 Dockerfile：从入门到系统化实践

Posted 2018-11-11 00:00 +0800 by ZhangJie ‐ 7 min read

分享:  

Dockerfile 是容器化应用的第一道门槛，也是 Docker 工程师的核心基本功。本文从零开始，系统性梳理如何编写一个高质量的 Dockerfile —— 从基础语法到定制镜像，再到构建优化和最佳实践。

1 Dockerfile 是什么 #

Dockerfile 是一个文本文件，其中包含了一系列指令（Instruction），用于描述如何基于一个基础镜像构建出目标镜像。每一条指令都会在基础镜像上创建一个新的镜像层（Layer），最终叠加成一个完整的镜像。

相比于 docker commit 的手动、黑盒方式，Dockerfile 可重复、可审计、可版本化，是构建镜像的唯一推荐方式。

2 指令全览 #

Dockerfile 指令是构建块，下面按使用频率和重要性梳理每条指令的语义。

2.1 FROM — 指定基础镜像 ##

必须是 Dockerfile 的第一条非注释指令。

FROM ubuntu:22.04
FROM golang:1.21-alpine AS builder    # 多阶段构建时给阶段命名
FROM scratch                          # 从空镜像开始，用于构建极简镜像
  • 优先使用官方镜像（_/ubuntu、_/golang 等）
  • 指定精确 tag，不要用 latest（可重复性问题）
  • scratch 是 Docker Hub 上的保留名，代表一个完全空的镜像

2.2 RUN — 构建时执行命令 ##

在镜像构建阶段执行命令，通常用于安装系统包、编译代码等。

# shell 形式
RUN apt-get update && apt-get install -y nginx

# exec 形式
RUN ["apt-get", "update"]

关键原则：多条命令用 && 串联在同一个 RUN 中，并在同一层清理安装缓存。每多一个 RUN 就多一层镜像，会增加镜像体积。

# ✅ 推荐
RUN apt-get update \
    && apt-get install -y --no-install-recommends nginx \
    && rm -rf /var/lib/apt/lists/*

# ❌ 避免
RUN apt-get update
RUN apt-get install -y nginx

2.3 CMD vs ENTRYPOINT — 容器启动行为 ##

这是 Docker 里最容易混淆的一对指令，但理解之后就能灵活控制容器的启动行为。

CMDENTRYPOINT
docker run image arg替换 CMD追加到 ENTRYPOINT 后面
定位提供默认行为/参数定义容器的"主程序"
覆盖方式docker run 命令行参数--entrypoint 标志

三种写法：

# exec 形式（推荐）— JSON 数组，不会启动 shell
CMD ["nginx", "-g", "daemon off;"]

# shell 形式 — 实际执行 /bin/sh -c "nginx ..."
CMD nginx -g "daemon off;"

# 作为 ENTRYPOINT 的默认参数
ENTRYPOINT ["nginx"]
CMD ["-g", "daemon off;"]

推荐实战模式：ENTRYPOINT 指定主程序 + CMD 提供默认参数。用户可以通过 docker run 传参覆盖默认行为，但不会改变入口程序本身。

ENTRYPOINT ["docker-entrypoint.sh"]
CMD ["--config", "/etc/app/default.conf"]

容器作为可执行工具：当希望容器表现得像一个命令行工具时，用 ENTRYPOINT：

FROM alpine
RUN apk add --no-cache curl
ENTRYPOINT ["curl"]
# docker run --rm my-curl -s https://example.com

信号处理注意：exec 形式下容器内 PID 1 就是你的程序，可以正常接收 SIGTERM；shell 形式下 PID 1 是 /bin/sh，不会转发信号，导致 docker stop 超时后才 SIGKILL。

2.4 ENV vs ARG — 环境变量与构建参数 ##

# ENV — 构建时和运行时都有效
ENV APP_HOME=/app \
    NODE_ENV=production

# ARG — 只在构建时有效，运行容器时不可见
ARG VERSION=1.0.0
RUN echo "Building version: $VERSION"
  • ENV 影响运行时的容器行为，适合配置应用环境
  • ARG 适合构建时参数化（版本号、平台架构等），也可用来避免将敏感信息写入镜像层
  • 可以在 docker build 时传入：docker build --build-arg VERSION=2.0.0 .

⚠️ ARG 不适合传密钥/密码，因为 docker history 可以看到。传密钥应使用 BuildKit 的 --secret。

2.5 COPY vs ADD — 文件复制 ##

# COPY — 简单、透明，推荐优先使用
COPY ./app /app
COPY --chown=1000:1000 ./app /app    # 同时设置文件所有者

# ADD — 额外支持 tar 自动解压和 URL 下载
ADD archive.tar.gz /app/             # 自动解压
ADD https://example.com/file /app/   # 下载远程文件

原则：始终优先使用 COPY，除非你明确需要 tar 自动解压或 URL 下载的功能。ADD 的"聪明"行为有时会带来意想不到的结果（如 URL 下载的文件权限问题）。

2.6 WORKDIR — 工作目录 ##

WORKDIR /app
  • 为后续的 RUN、CMD、ENTRYPOINT、COPY、ADD 设置工作目录
  • 优于 RUN cd /app && ... 的方式
  • 如果目录不存在，Docker 会自动创建

2.7 USER — 切换用户 ##

# 创建用户并切换到该用户
RUN groupadd -r appuser && useradd -r -g appuser appuser
USER appuser
  • 避免以 root 运行容器，遵循最小权限原则
  • 注意切换 USER 的时机：安装系统包需要 root，运行应用时用普通用户

2.8 EXPOSE — 声明端口 ##

EXPOSE 8080
EXPOSE 8080/udp

EXPOSE 仅声明容器监听的端口，不会实际将端口发布到宿主机。实际映射需在 docker run -p 或 compose 中指定。它的主要作用是作为镜像的元数据文档，供编排系统读取。

2.9 VOLUME — 声明数据挂载点 ##

VOLUME /data
VOLUME ["/var/log", "/var/db"]
  • 将指定目录标记为外部存储挂载点，防止数据写入容器的可写层
  • 会创建一个匿名卷（anonymous volume），即使容器删除了，卷仍在

2.10 HEALTHCHECK — 健康检查 ##

HEALTHCHECK --interval=30s --timeout=5s --retries=3 --start-period=10s \
  CMD curl -f http://localhost:8080/health || exit 1
  • Docker 根据健康检查的状态标记容器为 healthy、unhealthy 或 starting
  • 编排系统（Swarm、K8s）会依据健康状态做出调度决策

2.11 其他指令 ##

指令说明
LABEL添加元数据标签，如LABEL version="1.0" maintainer="..."
SHELL指定 shell 形式的默认 shell，如SHELL ["/bin/bash", "-c"]
STOPSIGNAL设置docker stop 发送的信号，默认 SIGTERM
ONBUILD定义触发器，当此镜像作为其他镜像的基础镜像时自动执行

3 定制自定义镜像 #

这部分是实操核心：如何从零定制一个适合自己的 Linux 开发镜像或应用镜像。

3.1 选择基础镜像 ##

根据需求场景，在轻量与完整之间做权衡：

基础镜像大小适用场景注意事项
Alpine~5 MB生产运行镜像musl libc，部分二进制不兼容
Debian slim~80 MB通用生产环境去除文档和 man，比完整版小很多
Ubuntu~77 MB需要丰富 apt 生态开发环境优先考虑
Distroless~10 MB高安全要求的运行环境没有 shell/包管理
CentOS / Rocky~200 MBRHEL 兼容环境体积较大，企业场景常见

开发镜像推荐 Ubuntu/Debian，方便调试；生产运行镜像推荐 Alpine 或 Distroless。

3.2 定制 Linux 开发镜像 ##

一个实用的 Linux 开发镜像通常需要：

  • 系统工具链：gcc、make、cmake、autoconf
  • 运行时：Python、Node.js、Go 等
  • 调试工具：gdb、strace、ltrace、tcpdump、curl、vim
  • Shell 配置：zsh/bash 优化，oh-my-zsh、dotfiles
  • 开发辅助：git、ssh、rsync
FROM ubuntu:22.04

# 1. 换国内镜像源（按需）
RUN sed -i 's/archive.ubuntu.com/mirrors.tuna.tsinghua.edu.cn/g' /etc/apt/sources.list

# 2. 安装系统基础包 — 用 --no-install-recommends 减少体积
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential gdb strace \
    git curl wget vim zsh \
    python3 python3-pip \
    && rm -rf /var/lib/apt/lists/*

# 3. 安装 oh-my-zsh
RUN sh -c "$(curl -fsSL https://raw.github.com/ohmyzsh/ohmyzsh/master/tools/install.sh)" "" --unattended

# 4. 复制个性化 dotfiles
COPY .zshrc /root/.zshrc
COPY .vimrc /root/.vimrc
COPY .gitconfig /root/.gitconfig

# 5. 额外工具 — 如安装特定版本的 Go
ARG GO_VERSION=1.21.6
RUN curl -fsSL "https://go.dev/dl/go${GO_VERSION}.linux-amd64.tar.gz" \
    | tar -xz -C /usr/local
ENV PATH=$PATH:/usr/local/go/bin

WORKDIR /workspace
CMD ["/bin/zsh"]

3.3 定制应用运行镜像 ##

应用镜像要贯彻"最小化"原则，只包含运行时必需的内容：

# ========== 构建阶段 ==========
FROM golang:1.21-alpine AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o /app .

# ========== 运行阶段 ==========
FROM alpine:3.18
# 添加必要的运行时依赖（如 ca-certificates）
RUN apk add --no-cache ca-certificates tzdata
# 创建非 root 用户
RUN adduser -D -u 1000 appuser

COPY --from=builder /app /app
USER appuser
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=5s CMD wget -qO- http://localhost:8080/health || exit 1
ENTRYPOINT ["/app"]

3.4 自定义软件包、服务和配置 ##

在 Dockerfile 中加入自定义软件和配置的常见模式：

# ===== 模式 1：安装特定版本的软件包 =====
ARG MYSQL_VERSION=8.0
RUN apt-get update && apt-get install -y mysql-server=${MYSQL_VERSION}*

# ===== 模式 2：从源码编译安装 =====
RUN git clone --depth 1 --branch v1.2.3 https://github.com/xxx/xxx.git \
    && cd xxx && make && make install \
    && cd .. && rm -rf xxx

# ===== 模式 3：配置文件通过 COPY 注入 =====
COPY nginx.conf /etc/nginx/nginx.conf
COPY supervisord.conf /etc/supervisor/conf.d/

# ===== 模式 4：初始化脚本在 ENTRYPOINT 中执行 =====
COPY docker-entrypoint.sh /usr/local/bin/
RUN chmod +x /usr/local/bin/docker-entrypoint.sh
ENTRYPOINT ["docker-entrypoint.sh"]

# 一个典型的 entrypoint 模板：
# #!/bin/bash
# set -e
# # 运行时动态生成配置
# envsubst < /etc/nginx/nginx.conf.template > /etc/nginx/nginx.conf
# # 执行传入的命令
# exec "$@"

docker-entrypoint.sh 是容器化应用里的常见模式：在启动主程序之前，先执行初始化逻辑（检查环境变量、生成配置、等待依赖服务就绪等），最后 exec "$@" 来运行 CMD 指定的主程序。

4 构建优化 #

Docker 18.09 引入了 BuildKit 作为新一代构建引擎（通过 DOCKER_BUILDKIT=1 开启，Docker 23.0+ 已默认启用），它带来了多项构建优化能力。

4.1 层缓存优化 ##

Docker 构建时逐层缓存：如果某层指令及其输入没有变化，则命中缓存。

# ❌ 缓存利用率低 — COPY . . 之后，任何源码变更都会导致后续全部重建
COPY . .
RUN go mod download
RUN go build -o app .

# ✅ 先复制依赖文件，利用缓存
COPY go.mod go.sum ./
RUN go mod download          # 依赖不变则命中缓存
COPY . .
RUN go build -o app .        # 只重建这层

通用原则：把不易变的内容放前面（操作系统包安装 → 语言依赖 → 源码复制）。

4.2 .dockerignore ##

在构建上下文发送给 Docker daemon 之前过滤文件，减少传输量并防止误复制：

# .dockerignore
.git
.gitignore
node_modules
*.log
.env
dist/
Dockerfile
docker-compose.yml
README.md

4.3 多阶段构建 ##

多阶段构建是最重要的镜像减肥手段，让编译环境和运行环境分离：

# 阶段 1：编译
FROM maven:3.9 AS build
COPY . .
RUN mvn package -DskipTests

# 阶段 2：运行（只包含 JAR 文件，不包含 Maven/源码）
FROM openjdk:17-slim
COPY --from=build target/app.jar /app.jar
CMD ["java", "-jar", "/app.jar"]

还支持从多个构建阶段复制：

COPY --from=builder /app/backend /app/backend
COPY --from=frontend-builder /app/dist /app/static

4.4 BuildKit 缓存挂载 ##

这是 BuildKit 引入的最实用的特性之一：构建时挂载缓存目录，跨构建复用，不写入镜像层。

# ===== 缓存包管理器下载 =====
# apt 缓存
RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt \
    apt-get update && apt-get install -y nginx

# pip 缓存
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# go module 缓存
RUN --mount=type=cache,target=/go/pkg/mod \
    --mount=type=cache,target=/root/.cache/go-build \
    go build -o /app .

# npm 缓存
RUN --mount=type=cache,target=/root/.npm \
    npm ci

这意味着即使不同镜像构建之间，包管理器的下载缓存也可复用，极大加速重复构建。

4.5 BuildKit Secret 挂载 ##

安全地传递构建时的敏感信息（密钥、Token、证书），不会写入镜像层：

# 用法：构建时 --secret id=myssh,src=$HOME/.ssh/id_rsa
RUN --mount=type=secret,id=myssh \
    mkdir -p ~/.ssh \
    && cp /run/secrets/myssh ~/.ssh/id_rsa \
    && chmod 600 ~/.ssh/id_rsa \
    && git clone git@github.com:private/repo.git

# 用法：构建时 --secret id=npmrc,src=.npmrc
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm install --production

# 构建命令
docker build --secret id=myssh,src=$HOME/.ssh/id_rsa -t myapp .

4.6 BuildKit SSH 挂载 ##

让构建过程可以使用宿主机的 SSH agent，便于 clone 私有仓库：

# 需要先 eval $(ssh-agent) && ssh-add
RUN --mount=type=ssh \
    git clone git@github.com:private/repo.git

docker build --ssh default -t myapp .

4.7 并行构建与输出 ##

# 构建并直接提取产物到宿主机（不产生镜像）
docker build --output type=local,dest=./out .

# 指定目标阶段
docker build --target builder -t myapp:builder .

5 最佳实践清单 #

整合 Docker 官方和社区总结的生产级实践经验：

编号实践项说明
1一个容器一个进程如果需要多进程管理，用 supervisor 或拆分为多个容器
2使用精确 tagFROM node:18.17.0，而非 node:latest
3合并 RUN 并在同层清理&& 串联，清理包管理缓存
4先复制依赖文件再复制源码最大化层缓存命中率
5用 .dockerignore 减少上下文排除.git、node_modules、构建产物
6优先 COPY 而非 ADD除非需要 tar 解压或 URL
7使用非 root 用户运行USER 切换到普通用户
8EXPOSE 端口声明作为功能和文档约定
9用 exec 形式写 CMD/ENTRYPOINT正确处理信号
10最小化镜像选 Alpine/Distroless，多阶段构建，去除调试工具
11不写密钥进镜像用 BuildKit--secret 或运行时注入
12镜像加标签和元数据LABEL maintainer、version、commit
13配置 HEALTHCHECK让编排系统知道容器状态
14将日志输出到 stdout/stderr不要写入容器内文件
15按字母排序多行包名apt-get install -y \n aaa \n bbb 方便 diff

6 完整示例 #

结合以上所有技巧，一个生产级的 Go Web 应用 Dockerfile：

# ====== 语法检查参考 ======
# docker build --secret id=netrc,src=$HOME/.netrc -t myapp:latest .
# docker run -p 8080:8080 --read-only --tmpfs /tmp myapp:latest

# ====== 阶段 1: 下载依赖 ======
FROM golang:1.21-alpine AS deps
WORKDIR /src
COPY go.mod go.sum ./
RUN --mount=type=cache,target=/go/pkg/mod \
    go mod download

# ====== 阶段 2: 构建 ======
FROM golang:1.21-alpine AS build
WORKDIR /src
COPY --from=deps /go/pkg/mod /go/pkg/mod
COPY . .
RUN --mount=type=cache,target=/go/pkg/mod \
    --mount=type=cache,target=/root/.cache/go-build \
    CGO_ENABLED=0 go build \
    -ldflags="-s -w -X main.version=$(git describe --tags --always)" \
    -o /app .

# ====== 阶段 3: 运行 ======
FROM alpine:3.18

# 安装运行时依赖
RUN apk add --no-cache ca-certificates tzdata curl

# 创建非 root 用户
RUN adduser -D -u 1000 -h /app appuser

# 复制二进制
COPY --from=build /app /app/app
RUN chown -R appuser:appuser /app

# 切换用户和目录
USER appuser
WORKDIR /app

# 声明端口和健康检查
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD wget -qO- http://localhost:8080/health || exit 1

# 入口
ENTRYPOINT ["/app/app"]

💡 这个 Dockerfile 展示了从依赖下载到构建再到运行的完整流程，使用缓存挂载加速构建，最终镜像只有十几 MB，并且以非 root 用户运行。

总结 #

Dockerfile 的编写看似简单，但要写出安全、高效、可维护的 Dockerfile，需要对指令语义、层缓存机制、构建优化有系统的理解。核心思路可以归结为三点：

  1. 指令正确：理解 CMD/ENTRYPOINT、ENV/ARG、COPY/ADD 等底层差异，写出语义正确的 Dockerfile
  2. 构建高效：利用层缓存、多阶段构建、BuildKit 特性（cache/secret/ssh 挂载）优化构建速度
  3. 运行安全：最小镜像、非 root 用户、HEALTHCHECK、信号处理——让容器在生产环境中可靠运行

当你下次需要一个新环境的 Dockerfile 时，不用再从零搜索——回来参考这篇文章的结构和模板即可。
