wait 的唤醒机制与 SIGCHLD 信号的深度解析

Posted 2025-07-17 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

在日常工作中经常会用到 wait/waitpid 来等待子进程退出。有一个困惑了我很久的问题：SIGCHLD 信号是不是用来唤醒 wait 的？毕竟按直觉理解，子进程退出时内核发送 SIGCHLD，然后父进程从 wait 中醒来——这个因果关系看起来非常自然。

但深入研究内核源码后发现，事实并非如此。wait 和 SIGCHLD 是两条独立、互补的通道，它们的角色和适用场景完全不同。

核心结论 #

  • wait 的唤醒：由内核内部的 __wake_up_parent 直接完成，这是一条专门为父子进程同步设计的"VIP 唤醒通道"，不是 SIGCHLD 信号触发的。
  • SIGCHLD 的角色：是一个异步通知，主要服务于那些没有阻塞在 wait 调用中的父进程，或者用于中断父进程正在执行的其他可中断系统调用。

⚠️ 补充说明：SIGCHLD 确实可以唤醒阻塞在 wait 上的父进程（因为 wait 进入的是 TASK_INTERRUPTIBLE 睡眠），但前提是子进程状态真的发生了变化。手动 kill(parent, SIGCHLD) 虽然发送了信号，但子进程状态未变，父进程从信号处理中返回后检查条件不满足，会继续阻塞。实际唤醒起作用的还是 __wake_up_parent。

wait：同步阻塞与 VIP 唤醒通道 #

工作模式 ##

当进程调用 wait() 或 waitpid() 且暂时没有子进程状态变化时：

  1. 内核将该进程状态设置为 TASK_INTERRUPTIBLE（可中断睡眠）。
  2. 将其放入一个专门为等待子进程而设的等待队列：current->signal->wait_chldexit。
  3. 调度器将其换出 CPU，进程开始睡眠等待。

唤醒机制 ##

当子进程状态发生变化（退出、停止等）时：

  • 内核相关代码调用 __wake_up_parent()。
  • 此函数直接、高效地定位到在 wait_chldexit 等待队列上睡眠的父进程。
  • 将父进程状态改回 TASK_RUNNING，使其可以被调度器再次调度执行。

这是一条为 wait 专门设计的"VIP 通道"，确保等待父进程能被精确唤醒，不经过信号传递的通用路径。

与 SIGCHLD 的关系 ##

对于正在 wait() 的进程，SIGCHLD 信号是一个并行的、冗余的通知：

  • 父进程已经被 __wake_up_parent 唤醒，SIGCHLD 本身并没有起到"唤醒 wait“的作用。
  • 如果父进程注册了 SIGCHLD handler，它会在 wait 返回后被调用，但并不影响 wait 本身的唤醒。

SIGCHLD：异步通知与中断机制 #

SIGCHLD 信号服务于那些不想（或不能）阻塞在 wait 上的父进程，主要用于以下两种场景：

场景一：中断其他系统调用 ##

如果父进程正在执行其他可中断的系统调用（如 sleep()、select()、poll()、阻塞的 read()），SIGCHLD 的到来会中断这些调用，使其提前返回，errno 被设为 EINTR。

这给了父进程一个机会去处理子进程事件——通常在收到 EINTR 后再调用 waitpid(..., WNOHANG) 来回收子进程。

while (1) {
    int ret = select(nfds, &rfds, NULL, NULL, NULL);
    if (ret < 0 && errno == EINTR) {
        // 可能是 SIGCHLD 打断了 select
        while (waitpid(-1, &status, WNOHANG) > 0) {
            // 回收已退出的子进程
        }
        continue;
    }
    // 正常处理 I/O 事件
}

场景二：触发信号处理函数 ##

如果父进程正在用户空间执行代码，SIGCHLD 会被标记为 pending。在下一次从内核态返回用户态时，内核执行注册的 SIGCHLD 处理函数：

void sigchld_handler(int sig) {
    int status;
    pid_t pid;
    while ((pid = waitpid(-1, &status, WNOHANG)) > 0) {
        printf("child %d exited\n", pid);
    }
}

// 注册 handler
signal(SIGCHLD, sigchld_handler);

两种机制对比 #

特性wait 系统调用SIGCHLD 信号
模型主动、同步的等待被动、异步的通知
唤醒者内核内部的 __wake_up_parent信号传递机制
主要目标阻塞当前进程，等待子进程变化通知正在做其他事的进程
睡眠方式TASK_INTERRUPTIBLE不睡眠（或中断其他睡眠）
典型场景waitpid(-1, &status, 0) 阻塞回收signal(SIGCHLD, handler) + waitpid(WNOHANG)

总结 #

wait 和 SIGCHLD 是两种并行且互补的机制：

  • wait 适合同步等待：父进程的主要任务是等待子进程，没别的事可做。
  • SIGCHLD 适合异步通知：父进程有主循环正在做其他事（如事件循环），需要被中断才能去处理子进程。

理解这两种机制的分工，才能在实际编程中选择合适的子进程回收策略，避免僵尸进程或丢失子进程退出通知。
