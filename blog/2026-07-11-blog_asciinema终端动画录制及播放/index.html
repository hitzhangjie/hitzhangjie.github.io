asciinema 与 asciinema-player 的核心工作原理

Posted 2026-07-11 16:26 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 #

想在网页上展示一段「终端里正在跑命令」的效果，常见做法有两种：录一段屏幕视频，或者用 GIF。前者体积大、无法选中文字；后者画质差、帧率低。

asciinema 走的是另一条路：不录像素，只录终端字节流。录制产物是轻量的 .cast 文本文件；在浏览器里由 asciinema-player 按时间戳回放。博客首页的 git log 滚动动画，就是这条链路。

本文顺着「录制 → 存储 → 播放」把原理拆开，回答两个问题：

  1. asciinema rec 到底捕获了什么？
  2. 浏览器里的 player 如何把一串字节「演」成终端画面？

录制：PTY 上的透明中间人 #

和录屏的本质区别 ##

传统录屏工具（OBS、ffmpeg 抓屏）记录的是光栅帧：每一帧是一块位图。asciinema 记录的是终端会话的原始输出——普通字符，加上控制光标、颜色、清屏的 ANSI/VT escape 序列。

因此：

  • 文件通常只有几 KB 到几百 KB，gzip 后往往还能再压到约 8%
  • 播放时可复制文字、可缩放、可换主题
  • 代价是：只能还原「终端能表达」的内容，不能录 GUI

伪终端（PTY）是关键 ##

录制机制和 ssh、script、screen 同源，都依赖 Unix 伪终端（PTY）。一对 PTY 由两个端组成（若对 TTY / PTY / Console / 终端模拟器这些词本身发懵，可先看 终端、TTY、PTY、Console：把这些概念一次理清）：

端角色
slave对进程表现为「真实终端」：isatty() 为真，支持窗口大小、TERM 等
master控制端：谁持有 master，谁就能读写 slave 上进程的全部 I/O

asciinema rec 时大致是这样：

键盘 / 用户输入
       │
       ▼
┌──────────────────┐
│  asciinema       │  ← 充当 PTY master
│  (rec / stream)  │
└────────┬─────────┘
         │ 透传 I/O，同时打时间戳落盘
         ▼
┌──────────────────┐
│  shell / 命令     │  ← 跑在 PTY slave 上
│  (stdout/stderr) │
└──────────────────┘

因为站在 master 上，asciinema 能透明拿到：

  • 程序写出的全部原始字节（含不可见 escape）
  • 终端窗口 resize 事件
  • （可选）键盘 input 事件

程序并不知道自己被录了——它以为自己连着一台普通终端。这也是为什么带颜色的 TUI（如 htop、sl）能被完整还原：颜色和光标移动本来就编码在输出字节里。

本地重放 vs 浏览器重放 ##

asciinema play demo.cast 很简单：把事件流按时序写回你当前的真实终端，由终端自己解释 ANSI。录制端几乎不做「渲染」。

浏览器里没有真实终端，所以 player 必须自带一套 VT 模拟器——后面会细说。

存储：asciicast 事件流 #

文件长什么样 ##

asciicast 是 newline-delimited JSON。第一行是 header，后续每行一条事件。当前常见的是 v2（v3 把时间戳改成了相对上一事件的 interval，语义类似）。

一个简化的 v2 片段：

{"version": 2, "width": 80, "height": 24, "timestamp": 1783757676, "env": {"SHELL": "/bin/bash", "TERM": "xterm-256color"}}
[0.02197, "o", "* 77ffc6f ... feat: replace homepage ...\r\n"]
[0.173439, "o", "* 93b1ec8 ... fix/frontmatter ...\r\n"]
[0.324163, "o", "* 82f038f ...\r\n"]

Header ##

字段含义
version格式版本（2 或 3）
width / height录制时的终端列数 / 行数
timestamp录制开始的 Unix 时间（可选）
env如 SHELL、TERM，影响程序行为和颜色能力

事件三元组 ##

每条事件是 [time, code, data]：

字段含义
timev2：相对录制开始的秒数；v3：相对上一条事件的间隔
code事件类型
data该类型对应的载荷字符串

常见 code：

code含义
"o"output：终端输出字节（主体）
"i"input：用户按键（可选录制）
"r"resize：窗口大小变化
"m"marker：书签/章节标记
"x"扩展事件（实现方可忽略未知类型）

data 里经常直接带着 \r\n、\x1b[31m 这类序列——原样保存，不做语义解析。解析留给播放端。

为什么「节奏」也能保留 ##

终端演示好看，往往不在于内容，而在于打字速度、停顿、滚动节奏。事件上的时间戳把这些都编码进去了：两条 "o" 之间隔 0.15 秒，播放时就会像一行行刷出来。

例如用脚本把 git log 按行延迟写出，再交给 asciinema rec -c ...，得到的 .cast 播放起来就是平滑滚动——本质是人为控制了 "o" 事件的时间间隔，而不是改了 player。

播放：浏览器里的 VT 模拟器 #

整体流水线 ##

asciinema-player 要做的，不是「播视频」，而是：

.cast 文件
    │
    ▼
 Parser          解析 header + 事件流 → 内部 Recording 模型
    │
    ▼
 Scheduler       按时间戳调度下一条事件
    │
    ▼
 Terminal emu    把字节喂进 VT 状态机，更新字符网格
 (avt / WASM)
    │
    ▼
 Renderer        把网格画到 DOM / Canvas
    +
 Control UI      播放 / 暂停 / 倍速 / 进度条 / 全屏

对外 API 通常一行就够：

AsciinemaPlayer.create('/casts/demo.cast', document.getElementById('demo'));

为什么必须自带终端模拟器 ##

浏览器拿到的只是字符串。要正确显示，必须理解例如：

  • \x1b[2J 清屏
  • \x1b[31m 前景变红
  • \x1b[1;1H 光标移到左上角
  • \r 回车不换行、\n 换行滚动

player 内嵌的终端模拟器叫 avt（用 Rust 实现，编译成 WASM），其解析思路可追溯到 Paul Williams 的 ANSI 兼容视频终端状态机。它专注显示侧仿真：维护 cols × rows 的字符单元网格，处理颜色、粗体、下划线、反色、光标与滚动，从而在网页上复刻当时终端窗口的样子。

对比一下：

场景谁负责解释 ANSI
asciinema play本机真实终端（xterm、iTerm、Windows Terminal…）
网页嵌入player 内的 avt（WASM）

实现分层（JS + WASM） ##

现代 player（3.x）大致拆成：

  • JavaScript：加载资源、UI、定时调度、主题与交互
  • Rust → WASM：解析与终端仿真（CPU 密集）
  • 可选 WebWorker：把解析/仿真放到后台线程，高带宽、高帧率录制时 UI 更顺；普通 demo 通常不必折腾 split 构建

解析器默认吃 asciicast v1/v2/v3，也支持通过自定义 parser 接入 ttyrec 等其它格式——内部统一成 { cols, rows, events } 模型后再驱动播放。

和「假终端动画」的差别 ##

有些页面用 CSS/setInterval 往 <pre> 里塞文本，看起来也像终端。和 asciinema 的差别在于：

  • 假动画通常没有完整 VT 状态机，复杂 TUI、颜色、光标跳转容易露馅
  • asciinema 保留的是真实程序在真实 PTY 上的输出，保真度由协议保证，而不是由前端手写动画保证

一条完整链路（以博客首页为例） #

把上面串起来，就是博客首页动画在做的事：

  1. 录制：asciinema rec -c '<command>' --cols N --rows M，在 PTY 上跑命令，捕获带时间戳的输出
  2. 落盘：得到 static/casts/animation.cast（header + "o" 事件流）
  3. 嵌入：页面挂一个容器，加载 asciinema-player 的 JS/CSS
  4. 播放：AsciinemaPlayer.create(castUrl, el, options) 拉取 .cast，按时序喂给 WASM 终端模拟器并渲染

录制脚本里若用「按行 sleep 再打印」的方式控制节奏，改的是事件时间轴；player 本身只是忠实回放。

总结 #

asciinema 的核心可以压成两句话：

  1. 录制：以 PTY master 身份夹在用户与 shell 之间，把原始终端 I/O 打成带时间戳的事件日志（asciicast），而不是录视频帧。
  2. 播放：在浏览器里用自研 VT 模拟器（avt / WASM）解释这些字节，重建字符网格并渲染；终端内重放则直接把字节喂回真实终端。

理解这条链路之后，很多工程选择就顺理成章了：为什么文件极小、为什么能复制文字、为什么必须声明 cols/rows、为什么复杂 TUI 也能还原、以及为什么「调 delay」能改变动画观感——改的是事件时间，不是播放器魔法。
