Go 网络 IO 与 SIGPIPE：closed connection 为何不会把进程打挂

Posted 2019-04-18 12:00 +0800 by ZhangJie ‐ 8 min read

分享:  

背景 ##

SIGPIPE 的默认行为是让进程 crash。进程需要通过 sigaction 显式忽略或自行处理该信号；如果语言运行时接管了信号处理，行为就会不同。

本文从网络 IO 场景出发，看 Go 如何优雅处理写 closed TCP 连接时的 SIGPIPE 问题，避免服务进程因此挂掉——也能再次体会 Go 在工程细节上的用心。

问题场景 ##

对一个已关闭的 tcpconn 执行写操作，Go 里可能有不同表现。典型场景是 client 通过连接池访问 server：

  1. Server 主动关闭连接：server 认为连接空闲，执行 conn.Close()；client 端写一次、再写一次，会报 broken pipe。
  2. 本端已 Close 再写：server 侧已 conn.Close()，再对本连接写一次或多次，会报 use of closed network connection。
  3. SIGPIPE 疑问：出现 broken pipe 时，进程为什么没有被 SIGPIPE 打死？

broken pipe vs use of closed network connection ##

场景一（broken pipe）

Client 第一次写时，server 会回 FIN 或 RST，client 刚感知到对端连接销毁，但上层应用还没有调用 conn.Close() 销毁本端 fd（见下文 mutexClosed 标记）。第二次写会走到 syscall.Write(...)，返回 EPIPE，上层 err.Error() 即：

errors[EPIPE] = "broken pipe"

场景二（use of closed network connection）

本端执行 conn.Close() 会标记并销毁 fd（mutexClosed）。之后再 conn.Write(...) 时，可直接检测到该标记，返回 use of closed network connection，根本不会进入下层 syscall.Write(...)。

一句话区分

错误本质
use of closed network connection应用层显式 conn.Close()，fd 已标记关闭，写操作在 Go 层被拦住
broken pipe协议栈发现连接已不可写，syscall.Write 返回 EPIPE

另外，Go 对 SIGPIPE 做了特殊处理，尽量避免服务端因「写 closed tcpconn」而挂掉。use of closed network connection 本身也是在连接关闭时拦住写操作，从而避免走到 SIGPIPE；更大的好处是：无需进入内核。

官方文档：Go 如何约定 SIGPIPE ##

源码注释见 os/signal 包文档（signal/doc.go）：

SIGPIPE

When a Go program writes to a broken pipe, the kernel will raise a
SIGPIPE signal.

If the program has not called Notify to receive SIGPIPE signals, then
the behavior depends on the file descriptor number. A write to a
broken pipe on file descriptors 1 or 2 (standard output or standard
error) will cause the program to exit with a SIGPIPE signal. A write
to a broken pipe on some other file descriptor will take no action on
the SIGPIPE signal, and the write will fail with an EPIPE error.

If the program has called Notify to receive SIGPIPE signals, the file
descriptor number does not matter. The SIGPIPE signal will be
delivered to the Notify channel, and the write will fail with an EPIPE
error.

This means that, by default, command line programs will behave like
typical Unix command line programs, while other programs will not
crash with SIGPIPE when writing to a closed network connection.

要点：

  • 未 signal.Notify 收 SIGPIPE 时：写 stdout/stderr（fd 1/2） 的 broken pipe 会让进程以 SIGPIPE 退出；写其他 fd（如网络连接）只返回 EPIPE，不杀进程。
  • 已 Notify 收 SIGPIPE 时：信号进 channel，写操作仍以 EPIPE 失败，与 fd 编号无关。
  • 因此默认下：命令行程序行为像传统 Unix 工具；网络服务写 closed connection 不会因 SIGPIPE 崩掉。

下面按调用链结合源码回答三个问题：

  1. Go 如何处理 EPIPE + SIGPIPE？
  2. Go 如何处理 tcpconn.Write()？
  3. Go 如何处理 conn.Close()？

源码摘录按实际执行路径组织，建议对照源码跳转阅读。

How Go handle SIGPIPE？ ##

epipecheck：仅对 stdout/stderr 升格为 SIGPIPE ##

os/file_unix.go：

// epipecheck raises SIGPIPE if we get an EPIPE error on standard
// output or standard error. See the SIGPIPE docs in os/signal, and
// issue 11845.
func epipecheck(file *File, e error) {
    // 如果 write pipe 时操作系统返回 EPIPE，libc 往往会给进程发 SIGPIPE。
    // Go 按场景做了区分：
    // - 命令行程序写 stdout/stderr 出现 EPIPE → 可挂掉进程（贴近 Unix 习惯）；
    // - 网络服务写 closed tcpconn 返回 EPIPE → 返回 error "broken pipe"，
    //   而不是把 SIGPIPE 直接交给默认行为打死进程。
    if e == syscall.EPIPE && file.stdoutOrErr {
       sigpipe()
    }
}

How Go handle tcpconn.Write()？ ##

上层 Write：写完再 epipecheck ##

os/file.go：

// Write writes len(b) bytes to the File.
// It returns the number of bytes written and an error, if any.
// Write returns a non-nil error when n != len(b).
func (f *File) Write(b []byte) (n int, err error) {
	if err := f.checkValid("write"); err != nil {
		return 0, err
	}
    // 执行数据写操作
	n, e := f.write(b)
	if n < 0 {
		n = 0
	}
	if n != len(b) {
		err = io.ErrShortWrite
	}
	// 检查系统调用是否返回 syscall.EPIPE（见上文 epipecheck）：
    // - f 是 stdout/stderr → 可能挂掉进程；
    // - 否则 → 返回 error "broken pipe"。
	epipecheck(f, e)

	if e != nil {
		err = f.wrapErr("write", e)
	}

	return n, err
}

下沉到 poll.FD ##

os/file_unix.go：

// write writes len(b) bytes to the File.
// It returns the number of bytes written and an error, if any.
func (f *File) write(b []byte) (n int, err error) {
	n, err = f.pfd.Write(b)

internal/poll/fd_unix.go：

// Write implements io.Writer.
func (fd *FD) Write(p []byte) (int, error) {
    // 写前抢 fd 写锁，保证当前 p []byte 尽量连续写出
    //（syscall.Write 不一定一次写完，需考虑发送缓冲）。
	if err := fd.writeLock(); err != nil {
		return 0, err
	}
	defer fd.writeUnlock()
	if err := fd.pd.prepareWrite(fd.isFile); err != nil {
		return 0, err
	}
	var nn int
	for {
		max := len(p)
		if fd.IsStream && max-nn > maxRW {
			max = nn + maxRW
		}
        // fd 有效则进入内核写；连接/流已关闭时，内核返回 EPIPE，
        // 上层表现为 error "broken pipe"。
		n, err := syscall.Write(fd.Sysfd, p[nn:max])
		if n > 0 {
			nn += n
		}
		if nn == len(p) {
			return nn, err
		}
		if err == syscall.EAGAIN && fd.pd.pollable() {
			if err = fd.pd.waitWrite(fd.isFile); err == nil {
				continue
			}
		}
		if err != nil {
			return nn, err
		}
		if n == 0 {
			return nn, io.ErrUnexpectedEOF
		}
	}
}

写锁与 mutexClosed：Close 后写直接失败 ##

internal/poll/fd_mutex.go：

// writeLock adds a reference to fd and locks fd for writing.
// It returns an error when fd cannot be used for writing.
func (fd *FD) writeLock() error {
    // 加锁失败只有一种情况：fd 已被关闭
	if !fd.fdmu.rwlock(false) {
		return errClosing(fd.isFile)
	}
	return nil
}

// lock adds a reference to mu and locks mu.
// It reports whether mu is available for reading or writing.
func (mu *fdMutex) rwlock(read bool) bool {
	var mutexBit, mutexWait, mutexMask uint64
	var mutexSema *uint32
	if read {						// 读锁相关
		...
	} else {						// 写锁相关
		mutexBit = mutexWLock
		mutexWait = mutexWWait
		mutexMask = mutexWMask
		mutexSema = &mu.wsema
	}
	for {							// 自旋方式加锁
		old := atomic.LoadUint64(&mu.state)
		if old&mutexClosed != 0 {	// fd 已经关闭
			return false
		}
		var new uint64
		if old&mutexBit == 0 {		// 锁空闲，尝试获取
			// Lock is free, acquire it.
			new = (old | mutexBit) + mutexRef
			...
		} else {					// 被他人持有，等待
			// Wait for lock.
			new = old + mutexWait
			...
		}
        // CAS 获取锁
		if atomic.CompareAndSwapUint64(&mu.state, old, new) {
			if old&mutexBit == 0 {
				return true
			}
			runtime_Semacquire(mutexSema)
			// The signaller has subtracted mutexWait.
		}
	}
}

错误文案定义在 internal/poll/fd.go：

// ErrNetClosing is returned when a network descriptor is used after
// it has been closed. Keep this string consistent because of issue
// #4373: since historically programs have not been able to detect
// this error, they look for the string.
var ErrNetClosing = errors.New("use of closed network connection")

// ErrFileClosing is returned when a file descriptor is used after it
// has been closed.
var ErrFileClosing = errors.New("use of closed file")

// Return the appropriate closing error based on isFile.
func errClosing(isFile bool) error {
	if isFile {
		return ErrFileClosing
	}
	return ErrNetClosing
}

系统调用与 “broken pipe” 字符串 ##

syscall/syscall_unix.go：

func Write(fd int, p []byte) (n int, err error) {
	if race.Enabled {
		race.ReleaseMerge(unsafe.Pointer(&ioSync))
	}
	n, err = write(fd, p)
	if race.Enabled && n > 0 {
		race.ReadRange(unsafe.Pointer(&p[0]), n)
	}
	if msanenabled && n > 0 {
		msanRead(unsafe.Pointer(&p[0]), n)
	}
	return
}

How Go handle conn.Close()？ ##

Close：标记 closed + 唤醒阻塞 IO ##

internal/poll/fd_unix.go：

// Close closes the FD. The underlying file descriptor is closed by the
// destroy method when there are no remaining references.
func (fd *FD) Close() error {
	if !fd.fdmu.increfAndClose() {
		return errClosing(fd.isFile)
	}

	// Unblock any I/O.  Once it all unblocks and returns,
	// so that it cannot be referring to fd.sysfd anymore,
	// the final decref will close fd.sysfd. This should happen
	// fairly quickly, since all the I/O is non-blocking, and any
	// attempts to block in the pollDesc will return errClosing(fd.isFile).
	fd.pd.evict()

	// The call to decref will call destroy if there are no other
	// references.
	err := fd.decref()

	...

	return err
}

// Shutdown wraps the shutdown network call.
func (fd *FD) Shutdown(how int) error {
	if err := fd.incref(); err != nil {
		return err
	}
	defer fd.decref()
	return syscall.Shutdown(fd.Sysfd, how)
}

increfAndClose：打上 mutexClosed ##

internal/poll/fd_mutex.go：

// increfAndClose sets the state of mu to closed.
// It returns false if the file was already closed.
func (mu *fdMutex) increfAndClose() bool {
	for {
		old := atomic.LoadUint64(&mu.state)
        if old&mutexClosed != 0 {						// 重复 close 同一 fd
			return false
		}
		// Mark as closed and acquire a reference.
		new := (old | mutexClosed) + mutexRef			// 标记 mutexClosed
		...
		// Remove all read and write waiters.
		new &^= mutexRMask | mutexWMask
		if atomic.CompareAndSwapUint64(&mu.state, old, new) {
			// Wake all read and write waiters,
			// they will observe closed flag after wakeup.
           	...
			return true
		}
	}
}

// decref removes a reference from fd.
// It also closes fd when the state of fd is set to closed and there
// is no remaining reference.
func (fd *FD) decref() error {
	if fd.fdmu.decref() {
		return fd.destroy()
	}
	return nil
}

总结 ##

  1. use of closed network connection：本端已 Close，fdMutex 打了 mutexClosed，写锁阶段直接失败，不进内核。
  2. broken pipe：本端 fd 仍可用，内核写发现 pipe/连接已断，返回 EPIPE，映射为 "broken pipe"。
  3. SIGPIPE：Go 默认只在 stdout/stderr 写 EPIPE 时走 sigpipe()（未 Notify 则可能退出）；网络 fd 写 EPIPE 一般只返回 error，服务进程不会被默认 SIGPIPE 行为打挂。
  4. 这套分层（应用层 closed 标记 → 系统调用 EPIPE → 按 fd 类型决定是否升格 SIGPIPE）让 CLI 保持 Unix 习惯，又让网络服务更稳——是典型的工程化取舍。
