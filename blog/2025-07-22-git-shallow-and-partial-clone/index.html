Git 浅克隆与部分克隆：原理与实践

Posted 2025-07-22 00:00 +0800 by ZhangJie ‐ 4 min read

分享:  

背景 #

克隆一个大型仓库时，我们经常会遇到这样的问题：历史太深、对象太多，git clone 慢得让人焦虑。团队仓库跑了好几年，tag 上百个，branch 几十条，.git 动不动就上 GB——如果只是想把代码拉下来改两行提个 MR，等一个完整克隆实在太不划算了。

Git 提供了两类机制来解决这个问题：

  • 浅克隆（shallow clone）：只拉取最近若干次提交的历史，用 --depth 控制深度。
  • 部分克隆（partial clone）：只拉取你需要的对象类型，用 --filter 控制过滤条件。

两个机制也可以组合使用。但在讲「怎么用」之前，有必要先搞清楚它们「怎么工作」——理解了原理，才能判断在 CI、本地开发等不同场景下该选哪个。

本文假定你已经了解 Git 对象数据库的基本概念（blob、tree、commit、tag 四种对象的含义和关系）。如果还不熟悉，建议先阅读 Git 对象数据库：从原理到实操。

Shallow Clone：截断历史深度 #

基本用法 ##

# 只克隆最近 1 次提交
git clone --depth 1 <url>

# 只克隆最近 5 次提交
git clone --depth 5 <url>

# 只克隆某个日期之后的提交
git clone --shallow-since="2024-01-01" <url>

# 只克隆某个 tag 到当前的所有提交
git clone --shallow-exclude="v2.0" <url>

这些选项可以组合。比如 --depth 10 --shallow-since="2024-06-01"，最终的提交数量就是两个条件的交集。

底层机制 ##

--depth 工作流程 ##

Shallow clone 的核心行为在 fetch 阶段。以 git clone --depth 3 为例：

  1. 客户端在 fetch 请求中声明 depth 3，想让服务端只发送最近 3 个 commit。
  2. 服务端从远端分支 HEAD 开始，沿着 parent 链往前数 3 个 commit，打包这些 commit 以及关联的 tree 和 blob，发给客户端。
  3. 客户端接收到之后，在本地写一个 .git/shallow 文件，里面记录截断点——也就是第 4 个 commit 的 SHA1。

这个 .git/shallow 文件的作用是什么呢？它告诉本地的 Git：这个 commit 的 parent 我们不知道，当它没有 parent 就行了。正常情况下，Git 认为一个 commit 要么没有 parent（初始提交），要么 parent 一定存在。但 shallow clone 打破了这条规则——浅克隆的根 commit 被标记在 .git/shallow 中，Git 就知道「到这里就到底了，别再往上追溯」。

文件内容很简单，每行一个 commit SHA1：

# .git/shallow
a1b2c3d4e5f6789012345678901234567890abcd

注意：--depth 的数值本身不会被保存——Git 不记录「深度 = 3」，只记录截断点的 commit SHA1。深度只是 clone/fetch 时告诉服务端「往回走几步」的一个一次性参数，传完就扔了。后续一切加深操作都以 .git/shallow 里当前的截断点为准，和之前的深度数字毫无关系。

截断点的影响 ##

因为历史在截断点被切断，很多依赖于完整历史的操作会受影响：

操作能否正常工作
git log✅ 只能看到浅层内的提交
git diff✅ 只要涉及的 commit 都在浅层内
git blame⚠️ 追不到浅层之外，可能显示截断点 commit 的 author
git log -- <path>⚠️ 文件历史在浅层外就看不到了
git merge 与远程分支✅ 正常，但 merge base 如果落到浅层外会失败

这里有个坑：在 shallow clone 的仓库里 push 新 commit 是允许的，但 push 到「浅层之外的分支」会失败，因为本地缺少 remote 分支的全部历史。

加深和解除 ##

如果需要更多历史，可以动态加深：

# 再多拉 10 个 commit
git fetch --deepen=10

# 拉到某个时间点
git fetch --deepen-since="2023-01-01"

# 完全解除浅克隆，拉取全部历史
git fetch --unshallow

--unshallow 本质上是把 .git/shallow 清空，然后做一次完整 fetch。这之后仓库就跟普通克隆一样了。

加深的方向是往过去追溯，而不是从远程 HEAD 重新截取。 举个例子，假设远程历史是这样的：

A <- B <- C <- D <- E <- F <- G <- H  (origin/main)

你 git clone --depth 3 之后，本地只有 F <- G <- H，.git/shallow 记录着截断点 F。此时执行 git fetch --deepen=2，Git 会从截断点 F 往过去再追溯 2 个 commit，得到 D <- E <- F <- G <- H，截断点更新为 D。

那如果远程在 clone 之后又有新提交（比如 I、J）呢？git fetch 会正常拉取这些新 commit，--deepen 只负责「额外再往深挖 N 个」——两者独立，互不干扰。加深操作立即生效，fetch 一完成，.git/shallow 就更新了。

Shallow Clone 的局限 ##

Shallow clone 的核心局限在于：它是按「提交次数」截断的，而非按「内容大小」截断。

举个例子：一个仓库最近 5 次提交中改了 5GB 的二进制文件，--depth 5 仍然要拉这 5GB。反过来，如果 100 次提交全都是改 README（每个 blob 几 KB），拉取全量历史也很快。

这就是为什么 shallow clone 不能解决所有场景的仓库膨胀问题——它只减少了 commit 和历史 tree 的拉取，不减少 blob 的拉取。该拉的大文件一个也跑不掉。

Partial Clone：按需过滤对象 #

Partial clone 的思路不同：不是截断历史，而是跳过某些不立即需要的对象类型，真正使用时再按需拉取。

Partial clone 依赖于 gitprotocol-v2（Git 2.18+ 默认启用），通过新协议实现了「服务端过滤 + 客户端按需补充」。

三种常用 filter ##

--filter=blob:none（blobless clone） ##

最常用的选项，只拉取 commit 和 tree，不拉取任何 blob 文件内容：

git clone --filter=blob:none <url>

它是什么意思呢？服务端会发送所有 commit 和 tree 对象（所以能看到完整历史），但 blob 对象（也就是文件的具体内容）一个都不发。当你在本地 git checkout、git diff 或 git show 时，Git 发现某个 blob 本地不存在，会自动向服务端发送一个 fetch 请求，按需拉取这个 blob。

效果：

  • 历史完整：git log 全量可用。
  • 初始克隆快：不传输大文件，.git 目录初始很小。
  • 使用时延迟：第一次 checkout 或 diff 大文件时会有网络请求延迟。

--filter=tree:0（treeless clone） ##

跳过所有 tree 和 blob，只拉取 commit：

git clone --filter=tree:0 <url>

这个更极端，clone 下来基本上只有一个 commit graph。几乎做不了什么操作——连 git checkout 都会需要按需拉取大量 tree 和 blob。主要用于自动化场景，比如写一个脚本只检查 commit message 格式，不需要任何文件内容。

--filter=blob:limit=<size>（按大小过滤） ##

只跳过超过指定大小的 blob：

# 跳过大于 1MB 的 blob
git clone --filter=blob:limit=1m <url>

这个适合「拉取全部小文件，跳过大文件」的场景。大文件可以在第一次需要时按需拉取。

底层机制：Promisor Remote ##

Partial clone 的按需拉取，基于一个叫 promisor remote 的概念。

克隆时使用了 --filter，Git 会在 .git/config 中记录：

[remote "origin"]
    url = https://example.com/repo.git
    fetch = +refs/heads/*:refs/remotes/origin/*
    promisor = true
    partialclonefilter = blob:none

promisor = true 标记了这个 remote 是个「承诺提供者」——Git 被告知：「你可以先不拉全部对象，需要时来找我」。

当本地缺少某个对象时，Git 不会报错，而是触发一个懒加载：向 promisor remote 发送一次窄请求，只拉回这个对象。这个过程对用户是透明的——除了第一次访问会有网络延迟，后续就跟本地对象一样了。

注意：如果脱离网络（比如在飞机上），promisor remote 无法访问，任何涉及缺失对象的操作都会失败。Partial clone 的仓库需要一定的网络可达性。

部分克隆对象的状态管理 ##

Partial clone 的仓库中，对象有三种状态：

状态说明
存在（present）对象在 .git/objects/ 中，正常可用
缺失（missing）对象不在本地，需要通过 promisor remote 拉取
承诺存在（promised）对象虽然不在本地，但 Git 知道它在 promisor remote 那边，不会报错

Git 通过 .git/objects/info/exclude 或 .promisor 标记来管理这些状态。每次本地需要缺失的对象时，Git 都会自动触发一次窄 fetch，只拉这一个对象。

限制拉取的分支范围 #

除了深度（--depth）和对象类型（--filter），还有第三个维度可以控制 clone 的体积：限制拉取哪些分支。

在一个分支泛滥的仓库里——每个人都在自己的 feature 分支上工作，分支长期不删、成百上千——如果你 git clone 不加限制，默认会把所有分支的 ref 和它们指向的历史全部拉下来。但实际上你可能只关心 main 和当前在做的两三条分支。

--single-branch ##

clone 时最简单的做法：

git clone --single-branch --branch=main <url>

这只会拉取 main 分支的历史，其他分支的 ref 完全不会出现在 refs/remotes/origin/ 下。它的原理很简单：把默认的 fetch refspec 从 +refs/heads/*:refs/remotes/origin/* 改为 +refs/heads/main:refs/remotes/origin/main。

之后如果需要添加其他分支，可以手动 fetch：

git fetch origin feature-xxx:refs/remotes/origin/feature-xxx

或者直接改 .git/config 把 refspec 加回去。

自定义 fetch refspec ##

.git/config 中的 fetch 行本质上就是一个 refspec，完全可以按需定制。比如只拉 main 和所有 feature/ 前缀的分支：

[remote "origin"]
    url = https://example.com/repo.git
    fetch = +refs/heads/main:refs/remotes/origin/main
    fetch = +refs/heads/feature/*:refs/remotes/origin/feature/*

这样 git fetch 就只会同步这两类分支，其他几百条个人分支一根毛都不会碰。

也可以通过命令行操作：

# 查看当前 fetch refspec
git config --get-all remote.origin.fetch

# 设为只拉 main
git remote set-branches origin main

# 添加更多分支
git remote set-branches --add origin feature-xxx

三个维度的组合 ##

Shallow clone（深度）、partial clone（对象类型）、fetch refspec（分支范围）三者完全正交，可以任意组合：

git clone --depth 1 --filter=blob:none --single-branch --branch=main <url>

这行命令的意思是：只拉 main 分支、只拉到最新一个 commit、不拉 blob 文件内容。对于一个分支上百、历史数万、文件 GB 级的仓库，这几乎是把 clone 开销压到了理论最低。

Shallow Clone vs Partial Clone：本质区别 #

很多人觉得这两个差不多——都是「不完全克隆」。但它们解决的问题不同，底层机制也完全不一样。

维度Shallow Clone (--depth)Partial Clone (--filter)
核心思路截断提交历史跳过某些对象类型
历史不完整，只有最近 N 条完整，所有 commit 都在
文件内容浅层内的文件全量拉取按需拉取，不用的不拉
依赖的 Git 版本任意版本Git 2.18+（需要 protocol v2）
离线可用性好（所有内容都已拉取）差（缺失对象需要网络）
典型场景CI 构建、一次性 clone本地开发、大文件仓库
内部标记.git/shallow 文件promisor = true 配置

一个直觉上的判断：

  • 如果你的仓库历史太长但单次提交不大（比如高频提交的小改动），用 --depth 减少 clone 大小。
  • 如果你的仓库单次提交很大但你对历史不感兴趣（比如有巨大二进制文件的仓库），用 --filter=blob:none，这是性价比最高的做法。
  • 如果两者都有，可以组合使用：
    git clone --depth 100 --filter=blob:none <url>

实际场景选择 #

CI 流水线 ##

CI 一般只需要最新的代码，不需要历史，而且经常在一台新机器上从头拉——--depth 1 是最佳选择，简单、高效、兼容性好。

如果你的 CI 还做 git blame 或跑基于历史数据的覆盖率分析，那 --depth 1 不够，考虑 --depth 100 或换用 --filter=blob:none。

本地开发 ##

本地开发需要完整历史来做 git log、git blame、git bisect 等操作。--filter=blob:none 是最推荐的——历史完整，大文件按需拉取，clone 快，日常操作几乎无感。

像 GitHub 这样的大型仓库，blobless clone 的初始传输量只有完整 clone 的几百分之一，而 git log 等日常操作完全不受影响。

巨型单体仓库（monorepo） ##

对于 Android、Chromium 这种量级的 monorepo，单独 shallow clone 或 partial clone 可能都不够。通常的组合是：

git clone --depth 1 --filter=blob:none --single-branch --branch=main <url>
  • --depth 1：只要最近一条历史。
  • --filter=blob:none：不拉 blob。
  • --single-branch：只拉一个分支。

这能把初始 clone 时间从几小时压缩到几分钟。

总结 #

Shallow clone、partial clone 和分支范围限制，解决的是 Git 仓库膨胀的三个不同维度：

  • 分支范围（fetch refspec / --single-branch） 截断的是广度——拉哪些分支。限制 refspec 让它只同步你关心的分支。
  • Shallow clone（--depth） 截断的是时间——历史深度。底层用 .git/shallow 文件标记截断点，让 Git 知道「再往上就不要追溯了」。
  • Partial clone（--filter） 截断的是空间——对象类型。底层用 promisor remote 机制实现按需懒加载，没拉到的对象在用到时自动补。

三者正交，可以任意组合。选择的关键是搞清楚你的瓶颈在哪里：分支太多？历史太长？还是文件太大？不同的答案对应不同的策略。
