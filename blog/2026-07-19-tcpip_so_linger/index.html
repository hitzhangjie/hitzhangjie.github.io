tcp状态机：SO_LINGER 精确控制 close 发送 FIN/RST

Posted 2026-07-19 01:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

在 [TCP 连接终止场景分析] 里，我们梳理过几类「关连接到底发 FIN 还是 RST」的情况：

  • 进程被 kill -9 后，内核代为关掉 socket：和主动/被动打开无关，默认往往走 FIN；
  • 关闭瞬间接收缓冲区还有未读数据时，内核倾向于发 RST，因为这些数据注定被丢弃；
  • 应用层 Write() 成功，也不代表对端一定还活着。

文中还顺带提到另一种会触发 RST 的路径：SO_LINGER 设为「开启且等待时间为 0」。
这和「缓冲里有脏数据」不同——它是应用主动选择的关闭策略。本文单独把 SO_LINGER 说清楚：它管的是什么、三种配置分别怎样、什么时候值得用、又会踩什么坑。

正文 #

1. close() 默认在干什么？ ##

对 TCP 而言，应用调用 close(fd)（引用计数归零）后，内核大致会：

  1. 若接收缓冲里还有未读数据 → 常直接 RST（与 linger 无关，见前述场景二）；
  2. 否则进入 tcp_close 一类路径，尽量把发送缓冲里的数据发完，再发 FIN，进入正常挥手；
  3. 在默认 linger 语义下，close() 不必等 对端 ACK 完整个挥手过程——未发完的数据可交给内核在后台继续处理（socket 变成孤儿套接字等，细节见 [close 后 socket 是否立刻销毁]）。

因此默认行为更接近：尽快把 fd 从进程里摘掉，优雅关闭尽量交给协议栈。
SO_LINGER 则允许你改写第 2、3 步的语义——包括「要不要阻塞等发送结束」，以及「干脆别 FIN，直接 RST」。

2. SO_LINGER 是什么？ ##

SO_LINGER 是 SOL_SOCKET 层选项，用一个 linger 结构描述关闭策略：

struct linger {
    int l_onoff;   /* 是否启用 linger 语义：0=默认，非 0=启用 */
    int l_linger;  /* 等待时间，单位秒 */
};

struct linger lg = { .l_onoff = 1, .l_linger = 0 };
setsockopt(fd, SOL_SOCKET, SO_LINGER, &lg, sizeof(lg));
close(fd);

Go 里可用 syscall / golang.org/x/sys/unix 设置，或在 net.TCPConn 上通过底层 fd 操作（标准库没有一等公民的 SetLinger 包装在所有版本里都同样显眼，但语义与 C 一致）。

可以把它理解成一句话：

close() 时要不要“磨蹭/等待”，以及要不要用 RST 秒断。

3. 三种典型配置 ##

配置close() 行为对端通常看到
默认（l_onoff = 0）几乎立刻返回；发送缓冲由内核后台尽量发完，然后 FIN正常 EOF / 四次挥手
l_onoff = 1, l_linger > 0阻塞，最多等 l_linger 秒，尽量把数据发完并完成关闭；超时后行为与实现/剩余数据有关，不保证对端一定收齐多半仍是 FIN；超时边界需实测
l_onoff = 1, l_linger = 0立刻返回；丢弃未发送/未读数据，发 RST 中止，不走完整挥手connection reset by peer，不是干净 EOF

注意优先级感：若关闭时接收缓冲已有未读数据，即便你没设 linger=0，Linux 上也常常已经走 RST。
linger=0 是在「缓冲未必有未读数据」时，仍然强制 RST 的开关。

4. 和「连接终止场景」怎么对上号？ ##

把前文几个场景和 linger 放在一张表里，边界会更清晰：

场景更可能发什么原因
kill -9，接收缓冲空，默认 lingerFIN内核退出路径上 close socket，等价于普通关闭
kill -9 或 close()，接收缓冲有未读数据RST数据将被丢弃，异常关闭
应用 setsockopt(SO_LINGER, {1,0}) 再 close()RST主动选择 abort，与角色（client/server）无关
对端连接已不存在后再写数据本端后续感知 RST / EPIPE 等对端 TCB 已无，属另一条路径

所以：文章里写「SO_LINGER(on, 0) → RST」，不是第三种神秘魔法，而是 POSIX/实现约定的 abortive close（中止式关闭）。
它和「server 被杀就 RST、client 被杀就 FIN」这类按角色划分的说法无关——和 kill -9 一样，看的是关闭瞬间的策略与 socket 状态，不是主动打开还是被动打开。

5. linger=0 常见用途与代价 ##

有人会用它的原因：

  1. 尽快释放本端资源，避免卡在 close() 或长时间 FIN_WAIT / 半关闭；
  2. 减少本端 TIME_WAIT（谁主动发 FIN 完成关闭，谁更容易进 TIME_WAIT；RST 掐断往往不按完整主动关闭路径走）；
  3. 明确告诉对端：这是异常结束，不要当成业务层的正常 EOF 去解析半包。

代价同样明显：

  • 发送缓冲里尚未送达/未确认的数据可能直接丢；
  • 对端 Read 得到的是 reset，而不是 EOF，错误处理路径和「对端正常 close」不同；
  • 用 RST 规避 TIME_WAIT 往往是治标：连接异常增多、排障更难，高并发服务更稳妥的做法通常是调 TIME_WAIT 相关参数、连接复用、或架构上减少短连接风暴，而不是全局 linger=0。

经验上：库、框架、中间件默认很少开 linger=0；只有明确知道「这条连接废了，丢数据可接受，且希望对端立刻失败」时才值得设。

6. linger>0：阻塞 close 的适用面 ##

l_onoff=1 且 l_linger>0 适合这类需求：

  • 发送完最后一截响应后，希望 close() 返回时数据已尽量交给协议栈并发出；
  • 进程马上要退出，担心默认「异步收尾」来不及把尾包刷出去。

但要注意：

  • 阻塞的是调用 close() 的线程，高 QPS 服务里随意加大 linger 可能拖垮工作线程；
  • 「等了 N 秒」≠「对端应用一定 read 到并处理完」——只是本端发送/关闭进度的上限等待；
  • 与 shutdown(SHUT_WR) + 读到对端 FIN/EOF 的「优雅双端确认」相比，linger 仍是更粗的旋钮。

若目标是半关闭、连接池回收，往往应优先考虑 shutdown 与协议层结束标记，而不是只拧 linger。相关讨论见同目录半关闭/写关闭相关文章。

7. 最小实验思路 ##

不必上完整业务，本地即可验证 linger=0：

  1. 起一对 TCP 连接，接收缓冲保持为空；
  2. 一端默认 close() → 对端 Read 多半是 EOF；
  3. 另一端先 SO_LINGER{1,0} 再 close() → 对端 Read 多半是 connection reset by peer。

这与 TCP 连接终止场景分析 同目录下的 kill9_fin_rst_demo.go 判定方式相同：存活侧用 Read 错误类型区分 FIN 与 RST。
差别只在于：demo 用 SIGKILL 模拟进程崩溃；这里是应用显式选择 abortive close。

总结 #

  • SO_LINGER 控制的是 close() 的收尾策略：是否阻塞等待、是否在 l_linger=0 时用 RST 中止连接。
  • 默认关闭 + 接收缓冲为空 → 通常 FIN；缓冲有未读数据 → 常 RST；linger=0 → 强制 RST。
  • 它和 kill -9、主动/被动打开无绑定关系；和「连接终止」其它场景是同一张决策表上的不同输入。
  • linger=0 能快速掐断、有时避开 TIME_WAIT，但以丢数据和更差的对端错误语义为代价，应谨慎、按连接使用，而不是当全局默认。
