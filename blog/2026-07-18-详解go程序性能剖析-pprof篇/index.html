详解Go程序性能剖析：pprof篇（译）

Posted 2026-07-18 10:30 +0800 by ZhangJie ‐ 8 min read

分享:  

原文：Profiling Go Programs — Russ Cox, 2011 年 6 月；马盛浩更新于 2013 年 5 月

2011 年的 Scala Days 上，Robert Hundt 发表了一篇题为 Loop Recognition in C++/Java/Go/Scala 的论文。文中用 C++、Go、Java、Scala 四种语言实现了同一个循环查找算法——编译器流分析中常见的那种——然后用这些程序的运行数据来讨论各语言的典型性能特征。其中 Go 的那个版本运行相当慢，这恰好给了我们一个绝佳的机会，来展示如何借助 Go 的剖析工具，把一个慢程序一步步变快。

借助 Go 的剖析工具识别并修正具体的性能瓶颈，我们可以让这个循环查找程序快上一个数量级，同时将内存消耗降至原来的 1/6。（更新：由于 gcc 的 libstdc++ 近期做了优化，内存降幅调整为 1/3.7。）

Hundt 的论文没有注明各语言编译器及工具链的具体版本。本文中，我们使用 6g Go 编译器的最新 weekly 快照版本，以及 Ubuntu Natty 发行版附带的 g++ 版本。（我们不会涉及 Java 和 Scala，因为我们对这两种语言的高效编程并不精通，放在一起比较有失公允。既然论文中 C++ 是跑得最快的，这里拿 C++ 来对比就够了。）更新：本文更新版使用 Go 编译器在 amd64 上的最新开发快照，以及 2013 年 3 月发布的 g++ 4.8.0。

$ go version
go version devel +08d20469cc20 Tue Mar 26 08:27:18 2013 +0100 linux/amd64
$ g++ --version
g++ (GCC) 4.8.0
Copyright (C) 2013 Free Software Foundation, Inc.

测试机配置：3.4GHz Core i7-2600 CPU，16 GB RAM，Gentoo Linux 3.8.4-gentoo 内核，已禁用 CPU 频率调节。

我们取了 Hundt 的基准测试程序的 C++ 和 Go 版本，将每个版本合并为单个源文件，并只保留了一行输出。使用 Linux 的 time 工具计时，输出格式包括用户时间、系统时间、实际时间和最大内存占用。

$ make havlak1cc
g++ -O3 -o havlak1cc havlak1.cc
$ ./xtime ./havlak1cc
# of loops: 76002 (total 3800100)
loop-0, nest: 0, depth: 0
17.70u 0.05s 17.80r 715472kB ./havlak1cc

$ make havlak1
go build havlak1.go
$ ./xtime ./havlak1
# of loops: 76000 (including 1 artificial root node)
25.05u 0.11s 25.20r 1334032kB ./havlak1

C++ 程序跑了 17.80 秒，占用了 700 MB 内存。Go 程序跑了 25.20 秒，占用了 1302 MB 内存。（这些数据与论文原文难以直接对齐，但本文的主旨是演示 go tool pprof 的用法，而不是复现论文数据。）

启用剖析 ##

要开始调优 Go 程序，首先得把剖析打开。如果你的代码使用了 Go testing 包的基准测试支持，那直接用 go test 的 -cpuprofile 和 -memprofile 参数就行了。但在像这样一个独立运行的程序中，就需要手动引入 runtime/pprof 并加上几行代码：

var cpuprofile = flag.String("cpuprofile", "", "write cpu profile to file")

func main() {
    flag.Parse()
    if *cpuprofile != "" {
        f, err := os.Create(*cpuprofile)
        if err != nil {
            log.Fatal(err)
        }
        pprof.StartCPUProfile(f)
        defer pprof.StopCPUProfile()
    }
    // ...
}

新增的代码定义了一个 cpuprofile 命令行参数，调用 Go flag 库解析命令行，如果用户指定了 -cpuprofile，就启动 CPU 剖析并将数据重定向到指定文件。剖析器要求在程序退出前调用一次 StopCPUProfile，以保证所有待写数据都被刷入文件；这里通过 defer 确保它在 main 返回时执行。

加上以上代码后，我们就可以带 -cpuprofile 参数运行程序，然后用 go tool pprof 来解读剖析结果了。

$ make havlak1.prof
./havlak1 -cpuprofile=havlak1.prof
# of loops: 76000 (including 1 artificial root node)
$ go tool pprof havlak1 havlak1.prof
Welcome to pprof!  For help, type 'help'.
(pprof)

go tool pprof 本质上是 Google 的 pprof C++ 剖析器的一个变体。最重要的命令是 topN，它展示剖析数据中占比最高的前 N 个采样点：

(pprof) top10
Total: 2525 samples
     298  11.8%  11.8%      345  13.7% runtime.mapaccess1_fast64
     268  10.6%  22.4%     2124  84.1% main.FindLoops
     251   9.9%  32.4%      451  17.9% scanblock
     178   7.0%  39.4%      351  13.9% hash_insert
     131   5.2%  44.6%      158   6.3% sweepspan
     119   4.7%  49.3%      350  13.9% main.DFS
      96   3.8%  53.1%       98   3.9% flushptrbuf
      95   3.8%  56.9%       95   3.8% runtime.aeshash64
      95   3.8%  60.6%      101   4.0% runtime.settype_flush
      88   3.5%  64.1%      988  39.1% runtime.mallocgc

CPU 剖析开启后，Go 程序每秒大约暂停 100 次，每次都记录当前 goroutine 栈上的程序计数器，形成一个采样点。这份剖析数据共 2525 个采样点，所以程序大约运行了 25 秒出头。在 go tool pprof 的输出中，每个出现在采样中的函数各占一行。前两列分别表示该函数正在执行（而非等待被调用函数返回）的采样点数量，以及该数量占总采样点的百分比。runtime.mapaccess1_fast64 在执行期间被采到了 298 次，占 11.8%。top10 的输出按这个计数降序排列。第三列是列表中的累计占比：前三行加起来已经覆盖了 32.4% 的采样点。第四、第五列则表示该函数出现在调用栈中（无论是正在执行还是在等待被调用函数返回）的采样点数量和占比。main.FindLoops 有 10.6% 的采样点命中了它自身的执行，但它在 84.1% 的采样点中都以某种方式出现在了调用栈上——要么是它自己在跑，要么是它调用的函数在跑。

如果想按第四、第五列排序，加上 -cum（cumulative，累计）参数即可：

(pprof) top5 -cum
Total: 2525 samples
       0   0.0%   0.0%     2144  84.9% gosched0
       0   0.0%   0.0%     2144  84.9% main.main
       0   0.0%   0.0%     2144  84.9% runtime.main
       0   0.0%   0.0%     2124  84.1% main.FindHavlakLoops
     268  10.6%  10.6%     2124  84.1% main.FindLoops

实际上 main.FindLoops 和 main.main 的累计占比加起来应该是 100%，但每个栈采样只会记录栈底的 100 帧。大约有四分之一的采样中，递归函数 main.DFS 比 main.main 深了超过 100 帧，导致完整栈迹被截断了。

栈采样数据包含了比文本列表更丰富的函数调用关系信息。web 命令会将剖析数据以 SVG 格式绘制成图并在浏览器中打开。

(pprof) web

图中的每个方框对应一个函数，方框大小正比于该函数正在执行的采样点数量。从 X 到 Y 的边表示 X 调用了 Y，边上的数字表示该调用在采样中出现的次数。如果一次采样中同一条调用路径出现多次——比如递归调用——每次出现都会被计入边的权重。这就是为什么 main.DFS 到自身的回边上会标着 21342。

仅凭一张宏观图我们就能看出，程序把大把时间花在了哈希操作上，而这正是 Go 的 map 带来的。我们还可以让 web 只显示涉及某个特定函数的采样，比如 runtime.mapaccess1_fast64，从而滤除掉图中的大量噪音。

如果再细看，你会发现对 runtime.mapaccess1_fast64 的调用大多来自 main.FindLoops 和 main.DFS。

现在我们有了大局观，是时候聚焦到某个具体函数了。先看 main.DFS，因为它体量相对较小：

(pprof) list DFS
Total: 2525 samples
ROUTINE ====================== main.DFS in havlak1.go
   119    697 Total samples (flat / cumulative)
     3      3  240: func DFS(currentNode *BasicBlock, nodes []*UnionFindNode, number map[*BasicBlock]int, last []int, current int) int {
     1      1  241:     nodes[current].Init(currentNode, current)
     1     37  242:     number[currentNode] = current
     .      .  243:
     1      1  244:     lastid := current
    89     89  245:     for _, target := range currentNode.OutEdges {
     9    152  246:             if number[target] == unvisited {
     7    354  247:                     lastid = DFS(target, nodes, number, last, lastid+1)
     .      .  248:             }
     .      .  249:     }
     7     59  250:     last[number[currentNode]] = lastid
     1      1  251:     return lastid

这个列表展示了 DFS 函数（以及所有匹配 DFS 正则表达式的函数）的源码。前三列分别是在该行上采到的样本数、在该行及其调用的代码中采到的样本数、以及行号。相关命令 disasm 输出函数的反汇编而非源码列表；当采样足够密集时，这能帮你看出究竟哪些指令在吃 CPU。weblist 则将两种模式融合：展示源码列表，点击某一行就能展开对应的反汇编。

既然我们已经知道时间都耗在哈希运行时实现的 map 查找上了，那最值得关注的自然就是第二列。大量的时间耗在了 DFS 的递归调用上（第 247 行），这对于一个递归遍历来说理所当然。刨去递归本身，耗时主要集中在对 number 这个 map 的几次访问上——第 242、246、250 行。对于这种场景来说，map 并非最高效的选择。就像在编译器中一样，基本块结构都被分配了唯一的序号。与其用 map[*BasicBlock]int，不如用一个 []int 切片，用块序号做索引。能用数组或切片解决的事情，就不要动用 map。

将 number 从 map 换成切片，只需改动程序的七行代码，运行时间近乎减半：

$ make havlak2
go build havlak2.go
$ ./xtime ./havlak2
# of loops: 76000 (including 1 artificial root node)
16.55u 0.11s 16.69r 1321008kB ./havlak2

我们再跑一次剖析来确认 main.DFS 已经不再是瓶颈：

$ make havlak2.prof
$ go tool pprof havlak2 havlak2.prof
(pprof) top5
Total: 1652 samples
     197  11.9%  11.9%      382  23.1% scanblock
     189  11.4%  23.4%     1549  93.8% main.FindLoops
     130   7.9%  31.2%      152   9.2% sweepspan
     104   6.3%  37.5%      896  54.2% runtime.mallocgc
      98   5.9%  43.5%      100   6.1% flushptrbuf

main.DFS 已经从 profile 中消失了，程序的整体运行时间也降了下来。现在程序大部分时间都耗在了内存分配和垃圾回收上——runtime.mallocgc（这个函数既负责分配又负责周期性 GC）吃掉了 54.2% 的时间。要弄清为什么 GC 如此繁忙，我们需要找出到底是谁在大量分配内存。一个办法是给程序加上内存剖析。

我们在程序中加入：如果用户指定了 -memprofile 参数，程序就在完成一轮循环查找后停下来，把内存剖析数据写入文件然后退出：

var memprofile = flag.String("memprofile", "", "write memory profile to this file")

    FindHavlakLoops(cfgraph, lsgraph)
    if *memprofile != "" {
        f, err := os.Create(*memprofile)
        if err != nil {
            log.Fatal(err)
        }
        pprof.WriteHeapProfile(f)
        f.Close()
        return
    }

然后带 -memprofile 运行程序来生成剖析文件：

$ make havlak3.mprof
go build havlak3.go
./havlak3 -memprofile=havlak3.mprof

我们用完全一样的方式运行 go tool pprof。只不过这次分析的不再是时钟采样，而是内存分配。

$ go tool pprof havlak3 havlak3.mprof
Adjusting heap profiles for 1-in-524288 sampling rate
Welcome to pprof!  For help, type 'help'.
(pprof) top5
Total: 82.4 MB
    56.3  68.4%  68.4%     56.3  68.4% main.FindLoops
    17.6  21.3%  89.7%     17.6  21.3% main.(*CFG).CreateNode
     8.0   9.7%  99.4%     25.6  31.0% main.NewBasicBlockEdge
     0.5   0.6% 100.0%      0.5   0.6% itab
     0.0   0.0% 100.0%      0.5   0.6% fmt.init

go tool pprof 告诉我们，FindLoops 分配了 82.4 MB 内存中的大约 56.3 MB，CreateNode 又占了 17.6 MB。为了降低开销，内存剖析器大约每分配 0.5 MB 才记录一次（即"1/524288 采样率"），所以这些数字是真实值的近似。

显然，当前瓶颈和前一次如出一辙：在简单数据结构就够用的场景下使用了 map。光是 FindLoops 就为 map 分配了大约 29.5 MB。

顺便提一句，如果用 --inuse_objects 参数运行 go tool pprof，它会报告分配对象的个数而非大小：

$ go tool pprof --inuse_objects havlak3 havlak3.mprof
(pprof) list FindLoops
Total: 1763108 objects
...
311296 311296  278:             nodes[i] = new(UnionFindNode)
409600 409600  288:             nonBackPreds[i] = make(map[int]bool)
...

大约 20 万个 map 占了 29.5 MB，算下来每个 map 的初始分配大约是 150 字节。如果 map 确实用来存键值对，这个开销说得过去；但在这里，map 只是被当作一个简单的集合来用——这就很不划算了。

与其用 map，不如用一个简单的切片来存元素列表。在几乎所有用到 map 的地方，算法都不可能插入重复元素。唯一一个可能出现重复的场景，我们写了一个 appendUnique 的小函数来应对：

func appendUnique(a []int, x int) []int {
    for _, y := range a {
        if x == y {
            return a
        }
    }
    return append(a, x)
}

把这些 map 换成切片，只需改动寥寥几行代码：

$ make havlak4
go build havlak4.go
$ ./xtime ./havlak4
# of loops: 76000 (including 1 artificial root node)
11.84u 0.08s 11.94r 810416kB ./havlak4

我们已经比初始版本快了 2.11 倍。再看一下 CPU profile：

$ make havlak4.prof
$ go tool pprof havlak4 havlak4.prof
(pprof) top10
Total: 1173 samples
     205  17.5%  17.5%     1083  92.3% main.FindLoops
     138  11.8%  29.2%      215  18.3% scanblock
      88   7.5%  36.7%       96   8.2% sweepspan
      76   6.5%  43.2%      597  50.9% runtime.mallocgc
      75   6.4%  49.6%       78   6.6% runtime.settype_flush
      74   6.3%  55.9%       75   6.4% flushptrbuf
      64   5.5%  61.4%       64   5.5% runtime.memmove

      51   4.3%  71.1%       51   4.3% main.DFS
      50   4.3%  75.4%      146  12.4% runtime.MCache_Alloc

内存分配以及伴随而来的垃圾回收（runtime.mallocgc）现在占到了 50.9%。我们可以用 web mallocgc 来看看究竟是谁在驱动这些分配。图中节点太多，小样本的节点遮蔽了大头，这时可以用 --nodefraction=0.1 过滤掉占比不到 10% 的节点：

$ go tool pprof --nodefraction=0.1 havlak4 havlak4.prof
(pprof) web mallocgc

现在粗箭头一目了然：FindLoops 是绝大部分 GC 压力的源头。列表查看一下这个函数，发现大量的分配集中在函数开头：

(pprof) list FindLoops
...
     .    145  277:     nonBackPreds := make([][]int, size)
     .      9  278:     backPreds := make([][]int, size)
     .      1  280:     number := make([]int, size)
     .     17  281:     header := make([]int, size, size)
     .      .  284:     nodes := make([]*UnionFindNode, size, size)
     2     79  287:             nodes[i] = new(UnionFindNode)
...

每次调用 FindLoops 都会分配一大坨记账用的数据结构。因为基准测试调用了 FindLoops 多达 50 次，这些分配累加起来产生了大量的垃圾，进而给 GC 带来了沉重的负担。

有垃圾回收的语言不意味着你可以无视内存分配。在这个场景中，最直接的方案是引入一个缓存，让每次 FindLoops 调用尽可能复用上一轮已经分配好的存储空间。（事实上，Hundt 在他的论文中提到 Java 版本正是做了这个改动才获得了可接受的性能，但他并未对其他有 GC 的语言实现做同样的优化。）

我们添加一个全局的 cache 结构体来持有这些可复用的存储空间，然后让 FindLoops 在分配之前先去 cache 里查找。当然，全局变量从软件工程角度看是坏实践——它意味着对 FindLoops 的并发调用不再安全。但在这里，我们只做最小的改动来验证什么对性能至关重要；最终版本会用一个独立的 LoopFinder 实例来管理这些内存，恢复并发安全。

$ make havlak5
go build havlak5.go
$ ./xtime ./havlak5
# of loops: 76000 (including 1 artificial root node)
8.03u 0.06s 8.11r 770352kB ./havlak5

还有一些可以进一步清理和优化的地方，但所用的剖析技巧都已经在前面演示过了。内层循环的工作列表可以在迭代间乃至 FindLoops 的多次调用间复用，还可以与当轮生成的"节点池"合并。类似地，循环图的存储也可以在每轮迭代中复用而非重新分配。最终版本采用了地道的 Go 风格编写，使用了数据结构和方法的组合。风格改动对运行时间的影响微乎其微：算法和约束保持不变。

最终版本 2.29 秒跑完，内存占用 351 MB：

$ make havlak6
go build havlak6.go
$ ./xtime ./havlak6
# of loops: 76000 (including 1 artificial root node)
2.26u 0.02s 2.29r 360224kB ./havlak6

这比我们最初出发时的版本快了 11 倍。即便禁用循环图缓存（只复用记账用的那部分内存），程序也比原始版本快 6.7 倍，且内存反而省了 1.5 倍。

$ ./xtime ./havlak6 -reuseloopgraph=false
# of loops: 76000 (including 1 artificial root node)
3.69u 0.06s 3.76r 797120kB ./havlak6

当然，此时再把 Go 程序和原始的 C++ 程序放在一起比较已经不太公平了——原版 C++ 也用了一堆 set，而更合适的选择显然是 vector。为了做个理智检验，我们把最终版的 Go 程序翻译成了等效的 C++ 代码，其运行时间与 Go 版非常接近：

$ make havlak6cc
g++ -O3 -o havlak6cc havlak6.cc
$ ./xtime ./havlak6cc
# of loops: 76000 (including 1 artificial root node)
1.99u 0.19s 2.19r 387936kB ./havlak6cc

Go 程序跑得几乎和 C++ 一样快。由于 C++ 用的是自动析构和分配而非显式缓存，C++ 代码稍短、写起来也稍微省心一些，但差别远没有到夸张的程度：

$ wc havlak6.cc; wc havlak6.go
 401 1220 9040 havlak6.cc
 461 1441 9467 havlak6.go

基准测试的价值和其所度量的程序质量同等重要。我们通过 go tool pprof 审视了一个低效的 Go 程序，然后将其性能提升了一个数量级，内存消耗降到原来的 1/3.7。随后与同等优化过的 C++ 程序的对比表明，只要程序员用心控制内层循环产生的垃圾量，Go 完全可以和 C++ 在性能上一较高下。

本文中用到的程序源码、Linux x86-64 二进制及剖析文件，均可在 GitHub 上的 benchgraffiti 项目找到。

如前所述，go test 已经内置了这些剖析参数：只要定义好基准测试函数，一切就绪。此外，还有一个标准的 HTTP 接口对外暴露剖析数据。在一个 HTTP 服务器中，只需添加：

import _ "net/http/pprof"

就会自动在 /debug/pprof/ 下注册若干个 handler。然后你可以直接用 go tool pprof 加上服务器 URL 来下载并分析一份实时剖析数据：

go tool pprof http://localhost:6060/debug/pprof/profile   # 30 秒 CPU profile
go tool pprof http://localhost:6060/debug/pprof/heap      # 堆内存 profile
go tool pprof http://localhost:6060/debug/pprof/block     # goroutine 阻塞 profile

goroutine 阻塞 profile 将留待后续文章讲解。敬请期待。

补充：更轻量的诊断工具 gops ##

如果你觉得手动编写 runtime/pprof 的采样代码有些繁琐，或者希望在不修改代码的情况下快速查看系统中运行的 Go 程序的状态，Google 开源的 gops 是一个很好的选择。

gops 是一个简化版的 Go 程序诊断工具，它由一个命令行工具 gops 和一个可嵌入 Go 程序的轻量库组成。程序只需引入 gops 库，就能在本地启动一个 agent，然后你可以通过 gops 命令行工具与之交互：

import "github.com/google/gops/agent"

func main() {
    // 启动 gops agent，监听本地环回地址
    if err := agent.Listen(agent.Options{}); err != nil {
        log.Fatal(err)
    }
    // ... 你的业务逻辑
}

然后通过命令行：

# 列出系统中所有集成了 gops 的 Go 进程
$ gops
98324   go1.18  /usr/local/bin/myapp
98325   go1.21  /usr/local/bin/another

# 查看进程的调用栈（类似 pprof goroutine）
$ gops stack 98324

# 查看内存统计
$ gops memstats 98324

# 直接生成 CPU profile（无需提前在代码中添加采样逻辑）
$ gops pprof-cpu 98324

# 查看当前运行的 Go 版本
$ gops stats 98324

gops 的核心优势在于零配置诊断——只要程序集成了 gops agent，无需在代码中预先添加 runtime/pprof 的采样开关，也不需要暴露 HTTP pprof 端点，就可以随时通过命令行对运行中的程序进行诊断。这对于长期运行的后台服务、不方便重启的生产环境排查尤其有用。

gops 本质上是对 runtime/pprof 和 net/http/pprof 的轻量封装，它的 agent 监听在本地 TCP 端口上（默认未公开），通过简单的文本协议通信，比 HTTP pprof 端点更轻量，也不需要引入 net/http 依赖。
