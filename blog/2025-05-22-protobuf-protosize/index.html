理解 Protobuf 的 proto.Size 与非最小化编码

Posted 2025-05-22 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

在使用 Go 的 Protobuf 库时，proto.Size(m) 是一个常用的辅助函数，用于预先计算某个 message 序列化后占用的字节数。但官方文档特别提醒：proto.Size 的返回值可能比实际序列化后的数据更大。

这听起来有些反直觉——为什么要返回一个偏大的值？这涉及到 Protobuf wire format 中"非最小化编码"（non-minimal wire format）的概念。

proto.Size 返回更大值的原因 #

1. Varint 的非最小化编码 ##

Protobuf 使用 varint 编码整数。同一个数值可以用多种不同长度的字节序列表示：

数字 1 的 varint 编码：
- 最小化编码：0x01          （1 字节）
- 非最小化：  0x81 0x00     （2 字节）
- 非最小化：  0x81 0x80 0x00 （3 字节）

proto.Size() 在计算时可能采用非最小化编码的尺寸，而实际 marshal 过程会使用最小化编码，导致 Size 返回值偏大。

2. Repeated 字段的 packed vs unpacked ##

Repeated 字段有两种序列化方式：

message Example {
    repeated int32 values = 1;
}

对于 values = [1, 2, 3]：

  • Unpacked（每个值独立编码）：

    tag(1) + value(1) + tag(1) + value(2) + tag(1) + value(3) = 6 字节

  • Packed（所有值打包到一组）：

    tag(1) + length + value(1) + value(2) + value(3) = 5 字节

proto.Size() 可能按 unpacked 方式估算，而实际 marshal 使用 packed 方式，导致结果偏大。

注意：proto3 中 repeated 标量类型默认使用 packed 编码。

3. 字段顺序的灵活性 ##

Protobuf 允许字段以任意顺序编码。proto.Size() 可能假定一种较宽松的顺序，实际 marshal 过程可能选择更紧凑的排列。

为什么 Protobuf 允许非最小化编码 #

这是设计上的有意为之。Protobuf 规范允许解析器接受非最小化编码的数据，主要原因：

  1. 兼容性：不同语言、不同版本的实现可能产生不同形式的编码
  2. 性能权衡：强制要求最小化编码会增加序列化 / 反序列化的计算成本

但这种灵活性也带来了安全风险——攻击者可以构造包含非最小化编码的恶意消息，导致解析器误判数据大小。

安全影响 #

恶意构造的 Protobuf 消息可以通过非最小化编码大幅膨胀消息体的解析开销：

  • 一个实际只有几字节的小消息，通过非最小化编码可以声明自己是数 GB 的数据
  • 如果服务端盲目按 proto.Size() 的值分配缓冲区，可能因内存耗尽而 OOM

官方文档中明确提到了这种攻击场景：

Background: Non-minimal wire format — a malicious actor could craft Protobuf messages specifically to trigger crashes over the network.

最佳实践 #

  1. 不要把 proto.Size() 当作精确值：它只是一个估算值，用于预分配缓冲区的参考
  2. 为缓冲区追加 padding：如果使用 proto.Size() 分配空间，额外多分配一些余量
  3. 对输入消息设置大小限制：在反序列化前先检查消息字节流的大小，拒绝异常大的请求
  4. 使用 proto.Buffer 进行流式编码：避免一次性分配大块内存
// 不推荐：精确分配
buf := make([]byte, proto.Size(m))

// 推荐：追加余量
buf := make([]byte, proto.Size(m)+64)

// 更推荐：由 marshal 内部管理缓冲区
data, err := proto.Marshal(m)

总结 #

proto.Size() 返回值偏大并非 bug，而是 Protobuf 非最小化编码机制的副产品。理解这个特性很重要——既不要在业务代码中过度依赖其精确性，也要意识到它可以作为恶意消息膨胀的载体。对任何从不可信来源接收 Protobuf 消息的服务，都应该增加大小校验来防止相关攻击。
