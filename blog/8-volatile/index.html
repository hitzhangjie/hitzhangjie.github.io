8 Volatile

Posted 2018-12-27 16:26 +0800 by ZhangJie ‐ 3 min read

分享:  

Java 中的 volatile 关键字用于标记一个变量"存储在主内存中"。更准确地说，对 volatile 变量的每次读都将从主内存中读取，而不是从 CPU 缓存中读取；每次对 volatile 变量的写都将立即写回主内存。

从 Java 5 开始，volatile 关键字除了保证变量的可见性之外，还提供了 happens-before 语义，能够限制指令重排序，这使得它的作用不止于"从主存读写"那么简单。

变量可见性问题 #

在多线程应用中，线程出于性能考虑，可能会将非 volatile 变量从主内存拷贝到 CPU 寄存器中进行操作。在多 CPU 系统中，每个线程可能在不同的 CPU 上运行，各自持有变量在 CPU 缓存中的不同拷贝。这就产生了可见性问题。

下图展示了这一情形：多个线程持有来自主内存的变量拷贝（位于 CPU 缓存中），但对变量的修改何时写回主内存是不确定的。

考虑一个包含非 volatile 计数器的共享对象：

public class SharedObject {
    public int counter = 0;
}

假设 Thread 1 和 Thread 2 都读取了 counter 的值，然后 Thread 1 对 counter 执行了 +1 操作，但修改后的值仍停留在 Thread 1 所在 CPU 的缓存中。此时 Thread 2 读取到的 counter 值仍然是旧值 0。

下图对比了 CPU 缓存中的值与主内存中 counter 的差异，这种一个线程的更新对其他线程不可见的问题，就是可见性问题（Visibility Problem）。

volatile 的可见性保证 #

将变量声明为 volatile 后，对该变量的所有写操作都会立即写回主内存，所有读操作都会从主内存中读取。

public class SharedObject {
    public volatile int counter = 0;
}

完整的 volatile 可见性保证 ##

实际上，volatile 的可见性保证不仅限于 volatile 变量本身，它具有更广泛的含义：

  1. 如果 Thread A 写入了一个 volatile 变量，随后 Thread B 读取了同一个 volatile 变量，那么 Thread A 在写入 volatile 变量之前可见的所有变量，在 Thread B 读取 volatile 变量之后也将对 Thread B 可见。
  2. 如果 Thread A 读取了一个 volatile 变量，那么 Thread A 在读取该 volatile 变量时可见的所有变量，也会从主内存中被重新读取。

下面通过一个示例来说明：

public class MyClass {
    private int years;
    private int months;
    private volatile int days;

    public void update(int years, int months, int days) {
        this.years  = years;
        this.months = months;
        this.days   = days;
    }
}

在 update() 方法中，years 和 months 先被赋值，days（volatile 变量）最后被赋值。当 days 被写入时，years 和 months 的值也会一并被刷新到主内存中，从而对其他线程可见。

同样地，在读取时：

public int totalDays() {
    int total = this.days;
    total += months * 30;
    total += years * 365;
    return total;
}

由于 days 是 volatile 变量，首先读取 days 会确保 months 和 years 也从主内存中重新读取，从而保证读到的是最新的值。

补充说明：这里 volatile 变量的读写形成了一个"内存屏障"效果。对 volatile 变量的写，会将之前所有修改都刷新到主内存；对 volatile 变量的读，会使后续读取都从主内存获取。这也是为什么前面的文章中提到，synchronized 通过 monitorenter / monitorexit 配合底层 lock 指令，本质上也是利用了类似的机制来实现可见性。

指令重排序带来的挑战 #

JVM 和 CPU 为了提升性能，可能会对指令进行重排序，只要程序的语义不被改变。例如：

int a = 1;
int b = 2;
a++;
b++;

可能被重排序为：

int a = 1;
a++;
int b = 2;
b++;

但在引入 volatile 后，指令重排序可能导致严重问题。考虑前面的 MyClass 示例，如果 update() 方法中的指令被重排序，使得 days 的赋值被移到了 months 和 years 的赋值之前，那么当 days 的新值被写回主内存时，months 和 years 的新值还尚未写入——其他线程此时读取 days 会得到新值，但读取 months 和 years 得到的却是旧值，程序的语义被破坏了。

volatile 的 Happens-Before 保证 #

为了解决上述指令重排序引发的问题，volatile 提供了 happens-before 语义，具体包括：

  • 对于 volatile 变量的写操作：原本出现在 volatile 写之前的、对其他变量的读写操作，不能被重排序到 volatile 写之后。也就是说，volatile 写之前的所有操作，一定会发生在 volatile 写之前（但 volatile 写之后的指令可以被重排序到 volatile 写之前，前提是这不会受 volatile 写影响）。
  • 对于 volatile 变量的读操作：原本出现在 volatile 读之后的、对其他变量的读写操作，不能被重排序到 volatile 读之前。也就是说，volatile 读之后的所有操作，一定会发生在 volatile 读之后（但 volatile 读之前的指令可以被重排序到 volatile 读之后）。

从 happens-before 的视角来看：

  • volatile 变量的写 happens-before 后续对这个 volatile 变量的读，即前一个线程对 volatile 变量的写，对后续读取该 volatile 变量的线程是可见的。

补充说明：关于这一点，前面 JMM 内存模型文章中已有更详细的讨论。volatile 底层通过 CPU 的内存屏障（Memory Barrier）实现——写 volatile 时 CPU 发出 lock 指令，将 cache 中的数据立即写入内存，并通知其他 CPU 使其 cache 失效、重新读取。intel CPU 虽然是强一致的，即使不加 volatile 也能保证可见性，但为了跨平台兼容性，我们仍然应该正确使用 volatile 和其他同步措施，真正做到"一次编译、到处运行"。

volatile 何时是不够的 #

在某些场景下，仅仅使用 volatile 是不够的。典型的问题是：一个线程需要先读取 volatile 变量的值，基于该值生成新值，然后再写回。

典型的例子是计数器递增操作：

volatile int counter = 0;
// 多个线程执行 counter++;

counter++ 并非原子操作，它实际上包含三个步骤：读取、加 1、写入。如果两个线程几乎同时读取 counter 的值为 0，然后各自加 1 得到 1，再各自写回 1，最终 counter 的值是 1 而不是期望的 2。

下图展示了这种竞态条件：

两个线程都从主内存中读取了 counter 的初始值 0，保存在各自的 CPU 缓存中，各自 +1 后得到 1，再各自写回主内存。最终主内存中的值只是 1，丢失了其中一次递增。

这种情况下就需要使用 synchronized 块或者 java.util.concurrent 包中的原子类型（如 AtomicLong、AtomicReference 等）。

volatile 何时是足够的 #

volatile 在以下场景中是足够的：

  • 只有一个线程写入 volatile 变量，其他线程只读取。这种情况下，volatile 保证了读取线程总是能看到写入线程的最新值，且不存在多线程同时写入引发的竞态条件。
  • volatile 变量作为状态标志，例如用于控制线程是否继续运行、是否已经完成初始化等。

如果涉及多个线程同时对变量进行"读-改-写"操作，则需要使用 synchronized 或者 java.util.concurrent.atomic 包中的原子类来保证原子性。

volatile 关键字在 32 位和 64 位变量上都能正常工作。

性能考量 #

对 volatile 变量的读写直接命中主内存（而不是 CPU 寄存器），这比读写非 volatile 变量更昂贵。同时，访问 volatile 变量还会阻止指令重排序，而指令重排序是一种常规的性能优化技术。因此，应该只在确实需要保证变量可见性的时候，才使用 volatile。

在实际中，CPU 寄存器的值通常只会写入 CPU 的 L1 缓存，访问 L1 缓存的速度也是相当快的，而缓存到主内存的同步则由独立的芯片负责处理。所以 volatile 的性能开销在大多数场景下是可以接受的，但仍应避免滥用。

总结 #

volatile 关键字在 Java 并发编程中承担了两个重要角色：

  1. 保证可见性：对 volatile 变量的写立即刷新到主内存，读从主内存获取最新值，并且连带保证写线程在写 volatile 之前对其他变量的修改也对后续的读线程可见。
  2. 禁止指令重排序：通过 happens-before 语义，确保 volatile 写之前的操作不会被移到写之后，volatile 读之后的操作不会被移到读之前。

但 volatile 不能保证原子性。对于"读-改-写"复合操作，仍需要 synchronized 或原子类来保证线程安全。正确理解 volatile 的能力边界，是写好 Java 并发程序的基础。
