sync.Once：类型结构体中作为值字段还是指针字段？

Posted 2025-12-26 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

TL;DR ##

  • Embed sync.Once by value — 它是一个小结构体，零值即可使用。
  • 不要用 *sync.Once 除非你真的需要在多个结构体间共享同一个 Once（或者你想避免复制整个结构体，但这种情况很少见）。
  • 永远不要复制已经调用过 Do 的结构体 — sync.Once 在使用后就不再安全复制了。

为什么 sync.Once 通常是值字段 ##

type MyService struct {
    once sync.Once
    // other fields …
}
  • sync.Once 只是一个小结构体（目前只有一个指向内部状态的指针），复制开销极低。
  • sync.Once 的零值已经是一个功能完备的"once"对象，不需要额外分配或初始化。
  • 嵌入它（或作为命名字段）提供了地道的 foo.once.Do(...) 用法。
  • Go 标准库和实际代码中的大多数示例都使用值类型。

什么时候考虑用指针 ##

场景为什么用指针可能有帮助代价
在多个结构体间共享同一个 Once所有结构体都调用同一个 once 实例，初始化只运行一次失去 Do 的自动提升，需要管理指针的生命周期
避免复制包含 sync.Once 的结构体如果复制了已调用过 Do 的结构体，副本可能会再次运行初始化或导致竞态。使用指针可以保持单实例仍要小心不要复制结构体本身；Once 的指针不会魔术般地让结构体安全复制
需要延迟创建 Once*sync.Once 允许你在需要前设为 nilsync.Once 零值已经是可用的，很少需要这个

在绝大多数典型场景中，你都会看到 sync.Once 作为值嵌入。

“使用后复制"的陷阱 ##

type Cache struct {
    once sync.Once
    data map[string]int
}

func (c *Cache) Init() {
    c.once.Do(func() {
        c.data = make(map[string]int)
    })
}

func main() {
    var c1 Cache
    c1.Init()

    // BAD: 复制已经使用过 Once 的结构体
    c2 := c1 // 复制了 Once！
    c2.Init() // 可能会再次运行初始化 — 未定义行为
}

sync.Once 包含一个状态标志，在第一次 Do 之后就不再安全复制了。如果你需要一个可以复制的结构体，请保持指向整个结构体的指针，或确保在使用 Once 后永不复制。

结论 ##

  • 将 sync.Once 作为值字段放在结构体里。
  • 避免使用 *sync.Once，除非你有非常特殊的需求。
  • 永远不要复制已经使用过 sync.Once 的结构体。如果需要可复制的类型，改为复制指向结构体的指针。

这就是 Go 中最地道、最安全的模式。
