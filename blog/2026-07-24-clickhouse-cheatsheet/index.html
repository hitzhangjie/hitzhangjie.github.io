ClickHouse 速查手册：场景、架构、存储、部署与成本

Posted 2026-07-24 00:00 +0800 by ZhangJie ‐ 4 min read

分享:  

前言 #

这不是一篇入门教程，也不是某个特性的深入分析。这是一份面向架构师的系统性速查手册，目标是让你在对 ClickHouse 没有系统性学习的情况下，仍能做出合理的技术决策——判断它是否适合当前场景、理解它的核心架构、估算部署和运维成本。

一、适用场景决策树 #

✅ ClickHouse 最适合：

  • 日志分析（应用日志、访问日志、审计日志）
  • 监控指标存储与查询（时序数据，如 Prometheus 远程存储）
  • 实时 OLAP 报表（用户行为分析、漏斗分析、留存分析）
  • 大规模数据聚合（GROUP BY 亿级行数据在秒级返回）
  • 事件追踪与分析（点击流、用户轨迹）

⚠️ 勉强可用但不擅长：

  • 频繁更新已存在数据（如用户状态变更）
  • 单行点查询（按 ID 查一条记录的全部字段）
  • 高并发短查询（QPS > 100）

❌ 完全不适合：

  • OLTP 业务（订单交易、库存扣减）
  • 需要 ACID 事务的场景
  • 需要外键约束、唯一约束的场景
  • 高频单行 DELETE/UPDATE 操作

决策公式：

写入量大 + 查询重分析 + 很少更新 + 并发不高 ≈ ClickHouse

二、架构设计 #

整体架构 ##

┌─────────────────────────────────────────────┐
│                  Client (SQL)               │
└─────────────────┬───────────────────────────┘
                  │
┌─────────────────▼───────────────────────────┐
│            Distributed Table (代理层)        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │ Shard 1 │  │ Shard 2 │  │ Shard N │     │
│  │┌───────┐│  │┌───────┐│  │┌───────┐│     │
│  ││Replica││  ││Replica││  ││Replica││     │
│  ││   A   ││  ││   A   ││  ││   A   ││     │
│  ││Replica││  ││Replica││  ││Replica││     │
│  ││   B   ││  ││   B   ││  ││   B   ││     │
│  │└───────┘│  │└───────┘│  │└───────┘│     │
│  └─────────┘  └─────────┘  └─────────┘     │
└─────────────────────────────────────────────┘
  • Shard（分片）：数据水平切分，不同分片存储不同数据，用于扩展写入吞吐和存储容量。
  • Replica（副本）：同一分片的数据冗余，用于高可用和读扩展。
  • Distributed Table：逻辑代理层，对用户透明，内部将查询路由到各分片并汇总结果。

核心计算引擎特性 ##

MPP（大规模并行处理） ##

查询被分解为多个子查询，分发到各计算节点并行执行，各节点只处理自己分片内的数据，最后汇总。

  • 单机并行：单个节点上查询自动利用所有 CPU 核心。
  • 跨节点并行：通过 Distributed 引擎将查询推到每个分片并行执行。

向量化执行 ##

不同于传统数据库的"逐行处理"（Volcano 模型），ClickHouse 采用批量处理——一次处理一批数据（如 8192 行），充分利用 CPU 的 SIMD 指令（SSE/AVX/NEON）做批量计算。

对架构师的意义：同一份 SQL，在 ClickHouse 上跑可能比在 MySQL 上快 100 倍以上，原因是架构级的差异，而非配置调优。

代码生成（JIT） ##

对查询中的复杂表达式进行 JIT 编译，生成机器码直接执行，消除解释开销。适用于计算密集型查询。

三、存储引擎设计 #

MergeTree 核心机制 ##

MergeTree 是 ClickHouse 所有高级引擎的基类，理解它就理解了 ClickHouse 存储层的核心。

数据组织：Part ##

Table
├── Partition 2024-01
│   ├── Part_001 (已经合并，不可变)
│   └── Part_002 (活跃写入)
├── Partition 2024-02
│   ├── Part_003
│   └── Part_004
  • 每次 INSERT 创建一个新的 Part。
  • Part 是不可变的（写入后即不可修改）。
  • 后台自动将小 Part 合并为大 Part，减少文件数量、提升查询效率。

稀疏索引 ##

不同于 MySQL 的 B+Tree 密集索引（每行数据都有索引条目），ClickHouse 使用稀疏索引——每隔 N 行（由 index_granularity 控制，默认 8192）记录一个索引标记。

  • 优点：索引文件极小，可常驻内存。
  • 缺点：不适合点查询——因为一个索引标记代表 8192 行，查询时需要扫描一个 granule 内的所有行。
  • 主键允许重复：因为主键在这里的作用是排序和索引，而非唯一性约束。

列式存储与压缩 ##

行式存储 (MySQL)：Row1[colA,colB,colC] Row2[colA,colB,colC] ...
列式存储 (ClickHouse)：colA[val1,val2,...] colB[val1,val2,...] ...

列式存储适合 OLAP 的两大原因：

  1. 查询只读需要的列：SELECT count(colA) 不需要碰 colB、colC 的数据文件。
  2. 同列数据相似度高，压缩比极高：同一列的数据类型一致、值域集中，压缩算法可达到 5-20 倍压缩比。

每列数据存储在独立的 .bin 文件中，默认使用 LZ4 压缩，可在建表时按列指定压缩算法（如 CODEC(ZSTD(10))）。

MergeTree 家族 ##

引擎用途
MergeTree基础引擎，高速插入+复杂查询
ReplacingMergeTree按主键去重（合并时保留最新版本）
SummingMergeTree自动对数值列求和聚合
AggregatingMergeTree配合物化视图，存储预聚合结果
CollapsingMergeTree通过 sign 标记实现"逻辑删除"
VersionedCollapsingMergeTree带版本号的逻辑删除
Replicated*以上所有引擎的副本版本（通过 ZK 同步）

数据写入路径 ##

INSERT → 内存 Buffer → 刷盘生成新 Part → 后台 Merge
  • 无 WAL：数据先写内存，再定期刷盘。注意：如果节点宕机，未刷盘的数据会丢失。生产环境建议配置副本。
  • 批量写入为王：ClickHouse 不喜欢逐行 INSERT，每次 INSERT 应该攒够一批（几千到几万行），否则会产生大量小 Part，触发 Too Many Parts 错误。

数据读取路径 ##

查询 → 解析 → 优化 → 分布式路由 → 各节点扫描 Part → 过滤 → 聚合 → 汇总 → 返回
  • 先在本地 Part 上利用稀疏索引定位数据范围。
  • 读取相关列的 .bin 文件（跳过不相关的列）。
  • 解压、过滤、聚合。
  • 分布式查询时，各节点结果汇集到初始节点做最终合并。

四、部署拓扑 #

最小生产部署 ##

┌────────────────────────────────────┐
│  3 × ClickHouse 节点               │
│  (2 分片 × 2 副本，可交叉部署)      │
│  + ClickHouse Keeper（3 节点）     │
└────────────────────────────────────┘
  • 分片数：取决于数据量和写入吞吐需求。数据量 < 1TB，1 分片即可；每增加 ~1TB，增加一个分片。
  • 副本数：建议 ≥ 2，保证高可用。副本数越多，读能力越强，但写延迟越高（Quorum 场景下）。
  • Keeper 节点：3 节点（奇数个），用于副本协调。可用 ClickHouse Keeper 替代 ZooKeeper。

硬件选型建议 ##

组件建议
CPU高主频优先于多核心（查询是单线程瓶颈）；核心数影响并行度
内存至少 32GB，越大越好；稀疏索引和查询中间结果都在内存中
磁盘NVMe SSD 优先（写入和合并操作 I/O 密集）；HDD 仅适合冷数据归档
网络千兆起步，万兆推荐（分布式查询涉及节点间数据传输）

集群部署方式 ##

  1. 手动部署：RPM 安装 + 配置文件管理。适合少量节点、对定制化要求高的场景。
  2. Ansible/Chef：中大规模集群，可复用 playbook。
  3. Kubernetes：通过 clickhouse-operator 或 Altinity operator 部署。适合已有 K8s 基础设施的团队。
  4. ClickHouse Cloud：托管服务，免运维，按量付费。

五、成本估算 #

硬件成本（自建） ##

以每天写入 500GB、保留 30 天、1 副本为例：

  • 存储总量：500GB × 30 × 压缩比(1/5) × 2(副本) ≈ 6TB 有效存储 + 冗余
  • 建议配置：3 节点 × 4TB NVMe SSD × 64GB RAM × 16 vCPU
  • 粗略硬件成本：约 ¥3-5 万/年（裸金属租赁或云主机，不含网络和运维人力）

运维成本 ##

  • 日常运维：集群扩缩容、备份恢复、版本升级、监控告警。至少需要一个 0.5 人力持续投入。
  • 查询优化：ClickHouse 查询对 SQL 写法敏感（JOIN 顺序、子查询、PREWHERE 等），需要团队有 ClickHouse 经验。
  • 数据治理：TTL 策略、冷热分层、过期数据清理，需要持续管理。

云托管成本 ##

ClickHouse Cloud（及阿里云/腾讯云托管版）按计算单元和存储量计费，生产级实例约 ¥3000-10000/月起步。优势是免运维，劣势是单价高于自建。

降本策略 ##

  • TTL 自动过期：按时间或数据量设置 TTL，自动删除旧数据或将其转移到冷存储（S3/OSS）。
  • 压缩优化：对大数据量列用 CODEC(ZSTD) 进一步压缩，存储减少 30-50%。
  • 冷热分层：热数据放 SSD，冷数据放 HDD 或对象存储，ClickHouse 支持多卷存储策略。
  • 合理分区：按天或按月分区，避免分区过大（扫描放大）或过小（Part 过多）。

六、运维关键点 #

Too Many Parts ##

最常见的生产事故。如果单表 Part 数超过 parts_to_throw_insert（默认 3000），ClickHouse 会拒绝写入。

原因：频繁小批量 INSERT，后台 Merge 速度跟不上 Part 生成速度。

解决：

  • 应用层攒批写入，每次 ≥ 10000 行。
  • 降低插入频率，使用 Buffer 引擎做写入缓冲。
  • 适当调整 Merge 参数（max_bytes_to_merge_at_max_space_in_pool）。

ZooKeeper 压力 ##

大量表（数百张）的副本同步压力会导致 ZK 性能瓶颈。

表现：副本延迟持续增大，写入变慢。

解决：

  • 迁移到 ClickHouse Keeper（性能优于 ZK）。
  • 控制 ReplicatedMergeTree 表数量（尽量合并表，而非无限建表）。
  • ZK 独立部署，不和 ClickHouse 混部。

查询性能陷阱 ##

  • SELECT *：列存下仍然昂贵，因为需要读取+解压所有列文件。只 SELECT 需要的列。
  • 大 JOIN：ClickHouse 的 JOIN 是 Broadcast Hash Join（右表广播到所有节点），大表 JOIN 大表性能极差。应尽量在 ETL 阶段做宽表。
  • 不带分区键的查询：全表扫描所有 Partition。WHERE 条件应尽量命中分区键。
  • 大量 GLOBAL IN：结果集在网络中广播，容易打满带宽。

备份与恢复 ##

  • ALTER TABLE … FREEZE：创建文件系统级别的快照。
  • clickhouse-backup：社区最常用的备份工具，支持全量和增量备份到 S3/本地。
  • 复制副本本身就是备份：多副本在不同物理机上，本身就是一种容灾手段——但不能代替异地备份。

│   ├── MPP 并行计算（单机+分布式双档并行）

│   ├── Part + 后台 Merge（核心机制）

│   ├── 分片×副本 拓扑

│   ├── 自建：¥3-5万/年（3节点起步）

└── 运维
    ├── 危险信号：Too Many Parts、ZK压力、大JOIN
    └── 必备工具：clickhouse-backup、Grafana、exporter
