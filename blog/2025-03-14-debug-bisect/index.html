bisect：二分定位多特性开关中引入的bug

Posted 2025-03-14 00:00 +0800 by ZhangJie ‐ 3 min read

分享:  

开发过程中经常遇到这种情况：项目里有多个特性开关，某次发布后测试报了个 bug，但你不知道是哪个开关、或者哪几个开关的组合导致的。如果只有两三个开关，手动 enable/disable 一个个试也行。但如果开关数量多、迭代频率高、开关之间还相互影响，手动排查的效率就完全跟不上了。

这个问题 Go 编译器团队也遇到过——编译器的每个优化 pass 都可能引入 bug，而且这些优化之间会相互触发。他们搞了一套叫 bisect 的定位方法，Russ Cox 专门写了篇文章介绍1，我在 golang-debugger-book 里也做过系统性的拆解。后来这套方法被沉淀成了 Go 官方的 bisect 工具和库2，任何 Go 项目都能直接用。

本文通过一个实际的例子演示怎么把 bisect 接入项目，以及怎么用它快速定位问题开关。

场景：UDP echo server 改造为支持 TCP #

假设我们维护着一个 UDP echo server，现在要做一次改造，让它也支持 TCP。改动的点比较多：

  • 新增 TCP listener 和 accept 循环
  • 连接处理中引入 buffered write
  • 新增字符串反转功能（产品要求）
  • 还顺手调整了连接关闭逻辑

为了控制上线风险，每个改动点都有一个开关。代码大概长这样：

var (
    useTCP           = true  // 开关 1：是否启用 TCP
    useBufferedWrite = true  // 开关 2：是否使用 buffered write
    reverseEcho      = true  // 开关 3：是否反转回显内容
    skipClose        = false // 开关 4：是否跳过连接关闭
)

func main() {
    if useTCP {
        go startTCPListener()
    }
    go startUDPListener()
    select {}
}

func handleConnection(conn net.Conn) {
    defer func() {
        if !skipClose {
            conn.Close()
        }
    }()

    buf := make([]byte, 1024)
    for {
        n, err := conn.Read(buf)
        if err != nil {
            return
        }
        response := processResponse(string(buf[:n]))

        if useBufferedWrite {
            writer := bufio.NewWriter(conn)
            writer.WriteString(response)
            // BUG: 忘了 writer.Flush()
        } else {
            conn.Write([]byte(response))
        }
    }
}

func processResponse(data string) string {
    if reverseEcho {
        words := strings.Split(data, " ")
        for i := range words {
            words[i] = reverseString(words[i])
        }
        return strings.Join(words, " ")
    }
    return data
}

func reverseString(s string) string {
    runes := []rune(s)
    // BUG: 循环条件应该是 len(runes)/2
    for i := 0; i < len(runes)/2-1; i++ {
        runes[i], runes[len(runes)-1-i] = runes[len(runes)-1-i], runes[i]
    }
    return string(runes)
}

改造完成后一跑测试：client 发过去的数据，server 没回。这里有两个 bug——buffered write 忘了 flush，字符串反转的循环条件也写错了。

四个开关中两个有问题，可能分别是哪个导致的，也可能需要两个同时打开才触发——这就不是手工试几下能搞定的了。

接入 bisect #

原理 ##

bisect 的核心是一个简单的协议：程序在运行时会输出带有 hash 标记的调试信息，告诉外部工具"我这里打开（或关闭）了一个开关点"；bisect 工具收集这些 hash，然后通过环境变量 BISECT_PATTERN 下发不同 pattern，控制哪些点启用、哪些点禁用，像二分查找一样不断缩小故障范围。

接入 bisect 需要做两件事：

  1. 在代码中引入 golang.org/x/tools/internal/bisect 库，给每个开关点分配 ID 并随 pattern 控制启停
  2. 写一个测试脚本，bisect 工具通过反复调用这个脚本来判断当前 pattern 下程序行为是否正常

步骤 1：改造代码 ##

import "golang.org/x/tools/internal/bisect"

var matcher *bisect.Matcher

func init() {
    pattern := os.Getenv("BISECT_PATTERN")
    if pattern != "" {
        var err error
        matcher, err = bisect.New(pattern)
        if err != nil {
            log.Fatal(err)
        }
    }
}

// shouldEnable 包装了 bisect 的开关逻辑
func shouldEnable(id uint64) bool {
    if matcher == nil {
        return true // 没有设置 BISECT_PATTERN 时，默认全部启用
    }
    enabled := matcher.ShouldEnable(id)
    if matcher.ShouldReport(id) {
        fmt.Fprintf(os.Stderr, "%s %d\n", bisect.Marker(id), id)
    }
    return enabled
}

func main() {
    useTCP           = shouldEnable(1)
    useBufferedWrite = shouldEnable(2)
    reverseEcho      = shouldEnable(3)
    skipClose        = shouldEnable(4)

    // 后续逻辑不变
}

改造量很小：每个开关点分配一个数字 ID（1, 2, 3, 4），初始化时读 BISECT_PATTERN 环境变量，然后用 shouldEnable 决定启停。ShouldReport 那行是关键——它输出带 hash 的 marker，bisect 工具就是靠解析这些输出来知道当前有哪些开关被命中的。

步骤 2：写测试脚本 ##

#!/bin/bash

# test_server.sh — bisect 的 target script
# 返回 0 表示通过，非 0 表示失败

./server &
PID=$!
sleep 1

# 发个请求，检查是否收到正确回显
echo "Hello World" | nc localhost 8080 > output.txt 2>/dev/null

kill $PID
wait $PID 2>/dev/null

if grep -q "Hello World" output.txt; then
    exit 0
else
    exit 1
fi

bisect 工具会反复执行这个脚本，每次设置不同的 BISECT_PATTERN，根据脚本的退出码判断"当前这组开关组合下程序是否正常"。

二分定位 #

一切就绪，运行 bisect：

go build -o server .
bisect -timeout=5s ./test_server.sh

输出大致这样：

bisect: checking target with all changes disabled
bisect: run: BISECT_PATTERN=n test_server.sh... ok
bisect: checking target with all changes enabled
bisect: run: BISECT_PATTERN=y test_server.sh... FAIL

bisect: target succeeds with no changes, fails with all changes
bisect: searching for minimal set of enabled changes causing failure
bisect: run: BISECT_PATTERN=-0 test_server.sh... ok
bisect: run: BISECT_PATTERN=-0,1 test_server.sh... FAIL
bisect: run: BISECT_PATTERN=-0,1,2 test_server.sh... FAIL
bisect: run: BISECT_PATTERN=-0,1,2,3 test_server.sh... ok
...

--- change set #1 (enabling changes causes failure)
[bisect-match 0x...] 2
---

bisect 的执行流程：

  1. 确立基线：先跑 BISECT_PATTERN=n（关掉所有开关），确认程序本身没问题；再跑 BISECT_PATTERN=y（打开所有开关），确认 bug 确实能被触发。
  2. 二分搜索：逐次将开关集合减半，找出"打开哪些开关能触发故障"的最小集合。每次运行它都解析程序输出的 marker，比对当前的 pattern 是否按预期命中了对应开关。
  3. 输出结论：给出故障开关的 ID 列表。

结果告诉我们 ID 2（useBufferedWrite）导致了问题。修掉那个漏掉的 Flush() 之后重新跑，又定位到 ID 3（reverseEcho）的循环边界问题。

ps: 实际的 bisect 输出更详细，它会显示每次二分时的 pattern 和匹配的 marker 数量。Hash 的位数会根据总点位数量自动调整——3 个点位可能 1 位 hex 就够了，上百个点位就需要更多。

什么时候值得用 #

不是所有场景都需要 bisect。如果就两三个开关、改几下就试出来了，手动排查更快。

但如果满足以下条件之一，bisect 的收益就很大了：

  • 开关数量多（10 个以上），手动遍历组合的时间是 O(2^n)，而 bisect 是 O(log n)
  • 迭代频率高，每次发布都有多个开关变更，开关之间还可能相互影响
  • bug 由组合触发，不是某一个开关的问题，而是开 A + 开 B 才出 bug——手工几乎不可能穷举

ps: Go 编译器里还有一层更精致的机制叫 hash-debug，编译器内的每个开关点在编译时会根据文件名+行号计算 hash，实现更细粒度的按点位控制。当前应用的场景是日常开发中的特性开关，用 ID 分配的方式就够用了，不需要这么精细。

另外，bisect 工具和 target 程序之间的通信全部走 stderr 和退出码，不依赖任何额外的基础设施，接入零成本。CI pipeline 里也可以直接用——把测试脚本挂上去，出了 regression 自动跑一遍 bisect，比翻日志定位快得多。
