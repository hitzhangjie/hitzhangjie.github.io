Etcd 架构设计详解

Posted 2026-07-19 10:00 +0800 by ZhangJie ‐ 3 min read

分享:  

1. Etcd 是什么 ##

Etcd 是一个分布式、强一致的键值存储系统，由 CNCF（云原生计算基金会）托管。它基于 Go 语言开发，使用 Raft 共识算法保证多节点间的数据一致性，即使在节点失效（包括 Leader 失效）的情况下也能可靠地提供服务。

从简单的 Web 应用到复杂的 Kubernetes 集群，都可以通过 Etcd 进行服务发现、配置管理和分布式协调。其核心设计围绕 强一致性、高可用 和 简单可靠 展开。

项目地址：etcd-io/etcd

2. 核心特性 ##

Etcd 提供以下关键能力：

  • 强一致性：基于 Raft 协议，所有节点看到的数据严格一致，支持线性一致性读（Linearizable Read）。
  • MVCC 多版本控制：每次写入生成新版本，旧版本保留一段可配置的时间，支持历史版本查询。
  • Watch 监听机制：客户端可订阅某个键或前缀的变更事件，通过 gRPC 流实时推送，是 Kubernetes 声明式状态同步的基石。
  • Lease 租约：带 TTL 的轻量级对象，可用于服务心跳保活和自动过期键值对。
  • gRPC API：v3 API 使用 gRPC + Protobuf，兼顾性能与跨语言支持，同时提供 HTTP/JSON 网关。
  • 简单可靠：单二进制部署，内置监控指标，操作门槛低。

3. 典型应用场景 ##

  • Kubernetes：使用 Etcd 存储集群所有资源对象（Pod、Service、ConfigMap 等）的状态。
  • CoreDNS：支持将 Etcd 作为后端存储，实现动态 DNS 记录管理。
  • 服务发现：微服务注册实例信息，客户端通过 Watch 感知实例上下线。
  • 分布式协调：通过 Lease 实现分布式锁、Leader 选举等原语。
  • 配置中心：集中管理应用配置，客户端监听配置变更并热更新。

4. 整体架构 ##

Etcd 集群由多个节点组成，每个节点运行完整的 Etcd 实例。架构上分为三层：API 层 → Raft 共识层 → 存储层。

        +---------------------------------+
        |          Client / App           |
        +----------------+----------------+
                         |
        +----------------+----------------+
        |           API 层                 |
        |    gRPC API    |  HTTP Gateway  |
        |  (Protobuf)    | (grpc-gateway) |
        +----------------+----------------+
                         |
        +----------------+----------------+
        |        Raft 共识层               |
        |  Leader 选举 · 日志复制 · 提交   |
        +----------------+----------------+
                         |
        +----------------+----------------+
        |        存储层                    |
        |  MVCC 索引 · bbolt · WAL · 快照  |
        +---------------------------------+

4.1 API 层 ##

Etcd v3 提供两套 API 入口：

  • gRPC API：原生高性能接口，使用 Protobuf 定义 Put、Get、Watch、Lease 等 RPC 方法。官方提供 Go、Java、Python 等语言的客户端库，内置连接管理、负载均衡、自动重试等能力。
  • HTTP/JSON Gateway：通过 grpc-gateway 将 HTTP/JSON 请求转换为 gRPC 调用，方便 curl 调试和非 gRPC 环境接入。注意这不是独立服务，而是 Etcd 进程内置的网关。
// Go 客户端示例：写入键值对
cli, _ := clientv3.New(clientv3.Config{
    Endpoints: []string{"http://127.0.0.1:2379"},
})
defer cli.Close()
_, err := cli.Put(context.TODO(), "/config/db", "postgres://...")

4.2 Raft 共识层 ##

Raft 是 Etcd 保证强一致性的基石。每个节点的 Raft 状态机负责：

  • Leader 选举：节点以 Follower 身份启动，若在选举超时内未收到 Leader 心跳，则转为 Candidate 发起投票，获得多数票者成为新 Leader。
  • 日志复制：所有写请求必须先到达 Leader，Leader 将操作封装为日志条目，追加到本地 WAL 并广播给 Follower；收到多数节点（Quorum）确认后提交（Apply）到状态机。
  • 安全保证：Raft 确保日志的连续性（Log Matching）和 Leader 日志的完整性（Leader Completeness），即使发生网络分区或节点宕机，最终也能恢复到一致状态。

节点角色：

角色职责
Leader处理所有客户端写请求，驱动日志复制，定期发送心跳
Follower被动接收 Leader 日志并响应心跳，不处理写请求
Candidate选举期间的临时角色，向其他节点请求投票

提示：当客户端向 Follower 发起写请求时，Follower 会返回 gRPC Redirect 错误，告知真正的 Leader 地址，客户端库通常会自动处理此重定向。

4.3 存储层 ##

存储层由三部分组成：

  • WAL（Write-Ahead Log）：Raft 日志条目在应用到状态机之前先写入 WAL，保证持久性和崩溃恢复。WAL 保存在磁盘的 wal/ 目录。
  • bbolt：Go 语言实现的嵌入式 B+ 树键值数据库，是 Etcd 的状态机存储。Etcd 维护自己的 bbolt 分支，提供稳定的事务型 KV 存储。数据文件保存在 member/snap/db。
  • MVCC 索引：在 bbolt 之上构建的多版本并发控制层。每个键对应一个单调递增的 revision，写操作生成新版本而非覆盖旧值。索引结构支持高效的按 revision 范围查询和 Watch 事件回溯。
# 示例：MVCC 历史版本读取
etcdctl put key "value1"
etcdctl put key "value2"
etcdctl get key --rev=1   # 返回 value1（历史版本）
etcdctl get key           # 返回 value2（最新版本）

4.4 Watch 机制 ##

Watch 是 Etcd 区别于普通 KV 存储的核心特性，它通过 事件流 实现 pub/sub 模式：

  1. 客户端通过 gRPC 流式 RPC 发起 Watch，指定监听的键或前缀。
  2. Etcd 将 watcher 注册到对应键的监听列表。
  3. 每次写操作提交后，相关的变更事件（PUT/DELETE）推送给所有匹配的 watcher。

典型应用：Kubernetes 的 Controller 通过 Watch Apiserver（Apiserver Watch Etcd）感知资源变化，驱动调谐（Reconcile）循环。

4.5 Lease 租约机制 ##

Lease 是一种带 TTL 的轻量级对象：

  • 客户端创建 Lease（如 TTL=10s），获取 Lease ID。
  • 将键绑定到该 Lease：键的有效期和 Lease 绑定。
  • 客户端需定期 KeepAlive 续约；若 Lease 到期未续，绑定的所有键被自动删除。
# 示例：服务注册
etcdctl lease grant 10              # 创建 10s 租约，返回 lease-id
etcdctl put /services/app 192.168.1.1 --lease=694d7c1c
# 每 1/3 TTL 左右执行一次续约
etcdctl lease keep-alive 694d7c1c

生产环境中，建议客户端库启用自动 KeepAlive（如 Go SDK 的 clientv3.WithLease 会自动维护），避免手动脉冲式的续约。

4.6 快照与压缩 ##

Etcd 提供两级空间回收机制：

  • Raft 日志快照：Raft 日志会随写入无限增长，Etcd 定期将状态机内存数据序列化到 .snap 文件，并丢弃快照点之前的 Raft 日志条目，控制磁盘占用。
  • MVCC 压缩（Compaction）：MVCC 的旧版本数据会占用 bbolt 空间，需要定期压缩。可通过 --auto-compaction-retention 配置自动压缩策略（如保留最近 2 小时的历史版本），超出部分在压缩时清除。

5. 典型部署拓扑 ##

下面是一个典型的三节点 Etcd 集群架构：

        +----------------------+
        |     Client App       |
        +----------+-----------+
                   |
           gRPC / 负载均衡
                   |
        +----------+-----------+
        |          |           |
        v          v           v
   +---------+ +---------+ +---------+
   | Node A  | | Node B  | | Node C  |
   | (Leader)| |(Follower)| |(Follower)|
   +----+----+ +----+----+ +----+----+
        |          |           |
        +----------+-----------+
                   | Raft 日志复制
                   v
        +----------+-----------+
        |  持久化存储（各节点独立）|
        |  bbolt + WAL + 快照   |
        +----------------------+

请求路径：

  • 写请求：必须发往 Leader。Leader 写入本地 WAL → 广播日志给 Follower → 等待多数确认 → 提交到状态机 → 返回客户端。
  • 线性一致性读：默认读也由 Leader 处理，通过 ReadIndex 机制在不写 WAL 的前提下保证线性一致性。
  • 串行化读（Stale Read）：允许从 Follower 读，延迟更低但可能返回过期数据，适合对实时性要求不高的场景。

6. 关键设计权衡 ##

6.1 强一致性 vs 可用性 ##

Etcd 是 CP 系统（CAP 定理），优先保证线性一致性。代价是当集群失去多数节点（如 3 节点中 2 节点宕机）时，剩余节点无法形成 Quorum，集群不可用。

影响：访问延迟对稳定性至关重要。Raft 要求写入必须等待多数节点的确认，因此集群整体延迟实际上由最慢的节点决定。在网络不稳定或磁盘 IO 抖动时，单个慢节点会拖慢整个集群。建议对延迟指标（backend_commit_duration_seconds、wal_fsync_duration_seconds）进行监控和告警。

6.2 读写分离的取舍 ##

模式一致性延迟吞吐适用场景
线性一致性读（Leader）强一致中等受 Leader 限制配置读取、分布式锁
串行化读（Follower）最终一致低高监控数据、非关键查询

建议默认使用线性一致性读，仅在确认可容忍延迟数据时开启串行化读。

6.3 性能优化方向 ##

  • 批量写入：合并多个写操作为一个事务提交，减少 Raft 共识轮次。
  • 磁盘 IO 隔离：使用高 IOPS 的 SSD/NVMe，避免与其他高 I/O 应用共用磁盘。
  • 网络平面隔离：将 Peer 通信端口（2380）与 Client 端口（2379）走不同网络，避免客户端大查询影响 Raft 心跳和日志复制。
  • 合理的快照与压缩策略：避免 WAL 和 bbolt 数据库无限增长。

7. 总结 ##

Etcd 的架构设计简洁而健壮：

  • Raft 提供强一致性共识基础，是分布式系统的安全网；
  • MVCC + bbolt 支持高效的持久化存储和历史版本访问；
  • Watch + Lease 满足动态服务发现和分布式协调的核心场景；
  • gRPC API 保证高性能和跨语言生态。

理解这些核心组件的设计意图和交互方式，有助于在实际系统中正确地使用和运维 Etcd 集群。关于具体场景的深入讨论，请参考本站 Etcd 系列的其他文章。
