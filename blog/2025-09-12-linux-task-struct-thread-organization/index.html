Linux 线程的组织结构与 task_struct 字段解析

Posted 2025-09-12 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

Linux 中线程和进程的实现都基于同一个数据结构 task_struct。每个线程有自己独立的 task_struct，但同一个进程的不同线程又需要共享地址空间、文件描述符等资源。

这种"独立但共享"的关系，通过在 task_struct 中区分共享字段和独有字段来实现。理解这个设计对掌握 Linux 多线程编程和内核行为很有帮助。

进程标识：pid 与 tgid #

这是理解线程模型最基础的一对字段：

pid_t pid;        // 线程 ID——每个线程独有
pid_t tgid;       // 线程组 ID——同一进程的所有线程相同
  • pid：每个线程有唯一的线程 ID（也是 gettid() 的返回值）。
  • tgid：同一进程的所有线程共享相同的线程组 ID。对用户空间来说，这就是进程 ID（getpid() 实际返回的是 tgid）。

线程组的组织 #

struct task_struct *group_leader;    // 指向线程组领导者（主线程）
struct list_head thread_group;       // 线程组链表节点
struct list_head thread_node;        // 在 signal->thread_head 中的节点

组织结构：

  • 主线程的 group_leader 指向自己。
  • 其他线程的 group_leader 指向主线程。
  • 所有线程通过 thread_group 链表连接在一起（双向循环链表）。

这样内核可以随时从任意线程遍历整个线程组。

共享字段 #

同一进程的所有线程共享以下资源：

内存管理 ##

struct mm_struct *mm;           // 内存描述符
struct mm_struct *active_mm;    // 活跃内存描述符

所有线程共享同一个虚拟地址空间。这意味着一个线程修改了全局变量，其他线程立即可见——不需要任何"同步机制"就能共享数据（当然，为了正确性仍需加锁）。

文件系统与文件描述符 ##

struct fs_struct *fs;           // 文件系统信息（根目录、当前工作目录等）
struct files_struct *files;     // 打开的文件描述符表

所有线程共享同一份 files_struct，因此文件描述符是进程级别的资源。线程 A 打开的文件，线程 B 可以直接用同一个 fd 读写。

命名空间 ##

struct nsproxy *nsproxy;        // 命名空间代理

命名空间（UTS、IPC、PID、网络、挂载等）是进程级别的，所有线程共享。

信号处理 ##

struct signal_struct *signal;   // 共享信号结构
struct sighand_struct *sighand; // 信号处理结构
  • signal 中包含了 shared_pending——进程级信号队列，这也是进程中任意线程都可以处理 SIGCHLD 等进程级信号的底层原因。
  • sighand 包含信号处理函数表，所有线程共享同一组 handler。一个线程修改了 handler（signal()/sigaction()），所有线程都受影响。（这是一个常见的坑——多线程程序中安装信号 handler 需要注意）

进程凭据 ##

struct cred *real_cred;         // 真实凭据（UID/GID 等）
struct cred *cred;              // 有效凭据

凭据是进程级别的，但内核允许线程临时修改自己的凭据（如通过 seteuid()），这会影响整个进程。

线程独有字段 #

每个线程拥有独立的以下资源：

线程标识 ##

pid_t pid;                      // 线程 ID
struct pid *thread_pid;         // 线程 PID 结构

调度信息 ##

volatile long state;            // 线程状态（TASK_RUNNING, TASK_INTERRUPTIBLE 等）
int prio;                       // 动态优先级
int static_prio;                // 静态优先级
int normal_prio;                // 普通优先级
struct sched_entity se;         // CFS 调度实体

调度是线程级别的——每个线程独立调度，有独立的优先级和时间片。

CPU 使用统计 ##

u64 utime;                      // 用户态 CPU 时间
u64 stime;                      // 内核态 CPU 时间
unsigned long nvcsw;            // 主动上下文切换次数
unsigned long nivcsw;           // 被动上下文切换次数

/proc/<pid>/stat 中看到的 CPU 时间就是这些字段的汇总。

线程特定信号 ##

sigset_t blocked;               // 阻塞的信号掩码
sigset_t real_blocked;          // 真实阻塞的信号掩码
struct sigpending pending;      // 私有待处理信号

每个线程可以通过 pthread_sigmask() / sigprocmask() 独立地阻塞或解除阻塞特定信号。这是多线程程序分配信号处理负载的关键机制——让不希望处理某些信号的线程屏蔽掉它们。

线程栈和寄存器 ##

void *stack;                    // 内核栈指针
struct thread_info thread_info; // 线程信息（包含寄存器状态等）

每个线程有独立的内核栈（通常 8KB 或 16KB），用来保存系统调用和中断上下文。

线程特定同步 ##

raw_spinlock_t pi_lock;         // 优先级继承锁
struct wake_q_node wake_q;      // 唤醒队列节点

总结 #

类别共享/独有典型字段
地址空间共享mm, active_mm
文件描述符共享files, fs
信号处理共享（handler）+ 独有（mask）signal, sighand + blocked, pending
凭据共享cred, real_cred
线程 ID独有pid, thread_pid
调度独有state, prio, se
CPU 时间独有utime, stime
内核栈独有stack, thread_info

这种设计使得 Linux 能高效地支持多线程程序：共享的资源避免了重复开销，独立的字段保证了每个线程的调度独立性和信号处理灵活性。理解这些区分，有助于在多线程编程中做出正确的决策——比如为什么在多线程程序中修改 signal() handler 是危险的、以及如何利用信号掩码来分配信号处理负载。
