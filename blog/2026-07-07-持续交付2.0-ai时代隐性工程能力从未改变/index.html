重读《持续交付2.0》：从EPC到AI Harness工程，隐性工程能力从未改变

Posted 2026-07-07 12:12 +0800 by ZhangJie ‐ 5 min read

分享:  

背景 #

2019 年到 2022 年，我深度参与了腾讯侧的技术治理工作：微服务框架、研发效能提升、统一协议托管、DevOps、测试左移、一站式研发发布平台……那几年对「软件工程」和「技术沉淀」的理解被拉高了一截。2022 年因团队变动，结合个人意愿，我加入了天美某在研的游戏后台团队，项目管理上实践良好，但是团队后台工程实践上 …… 也算是让我打开眼界了。

近几年 AI 大火，公司、团队、个人都在追问 AI 的能力边界，并希望把能力赋能到个人与组织——行业如此。我从 2021 年 GitHub Copilot 发布起就持续跟踪；ChatGPT 之后的一系列现象级产品与重大进展，常常让我生出「要不要 ALL IN AI」的冲动。从 Prompt 工程到 Context 工程，再到 Harness 工程，实践很热，但真正能驾驭 AI 的团队极其少见，营销反而更多。从 AI Assist 到 AI First，远不只是「把 Harness 搭好」这么简单。若真这么简单，Agent 内置软件工程领域的 SKILL 便可横扫一切——现实是没有任何一个 Agent 能独自搞定所有领域。也许需要时间，也许需要专用 Agent 的协作与编排；无论如何，至今还没有哪家企业能拍着胸脯说已经把 AI 用得很好，但这件事我认为会被实现，我这里欲扬先抑一下。

与此同时，常听到一种论调：「AI 总有搞不定的领域。」这不意外——对量化别人的产出，很多人毫无概念，要么笼统判断，要么在专业领域沉淀不够，找不到合理量化指标。「一切皆可计算」：若不能计算，人的活动便既盲目又冒险；审视研发是否科学，就是量化、发现问题、改进、再量化的循环。乔梁老师在《持续交付 2.0：引领业务的 DevOps 精要》里，对这条主线讲得系统而具体。当然，这里的量化指标是用来验证我们的假设，改进我们的流程，而不是我们的终极目标。

不少参与过研发效能（EPC，工程效能胜任力）落地的人，对乔梁老师存在很深误解：理念死板、理论大于实践、花架子。多年之后，当我思考如何做出更好的 AI Agent、如何驾驭 Agent 时，突然发现：研发活动里的种种 SKILL，与过去沉淀的一些隐形技能之间，存在清晰映射。于是从书架取下这本书——当年还是腾讯技术副总裁曾宇与腾讯特邀专家乔梁老师亲笔签名本——耐着性子读了两周。越往后感触越深：过去的判断更详实了，也修正了一些认识误区。内网仍可搜到曾宇关于服务器开发的 PPT，言简意赅、有洞见，很多 2026 年的高级工程师也未必能写出那种密度。

有人说他们「空有其表」——在我看来，说这话的人才是真正的傲慢与无知。若真要埋怨，应埋怨当年 EPC 落地中某些团队的务虚作风，而不是理念本身。时至今日，这些理念对许多业务仍有牵引、指导作用。理念像水，没有绝对最好，只有更合适；唯一不变的是变：持续优化可维护性是变，满足于现状沦为屎山也是变。有洞见的 TL 能观察微妙变化，立规矩、建流程与质量门禁，让隐晦不可见的风险点变得可见，并专注培养人与提振技术氛围。若连学习进步的欲望都没有，工程上的怠惰难免迁移到业务领域探索优化上的怠惰。和大范围用 AI 一样，我们工作不只为了解决问题，也在培养与传承。许多感动用户的设计与高可用，是在思辨进步的氛围里一点点夯出来的——AI 不会给你一切；若本身不是那块材料，即便 AI 给出更好方案，也不敢采纳，因为已超出认知。

下文按全书重要方向展开，并穿插上述个人经历与 AI 时代的对照。各章并非读书笔记摘抄，而是「书里的骨架 + 落地时的骨肉记忆」。

一、持续交付 2.0 在回答什么问题（第 1 章） #

1.1 为什么软件工程课像「斗兽场」——以及后来我才读懂的第 1 章 ##

从本科接触《软件工程》起，我就记得一种很荒诞的课堂景观：大家平时爱写代码、爱拿算法解点实际问题，一到这门课，教室自动变成古罗马斗兽场——老师在场地中央，同学们能往后坐就往后坐，能贴墙就贴墙。为什么？因为彼时脑海里、实践里都还没真正碰过复杂工程：写的多是玩具，也还不具备中长期维护一个中大型项目的能力。软件工程在课上显得枯燥，本质是痛苦还没到账。

瀑布那套重文档（需求分析、概要设计、详细设计、任务分解与实现），在课程大作业里也不是人人当真。我算相对较认真的一类——之前接触过稍大一点的课题，隐约知道无规矩不成方圆：软件工程研究的核心课题，历来就是如何在风险与成本可控的前提下，保质保量交付。书第 1 章把瀑布、敏捷、DevOps、持续交付 1.0 串成一条线，我重读时不再觉得「背考点」，而是在对号入座：瀑布的阶段性在规范什么；敏捷在回应什么；当拥抱变化的需求、插入需求、临时需求把文档驱动拖垮之后，团队为何转向更敏捷的协作与交付。

这里没有银弹。对当时的团队来说，合适的就是不错的。但若目标变成全流程提效，就不能只优化「写代码」那一截，而要正视整条研发链：有哪些环节、环节之间的边界在哪里（参与人、职能）、哪些可以度量、度量之后如何改进——这正是第 1 章引出 CD 2.0、双环模型、持续交付七巧板 的背景。概念本身不必像教科书那样展开，重要的是理解它们为何合理、在解决什么别扭。

1.2 双环模型：探索环 + 验证环 ##

持续交付 2.0 相对 1.0 的关键增量，是把「发布管线」抬升到与「价值发现」并列的双环模型：

  • 价值探索环：问对问题、锚定假设、共创方案、精炼到可验证粒度；
  • 快速验证环：构建、运行、监测、决策——用运行态数据与用户反馈闭合环路。

四个核心原则（价值导向、快速验证、持续学习、弹性稳健）和后面的组织文化、度量改善（第 4 章）是咬合的。双环模型的理论源头是精益思想——消除价值流中的浪费、拉动式交付、持续改善——探索环解决「做正确的事」，验证环解决「正确地做事」，少一条优化都是局部的。EPC 年代若只抄工具、不抄双环，很容易变成「Jira 很满、价值很虚」：探索环回答「做对了没有」，验证环回答「做得快不快、稳不稳」。从本科「嫌软件工程枯燥」到治理期「天天争论门禁与发布频率」，中间缺的那一课，其实就是两个环都要闭合——只快不对，只对不快，都撑不起长期维护。

放到今天：AI 时代更应警惕只加强验证环的幻觉（生成代码、生成流水线很快），却跳过探索环的锚定。模型再强，锚定错了，只会更快地产出错误。

1.3 持续交付七巧板 ##

「七巧板」是把实践模块化的隐喻：组织文化、架构、协作、流水线、分支与集成、测试、发布与监测……拼起来才是一张可落地的图。我的体会和课堂年代相反——年轻时觉得「规矩」多余，做过平台与治理之后才承认：缺一块就补一块的「工具项目」，和按七巧板系统拼图，成功率差一个数量级。

后来做 Agent 的 Skill 编排，本质也是七巧板：载体从 Jenkins、一站式发布平台，变成了 Markdown 与 MCP。但拼图顺序没变——先弄清环节与边界（第 1 章、双环），再谈自动化与模型。否则 Skill 越多，只是在玩具工程上贴更多贴纸。

二、价值探索环：用户说的「我想要」，往往不是答案（第 2 章） #

第 2 章在书里写成探索环四步（提问、锚定、共创、精炼）和一堆方法——装饰窗（放个假入口测用户兴趣）、最小可行特性（只交付能验证假设的最小切片）、特区法（隔离一小撮用户控制爆炸半径）、定向探索法（针对特定用户群深挖）、稻草人法（扔一个粗糙方案激出真实反馈）、MVP（最小可行产品验证核心假设）——名字各异，逻辑一致：用最小代价验证最关键假设。我落地时的理解更直白：探索环关心的是，如何理解真正的用户需求。

用户说「我想要什么」，在流程上只是收集到的问题反馈——它既不是最终解决方案，也常常不是最本质的需求。团队要对反馈做深入思考，挖出背后用户真正要解决的问题，再和研发一起讨论：我们准备怎样交付这个需求。探索环里典型工作包括：挖掘需求、细化需求、为效果做数据埋点与分析设计等。若把用户原话直接排进 Sprint，探索环等于没走。

书里的原则（分解快速试错、一次只验证一点、允许失败）都是在约束这类跳跃。腾讯治理期我见过太多反面教材：需求包太大、一次想验证五个假设、考核只认「上线成功」不认「假设被证伪」。顶流项目压力大时，也更容易跳过锚定直接堆人天——探索环最先被牺牲。

与 AI 的交界：模型很擅长把「用户原话」扩写成 PRD、故事地图，但替你做需求考古和承担锚定后果仍靠人。若 Context 里没有「我们假设用户要的是 A，成功指标是 B，证伪条件是 C」，Agent 只会把错误方案写得更漂亮。探索环的做事规则应写进协作规范，而不是指望AI自行发挥。

三、快速验证环：交付、反馈与再进入探索（第 3 章） #

验证环则是：如何把已经澄清的需求交付出去。 在我经历过的团队里，它至少覆盖这些动作（与书里的构建、运行、监测、决策一一呼应，只是用工程语言说）：

  • 需求方案评审：设计是否可分批交付、风险点是否说清；
  • 分批交付与快速验证：先上最小切片，而不是一次押满；
  • 快速质量反馈：CI、测试左移、门禁——让问题尽早暴露；
  • 更稳妥的测试管理、制品管理；
  • 发布上线、灰度测试，以及根据线上信号快速调整需求；
  • 科学埋点，方便产品侧做数据度量，对需求与效果做综合分析，用来指导下一轮生产活动。

这里有一个容易被跳过的关键规则：大需求必须拆成小需求、小任务，分批交付、分批验证，而不是攒到最后集中式大量交付。验证环追求的核心指标不是「一次交付了多少功能」，而是「多快能拿到一次可行动的反馈」。每推迟一天验证，代码与真实需求的偏差就多累积一天；越早交付可运行的最小切片、越早用真实数据或真实用户验证，质量反馈就越及时——此时距离改动提交的时间点最近，上下文还在开发者脑子里热着，定位问题、修正问题的成本最低。反之，集中式大量交付会让问题堆积、错误叠加，排查时已记不清是哪次改动引入的，定位成本指数级上升。

这既是第 2 章探索环「精炼」步骤的真谛——把需求精炼到一次只验证一个假设的粒度，用 MVP、最小可行特性、特区法等手段切出最小可验证单元；也是第 3 章验证环「小批量」原则的操作纪律：一次只交付能验证那个假设的最小集合，让「改一小块代码 → 跑一次验证 → 拿一次反馈」的闭环转得足够快。验证环不是「等所有东西都做完再一起测」，而是把交付拆碎，把验证前移——这和第 10 章测试左移是一脉相承的。

验证环的原则——质量内建、消除等待、重复事务自动化、监测一切——在我参与的一站式发布、测试左移、流水线里都是具体零件。等待（等环境、等审批、等测试排队）是验证环的敌人；云平台、分级构建、个人流水线（第 7 章）都是为了压缩等待。

探索环与验证环不是「先探索完再验证」的一次性流水线，而是研发活动里循环交替：上一圈验证环产出的数据与灰度结论，应回到下一圈探索环，修正我们对用户需求的理解，再进入新一轮交付。CD 2.0 的双环，说的正是这种认知—交付—度量—再认知的转速；只加快验证、不闭合探索，就会陷入「发得越多，偏得越远」。

AI 可以生成用例、流水线配置、日志解读，但若方案评审、制品边界、灰度决策、埋点与效果复盘仍断裂，Harness 再强也只是验证环里的局部加速器——不是 CD 2.0 意义上的整环。

四、组织文化、度量与改善套路（第 4 章） #

一家企业的企业文化非常重要。我理解其中最核心的一条，可以概括成 Context, not Control（给上下文，而不是用控制欲替代思考）：首先要承认——员工是企业创新的源泉。这里的「员工」今天是以人为主体；往后团队里很可能还有 AI 成员，但逻辑不变：你想让有天分、有潜力的人（以及被人正确驾驭的 Agent）充分发挥主观能动性，就必须正视文化，而不是多买几张高性能GPU 或多招几轮优质校招就完事。

具体意味着什么？允许不同员工发出自己的声音、表达观点；在风险可控的范围内允许探索。一旦有能力的人开始觉得「我没有任何自主权」，他们就会停止主动做决策——组织慢慢变成一言堂：花了那么大代价招来的人，或花了那么大精力搭好的 AI 算力与平台，最终方案权和拍板权却攥在少数人手里。这是最傻的组织形态之一，企业的发展天花板会很快顶到。

乔梁老师在第 4 章写的安全与信任（失败是安全的）、相互信任、持续改善，以及会议讨论、问题复盘要从学习型组织出发——允许成员自由表达，就事论事：少站在「谁该为问题负责」的立场，多站在「如何解、如何优化」的立场。这样对犯错的人、发现问题的人、拿出方案的人都是好事：大家才不担心被惩罚、被孤立、被埋怨，主观能动性才能得到鼓励和释放。度量原则（度量是为了改善，不是 KPI 酷刑）和改善套路（Kata）要落在同一套文化上；EPC 被误读，多半是度量当鞭子、文化当海报，把「改善」做成了「问责」。书中引用的古德哈特定律——「当一项指标成为目标时，它就不再是好的指标」——恰好是这道伤口的病理学诊断。书中以谷歌（代码审查强制、开发自测、无独立测试团队）和 Etsy（无指责复盘、开发者自主上线、持续试验）为例，说明文化不是口号，是每天在做的事。

这和「一切皆可计算」不矛盾：计算的是趋势与瓶颈，用来开复盘、调流程，不是用单一数字否定人。而量化指标也不是最终目标，而是我们用来验证假设、进一步改善的牵引性的参考数据。

我有时会想 AI First 转型里的镜像场景：当 AI（或自动化门禁、静态扫描、测试报告）指出——你的代码结构差、连基本测试都没写——你还会把自己当成受害者吗？AI 并不是要来「攻击」你；若你仍用旧文化的防御姿态对抗反馈，你反而成了把事情做好的瓶颈。文化若不能让人（以及让人愿意采纳 Agent 的输出）把反馈当成改进素材，Assist 会停在补全，First 更无从谈起。

所以：给员工（给团队）更多信息，允许一定程度的发言权与探索自主权——用那点芝麻大的权力傲慢去「压人」，不是管理。

五、架构、需求与技术债（第 5–6 章） #

以下不追求与原文逐句对齐，而是第 5 章「大系统小做」、常见架构模式与改造手法，加上第 6 章里「技术债也是需求」在我自己做团队教练、做治理时的理解。

5.1 业务架构、团队架构、技术架构：三个维度 ##

有经验的工程师能分清：业务架构和技术架构不是同一维度；再往前一步，往往是 业务架构、团队架构、技术架构 三套东西叠在一起。

  • 业务架构、团队架构通常大家能直接感知：部门怎么梯队化、有几个大方向、各方向谁负责、下面有哪些职能小组——开会、汇报、考核都围着这套结构转。
  • 技术架构是真正交付出来的系统形态：怎么划分子系统、谁对哪块负责、边界上承诺什么能力。它在一定程度上折射业务与团队，但又不等于 org chart——错配时就会出现「组织拆散了，代码还耦在一起」或反过来。

架构师要持续看清这几层之间的连接与张力（康威定律不是背下来就算数，而是用来对账的）。书里的微内核、微服务、单体以及拆迁者、绞杀者、修缮者模式，本质都是在动技术边界时，顺便把团队能独立验证、独立发布的粒度对齐。

5.2 真实复杂度与「必然屎山」的借口 ##

业务变复杂，业务复杂度往往不可避免——这是真实世界塞进产品里的。但技术复杂度不然：有经验的架构师应当用设计把实现复杂度管住，而不是放任它在代码层、架构层无节制生长。

行业里常听到「代码屎山不可避免」「技术架构一定越来越复杂」——我更愿意把它理解成：很多人功力不够，管控不了复杂度，于是把失败说成命运。管控手段之一，就是架构演进：从单体到 SOA 再到微服务，书第 5 章写的正是这类路径上的模式与拆法。

5.3 「形」与「意」：跟风微服务与分布式单体 ##

单体 → SOA → 微服务，都是手段，不是勋章。不少人被动采纳，只因道听途说或跟风，缺乏判断力，分不清 形（服务切了几刀、注册中心上了没有）和 意（边界清晰、存储归属、接口契约、可演进）。

反面教材我见过不少所谓的「微服务」：各服务把存储暴露给别人读甚至写，读写绕开接口层，存储设计凌乱，重构牵一发动全身，事务一致性更是无从谈起——本质是分布式单体，只是代码、部署形态看上去像微服务。边界要能在接口与数据归属上说清楚，而不是在 PPT 上多画几个框。

5.4 技术债：能跑就行，是对团队最大的不负责任 ##

存量系统背着技术债是常态。糟糕的做法是「能跑就行」：不是我写的绝不主动改——好听点叫怕改出问题，难听点叫不愿碰模糊边界，没有持续重构、优化、减债、止损的意识。我认为这是对团队最大的不负责任。

技术债必须被当作要还的钱来排期（第 6 章把它写进需求体系是对的）。熵增不会因为你无视就暂停；负债总有一天会以更贵、更痛的形式冒出来——线上事故、招聘留不住人、AI 生成的补丁越贴越乱，都是同一种账。

我个人的用人偏好很直白：宁可要一个会持续重构、会主动碰边界的人（哪怕会犯错），也不要一个事不关己、高高挂起的人。 这和第 4 章的文化是咬合的：若复盘只会追责，没人会主动还债。

5.5 第 6 章补一笔：需求协作与技术债进 backlog ##

需求侧的 INVEST、拆分技法、故事地图、依赖图、「完成」的定义、故事验证——探索环做得好不好，一半看这里。需求拆不好，探索环再漂亮也只是 UI 稿。

AI 可以批量生成用户故事；识别依赖、界定 Done、把技术债翻译成可排期、可验收的条目，仍是资深工程师与架构师的隐形技能。Agent 写代码越快，边界与负债若无人管，验证环只是在加速问题堆积而已，苦日子在后面。

六、高质量落地：流水线、分支、CI 与测试（第 7–10 章） #

前面几章更像把大家从「蛮荒工程」领进软件工程的大门——让有多年经验的开发、管理重新思考：探索环、验证环、文化、架构、需求与技术债，哪些做得好、哪些一直在裸奔。第 7～10 章则落在每天在做、却常常没做好的实处：部署流水线、分支策略、持续集成、自动化测试。

6.1 质量不可妥协；效率必须站在质量之上 ##

有技术实力的架构师通常认同一句话：质量是不可妥协的要素，效率必须建立在质量之上——不是什么「为了效率可以让渡一点质量」。没有质量，就没有效率；没有质量的效率也没有意义。

这放在 AI 时代特别好懂：模型可以以人类写代码速度的几十倍、上百倍吐代码——若产物里塞满缺陷，你敢合进主干吗？敢灰度吗？问题不难，难的是组织里仍有人用错误判断给坏习惯买单。

6.2 两种不称职的 TL，和一种危险的口号 ##

我见过两类问题突出的 TL（见识窄的那类）：

  • 一种放任不管：代码规范、可读性、可测试性全无，能跑就行。
  • 另一种形式主义：CI 跑一跑规范扫描、安全扫描（有没有密码泄露之类），有问题也不设质量门禁——红灯照样合、照样发。还常挂一句：「最挣钱的业务代码一定是最烂的」——我认为大错特错。

首先，这类 TL 不称职：没有专业能力压住坏习惯带来的风险，还把规范、评审、测试当成花架子。真出了事，往往让写代码的人背锅，自己则躲的远远的，实际证明也往往确实是这样；在上级大领导追责之前，他们常常也不会被追责。其次，他们建不起像样的 CI：不懂快速质量反馈的意义——那既是为了项目的可维护性、健壮性，也是为了保护团队成员别因低级错误背事故锅；更谈不上对用户体验的敬畏。

6.3 一线研发与管理晋升：混乱的温床 ##

很多一线研发——尤其初级工程师——暂时没有 CI 意识、没有科学的分支习惯，尚可理解：没经历过事故，没被人用门禁「拦过一次」。

但若这种人升了管理，技术视野又窄，再搞信息墙、搞 PUA、要求下属按他的惯性思维办事，叠加下面几条盲区，终局往往是混乱：

  • 长时间不遵守Git分支规范，不懂超长生命周期分支的合入风险；
  • 不知道Git冲突解决是启发式的，并非 100% 可靠，多线修改合并的不良实践；
  • 不懂测试覆盖率的意义，不懂测试是资产——重构时的守门员；
  • 认为代码规范、可读性设计都是形式主义，认为能读懂代码、猜出垃圾隐式逻辑才是能力，而不是能写出可读性好、可维护性好的代码；

第 8 章的分支策略、第 9 章的六步提交与「视而不见的扫描问题」、第 10 章的测试分层与维护共担，都是在给这类混乱立规矩。不是教条，是避免用肉身去重复踩坑，然后再出卖体力修复，简直是“卖肉”行径。

6.4 第 7～10 章在工程里各自卡住什么 ##

结合书和平台实践，我把四块分工记成下面这张表（便于和日常对齐）：

章节在解决什么做不好时典型症状
第 7 章 流水线可重复构建、制品、环境、自助工具人肉发包、制品不可追溯、环境「雪花」
第 8 章 分支集成节奏与发布周期匹配长期分支、合并地狱、发布火车撞车
第 9 章 持续集成提交即集成、快速失败、分级构建CI 红灯习惯无视、集成晚于联调
第 10 章 自动化测试金字塔、左移、共担维护、质量内建无测试资产、上线前手工点点点

没有质量门禁的 CI，只是表演；没有测试的 CD，只是更快地上线事故。第 10.5.3 节提到 AI 在测试中的应用——今天可以用AI延伸到生成用例，但前提仍是：先有可维护的测试策略，否则 AI 放大的是垃圾用例的维护成本。比如测试是否正交、数量应该少而精、测试用例应该稳定，如果连这点基本的原则都不清楚，只是一味地要求提高代码测试覆盖率，那真是噩梦。

制品与需求的双向追溯（与第 7 章流水线一体） ##

严格说，制品怎么管、怎么串回需求，是流水线与 CI/CD 的延伸，我常和第 7～10 章放在一起想：CI 末尾产出制品，经检验（CI 冒烟或 CD 流水线冒烟）后，制品应打上完整标签——分支、构建时 Git commit SHA、构建人、描述里带上 commit message 等。理想状态下：

  • 制品 → 代码 → 需求：从制品信息能查到 commit 日志；commit message/footer 里带有 --story / --bug 等关键字，能对应到需求或缺陷。
  • 需求 → 代码 → 制品：反过来查某个需求进展——有没有对应提交、合入、MR、bugfix、是否已进入某次发布的制品。

探索环、验证环要闭合，离不开这条链。人肉发布时代几乎做不到；自动化之后若仍不打 tag、不写 footer，只是换了一种「不可追溯」。

6.5 环境准备、测试左移 / 右移、A0～A3 与性能门禁 ##

前面表格和第 10 章讲了「要测试」，但书里容易一笔带过的，是环境从哪来、测试在发布前还是发布后、测到哪一层、性能要不要挡在 CD 门口。这些我在治理期和项目里都有接触，和验证环、第 11 章环境管理相关。

环境：研发、测试、类生产 ##

至少要分清几类环境的职责，而不是「有一台测试机」就算有测试：

  • 研发环境：日常开发、调试，允许脏、允许频繁变更。
  • 测试 / 特性环境：承载基线或特性部署，供接口级、场景级验证；依赖可部分真实、部分mock。微服务框架层可能要支持多环境路由。
  • 正式测试 / 类生产环境：配置与数据规模尽量贴近生产，用于发布前最后一道集成、性能与回归——不是「正式服」，但要比研发机严肃得多。

环境准备本身应是 XaC（第 11 章）：装什么组件、什么版本、谁有权改变环境，都应可追溯。环境「雪花化」时，A2/A3 用例会 flaky，CI 绿灯也没有可信度。

测试左移：上线之前就把证据攒够 ##

测试左移指在发布上线之前完成尽可能多、尽可能自动的验证——单测、接口测试、集成测试、E2E 等。QA 可以提前介入用例与场景设计；开发也应主动写 A0/A1，而不是等「测试阶段」再补。这和第 3 章验证环里的「快速质量反馈」、第 10 章「测试活动左移」是一件事。

测试右移：上线之后仍在验证环里 ##

测试右移不是「上线就不管了」，而是在正式环境继续做受控验证，例如：

  • 混沌工程（故障注入、依赖延迟、实例摘除）——看系统在真实流量与真实拓扑下是否还站得住；
  • ABTest / 灰度（第 12～13 章）——用生产数据验证方案，属于探索环 + 验证环的延长线。

左移把缺陷成本压在合并前；右移把未知压在可控的生产实验里。二者对立统一，不是二选一。

A0～A3：从函数到场景的分层（我团队常用的说法） ##

在左移里，我们把自动化用例粗分为四层，便于决定跑在哪、依赖怎么处理：

层级测什么依赖与环境
A0函数、方法、类对程序的方法、类进行测试；只依赖被测试的代码；测试用例和被测服务同进程内进行；建议单用例执行耗时上限在 1s 以内（SDK 同样适用）
A1函数、方法、类（模块间集成）对程序的方法、类进行测试，注重服务内部模块间的集成验证；除依赖被测代码以外，还可以依赖本地资源（“本地"即访问网络为 localhost，包括本地数据库、本地文件系统、同进程 mock 或者本地测试替身等）；测试用例和被测服务同进程；建议单用例执行耗时上限在 3s 以内（SDK 同样适用）
A2单服务接口对服务接口进行测试；建议全部依赖真实链路，或部分依赖测试替身（但不鼓励）；测试用例和被测对象处于不同进程；需要部署启动被测服务；建议单用例执行耗时上限在 10s 以内
A3多服务、多接口串联对多服务多接口串联完成对某个业务场景的测试，主要是模拟客户端（如 app、H5、Web）调用，部署真实服务、全部依赖真实下游的全链路端到端测试；测试用例和被测对象均处于不同进程；建议单用例执行耗时在 1s 以上且控制在 10 分钟以内

A0/A1 适合压在 CI 前几段（快、稳）；A2 依赖环境供给与 Mock 策略；A3 更吃环境编排与数据准备，往往落在 CD 前期或类生产环境。书里的测试金字塔与此同构——层数不对、全挤在 A3，就会又慢又脆。

性能、负载、压力、稳定性：也应挡在「继续 CD」之前 ##

除了功能，上线前还应问：性能有没有回退？常见四类（名称在不同团队略有出入，语义如下）：

  1. 性能测试：以系统设计初期规划的性能指标为预期目标，对系统不断施加压力，验证系统在资源可接受范围内，是否能达到性能预期。
  2. 负载测试：对系统不断地增加并发请求以增加系统压力，直到系统的某项或多项性能指标达到安全临界值。当并发请求数量超过安全临界值之后，系统吞吐量不升反降。
  3. 压力测试：超过安全负载的情况下，对系统继续施加压力，直到系统崩溃或不能再处理任何请求，以此获得系统最大压力承受能力。
  4. 稳定性测试：被测试系统在特定硬件、软件、网络环境条件下，给系统加载一定业务压力，运行一段较长时间，以此检测系统是否稳定。

通过以上测试，我们期望获得三个关键点：系统最佳运行点、系统最大负载点、系统崩溃点——它们共同构成系统的性能画像，也是容量规划与弹性伸缩策略的基础输入。

理论上，这些应在 CI 末尾或 CD 中冒烟通过之后跑一轮（或采样对比基线）：无明显性能回退，才进入后续晋升、灰度、全量。把性能丢给「上线后看监控」，和把功能丢给「上线后用户报 bug」一样，是对验证环的偷工减料。

AI 可以生成 A0/A1 骨架用例、帮忙读性能报告，但环境契约、A2 Mock 边界、A3 场景数据、性能基线谁定——仍是团队资产，宜写进 Skill / 流水线配置，而不是每次让模型猜。

ps：有时会提到测试用例是否稳定（flaky tests），和这里的服务、系统稳定性不是一个概念。

6.6 AI、噪音与 Harness：烂摊子上加速，只会更乱 ##

现在很多团队背着 AI 提效的压力和指标。当 AI 介入一个本已混乱的项目时，大模型会吃进大量不良实践的噪音。LLM 用注意力机制在多个约束间加权，但无论怎么加权，劣质上下文都是噪声——发挥不及预期，反而是常态。

在基础模型能力到某个台阶之前，还要靠 Agent 兜底；Agent 也不可能覆盖所有坏习惯：你不分包分文件、函数冗长、圈复杂度夸张，Agent 得用尽花样才能绕过你的 bad case 捞到一点有效信息，还要担心上下文爆掉、幻觉、多轮交互——算力、时间、钱都烧在修修补补上。实践良好的仓库，往往用更少的成本就能让 AI 稳定交付。

很多人觉得上了 AI 就能收拾烂摊子——至少发展到现阶段，我的判断是：意识不到位时，AI 的交付速度只会加快混乱。所以才会天天谈 Harness engineering、天天想蒸馏高级工程师的 Skill：本质上是把「已经证明有效的实践」规范化，让人和 Agent 在同一条验证环上干活。如果你的团队工程素养很好、上述实践做的也比较好，你也不会因为AI焦虑，因为你已经做好了Harness工程领域里面最重要的细活部分。

若把四块映射到 Agent：流水线 = 可重复的 Tool 链；分支策略 = 变更与集成的边界；CI = 行动后的自动验真；测试 = 对 Agent 产出的守门员。Harness 调度的是已有纪律——纪律不存在，调度越快，塌得越快。

七、配置、发布与监测：让不可见者可见（第 11–13 章） #

第 7～10 章讲的是「每天怎么写、怎么并、怎么测」；第 11～13 章再上移一层：构建并交付过程中，所有隐式依赖、人肉步骤、审批与漂移，如何被收拢成可观测、可度量、可改进的对象。我在腾讯快十年了，刚来时仍经历过一段「蛮荒」——和书里的现代化流水线对照，印象特别深。

7.1 蛮荒发布：每个手动步骤都是在掷硬币 ##

刚来腾讯时，发一个服务往往是：自己在构建机上手动编译，再上传二进制，填发布说明、代码版本——全流程人肉。没有合适流程自动化时，每一个环节都是变量：编译器选项对吗？依赖版本对吗？描述和代码版本一致吗？这次发布真的对应某个需求吗？……每个手动操作至少隐含「对 / 错」两个维度，组合起来发布失败就很常见。

我见过的典型翻车包括：配置错了（忘了打进包、配置版本和二进制版本对不上）；代码里硬编码配置，线上不能靠 reload 救；配置修改不可追溯；上线后配置漂移——被人肉改过，和刚发布时不一致。这些痛，后来都推动我们去做制品、配置、环境的一体化治理——也就是第 11 章「将一切纳入配置管理」的现实来源。

7.2 配置不只是配置文件：Everything as Code（XaC） ##

今天说的「配置」，远不止是几个 .yaml：

  1. 配置与二进制分离（改配置不必重编译，且版本可对照）
  2. 业务逻辑配置（开关、策略、阈值）
  3. 组件配置（中间件、框架参数）
  4. 依赖配置（版本锁、传递依赖）
  5. 运行环境与组件安装、启动（机器上到底有什么）
  6. CI 流水线本身——工具链、构建依赖、阶段定义
  7. CD 流水线——部署策略、门禁、环境改变

业界把这类实践统称 X as Code（Infrastructure / Pipeline / Policy …）。第 11 章想强调的核心就一句：构建并交付里所有原本靠默契、靠人肉、靠「老张记得」的东西，都要有着落——可见、可观测、可度量、可改进。我后来在博客写配置漂移、优先级覆盖，也是同一条战线的延伸。

7.3 低风险发布：灰度、金丝雀、蓝绿与 A/B ##

第 12 章讨论如何在高频发布下控风险。结合业务场景，我常用的脑子里的分法：

先验证「方案对不对」（探索环）

  • 灰度发布：先让一小撮用户吃新版本——按尾号、区域、等级切流，或按 ABTest 标签 圈人。适合「特性已开发完，要看真实数据与反馈再放大」。
  • A/B 对比：几种实现并存，不同用户走不同方案，用数据比效果。强调的不只是「少宕机」，而是哪个方案更对。

再验证「发布过程稳不稳」（验证环）

  • 金丝雀（Canary）：先灰度少量实例，控制流量观察错误率、延迟、业务指标；没问题再滚动其余实例。戏剧性的名字，本质是「用生产流量做受控实验」——和探索环的「一次只验证一点」同构。
  • 蓝绿（Blue-Green）：两套完整环境，切流量是一次「大刀」——回滚快（切回旧环境），但资源双倍、数据库与有状态组件要单独设计，否则蓝绿只是表象。
方式典型优点典型代价 / 注意点
金丝雀 / 滚动资源省、贴近日常扩缩容；可渐进要观测与自动熔断；有状态服务要小心
蓝绿切换与回滚干脆双倍资源；数据层与缓存一致性
灰度 / AB（产品向）绑探索环指标埋点、分流、伦理与合规要想清楚

顶流项目发布频繁时，若只有「全量一键发」、没有开关与灰度，验证环在发布这一截是断的。功能开关、数据迁移、书里的抽象分支等，都是为了让「发」和「证」能拆开。

7.4 监测的两条线：产品验证与服务健康 ##

第 13 章的「监测与决策」，我拆成两条（都服务双环，但观众不同）：

产品 / 需求线（探索环的燃料） 埋点设计、实现、采集、分析——验证方案有没有解决问题、指标是否朝假设走。没有这条，验证环末尾的「决策」只能凭感觉。

服务 / 系统线（验证环的护栏） 对运行态本身：成功率、错误、延迟、队列长度、任务数、worker 线程、CPU、内存、事件、数据大小……告警治理、生产探测、混沌工程，都是为了让发布与变更不会悄悄把系统推过悬崖。书里说「向东还是向西」——用数据决策，而不是用会议室嗓门决策。

AI 能帮忙写 PromQL、总结日志、草拟埋点方案，但埋点定义权、发布是否继续、是否回滚，仍应在人（与组织文化）手里——否则只是把告警海洋换成「模型幻觉海洋」。

7.5 和 AI 的交界：这一层为什么更难被模型「一笔带过」 ##

配置漂移、制品追溯、灰度策略、双轨监测，恰恰是 Agent 最难自动猜对 的领域：它们依赖组织特有的命名规范、需求体系、监控基线与合规要求。Harness 和 Skill 在这里的价值，是把 XaC、发布剧本、观测面板 写清楚，减少上下文噪音。

这几章讲的，是「另一个层面」：不是再写一段代码，而是让交付全过程可复盘、可追责、可改进。

八、案例篇：FT 化与小团队逆袭（第 14–16 章） #

第 14 章大型互联网 FT（Feature Team）化：架构解耦、组织解耦、流程再造、自动化一切——是双环模型在大体量下的组织投影。

第 15 章小团队从「死亡行军」到无缺陷交付：故事墙、测试左移、代码评审、关注过程——说明 CD 2.0 不只属于巨头。

第 16 章研发推动的 DevOps：敏捷 101 到 DevOps 转型，运维冲突、高频发布障碍、整体方案设计——不是某一个工具 winner，而是目标、节奏、责任的再分配。

好的实践，好的沉淀，收获复利！

九、持续保持：第 17 章的六条必经之路 #

第 17 章「研发组织效能提升的必经之路」给出了持续保持高水准的框架，这里我还是说说我的观点吧。

  • 跨职能分享让不同角色之间互相学习。对探索环+验证环进行系统性经验分享、优化加速，而不只是某一个环节；
  • 深入一线量化，让不可见变可见。了解一线同学面临的挑战和问题，发现问题、量化问题、解决问题；
  • 立质量红线与门禁。无规矩不成方圆，思考团队成员之间的协作效率问题，长期看可维护性风险问题；
  • 持续重构、提升技术视野。保持学习心态，先别急着抵触，是骡子是马拉出来遛遛就知道了；
  • 周期性 review 与复盘改进。要让大家知道哪里做的好，也要知道哪里做的不好，知道不好还不够要给出方案；
  • 以及最重要的——疏导胜过强制，从宽松门禁起步，让大家先体会到价值再增量提高。

这些思考与书籍内容不完全一一对应，是我这些年在腾讯工作期间，对软件工程、工程素养以及在团队中把事情做成的一些很现实的思考。这六条对 AI 在引入也是同理，Agent 不是一次性部署完就完毕，而是持续的认知升级和流程适配。

十、从书的结构到 AI：Skill、Harness 与「隐形技能」 #

全书目录（从第 1 章软件工程概述到第 16 章 DevOps 转型，再到附录的三次进化与估算方法）构成一张完整的工程地图。重读之后我越来越确定：这些章节不是在讲「用什么工具」，而是在讲「怎么想、怎么管、怎么改进」——这些思考框架不会因为 AI 出现而过时，反而因为 AI 放大了执行速度，变得更加关键。

下面这张表不是在做生硬的「映射」，而是在追问同一个问题：每一条工程原则，到了 AI 时代为什么反而更重要了？

书中大方向核心工程原则AI 时代为什么更重要
价值探索环先锚定假设、定义成功指标，再动手交付模型能秒出方案和代码，但锚定错了，AI 只会更快地产出错误——探索环的提问和判断，仍然靠人
快速验证环小批量交付、快速反馈、质量内建Agent 生成速度远超人类，验证周期必须更短、更自动；没有质量门禁的 AI 加速，等于加速累积缺陷
文化与度量度量为了改善，信任与安全让人敢说真话AI 产出天然可度量（生成行数、采纳率），但「度量什么」的方向感不能外包；复盘文化决定了团队能从 AI 反馈中学习，还是把它当成甩锅对象
架构与需求边界清晰、职责明确、技术债进 backlog架构边界的清晰程度，直接决定 Agent 的 Tool 和 Skill 能否独立理解、独立验证；耦合严重的系统里，喂给模型的上下文全是噪音
流水线/分支/CI可重复构建、频繁集成、快速失败AI 工程的 Harness 编排的是团队已有的规范和纪律——规范本身不存在，编排越快越激进越危险，比如零自动化测试下的CI/CD
测试与配置测试是资产、一切皆代码、环境可追溯AI 生成用例和配置很快，但测试策略怎么定、环境契约怎么设计、基线谁来维护、特性环境如何搭建、mock如何支持——这些「慢功夫」仍靠团队长期投入，生成速度帮不上忙
发布与监测低风险发布、数据驱动决策、双轨监测灰度是否继续、是否回滚，本质是人对风险的判断和承担责任；模型可以提供数据摘要，不能替代拍板

这张表想说的其实是同一件事：原则没有变，变的是违反原则的代价。人肉开发时代，跳过探索环、忽视测试、容忍技术债，后果来得慢、看起来还能扛；AI 时代代码产出速度放大几十倍，同样的坏习惯会在更短时间内堆出更大的烂摊子。

公开圈里流行「只要 Skill 够多，Agent 就能做软件工程」。乔梁老师若写续集，大概会强调：Skill 是显性化了的实践片段；真正决定成败的，是探索环的判断、验证环的纪律、文化里的信任与度量上的诚实——这些 largely 仍是隐形技能，父传子、TL 传团队，写进 SKILL.md 也只是冰山一角。

我仍相信 AI First 会被实现，但路径是：Assist 阶段把重复事务自动化（书第 3 章原则）→ First 阶段把双环闭合交给人机共驾 → 专用 Agent 分工（探索辅助、验证执行、监测摘要）。不是单个超级 Agent 内置全书目录就完事。

十一、给同行者的几个实操建议 #

以下建议不是读后感清单，而是我自己在带团队、做治理、尝试 AI 工程落地过程中反复验证过的判断。每条都对应书中一个核心章节，如果你正在推动团队的 AI 转型，不妨对照着审视：

  1. 先量化瓶颈，再选 AI 工具。在你引入任何 AI 编码助手或 Agent 平台之前，先问一个笨问题：团队目前的交付瓶颈到底在哪？是需求澄清慢、集成痛苦、还是上线后回滚率高？没有这个答案，AI 工具就是在随机位置上加速——很可能加速的是你本来就不该做的事。度量的目的不是给团队打分，而是找到改善的起点。
  2. 把你的 CI 当成 Agent 产出的第一道守门员。如果你的 CI 现在是「红灯照样合」，Agent 生成的代码会毫无障碍地流入主干。先让门禁起作用——哪怕从最宽松的规则开始（编译通过、无明文密码、单测覆盖率不下降），再逐步收紧。门禁保护的不是「流程」，是写代码的人和接手维护的人。
  3. 一次只让 AI 帮你验证一个假设（第 2 章）。不要让 Agent 一次性生成整个功能模块然后直接上线——这不是 AI First，这是赌命。把需求切成可验证的切片，每切一片、生成一片、验证一片。探索环的锚定必须来自业务理解和用户数据，不能来自 prompt 炫技。
  4. 技术债不进 backlog，AI 生成的代码就是在屎上雕花。Agent 不会主动说「这坨代码该重构了」——它只会基于你喂给它的上下文继续生成风格一致的代码。如果存量代码本身质量差，AI 的补丁放大的是坏味道，不是生产力。技术债是需求，要排期、要验收、要在 AI 成倍放大它之前还掉。
  5. 培养人的判断力，比攒 Skill 文件更重要。Skill 可以拷贝，判断力不能。代码评审时为什么拒绝一个 AI 生成的方案、灰度数据摆在面前时是继续放量还是回滚、技术债还到什么程度算「够了」——这些决策靠的不是更多的 SKILL.md，而是团队里有人经历过、踩过坑、形成了品味。

理念像水，也许后续还应该写些 场景配方。

总结 #

重读《持续交付 2.0》，不是为了怀旧腾讯技术治理的岁月，而是为了在 AI 噪声里找回主轴：双环闭合、质量内建、度量为了改善、让不可见变得可见。我一直认可EPC的理念，但是对当时落地的僵硬过程仍感觉心有余悸。近期跟进AI过程中，尤其是如何促进组织从AI Assist向AI First模式转型，这个思考、实践过程中让我对如何更好地驾驭AI产生了很多思考。当我从0到1理解LLM如何训练的、如何推理的，AI Agent是如何实现的 …… 很自然地你回联想到不良的工程实践，会不会给AI First模式带来些许阻碍。于是，你会进一步探索Transformer架构中的多头注意力机制，以及不良工程实践产生的噪音给AI Agent + LLM本身带来的干扰，以及单纯靠SKILLS能否稳定地驾驭这种干扰。

如果你不关心技术细节，你也是很难思考到这个程度的，当然也很难得出一个令自己信服的判断。一切皆可计算，如果你的结果是否，那说明理解还不够，还没有深入到可计算、可量化的程度。我对AI的判断就是AI很强，但是如果你的团队在软件工程方面做得很差劲，想转型AI First组织会面临很多苦难；反之，则会走的比较顺利。

AI时代，所要求的那些工程隐中的隐性技能，并没有发生质的改变。有些大佬说将来的职业不会划分的像现在PM、研发、QA、运维等这么具体，可能是原型师、构建者、清理师、增长师、维护者，我觉得他们说的对，但是这并没有改变这里的判断，不管岗位职责如何设置，但是高质量高效率交付所要求的隐性技能依然是文中提及这些。
