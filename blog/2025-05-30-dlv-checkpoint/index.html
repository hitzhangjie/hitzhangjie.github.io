Delve checkpoint：调试时的存档与读档

Posted 2025-05-30 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

背景 #

调试复杂 bug 时，我们经常需要反复回到某个关键位置，尝试不同的执行路径来定位问题。传统做法是重新启动程序、重新设置断点、重新触发条件——这个机械性的重放过程既耗时又容易遗漏关键状态。Delve 的 checkpoint 功能正是为了解决这个问题而设计的：它允许你随时"存档"，并在需要时"读档"，回到之前的程序状态继续调试。

checkpoint 是什么 #

在 Delve 中，checkpoint 是一个调试时的"快照"功能：

  • 保存程序状态：你可以在某个时刻创建一个 checkpoint，相当于拍了一个快照，记录下此时程序的完整执行状态
  • 恢复到任意 checkpoint：之后你可以随时让程序回到某个 checkpoint 时的状态继续调试

这类似于游戏的"存档/读档"机制，或者编辑器的 undo/redo 操作。它的核心价值在于：避免重复的重启调试才能进行后续排查的方式，让你专注于真正需要探索的执行路径。

典型使用场景 #

调试逆向执行

在调试中走错方向是常有的事。比如你 step into 了一个不相关的函数，或者跳过了关键代码。有了 checkpoint，直接回到之前的检查点即可，无需重启进程、重新触发 bug。

多路径探索

在某个分支点创建 checkpoint 后，你可以：

  1. 先探索路径 A，观察程序行为
  2. 恢复到 checkpoint，再探索路径 B
  3. 反复对比两条路径的状态差异

这在排查条件分支相关的 bug 时特别有用。

底层依赖：录制/回放技术 #

checkpoint 功能并不是免费的。它的实现依赖于底层的**录制/回放（record/replay）**技术，例如 Mozilla rr。

原理是：

  • 程序的完整执行过程被录制下来（包括所有非确定性事件，如信号、系统调用返回值等）
  • checkpoint 本质上是在录制的时间线上标记一个位置
  • 恢复时，回放引擎从记录的初始状态重放到 checkpoint 标记的位置

如果没有录制支持（如普通的 live 调试模式），checkpoint 功能就不可用。 这也是为什么你在直接 dlv debug 时看不到 checkpoint 命令，而需要先通过 dlv record 或在 rr 环境下启动。

相关命令 #

在 Delve 命令行中，checkpoint 相关的命令有：

命令作用
checkpoint在当前位置创建一个新的 checkpoint
checkpoints列出所有已创建的 checkpoint
restart <id>恢复到指定 checkpoint 继续调试

总结 #

checkpoint 是 Delve 在录制/回放模式下提供的"快照"能力，能让调试者随时存档并恢复到任意历史状态。它的底层依赖是 rr 等录制/回放技术——程序执行被完整记录后，checkpoint 只是时间线上的一个索引。如果你还没有尝试过这种调试方式，下次遇到需要反复复现的复杂 bug 时，不妨试试 dlv record + checkpoint 的组合。
