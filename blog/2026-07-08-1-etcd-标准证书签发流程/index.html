标准 etcd 证书签发流程 #

本文档描述 etcd 集群 TLS 的标准证书签发与部署流程，可作为续签、轮换或新建集群时的操作参考。

密钥对关系概览 ##

etcd 集群使用 两套独立的 RSA 密钥对：CA 密钥对用于建立信任链并签发节点证书，节点密钥对只提供节点身份的公钥材料（私钥不下发到对端）。

密钥对 A（CA，仅证书管理机）              密钥对 B（etcd 节点）
──────────────────────────────            ──────────────────────
ca-key.pem（私钥）                         etcd-key.pem（私钥）
    │ 自签                                      │ 公钥写入 CSR
    ▼                                           ▼
ca.pem（CA 证书）                        etcd.csr
    │                                           │
    │  openssl x509 -req                        │
    │    -CA ca.pem                             │
    │    -CAkey ca-key.pem  ────────────────────┘
    │              │
    │              ▼  CA 签名
    │         etcd.pem（节点证书）
    │              │
    └──────────────┘  验签时：用 CA 证书（默认 ca.pem）验证 etcd.pem 的签名

要点：

  • etcd-key.pem 不负责签发 etcd.pem，只与证书中的公钥配对，用于 TLS 握手
  • 签 CA：ca-key.pem → 自签 → ca.pem（续签后可替换为新的 CA 证书文件）
  • 签节点证：ca.pem + ca-key.pem → 签发 → etcd.pem（证书内嵌的是节点公钥，来自 etcd-key.pem）
文件角色是否下发到 etcd 节点
ca-key.pemCA 私钥：自签 CA，并签发节点证书❌ 严禁下发，仅证书管理机持有
ca.pem（或续签后的新 CA 文件）CA 公钥证书：签发时作 -CA，运行时验签✅（用于验证对端）
etcd-key.pem节点私钥：与 etcd.pem 中公钥配对✅
etcd.pem节点证书：由 CA 签发，含节点公钥✅

1. 生成 CA（仅在证书管理机执行一次） ##

# 生成 CA 私钥
openssl genrsa -out ca-key.pem 2048

# 自签 CA 根证书
openssl req -x509 -new -nodes \
  -key ca-key.pem \
  -days 3650 \
  -out ca.pem \
  -subj "/C=CN/ST=shenzhen/L=shenzhen/O=etcd/OU=System/CN=etcd"

说明：

  • -nodes：私钥不以口令加密，便于自动化签约（生产环境可按策略加密保管）
  • -days 3650：CA 有效期约 10 年；到期前应续签（见下文）
  • -subj：Subject 中的 DN 会写入 CA 证书；建议 CA 与节点使用不同 CN（如 CN=etcd-ca vs CN=etcd），避免 OpenSSL 等工具误判为自签名（详见 VERIFICATION.md）

2. 生成节点证书（应由 CA 签发） ##

# 生成节点私钥
openssl genrsa -out etcd-key.pem 2048

# 生成 CSR
openssl req -new \
  -key etcd-key.pem \
  -out etcd.csr \
  -subj "/C=CN/ST=shenzhen/L=shenzhen/O=etcd/OU=System/CN=etcd"

# 由 CA 签发节点证书（标准做法）
openssl x509 -req \
  -in etcd.csr \
  -CA ca.pem \
  -CAkey ca-key.pem \
  -CAcreateserial \
  -days 36500 \
  -out etcd.pem

说明：

  • 节点证书必须由 CA 私钥签名；不要用 etcd-key.pem 自签节点证书
  • 生产环境通常还需在签发时加入 SAN（Subject Alternative Name）（IP / DNS），与集群成员地址一致；可用 -extfile / -extensions 传入扩展配置
  • -days 36500 为超长有效期示例；实际应按组织策略设定，并记录到期提醒

含 SAN 的签发示例 ##

# 扩展配置文件 openssl-san.cnf（示例）
cat > openssl-san.cnf <<'EOF'
[req]
distinguished_name = req_distinguished_name
req_extensions = v3_req
prompt = no

[req_distinguished_name]
C = CN
ST = shenzhen
L = shenzhen
O = etcd
OU = System
CN = etcd

[v3_req]
basicConstraints = CA:FALSE
keyUsage = digitalSignature, keyEncipherment
extendedKeyUsage = serverAuth, clientAuth
subjectAltName = @alt_names

[alt_names]
IP.1 = 127.0.0.1
# IP.2 = <节点 IP>
# DNS.1 = <节点主机名>
EOF

openssl req -new \
  -key etcd-key.pem \
  -out etcd.csr \
  -config openssl-san.cnf

openssl x509 -req \
  -in etcd.csr \
  -CA ca.pem \
  -CAkey ca-key.pem \
  -CAcreateserial \
  -days 36500 \
  -out etcd.pem \
  -extfile openssl-san.cnf \
  -extensions v3_req

3. 部署到节点 ##

文件是否下发到 etcd 节点
etcd.pem✅
etcd-key.pem✅（权限建议 600）
ca.pem（或续签后的新 CA 文件）✅（公钥，用于验证对端）
ca-key.pem❌ 严禁下发

etcd 侧典型配置（示例使用 ca.pem）：

client-transport-security:
  cert-file: /etc/etcd/ssl/etcd.pem
  key-file: /etc/etcd/ssl/etcd-key.pem
  trusted-ca-file: /etc/etcd/ssl/ca.pem
  client-cert-auth: true
  auto-tls: false
peer-transport-security:
  cert-file: /etc/etcd/ssl/etcd.pem
  key-file: /etc/etcd/ssl/etcd-key.pem
  trusted-ca-file: /etc/etcd/ssl/ca.pem
  client-cert-auth: true
  auto-tls: false
配置项作用
cert-file / key-file本节点 TLS 身份（出示 etcd.pem + etcd-key.pem）
trusted-ca-file验证对端证书是否由受信 CA 签发（client-cert-auth: true 时）

部署后建议用 etcdctl endpoint health（带 --cacert / --cert / --key）做连通性验收，不要单独依赖 openssl verify（见 VERIFICATION.md）。

4. CA 证书续签 ##

当 CA 证书临近到期时，使用 同一把 ca-key.pem 续签，生成有效期更长的新 CA 证书：

# 将现有 CA 证书转为 CSR（变量名 new-server.csr 易误导，实为 CA 的 CSR）
openssl x509 -x509toreq -in ca.pem -signkey ca-key.pem -out new-server.csr

# 用 CA 私钥签发续签后的 CA 证书
openssl x509 -req -days 36500 -in new-server.csr -signkey ca-key.pem -out new-ca.pem

要点：

  • 续签后新旧 CA 证书（如 ca.pem 与 new-ca.pem）公钥相同，均为同一 CA 密钥对
  • 用同一 CA 私钥签发的历史节点证书，在信任续签后的 CA 文件后仍可验签通过
  • 轮换时需同步更新各节点及依赖服务的 trusted-ca-file（或等价 CA 信任文件）

下发命名注意 ##

部分服务配置中 CA 路径可能写死为 ca.pem。为降低变更影响，建议在各节点统一使用约定文件名（例如统一落地为 ca.pem），并确保内容是当前生效的 CA 证书。

5. 流程检查清单 ##

  • ca-key.pem 仅在证书管理机，未出现在 etcd 节点
  • 节点证书由 CA 签发（非自签），且含所需 SAN
  • etcd.pem 与 etcd-key.pem 公钥模数一致
  • 各节点已下发：etcd.pem、etcd-key.pem、CA 公钥证书
  • etcd trusted-ca-file 指向实际使用的 CA 文件
  • etcdctl endpoint health（mTLS）验收通过
