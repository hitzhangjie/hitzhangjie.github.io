Delve 的 I/O 重定向机制：分离调试器与被调试进程的标准流

Posted 2025-05-23 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

Delve 作为交互式调试器，自身要用 stdin/stdout/stderr 和用户说话——读命令、打变量、显示调用栈。被调试的 target（tracee）同样可能读写标准流。两边若挂在同一条终端链路上，就会出现两个问题：

  1. 输入归谁：这行字是给 (dlv) 的，还是给程序里的 Scan / scanf？
  2. 输出怎么分：程序的 Println 和调试器的 prompt 混在同一窗口，自动化场景里几乎无法解析。

很多人第一反应是：那 Delve 必须把 target 的 IO 全部重定向走。其实默认并不是这样——不指定 -r / --tty 时，target 常常和 dlv 共享同一终端。重定向是可选的隔离手段，在手工调试时一般用不到，在自动化调试时几乎刚需。

默认共享终端时，为什么多数时候“不太乱” #

同一次终端会话里，shell、dlv、以及 dlv 拉起的 target，默认往往指向同一个 PTY slave。进程各自有 fd 表，但 0/1/2 背后的设备可以是同一个 /dev/pts/N。

PTY、前台进程组、后台读终端收到 SIGTTIN 等细节，见 终端、TTY、PTY、Console：把这些概念一次理清 中「同一会话里：谁共享 PTY slave？」一节。这里只留和调试相关的结论：

  • 输出：多个进程写同一 slave，字节在窗口里会交错——这是真共享，不是显示 bug。
  • 输入：内核把键盘交给当前前台进程组；不是“每个进程各有一份 stdin 副本”。
  • 调试时 dlv 与 target 对终端的使用，多数是时分交错，而不是长时间并行抢读：
停在断点 / 停在 (dlv) 提示符
  → 主要是调试器在读命令、打输出

continue / step 跑起来
  → target 在执行，可能读输入、写输出

再断下
  → 又回到调试器

所以手动调试时，你敲 next 的时候 target 往往停着；target 在 fmt.Scanln 等输入时，你也很少同时在敲 Delve 命令。交替执行掩盖了共享终端的本质矛盾，体验上“大多时候还行”。

一旦出现下面任一情况，共享就会刺手：

  • target 在跑的同时你还想稳定地给 dlv 下命令（或反过来）
  • 两边输出缠在一起，人眼都难拆，脚本更难拆
  • 自动化 / AI agent 驱动调试：必须事先知道「这路字节是调试协议」还是「目标程序的业务 IO」

三种布局：默认、文件重定向、独立 TTY #

布局行为适用
默认（不传-r / --tty）target 与 dlv 共享（或继承）当前终端流；跑起来时 Delve 还可能配合 job control / foreground 把终端交给 target交互式人手调试，能接受交错
-r / --redirect只改 target 的 stdin/stdout/stderr 到文件；dlv 自己仍用当前终端做调试会话固定输入、落盘输出、脚本/agent 要解析两侧 IO
--tty <path>把 target 的三个标准流接到另一个终端设备，并可Setsid + 设 controlling TTYtarget 依赖真 TTY（isatty、全屏 TUI 等），又不能和 dlv 抢同一窗口

官方说明写得很直白：-r 与 --tty 控制的是 target process 的标准文件描述符，不是给 Delve 自己换一套 shell 重定向。

实现上，native 后端在 exec 前设置 process.Stdin/Stdout/Stderr：指定了路径就打开文件；foreground 且未指定 stdin 时 stdin 才退回 os.Stdin，stdout/stderr 默认也可落到 os.Stdout / os.Stderr。--tty 路径上则打开该设备并挂到 target 的 0/1/2。

默认：
  [你的终端] ←→ dlv 与 target 的 0/1/2（共享同一 PTY slave）

-r：
  [你的终端] ←→ 仅 dlv
  target: 0→input.txt  1→output.txt  2→error.txt

--tty /dev/pts/X：
  [你的终端] ←→ dlv
  [/dev/pts/X] ←→ target 的 0/1/2

重定向的格式 #

stdin:<filename>   → target 的标准输入来自文件
stdout:<filename>  → target 的标准输出写入文件
stderr:<filename>  → target 的标准错误写入文件

省略前缀时默认是 stdin：

# stdin ← input.txt，stdout → output.txt
dlv debug --redirect stdin:input.txt --redirect stdout:output.txt

# 省略前缀，等价于 stdin:input.txt
dlv debug --redirect input.txt

restart 时也可以改 argv 和重定向（CLI 里还支持类似 shell 的 <in >out 2>err 写法，见 dlv help / CLI 文档）。

从测试用例看解析细节 #

解析结果固定为 [stdin, stdout, stderr] 三元组，与命令行参数顺序无关。

简单场景 ##

// 单个文件 → 默认 stdin
[]string{"one.txt"},
[3]string{"one.txt", "", ""},  // [stdin, stdout, stderr]

多重重定向 ##

[]string{"stdout:one.txt", "stderr:two.txt", "stdin:three.txt"},
[3]string{"three.txt", "one.txt", "two.txt"},

混合模式 ##

[]string{"stdout:one.txt", "stderr:two.txt", "three.txt"},
[3]string{"three.txt", "one.txt", "two.txt"},
// three.txt 无前缀 → stdin

错误处理 ##

[]string{"one.txt", "two.txt"},
[3]string{},
"redirect error: stdin redirected twice",

同一标准流不能重定向两次，解析阶段直接报错。

自动化调试：为什么必须能把 IO 拆开 #

人手调试可以靠“停 / 跑交替”混用一个终端；自动化不行。

无论是脚本、CI，还是 AI agent 调 Delve，控制面通常需要：

  1. 稳定地向调试器下发命令、解析调试器输出（命中断点、变量值、错误信息）
  2. 独立地给 target 准备 stdin、收集 stdout/stderr（业务输入、程序日志、断言输出）

若两边仍共享同一 PTY：

  • agent 发给“调试会话”的文本可能被 target 的 Read 吃掉
  • target 的业务输出会插进调试协议流，解析状态机极易炸

这时组合使用：

  • -r：把 target 的标准流落到文件（或由上层再包装成管道），输入可预置、输出可事后 diff
  • --tty：target 需要真实终端语义时，单独开一个 pts，与 dlv 的控制终端分开

两者改的都是 target 侧；Delve 自己的交互通道保持干净。这是文件重定向看起来“土”、却在自动化里非常管用的原因。

ps: Starlark 等脚本能力解决的是「调试步骤怎么编排」；-r / --tty 解决的是「字节流属于谁」。自动化调试里这两层经常一起用。

总结 #

  • 默认：dlv 与 target 常共享终端；交互调试多靠停跑交替，所以多数时候不显得乱。
  • -r / --redirect：改的是 target 的 0/1/2 到文件，用来和调试会话彻底拆流。
  • --tty：给 target 单独终端设备，适合依赖 TTY 的程序。
  • 自动化 / agent：必须能区分“调试器 IO”和“目标进程 IO”，redirect 与独立 tty 是底座，不是锦上添花。

共享为何会发生、前台进程组如何约束读端，原理在 TTY/PTY 一文；本文只关心 Delve 在这套模型上提供了哪些旋钮。
