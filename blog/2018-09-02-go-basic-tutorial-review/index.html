Tutorialspoint：Go核心特性 & 与C/C++关键差异

Posted 2018-09-02 12:00 +0800 by ZhangJie ‐ 11 min read

分享:  

说明：早年刚接触 Go 时，曾对照 Tutorialspoint Go Tutorial 整理过一批学习笔记。原材料偏照搬、且已过时不全。本文不再逐页复制，而是按该站最新整理的内容进行更系统性总结：每节用尽量短的文字概括要点，并附 与 C/C++ 的对比，方便有 C/C++ 背景时快速捡起细节。

整理基准时间可视为 2018-09-02 前后的学习阶段，但实际包含了后续年份版本的内容，算是为了纪念当时的学习历程；文中个别处会标注后续语言演进（如 Generics）。

一、入门与语言基础 #

Go - Overview ##

要点：Go 由 Google 在 2007 年由 Robert Griesemer、Rob Pike、Ken Thompson 设计，2009 年公开。静态强类型、自带 GC、原生并发、包管理依赖、传统编译链接生成静态二进制。刻意省略：类型继承、方法/运算符重载、包循环依赖、指针运算、断言；早期材料还写“无泛型”，Go 1.18 起已有 Generics。

vs C/C++：

维度GoC/C++
内存GC 自动回收手动 / RAII / smart pointer
并发语言级 goroutine + channel库级 pthread / std::thread
类型系统接口 + 组合；无类继承C 无 OOP；C++ 多重继承等
产物默认静态链接、部署简单动态库/ABI 问题更常见

Go - Environment Setup ##

要点：安装 Go 工具链，配置 GOROOT（安装根）、GOPATH（工作区，旧工作流）或 module 模式（go.mod，现代默认）。go run / go build / go install 覆盖编译运行；go fmt 统一格式。

vs C/C++：类似装 gcc/clang 与设置 PATH。Go 工具链一体（编译、测试、依赖、格式化），不像 C/C++ 常拆成编译器 + CMake + 包管理器多套。Module 对标 vcpkg/conan，但更“官方默认”。

Go - Program Structure ##

要点：源文件 .go；必须有 package 声明；可执行程序 package main + func main()；import 引入包；语句块用 {}。

vs C/C++：package main + main ≈ int main()。Go 无头文件，导入的是包而不是声明文件；可见性靠标识符首字母大小写，不是 static/ extern / 头文件暴露。

Go - Basic Syntax ##

要点：

  • 语句末尾可不写分号（编译器自动插入）
  • 标识符：Unicode 字母/数字/_，首字母大小写决定导出
  • 注释：// 与 /* */
  • 花括号风格：if/for 的 { 必须与条件同行（强制 gofmt 风格）

vs C/C++：C/C++ 分号强制、大括号位置自由。Go 无预处理器（无 #define/#include 宏魔法）；导出规则类似“默认 private，大写 public”，比 public:/private: / friend 更粗粒度。

Go - Data Types ##

要点 常用类型：

  • 布尔：bool（true/false）
  • 整数：int/uint（平台相关）、int8…int64、uint8…uint64、byte(=uint8)、rune(=int32，Unicode 码点)
  • 浮点：float32/float64
  • 复数：complex64/complex128
  • 字符串：string（不可变字节序列，语义上 UTF-8）
  • 派生：数组、切片、结构体、指针、函数、接口、map、channel

vs C/C++：

  • Go 的 int 随架构变宽，类似 C 的 int/long 平台相关；需要固定宽度用 int32/int64。
  • 无隐式数值类型转换：int 与 int32 也不能直接混算，必须显式转换（比 C 严格得多）。
  • string 是一等类型且不可变；C 是 char*/std::string。
  • rune 对标 Unicode code point，不是 C 的 char。

Go - Variables ##

要点：

var a int = 10
var b = 10          // 类型推断
c := 10             // 短声明，函数内
var d, e int        // 零值初始化

零值：数值 0、bool false、string “"、指针/切片/map/channel/函数/接口为 nil。

vs C/C++：C 局部变量未初始化是未定义行为；Go 永远有零值。短声明 := 类似 C++ auto，但只能在函数内。多返回值赋值是 Go 常态，C 需结构体或指针出参。

Go - Constants ##

要点：const 声明；可用 iota 生成枚举式递增常量；常量在编译期确定；无类型常量（untyped const）在赋值时更灵活。

const Pi = 3.14
const (
    A = iota // 0
    B        // 1
    C        // 2
)

vs C/C++：类似 const / enum / #define。iota 比 C 的 enum 更灵活（可跳步、表达式）。Go 没有 C++ 那种对象级 constexpr 方法体系，但常量表达式足够覆盖枚举与配置。

Go - Identifiers ##

要点：命名规则与导出：Name 包外可见，name 包内私有。预声明标识符：true/false/nil/append/len/make/new/panic/recover 等（不是关键字，可被遮蔽，但不建议）。

vs C/C++：C 可见性靠翻译单元 + static / extern；C++ 有命名空间、 public/private/friend。Go 用大小写 + 包边界，规则更简单也更粗。

Go - Keywords ##

要点：约 25 个关键字，例如：break case chan const continue default defer else fallthrough for func go goto if import interface map package range return select struct switch type var。没有 while/class/try/catch。

vs C/C++：C/C++ 关键字更多（尤其 C++）。Go 刻意少关键字；go/defer/chan/select/fallthrough 是特色。

二、Go Operators #

Go - Operators（总览） ##

运算符分类：算术、关系、逻辑、位运算、赋值、杂项（&/* 取址与解引用）。无三元运算符 ?:。

vs C/C++：大体同源，但去掉 ?:，++/-- 只能作语句不能作表达式，且只有后缀。

Go - Arithmetic Operators ##

要点：+ - * / %，以及语句形式 ++ / --。整数除法截断；% 仅整数。

vs C/C++：几乎相同。差异：

  • Go：i++ 是语句，不能写 a = i++ 或 ++i
  • 无逗号运算符用于表达式序列（赋值里的多值是另一套语法）

Go - Assignment Operators ##

要点：= += -= *= /= %= <<= >>= &= |= ^=。另有短变量声明 :=（声明+赋值）。

vs C/C++：复合赋值一致。Go 额外有 :=；没有 C 的逗号表达式赋值技巧。多赋值 a, b = b, a 可直接交换，无需临时变量语法糖层面的折腾。

Go - Relational Operators ##

要点：== != < <= > >=，结果为 bool。可比较类型需满足语言规则（如切片/map/函数不能直接 ==，除与 nil 比）。

vs C/C++：C 中关系运算在 int 语境下常得 0/1；Go 严格是 bool，不能写 if (a == b == c) 这种链式坑（本身也非法）。结构体可比较当且仅当所有字段可比较。

Go - Logical Operators ##

要点：&& || !，短路求值。操作数必须是 bool。

vs C/C++：C 允许任意标量当真假；Go 不允许 if (ptr) 或 if (n)，必须 if ptr != nil / if n != 0。

Go - Bitwise Operators ##

要点：& | ^ << >>，以及 AND NOT &^（位清除）。^ 作一元是按位取反。

vs C/C++：C 用 ~ 取反，Go 用一元 ^。&^ 是 Go 特有写法（C 里通常 a & ~b）。移位时类型与溢出规则更明确，无 C 那种有符号右移实现定义的老坑那么多（仍要注意类型宽度）。

Go - Miscellaneous Operators ##

要点：

  • &v：取地址
  • *p：解引用
  • 没有 sizeof 运算符（用 unsafe.Sizeof 或 reflect）
  • 没有 ?:

vs C/C++：取址/解引用同形。C 的 sizeof、->、.* 等在 Go 中：结构体字段用 .（指针会自动解引用，p.Field 即可，不必 p->Field）。

Go - Operators Precedence ##

要点：优先级大致：一元 > 乘除模 > 加减 > 移位 > 比较 > ==/!= > & > ^ > | > && > ||。拿不准就加括号。

vs C/C++：整体类似，细节表略有不同；Go 无逗号运算符、无 ?:，表更短。实践上两边都建议少炫优先级。

三、Go Decision Making #

Go - Decision Making（总览） ##

分支：if / if-else / 嵌套 if / switch；另有面向 channel 的 select（常归并发，但教程放在决策类）。

vs C/C++：不像c/c++中那样“非零即真”；条件必须是 bool。

Go - If Statement ##

if x > 0 {
    // ...
}
// 可带简单语句
if v := f(); v > 0 {
    // v 作用域限于 if-else 链
}

vs C/C++：条件无括号（if x > 0 不是 if (x > 0)）；{ 强制。init 语句类似 C++17 if 带初始化，但 Go 更早就有。

Go - If Else Statement / Nested If ##

要点：else if 链式；嵌套同其他语言。else 必须与前一个 } 同行（gofmt 规则）。

vs C/C++：语义相同；风格强制更严，减少 “dangling else” 争议写法。

Go - Switch Statement ##

要点：

  • case 自动 break，需要贯穿用 fallthrough
  • case 可以是表达式，不一定是常量
  • 可无条件：switch { case x > 0: ... } 替代长 if-else
  • 类型 switch：switch v := x.(type)

vs C/C++：C/C++ 默认贯穿、易漏 break；Go 相反。Go switch 更强（表达式 case、无表达式 switch、type switch）。C++20 仍无 type switch 这般直接。

Go - Select Statement ##

要点：在多个 channel 发送/接收上等待，类似“channel 版 switch”。可 default 实现非阻塞；随机选择多个就绪 case。

vs C/C++：无语言级对等物。大致类似多路 IO 的 select/poll/epoll，或手写条件变量组合；Go 把 CSP 风格通信做成语法。

四、Go Control Flow Statements #

Go - For Loop ##

要点：Go 只有 for，扛下 while / do-while / for：

for i := 0; i < n; i++ { }
for cond { }          // while
for { }               // 无限循环
for i, v := range xs { }

vs C/C++：无 while/do-while。range 对标 range-for，但还能遍历 map、channel、字符串（得 rune）。

Go - Nested for Loops ##

要点：多重循环普通嵌套；可用 label 配合 break/continue 跳出外层。

vs C/C++：相同。C 也可用 label；日常更少用。

Go - Break Statement ##

要点：跳出最内层 for/switch/select；break Label 可以跳出外层。注意for内部嵌套switch+break的情况，break不带label只能跳出当前switch-case。

vs C/C++：同。注意 Go 的 switch 自带 break，显式 break 多用于 for。

Go - Continue Statement ##

要点：进入下一轮循环；可 continue Label。

vs C/C++：同。

Go - Goto Statement ##

要点：goto Label 跳到同函数内标签；不能跨越变量声明导致未定义等限制（编译器会拦）。linux内核有大量goto用法，仅限于往前跳、清理工作。

vs C/C++：都有 goto，都不鼓励滥用。Go 的 goto 约束更严，场景更少（错误处理多返回或 defer，而不是 goto cleanup——尽管早期也有人用）。

五、Go Functions #

Go - Functions ##

func add(a, b int) int { return a + b }
func div(a, b int) (int, error) { ... }     // 多返回值
func split(sum int) (x, y int) { y = sum - x; return } // 命名返回值

vs C/C++：多返回值是一等公民，C 靠出参指针、C++ 靠 std::tuple/struct。无默认参数、无函数重载、无可变模板那套（有 ...T 的变参 ...interface{}/...T 泛型变参）。

Go - Call by Value ##

要点：参数默认值拷贝。传 slice/map/channel 时拷贝的是描述符（底层数据仍共享），看起来像“引用语义”。

vs C/C++：同 C 默认按值。C++ 有引用参数 &；Go 要改调用方可见对象，用指针或本身是引用类型的描述符。

Go - Call by Reference ##

要点：通过指针参数修改：func f(p *T)。没有 C++ 那种“引用类型参数”语法。

vs C/C++：*T ≈ C 指针参数；没有 T&。数组在 Go 中是值类型（整表拷贝），和 C 数组退化成指针不同——Go 数组传参要小心，切片才是常用方式。

Go - Functions as Values ##

要点：函数是一等值，可赋给变量、作参数/返回值。

vs C/C++：C 函数指针；C++ 函数指针/std::function/lambda。Go 更干净，无函数指针类型语法噪音。

Go - Function Closure ##

要点：闭包捕获外部变量（按引用捕获变量本身）。循环里起 goroutine 时经典坑：捕获的是同一循环变量（Go 1.22 起循环变量语义有改进，旧代码仍需留意）。严格来说，Go编译器会检查被捕获变量是否有读写操作，如果只读且被捕获变量size不大就是值捕获方式；如果有写操作或者虽然为读但是被捕获变量size过大，就使用引用捕获。关于闭包实现的更多细节，可以参考我这里的总结 Go闭包实现。

vs C/C++：C 无闭包；C++ lambda 可 [=]/[&] 明确捕获方式。Go 无捕获列表，语义接近“始终按引用捕获变量”。

Go - Function Method ##

要点：方法 = 带 receiver 的函数：

func (p Point) Dist() float64 { ... }   // 值接收者
func (p *Point) Move(dx float64) { ... } // 指针接收者

不支持类，方法可定义在同一包内任意命名类型上（非指针/接口等限制类型除外）。

要特别注意：

  • nil receiver，允许nil，但是访问字段时会panic。有些语言对nil类型的字段访问时的处理进行了优化，比如zig。
  • 值receiver字段赋值问题，可能误认为字段赋值了实际上无效。rust 会阻止非mutable字段赋值。 例子：ins.Method(...)，实际上是Method(ins, ...)，而这里的ins是一份struct value copy，字段赋值并不会影响到原来的ins。

vs C/C++：C 无方法；C++ 成员函数在类内。Go 的 receiver 显式，值/指针语义更直观；无 this 隐式，无虚函数表那套（接口动态派发另说）。

Go - Anonymous function ##

要点：func(x int) int { return x+1 }(3) 或赋给变量。常与 goroutine、defer、回调一起用。

vs C/C++：对标 C++ lambda；C 无原生匿名函数（GCC 嵌套函数属扩展）。

六、Go Strings #

Go - Strings ##

要点：string 不可变；底层为只读字节序列，习惯 UTF-8。字面量 "..." 或反引号原始字符串 `...`。遍历：

  • for i := 0; i < len(s); i++ → byte
  • for i, r := range s → rune + 字节索引

remark：Go 为什么约定 string 为 UTF-8？因为 Go 源文件必须是 UTF-8 编码，源文件中的字符串字面量自然也是 UTF-8 编码存储的。编译时这些 UTF-8 字节数据直接写入二进制，运行时加载到进程内存中仍保持 UTF-8 编码。因此 Go 的 string 在语义上就是一段 UTF-8 编码的字节序列。这也解释了为什么 len(s) 返回字节数而非字符数，以及 for range 能按 rune（Unicode 码点）解码遍历——它底层就是在解码 UTF-8。

vs C/C++：不是以 \0 结尾的 C 字符串；更接近 std::string 的值语义，但不可变（修改需造新串）。无隐式 encoding 假设之外的 wchar 混乱——惯例 UTF-8。

Go - String Length ##

要点：len(s) 是字节数；Unicode 字符数用 utf8.RuneCountInString(s) 或 range 计数。

vs C/C++：strlen 计到 \0 的字节；std::string::size() 也是字节/码元层面。多字节字符同样要专门 API。

Go - String Concatenation ##

要点：+ 连接；循环内大量拼接用 strings.Builder（或 bytes.Buffer）。fmt.Sprintf 做格式化拼接。

vs C/C++：+ 像 std::string 拼接；C 用 strcat 易溢出。Builder 对标反复 append/reserve。

Go - Compare Strings ##

要点：==/!=/< 等按字节序；strings.Compare/EqualFold（大小写不敏感）等在 strings 包。

vs C/C++：strcmp / std::string::compare。注意本地化/语言敏感排序两边都不是默认行为。

Go - Split String ##

要点：strings.Split / SplitN / Fields（空白分割）。

vs C/C++：C 的 strtok（有状态、非线程友好）；C++ 需手写或用第三方。Go 标准库更直接。

Go - Substring Extraction ##

要点：切片语法 s[i:j] 得到子串（仍共享底层字节，因不可变所以安全）。按 rune 截取需先转 []rune。

vs C/C++：类似 substr，但按字节下标；多字节字符边界要自己处理。

Go - String Replacement ##

要点：strings.Replace / ReplaceAll / Replacer；正则用 regexp。

vs C/C++：无统一标准一句话 API（C 更痛苦）；逻辑类似。

Go - String Interpolation ##

要点：无运行时字符串插值语法糖（至少经典 Go）；用 fmt.Sprintf("a=%d", x) 或 Go 1.22+ 的 fmt.Appendf 等。模板见 text/template。

vs C/C++：C 的 printf；C++20 std::format。Go 的 fmt 动词与 printf 亲缘近（%v %T 等是扩展）。

Go - Parse Date Strings ##

要点：time.Parse(layout, value)，layout 用参考时间 Mon Jan 2 15:04:05 MST 2006 的记忆法，而不是 YYYY-mm-dd 这种占位符。

vs C/C++：strptime / std::get_time 用 %Y-%m-%d。Go 的 layout 是最大记忆点差异之一。

七、Go Arrays #

Go - Arrays ##

要点：var a [N]T，长度是类型一部分：[3]int 与 [4]int 不同类型。值语义，赋值/传参拷贝整个数组。

vs C/C++：C 数组传参退化为指针；Go 数组是真值类型（更像 C++ std::array）。日常更常用 slice。

这点需要注意，很多C/C++开发者迁移到Go后，初期会以为数组是引用类型，Go里面是值类型，传参copy整个数组。

另外在C/C++中常有变长数组的写法，如:

struct XX {
  int len;
  char dat[0];
}

XX xx;
xx.len = 1024;
xx.dat = malloc(sizeof(char), 1024)

Go总这样写也是不行的，因为数组维度是类型的一部分，所以也不能写：

type XX struct { int len dat [0]byte }

var xx XX xx.len = 1024 xx.dat = [1024]byte{….} // WRONG, mismatched datatypes // cannot use [1024]byte{} (value of type [1024]byte) as [0]byte value in assignment

Go - Multidimensional Arrays ##

要点：[3][4]int 等；字面量嵌套初始化。

vs C/C++：同形。Go 没有 VLA（C99 可变长数组）。

Go - Passing Arrays to Functions ##

要点：按值拷贝；要避免拷贝可传 *[N]T，或更常见传 slice。

vs C/C++：C 实际传指针；Go 若写 func f(a [1000]int) 真会拷贝 1000 个元素——这是从 C 转过来最容易踩的坑之一。

是的！有些开发会不解，实际上这样更加简单：Go 将数组设计为 值类型 （Value Type），主要是为了保证 内存连续性、安全隔离与确定性的垃圾回收 。在 Go 中，数组是一个大小固定、分配在连续内存块上的底层实体，这使得它可以直接嵌入到结构体中以大幅提升 CPU 缓存命中率，且小数组能直接高效地分配在栈上。作为值类型，数组在赋值或函数传递时会进行全量数据复制，实现了天然的并发数据隔离，杜绝了隐式指针修改带来的副作用；同时，这也赋予了数组直接使用 == 进行比较的能力，使其可以作为 Map 的键。这种低级的、完全掌控内存的底层设计，也完美地为上层更常用的动态引用结构——切片（Slice）——奠定了坚实的存储基础。

八、Go Pointers #

Go - Pointers ##

要点：var p *T = &v；*p 读写。无指针运算（不能 p++ 扫数组）。new(T) 分配并返回 *T（零值）。

vs C/C++：有指针、无运算 → 更安全，也不能做 C 那套地址把戏。无 ->，用 .。垃圾回收下指针不能随意转整数乱跳（unsafe 除外）。

为了确保”pointer值“不会变成野指针，有时候可能会将pointer和uint64之间转换，如何避免uint64指针值因为无效（如对象被GC掉了)的呢？Go定义了一些unsafe规则，在unsafe规则声明下的操作，是可以避免这个问题的；但是肆意跳出这个规则的，就没有任何保证。 read more: github.com/golang/go/blob/master/src/unsafe/unsafe.go?name=release ，unsafe包没有Go1兼容性承诺，当尝试将uintptr转换为unsafe.Pointer时，有几条特殊的case，编译器会特殊注意，确保转换时不会出现问题，但是其他cases就没这个保证，要注意避免。 简单说，Go提供了指针，但是不支持指针运算。Go提供了unsafe.Pointer和uintptr，允许你打印uintptr，并且允许极少数场景下的uintptr像unsafe.Pointer转换时不出错。uintptr本质上是一个值类型，而不是一个对象引用。GC的时候也不会像Java那样用类似”读屏障“的方式来更新这个uintptr，Go中对象也不支持移动，Go中对象被销毁时也不会更新uintptr。理解这里的设计，避免犯错。

Go - Array of pointers ##

要点：[N]*T，元素是指针，可省大对象拷贝、可表示可选（nil）。

vs C/C++：同。注意 GC 下指向的对象只要可达就不会被收。

Go - Pointer to pointer ##

要点：**T 合法，但实际代码远少于 C（多返回值、切片等减少了二级指针需求）。

vs C/C++：C 里 ** 很常见（出参改指针、字符串数组）；Go 较少见。

Go - Passing pointers to functions ##

要点：用于修改调用方变量或避免大结构体拷贝。receiver 选 *T 同理。

vs C/C++：同 C 出参模式。C++ 更常用引用；Go 坚持指针可见性（看得到 * 就知道可能被改）。

Go - Dereferencing Pointer ##

要点：*p；对结构体指针 p.Field 自动解引用。解引用 nil 会 panic。

vs C/C++：解引用空指针是未定义行为；Go 是确定的 panic。自动 . 比 C 的 -> 省事。

九、Go Advanced Control Structures #

Go - Scope Rules ##

要点：包级、函数级、块级作用域。if/for/switch 的 init 变量有局部作用域。导入的包名也在文件作用域。

vs C/C++：块作用域类似。Go 无链接期的 internal/external 那套复杂故事，主要靠包 + 大小写。

Go - Structures ##

type Person struct {
    Name string
    Age  int
}
p := Person{Name: "A", Age: 18}

支持嵌入（embedding）实现组合：

type Employee struct {
    Person      // 嵌入
    EmployeeID string
}

vs C/C++：像 C struct；无构造函数（惯用工厂函数 NewXxx）。嵌入 ≈ 匿名成员 + 提升字段/方法，不是 C++ 继承（无多态基类构造顺序那套）。

Go - Slice ##

要点：动态长度视图：[]T，三元组（ptr, len, cap）。make([]T, len, cap)；append 可能分配新底层数组。切片表达式 a[i:j]。

vs C/C++：对标 std::vector 的常用子集 + 数组视图。比 C 手动 realloc 友好。注意 append 后旧切片可能指向旧数组——共享底层的别名问题类似 vector 迭代器失效，但规则不同。

Go - Slice of Slices ##

要点：[][]T 可做锯齿数组（jagged），每行长度可不同；不是必然连续的二维块。

vs C/C++：像 vector<vector<T>>，不是 T[n][m] 连续二维数组。要连续大块需自己设计。

Go - Range ##

要点：for i, v := range coll。可用于数组/切片/字符串/map/channel。只要索引或值用 _ 丢弃。range 出的 v 是拷贝。

vs C/C++：range-for 类似；Go 还能 range channel（读到关闭为止）。改元素要靠索引 s[i] = ... 或指针切片。

Go - Maps ##

m := make(map[string]int)
m["a"] = 1
v, ok := m["a"]  // 逗号 ok 判断存在
delete(m, "a")

无序；可 == nil；读写 nil map 会 panic（写），读返回零值。

vs C/C++：std::unordered_map / std::map。Go map 非线程安全（并行写要自己同步）；C++ 容器同样默认非线程安全。无树/哈希接口选择，语法内建一种。

Go - Recursion ##

要点：函数调自己；注意栈深度（goroutine 栈可增长，但仍非无限）。尾递归不保证优化。

vs C/C++：相同模式。C/C++ 某些编译器可做 TCO；Go 不把 TCO 当语言保证。

比如执行下面代码：

func main() {
  main()		// WRONG: 递归没有合适的终止条件
  				// runtime: goroutine stack exceeds 1000000000-byte limit**
}

Go - Type Casting（类型转换） ##

要点：T(x) 显式转换；仅预定义的兼容转换。接口用 type assertion：x.(T)。无 C 风格强制拧比特的随便转（那是 unsafe）。

vs C/C++：C 强制转换过宽；C++ 有 static_cast 等。Go 默认更严，int(float64Val) 是显式截断转换。

Go - Interfaces ##

type Reader interface {
    Read(p []byte) (n int, err error)
}
// 隐式实现：类型拥有全部方法即实现接口，无需 implement 关键字

vs C/C++：C 无；C++ 抽象基类 + 虚函数需显式继承。Go 结构化类型（duck typing 的静态版）：适合解耦，但也更依赖文档与接口粒度设计。空接口 interface{} / any ≈ void* 的类型安全版 + 反射。

”组合优于继承“，并不是Go这样做，而是Go将设计模式中的一些良好设计借鉴了过来。既然继承有那么问题，那就索性不支持继承，而是用组合的思路来解决复杂问题。并不是说组合就不能实现继承，比如通过匿名嵌套，配合编译器的一些字段、方法的自动提升，是可以模拟继承的效果。另外，设计模式里面非常经典的，继承关系链过长、类数量爆炸问题，组合才是终极解决方案。

Go - Type Assertion ##

s := i.(string)        // 失败则 panic
s, ok := i.(string)    // 安全形式
switch v := i.(type) { // 类型 switch
case int:
case string:
}

vs C/C++：像 dynamic_cast + std::any_cast 的使用场景。失败默认 panic 而不是空指针，所以生产代码优先用 , ok。

Go - Error Handling ##

要点：错误是值：error 接口。惯用多返回值 (T, error)，调用方 if err != nil。panic/recover 用于真正异常情况，不替代日常错误。

vs C/C++：C 返回码 / errno；C++ 异常。Go 明确拒绝把异常当控制流，错误处理冗长但路径清晰。defer 做清理，对标 finally/RAII 的一部分职责。

Go - Concurrency ##

要点：

  • go f()：启动 goroutine（M:N 调度，轻量）
  • chan T：通信与同步；make(chan T, n) 缓冲
  • select：多路 channel
  • sync.WaitGroup、sync.Mutex / RWMutex 等

哲学：Don’t communicate by sharing memory; share memory by communicating.（也可用锁，看场景）

Go channel支持了CSP的理念，通过通信来共享，而非通过共享内存来通信。有了channel的支持，也可以很容易实现类似future/promise的操作，只是我们再也不需要封装成future/promise的形式。future/promise本身是一种处理异步任务的模式或者方案，在goroutine+channel基础上，可以非常轻松的实现同步转异步、异步转同步。因此，直接使用go惯用法来写就可以，没有必要额外包装future/promise。这个在go issues中维护人员也多次提及这点。 如果你用过一些事件驱动的网络框架，对这个应该不会感到陌生，对于一个异步任务，你创建一个promise用来更新对象，然后用一个future对象来只读查询这个对象。这依然是一个不错的执行异步任务、查询结果的方式，但是在Go中，你不需要用它。

vs C/C++：

GoC/C++
线程goroutine 极轻OS 线程 / 线程池成本高
通信channel 语法级队列库 / 条件变量手写
数据竞争go run -raceTSan 等，非默认开发路径那么顺
模型CSP 风格一等公民锁+共享内存为主流

Go - Regular Expression ##

要点：标准库 regexp（RE2 语法，无回溯爆炸那类 Perl 特性）。Compile / MustCompile，Match / Find / Replace。

vs C/C++：C 无标准正则；C++ <regex> 有但实现与语法方言不一。Go RE2 保线性时间，丢弃部分高级特性换可预测性能。

Go - Inheritance ##

要点：没有经典继承。用 组合 + 接口 + 结构体嵌入 提升字段/方法。多态走接口。

vs C/C++：C 无继承；C++ 单/多继承、虚继承复杂度高。Go 刻意用组合替代 is-a 继承树。

Go - Packages ##

要点：每个目录一个包（同目录同 package 名）；导入路径；导出靠大写。现代用 Go Modules：go.mod / go.sum。init() 包初始化函数。

vs C/C++：比头文件+翻译单元清晰。无环形 import。Module ≈ 官方依赖管理；C/C++ 生态长期碎片化。

Go - Templates ##

要点：text/template、html/template（自动转义防 XSS）。{{.Field}}、管道、range、if 等动作。

vs C/C++：无标准模板引擎；需第三方。html/template 的上下文感知转义是 Web 向特色。

Go - Reflection ##

要点：reflect 包：TypeOf / ValueOf，运行时看类型、改可设值、调方法。代价：性能与可维护性。接口、编码库（json）大量用反射。

vs C/C++：C 基本无；C++ RTTI/typeid 有限，反射远弱于 Go（除非自己 codegen）。Go 反射强但“能不用则不用”。

Go - Generics ##

要点：Go 1.18+：

func Max[T constraints.Ordered](a, b T) T {
    if a > b { return a }
    return b
}

类型参数、约束（interface 作约束）、类型集合。标准库逐步泛型化（如 slices、maps 包）。

vs C/C++：对标 templates，但刻意更简单：无特化迷宫、无 SFINAE。表达力较弱，可读性/编译错误通常更友好。早期 Tutorialspoint Overview 写“无泛型”已过时。

十、Go File Handling #

Go - Read File By Word / By Line ##

要点：

  • 整文件：os.ReadFile
  • 按行：bufio.Scanner
  • 按词：Scanner + Split(bufio.ScanWords) 或 strings.Fields

vs C/C++：fscanf / getline / std::ifstream。Go 的 defer f.Close() 清理很干净。

Go - Read CSV Files ##

要点：encoding/csv：csv.NewReader(f).ReadAll() 或流式 Read()。

vs C/C++：无标准 CSV 库，需手写或第三方。

Go - Delete File ##

要点：os.Remove(path) / os.RemoveAll。

vs C/C++：remove / std::filesystem::remove。

Go - Rename & Move File ##

要点：os.Rename(old, new)（同一文件系统上常即 move）。

vs C/C++：rename / filesystem::rename。跨盘移动可能需拷贝+删除，各平台行为需注意。

Go - Truncate a File ##

要点：os.Truncate(path, size) 或已打开文件 f.Truncate(size)。

vs C/C++：ftruncate / 相应 filesystem API。

Go - File Read-Write Mode W/O Truncation ##

要点：打开标志 os.O_RDWR 等组合；避免默认 O_TRUNC 除非想清空。os.OpenFile(name, flag, perm) 精细控制。

vs C/C++：fopen 的 "r+" / "rb+" 等；或 open 的 O_RDWR。语义同属 Unix 风格标志位思维。

十一、Go Miscellaneous #

Go - defer Keyword ##

要点：defer f() 在函数返回前按 LIFO 调用。参数当时求值。常用于 Unlock、Close、计时收尾。

vs C/C++：接近 finally 或 RAII 析构；但 defer 是函数作用域级，不是块结束即跑（与 C++ 局部对象析构时机不同）。循环里 defer 会堆积到函数结束——要小心。

通过GCC扩展，实际上可以实现作用域级别的对象销毁->触发清理能力。可以查看我的博客中模拟go，java，cpp在如何模拟”defer“能力这块的对比。

Go - Fmt Package ##

要点：Print / Println / Printf；Sprint* 返回字符串；Fprint* 写 io.Writer。常用动词：%v %+v %#v %T %d %s %q %p。

vs C/C++：printf 家族；Go 的 %v/%T 对调试更友好。类型不匹配不会像 C 那样未定义行为，而是报告格式问题。

go不仅提供了 gofmt goimports 工具来方便大家对写好的代码进行格式化，实际上写工具生成某些桩代码时，也可以调用 source.Format(...) 进行格式化，很多类似工具 protoc-gen-go以及其他go桩代码生成工具都会调用格式化函数。go的这种哲学，将业界最推荐的一些工程规范内置在工具中，而不是大家靠自觉、靠部分团队口口相传、纸面上的规范，这极大减轻了go代码可维护性，降低了协作负担。 这让我联想到，当年刚入职写代码时，不同同事使用不同公司的规范，不一致，但是他们说起来理由都是振振有词，都觉得自己的规范好 :(

Go - Zero Value ##

要点：变量声明未显式初始化即为零值——语言核心特性，减少“未初始化”类 bug。

vs C/C++：C 局部变量垃圾值；静态存储区的才会零初始化。C++ 有默认初始化/值初始化规则，复杂度更高。Go 规则一句话：默认零值，且设计上应零值可用。更简单！

Go - Import ##

import (
    "fmt"
    m "math"          // 别名
    _ "net/http/pprof" // 只跑 init
    . "path"          // 导入到当前命名空间（慎用）
)

vs C/C++：#include 是文本包含；import 是包依赖。未使用的 import 是编译错误（逼你干净）。无宏包含顺序玄学。

小结 #

Tutorialspoint 的 Go 教程侧边栏覆盖了从语法到文件、从指针到并发的一条完整入门链。对有 C/C++ 背景的人，最大的“肌肉记忆迁移税”通常在：

  1. 没有隐式真假与隐式类型转换
  2. 数组是值，切片才是日常
  3. 错误是值，不是异常
  4. 接口隐式实现，组合代替继承
  5. go/chan/select/defer 是语法级特性
  6. 字符串按字节与按 rune 两套心智

把上面六条内化，再按本文目录扫一遍侧栏知识点，基本就能把早年 tutorial 级细节快速捡回来。
