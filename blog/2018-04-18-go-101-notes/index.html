Go 101 学习笔记汇总

Posted 2018-04-18 12:00 +0800 by ZhangJie ‐ 4 min read

分享:  

写在前面 ##

这部分内容，是我早年学习 Golang 时的进阶材料与批注式笔记的汇总。2026年对这些散落的知识进行一些总结归档 :)

一开始接触的是各类初级入门介绍。后来了解到社区贡献的 Go 101 项目：内容涵盖 Go fundamentals、generics、optimizations、details & tips、quizzes、bugs、Q&A、apps & libs、blogs 等，从基础到深入覆盖都很扎实，对我帮助很大。当时做了大量笔记，而且多是在网页上通过 Diigo 做的批注。尽管后来 go101 的页面结构有变化，这些笔记仍然带来了很好的记忆锚点——虽然现在看起来略显零散。

我强烈建议 Go 初学者把 go101 作为精进理解的备选材料。可以说，我对 Go 的认识早已不是多年前那个初学者所能企及的了。如今再回看这些知识，对今天的我帮助已经不多，但可能对读者仍有价值。因此不再罗列当年那一长串章节笔记，而把这里提及的要点汇总成一篇文章。

说明：笔记形成于 Go 1.10 前后，部分细节（如泛型尚未正式落地、defer 性能、某些编译器优化）与现行版本可能有出入；文中只保留仍有助于理解语言语义与实践习惯的要点。更完整、更新的内容请以 go101.org 与官方文档为准。

1. Go 是什么：语言画像 ##

Go 是编译型、静态类型语言，源自 Google。它把若干“够用的现代特性”压进一套刻意克制的语法里：

  • 并发一等公民：goroutine、channel、select，CSP 风格——“用通信共享内存，而不是用共享内存通信”
  • 内置容器：slice、map
  • 接口驱动多态 + value boxing，并由此支撑反射
  • 指针（受限制，更安全）、闭包、方法、defer
  • 类型嵌套 / 组合（匿名嵌入）替代经典继承
  • 类型安全、内存安全、GC、类型推断
  • 语法简洁，标准库与社区生态活跃

当时的笔记里还写过“尚不支持自定义泛型，是否采用需权衡”——这一点在 Go 1.18 之后已经改写，但 go101 仍适合用来补语言细节，而不是当作版本变更史。

工具链上：go run 适合简单片段；复杂工程用 go build / go install。go vet 能报出 run/build 常忽略的可疑用法。

2. 源码与语言骨架 ##

2.1 分号、换行、标识符 ##

编译器会按规则自动插入分号。实践上：只在二元运算符、赋值号、点号、逗号、分号，以及 { [ ( 之后断行，可少踩换行陷阱。

关键字大致四类：声明（package/import/const/var/type/func）、复合类型（chan/interface/map/struct）、控制流、以及 go/defer 这类特殊修饰。标识符可导出看首字母是否大写；_ 是 blank identifier。

2.2 常量、变量、iota ##

  • 无类型常量 vs 有类型常量：字面量常有默认类型；nil 没有默认类型。
  • 变量必须始终有确定类型；包级变量按依赖初始化，禁止环依赖。
  • 支持多值赋值与交换 x, y = y, x，不支持链式赋值 a = b = 1。
  • 短变量声明 := 在当前作用域未声明的名字会新建变量，嵌套 block 里容易“阴影”外层变量。
  • 所有变量可寻址，所有常量不可寻址（复合字面量取地址是语法糖）。

2.3 表达式、简单语句、控制流 ##

if/for/switch 的某些槽位要求 simple statement（短变量声明、赋值、函数调用、channel 收发、x++/x-- 等）。注意：++/-- 是语句不是表达式，且只有后缀形式。

六类控制流：if-else、for、switch-case，以及与类型相关的 for-range、type-switch、select-case。除 if 外多可 break；switch/select/type-switch 默认不贯穿，需 fallthrough（且不能出现在最后一个分支）。

2.4 包、导入、init ##

  • 导入路径分隔符始终 /；匿名导入 _ "..." 只为执行 init。
  • 不支持循环依赖；加载按依赖序，main 最后。
  • 同包可有多个 init，无参无返回，各执行一次；依赖包的 init 先于导入方。

2.5 函数、defer、panic/recover、goroutine 入门 ##

  • 函数退出有 exiting phase：defer 按 LIFO 执行；有名返回值可在 defer 中修改。
  • go f() / defer f() 的返回值必须丢弃；参数在推入/创建时求值，不是执行时再求。
  • 用户态不直接创建 OS 线程，程序内并发主要靠 goroutine。
  • 未 recover 的 panic 会打崩整个进程；服务侧在关键路径上 recover 往往必要。
  • 全程序 goroutine 都永久阻塞时，runtime 会判定死锁并崩溃。
  • GOMAXPROCS：1.5 起默认约等于可用逻辑 CPU 数。

3. 类型系统核心：Value Parts ##

go101 对理解 Go 帮助最大的框架之一，是把值拆成：

  • direct part：值“头部”，unsafe.Sizeof 量的就是它
  • indirect part：被头部指针引用的底层数据

大致分类：

只有 direct part含 indirect part
数值、bool、数组、struct 字段布局、安全指针等string、slice、map、channel、function、interface

赋值、传参、返回值：一律浅拷贝 direct part。带 indirect 的类型赋值后，往往共享底层数据——这是 slice 别名、map 共享、channel 是同一对象等现象的统一解释。

内部形状（运行时视角，非用户可见类型）：

  • map / chan / func ≈ 指向实现结构的指针
  • slice：{data, len, cap}
  • string：{data, len}
  • 空接口 eface：{_type, data}；非空接口 iface：{itab, data}

3.1 可比较性与取地址限制（高频踩坑） ##

不可比较（因而也不能做 map key）：slice、map、function，以及含不可比较字段/元素的 struct、array。

常不可取地址：

  • string 中的字节、map 元素、接口的动态值
  • 常量、多数中间表达式结果
  • 包级函数、方法值本身等

复合字面量可取地址是语法糖（先建临时变量再取址）。

3.2 容器：array / slice / map ##

  • array 整体拷贝；slice/map 赋值共享底层。
  • append 可能换底层数组；容量不足时扩容策略与共享关系要心里有数。
  • subslice / 再切片后的别名与泄漏风险见后文内存一节。
  • for range 遍历的是 direct part 的拷贝；大数组应用指针或切片。
  • 容器操作本身无同步；并发写必须自己加锁或改设计。
  • 可用 map[K]struct{} 模拟 set。

3.3 string ##

  • 不可变 UTF-8 字节序列；赋值共享底层；s[i] 不可取址修改。
  • for range 按 rune（码点）走，index 是字节下标，可不连续。
  • string ↔ []byte 通常深拷贝（可变与不可变语义冲突）；编译器在特定场景有优化（如 for range []byte(s)、用作 map key 的转换、与常量拼接等，随版本可能变）。
  • 拼接：+、strings.Builder、bytes.Buffer 等；大量拼接优先 Builder。

3.4 方法 ##

  • 只能给 已定义的非指针、非接口类型 T 或 *T 声明方法（同包）。
  • value / pointer receiver 选择：要改状态、大对象、嵌套 sync 类型 → 倾向 pointer；小对象可 value，但同一类型别混用两种风格，以免并发语义混乱。
  • 编译器为 value 方法生成 pointer 包装；对可寻址值调用 pointer 方法则是取址语法糖。
  • receiver 同样按值传递（指针是传指针值）。

3.5 接口 ##

  • 方法集超集 → 隐式实现接口（无 implements 关键字）。
  • 赋值接口 = value boxing：写入 dynamic type / dynamic value，支撑多态与反射。
  • 类型断言 i.(T) 单值失败会 panic，双值形式更安全；type switch 是其增强版。
  • 接口比较本质是比 dynamic type + dynamic value。
  • 即便 T 实现 I，[]T 也不能直接转 []I，需逐元素装盒。
  • 早年用反射部分弥补“缺自定义泛型”；今日有真正泛型，反射仍用于 encoding/fmt 等动态场景。

3.6 unsafe 与反射（点到为止） ##

  • unsafe.Pointer ↔ 安全指针 / uintptr 有固定转换规则；uintptr 不持有引用，可能被 GC，常需 runtime.KeepAlive 或在同一表达式内完成转换与使用。
  • 官方文档列出若干合法 pattern；越界乱用会破坏类型与内存安全。
  • reflect.Type / reflect.Value：标准库大量使用；能做许多动态操作，代价是性能与可读性。

4. Channel：语义、规则与用法 ##

4.1 模型 ##

Channel 可看作同步 FIFO：内部大致有 缓冲环、发送等待队列、接收等待队列。发送/接收/关闭本身已同步，但“收到值再赋给共享变量”并不自动线程安全。

nil / 未关闭 / 已关闭 行为简表：

操作nil未关闭已关闭
closepanic成功panic
send永久阻塞可能阻塞panic
recv永久阻塞可能阻塞立即返回零值；ok==false（缓冲里旧数据先排空）

close 不能用于 receive-only channel。v, ok := <-ch 中 ok 表示是否在关闭前发送。

4.2 select ##

  • 无表达式；case 只能是 send/recv；不可 fallthrough。
  • 多 case 就绪时伪随机选一（实现上涉及按 channel 地址排序加锁防死锁、CAS 保证只赢一个 case）。
  • 无 default 且都不可用 → 阻塞；select{} 永久阻塞。
  • 已关闭 channel 在 select 里永远可读：常把不用的 case 对应 channel 置 nil 以禁用该 case。
  • select { case ch<-x: default: } / 对称的 try-recv 永不阻塞，编译器会特别优化。

4.3 典型用法（笔记中的模式） ##

  1. Future/Promise：返回 <-chan T 或传入 chan<- T，异步填结果。
  2. 通知：chan struct{}；一对一发送/接收；close 做 1 对 N 广播；缓冲容量设计可避免“只取第一个结果”时泄漏 worker。
  3. 当 mutex / 计数信号量：cap==1 互斥，cap==N 限流（通常不如 sync 包高效）。
  4. Ping-pong：两 goroutine 来回传数据。
  5. channel of channel：把回信地址随任务传递。
  6. 数据流管线：生成 → 聚合/拆分 → 组合/分解 → 过滤/计算 → 输出；用方向性 channel 约束模块角色。

Rob Pike 的建议仍适用：优先用通信表达所有权转移；但 channel 不是万能——单纯保护共享状态时，Mutex/atomic 往往更合适。

5. 其他同步原语 ##

  • sync.WaitGroup：Add / Done / Wait，等待一组任务结束。
  • sync.Once：保证只执行一次初始化。
  • Mutex / RWMutex：互斥与读写锁。
  • sync.Cond：与锁配合的条件等待（Wait / Signal / Broadcast），务必在循环里检查条件。
  • sync/atomic：无锁原子读写与 CAS。

单进程内优先这些；跨进程/分布式才考虑网络或文件锁这类更重的同步。

并发编程建议同步阅读 Go memory model：原语帮你写对，但误用仍会 race。

6. 内存：泄漏形态与习惯 ##

有 GC 不代表不会“逻辑泄漏”：

  1. 子串/子切片持有大底层s0 = big[:50] 可能让整块底层活着。应对：string([]byte(sub)) 强制拷贝、strings.Builder、或拷贝到新 slice（append([]T(nil), s...)）。部分字符串拼接优化 historically 存在，不可当作可移植保证。
  2. slice 中“死”元素仍持有指针缩短 len 不清理尾部指针时，被引用对象无法回收——需要显式置 nil 或覆盖。
  3. 挂起的 goroutine 永久阻塞且仍被引用的 channel/闭包会拖住整条引用链，属于真实泄漏。

其他习惯：大结构体传指针要权衡 GC；for range 大元素用下标访问；关注值拷贝成本与 BCE（边界检查消除）等编译器优化——可用 go build -gcflags="-d=check_bce" 观察。

7. 杂项速查（当年最有用的“清单感”） ##

可用 T{...} 复合字面量的类型：struct、array、slice、map。

零值可为 nil 的类型：pointer、slice、map、channel、function、interface。不同 kind 的 nil 尺寸不同，且未必可比较。

编译期可求值：unsafe 包函数；某些 len/cap（如数组）等，结果可作常量。

永远阻塞当前 goroutine：读/写永不就绪的 channel、对 nil channel 收发、select{}。

defer 注意点（当时笔记）：

  • 多数有返回值的内建函数不能直接 defer（recover/copy 例外）；可包一层匿名函数。
  • defer 的函数值与参数在 注册时 求值；var f func(); defer f() 会对 nil 调用。
  • defer 提升可读性与正确性（关文件、解锁），热路径上仍有开销，需按场景取舍。

recover 有效条件（规格要点）：

  • 一般须由 deferred 函数直接调用；
  • panic 沿调用栈向上传播，不会向下；
  • 同深度新 panic 可压制旧 panic；不同深度可并存多个 active panic。

nil 的诡异但合法行为：

  • 读 nil map 不 panic（得零值）；写 nil map 会 panic。
  • range nil map/slice：0 次；range nil channel：永久阻塞；range 数组指针：按数组长度。

包作用域：universal / package / file / local；if/for 等带有隐式 block——for k, v := range m { k, v := k, v } 里内外层是不同作用域。

8. 结语 ##

go101 的价值，不在于再讲一遍 “Hello World”，而在于把 类型语义、值表示、channel 规则、接口装箱、defer/panic 细节 等“写对 Go 必须清楚、却又容易被入门教程略过”的内容，用可验证的例子串起来。

当年这些零散批注，是理解从“会写”到“写对、写稳”的脚手架。今天把它们收成一篇，既是对自己学习路径的归档，也希望给后来者一张更短的地图：去读原站，去写代码，去踩坑，再回来对照。

若只带走三句话：

  1. 先建立 value parts 心智模型，再谈 slice/map/interface 行为。
  2. channel 规则表 + select 语义 比背十个模式更重要。
  3. GC 管的是可达性，管不了你的逻辑泄漏与死锁。
