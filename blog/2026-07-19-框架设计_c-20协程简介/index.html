C++20 协程简介：语言级无栈协程与 co_await

Posted 2026-07-19 17:39 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

C++ 社区很早就有协程实践：Boost.Coroutine、各家 stackful 库、微信 libco 等。它们多数是 有栈协程——每条协程一块栈，切换时换上下文。

C++20 把另一条路线写进了语言标准：无栈（stackless）协程。标准委员会提供的是：

  • 语言层的挂起/恢复机制（co_await / co_yield / co_return）；
  • 编译器生成的 协程帧 与一组可定制的 promise / awaitable 钩子；
  • 不 附带完整的「官方调度器 + 通用任务运行时」（与 Go 不同）。

一句话：100% 语言构造支持 + 0%（几乎）标准化执行层。强大且灵活，学习曲线也因此偏陡。本文做技术向简介，帮助建立正确心智模型。

一、什么函数算协程 #

在 C++20 中，函数体里若出现下列之一，该函数就是协程：

关键字作用
co_await等待一个 awaitable，可能挂起当前协程
co_yield产出一个值并挂起（生成器常见）
co_return结束协程并返回值（或 void）

特征：

  • 协程可以 挂起 后把控制权交还调用方/调度方，之后再 从挂起点恢复；
  • 恢复所需的状态不放在「普通调用栈的长期占用」里，而在编译器生成的 coroutine frame（通常堆分配，可优化）中；
  • 因此标准明确：C++20 协程是 stackless 的——多个协程共享当前线程的 OS/调用栈，各自帧在别处。

二、无栈意味着什么 #

2.1 和有栈库的差别 ##

有栈（Boost.Coroutine2 部分模式、libco、libmill）C++20 无栈
每任务栈独立栈空间无完整私有栈
切换成本保存寄存器 + 换栈保存帧内状态 + 返回/跳转
任意函数 yield栈上任意深度通常可切只有协程函数 能 co_await；普通函数不能中途挂起
与现有代码有时对「普通调用链」更宽容需要 await 风格改造调用链
实现位置库 + 汇编/ucontext 等编译器 切分状态机

无栈更难「第一眼看懂」的原因：源码仍是自上而下的函数，编译后却变成了按挂起点切开的状态机；局部变量可能被「提升」到堆上的帧里，生命周期规则也与普通函数不同。

2.2 co_await 在干什么（直觉版） ##

对表达式 co_await a;，可以极度简化地理解为：

  1. 确保当前协程中仍需要的局部状态在 协程帧 里（常在堆上）；
  2. 构造一个「恢复点」——之后调用它可以从 co_await 之后继续；
  3. 询问 awaitable a：现在能不能直接给出结果？
    • 能：取出结果，协程继续同步往下跑；
    • 不能（例如等 I/O）：挂起当前协程，把恢复句柄交给 a，控制流回到 启动/恢复该协程的那一层；
  4. 当异步结果就绪，某人或调度器再次 resume 该协程，从挂起点继续。

注意：步骤 3 里 awaitable 的方法返回时，往往并不是「返回给 co_await 下一行」这种普通函数语义，而是可能直接把 CPU 交给调度器或其他协程。这也是初学者容易晕的地方。

三、定制点：promise_type 与 awaitable #

C++20 协程是 机制，不是完整 框架。行为由你（或库）提供的类型决定。

3.1 promise_type ##

协程返回类型 R 关联的 R::promise_type（或特化）控制例如：

  • 协程启动时是否立刻挂起（initial_suspend）；
  • 结束时如何挂起/销毁（final_suspend）；
  • co_return 的值如何存储；
  • 未捕获异常如何处理；
  • 如何暴露 coroutine_handle 给外界 resume。

常见库会提供 task<T>、generator<T> 等，把 promise 细节包起来。自己从零写一个可用的 task 是理解协程的「成年礼」，也是劝退点。

3.2 Awaitable 协议 ##

一个类型要能被 co_await，需满足 awaitable 协议（可直接实现或通过 operator co_await / await_transform 转换），核心是三个操作的语义：

操作含义
await_ready是否已就绪，避免无谓挂起
await_suspend未就绪时挂起，登记 resume 回调
await_resume恢复后取结果（或抛异常）

异步 I/O、定时器、channel 接收、锁的异步获取……都表现为不同的 awaitable。

3.3 coroutine_handle<> ##

std::coroutine_handle<> 是指向协程帧的句柄：可 resume()、destroy()，可判断 done()。它是 执行层 的拼装积木——线程池、事件循环、串行 strand，都靠「谁在何时 resume 哪个 handle」搭起来。

四、最小概念示例 #

生成器（co_yield） ##

// 示意：具体 generator 类型需自带 promise_type
generator<int> range(int n) {
    for (int i = 0; i < n; ++i)
        co_yield i;   // 产出 i 并挂起，下次 resume 从这里之后继续
}

控制流在「调用方拉取下一个值」与「生成器协程」之间切换——正是协程最早在编译器/迭代器场景里的用法。

异步任务（co_await） ##

task<Response> handle(Request req) {
    auto body = co_await async_read(req.socket); // 可能挂起
    auto res  = co_await async_process(body);
    co_return res;
}

源码像同步，编译后在两次 co_await 处切开；中间状态进协程帧。
调用 handle 的人 拿到的通常是一个「未完成任务」对象，而不是已经算完的 Response——除非库把接口设计成同步等待。

五、和周边技术的坐标 #

技术栈模型谁提供调度
C++20 协程无栈你自己 / 第三方（Asio、各种 task 运行时）
Boost.Asio + C++20无栈 + 成熟 I/OAsio executor
libmill / libco有栈库内调度
Go goroutine有栈（分段栈/增长栈）Go runtime
Java Kilim/Quasar无栈 + 字节码库内调度
Java 虚拟线程轻量线程 + 可迁移栈状态JVM

C++20 的选择偏向：把最小机制放进语言，把策略留给库。优点是零开销抽象潜力大、可嵌入各种运行时；缺点是「写出能用的协程」门槛高于 Go/async Python。

六、实践中的坑（简表） #

  1. 生命周期：协程帧与 lambda 捕获、引用参数悬空——挂起后引用的栈对象可能已销毁。
  2. 对称 vs 非对称：标准协程偏 非对称（挂起后回到 resume 的调用方），不是任意协程间直接 yieldto。
  3. 调试与性能：帧分配、异常路径、final_suspend 是否挂起，都影响性能与泄漏。
  4. 与异常模型交互：异常要跨挂起点传递，promise 必须正确处理。
  5. 不要指望「一个 co_await 自动让全进程变成 Go」——没有 runtime 就没有自动 M:N。

总结 #

C++20 协程是 语言级无栈协程：

  • 用 co_await / co_yield / co_return 表达挂起与恢复；
  • 编译器生成 状态机 + 协程帧；
  • 通过 promise_type 与 awaitable 高度可定制；
  • 调度与 I/O 模型 不在标准核心里，需库补齐。

它和 libmill/Go 的有栈模型、和 Java 织入式 Fiber、和虚拟线程，都共享「并发任务可让出执行权」的思想，但实现落点不同。若要系统对比 有栈 vs 无栈 的优劣与理解难度，见同目录对比专文。
