Intel LOCK 指令前缀：硬件原子性的基石

Posted 2026-07-16 20:00 +0800 by ZhangJie ‐ 3 min read

分享:  

多核时代，一个最基础的问题：两个 CPU 同时写同一个内存地址，谁能赢？答案依赖原子操作。在 Intel x86 上，原子操作的硬件根基就是一条不起眼的前缀——LOCK（0xF0）。

这篇文章聚焦 LOCK 前缀本身：它做了什么、怎么做的、代价有多大，以及上层语言（Java、Go、C）的原子/屏障语义是如何最终"落地"到这条前缀上的。

背景：为什么需要 LOCK？ ##

看一个最简单的例子——两个线程同时执行 counter++：

线程 A: 读取 counter (100) → 加 1 → 写回 (101)
线程 B: 读取 counter (100) → 加 1 → 写回 (101)

如果没有原子性保证，两次自增最终可能得到 101 而不是 102。这是因为 counter++ 在指令层面是三条指令：读、改、写。如果这三步中间被另一个核插入，就产生了丢失更新。

LOCK 前缀解决的就是这个问题——它让读-改-写三个步骤"打包"成一个不可分割的整体。

LOCK 前缀是什么？ ##

LOCK 是 x86 指令的前缀字节（0xF0），可以修饰特定的内存操作指令。被 LOCK 修饰的指令在执行期间：

  1. 独占内存操作数所在的内存区域（总线锁或缓存锁）
  2. 该指令的读-改-写成为一个原子事务——期间任何其他总线主设备（其他 CPU 核、DMA 控制器等）都不能访问该内存区域
  3. 充当全内存屏障——指令完成后，store buffer 被 drain，之前的 store 全局可见，之后的 load/store 不会被重排到 LOCK 指令之前

关键约束：LOCK 只能修饰读-改-写指令（ADD、XCHG、CMPXCHG、INC、DEC、XADD 等），不能修饰 MOV。尝试在 MOV 前加 LOCK 会触发 #UD（未定义指令异常）。MOV 到内存只是写，没有"改"这一步，不存在中间状态需要保护。

例外：XCHG 指令在手册中有一个特殊待遇——无论是否显式写出 LOCK 前缀，只要操作数涉及内存，XCHG 就自动隐含 LOCK 语义。这也是为什么 Go 的 atomic.Value.Store 在 x86 上使用 XCHGQ 而不是 MOV + LOCK。

工作机制：从总线锁到缓存锁 ##

总线锁（Bus Lock）—— 早期实现 ##

最直观的互斥方式：LOCK 指令执行期间，拉高 LOCK# 引脚信号，物理锁定整个系统总线。其他所有设备在此期间无法发起内存访问。

   CPU 0                    CPU 1
     │                        │
     │ LOCK INC [addr]        │
     │ ┌──────┐               │ MOV [other], val
     │ │ LOCK#│  ──总线──      │   ↑ 被阻塞！
     │ │ assert│              │
     │ └──────┘               │
     │ INC 执行完成            │
     │ 释放 LOCK#              │ ──→ MOV 得以执行

总线锁的优点是简单可靠，缺点是代价极大——整个总线上所有数据传输都被阻塞，哪怕被锁的地址和正在访问的地址毫不相干。

缓存锁（Cache Lock）—— 现代实现 ##

从 Intel P6（Pentium Pro，1995） 开始，处理器引入了一种更高效的方式：如果被 LOCK 的操作数完全包含在一个缓存行内（且地址对齐），处理器不会锁总线，而是通过 MESI 缓存一致性协议 锁定该缓存行：

1. CPU 0 声明对该缓存行的"独占"（Exclusive/Modified 状态）
2. 在 LOCK 操作完成前，不响应其他核对该缓存行的 snoop 请求
3. LOCK 指令执行完毕后，恢复正常 MESI 状态转换

这相当于把锁的粒度从"整个总线"缩小到了"一个缓存行"。其他核仍然可以访问不同的缓存行，性能大幅提升。

什么情况下会回退到总线锁？

  • 操作数跨越了两个缓存行（未对齐的 8 字节操作可能跨 cache line 边界）
  • 操作数不在可缓存内存中（如 MMIO 区域）
  • 老处理器不支持缓存锁

这也是为什么 Linux 内核中 atomic_t 要求自然对齐——不对齐可能导致一次原子操作退化到总线锁，影响整个系统的性能。

LOCK 作为全内存屏障 ##

LOCK 前缀的另一个关键作用是内存屏障。在 x86 上，任何 LOCK 前缀的指令都充当全屏障（full barrier / mfence 等效）。

具体来说：

Store A
─────────── LOCK 指令 ───────────
                                   Store B
                                   Load C
  • LOCK 之前的 所有 store 必须 drain 出 store buffer（全局可见）之后，LOCK 指令才能开始执行
  • LOCK 执行完成之后，后续的 load/store 一定在 LOCK 完成后才开始——防止 Store→Load 重排

这就是你在姊妹文章中看到的结论——x86 上 LOCK 前缀指令 = 全屏障。Java volatile 写、Go atomic.Store、C memory_order_seq_cst store，在 x86 上最终都会生成一条带 LOCK 语义的指令。

Java volatile write
  → JVM: StoreStore + StoreLoad barrier
  → x86: LOCK ADD 或 XCHG

Go sync.Mutex.Unlock
  → runtime: atomic CAS (LOCK CMPXCHG)
  → x86: 全屏障

C++ std::atomic seq_cst store
  → x86: XCHG（隐式 LOCK）

常见的 LOCK 指令 ##

以下是日常编程中通过 LOCK 前缀实现的典型指令：

指令操作上层对应
LOCK CMPXCHG原子 CAS（Compare-and-Swap）JavaAtomicInteger.compareAndSet、Go sync.Mutex、Rust AtomicBool::compare_exchange
LOCK XADD原子 fetch-and-addJavaAtomicInteger.getAndAdd、Go sync.WaitGroup.Add
LOCK XCHG原子交换Goatomic.Value.Store（x86 AMD64）
LOCK INC/DEC原子增减编译器优化a++ 为 LOCK INC（无竞争时）
LOCK ADD原子加JVM 常用LOCK ADDL $0, (SP) 作为 StoreLoad barrier
LOCK BTS/BTR原子位测试并置位/复位内核中 bit 标志位原子操作

一个有趣的实现细节：JVM 的空操作屏障 ##

你可能在 JVM 生成的汇编中看到过这样的指令：

LOCK ADDL $0, 0(%rsp)

这条指令往栈上（%rsp）加 0——对程序语义完全无影响。它的唯一目的就是利用 LOCK 的全屏障语义，充当 StoreLoad barrier。为什么不用 MFENCE？因为在某些老处理器上 LOCK ADD 比 MFENCE 更快。

LOCK 的性能代价 ##

LOCK 不是免费的。即使走缓存锁，代价也比普通指令高一个数量级：

操作近似延迟（Skylake，L1 命中）
普通MOV（load/store）~4-5 cycles
LOCK CMPXCHG（无竞争）~15-20 cycles
LOCK CMPXCHG（有竞争）数十到上百 cycles

代价来源：

  1. 缓存一致性协议开销——需要从其他核"抢"缓存行的 Exclusive 状态
  2. Store buffer drain——必须等待之前的 store 全部完成，无法利用 store buffer 的异步性
  3. 指令流水线冲刷——LOCK 指令强制序列化执行，打乱乱序执行的流水
  4. 竞争时的 cache line bouncing——多个核反复争抢同一个缓存行，产生大量的 snoop 流量

这也是无锁编程中要精打细算的地方——能用 release/acquire 单向屏障就不要用 seq_cst 全屏障，能用 XCHG 就不在循环里反复 LOCK CMPXCHG。

与 ARM 的对比 ##

x86 用 LOCK 前缀实现原子性和全屏障，ARM 走了另一条路：

x86ARM
原子 RMWLOCK 前缀 + 普通指令（如 LOCK ADD）专用原子指令：LDXR/STXR（Load-Linked / Store-Conditional）
store-release无需特殊指令（TSO 模型下 store 天然保序）STLR（Store-Release）——保证之前 store 对该 store 之后的 load 可见
load-acquire无需特殊指令LDAR（Load-Acquire）
全屏障LOCK 前缀或 MFENCEDMB SY（Data Memory Barrier）

本质差异：x86 的 TSO 内存模型本身比 ARM 更强——x86 store 天然是 release 语义，load 天然是 acquire 语义——所以 x86 通常只需要在 store 侧的 StoreLoad 重排上做文章，也就是 LOCK 的主场。

总结 ##

  1. LOCK 前缀是 x86 原子操作的硬件根基——它让读-改-写指令成为不可分割的事务
  2. 现代处理器默认走缓存锁（通过 MESI 协议锁单个缓存行），仅在跨缓存行、MMIO 等场景回退到总线锁
  3. LOCK 指令天然是全内存屏障——不仅保证原子性，还 drain store buffer、禁止前后指令重排
  4. 上层语言最终都映射到它：Java volatile 写 → LOCK ADD/XCHG，Go atomic → LOCK CMPXCHG/XCHG，C seq_cst→XCHG`
  5. 代价不低但可接受：无竞争时 15-20 cycles，远低于总线锁，但比普通 load/store 贵 3-5 倍
  6. 与 ARM 的设计哲学不同：x86 用一个前缀把原子性和内存屏障打包解决，ARM 用 LL/SC + 独立屏障指令分别应对

理解 LOCK 前缀，就能理解为什么 x86 上 Java volatile 写有开销而 volatile 读几乎免费，也能理解为什么 lock-free 编程中"减少 contended CAS"是首要原则——每多一次 LOCK CMPXCHG，就是在跟缓存一致性协议抢资源。
