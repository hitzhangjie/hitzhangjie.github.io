Linux 信号处理流程详解

Posted 2025-07-17 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

背景 #

信号是 Linux 系统中一种经典的异步通知机制。无论是 Ctrl+C 触发 SIGINT，还是子进程退出时的 SIGCHLD，或是代码中的 SIGSEGV，信号在系统编程中无处不在。

但信号的处理时机和流程，很多人只有一个模糊的印象——“信号会中断当前执行，跳转到 handler”。具体是怎么中断的？内核是如何在恰当的时机插入信号处理的？handler 执行完后又是如何恢复原来的代码继续执行的？

本文梳理信号从产生到处理完成的完整流程，厘清关键的内核机制。

核心思想 #

Linux 内核在进程从内核态返回用户态的瞬间，检查并处理该进程的挂起信号。

进程在用户态正常执行时，即使有信号到达，也不会被立即处理。信号处理的"窗口"只存在于内核态→用户态返回的边界上。

触发信号处理的两个时机 #

信号处理只在以下两个时机被触发：

  1. 系统调用返回时 进程执行 read()、write() 等系统调用，在内核态完成任务、准备返回用户态之前。

  2. 中断或异常处理返回时 硬件中断（时钟中断、键盘输入等）或异常（缺页异常等）处理完成后，内核准备恢复用户态执行之前。

这两个时机的共同点是：都要从内核态返回用户态。

详细处理流程 #

第一步：信号产生 ##

某个事件导致内核决定向目标进程发送信号。例如：

  • 用户按下 Ctrl+C，终端驱动向当前前台进程组发送 SIGINT。
  • 子进程退出，内核向父进程发送 SIGCHLD。
  • 进程执行了非法操作，内核向其发送 SIGSEGV。

此时内核做的事情是：在目标进程的 task_struct 中将对应的信号标记为 pending（挂起） 状态。

信号仅仅被记录，不会被立即处理。

第二步：进程陷入内核态 ##

进程因系统调用或中断而陷入内核态执行。在内核态执行期间，pending 信号依然不会被处理——内核忙着处理当前的任务。

第三步：返回用户态前的检查 ##

这是信号处理中最关键的一步。

内核态任务完成后，在 ret_to_user（返回用户态）的路径上，内核检查当前进程是否有待处理的信号。这个检查会跳过被进程阻塞（blocked）的信号，只处理未被阻塞的 pending 信号。

第四步：准备处理信号 ##

如果找到了需要处理的信号，内核执行以下操作：

  1. 在进程的用户态栈上分配空间，构造一个新的栈帧（sigframe）。
  2. 该栈帧保存两项关键信息：
    • 上下文信息：当前的寄存器状态、程序计数器（PC）、栈指针等，用于后续恢复。
    • 信号信息：信号编号、发送者信息等（用于 sigaction 注册的 handler 接收 siginfo_t）。
  3. 修改进程的程序计数器（PC），使其指向用户空间注册的信号处理函数地址。

第五步：执行信号处理函数 ##

内核返回到用户态——但不是回到被中断时的代码位置，而是开始执行信号处理函数。

从进程的视角看，就好像在它正常执行的过程中，被"强行插入"了一段信号处理代码。

第六步：从信号处理函数返回 ##

信号处理函数执行完毕后，会执行 sigreturn 系统调用（glibc 会自动在 handler 返回后调用，对用户透明）：

  1. 进程再次进入内核态。
  2. 内核利用之前保存在用户栈上的上下文信息，将寄存器和程序计数器恢复到被中断前的状态。

第七步：恢复正常执行 ##

内核再次返回用户态，这次是回到最初被中断的位置。进程继续执行原来的代码，就像什么都没发生过一样。

流程图 #

用户态执行 → 陷入内核态 → 内核态处理 → 返回用户态前检查 pending 信号
                                              ↓
                                    有未阻塞的 pending 信号？
                                   ↙                    ↘
                              是                          否
                              ↓                           ↓
                   构造 sigframe + 修改 PC        直接返回原执行流
                              ↓
                    返回用户态执行 handler
                              ↓
                    handler 结束 → sigreturn
                              ↓
                    内核恢复上下文 → 返回原执行流

总结 #

  1. 信号不是"立即"处理的，而是在从内核态返回用户态的边界上被检查和处理的。
  2. 本质上是利用内核态→用户态的切换，在用户栈上构造一个"假的"返回点，让进程先去执行信号 handler，handler 结束后再通过 sigreturn 恢复现场。
  3. 理解这个机制有助于理解很多相关行为：为什么某些系统调用会被信号中断（EINTR）、为什么在多线程程序中信号只被一个线程处理、以及调试器中信号拦截的原理。
