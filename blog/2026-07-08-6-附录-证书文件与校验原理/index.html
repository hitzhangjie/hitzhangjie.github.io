附录：证书文件与校验原理 #

三类文件都包含什么，以及如何查看 ##

TLS 场景里最常见的三类文件是：key（私钥）、CA 证书、普通证书（节点/服务证书）。下面按这三类统一说明。

1) key 文件（如 etcd-key.pem / ca-key.pem） ##

包含的信息：

  • 私钥参数（如 RSA 的 d/p/q 等）
  • 对应公钥参数（如 n/e，属于密钥结构的一部分）
  • 不包含 Subject、Issuer、有效期、SAN、签名等证书字段

常用查看命令：

# 查看私钥基本信息（建议优先用 pkey，兼容 RSA/ECDSA）
openssl pkey -in etcd-key.pem -text -noout

# 仅导出公钥（用于后续比对）
openssl pkey -in etcd-key.pem -pubout -outform pem

私钥可以直接导出公钥，本质上公钥就是由私钥参数推导出来的。但是不能反过来，不能从公钥推导出私钥。

重点看什么：

  • 算法类型（RSA / EC）
  • 密钥位数（如 2048/4096）
  • 是否与证书公钥匹配（见“配对与签发关系检查”文档）

2) CA 证书文件（如 ca.pem / new-ca.pem） ##

包含的信息：

  • CA 的公钥（用于验证被它签发的证书）
  • CA 自身身份信息（Subject）
  • 颁发者信息（自签时 Subject=Issuer，续签/交叉签名时可能不同）
  • 有效期、序列号、签名算法
  • 关键扩展（通常应有 Basic Constraints: CA:TRUE、证书签发用途）

重点看什么：

  • 是否真的是 CA 证书（CA:TRUE）
  • 是否具备签发用途（Certificate Sign）
  • 到期时间和轮换窗口

3) 普通证书文件（如 etcd.pem） ##

包含的信息：

  • 节点/服务公钥
  • Subject（证书主体身份）
  • Issuer（签发者，一般为 CA）
  • 有效期、序列号、签名算法
  • 扩展用途（Key Usage / EKU）
  • SAN（DNS/IP 列表）

重点看什么：

  • Basic Constraints 一般应为 CA:FALSE
  • EKU 是否包含 TLS Web Server Authentication / TLS Web Client Authentication
  • SAN 是否覆盖实际访问域名/IP

SAN 在验证过程中如何校验（LB 场景） ##

SAN 校验的核心是：客户端用“自己实际访问的目标”去匹配证书 SAN。

  • 用域名访问（如 https://etcd-lb.example.com）时，匹配 SAN: DNS:etcd-lb.example.com
  • 用 IP 访问（如 https://21.66.xxx.xx）时，匹配 SAN: IP Address:21.66.xxx.xx
  • 匹配动作由客户端本地完成；不匹配就会报证书名称不匹配并拒绝连接

在“业务访问 LB，LB 再转发到后端 etcd 节点”的四层转发场景中：

  • 客户端校验对象仍是“它连接的 LB 地址”
  • 后端真实节点 IP 是否在 SAN，不影响这次以 LB 地址发起的校验
  • 若未来改为域名访问 LB，则需在 SAN 中补充对应 DNS 项

补充说明：

  • 域名访问时，客户端通常会在 TLS 握手中发送 SNI（服务器名）帮助服务端选证书
  • 但 SAN 是否匹配，仍以客户端本地“实际连接目标”作为验证依据
  • 因此 SAN 的安全意义是：限制证书只能用于被授权的 DNS/IP，降低证书被挪到其他机器滥用的风险

以 etcd 为例，etcd cluster 内部可以配置一套证书，etcd cluster 和 etcd client 之间可以配置另一套证书，可以是两套独立的证书。但是有时候运维为了省事，会使用同一套证书，这时候需要特别注意 SAN 的校验，确保 SAN 中包含了业务侧连接时的 CLB 地址，而不只是 etcd cluster members 地址。

补充：client 身份校验（mTLS + RBAC） ##

  • etcd 可以验证连接它的 client。开启 --client-cert-auth=true 并配置 --trusted-ca-file 后，etcd 会在 TLS 层校验客户端证书链（mTLS）。
  • 除了证书校验，建议同时启用 etcd 内置认证与授权（用户名/密码 + RBAC），实现“证书确认身份 + 权限控制操作范围”。
  • 生产实践建议两层同时开启：mTLS 解决“谁可以连进来”，RBAC 解决“连进来后能做什么”。
  • 常见代码疑问：业务侧 transport.TLSInfo 未显式设置 ClientCertAuth（默认 false）通常不影响作为 client 访问 etcd；该字段主要用于服务端配置语义。
  • 对 client 而言，关键是是否正确提供 CertFile/KeyFile（向服务端出示客户端证书）以及 TrustedCAFile（校验服务端证书）；服务端是否强制校验客户端证书由 etcd 的 client-cert-auth: true 决定。
  • 职责边界：TLS 握手流程（何时发证书、如何校验证书链、握手报文行为）主要由 Go crypto/tls 实现与决定。
  • etcd 的作用是把配置转成 tls.Config（例如证书路径、受信 CA、是否要求对端证书）；再由 crypto/tls 按该配置执行具体握手。
  • 可理解为：“TLS 行为引擎”是 crypto/tls，“策略与参数输入”由 etcd 配置/代码提供。

常见理解误区 ##

  1. 节点私钥 ≠ CA 私钥 — etcd-key.pem 只配对 etcd.pem；ca-key.pem 只配对 ca.pem / new-ca.pem，二者模数不同。
  2. 证书里没有私钥 — 私钥只存在于 *-key.pem 文件中，不能从证书反推私钥。
  3. 看不到 ca-key.pem 是正常的 — CA 私钥仅在签发/续签时使用，不应出现在 etcd 节点上。
  4. 节点证书与 CA 证书职责不同 — etcd.pem 标识节点身份；new-ca.pem 用于验证对端证书的信任链。
  5. 私钥里出现 n/e 不代表它是证书 — RSA 私钥结构会包含公钥参数，但没有 Issuer、有效期、SAN、签名等证书字段。
  6. 证书不等于“只有公钥” — 公钥只是证书的一部分，证书本体还包含身份信息、有效期、扩展用途和签名。
  7. openssl verify 报错不一定等于业务不可用 — 同 DN 等边界场景下可能出现工具误判，需结合 etcd/Go 实际验签结果判断。
