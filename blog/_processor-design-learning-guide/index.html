处理器设计学习指南 #

个人学习参考文档，目标是"吃透处理器设计，至少核心理念理解到位"。 本文档基于对 BOOM (Berkeley Out-of-Order Machine) 文档的评估， 梳理了学习现代处理器设计所需的核心概念、推荐资源和学习路径。

0. 为什么选 BOOM 作为入门 ##

特点说明
开源可综合可以读 Chisel 源码理解硬件实现
RISC-V ISA指令简单，每条指令拆 1 个 uop，无微码
7 级流水线结构清晰，x86 动辄 20+ 级难以入门
完整 OoO包含 ROB、寄存器重命名、Issue Queue、LSU 等核心结构
文档友好每个子系统有专门章节，篇幅适中

核心策略：先用 BOOM 建立 OoO 流水线的完整心智模型（1-2 天），再对照 Intel 手册和教材补充 x86 特定内容。

3. 关键概念详解 ##

3.1 寄存器重命名：OoO 的基石 ##

解决的问题：消除 WAW (Write-After-Write) 和 WAR (Write-After-Read) 假依赖。

逻辑寄存器 (x1, x2, ...  ← RISC-V 32 个)
     │
     │  Rename Map Table: x1 → p5, x2 → p7, ...
     ▼
物理寄存器 (p0, p1, ...  ← BOOM 有 100+ 个)

关键数据结构：

  • Rename Map Table：每条指令 decode 时，逻辑目标寄存器映射到空闲物理寄存器；逻辑源寄存器查找对应的物理寄存器
  • Free List：跟踪哪些物理寄存器是空闲的
  • Busy Table：跟踪物理寄存器的"写端口"是否忙（指令写回后变空闲）

误预测时的恢复：BOOM 在每条分支 rename 时拍 Rename Map Table 快照，误预测时恢复快照。

3.2 ROB 与 In-Order Commit 的意义 ##

为什么 stores 不能在 execute 时就写入内存？

因为 stores 是推测执行的——如果 store 在一条误预测的分支下，写内存是不可逆的副作用。所以：

  1. Execute 阶段：store 只计算地址，放入 STQ (Store Queue)
  2. Commit 阶段：store 到达 ROB 头部（确认不是推测路径），才真正写入内存

STQ (Store Buffer) → L1 写入的过程是不可逆的，只能在 commit 时执行。

这也解释了为什么 Store Buffer 中可能有未提交的 store——它们还在等 ROB commit。

3.3 Load/Store Unit 与内存排序失败 ##

BOOM LSU 的核心逻辑：

LDQ (Load Queue)          STQ (Store Queue)
分配于 Decode 阶段         分配于 Decode 阶段
执行时发射 load            仅在 Commit 时发射 store

Load 的处理流程：

1. Load 到达 LSU → 乐观发射到内存
2. 同时比较：load 地址 vs STQ 中所有 older store 的地址
3. 匹配？
   → YES: 如果 store 数据已就绪 → 转发给 load（Store-to-Load Forwarding）
          如果 store 数据未就绪 → load 休眠，稍后重试
   → NO:  load 从内存读数据（可能读到脏数据！）

内存排序失败的场景（load 在 older store 之前执行了）：

store A=1 (STQ 中，未 commit)
load  A   (在 store 前发射，从内存读到旧值 0)
store commit 后检查 LDQ → 发现 load A 已经读了脏数据
→ 冲刷流水线，从 load 处重新执行！

这就是内存屏障在微架构层面的开销来源：sfence 排空 STQ，mfence 等所有 load 完成。

3.4 内存一致性模型 ##

模型示例架构允许重排要点
SC (Sequential Consistency)理想模型不允许任何重排编程最容易但性能最差，现代处理器几乎不用
TSO (Total Store Order)x86允许 Store→Load 重排Store Buffer 导致：写操作先入 buffer，后面的读可能先执行
RVWMO (Weak Memory Order)RISC-V大量重排需要 FENCE 指令显式约束；BOOM 使用此模型
RelaxedARM, Power更多重排Store→Store, Load→Load, Store→Load, Load→Store 全可重排

x86 TSO 的关键特性：只有 Store→Load 可以重排。这是因为 Store Buffer 的存在。你的三篇文章的核心话题（Store Buffer、内存屏障、LOCK 前缀）都在 TSO 模型的框架下。

3.5 为什么 LOCK 前缀是"全屏障" ##

LOCK CMPXCHG [addr], reg

硬件做的事情：
1. 排空 Store Buffer（所有 pending store 先 commit）
2. 加载缓存行到 L1，置为 M 状态 (RFO — Read For Ownership)
3. 锁定缓存行（LOCK# 信号，阻止其他 agent 读/写）
4. 在 M 状态缓存行上执行原子 RMW (Read-Modify-Write)
5. 解锁

开销排序：缓存锁 << 总线锁，但都比普通 load/store 贵得多

BOOM 不涉及 LOCK 前缀（RISC-V 用 LR/SC 或 AMO 指令）。这部分需要 Intel SDM Vol 3 Ch.8/11 + Agner Fog 的微架构手册来弥补。

3.6 MESI 协议核心状态 ##

      M (Modified)
     /  \
    /    \
   E      S (Shared)
    \    /
     \  /
      I (Invalid)
转换触发事件总线消息
I→E (或 S)读 missBusRd (读请求)
I→E写 missBusRdX / RFO (独占读)
E→M本核写命中 E 状态无外部消息
M→S其他核读 M 状态行写回 (Write-back) + 变 S
S→I其他核写本缓存行BusUpgr (升级) 导致 Invalidate
M→I其他核写本缓存行写回 (Write-back) + 变 I

LOCK 前缀的"缓存锁"模式：锁定缓存行本质上就是保持该缓存行在 M 状态，阻止其他核通过一致性协议访问它，直至原子操作完成。

4. 与其他主题的关联 ##

学习处理器设计是为了深入理解底层，而底层本质上是为上层软件服务的。以下是学习过程中自然产生的跨层联系：

软件层概念对应的硬件机制
Java volatile 写Store 后 + StoreLoad 屏障 (mfence/lock)
Java volatile 读Load 后 + LoadLoad + LoadStore 屏障
Go sync/atomic直接编译为 LOCK 前缀指令 (amd64)
C stdatomicmemory_order_seq_cst → mfence 或等效
pthread_mutex_lockCMPXCHG + LOCK 前缀 → 缓存锁
RCU (Read-Copy-Update)利用 Store Buffer + 宽限期 (grace period)
memcpy 大块非临时存储 (NT stores) 绕过缓存，用 WC buffer
无锁队列 (SPSC Queue)依赖 TSO 模型只重排 Store→Load 的特性
false sharing 性能下降MESI 中两个变量在同一缓存行的 Invalid bounce

第 1 周：流水线 + 乱序执行基础
├── BOOM Pipeline Overview    (1 天)

├── BOOM LSU                  (1 天)  ← 直接对应你的三篇文章

├── Intel SDM Vol 3 Ch.11    (查阅)
└── 目标：理解 Store Buffer 为什么存在，TSO 为什么是 x86 的选择

├── Intel SDM Vol 3 Ch.8     (查阅 LOCK 部分)
