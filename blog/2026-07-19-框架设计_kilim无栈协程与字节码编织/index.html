Kilim：Java 字节码编织实现无栈协程

Posted 2026-07-19 17:36 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 #

常用并发编程模型大致两类：并发 workers（线程池干活）和 pipeline（actor / channel 传消息）。线程创建、销毁、切换的代价相对固定，在 网络 I/O 密集 的服务里，往往大量时间卡在阻塞上，CPU 却吃不满。

协程把「可调度的执行单元」再拆细一档：创建、销毁、切换开销远小于线程，同一 OS 线程上可以挂成千上万个逻辑任务。问题是——Java / JVM 并不原生提供协程语义。在 Project Loom 普及之前，业界常见做法是 字节码增强（weaving）：编译后改 class，让方法可暂停、可恢复。

Kilim 就是这条路上的代表作之一。它是基于 actor 的并发框架：把线程进一步细分为协程（Task），用 Mailbox 做任务间通信，用 Weaver 对抛出 Pausable 的方法织入特殊字节码。本文整合当年的学习笔记，从模型、核心类到字节码与 CPS 思想，说明它如何在 无栈协程 语义下工作。

相关旧笔记见仓库 content/blog/middlewares/.../Coroutine/Kilim/；本文做了合并与重写，并补上与后续 Quasar / 虚拟线程对照的视角。

一、Kilim 在解决什么问题 #

维度传统线程模型Kilim 目标
执行单元java.lang.Thread / 线程池 Task轻量 Task（协程）
阻塞代价阻塞即占用 OS 线程可 pause Fiber，释放载体线程
通信共享内存 + 锁，或队列Mailbox（多生产者、单消费者）
编程形态回调 / Future 拆分控制流近似同步写法，I/O 处可暂停

一句话：在 不改 JVM 指令集、不依赖原生线程栈切换 的前提下，用 用户态调度 + 字节码改写，做出「看起来同步、实际可挂起」的并发模型。

二、核心抽象 #

2.1 Task：协程实体 ##

自定义协程需 继承 Task，在 execute() 中抛出 Pausable（检查型「伪异常」）。

Pausable 不是 运行时真要抛给业务处理的异常，而是给 kilim.tools.Weaver 用的 标记：

  • Weaver 通过签名 / 反射识别「可暂停方法」；
  • 对这些方法 编织（weave） 特殊字节码；
  • 运行期才能在指定点 暂停，并在之后 从同一逻辑位置恢复。

业务方法若需要 pause/resume 能力，定义时同样要声明抛出 Pausable。于是「可暂停性」沿调用链向上传播——这和后来 Quasar 的 @Suspendable、C#/JS 的 async 传染性是同一类问题。

2.2 Fiber 与 State：无栈现场 ##

Java 跑在 VM 上，不能像 C 协程那样直接 swapcontext 保存物理寄存器与整段栈。Kilim 的做法是 stackless（无栈）协程：

  • 多个协程 共享 当前载体线程的 JVM 栈；
  • 真正属于某个协程的「现场」，落在 Fiber / State 等堆上结构里。

要点：

组件作用
Fiber模拟一组活动记录（activation records），串起各 Pausable 方法的 pc / state
Statepc：当前停在第几个可暂停点；Object 等字段寄存恢复后仍需要的局部量

直觉上可以这样理解：

有栈协程：每个协程一块（或一段）私有栈，切换时改栈指针 + 寄存器。
无栈协程：yield 时 显式保存「调用关系 + 关键局部变量 + 恢复点」；resume 时 按保存的状态重新走调用链，在正确的 pc 处继续。

Kilim Weaver 会给每个 Pausable 方法 追加 Fiber 参数（整条可暂停调用链共享）。Fiber 通过 down() 进入下一层 Pausable 栈帧语义，返回时用 up() 回到上层。这就是在字节码层面「手工维护」协程帧。

2.3 Scheduler：把 Task 绑到线程池 ##

Scheduler 维护线程池，把 Task 分发给工作线程；每个线程有自己的任务队列，被调度时取 Task 执行。

在 HotSpot + Linux 上，平台线程是 1:1 映射到内核线程 的（现代实现如此；笔记里写的 1:m 是早期/广义用户线程模型说法，今日可理解为「用户态 Task 再多路复用到少量平台线程」）。协程调度发生在 用户态 Scheduler，真正占 CPU 的仍是底下的 OS 线程。

2.4 Mailbox：Task 之间的信道 ##

Task 间通信靠 Mailbox（类似 Go 的 channel）：多生产者、单消费者的消息队列，可放任意对象。

访问语义可粗分为（名称以 Kilim API 为准）：

操作族行为倾向
get / put不阻塞线程、也不 pause fiber（视实现与队列状态，语义偏「尝试」侧）
getb / putbblock 线程——代价重，尽量避免
getnb / putnb非阻塞线程；配合 fiber pause 才是协程友好路径

设计意图很清楚：协作式让出 用 pause fiber；真正堵死 OS 线程 是退化路径。

三、字节码为什么是关键路径 #

3.1 JVM 字节码基础（与 Weaver 的关系） ##

JVM 指令是 一字节 opcode，约 200 个在用。业务代码编译成 class 后，结构可用 javap 观察：

javac HelloWorld.java
javap -c -verbose HelloWorld.class

-verbose 会展开常量池、descriptor、stack / locals / args_size 等。Kilim / ASM 做的事，本质上是在这类结构上 改方法描述符、插入保存/恢复局部变量的指令、插入按 pc 跳转的表驱动控制流。

ASM 等库降低了「改 class」的工程成本，但 插哪些指令、在什么控制点插，仍由协程语义决定——Weaver 才是 Kilim 的编译器后端。

3.2 Weaver 做了什么（概念模型） ##

对每个 Pausable 方法，大致要完成：

  1. 签名改写：增加 Fiber 参数（或等价约定）；
  2. 状态机化：把方法体按 suspend 点切成多段，用 pc 选择从哪一段恢复；
  3. 保存活跃变量：pause 前把后续仍需要的 locals 写入 State；
  4. 恢复路径：resume 时从 State 读回，再 goto 正确 continuation。

用户源码仍可写成近似同步风格：

// 逻辑形态（示意）
void execute() throws Pausable {
    Msg m = mailbox.getb();   // 可能 pause
    process(m);
    reply.putb(result);       // 可能 pause
}

织入后，运行时在 getb 等点可能 返回到调度器，而不是占用线程傻等。

四、CPS 思想：为什么「无栈」还能恢复 #

4.1 Continuation 与 CPS ##

普通调用：压栈 → 执行 → 弹栈返回。若函数 a 对 b 的调用是 尾调用，理论上可复用栈帧；更一般地，若「算完之后做什么」也作为参数传下去，就得到 Continuation Passing Style（CPS）：

  • continuation ≈ 「后续计算过程 + 所需状态」；
  • 调用不再依赖「返回后继续」，而是 显式把 continuation 传给被调方。

Kilim 借鉴 CPS：用额外的 Fiber 参数携带协程栈状态（pc、各层 State），在 pause 时固化 continuation，在 resume 时继续。

4.2 Kilim 并未「丢掉 JVM 栈」 ##

重要补充：Kilim 不是 在 JVM 上彻底取消栈。JVM 执行模型依赖操作数栈与调用栈；作者也观察到完全绕开栈并不划算。实际策略是：

  • 运行时 仍用当前线程的 JVM 栈执行；
  • 挂起时 把逻辑调用链上的关键状态 拷贝/编码 到 Fiber 相关结构；
  • 恢复时 再通过织入的代码 重建 到正确的逻辑点。

所以它是 语义上的无栈协程（协程不长期持有独立完整栈），实现上是 字节码级状态机 + 共享载体栈。

4.3 和「有栈协程」对照一眼 ##

有栈（如 Go、libmill）无栈（Kilim）
每协程栈独立栈（可分段增长）无长期私有栈
切换改 SP/上下文保存/恢复 State + 调度
任意深度 yield相对自然调用链须全程可织入
与本地/JNI较直接未织入代码、本地帧难 pause

五、工程侧的其他边角 #

Kilim 代码里还有 HTTP 相关路径，协议解析用过 Ragel：用正则/状态机描述生成 C/C++/Java 等解析代码。这与协程主线无关，但说明框架目标是 高并发网络服务——协议状态机 + 协程调度是常见组合。

状态机实现的经典骨架（与 Ragel 生成物同构）可概括为：状态函数表 + 转移表，每次根据返回码查下一状态。协程的 pc 跳转，思想上与「显式状态机」是亲戚。

六、局限与历史位置 #

Kilim 的价值与代价都清晰：

价值

  • 在 Java 7 时代给出可用的 actor + 轻量任务 模型；
  • 用 Weaver 证明：无栈协程可以在字节码层落地；
  • Mailbox 对齐 CSP/actor 通信习惯。

代价 / 局限

  • 可暂停性传染：调用链上方法都要 Pausable + 织入；
  • 工具链负担：构建期 Weaver、与 IDE/调试的摩擦；
  • 与「普通」库互操作：第三方阻塞 API、未织入 class 会破坏模型；
  • 生态演进：后续 Quasar 走类似 instrumentation 路线；再往后 Project Loom 虚拟线程 把「轻量线程」做进 JDK，第三方织入方案的历史使命逐步结束。

即便如此，读 Kilim 仍有收益：它把 Fiber、State、pc、CPS、调度器、邮箱 拆得足够裸，是理解「JVM 上如何假扮协程」的极好标本。

总结 #

Kilim 用 Task + Scheduler + Mailbox 搭起 actor 式并发，用 Pausable 标记 + Weaver 字节码编织 + Fiber/State 实现 无栈协程 的 pause/resume。本质不是换一套 OS 线程，而是：

  1. 在用户态多路复用少量平台线程；
  2. 在 class 文件里把可暂停方法改成 显式状态机；
  3. 用堆上的 continuation 信息描述「协程现场」。

若你接着要看同路线的另一实现，可阅读同目录下的 Quasar 一文；若要对照 有栈 实现（Go、libmill）与 语言原生 方案（Java 虚拟线程、C++20 coroutine），见「有栈与无栈协程对比」一文。
