有栈协程与无栈协程：实现异同、优劣与典型代表

Posted 2026-07-19 17:40 +0800 by ZhangJie ‐ 3 min read

分享:  

背景 #

协程的本质只有一句话：控制流可以主动让出（yield），并在之后从让出点恢复（resume）。
实现这句话的路径却分成两大流派：

流派英文典型代表
有栈协程stackfulGo goroutine、libmill、libco、许多 C 协程库
无栈协程stacklessC++20 coroutine、C# async/await、Kilim、Quasar、Python generator（某种意义上）

无栈往往 更难建立直觉——源码仍是顺序函数，运行时却是状态机；有栈则更像「缩小版线程」。本文把两边的实现异同、优缺点和代表技术放到同一张图上，方便选型与读源码。

同系列已有 Kilim、Quasar、Java 虚拟线程、C++20 的分篇介绍；本文侧重 横向对比。

一、先对齐「栈」指什么 #

这里的「栈」不是 JVM 操作数栈的细节，而是：

协程在挂起时，是否拥有（或等价于拥有）一条 可独立切换的调用栈。

  • 有栈：每个协程有自己的栈空间（固定大小、或可增长的分段栈）。切换 ≈ 保存寄存器与栈指针，换到另一套。
  • 无栈：协程 不 长期持有完整私有调用栈；挂起时只把 恢复所需状态（局部变量、程序计数/状态号、后续续体）放到堆上帧里；多个协程 复用 当前线程的宿主栈。

两者都能实现「让出/恢复」，但 谁可以 yield、切换多贵、和 C 调用/本地代码怎么相处 差别很大。

二、有栈协程：像线程，但不经内核 #

2.1 工作原理（示意） ##

创建协程：分配栈 + 初始上下文（入口 PC、SP 等）
运行：  把 CPU 上下文指到该栈，执行用户函数
yield：  保存当前寄存器/SP/IP 到协程控制块
        恢复调度器或其他协程的上下文
resume： 对称操作

在 Linux/C 里常见积木：ucontext、汇编切换、setjmp/longjmp 变体、调整栈指针后「跳进」函数等。

2.2 libmill：C 语言有栈 + CSP ##

libmill（Martin Sustrik）用较少代码实现了 Go 风格 的协程与 channel：「通过通信共享内存」。

有栈相关的经典技巧包括：在运行时 分配一段内存当协程栈，配合上下文切换让 go(fn) 在新栈上跑起来。本仓库亦有源码分析草稿：languages/cc++/2017-12-03-go风格协程库libmill之源码分析.md（其中对 mill_go_、栈指针与 setjmp 一类路径有说明）。

特点：

  • 业务函数 不必 改成 async 风格；
  • 调用栈多深都可以在阻塞点 yield（只要运行时在该点切入调度）；
  • 与「普通 C 函数」互操作相对自然。

2.3 Go goroutine：语言 + 运行时把有栈做到极致 ##

Go 的 g 是 有栈 模型，但栈是 可增长 的（历史成分段栈，现为连续栈拷贝增长等策略），并有：

  • 用户态调度（G-M-P）；
  • 系统调用、channel、网络 poller 上的挂起点；
  • 日益增强的抢占，避免纯计算饿死调度。

对开发者：go func() 几乎零仪式感——这是有栈 + 全运行时支持的红利。

2.4 有栈的优点 ##

  1. 心智模型简单：就是轻量线程；
  2. yield 位置灵活：不必把整条调用链改成「可挂起函数」；
  3. 与现有同步代码、部分本地库更合拍；
  4. 调试栈 往往更接近真实调用关系。

2.5 有栈的缺点 ##

  1. 每协程栈的内存：即便初始很小，海量协程仍占可观虚拟/物理内存；增长栈有拷贝或映射成本；
  2. 切换要保存的上下文 通常多于无栈「只存活跃变量」的理论最小集；
  3. 在 JVM 等托管运行时上难做——不能随便改 SP 骗过 VM/GC（所以 Java 上 Kilim/Quasar 走了无栈织入）；
  4. 和宿主语言异常、栈展开、信号 的交互要小心实现。

三、无栈协程：状态机，而不是小线程 #

3.1 工作原理（示意） ##

编译器或织入工具：
  把函数按挂起点切成多段
  把跨挂起仍存活的局部变量放到「协程帧」（堆）
  用 state / pc 记录下一跳

运行时：
  调用协程 ≈ 执行状态机的一个片段
  co_await / pause ≈ 存状态，return 到调度器/调用方
  resume ≈ 再进状态机，switch(pc) 继续

没有「整段私有调用栈」可切——所以 不能在任意普通函数中间 挂起，除非它也被编译/织成协程。

3.2 Java：Kilim 与 Quasar ##

二者都是 无栈 + 字节码增强：

KilimQuasar
标记throws Pausable@Suspendable
状态载体Fiber / StateFiber + instrumentation
通信MailboxChannel 等
时代较早的 actor 框架更工程化的 Fiber 栈

共同点：

  • 可挂起性 向上传染；
  • 构建期/Agent 织入；
  • 共享载体线程的 JVM 栈，挂起时靠堆上状态恢复。

这比有栈更绕，因为你必须同时理解：Java 语法、字节码形态、调度器、以及「假异常/注解只是标记」。

3.3 C++20：语言正规军的无栈 ##

co_await / co_yield / co_return 由 编译器 生成协程帧与状态机；promise_type 与 awaitable 定制行为。标准 不 送完整 Go 式调度器。

理解难度高的原因：

  1. 机制与策略分离，文档碎片化；
  2. 对象生命周期与挂起交错（引用易悬空）；
  3. 调试时「栈」不连续；
  4. 要搭实用系统还得再学一个异步运行时。

3.4 无栈的优点 ##

  1. 内存可更紧：帧只保留活跃状态，不必预留完整调用栈；
  2. 切换理论上可更轻（只碰必要状态）；
  3. 适合深度嵌入语言（async/await 语法糖、生成器）；
  4. 在 JVM 上可行——不靠改原生 SP。

3.5 无栈的缺点 ##

  1. 传染性 / 颜色函数（function coloring）：async 与 sync 分裂；
  2. 实现与理解成本高（状态机、帧、await 协议）；
  3. 与未改写代码、JNI/本地帧 不友好；
  4. 调用栈可视化、性能分析 更绕；
  5. 若库生态不齐，易写出「半同步半回调」的大杂烩。

四、并排对比表 #

五、Java 虚拟线程放哪一侧？ #

Project Loom 虚拟线程 常被称作「Java 协程」，但归类要谨慎：

  • 对开发者：它是 轻量 Thread，不是 co_await 状态机 API；
  • 对实现：阻塞时 unmount，把执行状态（continuation / 栈段）从载体上卸到堆管理结构，事件就绪再 mount；
  • 相对 Kilim/Quasar： 没有 用户级 @Suspendable 织入，更不像经典无栈「方法染色」；
  • 相对 Go：同属 运行时 M:N 轻量线程 阵营，栈/continuation 由 VM 管理。

更公允的说法：

虚拟线程是 平台实现的轻量线程（带可迁移执行状态），解决问题的集合与协程高度重叠，但 不是 C++20/Kilim 那种无栈协程语法与模型。

对比教学时建议单独一列，避免和 Quasar 画等号。

                    用户 API 像线程？
                   /                \
                 是                  否（async/await、Pausable…）
                /                      \
        运行时 M:N                   编译器/织入状态机
       /         \                      /        \
     Go        Java VT            C++20      Kilim/Quasar
   (有栈)   (continuation/       (无栈)       (无栈)
            栈段，偏轻量线程)

六、为什么无栈往往更难懂？ #

结合工程经验，难点通常不在「yield 两个字」，而在：

  1. 双重语义：源码顺序执行 vs 编译后状态机；
  2. 状态放哪：哪些局部变量跨挂起、生命周期到哪；
  3. 谁 resume 我：调度器？I/O 线程？调用方？；
  4. 染色：为什么不能在普通工具函数里随便挂起；
  5. 错误栈：崩溃时栈轨迹与「逻辑调用链」不一致。

有栈模型用「每任务一个栈」把 1–4 大部分藏进了运行时，开发者付的是 内存与切换 账单，而不是 心智与工具链 账单。

七、怎么选（实用建议） #

场景更自然的选择
新写 Go 服务goroutine（有栈，别重复造轮子）
C 要 Go 风格 CSPlibmill / libdill 一类有栈库
C++ 要与 Asio/异步 I/O 深度集成C++20 无栈 + 成熟库
C++ 要「像线程一样切」的旧代码有栈协程库（评估与标准协程混用成本）
现代 Java I/O 密集虚拟线程（Java 21+）
读历史/做字节码研究Kilim、Quasar
生成器/流水线/编译器管道无栈或半协程（generator）往往够用

原则：

  • 要吞吐与同步写法，且语言有官方轻量线程：用官方的（Go、Java VT）；
  • 要零开销挂起点、可嵌入任意执行器：无栈语言机制（C++20、async）；
  • 要在无运行时支持的语言里快速模拟线程：有栈库往往更快落地。

八、一张总图（代表技术） #

技术栈模型挂起如何实现调度
libmill有栈上下文/栈切换库
Go有栈（可增长）runtime parkGo runtime
Kilim无栈Weaver + Fiber/State库

总结 #

  • 有栈协程：每任务一栈，切换像轻量线程；代表 Go、libmill。易懂、易与同步代码结合，成本在内存与原生实现难度。
  • 无栈协程：挂起只存帧状态，复用宿主栈；代表 C++20、Kilim、Quasar。省栈、宜做语言特性，但有 函数染色 与更高理解成本。
  • Java 虚拟线程：平台级轻量线程，结束织入式 Fiber 的历史使命，模型上更靠近 Go 而非 C++20。
  • 无栈更难懂，是因为它把「线程栈」拆成了 显式状态机；一旦把状态机、帧、resume 谁来调这三件事想通，两边其实是同一句协程定义的两种工程答案。

建议阅读顺序：本文建立坐标系 → Kilim / Quasar（Java 无栈史）→ 虚拟线程（Java 现在）→ C++20（语言无栈）→ 需要时再下钻 libmill/Go runtime 源码。
