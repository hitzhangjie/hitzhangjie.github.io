Git 对象数据库：从原理到实操

Posted 2025-10-29 00:00 +0800 by ZhangJie ‐ 4 min read

分享:  

Git 的核心是一个"内容寻址文件系统"（content-addressable filesystem）。你在仓库里的每一次 git add、git commit，都会在 .git/objects/ 下生成一些文件——Git 管它们叫对象（object）。

这些对象只有四种类型。所有文件内容、目录结构、提交历史、版本标签，都由它们组合而成。这篇先讲清楚这四种对象是什么、怎么组织成项目历史的，再用一个最小化的实操实验从头走一遍，验证每一步生成了什么。

四种对象类型 ##

Git 对象数据库只有四种对象：

类型存什么类比
blob文件内容（原始字节流），不存文件名和路径文件
tree目录清单：文件名、权限，及其对应的 blob 或子树目录
commit一次快照的元数据：指向哪个 tree、作者、时间、父 commit、提交说明快照
tag语义化引用，指向某个 commit 或其他对象，常用于标记发布版本标签

tag 分两种：轻量标签（lightweight tag）就是一条固定的 SHA-1 引用，不是独立对象；注解标签（annotated tag）才是独立的 tag 对象，包含作者、信息、签名。日常 git tag v1.0 创建的是轻量标签，加上 -a 才是注解标签。

这四种对象加在一起，能完整描述一个项目的所有版本历史：文件内容归 blob，目录结构归 tree，版本快照归 commit，发布标记归 tag。

对象之间怎么组织 ##

理解对象类型不难，关键是它们之间的引用关系。

引用链 ##

tag → commit → tree → blob
                  → tree → blob

一层层拆开：

  • commit → tree：每个 commit 指向一个 tree，这个 tree 就是该提交时的工作目录根快照。
  • commit → parent(s)：commit 有 0 个（初始提交）、1 个（普通提交）或多个（merge 提交）父 commit，形成历史链条。
  • tree → blob / tree：tree 里列出文件对应哪个 blob、子目录对应哪个子 tree。
  • tag → commit：注解标签指向某个 commit（也可以指向其他对象，但绝大多数情况是指向 commit）。

DAG：有向无环图 ##

所有 commit 对象通过父指针连在一起，形成一个有向无环图（DAG，Directed Acyclic Graph）。

        commit D (merge)
        /          \
   commit C       commit B
       |              |
   commit A ←──── commit A (同一个)
  • 普通提交让 DAG 线性延伸。
  • git merge 产生多父 commit，DAG 分叉再合并。
  • git branch 只是创建一个指向某个 commit 的可移动指针（refs/heads/xxx），不创建新对象。

Git 通过遍历这个 DAG，按拓扑顺序恢复项目的完整历史。git log 本质上就是在遍历这个图。

物理存储 ##

对象实际存在 .git/objects/ 下，分两种形态：

  • 松散对象（loose objects）：每个对象单独压缩成一个文件，路径如 .git/objects/ab/cdef...——SHA-1 hash 的前 2 位做目录名，后 38 位做文件名。
  • 打包文件（packfiles）：仓库大了之后，Git 通过 git gc 把大量松散对象压缩打包到 .git/objects/pack/pack-*.pack，内部做增量压缩（delta compression），同时生成 .idx 索引文件加快查找。

packfile 的增量压缩只是存储层优化，不影响 Git 对象模型的概念语义——从概念上讲，每个 commit 仍然是完整快照。

另外还有 reflog（.git/logs/），记录 refs 的变更历史，用于恢复误删分支或撤销 git reset 等操作。它不属于对象数据库，但也是理解 Git 数据模型的重要部分。

实操验证 ##

概念说完了，动手验证。从头走一遍 git init → git add → git commit → 子目录，看 .git/objects/ 里每一步多了什么对象，再用 git cat-file 把内容扒出来。

准备：干干净净开始 ##

$ mkdir git && cd git
$ git init
$ touch f1
$ tree .git/objects
.git/objects
├── info
└── pack          # 只有两个空子目录

刚 git init 完，objects/ 下只有 info 和 pack 两个空目录，没有任何松散对象。

git add：生成 blob ##

$ git add f1
$ tree .git/objects
.git/objects
├── e6
│   └── 9de29bb2d1d6434b8b29ae775ad8c2e48c5391
├── info
└── pack

多了一个 e6/9de29b... 文件，这就是一个 blob 对象。验证一下：

$ git cat-file -t e69de29bb2d1d6434b8b29ae775ad8c2e48c5391
blob
$ git cat-file blob e69de29bb2d1d6434b8b29ae775ad8c2e48c5391
# 空输出 —— f1 是空文件

git add 做的事：读取文件全部内容，压缩后算 SHA-1 hash，存到 objects/ 下。对象类型是 blob——只存内容，不存文件名、路径、权限。

e69de29bb2d1d6434b8b29ae775ad8c2e48c5391 是空内容的固定 hash。在任何一个 Git 仓库里，echo -n '' | git hash-object --stdin 都会输出这个值。

修改不 add：什么都没发生 ##

$ echo "hello" > f1
$ git cat-file blob e69de29bb2d1d6434b8b29ae775ad8c2e48c5391
# 还是空输出

工作区改了 f1，但 Git 对象数据库毫不知情。工作区的修改，不经过 git add 就不会生成任何 object。

再次 add：新 blob 生成，旧的不删 ##

$ git add f1
$ tree .git/objects
.git/objects
├── ce
│   └── 013625030ba8dba906f756967f9e9ca394464a    # 新 blob，"hello\n"
├── e6
│   └── 9de29bb2d1d6434b8b29ae775ad8c2e48c5391    # 旧 blob，空内容
├── info
└── pack

$ git cat-file blob ce013625030ba8dba906f756967f9e9ca394464a
hello

两个 blob 同时存在。旧 blob 没有被删除——这是 Git 对象模型的核心规则：对象一旦创建就不可变（immutable），只会追加新的，不会修改或删除已有的。

git commit：生成 tree 和 commit ##

$ git commit -m 'add f1'
$ tree .git/objects
.git/objects
├── b9
│   └── 4fe35f6b9fb42474cece84a668e6788c756803    # tree
├── ce
│   └── 013625030ba8dba906f756967f9e9ca394464a    # f1 的 blob
├── d1
│   └── ab65bb06cb7a01bca305f4740426a5228c766a    # commit
├── e6
│   └── 9de29bb2d1d6434b8b29ae775ad8c2e48c5391    # 旧 blob
├── info
└── pack

多了两个对象。逐个看：

# commit 对象
$ git cat-file -t d1ab65bb06cb7a01bca305f4740426a5228c766a
commit
$ git cat-file commit d1ab65bb06cb7a01bca305f4740426a5228c766a
tree b94fe35f6b9fb42474cece84a668e6788c756803
author hitzhangjie <hit.zhangjie@gmail.com> 1761797659 +0800
committer hitzhangjie <hit.zhangjie@gmail.com> 1761797659 +0800

add f1

# tree 对象
$ git cat-file -t b94fe35f6b9fb42474cece84a668e6788c756803
tree
$ git cat-file -p b94fe35f6b9fb42474cece84a668e6788c756803
100644 blob ce013625030ba8dba906f756967f9e9ca394464a    f1

三种对象的关系一目了然：

commit (d1ab65...)
  └── tree (b94fe35...)
        └── blob (ce0136...)  ← f1 的内容 "hello\n"
  • blob：只存内容，不知道文件名。
  • tree：存目录清单，把 f1 这个名字和 blob ce0136... 对应起来。100644 是 Unix 文件权限模式。
  • commit：指向这次快照的根 tree、记录作者时间信息。因为是第一次提交，没有 parent。

加入子目录：tree 指向 tree ##

再加点复杂度。创建三个目录 a、b、c，各放一个空文件：

$ mkdir a b c
$ touch a/f1 b/f1 c/f1
$ git add a b c
$ git commit -m 'add folders'

objects 下又多了几个对象。看下新的 root tree：

$ git cat-file -p 4f6a3e763c056f24f5a019591dfa0d280fd721b3
040000 tree 56e3dd6f60494c9bbe56ea178b9a86c91d3139c6    a
040000 tree 56e3dd6f60494c9bbe56ea178b9a86c91d3139c6    b
040000 tree 56e3dd6f60494c9bbe56ea178b9a86c91d3139c6    c
100644 blob ce013625030ba8dba906f756967f9e9ca394464a    f1

两个发现：

第一，tree 可以递归指向子树。a、b、c 的类型是 tree（040000），因为它们本身是目录。子 tree 的内容：

$ git cat-file -p 56e3dd6f60494c9bbe56ea178b9a86c91d3139c6
100644 blob e69de29bb2d1d6434b8b29ae775ad8c2e48c5391    f1

**第二，三个子目录的 tree hash 完全相同。**都是 56e3dd6f...。因为它们内容一样——都只包含一个空文件 f1。这就是内容寻址的自然结果：相同内容 → 相同 hash → 只存一份。不管是 blob 还是 tree，Git 天生自带内容级去重。

这次提交后的对象图：

commit (3c9126...)
  └── tree (4f6a3e...)
        ├── tree (56e3dd...)  ← a 的目录清单
        │     └── blob (e69de2...)  ← 空文件
        ├── tree (56e3dd...)  ← b，同一个对象
        ├── tree (56e3dd...)  ← c，同一个对象
        └── blob (ce0136...)  ← f1 的 "hello\n"

常用查看命令 ##

实操中用到的几个命令，总结一下：

命令作用
git cat-file -t <hash>查看对象类型（blob / tree / commit / tag）
git cat-file -p <hash>以可读格式输出对象内容（tree 列清单，commit 显元信息）
git cat-file blob <hash>输出 blob 的原始内容
git cat-file commit <hash>输出 commit 的元信息
git hash-object --stdin从标准输入计算内容的 SHA-1 hash
git log --graph --oneline可视化 DAG 结构

总结 ##

  • 四种对象：blob（文件内容）、tree（目录清单）、commit（快照元数据）、tag（版本标签）。
  • 对象是不可变的：创建后不修改、不删除。历史通过追加新对象向前推进。
  • 组织方式：commit 通过 parent 形成 DAG，tree 递归嵌套描述目录结构，tag 锚定发布点。refs（分支、标签）是指向 DAG 中某个节点的可变指针。
  • 内容寻址即自动去重：相同内容产生相同 hash，相同 hash 只存一份。空文件的 blob、相同内容的子目录 tree，都天然去重。
  • 对象生成时机：工作区修改不生成对象；git add 生成 blob；git commit 生成 tree 和 commit。

理解了这个模型，git checkout、git reset、git rebase 这些操作本质上就是在 DAG 中移动 refs 指针——底层的数据结构其实非常简洁。
