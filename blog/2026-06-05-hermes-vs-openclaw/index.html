Agent 框架对比：Hermes Agent vs OpenClaw

二、架构设计：最底层的哲学分歧 ##

2.1 OpenClaw：中央网关型架构 ##

[WhatsApp] ─┐
[Telegram]  ─┤
[Discord]   ─┼─→ Gateway (WebSocket 长连接) ─→ Task Brain (SQLite Ledger) ─→ Agent 实例 ─→ Skills
[飞书]      ─┤
[钉钉]      ─┘

核心理念：广度优先。OpenClaw 的架构中心是一个长期运行的 Node.js 守护进程（Gateway），把所有消息渠道统一成一套内部协议，消息经过 Task Brain 路由分发到 Agent 实例。类比 K8s 的容器调度思路，所有子 Agent、Cron 任务、后台 CLI 进程全部统一在一个 SQLite Ledger 上。

优势：

  • 接新渠道只需写适配器，核心无改动
  • 多 Agent 主-子编排（master-sub agent）原生支持
  • 跨渠道会话状态统一

代价：

  • 所有会话共享同一进程空间，重度任务抢占 I/O
  • 协议层破坏性变更可能同时影响所有渠道
  • 历史越长，Markdown 全量加载越慢

2.2 Hermes：执行循环型架构 ##

[消息平台] ─→ Agent Runtime ─→ 任务执行 → 结果评估 → 模式提取 → Skill 写入
                    ↑                                                    │
                    └──────────── 下次检索复用 ←──────────────────────────┘

记忆三层：Session Memory → Persistent Memory → Skill Memory (FTS5 + LLM 摘要)

核心理念：深度优先。每个任务跑在独立的执行循环中，消息平台接入只是 Runtime 的外围功能。每个任务经历「输入→推理→工具使用→记忆→输出」的完整闭环。

优势：

  • 任务间互不污染，重复任务越做越高效
  • 六种沙箱后端（local / Docker / SSH / Daytona / Singularity / Modal）隔离执行
  • 自生成 Skill，经验自动沉淀

代价：

  • 接渠道和接模型都需要更多手工配置
  • 前期搭建成本高
  • 一次性非重复任务时学习机制无价值

2.3 一句话对比 ##

维度OpenClawHermes Agent
哲学广度优先深度优先
核心中央网关（Gateway）执行循环（Runtime Loop）
类比万能翻译插头每次任务开新工位
最佳场景多渠道协同长期复杂任务

三、功能完整度逐项拆解 ##

3.1 记忆系统 ##

层级OpenClawHermes Agent
存储格式Markdown 纯文本文件结构化多层存储
架构3 层：工作记忆 → 压缩记忆 → 持久记忆4 层：提示层 → 会话归档层 → 技能层 → 第三方扩展
检索机制向量检索 + 语义激活FTS5 全文检索 + LLM 摘要 + 用户画像建模
加载方式全量加载（历史越长越慢）分层精准调用，按需加载
实测效率基线同 1 个月记录，响应快 60%，Token 成本低 50%
可编辑性支持手动编辑 Markdown结构化编辑，支持 7+ 第三方记忆框架兼容

结论：Hermes 在记忆系统上全面领先，这是其"自我进化"能力的基石。

3.2 技能（Skills）系统 ##

维度OpenClawHermes Agent
技能来源人工编写 / ClawHub 市场下载（17,000+）Agent 运行中自生成 + 自优化
创建门槛需人工编写 SKILL.md零代码，5 次以上工具调用的复杂任务自动打包
进化能力无，依赖社区更新闭环学习，自动迭代优化
重复任务效率每次都从零开始重复任务效率提升 40%
生态规模极大规模（ClawHub + 社区）增长中，社区驱动
核心文件SKILL.md（YAML 元数据 + Markdown 正文）skill.md（自动生成）

结论：OpenClaw 的 Skills 生态规模碾压，但 Hermes 的技能自动生成能力是范式级创新——Agent 不再是"用技能"，而是"长技能"。这是两者最核心的差异之一。

3.3 消息渠道集成 ##

维度OpenClawHermes Agent
渠道数量50+ 平台~20 平台
国内平台飞书、钉钉、QQ、企业微信、微信公众号有限（主要是海外平台）
海外平台WhatsApp、Telegram、Discord、Slack、Signal、iMessage 等Telegram、Discord、Slack、WhatsApp、Signal、CLI
扩展方式官方适配器 + 社区插件需更多手工配置

结论：OpenClaw 在渠道覆盖上绝对领先，尤其对国内用户（飞书、钉钉、企业微信）是刚需优势。

3.4 模型支持 ##

维度OpenClawHermes Agent
海外模型GPT-4o、Claude 3.5 Sonnet、Gemini 等同样支持
国内模型豆包、文心、通义千问、DeepSeek、Kimi、混元等200+ 模型（OpenAI 兼容协议）
本地模型Ollama 兼容Ollama 兼容
模型路由按渠道路由到不同模型按任务类型选择模型

两者模型支持能力基本持平，都能灵活切换云端/本地模型。

3.5 执行沙箱与安全 ##

维度OpenClawHermes Agent
执行方式主进程内执行，依赖 OS 权限6 种沙箱后端隔离执行
CVE 漏洞9 个（含 CVSS 9.9 严重漏洞，2026 年 3 月 4 天内集中暴露）4 个（已修复，零高危）
技能市场安全ClawHub 扫出 341 个恶意技能（约 11.94%）内置五层纵深防御（授权→审批→隔离→审计→告警）
公网暴露13.5 万+ 实例公网暴露高风险操作默认人工审批，超时自动拒绝
敏感数据泄露约 7.1% 社区技能含明文 API 密钥容器隔离 + 权限白名单
风控事件Google Gemini API 大规模封禁（10 万+ 用户受影响）暂无公开风控事件

结论：Hermes 在安全架构上显著领先。OpenClaw 的生态规模反成了安全负担，技能投毒、公网暴露、API 封禁等问题频发。

3.6 多 Agent 协同 ##

维度OpenClawHermes Agent
协同模式主-子 Agent 分布式并行单 Agent 操作者 + 子 Agent 沙箱隔离
编排方式Task Brain 统一调度Agent Runtime 同步编排
人格定制每个 Agent 可独立配置 Soul.md + 权限通过记忆和 Skill 个性化

六、适用人群与场景 ##

6.1 选 OpenClaw 的场景 ##

  1. 团队多平台协作：团队沟通分散在飞书、Slack、WhatsApp、Discord 等多渠道，需要统一 Agent 响应
  2. 国内用户：需要飞书、钉钉、企业微信、QQ、微信公众号等本地化渠道
  3. 开箱即用：不想从零培养 Agent，需要 ClawHub 上现成的社区 Skills
  4. 确定性任务为主：群聊 @ 回复、文件总结、定时周报等不需要长期学习积累的任务
  5. 活跃的中文社区依赖：中文资料、教程、配置案例远比 Hermes 丰富

6.2 选 Hermes 的场景 ##

  1. 单人开发者或研究者：希望 Agent 长期积累工作习惯、代码风格、工具链，越用越顺手
  2. 强隔离需求：需要运行不受信任代码、处理敏感文件、同时运行多独立工作流
  3. 预算极度紧张：$5 VPS 或 Serverless + 自动休眠，长期 Token 成本更低
  4. 重复性深度任务：知识库维护、定期报告生成、长期数据监控等"持续优化型"任务
  5. 对安全有高要求：五层防御体系、沙箱隔离、零高危漏洞记录

6.3 双栈部署 ##

两者并非零和博弈。越来越多开发者在同时使用两个框架：OpenClaw 做多渠道编排入口，Hermes 做深度重复任务执行。Hermes 官方提供了 hermes claw migrate 一键迁移命令，可通过 --preset full --overwrite 完整导入 OpenClaw 的 persona、Skills、记忆和配置。

七、风险与局限 ##

风险维度OpenClawHermes Agent
安全风险技能投毒（11.94%）、公网暴露（13.5 万+）、9 个 CVE自动记忆可能保存过多上下文，需手动设淘汰策略
平台风控Google Gemini 封禁事件影响 10 万+ 用户尚无大规模风控事件
创始人风险创始人已离职加入 OpenAI团队稳定
生态锁定迁移回 OpenClaw 无官方反向工具迁移成本低（正向迁移一键完成）
中文适配成熟渠道覆盖是核心短板
学习曲线上手平缓（1 小时内可跑通），长期维护心智负担重上手陡峭（架构选择多），长期使用心智负担低

八、核心差异总结 ##

            OpenClaw                          Hermes Agent
            ─────────                         ─────────────
哲学：      广度 = 连接一切                    深度 = 越用越聪明
架构：      Gateway 中央路由                   Runtime 执行循环
技能：      人写 → Agent 用                    Agent 用 → Agent 写
记忆：      Markdown 文件 = 会话粘合剂         结构化存储 = 个人知识库
安全：      事后修补（CVE 频发）               设计即安全（五层防御）
生态：      成熟、庞大、有隐患                  爆发式增长、待完善
用户：      团队、多渠道、快速上手              个人、长期、安全优先

一句话总结：OpenClaw 是更好的"AI 调度中心"，Hermes 是更好的"AI 数字分身"。不存在谁"完爆"谁，而是问自己：你的场景是广度优先还是深度优先？
