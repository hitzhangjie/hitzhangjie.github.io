微服务架构设计模式速查手册

Posted 2026-07-24 08:23 +0800 by ZhangJie ‐ 9 min read

分享:  

背景 #

微服务架构不是银弹。它解决了一部分问题（独立部署、团队自治、技术异构），同时引入了另一批问题（分布式事务、服务发现、数据一致性、调试困难）。Chris Richardson 在《Microservices Patterns》中系统化地整理了 40+ 种设计模式，构成了一门模式语言（Pattern Language）——模式不是孤立存在的，它们互相支撑、互相弥补。

本文的目标是给你一张"地图"，让你在架构设计时：

  • 知道面对什么问题该参考哪个模式
  • 理解模式之间的依赖和替代关系
  • 建立起上帝视角：让不可见的系统行为变得可见

本文的 pattern 分类体系参考了 Chris Richardson 的 microservices.io 和《Microservices Patterns》一书，结合了实际工程中的落地经验。

一、模式全景图 #

1.1 分层视图 ##

微服务模式可以分为五个层次，从底层的基础设施到顶层的业务决策：

┌──────────────────────────────────────────────────────────────┐
│  第 5 层 · 业务决策层                                          │
│  架构风格选择 → 拆分策略 → 迁移策略                              │
│  (Monolith vs Microservice → DDD/业务能力 → Strangler Fig)     │
├──────────────────────────────────────────────────────────────┤
│  第 4 层 · 服务协作层                                          │
│  通信方式 → API 网关 → 服务发现 → 数据一致性                      │
│  (RPI/Messaging → Gateway/BFF → Registry → Saga/CQRS)         │
├──────────────────────────────────────────────────────────────┤
│  第 3 层 · 韧性与安全层                                        │
│  熔断降级 → 限流过载保护 → 认证授权                              │
│  (Circuit Breaker → Rate Limiting → Access Token)             │
├──────────────────────────────────────────────────────────────┤
│  第 2 层 · 可观测性层                                          │
│  日志聚合 → 指标采集 → 分布式追踪 → 健康检查 → 告警              │
│  (Logging → Metrics → Tracing → Health Check → Alerting)      │
├──────────────────────────────────────────────────────────────┤
│  第 1 层 · 基础设施层                                          │
│  部署方式 → 配置管理 → 服务框架 → 测试策略                       │
│  (Container/K8s → Config → Chassis → Test Pyramid)            │
└──────────────────────────────────────────────────────────────┘

1.2 完整模式清单 ##

标记说明：⭐ = 核心必知模式，🔗 = 模式间有依赖关系

架构风格 ##

模式说明关系
Monolithic architecture单体架构，所有功能在一个部署单元中微服务的起点和对立面
Microservice architecture ⭐微服务架构，独立部署、松耦合目标架构，所有后续模式为此服务

服务拆分（Decomposition） ##

模式说明关系
Decompose by business capability ⭐按业务能力拆分从业务视角出发，与 DDD 互补
Decompose by subdomain ⭐按 DDD 子域拆分从领域模型出发，与业务能力互补
Self-contained Service自包含服务，不依赖同步调用其他服务🔗 降低耦合，为 Saga 铺路
Service per team按团队边界划分服务康威定律的体现

迁移模式（Refactoring） ##

模式说明关系
Strangler Fig ⭐绞杀者模式，逐步替换单体功能增量迁移的核心策略
Anti-corruption layer防腐层，隔离新服务与遗留系统🔗 与 Strangler Fig 配合使用

通信方式（Communication） ##

模式说明关系
Remote Procedure Invocation (RPI)同步 RPC/REST/gRPC 调用🔗 需要服务发现和熔断器配合
Messaging ⭐异步消息通信🔗 驱动 Saga、Event Sourcing、CQRS
Domain-specific protocol领域特定协议（如 Email、SMTP）特定场景下的通信选择
Idempotent Consumer幂等消费者🔗 异步消息场景的必要保障

外部 API ##

模式说明关系
API Gateway ⭐统一入口，路由、认证、限流微服务对外暴露的标准模式
Backend for Frontend (BFF) ⭐按客户端类型提供专属网关🔗 API Gateway 的特化，每端一个

服务发现（Service Discovery） ##

模式说明关系
Service Registry服务注册中心（如 Consul/Nacos/Eureka）🔗 服务发现的基础设施
Client-side discovery客户端从注册中心查询后直连客户端负责负载均衡
Server-side discovery负载均衡器从注册中心查询后代理客户端无感，如 K8s Service
Self registration服务实例自己注册去中心化的注册方式
3rd party registration第三方（如 K8s）代为注册与容器编排平台集成

数据一致性与事务（Data & Transactions） ##

模式说明关系
Database per Service ⭐每个服务独享数据库🔗 微服务数据管理的基石，引出 Saga、CQRS
Shared database多服务共享数据库反模式，但在迁移期可做过渡方案
Saga ⭐用一系列本地事务 + 补偿操作实现分布式事务🔗 解决 Database per Service 带来的分布式事务问题
CQRS ⭐读写分离，各自的模型和存储🔗 解决跨服务查询难题，常与 Event Sourcing 搭配
Event Sourcing ⭐用事件序列代替当前状态存储🔗 消费领域事件，驱动 CQRS 的查询视图
Domain event状态变更时发布事件🔗 事件驱动协作的基础，支撑 Saga、CQRS、Event Sourcing
API Composition从多个服务收集数据后在内存中组合🔗 简单查询模式，复杂度升高后升级为 CQRS

可靠消息传递（Transactional Messaging） ##

模式说明关系
Transactional outbox ⭐业务数据和消息写入同一数据库事务🔗 保证"写业务数据 + 发消息"的原子性
Transaction log tailing读取数据库事务日志捕获变更🔗 Outbox 的替代方案，如 Debezium + binlog
Polling publisher定时轮询未发送的消息比 Log Tailing 简单，但实时性差

可靠性（Reliability） ##

模式说明关系
Circuit Breaker ⭐熔断器，失败超过阈值时快速失败🔗 防止级联故障，与 RPI 通信方式紧密相关
Rate Limiting限流，拒绝超额请求防止资源耗尽
Retry重试，失败后自动重试需要幂等消费配合
Bulkhead舱壁隔离，为不同服务/资源池分配独立线程池防止一个服务拖垮整条链路
Timeout超时控制，超过约定时间直接失败最基础也是最容易被忽视的韧性手段

可观测性（Observability） ##

模式说明关系
Log aggregation ⭐集中式日志聚合（如 ELK/Loki）可观测性的基础
Application metrics ⭐应用指标采集（如 Prometheus）定量的健康度量
Distributed tracing ⭐分布式追踪（如 Jaeger/Zipkin）🔗 串联所有服务的请求链路
Health check API ⭐健康检查端点（Liveness/Readiness）🔗 K8s/负载均衡的自动运维入口
Audit logging审计日志，记录用户操作安全与合规需求
Exception tracking异常追踪，集中聚合和去重告警如 Sentry
Log deployments and changes部署和变更日志关联系统异常与变更记录
Synthetic testing合成测试，定期运行测试用例主动探测，在用户发现问题前发现

安全（Security） ##

模式说明关系
Access Token访问令牌（JWT/OAuth2）在服务间传递🔗 通常与 API Gateway 配合
mTLS服务间双向 TLS 认证🔗 常通过 Service Mesh 零代码实现

部署（Deployment） ##

模式说明关系
Service instance per Container ⭐每个实例一个容器🔗 当前主流，K8s 的标准部署方式
Service instance per VM每个实例一个虚拟机隔离性好，启动慢
Multiple service instances per host一个主机部署多个实例简单，有资源争抢风险
Serverless deployment无服务器部署极致弹性，事件驱动
Service deployment platform自动化部署平台🔗 编排以上所有模式

跨切面（Cross-cutting） ##

模式说明关系
Microservice Chassis微服务底座框架，封装通用能力🔗 如 Spring Boot、Go kit 等
Externalized configuration配置外化，代码与配置分离🔗 如配置中心、环境变量
Service Template服务模板，新服务快速启动降低新建服务的成本

测试（Testing） ##

模式说明关系
Consumer-driven contract test消费者驱动的契约测试🔗 如 Pact，保证接口兼容性
Service component test服务组件测试，隔离测试一个服务用 test double 替代依赖服务
Consumer-side contract test消费者端契约测试验证消费者对接口的兼容性

UI 组合 ##

模式说明关系
Server-side page fragment composition服务端组装页面片段后端驱动的 UI 聚合
Client-side UI composition客户端组装 UI 片段如微前端（Micro Frontends）

二、核心模式详解 #

2.1 Saga — 分布式事务的最终一致性方案 ##

解决的问题：Database per Service 场景下，一个业务流程跨多个服务，没有 ACID 事务保证。

核心思想：将分布式事务拆解为一组有序的本地事务，每个本地事务有对应的补偿操作。失败时逆序执行补偿。

正常流程：  T1 → T2 → T3 → T4
                ↓ (T3 失败)
补偿流程：  C1 ← C2 ← C3

两种实现方式：

方式特点适用场景
Choreography（编排）各服务通过事件驱动自主决策，无中心协调器简单流程、参与者少
Orchestration（编制）中心 Saga 协调器指挥每个参与者复杂流程、需显式状态管理

工程实践要点：

  1. 补偿操作的幂等性：补偿可能被重复调用，必须保证幂等
  2. 关键节点标记：某些步骤（如用户确认支付）完成后不应回滚，而是异步重试后续步骤
  3. 补偿失败的处理：补偿操作也可能失败，此时需要人工介入或异步死信重试
  4. 隔离性问题：Saga 在执行过程中，其他事务可能读到中间状态（读未提交），需要通过语义锁或补偿事务的隔离性策略缓解

2.2 CQRS — 读写分离的查询优化 ##

解决的问题：跨多个服务的复杂查询（join/聚合）在 Database per Service 下无法直接执行 SQL。

核心思想：将**命令（写）和查询（读）**分离为不同的模型，各自使用最优的存储方案。

写路径：  Command → MySQL（权威数据源）
                         ↓ binlog/CDC
读路径：  Query → ElasticSearch（查询优化视图）→ 返回结果

CQRS 的多种实现层次：

层次写模型读模型复杂度
简单MySQL 主库MySQL 从库低
中级MySQL（业务数据）专用查询视图（物化视图）中
高级Event Store（事件存储）多套物化视图（ES + Redis + OLAP）高

何时该用 CQRS：

  • 查询模式与写入模式差异大（如写入是表单式的，查询是多维分析）
  • 读远多于写，需要独立扩容
  • 需要从多个服务聚合数据来构建一个查询结果
  • 简单场景下先用 API Composition，复杂度升高再升级到 CQRS

2.3 Event Sourcing — 用事件记录代替状态存储 ##

解决的问题：需要审计跟踪、时间旅行查询、可靠的事件发布。

核心思想：不存储当前状态，而是存储导致状态变化的事件序列。当前状态通过重放事件计算得出。

传统方式：  ┌──────────┐
           │ order {   │
           │  id: 123  │
           │  status:  │
           │  "paid"   │
           │}          │
           └──────────┘

Event       ┌──────────────────────────────────────┐
Sourcing：   │ event_id │ type       │ data        │
            │ 1        │ Created    │ {...}       │
            │ 2        │ Confirmed  │ {...}       │
            │ 3        │ Paid       │ {...}       │
            └──────────────────────────────────────┘
            重放 1→2→3 即可得到当前状态：Paid

与 CQRS 的关系：Event Sourcing 天然适合作为 CQRS 的写模型 —— 每次事件写入后，异步投影（project）到各个查询视图。

适用场景：审计系统、金融交易、协作编辑、复杂领域逻辑。

2.4 API Gateway / BFF — 服务能力的统一出口 ##

解决的问题：客户端直接调用微服务导致耦合过紧、客户端需要知道所有服务地址、多次请求开销大。

API Gateway：单一入口，聚合后端服务，处理横切关注点（认证、限流、日志）。

BFF：按客户端类型（iOS / Android / Web）提供专属的聚合层，各自裁剪响应。

        iOS ──→ BFF-iOS ──→ ┌──────────┐
                             │ Service A │
Android ──→ BFF-Android ──→ │ Service B │
                             │ Service C │
  Web  ──→ BFF-Web ───────→ └──────────┘

选择建议：服务少（< 5 个）用 API Gateway 即可；服务多、客户端差异大（移动端 vs PC 端），用 BFF。

2.5 Circuit Breaker — 防止级联故障 ##

解决的问题：一个服务故障导致调用方线程阻塞堆积，进而拖垮上游，形成雪崩。

三种状态：

         ┌─────────┐
         │ CLOSED  │ ←── 正常状态，请求正常通过
         └────┬────┘
              │ 失败数超过阈值
              ▼
         ┌─────────┐
         │  OPEN   │  ←── 熔断状态，请求直接返回失败
         └────┬────┘
              │ 等待时间窗口结束
              ▼
         ┌─────────┐
         │HALF-OPEN│ ←── 半开状态，允许少量请求试探
         └────┬────┘
     成功      │      失败
      ↙        └──────→ 回到 OPEN
  回到 CLOSED

实践要点：

  • 熔断要配合超时：别让一个慢请求永远占着线程
  • 熔断后的降级：返回缓存数据 / 默认值 / 降级页，而不是直接报错
  • 慢调用比例熔断比仅看失败率更有效：慢请求就是正在积压的故障

2.6 Transactional Outbox — 可靠的消息发送 ##

解决的问题：如何保证"写业务数据"和"发消息"两个操作的原子性？如果先写数据再发消息，发消息失败会导致不一致。

核心思想：在同一个数据库事务中，将消息写入 outbox 表；一个独立的消息中继进程轮询 outbox 表并发送消息到消息代理。

BEGIN;
  INSERT INTO orders (...);
  INSERT INTO outbox (event_type, payload) VALUES ('order_created', {...});
COMMIT;
                    ↓
         [Message Relay 进程]
         轮询 outbox → 发送到 Kafka/RabbitMQ → 标记为已发送

替代方案：Transaction Log Tailing（如 Debezium 监听 MySQL binlog），更实时但运维更复杂。

四、上帝视角：可观测性体系 #

“You can’t fix what you can’t see.” — 微服务架构的核心挑战之一，就是让不可见变得可见。

4.1 三级可观测性模型 ##

┌────────────────────────────────────────────────────────────┐
│  第一级：用户感知层                                          │
│  用户体验指标：页面加载时间、API 响应时间、交易成功率              │
│  工具：Synthetic Testing, RUM (Real User Monitoring)         │
├────────────────────────────────────────────────────────────┤
│  第二级：系统层                                              │
│  三大支柱：Logs + Metrics + Traces → 统一可观测平台             │
│  工具：Prometheus + Grafana + Loki + Tempo/Jaeger             │
├────────────────────────────────────────────────────────────┤
│  第三级：基础设施层                                           │
│  资源指标：CPU、内存、网络、磁盘、容器/Pod 状态                  │
│  工具：Node Exporter, cAdvisor, K8s Metrics API              │
└────────────────────────────────────────────────────────────┘

4.2 三大支柱的职责分工 ##

场景：用户反馈"订单创建失败"

  1. 告警平台收到 "order_create_error_rate > 1%" 的告警（Metrics）
                              │
                              ▼
  2. 查看 Grafana 仪表盘，定位到 Order Service 的错误率飙升（Metrics）
                              │
                              ▼
  3. 用 Trace ID 关联到失败请求的完整链路，发现 Payment Service 超时（Tracing）
                              │
                              ▼
  4. 查看 Payment Service 的日志，发现数据库连接池耗尽（Logging）
                              │
                              ▼
  5. 查看 Deployment Log，发现 10 分钟前 Payment Service 刚发布新版本
                              │
                              ▼
  6. 回滚到上一个版本，问题解决 ← 整个排障链路在 5 分钟内完成

4.3 告警 vs. 调试的分层策略 ##

这是架构师建立上帝视角的关键洞察：不是所有的指标都应该触发告警。

信号类型用途示例
告警指标反映用户体验，触发 OnCallHTTP 错误率、响应延迟 P99、消息积压量
调试指标排障时使用，不需要告警CPU 使用率、内存、线程池状态

原则：告警应该基于 SLO（Service Level Objective，服务等级目标），而非基础设施指标。CPU 飙升不一定是问题（可能是正常扩容），但用户请求错误率上升一定是问题。

三级告警框架：

层级告警对象示例
系统级整个应用总错误率、总 QPS
子系统级一组相关服务订单子系统的成功率
服务级单个服务支付服务的 P99 延迟

这样在设计告警时，可以立刻知道影响范围——是某个服务坏了，还是整个子系统挂了，还是系统级的灾难。

4.4 分布式追踪：让请求链路可视化 ##

请求 Trace: abc123
│
├─ API Gateway (2ms)
│   └─ Order Service (45ms)
│       ├─ Inventory Service (30ms)  ← 瓶颈在这！
│       │   └─ DB Query (28ms)
│       └─ Payment Service (12ms)
│           └─ DB Query (10ms)
│
总耗时：47ms

追踪实现的关键点：

  1. Trace ID 贯穿所有服务：在 API Gateway 或首跳服务生成，通过 HTTP Header（如 X-Trace-ID）或消息 Header 传递
  2. Span 记录每个服务内的操作：包括调用下游、数据库查询、缓存操作等
  3. 采样策略：
    • Head-based：请求入口处随机采样（简单，但有概率遗漏错误）
    • Tail-based：在所有 Span 完成后决策（保留所有错误和慢请求）
  4. 关联日志和追踪：在每行日志中打印 Trace ID，可以实现"从告警到日志"的一键跳转

4.5 健康检查：自动运维的入口 ##

┌────────────────────────────────────────┐
│  GET /health/liveness                  │
│  → 200 OK / 503 Service Unavailable    │
│  "我活着吗？" → 不健康就 kill 重启       │
├────────────────────────────────────────┤
│  GET /health/readiness                 │
│  → 200 OK / 503 Service Unavailable    │
│  "我能接流量吗？" → 不健康就摘流           │
└────────────────────────────────────────┘

Readiness 检查什么：

  • 数据库连接是否正常
  • 缓存是否预热完毕
  • 依赖的下游服务是否可达
  • 关键线程池是否健康

反模式：把 Liveness 和 Readiness 设成同一个端点，或者让 Liveness 检查依赖服务（可能导致级联重启）。

4.6 上帝视角仪表盘设计 ##

一个好的架构师仪表盘应该能在 一个屏幕 内回答以下问题：

┌──────────────────────────────┐  ┌──────────────────────────────┐
│  🔴 总请求错误率: 0.2%       │  │  📊 各服务 QPS (排行榜)       │
│  🟢 P99 延迟: 120ms          │  │  order-svc:  850/s            │
│  🟡 消息积压: 120条 (正常)    │  │  pay-svc:    620/s            │
│  🟢 健康检查: 32/32 ✓        │  │  inventory:  340/s            │
├──────────────────────────────┤  ├──────────────────────────────┤
│                              │  │                               │
│  调用链路拓扑图 (Service Map) │  │  各服务 SLA 达成率             │
│  ┌─→ A ─→ B ─→ D            │  │  order-svc:  99.95%   ✅      │
│  │     ↓      ↑              │  │  pay-svc:    99.82%   ✅      │
│  └─→ C ──────┘              │  │  notify-svc: 97.30%   ❌      │
│                              │  │                               │
├──────────────────────────────┤  ├──────────────────────────────┤
│  最近部署记录                  │  │  实时告警列表                  │
│  pay-svc v2.3.1 10分钟前     │  │  🔴 notify-svc 错误率 > 1%    │
│  order-svc v1.8.0 2小时前    │  │  🟡 pay-svc P99 延迟升高      │
└──────────────────────────────┘  └──────────────────────────────┘

核心原则：从一个高层的告警出发，能层层下钻——点击错误率 → 找到异常服务 → 查看该服务的 Trace → 定位到具体 Span → 跳转到关联日志。

五、架构决策速查表 #

5.1 “我该用哪个模式？” ##

你面临的问题推荐模式注意事项
怎么把单体拆成微服务？Decompose by business capability先划分业务能力，再用 DDD 子域精化
如何逐步迁移单体？Strangler Fig用 ACL 隔离新老系统
跨服务查询/join 怎么办？先 API Composition，复杂后升级 CQRS不要过早优化
跨服务事务怎么办？Saga优先用 Orchestration，简单场景用 Choreography
服务间同步调用挂了怎么办？Circuit Breaker + Timeout超时是第一道防线，熔断是第二道
如何保证发消息不丢？Transactional OutboxCDC/Debezium 是更高级的方案
如何知道哪个服务出了故障？Service Registry + Health CheckK8s 自带就够用，自建服务可用 Consul
如何排查"请求为什么慢"？Distributed Tracing先在入口生成 Trace ID
如何让监控告警不烦人？基于 SLO 的告警告警"用户感知的异常"，而非基础设施波动
新服务怎么快速搭建？Service Template + Chassis模板里内含了可观测性、配置、健康检查

5.2 架构评审检查清单 ##

在一个微服务系统上线前，架构师应该检查以下问题：

拆分合理性

  • 每个服务的边界是否清晰？是否满足高内聚低耦合？
  • 是否有超大服务（“上帝服务"反模式）？是否有过小的纳米服务？

通信与韧性

  • 同步调用链路的超时时间是否合理配置？
  • 级联故障的风险点是否都有熔断保护？
  • 异步消息的消费者是否都是幂等的？

数据一致性

  • 一致性的 SLO 是什么？强一致还是最终一致？
  • Saga 的补偿操作是否完备且幂等？
  • 跨服务查询是否经过优化？（是否直接 join 了对方的数据库？）

可观测性

  • 是否所有服务都输出了结构化的日志（含 Trace ID）？
  • 是否有关键业务指标（请求量、错误率、延迟）采集？
  • Liveness 和 Readiness 探针是否合理配置？
  • 是否有一个统一视图能在一屏内看到系统全貌？

部署与运维

  • 是否可以独立部署？是否做到了真正的独立发布？
  • 配置是否外化？修改配置是否需要重新部署？
  • 是否有自动化回滚能力？

总结 #

微服务架构的本质不是"把服务拆小”，而是在分布式约束下，让系统仍然可控。

这张速查表的组织逻辑是：

  1. 从拆分开始：用 DDD 找到边界，用 Strangler Fig 平稳迁移
  2. 从通信深入：选择合适的同步/异步方式，用 Circuit Breaker 保护同步链路，用 Outbox 保证异步可靠
  3. 用 Saga 和 CQRS 解决数据问题：分布式事务和跨服务查询是微服务最痛的两块，也是最需要模式指导的领域
  4. 用可观测性建立上帝视角：Logs + Metrics + Traces 三位一体，SLO 驱动告警，Trace ID 贯穿始终

最后一点：模式是工具箱，不是教条。不是所有的系统都需要 CQRS 和 Event Sourcing。简单场景用简单方案，复杂度升高了再引入更高级的模式。架构师的判断力体现在：知道什么模式存在，更知道什么模式暂时不需要。
