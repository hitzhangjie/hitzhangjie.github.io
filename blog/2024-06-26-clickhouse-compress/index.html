ClickHouse 列压缩机制：从 CODEC 到 Part 层面的压缩策略

Posted 2024-06-26 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

背景 #

使用 ClickHouse 做分析型负载时，存储成本是一个绕不开的话题。ClickHouse 以列式存储著称，而列存天然适合压缩——同一列的数据类型相同、值域相近，压缩算法可以轻松达到高压缩比。但在实际使用中，关于压缩有几个常见疑问：业务侧自己压缩后写入，还是让 ClickHouse 自己压缩？如何在表定义中指定压缩算法？如果不指定，ClickHouse 会不会自动压缩？压缩后的列还能用来做查询过滤吗？

这篇文章逐一梳理这些问题，帮你理清 ClickHouse 的压缩机制。

ClickHouse 压缩机制全景 #

业务侧压缩 vs ClickHouse 内置压缩 ##

一个很自然的想法是：既然数据最终要压缩存储，那业务侧先把数据压了再写进去，不是更省带宽和磁盘吗？

但实际推荐的做法是方案 2——业务侧不压缩，直接写入，让 ClickHouse 利用自身的列压缩能力来压缩存储。原因有几个：

  1. 压缩效率更高：ClickHouse 内部使用高度优化的压缩算法，会根据列的数据特征自动选择最优策略，压缩效果通常好于业务侧的通用压缩。
  2. 简化业务逻辑：如果业务侧自行压缩，需要在代码中增加压缩/解压逻辑，增加维护成本。利用 ClickHouse 自身能力，业务侧可以更专注于业务本身。
  3. 保证一致性：所有数据由 ClickHouse 统一压缩，格式一致，有利于后续数据处理和分析。
  4. 灵活性：ClickHouse 提供丰富的压缩选项，可根据实际需求灵活配置。

当然也有例外——如果网络带宽是瓶颈而 CPU 充足，业务侧预压缩减少传输量也是一个可选的优化方向。但对大多数场景来说，把压缩交给 ClickHouse 就行。

表定义中指定列压缩：CODEC ##

如果某列数据量特别大（比如文章正文、日志内容），可以在建表时通过 CODEC 语法为它指定压缩算法：

CREATE TABLE articles (
    name String,
    sex String,
    age UInt8,
    article String CODEC(ZSTD(10))
) ENGINE = MergeTree()
ORDER BY tuple();

这里 article 列使用了 CODEC(ZSTD(10))：

  • ZSTD：Zstandard 压缩算法，ClickHouse 推荐的通用压缩算法。
  • 10：压缩级别，取值范围 1-22。数值越大压缩率越高，但压缩/解压耗时也越长。级别 10 是一个比较平衡的选择。

如果数据模式比较简单，也可以考虑用 LZ4 或 LZ4HC——压缩率不如 ZSTD，但压缩/解压速度更快。

没指定 CODEC 就不会压缩吗？ ##

很多人以为不给列指定 CODEC，ClickHouse 就不会压缩。实际上 ClickHouse 在 Part 层面会对数据进行默认压缩。

ClickHouse 的数据组织方式是 Table → Partition → Part。Part 是磁盘存储的基本单元。当数据写入后累积成 Part 时，ClickHouse 会在 Part 合并过程中进行优化：

  1. 列文件压缩：即使没有显式指定 CODEC，ClickHouse 也会对列数据进行压缩（生成 .bin 文件）。
  2. 列文件合并：将多个小列文件合并成大文件，减少文件数量。
  3. 索引标记：为列数据创建稀疏索引，提高查询效率。
  4. Part 元数据压缩：压缩 Part 的元数据信息。

默认使用的压缩算法是 LZ4（可在配置文件中修改）。LZ4 的压缩比不如 ZSTD，但压缩和解压速度极快，适合对大量数据做快速压缩。对于 String、Enum、UUID 类型的列，如果没有显式指定 CODEC，ClickHouse 就会用 LZ4 兜底。

总结一下区别：

  • 不指定 CODEC：使用默认 LZ4 压缩，在 Part 层面自动进行，压缩率一般但速度快。
  • 指定 CODEC（如 ZSTD）：在列级别用指定算法压缩，压缩率更高，适合大数据量的列。

压缩列还能用来查询过滤吗？ ##

能。ClickHouse 在执行查询时会自动解压列数据，获取原始内容用于过滤、排序、聚合。从查询语法和使用方式上看，压缩列和未压缩列没有区别：

-- 压缩列完全可以在 WHERE 中使用
SELECT name, article
FROM articles
WHERE article LIKE '%ClickHouse%';

不过需要注意，使用压缩列查询时因为要先解压，会增加 CPU 开销。如果查询过滤条件无法命中索引，就需要扫描并解压整个列数据，性能会受影响。所以如果某列频繁用于查询过滤，可以考虑不对它启用高压缩级别的 CODEC。

总结 #

ClickHouse 的压缩机制分两层：

  • 列级 CODEC：建表时显式指定，适合大数据量列，自定义压缩算法和级别。
  • Part 级默认压缩：LZ4 兜底，对未指定 CODEC 的 String/Enum/UUID 列自动压缩。

实践中，对数据量大的列指定 CODEC（如 ZSTD）可以显著节省存储，而查询时 ClickHouse 会自动解压，对 SQL 透明。合理的策略是：对不常参与查询过滤的大字段开高压缩比，对频繁查询的字段保持默认 LZ4 或干脆不压缩。
