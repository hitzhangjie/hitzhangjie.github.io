Linux 信号传递中的线程选择机制

Posted 2025-09-12 00:00 +0800 by ZhangJie ‐ 1 min read

分享:  

背景 #

在多线程程序中，当向一个进程发送信号时，你可能会想：信号会被哪个线程处理？答案是内核会根据发送方式的不同，将信号放入不同的队列，并智能地选择一个合适的线程来处理。

这背后的实现涉及 send_signal、complete_signal 等关键函数，以及 PIDTYPE_TGID 与 PIDTYPE_PID 两种信号目标类型的区分。本文从内核源码层面梳理这个机制。

信号发送路径 #

以子进程状态变化导致父进程收到 SIGCHLD 为例，完整的内核调用路径如下：

do_notify_parent_cldstop()
  -> __group_send_sig_info(SIGCHLD, &info, parent)
    -> send_signal(sig, info, p, PIDTYPE_TGID)
      -> __send_signal(sig, info, t, type, force)

信号队列的选择：共享 vs 私有 #

在 __send_signal 中，根据信号的目标类型决定将信号放入哪个队列：

pending = (type != PIDTYPE_PID) ? &t->signal->shared_pending :
                                  &t->pending;

两类队列的区别：

目标类型信号队列含义
PIDTYPE_PIDt->pending发送给特定线程，放入线程私有的 pending 队列
PIDTYPE_TGIDt->signal->shared_pending发送给整个进程（线程组），放入进程共享的 pending 队列

而在 __group_send_sig_info 中明确使用了 PIDTYPE_TGID：

return send_signal(sig, info, p, PIDTYPE_TGID);

这意味着对于 SIGCHLD 这类进程级信号，会被放入共享队列 signal->shared_pending，进程中的任意线程都可以处理它。

线程选择机制 #

信号已经放入了共享队列，但具体由哪个线程来处理呢？这是 complete_signal 函数的职责：

if (wants_signal(sig, p))
    t = p;  // 主线程优先
else {
    // 寻找其他合适的线程
    t = signal->curr_target;
    while (!wants_signal(sig, t)) {
        t = next_thread(t);
        // ...
    }
}

选择逻辑：

  1. 主线程优先：如果主线程（group_leader）没有阻塞该信号，就选它。
  2. 轮询查找：否则从 curr_target 开始，遍历线程组链表，找到第一个没有阻塞该信号且愿意接收的线程。
  3. 未阻塞条件：线程可以通过 sigprocmask 阻塞特定信号，被阻塞的信号不会被选中处理。

这种设计提供了很好的负载均衡效果——如果主线程繁忙或阻塞了某些信号，其他线程可以分担信号处理工作。

ptrace 场景下的信号拦截 #

当目标线程正在被调试器（ptracer）跟踪时，信号处理的行为会发生变化：

  1. 信号不直接执行：被跟踪线程收到信号后，不会立即执行信号 handler。
  2. 进入停止状态：线程进入 signal->deliver->stop 状态，暂停执行，同时通知 ptracer。
  3. ptracer 决定：调试器通过 ptrace(PTRACE_CONT, ...) 等请求来决定：
    • 忽略该信号，让目标线程继续执行（就像没收到过信号）。
    • 将信号注入目标线程，让其正常处理。
    • 修改寄存器或内存后继续执行。

这是调试器实现断点、单步等功能的底层基础。当你在 GDB 中看到程序收到 SIGSEGV 停住时，就是 ptrace 在信号传递路径上拦截了它。

总结 #

  1. 信号队列有两种：PIDTYPE_PID → 线程私有队列 t->pending；PIDTYPE_TGID → 进程共享队列 signal->shared_pending。
  2. 线程选择策略：主线程优先，再轮询线程组找第一个合适的线程，实现了基础的负载均衡。
  3. ptrace 拦截：被跟踪线程的信号会被暂停并交给调试器决定，这是调试器控制被调试程序的核心机制。
  4. 理解这些机制有助于在多线程程序和调试场景中预期信号的行为，避免信号丢失或被意外线程处理的困惑。
