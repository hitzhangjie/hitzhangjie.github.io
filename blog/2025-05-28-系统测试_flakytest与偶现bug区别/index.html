Flaky Test 与偶现 Bug：别再混为一谈

Posted 2025-05-28 00:00 +0800 by ZhangJie ‐ 2 min read

分享:  

背景 #

日常开发中经常会遇到两类令人头疼的问题：一是 CI 跑着跑着红了，重试一次又绿了；二是用户反馈了一个 bug，但自己怎么都复现不出来。前者我们习惯叫"flaky test"，后者叫"偶现 bug"。

它们表面上都是"偶现"，于是很多人（包括之前的我）会下意识地认为它们是一回事——flaky test 就是偶现 bug 在测试里的表现形式。但仔细想想，这个等式真的成立吗？

两个概念，分别定义 #

Flaky Test ##

Flaky test 是指在相同代码、相同环境下，执行结果不确定的测试——有时通过，有时失败。核心特征是测试结果的非确定性。

常见根因：

  • 测试隔离不足：测试之间共享可变状态（全局变量、数据库行、文件系统），执行顺序不同导致结果不同。
  • 时序依赖：测试中 hardcode 了 sleep 或超时时间，在负载波动的 CI 机器上可能不够用。
  • 外部依赖不稳定：依赖的第三方服务、网络、磁盘 I/O 存在抖动。
  • 测试数据未清理：前一次测试的残留数据影响了下一次测试的初始状态。
  • 断言不够精确：比如使用了宽松的 assertEventually 但超时设得太短。
  • 环境差异：本地能过，CI 上不能过——时间 zone、locale、CPU 架构、内存大小等微小差异。

关键点：多数 flaky test 的根因在测试侧——测试写得有问题，而非产品代码有 bug。

偶现 Bug ##

偶现 bug 是指产品代码中确实存在、但在特定条件下才触发的缺陷。核心特征是难以稳定复现。

常见根因：

  • 竞态条件：多 goroutine/线程对共享资源的访问顺序不确定，特定交错下出错。
  • 资源泄漏：内存、文件描述符、连接池耗尽，在长时间运行后才暴露。
  • 外部系统交互的时序问题：分布式系统中，消息到达顺序、网络分区、超时重试的边界情况。
  • 未处理的异常路径：罕见的错误码、异常输入组合、极限流量下的降级逻辑。
  • 内存损坏：C/C++ 中的 use-after-free、buffer overflow，Go 中 unsafe 包的不当使用。

关键点：偶现 bug 的根因一定在产品侧——代码本身有缺陷，只是触发条件苛刻。

核心差异：不止是"描述对象不同" #

表面上看，一个在说测试，一个在说产品，泾渭分明。但真正容易混淆的地方在于归因方向和发现场景。

归因方向不同 ##

Flaky Test偶现 Bug
首要怀疑对象测试代码、测试数据、测试环境产品代码
修复方式修测试（加固隔离、消除时序依赖、mock 外部服务）修产品代码
是否意味着产品有 bug不一定——大部分只是测试写得烂一定——否则不会有这个 bug

当一个测试时过时挂时，首先要排查的是测试本身——是不是共享了状态、是不是依赖了时序、是不是没做 cleanup。只有在排除了测试侧问题之后，才把目光转向产品代码。

反过来，用户反馈的偶现 bug 跟自动化测试没有半毛钱关系——它就是跑在真实环境里的真实问题。

发现场景不同 ##

这是最容易被忽略的维度：

  • Flaky test 的发现场景很窄——它只在自动化测试执行期间被观察到。你跑go test，它挂了；再跑一次，它绿了。你骂一句"flaky"，加个 retry，然后继续写代码。
  • 偶现 bug 的发现场景很广——它可能出现在任何地方：
    • QA 手动测试时偶然触发
    • 用户在生产环境中遇到，提了工单
    • 监控告警里出现了一次诡异的 spike，看日志才发现某个 corner case
    • 开发者在本地调试另一个功能时意外踩到

也就是说，偶现 bug 不依赖测试套件来暴露自己。很多偶现 bug 根本没有对应的自动化测试——它们潜伏在代码里，直到某个特定的用户操作、流量模式、系统状态组合把它唤醒。

两者的交集 ##

当然，它们确实有交集：一个 flaky test 有可能是触碰到了真实的偶现 bug。

比如你的 test 偶发性地过不了，排查了一圈发现测试没毛病，最后定位到是 sync.Map 在某个特定并发模式下有数据竞争——这就是一个通过 flaky test 暴露出来的偶现 bug。但需要注意的是：

  1. 这种情况在实际中占少数。大部分 flaky test 是测试自身的问题。
  2. 即便在这种情况下，flaky test 仍然是对测试行为的描述，偶现 bug 是对产品缺陷的描述——两个概念在范畴上没有合并。

现实中的判断流程 #

遇到一个"偶发失败"时，该怎么判断该往哪个方向排查？

偶发失败
  │
  ├─ 是在自动化测试中发现的吗？
  │   ├─ 是 → 优先排查测试侧
  │   │   ├─ 测试是否有共享状态？
  │   │   ├─ 测试是否有时序依赖？
  │   │   ├─ 测试数据是否正确 cleanup？
  │   │   ├─ 断言是否精确？
  │   │   └─ 以上都排除了 → 可能是产品 bug，进入偶现 bug 排查流程
  │   │
  │   └─ 否（用户反馈/QA/线上） → 这就是偶现 bug
  │       └─ 进入产品侧排查：日志、trace、core dump、压测复现

这个流程的核心思想就一句话：先怀疑自己写的测试，再怀疑自己写的代码。

总结 #

Flaky test 和偶现 bug 虽然都表现为"偶发"，但本质上是两个不同层次的概念：

  1. Flaky test 描述的是测试行为的不稳定，首要归因对象是测试套件本身（测试代码、测试数据、测试环境），发现场景限于自动化测试执行期间。
  2. 偶现 bug 描述的是产品代码的缺陷，归因对象一定是产品代码，发现场景不限于测试——用户反馈、QA 测试、线上监控都可能暴露它。
  3. 两者有交集（flaky test 可能触碰到了真实 bug），但交集占比不高，大部分 flaky test 就是测试写得不够好。
  4. 面对偶发失败，养成先排查测试、再排查产品的习惯，能省下大量在错误方向上钻牛角尖的时间。
